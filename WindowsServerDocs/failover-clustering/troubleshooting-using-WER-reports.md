# <a name="troubleshooting-a-failover-cluster-using-windows-error-reporting"></a>Устранение неполадок отказоустойчивого кластера с помощью отчетов об ошибках Windows 

> Относится к: Windows Server 2019 г., Windows Server 2016, Windows Server

Windows Error Reporting (WER) — это инфраструктура гибкие обратной связи на основе событий, предназначен для опытных администраторов или 3 уровня поддержки сбора сведений о проблемах оборудования и программного обеспечения, которые Windows может обнаруживать, сообщающие сведения в корпорацию Майкрософт, и предоставить пользователям все доступные решения. Это [ссылку](https://docs.microsoft.com/powershell/module/windowserrorreporting/) приводится описание и синтаксис всех командлетов WindowsErrorReporting.

Сведения об устранении неполадок, представленные ниже, будет полезно для устранения неполадок Дополнительные вопросы, переданных на обработку и потребовать данных, отправляемых в корпорацию Майкрософт для рассмотрения.

## <a name="enabling-event-channels"></a>Включение каналы событий

При установке Windows Server, многие каналы событий включены по умолчанию. Но иногда при диагностике проблем, мы хотим иметь возможность включить некоторые из этих каналов событий, так как он поможет в рассмотрение и диагностика проблем с системой.

Можно было включить каналы дополнительных событий на каждом серверном узле в кластере, при необходимости; Тем не менее это создает две проблемы:

1. Необходимо не забыть включить те же каналы событий на каждый новый узел сервера, добавляемого в кластер.
2. При диагностике, может быть неудобен для включения каналов определенного события, воспроизведите ошибку и повторите эту процедуру, пока не основная причина.

Чтобы избежать этих проблем, вы можете включить каналы событий при запуске кластера. Список каналов событий, включенные в кластере можно настроить с помощью открытого свойства **EnabledEventLogs**. По умолчанию включены следующие каналы событий:

```powershell
PS C:\Windows\system32> (get-cluster).EnabledEventLogs
```

Ниже приведен пример выходных данных.
```
Microsoft-Windows-Hyper-V-VmSwitch-Diagnostic,4,0xFFFFFFFD
Microsoft-Windows-SMBDirect/Debug,4
Microsoft-Windows-SMBServer/Analytic
Microsoft-Windows-Kernel-LiveDump/Analytic
```

**EnabledEventLogs** свойство является мультистроку (multistring), где каждая строка представляет в виде: **имя канала, уровень ведения журнала, ключевое слово маска**. **Маску ключевого слова** может быть шестнадцатеричного (префикс 0 x), восьмеричное (префикс 0) или номер десятичное число (без префикса). Например добавить новый канал событий в список и настроить оба **уровень ведения журнала** и **маску ключевого слова** запуском:

```powershell
(get-cluster).EnabledEventLogs += "Microsoft-Windows-WinINet/Analytic,2,321"
```

Если вы хотите задать **уровень ведения журнала** , но сохранить **маску ключевого слова** значение по умолчанию, воспользуйтесь одним из следующих команд:

```powershell
(get-cluster).EnabledEventLogs += "Microsoft-Windows-WinINet/Analytic,2"
(get-cluster).EnabledEventLogs += "Microsoft-Windows-WinINet/Analytic,2,"
```

Если вы хотите сохранить **уровень ведения журнала** значение по умолчанию, но задать **маску ключевого слова** можно выполните следующую команду:

```powershell
(get-cluster).EnabledEventLogs += "Microsoft-Windows-WinINet/Analytic,,0xf1"
```

Если вы хотите оставить оба **уровень ведения журнала** и **маску ключевого слова** значения по умолчанию, можно запустить любой из следующих команд:

```powershell
(get-cluster).EnabledEventLogs += "Microsoft-Windows-WinINet/Analytic"
(get-cluster).EnabledEventLogs += "Microsoft-Windows-WinINet/Analytic,"
(get-cluster).EnabledEventLogs += "Microsoft-Windows-WinINet/Analytic,,"
```

Эти каналы событий будет включена на каждом узле кластера, при запуске службы кластеров, или каждый раз, когда **EnabledEventLogs** изменении свойства.

## <a name="gathering-logs"></a>Идет сбор журналов

После включения каналы событий, можно использовать **DumpLogQuery** для сбора журналов. Свойство типа ресурса с общедоступным **DumpLogQuery** представляет собой значение mutistring. Каждая строка представляет [запрос XPATH, как описано здесь](https://msdn.microsoft.com/library/windows/desktop/dd996910(v=vs.85).aspx).

При устранении неполадок, если вам нужно собрать дополнительные каналы, вы можете изменить **DumpLogQuery** свойства, добавив дополнительные запросы или изменения списка.

Чтобы сделать это, сначала проверить запрос XPATH с помощью [get-WinEvent](https://docs.microsoft.com/powershell/module/Microsoft.PowerShell.Diagnostics/Get-WinEvent?view=powershell-5.1) командлет PowerShell:

```powershell
get-WinEvent -FilterXML "<QueryList><Query><Select Path='Microsoft-Windows-GroupPolicy/Operational'>*[System[TimeCreated[timediff(@SystemTime) &gt;= 600000]]]</Select></Query></QueryList>"
```

Затем добавьте запрос так, чтобы **DumpLogQuery** свойство ресурса:

```powershell
(Get-ClusterResourceType -Name "Physical Disk".DumpLogQuery += "<QueryList><Query><Select Path='Microsoft-Windows-GroupPolicy/Operational'>*[System[TimeCreated[timediff(@SystemTime) &gt;= 600000]]]</Select></Query></QueryList>"
```

И если вы хотите получить список запросов, выполните:
```powershell
(Get-ClusterResourceType -Name "Physical Disk").DumpLogQuery
```
