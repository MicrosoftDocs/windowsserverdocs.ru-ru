# Настройка контроль доступа пользователей и разрешения

>Относится к: Windows Admin Center, ознакомительная версия Windows Admin Center

Если вы еще не сделали этого, ознакомьтесь с [варианты управления доступа пользователей в Windows Admin Center](../plan/user-access-options.md)

>[!NOTE]
> Группы на основе доступа в Windows Admin Center не поддерживается в среде рабочей группы или доменах, не являющегося доверенным.

## Определения ролей шлюза доступа

Существует две функции для доступа к службе шлюз Windows Admin Center.

**Шлюз пользователи** могут подключаться к службе шлюз Windows Admin Center для управления серверами через этот шлюз, но не могут изменять разрешения на доступ, а также механизм проверки подлинности, используемый для проверки подлинности шлюза.

**Шлюз Администраторы** могут настроить пользователей, получающих доступ также как пользователи проходят проверку подлинности шлюза. Только администраторы шлюза можно просматривать и настроить параметры доступа в Windows Admin Center. Локальные администраторы на компьютере шлюза всегда являются Администраторы службы шлюз Windows Admin Center.

> [!NOTE]
> Доступ к шлюзу не подразумевает доступ к управляемым серверам видимым, шлюз. Для управления на целевом сервере, подключающееся пользователь должен использовать учетные данные (с помощью учетных данных Windows их передаются через или через учетные данные, предоставленные в сеансе Windows Admin Center, используя **Управление как** действие), имеющих административного доступа Этот конечный сервер.

## Active Directory или группы локального компьютера

По умолчанию Active Directory или группы локального компьютера используются для управления доступом шлюза. Если у вас есть домена Active Directory, вы можете управлять шлюза пользователей и администраторов доступ к с помощью интерфейса Windows Admin Center.

На вкладке " **Пользователи** " можно управлять доступом Windows Admin Center в качестве шлюза пользователя. По умолчанию и если не указать группу безопасности, любой пользователь, который получает доступ к URL-адрес шлюза имеет доступ. После добавления одного или нескольких групп безопасности в список пользователей, доступ ограничен члены этих групп.

Если вы не используете домена Active Directory в вашей среде, доступ управляется ```Users``` и ```Administrators``` локальных групп на компьютере шлюза Windows Admin Center.

### Проверка подлинности смарт-карты

Позволяют принудительно включить **проверку подлинности смарт-карты** , указав дополнительные _необходимые_ группы для группы безопасности на основе смарт-карты. После добавления группы безопасности на основе смарт-карты, пользователь доступны только службу Windows Admin Center если они являются членами группы безопасности и группу смарт-карты, включенных в список пользователей.

На вкладке " **Администраторы** " могут управлять доступом Windows Admin Center в качестве шлюза администратора. В группу локальных администраторов на компьютере всегда будут иметь доступ администратора и не может быть удален из списка. При добавлении группы безопасности, предоставляются членами этих групп права для изменения параметров шлюз Windows Admin Center. Список Администраторы поддерживает проверку подлинности смарт-карты в так же, как список пользователей: с условием AND для групп безопасности и смарт-карты.

## Azure Active Directory

Если ваша организация использует Azure Active Directory (Azure AD), вы можете добавить **дополнительный** уровень безопасности, требующих проверки подлинности Azure AD для доступа к шлюзу Windows Admin Center. Чтобы получить доступ к Windows Admin Center, пользовательской учетной **записи Windows** также должны иметь доступ к серверу шлюза (даже если используется проверка подлинности Azure AD). Если вы используете Azure AD, вы будете управлять разрешения на доступ пользователей и администраторов Windows Admin Center на портале Azure, а не с помощью пользовательского интерфейса Windows Admin Center.

### Доступ к Windows Admin Center, если включена проверка подлинности Azure AD

В зависимости от браузера используется, некоторые пользователи, доступ к Windows Admin Center с Azure AD проверка подлинности будет получать дополнительные запрос **из браузера** готовиться к предоставляют свои окна учетные данные учетной записи для компьютера, на котором Windows Admin Center установлен. После ввода этих данных, пользователи получат дополнительный запрос проверки подлинности Azure Active Directory, для которых требуется учетные данные учетная запись Azure, который был предоставлен доступ в приложение Azure AD в Azure.

> [!NOTE]
> Пользователи, кто учетной записи Windows имеет **права администратора** на компьютере шлюза будет не будет предложено для проверки подлинности Azure AD.

### Настройка проверки подлинности Azure Active Directory для ознакомительная версия Windows Admin Center

Перейдите к Windows Admin Center **Параметры** > **доступ** и использование переключателя для включения «использовать Azure Active Directory для добавления к шлюзу уровень безопасности». Если вы не выполнили регистрацию шлюза в Azure, вам будет предложено пройти для этого в данный момент.

По умолчанию все члены клиента Azure AD имеют доступ пользователей к службе шлюз Windows Admin Center. Только локальных администраторов на компьютере шлюза есть доступ с правами администратора к шлюзу Windows Admin Center. Обратите внимание, что права локальных администраторов на компьютере шлюза не может быть ограничен — локальные администраторы могут выполнять никаких действий независимо от того, используется ли Azure AD для проверки подлинности.

Если вы хотите предоставляют определенные Azure AD пользователей или групп шлюза пользователя или шлюза доступ с правами администратора к службе Windows Admin Center, выполните следующие действия.

1.  Перейдите в приложение Windows Admin Center Azure AD на портале Azure с помощью гиперссылки, приведенные в разделе параметры доступа. Обратите внимание, что гиперссылки доступен только в случае, если включена проверка подлинности Azure Active Directory. 
    -   Можно также найти ваше приложение на портале Azure, перейдя к **Azure Active Directory** > **корпоративных приложений** > **все приложения** и поиск **WindowsAdminCenter** (приложение Azure AD будет называться WindowsAdminCenter-<gateway name>). Если вы не дает результатов поиска, обеспечить **Показать** задано для **всех приложений**, **приложение** имеет статус **любого** и нажмите кнопку Применить, затем повторите поиск. После того, как найти приложение, перейдите к **пользователям и группам**
2.  На вкладке "Свойства" значение **требуется назначение пользователя** «Да».
    После этого, только члены, указанных на вкладке " **пользователи и группы** " будет иметь доступ к шлюзу Windows Admin Center.
3.  Пользователи и группы вкладка выберите **Добавить пользователя**. Необходимо назначить пользователя шлюза или шлюза роль администратора для каждой добавления пользователя или группы.

После включения проверки подлинности Azure AD, перезапускать службу шлюза и вам необходимо обновить браузер. Вы можете изменить доступ пользователя для малых и средних Предприятий Azure AD приложения на портале Azure в любое время.

## Настройка единого входа

**Единый вход при развертывании как служба в Windows Server**

При установке Windows Admin Center в Windows 10, оно будет готова для использования единого входа. Если вы собираетесь использовать Windows Admin Center в Windows Server, тем не менее, необходимо настроить какую-либо форму делегирование Kerberos в вашей среде, прежде чем использовать единый вход. Делегирование настраивает компьютер шлюза как доверенным для делегирования целевому узлу. 

Чтобы настроить [на основе ресурсов ограниченное делегирование](http://windowsitpro.com/security/how-windows-server-2012-eases-pain-kerberos-constrained-delegation-part-1) в вашей среде, выполните следующие командлеты PowerShell. (Будет знать, что для этого требуется контроллер домена под управлением Windows Server 2012 или более поздней версии).

```powershell
     $gateway = "WindowsAdminCenterGW" # Machine where Windows Admin Center is installed
     $node = "ManagedNode" # Machine that you want to manage
     $gatewayObject = Get-ADComputer -Identity $gateway
     $nodeObject = Get-ADComputer -Identity $node
     Set-ADComputer -Identity $nodeObject -PrincipalsAllowedToDelegateToAccount $gatewayObject
```

В этом примере шлюз Windows Admin Center установлен на сервере **WindowsAdminCenterGW**и целевое имя узла является **ManagedNode**.

Чтобы удалить связь, выполните следующий командлет:

```powershell
Set-ADComputer -Identity $nodeObject -PrincipalsAllowedToDelegateToAccount $null
```

## Управление доступом на основе ролей

Управление доступом на основе ролей позволяет предоставлять пользователям ограниченный доступ к компьютеру вместо внесения их полной локальных администраторов.
[Дополнительные сведения о доступных ролей и управления доступом на основе ролей.](../plan/user-access-options.md#role-based-access-control)

Настройка RBAC состоит из 2 действий: Включение поддержки на конечные компьютеры и назначение пользователей в соответствующие роли.

> [!TIP]
> Убедитесь, что у вас есть права локального администратора на компьютерах, где вы настраиваете поддержку для управления доступом на основе ролей.

### Применение элемента управления доступом на основе ролей на одном компьютере

Модели развертывания один компьютер идеально подходит для простых средах с небольшого количества компьютеров для управления.
Настройка компьютера с поддержкой для управления доступом на основе ролей приведет к следующие изменения:
-   Модули PowerShell с помощью функций, необходимого для Windows Admin Center будет устанавливаться на жестком диске, в разделе `C:\Program Files\WindowsPowerShell\Modules`. Все модули начнется с **Microsoft.Sme**
-   Желаемую конфигурацию состояние будет выполняться единовременной конфигурации для настройки конечной точки Just Enough Administration на компьютере, с именем **Microsoft.Sme.PowerShell**. Эту конечную точку определяет 3 роли, используемых Windows Admin Center и выполняются в качестве временного локального администратора при подключении к нему.
-   3 новые локальные группы создается для элемента управления, кто из пользователей ограниченный доступ к роли:
    -   Администраторы Windows Admin Center
    -   Администраторы Windows Admin Center Hyper-V
    -   Средства чтения с Windows Admin Center

Чтобы включить поддержку для управления доступом на основе ролей на одном компьютере, выполните следующие действия.

1.  Откройте Windows Admin Center и подключитесь к компьютеру, который вы хотите настроить с помощью управления доступом на основе ролей с помощью учетной записи с правами локального администратора на целевом компьютере.
2.  В средства " **Обзор** ", выберите **Параметры** > **управления доступом на основе ролей**.
3.  Щелкните **Применить** в нижней части страницы, чтобы включить поддержку для управления доступом на основе ролей на конечном компьютере. Процесс приложения включает копирования сценариев PowerShell и вызов конфигурации (с помощью Настройка требуемого состояния PowerShell) на целевом компьютере. Это может занять до 10 минут и произойдет перезапуск WinRM. Это будет временно отключить пользователей Windows Admin Center, PowerShell и WMI.
4.  Обновите страницу, чтобы проверить состояние элемента управления доступом на основе ролей. Когда приложение будет готово к использованию, состояние изменится на **применения**.

После применения конфигурации, вы можете назначать пользователям роли:

1.  Откройте средство **Локальные пользователи и группы** и перейдите на вкладку " **группы** ".
2.  Выберите группу, **Средства чтения с Windows Admin Center** .
3.  В области *сведений* , внизу нажмите кнопку **Добавить пользователя** и введите имя группы пользователей или безопасности, который должен иметь доступ только для чтения к серверу через Windows Admin Center. Пользователи и группы могут поступать из локального компьютера или домена Active Directory.
4.  Повторите шаги 2 – 3 для групп **Windows администратора центра Hyper-V Администраторы** **"** И" Windows Admin Center.

Вы также можете заполнить эти группы согласованно в домене, настроив объекта групповой политики с [Ограниченным доступом параметр политики группы](https://docs.microsoft.com/previous-versions/windows/it-pro/windows-server-2003/cc756802%28v=ws.10%29).

### Применение элемента управления доступом на основе ролей на нескольких компьютерах

В крупных корпоративных развертывания можно использовать существующие средства автоматизации передачи средства управления доступом на основе ролей на компьютерах, загрузив пакет конфигурации из шлюз Windows Admin Center.
Пакет конфигурации предназначен для использования с Настройка требуемого состояния PowerShell, но вы можете адаптировать его для работы с решением предпочтительный автоматизации.

#### Скачать конфигурацию управления доступом на основе ролей

Чтобы загрузить пакет конфигурации управления доступом на основе ролей, необходимо иметь доступ к Windows Admin Center и командную строку PowerShell.

Если вы используете шлюз Windows Admin Center в режиме службы в Windows Server, используйте следующую команду, чтобы загрузить пакет конфигурации.
Не забудьте обновить адрес шлюза с правильным для вашей среды.

```powershell
$WindowsAdminCenterGateway = 'https://windowsadmincenter.contoso.com'
Invoke-RestMethod -Uri "$WindowsAdminCenterGateway/api/nodes/all/features/jea/endpoint/export" -Method POST -UseDefaultCredentials -OutFile "~\Desktop\WindowsAdminCenter_RBAC.zip"
```

Если вы используете шлюз Windows Admin Center на компьютере с Windows 10, выполните следующую команду:

```powershell
$cert = Get-ChildItem Cert:\CurrentUser\My | Where-Object Subject -eq 'CN=Windows Admin Center Client' | Select-Object -First 1
Invoke-RestMethod -Uri "https://localhost:6516/api/nodes/all/features/jea/endpoint/export" -Method POST -Certificate $cert -OutFile "~\Desktop\WindowsAdminCenter_RBAC.zip"
```

При развертывании ZIP-архив, вы увидите следующую структуру папок:
- InstallJeaFeatures.ps1
- JustEnoughAdministration (каталог)
- Модули (каталог)
    - Microsoft.SME.\* (каталоги)
    - WindowsAdminCenter.Jea (каталог)

Чтобы настроить поддержку для управления доступом на основе ролей на узле, необходимо выполнить следующие действия:
1.  Скопируйте модули JustEnoughAdministration, Microsoft.SME.\* и WindowsAdminCenter.Jea в папку модуля PowerShell на целевом компьютере. Как правило, это находится в `C:\Program Files\WindowsPowerShell\Modules`.
2.  Обновите файл **InstallJeaFeature.ps1** в соответствии с желаемую конфигурацию для конечной точки RBAC.
3.  Запустите InstallJeaFeature.ps1 для компиляции DSC ресурса.
4.  Развертывание конфигурации DSC на все свои компьютеры, чтобы применить конфигурацию.

Как это сделать с помощью удаленного взаимодействия PowerShell объясняется в следующем разделе.

#### Развертывание на нескольких компьютерах

Для развертывания конфигурации можно скачать на нескольких компьютерах, вы должны обновлять **InstallJeaFeatures.ps1** сценарий, добавив нужные группы безопасности для вашей среды, копирования файлов на каждом компьютере и вызова сценарии настройки.
Можно использовать инструменты Ваш предпочитаемый автоматизации для этого, однако мы сконцентрируемся на чистом подход на основе PowerShell.

По умолчанию сценарий настройки создаст локальных групп безопасности на компьютере, чтобы управлять доступом к каждой роли.
Это подходит для рабочей группы и домена к домену компьютеров, но если вы выполняете развертывание в среде только для домена, которые вам может потребоваться непосредственно связать группу безопасности домена с каждой роли.
Для обновления конфигурации для использования групп безопасности домена, откройте **InstallJeaFeatures.ps1** и внесите следующие изменения:

1.  Удаление 3 **группы** ресурсов из файла.
    1.  «Группы MS-карт Group»
    2.  «Группы MS-Hyper-V-администраторам Group»
    3.  «Группы MS-администраторам Group»
2.  Удалите из свойства JeaEndpoint **DependsOn** 3 группы ресурсов
    1.  «[Группа] MS-карт Group»
    2.  «[Группа] MS-Hyper-V-администраторам Group»
    3.  «[Группа] группу администраторов MS»
3.  Измените имена групп в свойстве JeaEndpoint **RoleDefinitions** к группам нужные безопасности. Например, при наличии группы безопасности *CONTOSO\MyTrustedAdmins* , который должен быть назначен доступ к роли Windows Admin Center Администраторы, изменить `'$env:COMPUTERNAME\Windows Admin Center Administrators'` для `'CONTOSO\MyTrustedAdmins'`. Приведены три строки, которые необходимо обновить.
    1.  "$env: COMPUTERNAME\Windows администратора центра администраторов
    2.  "$env: COMPUTERNAME\Windows администратора центра администраторам Hyper-V
    3.  "$env: читателей COMPUTERNAME\Windows администратора центра

> [!NOTE]
> Убедитесь, что использование групп безопасности для каждой роли. Конфигурация завершится ошибкой, если для нескольких ролей назначен одной и той же группы безопасности.

В конце файла **InstallJeaFeatures.ps1** , добавьте следующие строки PowerShell в нижней части сценария:

```powershell
Copy-Item "$PSScriptRoot\JustEnoughAdministration" "$env:ProgramFiles\WindowsPowerShell\Modules" -Recurse -Force
$ConfigData = @{
    AllNodes = @()
    ModuleBasePath = @{
        Source = "$PSScriptRoot\Modules"
        Destination = "$env:ProgramFiles\WindowsPowerShell\Modules"
    }
}
InstallJeaFeature -ConfigurationData $ConfigData | Out-Null
Start-DscConfiguration -Path "$PSScriptRoot\InstallJeaFeature" -JobName "Installing JEA for Windows Admin Center" -Force
```

Наконец можно скопируйте папку, содержащую модули, DSC ресурсов и конфигурации для каждого целевого узла и запустите сценарий **InstallJeaFeature.ps1** .
Чтобы сделать это удаленно с рабочей станции администратора, необходимо выполнить следующие команды:

```powershell
$ComputersToConfigure = 'MyServer01', 'MyServer02'

$ComputersToConfigure | ForEach-Object {
    $session = New-PSSession -ComputerName $_ -ErrorAction Stop
    Copy-Item -Path "~\Desktop\WindowsAdminCenter_RBAC\JustEnoughAdministration\" -Destination "$env:ProgramFiles\WindowsPowerShell\Modules\" -ToSession $session -Recurse -Force
    Copy-Item -Path "~\Desktop\WindowsAdminCenter_RBAC" -Destination "$env:TEMP\WindowsAdminCenter_RBAC" -ToSession $session -Recurse -Force
    Invoke-Command -Session $session -ScriptBlock { Import-Module JustEnoughAdministration; & "$env:TEMP\WindowsAdminCenter_RBAC\InstallJeaFeature.ps1" } -AsJob
    Disconnect-PSSession $session
}
```
