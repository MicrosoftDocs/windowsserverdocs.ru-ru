# <a name="monitor-servers-and-configure-alerts-with-azure-monitor-from-windows-admin-center"></a>Мониторинг серверов и Настройка оповещений с помощью Azure Monitor из центра администрирования Windows

[Узнайте больше об интеграции Azure с центром администрирования Windows.](../plan/azure-integration-options.md)

[Azure Monitor](https://docs.microsoft.com/azure/azure-monitor/overview) — это решение, которое собирает, анализирует и взаимодействует с данными телеметрии из различных ресурсов, включая серверы и виртуальные машины Windows, как локально, так и в облаке. Хотя Azure Monitor извлекает данные из виртуальных машин Azure и других ресурсов Azure, эта статья посвящена тому, как Azure Monitor работает с локальными серверами и виртуальными машинами, в частности с центром администрирования Windows. Если вы хотите узнать, как можно использовать Azure Monitor для получения оповещений по электронной почте о кластере с поддержкой Hyper-in, прочитайте об [использовании Azure Monitor для отправки сообщений электронной почты для Служба работоспособности ошибок](https://docs.microsoft.com/windows-server/storage/storage-spaces/configure-azure-monitor).

## <a name="how-does-azure-monitor-work"></a>Как работает Azure Monitor?
![IMG](../media/azure-monitor-diagram.png) данные, созданные на локальных серверах Windows, собираются в Log Analytics рабочей области Azure Monitor. В рабочей области можно включить различные решения для мониторинга — наборы логики, которые предоставляют аналитические сведения о конкретном сценарии. Например, Azure Управление обновлениями, центр безопасности Azure и Azure Monitor для виртуальных машин — это все решения мониторинга, которые можно включить в рабочей области. 

Когда вы включаете решение мониторинга в рабочей области Log Analytics, все серверы, передающие данные в эту рабочую область, начинают сбор данных, относящихся к этому решению, чтобы решение могла формировать аналитические сведения для всех серверов в рабочей области. 

Чтобы выполнить сбор данных телеметрии на локальном сервере и отправить их в рабочую область Log Analytics, Azure Monitor требует установки Microsoft Monitoring Agent или MMA. Для некоторых решений мониторинга также требуется вторичный агент. Например, Azure Monitor для виртуальных машин также зависит от агента ServiceMap для реализации дополнительных функций, предоставляемых этим решением. 

Некоторые решения, например Azure Управление обновлениями, также зависят от службы автоматизации Azure, которая позволяет централизованно управлять ресурсами в среде Azure и средах, отличных от Azure. Например, Azure Управление обновлениями использует службу автоматизации Azure для планирования и организации установки обновлений на компьютерах в вашей среде централизованно из портал Azure.


## <a name="how-does-windows-admin-center-enable-you-to-use-azure-monitor"></a>Как центр администрирования Windows позволяет использовать Azure Monitor?

В ВАК можно включить два решения для мониторинга:

- [Управление обновлениями Azure](azure-update-management.md) (в средстве обновления)
- Azure Monitor для виртуальных машин (в параметрах сервера), a. k. а Virtual Machines Insights

Вы можете приступить к работе с Azure Monitor из любого из этих средств. Если вы никогда не использовали Azure Monitor, ВАК автоматически подготавливает Log Analytics рабочую область (и учетную запись службы автоматизации Azure, если это необходимо), а также устанавливает и настраивает Microsoft Monitoring Agent (MMA) на целевом сервере. Затем соответствующее решение будет установлено в рабочую область. 

Например, если вы впервые переходите к средству Updates to Setup Azure Управление обновлениями, ВАК выполнит следующие действия:

1. Установка MMA на компьютере
2. Создайте рабочую область Log Analytics и учетную запись службы автоматизации Azure (так как в этом случае требуется учетная запись службы автоматизации Azure).
3. Установите решение Управление обновлениями в созданной рабочей области.

Если вы хотите добавить еще одно решение для мониторинга из ВАК на том же сервере, ВАК просто установит это решение в существующую рабочую область, к которой подключен этот сервер. ВАК дополнительно установит все необходимые агенты.

Если вы подключаетесь к другому серверу, но уже настроили Log Analytics рабочую область (либо с помощью ВАК, либо вручную на портале Azure), можно также установить агент MMA на сервере и подключить его к существующей рабочей области. При подключении сервера к рабочей области он автоматически начинает собирать данные и создавать отчеты для решений, установленных в этой рабочей области.

## <a name="azure-monitor-for-virtual-machines-aka-virtual-machine-insights"></a>Azure Monitor для виртуальных машин ( Виртуальная машина Insights)
>Область применения: Предварительная версия Windows Admin Center

При настройке Azure Monitor для виртуальных машин в параметрах сервера центр администрирования Windows включает решение Azure Monitor для виртуальных машин, также известное как виртуальная машина Insights. Это решение позволяет отслеживать работоспособность и события сервера, создавать оповещения по электронной почте, получать объединенное представление о производительности сервера в среде и визуализировать приложения, системы и службы, подключенные к определенному серверу.

> [!NOTE]
> Несмотря на свое имя, программа VM Insights работает для физических серверов, а также для виртуальных машин.

Благодаря бесплатному объему данных в размере до 5 ГБ за использование Azure Monitor, вы можете легко испытать их на сервере или двух, не обращаясь к плате. Прочтите статью, чтобы узнать о дополнительных преимуществах адаптации серверов к Azure Monitor, таких как получение консолидированного представления о производительности систем на серверах в вашей среде.

### <a name="set-up-your-server-for-use-with-azure-monitor"></a>**Настройка сервера для использования с Azure Monitor**

На странице Обзор подключения к серверу нажмите кнопку создать оповещения или выберите параметры сервера > Мониторинг и оповещения. На этой странице Подключите сервер к Azure Monitor, щелкнув "настроить" и завершив область настройки. Центр администрирования отвечает за подготовку рабочей области Azure Log Analytics, установку необходимого агента и обеспечение настройки решения VM Insights. После завершения сервер отправляет данные счетчиков производительности в Azure Monitor, позволяя просматривать и создавать оповещения по электронной почте на основе этого сервера из портал Azure.

### <a name="create-email-alerts"></a>**Создание оповещений по электронной почте**

После подключения сервера к Azure Monitor можно использовать интеллектуальные гиперссылки на странице Параметры > Мониторинг и оповещения для перехода на портал Azure. Центр администрирования автоматически включает сбор данных счетчиков производительности, поэтому вы можете легко [создать новое оповещение](https://docs.microsoft.com/azure/azure-monitor/platform/alerts-log) , настроив один из нескольких предварительно определенных запросов или создав собственный.

### <a name="get-a-consolidated-view-across-multiple-servers-"></a>\* * Получите объединенное представление на нескольких серверах * *

При подключении нескольких серверов к одной рабочей области Log Analytics в Azure Monitor можно получить объединенное представление всех этих серверов из [решения Virtual Machine Insights](https://docs.microsoft.com/azure/azure-monitor/insights/vminsights-overview) в Azure Monitor.  (Обратите внимание, что только вкладки "производительность" и "карты" Virtual Machine Insights для Azure Monitor будут работать с локальными серверами — вкладка "работоспособность" работает только с виртуальными машинами Azure.) Чтобы просмотреть это в портал Azure, перейдите в раздел Azure Monitor > виртуальные машины (в разделе "аналитика") и перейдите на вкладки "производительность" или "карты".

### <a name="visualize-apps-systems-and-services-connected-to-a-given-server"></a>**Визуализация приложений, систем и служб, подключенных к определенному серверу**

Когда центр администрирования применяет сервер к решению VM Insights в Azure Monitor, также выключается возможность, называемая [сопоставление служб](https://docs.microsoft.com/azure/azure-monitor/insights/service-map). Эта возможность автоматически обнаруживает компоненты приложения и сопоставляет взаимодействие между службами, чтобы можно было легко визуализировать подключения между серверами с большим описанием портал Azure. Это можно найти, перейдя к портал Azure > Azure Monitor > виртуальные машины (в разделе "аналитика") и перейдя на вкладку "карты".

> [!NOTE]
> Визуализации для Virtual Machine Insights для Azure Monitor предоставляются в настоящее время 6 общедоступных регионов.  Последние сведения см. в документации по [Azure Monitor для виртуальных машин](https://docs.microsoft.com/azure/azure-monitor/insights/vminsights-onboard#log-analytics).  Необходимо развернуть рабочую область Log Analytics в одном из поддерживаемых регионов, чтобы получить дополнительные преимущества, предоставляемые решением Virtual Machine Insights, описанным выше.

## <a name="disabling-monitoring"></a>Отключение мониторинга

Чтобы полностью отключить сервер в рабочей области Log Analytics, удалите агент MMA. Это означает, что этот сервер больше не будет отсылать данные в рабочую область, и все решения, установленные в этой рабочей области, больше не будут получать и обрабатывать данные с этого сервера. Однако это не влияет на саму рабочую область — все ресурсы, сообщающие в эту рабочую область, будут по-прежнему работать. Чтобы удалить агент MMA в ВАК, перейдите в раздел приложения & компоненты, найдите Microsoft Monitoring Agent и нажмите кнопку Удалить.

Если вы хотите отключить определенное решение в рабочей области, необходимо [удалить решение для мониторинга из портал Azure](https://docs.microsoft.com/azure/azure-monitor/insights/solutions#remove-a-management-solution). Удаление решения для мониторинга означает, что аналитика, созданная этим решением, больше не будет создаваться для _всех_ серверов, отправляющих отчеты в эту рабочую область. Например, если я удаляю решение Azure Monitor для виртуальных машин, я больше не вижу сведений о производительности виртуальной машины или сервера на всех компьютерах, подключенных к моей рабочей области.
