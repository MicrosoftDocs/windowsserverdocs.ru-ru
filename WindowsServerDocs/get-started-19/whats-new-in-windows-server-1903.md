# <a name="whats-new-in-windows-server-version-1903"></a>Новые возможности в Windows Server версии 1903

>Относится к: Windows Server (Semi-Annual Channel)

В этом разделе описываются некоторые из новых функций в Windows Server версии 1903, которых — это выпуск в Semi-Annual Channel. Эти функции включают улучшения выполнения и управления ими, контейнеры, инструменты для работы в установках основных серверных компонентов и возможность миграции хранилища с устройств Linux.

Вместо этого новые возможности в последнем выпуске канал долгосрочного обслуживания (LTSC) для Windows Server, см. в статье см. в разделе [новые возможности в Windows Server 2019](../get-started-19/whats-new-19.md). Также см. в разделе [новые возможности в Windows 10, содержимое IT Pro версии 1903](https://docs.microsoft.com/windows/whats-new/whats-new-windows-10-version-1903).

Системные требования для этой версии не отличаются от Windows Server 2019, см. в разделе [требования к системе](../get-started-19/sys-reqs-19.md) Дополнительные сведения. Чтобы узнать, что является была недавно удалена, см. в разделе [Features Removed or запланировано для замены, начиная с Windows Server версии 1903](../get-started-19/removed-features-1903.md)

> [!NOTE]
> Контейнеры Windows необходимо использовать ту же версию Windows, что и сервер узла или *более ранних* версии. Например сервер под управлением выпущенной версии Windows Server, версии 1903 (сборка 18342) можно запускать контейнеры Windows Server с той же или более ранней версией и номер сборки (даже если контейнер использует Insider Preview версию Windows). Дополнительные сведения см. в разделе [совместимость версий контейнеров Windows](https://docs.microsoft.com/virtualization/windowscontainers/deploy-containers/version-compatibility).

## <a name="enhanced-support-for-non-microsoft-container-services"></a>Расширенная поддержка для служб контейнеров сторонних производителей

Мы улучшили возможности платформы для поддержки службы контейнеров Azure и служб сторонних производителей контейнеров.

- Мы интегрировали CRI containerd с узла вычислений службу HCS для поддержки модулей из контейнеров Windows и Linux на Windows (LCOW) в Azure.
- Мы сотрудничаем с сообщества Kubernetes, чтобы включить поддержку контейнеров Windows. С появлением Kubernetes v1.14 поддержка узлов Windows Server официально окончил бета-версии в постоянное. Дополнительные сведения см. в разделе [контейнеры Windows, теперь поддерживается в Kubernetes](https://cloudblogs.microsoft.com/opensource/2019/03/25/windows-server-containers-now-supported-kubernetes/).
- Кошка черепахового Tigera для Windows теперь общедоступна как часть подписки Tigera Essentials, так и с помощью предложения без наложения сети и сетевой политики с возможностью взаимодействия, в смешанных средах Linux и Windows.
- Доставлен усовершенствованиям возможностей масштабирования, улучшение наложения сети поддержка контейнеров Windows, включая интеграцию с Kubernetes благодаря последнему выпуску v1.14 Flannel и Kubernetes. Дополнительные сведения см. в разделе [Общие сведения о поддержке Windows в Kubernetes](https://kubernetes.io/docs/setup/windows/).

## <a name="add-task-scheduler-and-hyper-v-manager-to-server-core-installations"></a>Добавить планировщик задач и диспетчер Hyper-V для установки основных серверных компонентов

Как правило, рекомендуется использовать вариант установки основных серверных компонентов, при использовании Windows Server Semi-Annual Channel в рабочей среде. Тем не менее Server Core по умолчанию пропускает ряд средств управления полезны. Вы можете добавить, многие из наиболее широко используемых средств, установив компонент совместимости приложений, но по-прежнему будут внесены некоторые отсутствующие инструменты.

Таким образом на основе отзывов клиентов, мы добавили две дополнительные средства компонентов совместимость приложений в этой версии: Планировщик заданий (taskschd.msc) и диспетчер Hyper-V (virtmgmt.msc).

Дополнительные сведения см. в разделе [компонент совместимости основных серверных компонентов приложения](../get-started-19/install-fod-19.md).

## <a name="storage-migration-service-now-migrates-local-accounts-clusters-and-linux-servers"></a>Служба миграции хранилища теперь переносит локальных учетных записей, кластеры и серверы с Linux

Служба миграции хранилища упрощает перенос серверов до более новой версии Windows Server. Он предоставляет графическое средство, которое проводит инвентаризацию данные на серверах и передает данные и конфигурацию на более новых серверах — все это без приложений и пользователям придется внести изменения.

При использовании этой версии Windows Server для координации миграций, мы добавили следующие возможности:

- Перенос локальных пользователей и групп на новый сервер
- Миграция хранилища из отказоустойчивых кластеров
- Миграция хранилища на сервере Linux, использующий Samba
- Более легко синхронизировать перенесенных общих ресурсов в Azure с помощью службы синхронизации файлов Azure
- Перенос в новых сетей, таких как Azure

Дополнительные сведения о миграции службы хранилища, см. в разделе [Общие сведения о службе миграции хранилища](../storage/storage-migration-service/overview.md).

## <a name="system-insights-disk-anomaly-detection"></a>Обнаружение аномалий диска системы аналитики

[Insights системы](../manage/system-insights/overview.md) — это функция прогнозной аналитики, который локально анализирует данные системы Windows Server и позволяет отслеживать работу сервера. Предоставляется ряд встроенных возможностей, но мы добавили возможность устанавливать дополнительные возможности через Windows Admin Center, начиная с диске обнаружения аномалий.

Обнаружение аномалий диска — это новая возможность, которой выделяется при поведения диски *по-разному* чем обычно. Во время различных не всегда плохо, просматривать эти аномальных секунд может пригодиться при устранении неполадок в системах.

Эта возможность также доступна для серверов под управлением Windows Server 2019.

## <a name="setupdiag"></a>SetupDiag
[SetupDiag](https://docs.microsoft.com/windows/deployment/upgrade/setupdiag) доступна версия 1.4.1.

SetupDiag является средством командной строки, которые могут помочь установить причину не удалось выполнить обновление Windows. SetupDiag выполняет поиск в файлах журнала программы установки Windows. SetupDiag использует набор правил для обнаружения известных проблем. В текущей версии SetupDiag 53 правил, содержащихся в файле rules.xml, который извлекается при выполнении SetupDiag. Файл rules.xml будет обновляться по мере выхода новых версий SetupDiag.

## <a name="update-rollback-improvements"></a>Усовершенствования откат обновления

Серверы можно теперь автоматически восстановиться после сбоев при запуске путем удаления обновлений в случае сбоя запуска был после установки последних обновлений драйверов или качества. Если устройство не смогла запускается некорректно после последней установки качество обновления драйверов, Windows будет автоматически удалить обновлений необходимо перевести устройство резервного копирования и работать нормально.

Эта функция требует использовать вариант установки основных серверных компонентов с сервера [среды восстановления Windows](https://docs.microsoft.com/windows-hardware/manufacture/desktop/windows-recovery-environment--windows-re--technical-reference) секции.

## <a name="microsoft-defender-advanced-threat-protection-atp-improvements"></a>Усовершенствования Microsoft Defender Advanced Threat Protection (ATP)

Windows Server включает в себя Microsoft Advanced Thread Protection в Защитнике (Дополнительные сведения см. в разделе [антивирусной программы Защитника Windows в Windows Server](https://docs.microsoft.com/windows/security/threat-protection/windows-defender-antivirus/windows-defender-antivirus-on-windows-server-2016)). Этот выпуск включает следующие улучшения:

- [Атакам контактную зону](https://docs.microsoft.com/windows/security/threat-protection/windows-defender-atp/overview-attack-surface-reduction) — ИТ-администраторы могут настроить устройства с помощью расширенных веб-защиты, который позволит определить списки разрешенных и запрещенных для определенного URL-адреса и IP-адреса.
- [Защита следующего поколения](https://docs.microsoft.com/windows/security/threat-protection/windows-defender-antivirus/windows-defender-antivirus-in-windows-10) — элементы управления были расширены для защиты от программ-шантажистов, неправильное использование учетных данных и атак, которые передаются с помощью съемных носителей.
    - Возможности обеспечения целостности принудительного применения — Включение удаленного выполнения аттестации.
    - Проверки изменений возможностей — используется безопасность на основе виртуализации для изоляции важными функциями безопасности ATP от операционной системы и злоумышленников.
- ATP в Защитнике Microsoft технологии защиты следующего поколения:
    - **Advanced машинного обучения**: Улучшена с помощью сложного машинного обучения и модели искусственного Интеллекта, позволяющие ему защититься от злоумышленников вершине, с помощью инновационных уязвимость методов атак, средства и вредоносных программ.
    - **Защита аварийного эпидемии**: Обеспечивает защиту аварийного эпидемии, который автоматически обновит устройств новой аналитики, при обнаружении нового эпидемии.
    - **Сертификации ISO 27001 соответствия**: Гарантирует, что облачная служба проанализировать угрозы, уязвимости и влияние, и что риск средства управления и обеспечения безопасности применяются.
    - **Поддержка GeoLocation**: Поддерживает географическое расположение и независимости образец данных, а также политики хранения можно настроить.
