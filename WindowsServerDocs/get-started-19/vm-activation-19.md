# Активации автоматического виртуальной машины

> Область применения: Windows Server 2019, Windows Server Semi-Annual Channel, Windows Server 2016, Windows Server 2012 R2

Автоматической активации виртуальной машины (AVMA) выступает в качестве механизм подтверждении, помогая убедиться, что продукты Windows используются в соответствии с правами на использование продуктов и условия лицензионного соглашения на использование программного обеспечения корпорации Майкрософт.

AVMA позволяет устанавливать виртуальных машин на сервере надлежащим образом активации Windows без необходимости управление ключами продуктов для каждой отдельной виртуальной машины, даже в отключенных средах. AVMA активацию виртуальной машины с лицензиями виртуализации сервера и активирует виртуальной машины при запуске. AVMA также предоставляет отчеты в режиме реального времени об использовании и исторические данные о состоянии лицензий на виртуальной машине. Отчеты и данные по отслеживанию доступен на сервере виртуализации.

## Практическое применение

На серверах виртуализации, которые активируются с помощью корпоративного лицензирования или OEM-Лицензирование AVMA обеспечивает ряд преимуществ.

Диспетчеры сервера центра обработки данных можно использовать AVMA сделать следующее:

  - Активация виртуальных машин в удаленных расположениях

  - Активация виртуальных машин с помощью или без подключения к Интернету

  - Отслеживать использование виртуальной машины и лицензии с сервера виртуализации, не требуя никаких прав доступа на виртуализированные системы

Существует без ключей продукта для управления и не наклейки на серверах для чтения. Виртуальная машина активируется и продолжает работу, даже если переносится на разных массива виртуализации серверов.

Партнеры соглашение лицензии поставщика услуг (SPLA) и других поставщиков не нужно совместно использовать ключи продуктов с клиентами или получить доступ к виртуальной машины клиента активировать его. При использовании AVMA, активацию виртуальной машины является прозрачным для клиента. Поставщики услуг размещения можно использовать журналы сервера для проверки соответствия лицензии и вести журнал использования клиента.

## Требования к системе

AVMA требуется сервер виртуализации Майкрософт под управлением Windows Server 2019 Datacenter, Windows Server 2012 R2 или Windows Server 2016 Datacenter. 

Ниже приведены гостевые ОС, может активировать узлы другую версию.

|Версия сервера узла|Windows Server 2019 г.|Windows Server 2016|WindowsServer2012R2|
|-|-|-|-|
|Windows Server 2019 г.|X|X|X|
|Windows Server 2016| |X|X|
|Windows Server 2012 R2| ||X|

Обратите внимание, что эти активировать всех выпусков (Datacenter, Standard или Essentials).

Это средство не работает с другими технологиями виртуализации сервера.

## Как реализовать AVMA

1.  На сервере виртуализации Windows Server Datacenter установить и настроить роль сервера Hyper-V корпорации Майкрософт. Дополнительные сведения см. в разделе [Установка сервера Hyper-V](../virtualization/hyper-v/get-started/install-the-hyper-v-role-on-windows-server.md).

2.  [Создание виртуальной машины](../virtualization/hyper-v/get-started/create-a-virtual-machine-in-hyper-v.md) и установка поддерживаемой операционной системы на нем.

3.  Установка ключа AVMA на виртуальной машине. В командной строке с повышенными привилегиями выполните следующую команду:
    
    ``` 
    slmgr /ipk <AVMA_key>  
    ```

Виртуальная машина автоматически активируется лицензии на сервере виртуализации.


> [!TIP]
> Также можно использовать клавиши AVMA в любой файл установки Unattend.exe.

## Отслеживание и отчетность

На сервере виртуализации реестра (KVP) предоставляет данные отслеживания в режиме реального времени для гостевых операционных систем. Поскольку в разделе реестра перемещается с виртуальной машиной, вы можете получить также информацию о лицензии. По умолчанию KVP возвращает сведения о виртуальной машине, включая следующие:

  - Полное доменное имя

  - Операционной системы и установленных пакетов обновления

  - Архитектура процессора

  - Сетевые адреса IPv4 и IPv6

  - Адреса протокола удаленного рабочего СТОЛА

Дополнительные сведения о том, как получить эту информацию см. в разделе [Hyper-V сценария: смотрит на KVP GuestIntrinsicExchangeItems](http://blogs.msdn.com/b/virtual_pc_guy/archive/2008/11/18/hyper-v-script-looking-at-kvp-guestintrinsicexchangeitems.aspx).


> [!NOTE]
> KVP данные не защищены. Он может быть изменен и не отслеживать изменения.



> [!IMPORTANT]
> KVP данные должны быть удалены, если ключ AVMA заменяется другой ключ продукта (розничной торговли, изготовитель Оборудования или том, ключ лицензии).


Исторические данные о запросах AVMA доступна в файл журнала на сервере виртуализации (EventID 12310).

Так как процесс активации AVMA является прозрачным, сообщения об ошибках не отображаются. Тем не менее следующие события регистрируются в файл журнала на виртуальных машинах (EventID 12309).

|Уведомление|Описание|
|-|-|
|Успех AVMA|Виртуальная машина был активирован.|
|Недопустимая узла|Виртуализация сервер не отвечает. Это может произойти, если на сервере не установлена поддерживаемая версия Windows.|
|Недопустимые данные|Это обычно приводит к после сбоя в обмен данными между сервером виртуализации и виртуальной машины, часто причиной несоответствия повреждения, шифрование и данных.|
|Активация отклонен|Сервер виртуализации не удалось активировать гостевую операционную систему, так как идентификатор AVMA не соответствует.|
