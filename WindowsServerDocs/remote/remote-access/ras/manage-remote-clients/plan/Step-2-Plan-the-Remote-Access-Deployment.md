# <a name="step-2-plan-the-remote-access-deployment"></a>Шаг 2 Планирование развертывания удаленного доступа

>Область применения. Windows Server (полугодовой канал), Windows Server 2016

## <a name="plan-a-client-deployment-strategy"></a>Планирование стратегии для развертывания клиента  
При планировании развертывания клиентов необходимо принять три решения:  
  
1.  Будет ли DirectAccess доступен для мобильных устройств, или ко всем компьютерам в определенную группу безопасности?  
  
    При настройке клиентов DirectAccess в мастере настройки клиента DirectAccess, можно разрешить только мобильным компьютерам в указанных групп безопасности для подключения к серверу с помощью DirectAccess. Если вы запретите доступ для мобильных устройств, будет автоматически настроен фильтр WMI, чтобы убедиться, что объект групповой политики клиента DirectAccess применяется только к мобильным устройствам из указанных групп безопасности. Чтобы выполнить указанные настройки, администратору удаленного доступа требуется разрешение на создание или изменение фильтров WMI групповой политики.  
  
2.  В какие группы безопасности будут входить компьютеры клиентов DirectAccess?  
  
    Настройки DirectAccess хранятся в клиенте DirectAccess объекта групповой политики (GPO). Этот объект GPO применяется к компьютерам, входящим в группы безопасности, указанные в мастере настройки клиента DirectAccess. Можно указать группы безопасности, которые содержатся в любом поддерживаемом домене.
  
    Перед настройкой удаленного доступа, необходимо создать группы безопасности. После завершения развертывания удаленного доступа, можно добавить компьютеры в группу безопасности. Тем не менее если вы добавите клиентские компьютеры, которые находятся в домене, отличном от группы безопасности, GPO клиента не применяется для этих клиентов. Например если вы создали SG1 в домене А для клиентов DirectAccess и добавлении клиентов из домена Б в эту группу, GPO клиента не применяется для клиентов в домене б.  
  
    Чтобы избежать этой проблемы, создайте новую группу безопасности клиента для каждого домена, содержащего клиентские компьютеры. Кроме того, если вы не хотите создать новую группу безопасности, запустите **Add-DAClient** командлет Windows PowerShell с именем нового GPO для нового домена.  
  
3.  Какие параметры вы настроите для помощника подключения к сети в DirectAccess?  
  
    DirectAccess сети помощник по подключению к выполняется на клиентских компьютерах и предоставляет дополнительные сведения о подключении DirectAccess для конечных пользователей. В мастере настройки клиента DirectAccess можно настроить следующие компоненты:  
  
    -   **Средства проверки подключения**  
  
        Создается веб-проба по умолчанию, которую клиенты используют для проверки подключения к внутренней сети. Имя по умолчанию — `https://directaccess-WebProbeHost.<domain_name>`. Это имя следует вручную зарегистрировать в системе DNS. Вы можете создать дополнительные проверки подключения, использующие другие веб-адреса по протоколу HTTP или PING. Для каждого средства проверки подключения должна существовать DNS-запись.  
  
    -   **Справка адрес электронной почты службы поддержки**  
  
        Если для конечных пользователей возникают проблемы с подключением DirectAccess, они могут отправлять сообщения электронной почты, содержащее данные диагностики, администратору удаленного доступа, который можно решить эту проблему.  
  
    -   **Имя подключения DirectAccess**  
  
        Можно указать имя подключения DirectAccess, чтобы помочь конечным пользователям определить подключение DirectAccess на своем компьютере.  
  
    -   **Разрешить клиентам DirectAccess использовать локальное разрешение имен**  
  
        Клиентам необходим способ разрешения имен локально. Если вы позволяете клиентам DirectAccess использовать локальное разрешение имен, конечные пользователи смогут применять локальные DNS-серверы для разрешения имен. Когда конечные пользователи используют локальные DNS-серверы для разрешения имен, DirectAccess не отправляет запросы на разрешения для однокомпонентных имен на внутренний корпоративный DNS-сервер. Локальное разрешение имен вместо этого он использует (с помощью локальных многоадресной рассылки имя разрешения (LLMNR) и NetBios по протоколам TCP/IP).  
  
## <a name="plan-a-remote-access-server-deployment-strategy"></a>Планирование стратегии для развертывания сервера удаленного доступа  
Решения, которые необходимо принять при планировании развертывания сервера удаленного доступа включают:  
  
-   **Топология сети**  
  
    Существует две топологии при развертывании сервера удаленного доступа:  
  
    -   **Два адаптера**: С двумя сетевыми адаптерами один сетевой адаптер, подключенный непосредственно к Интернету можно настроить удаленный доступ, а второй к внутренней сети. Или в качестве альтернативы сервер устанавливается за периферийным устройством, например брандмауэром или маршрутизатором. В такой конфигурации один сетевой адаптер подключен к сети периметра, а второй — к внутренней сети.  
  
    -   **Один сетевой адаптер**: В этой конфигурации удаленного доступа установлен сервер за периферийным устройством, например брандмауэром или маршрутизатором. Сетевой адаптер подключается к внутренней сети.  

-   **Сетевые адаптеры**  
  
    Мастер установки сервера удаленного доступа автоматически обнаруживает сетевые адаптеры, настроенные на сервере удаленного доступа. Убедитесь, что выбраны правильные адаптеры.  
  
-   **Сертификат IP-HTTPS**  
  
    Мастер настройки сервера удаленного доступа автоматически обнаруживает сертификат для подключения IP-HTTPS. Имя субъекта выбранного сертификата должно соответствовать адресу ConnectTo. Если вы используете самозаверяющие сертификаты, вы можете использовать сертификат, который создается автоматически сервером удаленного доступа.  
  
-   **Префиксы IPv6**  
  
    Если мастер настройки сервера удаленного доступа определяет, что для сетевых адаптеров развернут протокол IPv6, он автоматически заполняет префиксы IPv6 для внутренней сети, префикс IPv6, который будет назначен клиентским компьютерам DirectAccess, и префикс IPv6, который будет назначен клиентским компьютерам VPN. Если автоматически созданные префиксы неверны для вашей собственной инфраструктуры IPv6 или ISATAP, следует вручную изменить их.  
  
-   **Authentication**  
  
    Можно выбрать один из следующих методов проверки подлинности клиентов DirectAccess к серверу удаленного доступа:  
  
    -   **Проверка подлинности пользователя**: Вы можете применять проверку подлинности с использованием учетных данных Active Directory или двухфакторную проверку подлинности.  
  
    -   **Проверка подлинности компьютера**: Вы можете настроить проверки подлинности компьютера для использования сертификатов. Или без сертификатов сервера удаленного доступа может выступать в качестве прокси-сервер для проверки подлинности Kerberos. 
  
    -   **Клиенты Windows 7** по умолчанию клиентские компьютеры под управлением Windows 7 не удается подключиться к среде удаленного доступа под управлением Windows Server 2012. Если у вас есть клиенты под управлением Windows 7 в вашей организации, которым требуется удаленный доступ к внутренним ресурсам, можно разрешить их подключение. Все клиентские компьютеры, которым требуется предоставить доступ к внутренним ресурсам, должны входить в группу безопасности, указанную в мастере настройки клиента DirectAccess.  
  
        > [!NOTE]  
        > Позволить клиентам под управлением Windows 7 подключаться с помощью DirectAccess необходимо использовать проверку подлинности сертификата компьютера.  
  
-   **Конфигурация VPN**  
  
    Перед настройкой удаленного доступа следует решите, если вы собираетесь предоставить VPN-доступ к удаленным клиентам. Необходимо предоставить доступ к VPN при наличии клиентских компьютеров в вашей организации, которые не поддерживают подключения DirectAccess (например, они неуправляемые или они работают на операционной системе, для которой DirectAccess не поддерживается). Мастер установки сервера удаленного доступа можно настроить как назначаются IP-адреса (с помощью DHCP или из пула статических адресов) и способ проверки подлинности VPN-клиентов (с помощью Active Directory или RADIUS-сервера).
