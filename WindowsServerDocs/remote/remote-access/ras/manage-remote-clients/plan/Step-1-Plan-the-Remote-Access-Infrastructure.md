## <a name="plan-network-topology-and-settings"></a>Планирование топологии сети и настроек  
При планировании сети, необходимо учитывать адаптера топологию сети, параметры для назначения IP-адресов, а также требования по ISATAP.  
  
### <a name="plan-network-adapters-and-ip-addressing"></a>Планирование конфигурации сетевых адаптеров и параметров IP-адресации  
  
1.  Определение топологии сети адаптера, который вы хотите использовать. Можно настроить удаленный доступ с любой из следующих топологий:  
  
    -   С двумя сетевыми адаптерами: Сервер удаленного доступа устанавливается на границе с одним сетевым адаптером, подключенных к Интернету, а другой — к внутренней сети.  
  
    -   С двумя сетевыми адаптерами: Сервер удаленного доступа устанавливается за устройством NAT, брандмауэром или маршрутизатором, один сетевой адаптер подключен к сети периметра, а другой — к внутренней сети.  
  
    -   Один сетевой адаптер: Сервер удаленного доступа устанавливается за устройством NAT, и один сетевой адаптер подключен к внутренней сети.  
  
2.  Определите требования к IP-адресации:  
  
    Для установки безопасного подключения клиентских компьютеров к внутренней сети корпорации в технологии DirectAccess используются протоколы IPv6 и IPsec. Однако для работы DirectAccess необязательно подключаться к Интернету по IPv6 или использовать во внутренних сетях оборудование с поддержкой IPv6. Вместо этого он автоматически настраивает и использует технологии туннелирования IPv6 для туннелирования трафика IPv6 через протокол Интернета IPv4 (6to4, Teredo или IP-HTTPS) и в интрасети только с IPv4 (NAT64 или ISATAP). Общие сведение об этих технологиях туннелирования см. в следующих разделах:  
  
    -   [Технологии туннелирования для IPv6](https://technet.microsoft.com/library/bb726951.aspx)  
  
    -   [Спецификация протокола туннелирования IP-HTTPS](https://msdn.microsoft.com/library/dd358571.aspx)  
  
3.  Выполните необходимые настройки адаптеров и адресации в соответствии со следующей таблицей. Для развертывания, которые находятся за устройством NAT с помощью одного сетевого адаптера, настройте IP-адреса, используя только **внутренний сетевой адаптер** столбца.

Если эти условия выполняются, дальнейшие настройки маршрута по умолчанию не требуются. В этом случае добавьте фильтры пакетов на контроллер домена в сети периметра, где предотвратить подключение к IPv6-адрес веб-интерфейсу сервера удаленного доступа.|Необходимо установить:<br/><br/>Если вы не используете Уровни предпочтений по умолчанию, настроить интерфейсы интрасети с помощью `netsh interface ipv6 set InterfaceIndex ignoredefaultroutes=enabled` команды. После выполнения этой команды дополнительные маршруты по умолчанию, указывающие на маршрутизаторы интрасети, не будут добавлены в таблицу маршрутизации IPv6.     |IPv4-Интернет и IPv6-интрасеть|Сервер удаленного доступа направляет трафик маршрута IPv6 по умолчанию с помощью интерфейса адаптера Microsoft 6to4 позволяет осуществлять ретрансляцию 6to4 в IPv4-Интернет.     > [!NOTE]  
    > -   Если клиент DirectAccess присвоен общедоступный IPv4-адрес, он будет использовать технологию ретрансляции 6to4 для подключения к интрасети. Если клиент назначен частный IPv4-адрес, он будет использовать Teredo. Если клиент DirectAccess не удается подключиться к серверу DirectAccess с помощью технологий 6to4 или Teredo, он будет использовать IP-HTTPS.  
    > -   Для применения Teredo необходимо настроить два последовательных IP-адреса на внешнем сетевом адаптере.  
    > -   Если сервер удаленного доступа имеет только один сетевой адаптер, не могут использовать Teredo.  
    > -   Клиентским компьютерам, поддерживающим IPv6 и подключающимся по этому протоколу к серверу удаленного доступа, не требуется выполнять туннелирование.  
  
### <a name="plan-isatap-requirements"></a>Планирование требований к ISATAP  
ISATAP необходим для удаленного управления DirectAccessclients, таким образом, серверы управления DirectAccess можно подключиться к клиентам DirectAccess, расположенными в Интернете. ISATAP не является обязательным для поддержки подключений, инициированных клиентскими компьютерами DirectAccess на IPv4-ресурсам в корпоративной сети. Для этого применяется NAT64/DNS64. Если для развертывания требуется ISATAP, используйте следующую таблицу для определения требований.  
  
|Сценарий развертывания ISATAP|Требования|  
|---------------|--------|  
|Существующего собственного интрасети IPv6 (не ISATAP не требуется)|С помощью существующих собственной инфраструктуры IPv6 укажите префикс организации во время развертывания удаленного доступа и сервера удаленного доступа настраивает сам маршрутизатора ISATAP. Выполните следующие действия.<br/><br/>1.  Чтобы убедиться, что клиенты DirectAccess можно получить доступ из интрасети, необходимо изменить маршрут IPv6, чтобы трафик маршрута по умолчанию перенаправляется на сервер удаленного доступа. Если в вашей интрасети пространства IPv6-адресов используется адрес, отличный от одного 48-битный префикс IPv6-адреса, необходимо указать префикс IPv6 соответствующие организации во время развертывания.<br/>2.  Если вы подключены к Интернету IPv6, необходимо настроить трафика маршрута по умолчанию, таким образом, чтобы он перенаправляется на сервер удаленного доступа, а затем настройте соответствующие подключения и маршруты на сервере удаленного доступа, таким образом, чтобы направлять по умолчанию трафик перенаправляется на устройстве, подключенном к Интернету IPv6.|  
|Существующее развертывание ISATAP|При наличии существующей инфраструктуры ISATAP, во время развертывания запрашиваются 48-битного префикса организации, и сервер удаленного доступа не изменяет сам маршрутизатора ISATAP. Чтобы убедиться, что клиенты DirectAccess можно получить доступ из интрасети, необходимо изменить инфраструктуру маршрутизации IPv6, чтобы трафик маршрута по умолчанию перенаправляется на сервер удаленного доступа. Это изменение должно выполняться на существующие маршрутизатора ISATAP, к которому клиенты интрасети должен уже пересылку трафика по умолчанию.|  
|Нет существующего подключения IPv6|Когда мастер настройки удаленного доступа обнаруживает, что сервер имеет не собственным или основанным на ISATAP IPv6-подключения, он автоматически наследуется на основе 6to4 48-битного префикса для интрасети и настраивает сервер удаленного доступа в качестве маршрутизатора ISATAP для предоставления IPv6 подключение к узлам ISATAP в интрасети. (Префикс основе 6to4 используется только в том случае, если сервер имеет общедоступных адресов, в противном случае префикс создается автоматически из диапазона уникальный локальный адрес).<br/><br/>Использование ISATAP выполните следующие действия.<br/><br/>1.  Регистрация имени ISATAP в DNS-сервер для каждого домена, на котором необходимо включить возможность подключения на основе ISATAP имя ISATAP разрешимое внутренний DNS-сервер на внутренний адрес IPv4 сервера удаленного доступа.<br/>2.  По умолчанию DNS-серверы под управлением Windows Server 2012, Windows Server 2008 R2, Windows Server 2008 или Windows Server 2003 препятствуют разрешения имени ISATAP, используя список блокировки глобальных запросов. Чтобы включить ISATAP, необходимо удалить имя ISATAP из списка блокировок. Дополнительные сведения см. в разделе [удалить ISATAP из DNS глобального списка блокировки запросов](https://go.microsoft.com/fwlink/p/?LinkId=168593).<br/><br/>Узлы ISATAP на основе Windows, которые могут разрешить имя ISATAP автоматически настроить адрес, с сервера удаленного доступа следующим образом:<br/><br/>1.  На основе ISATAP IPv6-адрес для интерфейса туннелирования ISATAP<br/>2.  64-разрядных маршрут, который обеспечивает обмен данными на другие узлы ISATAP в интрасети<br/>3.  Маршрут IPv6 по умолчанию, который указывает на сервере удаленного доступа. Маршрут по умолчанию гарантирует, что узлы ISATAP в интрасети можно связаться с клиентами DirectAccess<br/><br/>Когда узлы ISATAP на основе Windows получить адрес IPv6 на основе ISATAP, они начинают использовать инкапсулированные ISATAP трафик для связи, если местом назначения является также узел ISATAP. Так как ISATAP использует 64-разрядных подсеть для Вся интрасеть, связи переход из сегментированной модели IPv4 связи, модели обмена данными одной подсети с поддержкой IPv6. Это может повлиять на поведение некоторых служб Active Directory домена (AD DS) и приложения, зависящие от конфигурации Active Directory — сайты и службы. Например если вы использовали Active Directory — сайты и оснастку служб для настройки сайтов, подсетей на основе протокола IPv4, межсайтовом транспорте для пересылки запросов на серверы в сайтах, эта конфигурация не используется узлы ISATAP.<br/><br/><ol><li>Чтобы настроить Active Directory — сайты и службы для пересылки в пределах узлов для узлы ISATAP, для каждого объекта подсети IPv4, необходимо настроить эквивалентный объект подсети IPv6, в котором префикс IPv6-адреса для подсети выражает один и тот же узел ISATAP из диапазона адреса, что и подсеть IPv4. Например, для IPv4 192.168.99.0/24 подсети и 64-разрядный адрес ISATAP prefix 2002:836b:1:8000:: / 64, эквивалентные префикс IPv6-адреса для подсети IPv6 объекта является 2002:836b:1:8000:0:5efe:192.168.99.0/120. Для произвольной IPv4 префикс длины (значение 24, в примере) можно определить соответствующие длины префикса IPv6 из формулы 96 + IPv4PrefixLength.</li><li>Для IPv6-адресов клиентов DirectAccess добавьте следующие строки:<br/><br/><ul><li>Для клиентов DirectAccess на основе Teredo: Подсеть IPv6 для диапазона 2001:0:WWXX:YYZZ:: / 64, в какие WWXX: YYZZ версия двоеточие шестнадцатиричное первый выходом в Интернет IPv4-адрес сервера удаленного доступа. .</li><li>Для клиентов DirectAccess, на основе IP-HTTPS: Подсеть IPv6 для диапазона 2002:WWXX:YYZZ:8100:: / 56, в какие WWXX: YYZZ — это версия двоеточие шестнадцатиричное первый выходом в Интернет IPv4 адреса (w.x.y.z) сервера удаленного доступа. .</li><li>Для клиентов DirectAccess на базе 6to4: Ряд основе 6to4 префиксы IPv6, которые начинаются с 2002: и представляют региональных и открытым префиксы адресов IPv4, которые управляются назначенный IANA Internet Numbers Authority () и региональной реестров. Префикс основе 6to4 для общедоступный префикс w.x.y.z/n IPv4-адрес является 2002:WWXX:YYZZ:: / [16 + n], в какие WWXX: YYZZ — двоеточие шестнадцатиричное версию w.x.y.z.<br/><br/>        Например диапазон 7.0.0.0/8 управляется Американский реестр для Интернета номера (ARIN) для Северной Америки. Соответствующий префикс 6to4 этот общедоступный IPv6-адрес диапазона является 2002:700:: / 24. Сведения о пространство общедоступных адресов IPv4, см. в разделе [реестре пространства адресов IPv4 IANA](https://www.iana.org/assignments/ipv4-address-space/ipv4-address-space.xml). .</li></ul></li></ol>|  
  
> [!IMPORTANT]  
> Убедитесь, что у вас нет общедоступных IP-адресов во внутреннем интерфейсе сервера DirectAccess. Если у вас есть общедоступный IP-адрес во внутреннем интерфейсе, обмен данными через ISATAP может завершиться ошибкой.  
  
### <a name="plan-firewall-requirements"></a>Планирование требований к брандмауэру  
Если сервер DirectAccess расположен за пограничным брандмауэром и работает в IPv4-интернете, для трафика DirectAccess потребуются следующие исключения:  
  
-   Для IP-HTTPS: Порт назначения 443 для передачи управления протокола (TCP) и TCP-порт источника 443 для исходящих подключений.  
  
-   Для трафика Teredo: Порт назначения 3544 для пользователя датаграмм протокола (UDP) входящих подключений и UDP-порт источника 3544 для исходящих подключений.  
  
-   Для трафик 6to4. Протокол 41 для входящих и исходящих.  
  
    > [!NOTE]  
    > В отношении трафика Teredo и 6to4 эти исключения должны применяться для обоих последовательных общедоступных IPv4-адресов для выхода в Интернет на сервере удаленного доступа.  
    >   
    > Для IP-HTTPS исключения должны применяться для адреса, который регистрируется на общедоступных DNS-сервера.  
  
-   Если вы развертываете удаленный доступ с одним сетевым адаптером и установка сервера сетевых расположений на сервере удаленного доступа, TCP-порт 62000.  
  
    > [!NOTE]  
    > Это исключение устанавливается на сервере удаленного доступа, а предыдущие исключения, на пограничном брандмауэре.  
  
Следующие исключения для трафика удаленного доступа необходимы, когда сервер удаленного доступа находится в сети IPv6:  
  
-   Протокол IP 50  
  
-   UDP-порт назначения 500 для входящих подключений и UDP-порт источника 500 для исходящих подключений.  
  
-   Трафик ICMPv6 входящего и исходящего трафика, (только при использовании Teredo).  
  
Если вы используете дополнительные брандмауэры, примените следующие исключения брандмауэра внутренней сети для трафика удаленного доступа:  
  
-   Для ISATAP: Протокол 41 для входящих и исходящих  
  
-   Для всех типов трафика IPv4/IPv6: TCP/UD  
  
-   Для Teredo: Протокол ICMP для всех типов трафика IPv4/IPv6  
  
### <a name="plan-certificate-requirements"></a>Планирование требований к сертификатам  
Существуют три сценария, которым требуются сертификаты при развертывании одного сервера удаленного доступа.  
  
-   **Проверка подлинности IPsec**: Требования к сертификатам для IPsec включают сертификат компьютера, который используется клиентскими компьютерами DirectAccess при установке соединения IPsec с сервером удаленного доступа и сертификат компьютера, который используется серверами удаленного доступа для установки Соединения IPsec с клиентами DirectAccess.  
  
    Для установки DirectAccess в Windows Server 2012 использование сертификатов IPsec не является обязательным. Кроме того сервер удаленного доступа может действовать как прокси-сервер для проверки подлинности Kerberos без запроса сертификатов. Если используется проверка подлинности Kerberos, он работает по протоколу SSL и протокол Kerberos использует сертификат, который был настроен для IP-HTTPS. В некоторых корпоративных сценариях (включая многосайтового развертывания и проверки подлинности клиента одноразовых паролей) требуется использовать сертификаты, а не Kerberos.  
  
-   **Сервер IP-HTTPS**: При настройке удаленного доступа, сервер удаленного доступа автоматически настраивается в качестве веб-прослушивателя IP-HTTPS. Узлу IP-HTTPS требуется сертификат веб-сайта (CRL), клиентские компьютеры должны иметь доступ к узлу со списком отзыва (CRL) для этого сертификата.  
  
-   **Сервер сетевых расположений**: Сервер сетевых расположений — это веб-сайт, определяющий, находятся ли клиентские компьютеры в корпоративной сети. Серверу сетевых расположений требуется сертификат веб-сайта. Клиенты DirectAccess должны иметь доступ к сайту CRL для этого сертификата.  
  
В следующей таблице приводится сертификационным требованиям центра сертификации (ЦС) для каждого из этих сценариев.  
  
|Проверка подлинности IPsec|Сервер IP-HTTPS|Сервер сетевого размещения|  
|------------|----------|--------------|  
|Необходим внутренний центр сертификации для выдачи сертификатов сервера удаленного доступа и клиентам для проверки подлинности IPsec, если не используется протокол Kerberos для проверки подлинности.|Внутренний ЦС: Вы можете использовать внутренний ЦС для выдачи сертификата IP-HTTPS. Однако следует убедиться, что точка распространения CRL доступна из внешней сети.|Внутренний ЦС: Вы можете использовать внутренний ЦС для выдачи сертификата веб-сайта сервера сетевых расположений. Убедитесь в абсолютной доступности точки распределения списка отзыва сертификатов из внутренней сети.|  
||Самозаверяющий сертификат: Можно использовать самозаверяющий сертификат для сервера IP-HTTPS. Самозаверяющие сертификаты неприменимы к многосайтовым развертываниям.|Самозаверяющий сертификат: Можно использовать самозаверяющий сертификат сетевого расположения сервера веб-сайта; Тем не менее нельзя использовать самозаверяющий сертификат в развертываниях на нескольких сайтах.|  
||Общедоступный ЦС: Мы рекомендуем использовать общедоступный ЦС для выдачи сертификата IP-HTTPS, это гарантирует, что точка распространения CRL имеется внешний доступ.||  
  
#### <a name="plan-computer-certificates-for-ipsec-authentication"></a>Планирование сертификатов компьютера для проверки подлинности IPsec  
При использовании проверки подлинности IPsec на основе сертификата сервера удаленного доступа и клиенты необходимо получить сертификат компьютера. Чтобы установить сертификаты проще всего использовать групповую политику, чтобы настроить автоматическую регистрацию для сертификатов компьютеров. При этом все участники домена будут получать сертификат от корпоративного ЦС. Если у вас в вашей организации ЦС предприятия, см. в разделе [служб сертификатов Active Directory](https://technet.microsoft.com/library/cc770357.aspx).  
  
К сертификату применяются следующие требования:  
  
-   Сертификат должен иметь расширенное использование ключа (EKU) проверки подлинности клиента.  
  
-   Клиент и сертификаты сервера должны быть связаны с одного корневого сертификата. Корневой сертификат выбирается в параметрах конфигурации DirectAccess.  
  
#### <a name="plan-certificates-for-ip-https"></a>Планирование сертификатов для IP-HTTPS  
Сервер удаленного доступа действует в качестве прослушивателя IP-HTTPS, и вы можете вручную установить на сервер сертификат веб-сайта HTTPS. При планировании учитывайте следующее:  
  
-   Чтобы список отзыва сертификатов был легко доступен, рекомендуется использовать публичный ЦС.  
  
-   В соответствующем поле укажите IPv4-адрес Интернет-адаптера сервера удаленного доступа или полное ДОМЕННОЕ имя URL-адрес IP-HTTPS (адрес подключения). Если сервер удаленного доступа расположен за устройством NAT, следует указать общедоступное имя или адрес этого устройства.  
  
-   Общее имя сертификата должно совпадать с именем узла IP-HTTPS.  
  
-   Для **расширенное использование ключа** поле, используйте идентификатор объекта (OID) проверки подлинности сервера.  
  
-   В поле **Точки распространения CRL** укажите точку распространения CRL, доступную клиентам DirectAccess, подключенным к Интернету.  
  
    > [!NOTE]  
    > Это только необходимые для клиентов под управлением Windows 7.  
  
-   У сертификата IP-HTTPS должен быть закрытый ключ.  
  
-   Сертификат IP-HTTPS необходимо импортировать непосредственно в личное хранилище сертификатов.  
  
-   В имени сертификатов IP-HTTPS может быть постановочный знак.  
  
#### <a name="plan-website-certificates-for-the-network-location-server"></a>Планирование сертификатов веб-сайта для сервера сетевых расположений  
При планировании веб-сайта сервера сетевых расположение, учитывайте следующее:  
  
-   В поле **Субъект** введите IP-адрес интерфейса интрасети сервера сетевых расположений или полное доменное имя URL-адреса сервера сетевых расположений.  
  
-   Для **расширенное использование ключа** поле, используйте идентификатор Объекта проверки подлинности сервера.  
  
-   Для **точки распространения CRL** поле, используйте точку распространения списка отзыва Сертификатов, доступную клиентам DirectAccess, подключенным к интрасети. Точка распределения CRL должна быть недоступна за пределами внутренней сети.  
  
> [!NOTE]  
> Убедитесь, что сертификаты для IP-HTTPS и сервера сетевых расположений имеют имя субъекта. Если сертификат использует альтернативное имя, он не будет принят мастером удаленного доступа.  
  
#### <a name="plan-dns-requirements"></a>Планирование требования к DNS  
В этом разделе объясняется, требования к DNS для клиентов и серверов в развертывание удаленного доступа.  
  
##### <a name="directaccess-client-requests"></a>Клиентские запросы DirectAccess  
Служба DNS используется для разрешения запросов от клиентских компьютеров DirectAccess, расположенных вне внутренней сети. Клиенты DirectAccess пытаются подключиться к сервер сетевых расположений DirectAccess для определения того, что они расположены в Интернете или в корпоративной сети.  
  
-   Если подключение установлено успешно, клиенты не будут определены в интрасети, DirectAccess не используется и клиентские запросы разрешаются с помощью DNS-сервер, настроенный в сетевом адаптере клиентского компьютера.  
  
-   Если подключение не устанавливается, предполагается, что клиенты находятся в Интернете. Чтобы определить, какой DNS-сервер будет использоваться для обработки запросов имен, клиенты DirectAccess используют таблицу политики разрешения имен (NRPT). Вы можете указать, что для разрешения имен при доступе DirectAccess должен использоваться DNS64 или альтернативный внутренний DNS-сервер.  
  
При разрешении имен клиентами DirectAccess используется NRPT, чтобы определять, каким образом должен быть обработан запрос. Клиенты запрашивают полное доменное имя или однокомпонентное имя например <https://internal>. Если запрошено однокомпонентное имя, к нему дополняется DNS-суффикс, чтобы получить полное доменное имя. Если запрос DNS соответствует записи в таблице NRPT и для записи указан DNS4 или сервер DNS в интрасети, запрос отправляется для разрешения имен с помощью указанного сервера. Если совпадение существует, но сервер DNS не указан, правило исключения, а обычное разрешение имен применяется.  
  
Если для консоли управления удаленным доступом в NRPT добавляется новый суффикс, DNS-серверов по умолчанию для этого суффикса можно выполнить автоматическое обнаружение, щелкнув **найти** кнопку. Автоматическое обнаружение работает следующим образом:  
  
-   Если корпоративная сеть основана IPv4 или IPv4-и IPv6, адресом по умолчанию служит адрес DNS64 внутреннего адаптера сервера удаленного доступа.  
  
-   Если корпоративная сеть основана на IPv6, адресом по умолчанию служит IPv6-адрес DNS-серверов корпоративной сети.  
  
##### <a name="infrastructure-servers"></a>Серверы инфраструктуры  
  
-   **Сервер сетевых расположений**  
  
    Клиенты DirectAccess пытаются подключиться к серверу сетевых расположений, чтобы определить, находятся ли они во внутренней сети. Клиенты во внутренней сети должны иметь возможность разрешить имя сервера сетевых расположений и их необходимо запретить разрешения имени в том случае, если они расположены в Интернете. Для этого по умолчанию полное доменное имя сервера сетевых расположений добавляется в таблицу NRPT как правило исключения. Кроме того, при настройке удаленного доступа автоматически создаются следующие правила:  
  
    -   DNS-суффикс правило для корневого домена или имя домена сервера удаленного доступа, и IPv6-адресов, соответствующих DNS-серверы интрасети настроены на сервере удаленного доступа. Например, если сервер удаленного доступа причастен к домену corp.contoso.com, создается правило для суффикса DNS corp.contoso.com.  
  
    -   Правило исключения для полного доменного имени сервера сетевых расположений. Например, если URL-адрес сетевого расположения сервера <https://nls.corp.contoso.com>, будет создано правило исключения для полного доменного ИМЕНИ nls.corp.contoso.com.  
  
-   **Сервер IP-HTTPS**  
  
    Сервер удаленного доступа действует как прослушиватель IP-HTTPS и использует соответствующие сертификаты для проверки подлинности клиентов IP-HTTPS. Имя IP-HTTPS должно разрешаться клиентами DirectAccess, использующими общедоступные DNS-серверы.  
  
##### <a name="connectivity-verifiers"></a>Проверки подключения  
Служба удаленного доступа создает веб-проба по умолчанию, которая используется клиентскими компьютерами DirectAccess для проверки подключения к внутренней сети. Чтобы убедиться, что проба работает, как ожидается, необходимо вручную зарегистрировать следующие имена в DNS:  
  
-   **DirectAccess-webprobehost** должно разрешаться на внутренний адрес IPv4 сервера удаленного доступа или IPv6-адресом в среде IPv6.  
  
-   **DirectAccess-corpconnectivityhost** должно разрешаться в адрес локального узла (замыкание на себя). Необходимо создать записи A и AAAA. Значение записи A — 127.0.0.1, а значение записи AAAA состоит из префикса NAT64 с последними 32 разрядами, которые представляют 127.0.0.1. Префикс NAT64 можно получить, выполнив **Get-netnatTransitionConfiguration** командлета Windows PowerShell.  
  
    > [!NOTE]  
    > Это допустимо только в средах с поддержкой только IPv4. В IPv4 и IPv6 или среде только для IPv6, создать только запись AAAA с петлевым IP-адресом:: 1.  
  
Можно создать дополнительные проверки подключения, используя другие веб-адреса по протоколу HTTP или PING. Для каждого средства проверки подключения должна существовать DNS-запись.  
  
##### <a name="dns-server-requirements"></a>Требования к DNS-серверу  
  
-   Для клиентов DirectAccess необходимо использовать DNS-сервер под управлением Windows Server 2012, Windows Server 2008 R2, Windows Server 2008, Windows Server 2003 или любой DNS-сервер с поддержкой IPv6.  
  
-   Следует использовать DNS-сервер, который поддерживает динамические обновления. Можно использовать DNS-серверы, которые не поддерживают динамические обновления, но затем необходимо вручную обновить записи.  
  
-   Полное доменное имя точки распространения списка отзыва Сертификатов должно разрешаться с помощью DNS-серверы Интернета. Например если URL-адрес <https://crl.contoso.com/crld/corp-DC1-CA.crl> в **точки распространения CRL** поле сертификата IP-HTTPS сервера удаленного доступа, необходимо убедиться, что полное доменное имя crld.contoso.com разрешается с помощью DNS-серверы Интернета.  
  
#### <a name="plan-for-local-name-resolution"></a>Планирование локального разрешения имен  
При планировании для локального разрешения имен, учитывайте следующее:  
  
##### <a name="nrpt"></a>Таблица политики разрешения имен  
Может потребоваться создание дополнительных имен правила политики (разрешения имен NRPT) таблицы в следующих ситуациях:  
  
-   Необходимо добавить дополнительные суффиксы DNS для пространства имен интрасети.  
  
-   Полные доменные имена точек распространения CRL, основанные на пространства имен интрасети, необходимо добавить правила исключения для полных доменных имен точек распространения списка отзыва Сертификатов.  
  
-   Если используется разделенная среда DNS, необходимо добавить правила исключения для имен ресурсов, для которых требуется настроить клиенты DirectAccess, расположенных в Интернете для доступа к Интернет-версии, а не интрасети.  
  
-   Если вы перенаправляете трафик на внешний веб-сайт через прокси-сервера веб-серверов интрасети, внешний веб-сайт доступен только из интрасети. Чтобы разрешить входящие запросы в нем адреса прокси-сервера веб-серверов. В этом случае добавьте правило исключения для полного доменного ИМЕНИ внешнего веб-сайта и указать, что правило использует ваш сервер веб-прокси интрасети, а не IPv6-адреса DNS-серверов интрасети.  
  
    Например предположим, что вы тестируете внешних веб-сайт Test.contoso.com. Это имя не разрешается с помощью DNS-серверы Интернета, но сервер веб-прокси Contoso знает, как для разрешения имени и как направить запросы для веб-сайта на внешний веб-сервер. Чтобы не позволить пользователям за пределами интрасети Contoso получить доступ к сайту, внешний веб-сайт разрешает только запросы от IPv4-адреса веб-прокси Contoso в Интернете. Таким образом пользователи интрасети можно доступ к веб-сайта, поскольку они используют веб-прокси Contoso, но пользователи DirectAccess невозможно, так как они не используют веб-прокси Contoso. Если настроить правило исключения NRPT для сайта test.contoso.com, который использует веб-прокси Contoso, запросы веб-страницы для test.contoso.com направляются на сервер веб-прокси интрасети через IPv4-интернет.  
  
##### <a name="single-label-names"></a>Однокомпонентные имена  
Единый имена меток, таких как <https://paycheck>, иногда используются для серверов интрасети. Если запрашивается одно описательное имя и настроен список поиска суффикса DNS, DNS-суффиксы в списке будут добавляться на одно описательное имя. Например, когда пользователь на компьютере, является членом типов домена corp.contoso.com <https://paycheck> веб-браузера, полное доменное имя, сформированного как имя является paycheck.corp.contoso.com. По умолчанию добавляемый суффикс основан на основной DNS-суффикс клиентского компьютера.  
  
> [!NOTE]  
> В сценарии несвязанном имя пространства (где один или несколько компьютеров домена имеет DNS-суффикс, который не соответствует домену Active Directory, в которую входят компьютеры) следует убедиться, что список поиска настраивается в включают все необходимые суффиксы. По умолчанию мастер удаленного доступа настраивает Active Directory DNS имя основной DNS-суффикс на клиенте. Добавьте суффикс DNS, используемый клиентами для разрешения имен.  
  
Если несколько доменов и служба имя Windows Internet (WINS) развернуты в вашей организации, и вы подключаетесь удаленно, однокомпонентные имена можно устранить следующим образом:  
  
-   Развернув зона прямого просмотра WINS в DNS. При попытке разрешить имя computername.dns.zone1.corp.contoso.com, которому направлен запрос на WINS-сервер, который использует только имя компьютера. Клиент считает, что выдает обычный запрос записи DNS A, но это фактически это запрос NetBIOS.  
  
    Дополнительные сведения см. в разделе [Управление зоной прямого просмотра](https://technet.microsoft.com/library/cc816891(WS.10).aspx).  
  
-   Добавив DNS-суффикс (например, dns.zone1.corp.contoso.com) в объект групповой Политики домена по умолчанию.  
  
##### <a name="split-brain-dns"></a>Разделенная система DNS  
Разделенная система DNS — это использование одного домена DNS для разрешения имен Интернета и интрасети.  
  
Для развертываний с разделенной DNS необходимо указать полные доменные имена, которые дублируются в Интернете и интрасети и определить, какие ресурсы следует клиент DirectAccess reach интрасети или Интернет-версии. Если необходимо, чтобы клиенты DirectAccess для доступа к Интернет-версии, необходимо добавить соответствующее полное доменное имя как правило исключения NRPT для каждого ресурса.  
  
В среде с разделенной DNS если требуется, чтобы обе версии ресурс станет доступным, настройте ресурсы интрасети с именами, которые не дублируют имена, которые используются в Интернете. Предложите пользователям использовать альтернативное имя при доступе к ресурсу в интрасети. Например, настроить www\.internal.contoso.com для внутреннего имени www\.contoso.com.  
  
В среде без разделения DNS пространство имен Интернета отличается от пространства имен интрасети. Например, корпорация Contoso использует имя contoso.com в Интернете и имя corp.contoso.com в интрасети. Так как все ресурсы интрасети используют DNS-суффикс corp.contoso.com, правило NRPT для corp.contoso.com направляет все запросы DNS-имен для ресурсов интрасети на DNS-серверы интрасети. Запросы DNS-имен с суффиксом contoso.com не соответствуют правилу пространства имен интрасети corp.contoso.com в таблице NRPT и их отправки в DNS-серверы Интернета. В развертывании без разделения DNS из-за отсутствия дублирующихся полных доменных имен для ресурсов интрасети и Интернета, дополнительная настройка таблицы NRPT не требуется. Клиенты DirectAccess могут получить доступ к ресурсами интрасети и Интернета для своей организации.  
  
##### <a name="plan-local-name-resolution-behavior-for-directaccess-clients"></a>Планирование локальное разрешение имен для клиентов DirectAccess  
Если имя не может быть разрешена с помощью DNS, в службе клиента DNS в Windows Server 2012, Windows 8, Windows Server 2008 R2 и Windows 7 можно использовать локальное разрешение имен, с разрешения имени локальных многоадресной рассылки (LLMNR) и NetBIOS через TCP/IP protocols, чтобы устранить  Имя локальной подсети. Локальное разрешение имен обычно требуется для однорангового соединения, если компьютер находится в частных сетях, например в домашних сетях с одной подсетью.  
  
Если клиентская служба DNS выполняет локальное разрешение имен серверов интрасети, а компьютер подключен к общей подсети в Интернете, злоумышленники могут перехватывать запросы LLMNR и NetBIOS через TCP/IP сообщения, чтобы определить имена серверов интрасети. На странице "DNS" мастера установки сервера инфраструктуры можно настроить локальное разрешение имен на основе типов ответов, полученных из DNS-серверов интрасети. Доступны следующие параметры:  
  
-   **Использовать локальное разрешение имен, если имя не существует в DNS**: Этот параметр наиболее надежен, так как клиент DirectAccess выполняет локальное разрешение имен только для имен серверов, которые не удалось сопоставить с помощью DNS-серверов интрасети. Если DNS-серверы интрасети доступны, имена серверов интрасети разрешаются. Если DNS-серверы интрасети недоступны или возникли другие типы ошибок DNS, имена серверов интрасети не попадают в подсеть при локальном разрешении имен.  
  
-   **Использовать локальное разрешение имен, если имя не существует в DNS или DNS-серверы недоступны, когда клиентский компьютер находится в частной сети (рекомендуется)** : Рекомендуется использовать этот параметр, потому что он позволяет применять локальное разрешение имен в частной сети, только если DNS-серверы интрасети недоступны.  
  
-   **Использовать локальное разрешение имен для любых ошибок разрешения DNS (наименее безопасное)** : Это наименее безопасный вариант, так как имена серверов интрасети могут попасть в подсеть при локальном разрешении имен.  
  
#### <a name="plan-the-network-location-server-configuration"></a>Планирование конфигурации сети расположение сервера  
Сервер сетевых расположений — это веб-сайт, определяющий, находятся ли клиенты DirectAccess в корпоративной сети. Клиенты в корпоративной сети не используют DirectAccess для доступа к внутренним ресурсам; но подключаются к ним напрямую.  
  
Веб-сайта сервера сетевых расположение может размещаться на сервере удаленного доступа или на другом сервере в вашей организации. Если вы размещаете сервер сетевых расположений на сервере удаленного доступа, веб-сайт создается автоматически при развертывании удаленного доступа. Если вы размещаете сервер сетевых расположений на другом сервере под управлением ОС Windows, необходимо убедиться в том, установку на этом сервере Internet Information Services (IIS), и что создается веб-сайта. Удаленный доступ не настраивает параметры на сервер сетевых расположений.  
  
Убедитесь, что веб-сайта сервера сетевых расположение соответствует следующим требованиям:  
  
-   Имеет сертификат HTTPS сервера.  
  
-   Имеет высокий уровень доступности к компьютерам во внутренней сети.  
  
-   Нет доступа к клиентским компьютерам DirectAccess в Интернете.  
  
-  
  
Кроме того рассмотрите следующие требования для клиентов, когда вы настраиваете веб-сайта сервера сетевого расположения:  
  
-   клиентские компьютеры DirectAccess должны доверять ЦС, который выдал сертификат сервера для веб-сайта сервера сетевых расположений;  
  
-   клиентские компьютеры DirectAccess во внутренней сети должны иметь возможность для разрешения имени сайта сервера сетевых расположений;  
  
##### <a name="plan-certificates-for-the-network-location-server"></a>Планирование сертификатов для сервера сетевых расположений  
При получении сертификата веб-сайта для сервера сетевых расположений учитывайте следующее:  
  
-   В поле **Субъект** введите IP-адрес интерфейса интрасети сервера сетевых расположений или полное доменное имя URL-адреса сервера сетевых расположений.  
  
-   Для **расширенное использование ключа** поле, используйте идентификатор Объекта проверки подлинности сервера.  
  
-   Сертификат сервера сетевых расположений должен проверяться список отзыва сертификатов (CRL). Для **точки распространения CRL** поле, используйте точку распространения списка отзыва Сертификатов, доступную клиентам DirectAccess, подключенным к интрасети. Точка распределения CRL должна быть недоступна за пределами внутренней сети.  
  
##### <a name="plan-dns-for-the-network-location-server"></a>Планирование DNS для сервера сетевых расположений  
Клиенты DirectAccess пытаются подключиться к серверу сетевых расположений, чтобы определить, находятся ли они во внутренней сети. Клиенты во внутренней сети должны иметь возможность разрешения имени сервера сетевых расположений. Если клиент находится во внешней сети, эта возможность должна исключаться. Для этого по умолчанию происходит добавление полного доменного имени сервера сетевых расположений в качестве правила исключения в NRPT.  
  
### <a name="plan-management-servers-configuration"></a>Планирование конфигурации серверов управления  
DirectAccess клиенты инициируют взаимодействие с серверами управления, которые предоставляют службы, такие как Windows Update и антивирусных обновлений. Клиенты DirectAccess также используют протокол Kerberos для проверки подлинности на контроллеры домена, чтобы доступ к внутренней сети. При удаленном управлении клиентами DirectAccess серверы управления взаимодействуют с клиентскими компьютерами для осуществления функций управления, например оценки программных или аппаратных активов. Служба удаленного доступа может автоматически обнаруживать некоторые серверы управления, в том числе:  
  
-   Контроллеры домена: Автоматическое обнаружение контроллеров домена выполняется для доменов, содержащие клиентские компьютеры, а также для всех доменов в одном лесу с сервером удаленного доступа.  
  
-   Серверы System Center Configuration Manager  
  
Контроллеры домена и System Center Configuration Manager автоматически обнаруживаются, серверы первом DirectAccess настроен. Контроллеры домена не отображаются в консоли, но параметры могут быть получены с помощью командлетов Windows PowerShell. Если контроллер домена или серверы System Center Configuration Manager изменяются, щелкнув **обновления серверов управления** в консоли обновляет список серверов управления.  
  
**Требования к серверу управления**  
  
-   Серверы управления должны быть доступны через туннель инфраструктуры. Если добавить серверы в список серверов управления при настройке удаленного доступа, они автоматически становятся доступными через этот туннель.  
  
-   Серверы управления, которые инициируют подключения, чтобы клиенты DirectAccess должны полностью поддерживать IPv6, с помощью собственного IPv6-адрес или с помощью адреса, назначенного ISATAP.  
  
### <a name="plan-active-directory-requirements"></a>Планирование требований к Active Directory  
Удаленный доступ использует Active Directory следующим образом:  
  
-   **Проверка подлинности**: Туннель инфраструктуры использует проверку подлинности NTLMv2 для учетной записи компьютера, подключающегося к серверу удаленного доступа, и учетная запись должна быть в домене Active Directory. Туннель интрасети использует проверку подлинности Kerberos для пользователя для создания туннеля интрасети.  
  
-   **Объекты групповой политики**: Удаленный доступ конфигурационные параметры собираются в объектах групповой политики (GPO), которой применяются к серверов удаленного доступа, клиентов и внутренними серверами приложений.  
  
-   **Группы безопасности**. Удаленный доступ использует группы безопасности, чтобы собрать и идентифицировать компьютеры клиентов DirectAccess. Объекты групповой политики применяются к группам безопасности.  
  
При планировании среды Active Directory для развертывания удаленного доступа, рассмотрите следующие требования:  
  
-   По крайней мере на одном контроллере домена устанавливается в операционной системе Windows Server 2012, Windows Server 2008 R2 Windows Server 2008 или Windows Server 2003.  
  
    Если контроллер домена в сети периметра (и доступен с выходом в Интернет сетевого адаптера сервера удаленного доступа), заблокируйте доступ к нему сервера удаленного доступа. Необходимо добавить фильтры пакетов на контроллере домена, чтобы предотвратить подключение к IP-адрес Интернет-адаптера.  
  
-   Сервер удаленного доступа должен быть членом домена.  
  
-   Клиенты DirectAccess должны входить в состав домена. Клиенты могут принадлежать к:  
  
    -   Любому домену в одном лесу с сервером удаленного доступа.  
  
    -   Любому домену, имеющему двустороннее доверие с доменом сервера удаленного доступа.  
  
    -   Любому домену в лесу, имеющем двустороннее доверие с лесом с доменом сервера удаленного доступа.  
  
> [!NOTE]  
> -   Сервер удаленного доступа нее может быть контроллером домена.  
> -   Контроллер домена Active Directory, который используется для удаленного доступа не должен быть доступен из внешнего Интернет-адаптера сервера удаленного доступа (адаптер не должен быть в доменном профиле брандмауэра Windows).  
  
#### <a name="plan-client-authentication"></a>Планирование клиентской проверки подлинности  
Возможности удаленного доступа в Windows Server 2012 можно выбрать с помощью встроенной проверки подлинности Kerberos, который использует имена пользователей и пароли, или сертификаты для проверки подлинности IPsec.  
  
**Проверка подлинности Kerberos**: Если вы решили использовать учетные данные Active Directory для проверки подлинности, DirectAccess сначала использует проверку подлинности Kerberos для компьютера, и затем использует проверку подлинности Kerberos для пользователя. При использовании этого режима проверки подлинности, DirectAccess применяет один туннель безопасности, предоставляющий доступ к DNS-сервер, контроллер домена и любой другой сервер во внутренней сети  
  
**Проверка подлинности IPsec**: Если вы решили использовать двухфакторную проверку подлинности или защита доступа к сети, DirectAccess применяет два туннеля безопасности. Мастер настройки удаленного доступа настраивает правила безопасности подключений в брандмауэре Windows в режиме повышенной безопасности. Эти правила позволяют задать следующие учетные данные при согласовании безопасности IPsec для сервера удаленного доступа:  
  
-   Туннель инфраструктуры использует учетные данные сертификата компьютера для первой проверки подлинности и учетные данные пользователя (NTLMv2) для проверки подлинности. Учетные данные пользователя принудительно использовать проверку подлинности протокола IP (AuthIP), и они обеспечивают доступ к DNS-сервер и контроллер домена, прежде чем клиент DirectAccess может использовать учетные данные Kerberos для туннеля интрасети.  
  
-   Туннель интрасети использует учетные данные сертификата компьютера для первой проверки подлинности и учетные данные пользователя (Kerberos V5) для проверки подлинности.  
  
#### <a name="plan-multiple-domains"></a>Планирование нескольких доменов  
В список серверов управления должны входить контроллеры домена из всех доменов, содержащих группы безопасности с клиентскими компьютерами DirectAccess. Он должен содержать все домены, содержащие учетные записи пользователей, которые могут работать на компьютерах, которые настроены как клиенты DirectAccess. Это гарантирует, что пользователи, не размещенные в одном домене с клиентским компьютером, на котором они работают, проходят проверку подлинности на контроллере домена в домене пользователя.  
  
Проверка подлинности выполняется автоматически, если домены находятся в одном лесу. Если группа безопасности с клиентских компьютеров или серверов приложений, которые находятся в разных лесах, контроллеры домена этих лесов не обнаруживаются автоматически. Леса также не обнаруживаются автоматически. Задание можно запустить **обновления серверов управления** в **управления удаленным доступом** обнаруживать эти контроллеры домена.  
  
Если это возможно, общие суффиксы доменных имен должны быть добавлены в таблицу NRPT во время развертывания удаленного доступа. Например, если у вас два домена, domain1.corp.contoso.com и domain2.corp.contoso.com, вместо добавления двух записей в таблицу NRPT, можно добавить запись общего суффикса DNS с суффиксом доменного имени corp.contoso.com. Это происходит автоматически для доменов в том же корневом каталоге. Домены, которые не в том же корневом каталоге необходимо добавить вручную.  
  
### <a name="plan-group-policy-object-creation"></a>Планирование создания объекта групповой политики  
При настройке удаленного доступа DirectAccess параметры собираются в объектах групповой политики (GPO). Двух объектов групповой политики заполняются параметрами DirectAccess, и они распределяются следующим образом:  
  
-   **Объекты групповой Политики клиента DirectAccess**: Этот GPO содержит параметры клиента, включая настройки туннелирования для IPv6, записи NRPT и правила безопасности подключения для брандмауэра Windows в режиме повышенной безопасности. Этот GPO применяется к группам безопасности, указанным для клиентского компьютера.  
  
-   **Объект групповой Политики сервера DirectAccess**: Этот GPO содержит параметры конфигурации, которые применяются к любому серверу, который настроен в качестве сервера удаленного доступа в развертывании. Он также содержит правила безопасности подключения для брандмауэра Windows в режиме повышенной безопасности.  
  
> [!NOTE]  
> Конфигурации серверов приложений не поддерживается в удаленного управления клиентами DirectAccess, так как клиенты не может получить доступ к внутренней сети сервера DirectAccess, в которых размещаются на серверах приложений. Шаг 4 на экране настройки установки удаленного доступа недоступен для этого типа конфигурации.  
  
Объекты групповой политики можно настроить автоматически или вручную.  
  
**Автоматически**: При указании, что объекты групповой политики создаются автоматически, имя по умолчанию указывается для каждого объекта групповой Политики.  
  
**Вручную**: Можно использовать объекты групповой политики, предопределенные администратором Active Directory.  
  
При настройке объектов групповой политики, необходимо учитывайте следующие предупреждения:  
  
-   После настройки DirectAccess для применения определенных GPO, перейти на другие объекты невозможно.  
  
-   Используйте следующую процедуру для резервного копирования всех удаленного доступа объектов групповой политики, перед выполнением командлетов DirectAccess:  
  
    [Резервное копирование и восстановление конфигурации удаленного доступа](https://go.microsoft.com/fwlink/?LinkID=257928).  
  
-   Используете ли вы автоматически или вручную настраиваемые GPO, необходимо добавить политику обнаружения медленного канала связи, если ваши клиенты используют 3G. Путь к **политики: Настройка групповой политики обнаружения медленных** является:  
  
    **Computer configuration/Polices/Administrative Templates/System/Group Policy**.  
  
-   Если необходимые разрешения для связывания GPO не существуют, то выдается предупреждение. Удаленный доступ операция будет продолжена, но привязывание не будет выполнено. Если это предупреждение выдается, ссылки не создается автоматически, даже если разрешения будут добавлены позже. Администратору следует создать их вручную.  
  
#### <a name="automatically-created-gpos"></a>Автоматически созданных GPO  
Учитывайте следующее при использовании автоматически созданных GPO:  
  
Автоматически созданные GPO применяются в соответствии с расположение и цель ссылки, следующим образом:  
  
-   Для объекта групповой Политики сервера DirectAccess расположение и цель ссылки указывают домена, содержащего сервер удаленного доступа.  
  
-   При создании клиента и сервера приложений объекты групповой политики, положение задается одним доменом. Поиск имени GPO в каждом домене, а домен заполнение настройками DirectAccess, если он существует.  
  
-   GPO привязывается к корневому каталогу домена, в котором он был создан. Таким образом, объект групповой политики создается в каждом домене, в котором имеются клиентские компьютеры или серверы приложений, и привязывается к его корневому каталогу.  
  
При использовании автоматически созданных GPO для применения настроек DirectAccess, администратору сервера удаленного доступа необходимы следующие разрешения:  
  
-   Разрешения на создание объектов групповой политики для каждого домена.  
  
-   Разрешения для связывания всех выбранных клиентских корневых элементов доменов.  
  
-   Разрешения для связывания корневых элементов доменов GPO сервера.  
  
-   Разрешения безопасности на создание, изменение, удаление и изменение объектов групповой политики.  
  
-   Объект групповой Политики разрешения на чтение для каждого необходимого домена. Это разрешение не является обязательным, но рекомендуется, так как он обеспечивает удаленный доступ, чтобы убедиться, что при создании объектов групповой политики объекты групповой политики с одинаковыми именами не существует.  
  
#### <a name="manually-created-gpos"></a>Вручную созданных GPO  
При использовании вручную созданных GPO учитывайте следующее:  
  
-   Мастер настройки удаленного доступа следует запускать, когда объекты групповой политики уже существуют.  
  
-   Чтобы применить параметры DirectAccess, администратору сервера удаленного доступа требуется полные разрешения, чтобы создавать, изменять, удалять и изменять вручную созданных объектов групповой политики.  
  
-   Поиск выполняется для ссылки на объект групповой Политики во всем домене. Если связи нет, она создается автоматически в корневом элементе домена. Если требуемые разрешения для создания связи недоступны, формируется предупреждение.  
  
#### <a name="recovering-from-a-deleted-gpo"></a>Восстановление удаленного объекта групповой политики  
Если объект групповой Политики на сервере удаленного доступа, клиента или сервера приложений был удален случайно, появится следующее сообщение об ошибке: **Не удается найти объект групповой Политики (имя объекта групповой Политики)** .  
  
Если имеется резервная копия, вы сможете восстановить GPO. Если резервная копия недоступна, необходимо удалить параметры конфигурации и их настройке.  
  
###### <a name="to-remove-configuration-settings"></a>Чтобы удалить параметры конфигурации  
  
1.  Выполните командлет Windows PowerShell **Uninstall-RemoteAccess**.  
  
2.  Откройте **управление удаленным доступом**.  
  
3.  Отобразится сообщение об ошибке, говорящее, что GPO не найден. Щелкните **Удалить параметры конфигурации**. После завершения сервер будет восстановлен в ненастроенном состоянии, и можно будет заново настроить параметры.
