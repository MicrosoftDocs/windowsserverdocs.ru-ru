# <a name="deploy-multiple-remote-access-servers-in-a-multisite-deployment"></a>Развертывание нескольких серверов удаленного доступа в многосайтовом развертывании

>Область применения. Windows Server (полугодовой канал), Windows Server 2016

 Windows Server 2016 и Windows Server 2012 объединить в единую роль удаленного доступа DirectAccess и удаленного доступа (RAS) VPN. Развертывание удаленного доступа возможно в нескольких корпоративных сценариях. Данный обзор представляет собой введение в корпоративный сценарий развертывания серверов удаленного доступа в конфигурации с несколькими узлами.  
  
## <a name="BKMK_OVER"></a>Описание сценария  
При многосайтовом развертывании двух или более серверов удаленного доступа или кластеры серверов развертываются и настраиваются как разные точки входа в одном месте или в географически разнесенных расположениях. Развертывание нескольких точек входа в одном месте позволяет избыточности сервера или для выравнивания серверов удаленного доступа с существующая архитектура сети. Развертывание по географическому расположению гарантирует эффективное использование ресурсов, как удаленный клиентские компьютеры могут подключаться к ресурсам внутренней сети, используя точку входа к ним ближе всего. Трафик в многосайтовом развертывании можно распределять и балансировать с помощью внешней глобальной подсистемы балансировки нагрузки.  
  
Многосайтовое развертывание поддерживает клиентские компьютеры под управлением Windows 10, Windows 8 или Windows 7. Клиентские компьютеры под управлением Windows 10 или Windows 8 автоматически определить точку входа или пользователь может вручную выбрать точку входа. Автоматическое назначение происходит в следующем порядке:  
  
1.  Используйте точку входа вручную выбранного пользователем.  
  
2.  Используйте точку входа, определяется внешней глобальной подсистемы балансировки нагрузки, если один развертывается.  
  
3.  Используйте ближайший точка записи с клиентского компьютера с помощью механизма автоматического пробы.  
  
Поддержка клиентов под управлением Windows 7 необходимо включить вручную в каждой точки входа и выбора точки входа, эти клиенты не поддерживается.  
  
## <a name="prerequisites"></a>Предварительные требования  
Перед началом развертывания этого сценария ознакомьтесь со списком важных требований.  
  
-   [Развертывание одиночного сервера DirectAccess с расширенными параметрами](../../directaccess/single-server-advanced/Deploy-a-Single-DirectAccess-Server-with-Advanced-Settings.md) должны быть развернуты перед многосайтового развертывания.  
  
-   Клиенты Windows 7 всегда подключается к конкретному сайту. Они не смогут подключиться к ближайшему узлу, на основе расположения клиента (в отличие от клиентов Windows 10, 8 или 8.1).  
  
-   Изменение политик вне консоли управления DirectAccess или командлетов PowerShell не поддерживается.  
  
-   Должна быть развернута инфраструктура открытого ключа.  
  
    Подробную информацию см. в следующих разделах: [Проверьте мини-модуль руководства по лаборатории: Базовая PKI для Windows Server 2012.](https://social.technet.microsoft.com/wiki/contents/articles/7862.test-lab-guide-mini-module-basic-pki-for-windows-server-2012.aspx)  
  
-   Корпоративной сети должен быть включен протокол IPv6. Если вы используете протокол ISATAP, необходимо удалить его и перейти на собственный IPv6.  
  
## <a name="in-this-scenario"></a>Содержание сценария  
Сценарий многосайтового развертывания включает несколько этапов.  
  
1. [Развертывание одиночного сервера DirectAccess с расширенными параметрами](../../directaccess/single-server-advanced/Deploy-a-Single-DirectAccess-Server-with-Advanced-Settings.md). Перед настройкой многосайтового развертывания должны развертываться единого сервера удаленного доступа с расширенными параметрами.  
  
2. [Планирование многосайтового развертывания](plan/Plan-a-Multisite-Deployment.md). Для создания количество дополнительное планирование многосайтового развертывания с одного сервера действия являются обязательными, а также соблюдение стандартов с несколькими сайтами предварительные требования и планирование для групп безопасности Active Directory, объекты групповой политики (GPO), DNS и параметров клиента.  
  
3. [Настройка многосайтового развертывания](configure/Configure-a-Multisite-Deployment.md). Это состоит из нескольких этапов настройки, включая подготовку инфраструктуры Active Directory, конфигурации существующего сервера удаленного доступа и добавление нескольких серверов удаленного доступа в качестве точки входа для многосайтового развертывания.  
  
4. [Диагностика многосайтового развертывания](troubleshoot/Troubleshoot-a-Multisite-Deployment.md). Этот раздел по устранению неполадок описываются некоторые наиболее распространенные ошибки, которые могут возникнуть при развертывании удаленного доступа в многосайтовом развертывании.
  
## <a name="BKMK_APP"></a>Практическое применение  
Многосайтовое развертывание предоставляет следующие возможности.  
  
-   Улучшенная производительность A многосайтового развертывания позволяет клиентским компьютерам, доступ к внутренним ресурсам при помощи удаленного доступа для подключения с использованием точки входа ближайший и наиболее подходящий. Клиент получить доступ к внутренним ресурсам эффективно и Повышена скорость клиентского Интернет запросов маршрутизируются через DirectAccess. Трафик по точкам входа может быть сопоставлено с помощью внешней глобальной подсистемы балансировки нагрузки.  
  
-   Простота из управления Мультисайт позволяет администраторам согласовать развертывания удаленного доступа к развертыванию сайтов Active Directory, создания упрощенной архитектуры. Общие параметры можно легко установить на серверы точек входа или кластерах. Можно управлять параметрами удаленного доступа из любого из серверов в развертывание, или удаленно с помощью средств удаленного администрирования сервера (RSAT). Кроме того весь многосайтового развертывания можно отслеживать с помощью единой консоли управления удаленным доступом.

## <a name="BKMK_HARD"></a>Требования к оборудованию  
Для этого сценария действуют следующие требования к оборудованию.  
  
-   Минимум два компьютера удаленного доступа должны создаваться в многосайтовом развертывании.   
  
-   Для проверки данного сценария, необходим по крайней мере один компьютер под управлением Windows 8, настроенный в качестве клиента DirectAccess. Для тестирования сценария для клиентов под управлением Windows 7 необходим по крайней мере один компьютер под управлением Windows 7.  
  
-   Для балансировки нагрузки трафика между серверами точки входа, сторонних внешней глобальной подсистемы балансировки нагрузки является обязательным.  
  
## <a name="BKMK_SOFT"></a>Требования к программному обеспечению  
Для этого сценария действуют следующие требования к программному обеспечению.  
  
-   Требования к программному обеспечению для развертывания на одном сервере.  
  
-   Помимо требования к программному обеспечению для одного сервера существуют multisite-конкретных требований:  
  
    -   Проверка подлинности IPsec требования в многосайтового развертывания DirectAccess должны быть развернуты с помощью проверки подлинности сертификата машины IPsec. Параметр, чтобы выполнять проверку подлинности IPsec, используя сервер удаленного доступа в качестве прокси-сервер Kerberos не поддерживается. Внутренний ЦС необходим для развертывания сертификатов IPsec.  
  
    -   IP-HTTPS и сети расположение сервера требования сертификатов, необходимых для IP-HTTPS и сервера сетевых расположений должен быть выдан центром сертификации. Параметр, чтобы использовать сертификаты, которые автоматически выдаются и самостоятельно подписанный сервером удаленного доступа не поддерживается. Сертификаты могут быть выданы внутренним ЦС или по сторонних внешнего центра сертификации.  
  
    -   Требования Active Directory-необходим хотя бы один сайт Active Directory. Сервер удаленного доступа должен находиться на сайте. Сократить время обновления рекомендуется что каждый сайт имеет контроллером домена с возможностью записи, хотя это не обязательно.  
  
    -   Далее приведены требования группы безопасности.  
  
        -   Одной группе безопасности является обязательным для всех клиентских компьютеров Windows 8 из всех доменов. Рекомендуется создать группу безопасности из этих клиентов для каждого домена.  
  
        -   Группа безопасности, содержащая компьютеров под управлением Windows 7 является обязательным для каждой точки входа, которые настроены для поддержки клиентов Windows 7. Рекомендуется иметь группы безопасности, уникальные для каждой точки входа в каждом домене.  
  
        -   Компьютеры должны входить только в одну группу безопасности, включающую клиентов DirectAccess. Если клиенты включены в несколько групп, разрешение имен для клиентских запросов будет работать некорректно.  
  
    -   Требования к объектов групповой политики объекта групповой Политики можно вручную создать перед настройкой удаленного доступа или автоматически создается во время развертывания службы удаленного доступа. Требования таковы:  
  
        -   Объект групповой Политики уникальных клиентских является обязательным для каждого домена.  
  
        -   Объект групповой Политики сервера является обязательным для каждой точки входа в домен, в котором находится точка входа. Таким образом, если несколько точек входа, находятся в одном домене, объекты групповой политики (по одному для каждой точки входа) в домене с несколькими серверами.  
  
        -   Уникальный объект групповой Политики Windows 7 клиента является обязательным для каждой точки входа включена для поддержки клиента Windows 7, для каждого домена.  
  
## <a name="KnownIssues"></a>Известные проблемы  
Ниже перечислены известные проблемы при настройке сценарий с несколькими сайтами:  
  
-   **Несколько точек входа в той же подсети IPv4**. Добавление нескольких точек входа в той же подсети IPv4 приведет к появлению адрес конфликт IP-адрес и адрес DNS64 для точки входа не будет настроен должным образом. Эта проблема возникает, когда не был развернут IPv6 на внутренних интерфейсов серверов в корпоративной сети. Чтобы избежать этой проблемы, выполните следующую команду Windows PowerShell на все текущие и будущие серверы удаленного доступа:  
  
    ```  
    Set-NetIPInterface -InterfaceAlias <InternalInterfaceName> -AddressFamily IPv6 -DadTransmits 0  
    ```  
  
-   Если общедоступный адрес, указанный для клиентов DirectAccess для подключения к серверу удаленного доступа имеет суффикса, включенных в NRPT, DirectAccess может не работать должным образом. Убедитесь в наличии NRPT исключения для общедоступного имени. При многосайтовом развертывании исключения должны быть добавлены для общедоступных имен всех точек входа. Обратите внимание, что если Принудительное туннелирование включено эти исключения, добавляются автоматически. Они удаляются при отключении Принудительное туннелирование.  
  
-   При использовании командлета Windows PowerShell **Disable DAMultiSite**, параметры WhatIf и Confirm бесполезны и Мультисайт будет отключена, и объекты групповой политики для Windows 7 будут удалены.  
  
-   Когда клиенты Windows 7, с помощью приложения для устройства при Многосайтовом развертывании будут обновлены до Windows 8, помощник по подключению к сети не будет работать. Эту проблему можно устранить до обновления клиента путем изменения объектов групповой политики Windows 7, используя следующие командлеты Windows PowerShell:  
  
    ```  
    Set-GPRegistryValue -Name <Windows7GpoName> -Domain <DomainName> -Key "HKEY_LOCAL_MACHINE\SOFTWARE\Policies\Microsoft\Windows\NetworkConnectivityAssistant" -ValueName "TemporaryValue" -Type Dword -Value 1  
    Remove-GPRegistryValue -Name <Windows7GpoName> -Domain <DomainName> -Key "HKEY_LOCAL_MACHINE\SOFTWARE\Policies\Microsoft\Windows\NetworkConnectivityAssistant"  
    ```  
  
    В случае, если клиент уже была обновлена, затем переместите клиентский компьютер в группу безопасности Windows 8.  
  
-   При изменении параметров контроллера домена, с помощью командлета Windows PowerShell **Set-DAEntryPointDC**, если указан параметр ComputerName командлета — это сервер удаленного доступа в точку входа, за исключением последнего добавлен в Мультисайт развертывания, предупреждение будет отображаться, указывающее, что сервер, указанный не будут обновлены только при следующем обновлении политики. Фактический сервер (серверы), который не был обновлен можно увидеть при помощи **состояние конфигурации** в **панели МОНИТОРИНГА** из **консоли управления удаленным доступом**. Это не вызовет никаких функциональных нарушений, тем не менее, можно выполнить **gpupdate/force** на серверах, который не был обновлен, чтобы немедленно обновить состояние конфигурации.  
  
-   При развертывании многосайтового в только с IPv4 корпоративной сети, изменив внутренний сетевой префикс IPv6 также адрес DNS64 изменения, но не обновляет адрес в правила брандмауэра, разрешающие DNS-запросы к службе DNS64. Чтобы устранить эту проблему, выполните следующие команды Windows PowerShell после изменения префикса IPv6 внутренней сети:  
  
    ```  
    $dns64Address = (Get-DAClientDnsConfiguration).NrptEntry | ?{ $_.DirectAccessDnsServers -match ':3333::1' } | Select-Object -First 1 -ExpandProperty DirectAccessDnsServers  
  
    $serverGpoName = (Get-RemoteAccess).ServerGpoName  
  
    $serverGpoDc = (Get-DAEntryPointDC).DomainControllerName  
  
    $gpoSession = Open-NetGPO -PolicyStore $serverGpoName -DomainController $serverGpoDc  
  
    Get-NetFirewallRule -GPOSession $gpoSession | ? {$_.Name -in @('0FDEEC95-1EA6-4042-8BA6-6EF5336DE91A', '24FD98AA-178E-4B01-9220-D0DADA9C8503')} |  Set-NetFirewallRule -LocalAddress $dns64Address  
  
    Save-NetGPO -GPOSession $gpoSession  
    ```  
  
-   Если DirectAccess развернут в том случае, когда инфраструктура ISATAP присутствовал, при удалении точки входа, который был у узла ISATAP IPv6-адрес DNS64 службы будут удалены из адреса DNS-серверов все DNS-суффиксов в NRPT.  
  
    Чтобы устранить эту проблему в **установки сервера инфраструктуры** мастера **DNS** странице, удалите DNS-суффиксы, которые были изменены и снова добавить правильные адреса DNS-сервера, щелкнув  **Обнаружить** на **адреса DNS-серверов** диалоговое окно.
