# <a name="step-2-plan-the-multisite-infrastructure"></a>Шаг 2 плана многосайтового инфраструктуры

>Область применения. Windows Server (полугодовой канал), Windows Server 2016

Следующий шаг в развертывании удаленного доступа в многосайтового топологии — это выполнить планирование многосайтового инфраструктуры; включая Active Directory, группы безопасности и объектов групповой политики.  
## <a name="bkmk_2_1_AD"></a>2.1 Планирование Active Directory  
Мультисайтовое развертывание удаленного доступа можно настроить в ряд топологий:  
  
-   **Один сайт Active Directory, несколько точек входа**— в этой топологии у вас есть один сайт Active Directory для всей организации с помощью быстрого интранет-ссылки на веб-узле, но у вас есть развертывания нескольких серверов удаленного доступа в организации, каждого действующего в качестве точки входа. Географическое пример этой топологии является один сайт Active Directory для точки входа на восточном побережье "и" западное побережье США.  
  
    ![Многоузловой инфраструктуры](../../../../media/Step-2-Plan-the-Multisite-Infrastructure/RAMultisiteTopo1.png)  
  
-   **Множественные сайты Active Directory, несколько точек входа**— в этой топологии у вас есть два или несколько сайтов Active Directory с сервером удаленного доступа, развернутые как точку входа для каждого сайта. Каждый сервер удаленного доступа, связанный с контроллера домена Active Directory для сайта. Географическое пример этой топологии является сайт Active Directory для США и один для Европы с одной точкой входа для каждого сайта. Обратите внимание на то, что при наличии нескольких сайтов Active Directory вы не обязательно должны иметь точку входа, связанных с каждого сайта. Кроме того некоторые сайты Active Directory может иметь более одной точки входа, связанные с ней.  
  
    ![Топология с несколькими сайтами](../../../../media/Step-2-Plan-the-Multisite-Infrastructure/RAMultisiteTopo2.png)  
  
В точке входа можно настроить один сервер удаленного доступа, нескольких серверов удаленного доступа или кластеры серверов удаленного доступа.   
  
### <a name="active-directory-best-practices-and-recommendations"></a>Рекомендации по работе Active Directory и рекомендации  
Обратите внимание на то, следующие рекомендации и ограничения для развертывания службы Active Directory в сценарий с несколькими сайтами:  
  
1.  Каждый сайт Active Directory может содержать один или несколько серверов удаленного доступа или кластер серверов, работает в качестве точки входа для клиентских компьютеров. Тем не менее в сайт Active Directory не требуется иметь точку входа.  
  
2.  Точка входа может быть связан только с одного сайта Active Directory. При подключении клиентских компьютеров под управлением Windows 8 для конкретной точкой входа, то они рассматриваются как относящиеся к сайт Active Directory, связанный с этой точки входа.  
  
3.  Рекомендуется наличие контроллера домена для каждого сайта Active Directory. Контроллер домена может быть только для чтения.  
  
4.  Если каждый узел Active Directory содержит контроллер домена, объект групповой Политики для сервера в точке входа управляет один из контроллеров домена в сайте Active Directory, связанный с конечной точкой. Если нет ни одного контроллера домена, доступного для записи на этом сайте, объект групповой Политики для сервера осуществляется на контроллере домена, доступного для записи, которую можно найти ближайший к первому серверу удаленного доступа, настроенного в точке входа. Ближайший определяется ссылку расчет стоимости. Обратите внимание, что в этом случае после изменения конфигурации возможны задержки при репликации между контроллером домена, управление объект групповой Политики и контроллер домена только для чтения в Active Directory узла сервера.  
  
5.  Клиентские объекты групповой политики и сервер приложений необязательно, управляемых объектов групповой политики на контроллере домена под управлением в роли эмулятора основного контроллера домена (PDC). Это означает, что клиентские объекты групповой политики не находятся под обязательно в сайте Active Directory, содержащей точку входа для клиентов, которые подключиться.  
  
6.  Если контроллер домена для сайта Active Directory недоступна, сервер удаленного доступа будет подключаться к дополнительного контроллера домена на сайте (если доступно). В противном случае он подключается к контроллеру домена для другого сайта, чтобы получить обновленный объект групповой Политики и для проверки подлинности клиентов. В обоих случаях консоли управления удаленным доступом и командлеты PowerShell не может использоваться для извлечения или изменения параметров конфигурации, пока не будет доступен контроллер домена. Обратите внимание на следующее условия:  
  
    1.  В случае недоступности сервера, работающего в качестве эмулятора основного контроллера домена необходимо указать доступный контроллер домена с обновленным объекты групповой политики в роли эмулятора основного контроллера домена.  
  
    2.  В случае недоступности контроллера домена, который управляет объекта групповой Политики сервера используйте командлет PowerShell Set-DAEntryPointDC должен быть сопоставлен точку входа нового контроллера домена. Новый контроллер домена должен иметь актуальные объекты групповой политики перед выполнением командлета.  
  
## <a name="bkmk_2_2_SG"></a>2.2 Планирование групп безопасности  
Во время развертывания одного сервера с расширенными параметрами все клиентские компьютеры, доступ к внутренней сети через DirectAccess были собраны в группу безопасности. При многосайтовом развертывании эта группа безопасности используется для Windows 8 клиентских компьютеров. Для многосайтовом развертывании клиентские компьютеры Windows 7 будут собраны в разные группы безопасности для каждой точки входа многосайтового развертывания. К примеру Если вы ранее сгруппированы все клиентские компьютеры в группе DA_Clients, необходимо теперь удалить все компьютеры Windows 7 из этой группы и поместить их в группу безопасности. Например на нескольких сайтах Active Directory, несколько ярлыки топологии, создайте группу безопасности для точки входа United States (DA_Clients_US) и один для точки входа Европа (DA_Clients_Europe). Поместите клиентских компьютеров Windows 7, расположенный в United States DA_Clients_US группы и все находится в Европе, в группе DA_Clients_Europe. Если у вас все клиентские компьютеры Windows 7, планирование групп безопасности для компьютеров под управлением Windows 7 не нужно.  
  
Ниже приведены необходимые группы безопасности.  
  
-   Одну группу безопасности для всех клиентских компьютеров Windows 8. Рекомендуется создать группу безопасности для этих клиентов для каждого домена.  
  
-   Группа безопасности входят компьютеры клиентов Windows 7 для каждой точки входа. Рекомендуется создать группу, уникальный для каждого домена. Пример: Domain1\DA_Clients_Europe; Domain2\DA_Clients_Europe; Domain1\DA_Clients_US; Domain2\DA_Clients_US.  
  
## <a name="bkmk_2_3_GPO"></a>2.3. Планирование объектов групповой политики  
Настройки DirectAccess, выполненные во время развертывания удаленного доступа, собираются в объектах групповой политики. Развертывание одного сервера уже использует объекты групповой политики клиентов DirectAccess, сервера удаленного доступа и (необязательно) для серверов приложений. Многосайтовое развертывание требует следующие объекты групповой политики:  
  
-   Сервер объекта групповой Политики для каждой точки входа.  
  
-   Объект групповой Политики для каждого домена, содержащего клиентские компьютеры под управлением Windows 8.  
  
-   Объект групповой Политики для каждой точки входа и каждого домена, которые входят компьютеры клиентов Windows 7.  
  
Объекты групповой политики можно настроить следующим образом:  
  
-   **Автоматически**-можно указать, что объекты групповой политики создаются автоматически службой удаленного доступа. Имя по умолчанию указывается для каждого объекта групповой Политики и могут быть изменены.  
  
-   **Вручную**-можно использовать объекты групповой политики, которые были созданы вручную администратором Active Directory.  
  
> [!NOTE]  
> После настройки DirectAccess для использования определенные объекты групповой политики, его нельзя настроить другие объекты.  
  
### <a name="231-automatically-created-gpos"></a>2.3.1 автоматически созданных GPO  
При автоматическом создании объектов групповой политики учитывайте следующее:  
  
-   Автоматически созданные GPO применяются в соответствии с расположением и целевым параметром связи следующим образом:  
  
    -   Для GPO сервера, как расположение и параметры связи соответствуют к домену, в котором находится сервер удаленного доступа.  
  
    -   Для клиентских объектов групповой политики цель ссылки устанавливается для корневого домена, в котором был создан объект групповой Политики. Объект групповой Политики создается для каждого домена, содержащего клиентские компьютеры, а объект групповой Политики будет связан в корень каждого домена. .  
  
-   Для автоматически созданных GPO для применения настроек DirectAccess удаленного доступа администратор сервера необходимы следующие разрешения:  
  
    -   Разрешения для требуемые домены для создания GPO.  
  
    -   Разрешение на привязывание к корневым каталогам всех выбранных клиентских доменов.  
  
    -   Разрешение на создание фильтра WMI для объектов групповой политики является обязательным, если DirectAccess были настроены для работы только на мобильных компьютерах.  
  
    -   Разрешения ссылки на привязывание к доменов, связанных с точки входа (доменов GPO сервера)  
  
    -   Администратору удаленного доступа рекомендуется иметь разрешение на чтение GPO для каждого необходимого домена. Это позволяет убедиться, что не существует объектов групповой политики с одинаковыми именами при создании объектов групповой политики для многосайтового развертывания.  
  
    -   Разрешения репликации Active Directory в домене, связанном с точками входа. Это обусловлено тем, при первоначальном добавлении точек входа, групповой Политики сервера для точки входа создается на контроллере домена, ближайший к точке входа. Тем не менее с момента создания связей поддерживается только для эмулятора основного контроллера домена, объект групповой Политики должны быть реплицированы на контроллере домена, на котором он был создан на контроллер домена под управлением в роли эмулятора основного контроллера домена, перед созданием ссылки.  
  
Обратите внимание, что если правильные разрешения для репликации и связывания GPO не существуют, то выдается предупреждение. Удаленный доступ операция будет продолжена, но репликации и связывание не будет выполнено. Если предупреждение выдается ссылку ссылки не создается автоматически, даже в том случае, если требуются разрешения на месте. Вместо этого администратору придется создавать связи вручную.  
  
### <a name="232-manually-created-gpos"></a>2.3.2 вручную созданных GPO  
При создании объектов групповой политики вручную учитывайте следующее:  
  
-   Следующие объекты групповой политики, создаваемых для многосайтового развертывания вручную:  
  
    -   **Объект групповой Политики сервера**- объекта групповой Политики сервера для каждой точки входа (в домене, в котором находится точка входа). Этот объект групповой Политики применяются на каждом сервере удаленного доступа в точке входа.  
  
    -   **Объекты групповой Политики (Windows 7) клиента**- объект групповой Политики для каждой точки входа и каждого домена, которые входят компьютеры клиентов Windows 7, которые будут подключаться к точкам входа многосайтового развертывания. Например Domain1\DA_W7_Clients_GPO_Europe; Domain2\DA_W7_Clients_GPO_Europe; Domain1\DA_W7_Clients_GPO_US; Domain2\DA_W7_Clients_GPO_US. Если нет клиентских компьютеров Windows 7 будет подключаться к точкам входа, объекты групповой политики не являются обязательными.  
  
-   Не требуется для создания дополнительных объектов групповой политики для Windows 8 клиентских компьютеров. Объект групповой Политики для каждого домена, которые входят компьютеры клиентов уже был создан при развертывании одного сервера удаленного доступа. При многосайтовом развертывании этих клиентских объектов групповой политики будет работать в качестве клиентов объекты групповой политики для Windows 8.  
  
-   Объекты групповой политики должна существовать перед нажатием кнопки фиксации в мастерах многосайтового развертывания.  
  
-   При использовании объектов групповой политики, созданных вручную, выполняется поиск связи объекта по всему домену. Если объект групповой политики не связан с доменом, он автоматически привязывается к корневому каталогу. При отсутствии разрешения на создание связей выносится предупреждение.  
  
-   При использовании вручную созданных объектов групповой политики для применения настроек DirectAccess администратор сервера удаленного доступа требуются разрешения полного доступа (редактирование, удаление и изменение параметров безопасности) к GPO, созданные вручную.  
  
Обратите внимание, что если правильные разрешения для репликации и связывания GPO не существуют, то выдается предупреждение. Удаленный доступ операция будет продолжена, но репликации и связывание не будет выполнено. Если предупреждение выдается ссылку ссылки не создается автоматически, даже в том случае, если требуются разрешения на месте. Вместо этого администратору придется создавать связи вручную.  
  
### <a name="233-managing-gpos-in-a-multi-domain-controller-environment"></a>2.3.3 управления объектами групповой политики в среде с несколькими контроллерами домена  
Каждый объект групповой Политики будут управляться определенным контроллером домена, следующим образом:  
  
-   GPO сервера управляет один из контроллеров домена в сайте Active Directory, связанного с сервером, или если контроллеры домена на этом сайте доступны только для чтения, ближайший к первому серверу в записи контроллером домена, доступного для записи точек.  
  
-   Клиентские объекты групповой политики будет управлять контроллером домена, запущенным в роли эмулятора основного контроллера домена.  
  
Если вы хотите вручную Примечание параметры групповой Политики следующие режимы.  
  
-   Для сервера объектов групповой политики, чтобы определить, какой контроллер домена связан с конкретной точкой входа, выполните командлет PowerShell `Get-DAEntryPointDC -EntryPointName <name of entry point>`.  
  
-   По умолчанию при внесении изменений с помощью сетевых командлетов PowerShell или консоли управления групповыми политиками, используется контроллер домена, выполняющий роль эмулятора основного контроллера домена.  
  
-   Если вы измените параметры на контроллере домена, который не является контроллером домена, связанные с точки входа (для GPO сервера) или эмулятора PDC (для клиента групповой политики), обратите внимание на следующее:  
  
    1.  Перед изменением параметров, убедитесь, что контроллер домена будут реплицированы с помощью обновленного GPO, и [резервное копирование параметров объекта групповой Политики](https://go.microsoft.com/fwlink/?LinkID=257928), перед внесением изменений. Если объект групповой Политики не обновляется, во время репликации могут возникнуть конфликты объединения, что может привести к повреждению конфигурации удаленного доступа.  
  
    2.  После внесения изменений в параметры, необходимо дождаться изменения реплицируются на контроллер домена, который связан с объектами групповой политики. Не следует вносить дополнительные изменения, с помощью консоли управления удаленным доступом или командлеты PowerShell удаленного доступа, пока не закончится репликация. Если GPO изменяется на двух разных контроллеров домена до завершения репликации, могут возникнуть конфликты объединения, приводит к повреждению конфигурации  
  
-   В качестве альтернативы можно изменить параметр по умолчанию, используя **Изменение контроллера домена** диалоговое окно, показанное на консоли управления групповыми политиками или с помощью **Open-NetGPO** командлет PowerShell, чтобы изменения выполнено с помощью консоли или сетевых командлетов использовать контроллер домена, указанную вами.  
  
    1.  Для этого в консоли управления групповыми политиками, щелкните правой кнопкой мыши домен или контейнер сайтов и нажмите кнопку **Сменить контроллер домена**.  
  
    2.  Для этого в PowerShell, укажите параметр DomainController Open-NetGPO командлета. Например чтобы включить частные и общедоступные профили в брандмауэре Windows на объект групповой Политики с именем domain1\DA_Server_GPO _Europe, используя контроллер домена с именем europe-dc.corp.contoso.com, сделайте следующее:  
  
        ```  
        $gpoSession = Open-NetGPO -PolicyStore "domain1\DA_Server_GPO _Europe" -DomainController "europe-dc.corp.contoso.com"  
        Set-NetFirewallProfile -GpoSession $gpoSession -Name @("Private","Public") -Enabled True  
        Save-NetGPO -GpoSession $gpoSession  
        ```  
  
#### <a name="modifying-domain-controller-association"></a>Изменение сопоставления контроллера домена  
Для поддержания согласованности конфигурации в многосайтовом развертывании важно, чтобы каждым объектом групповой политики управлял один контроллер домена. В некоторых случаях он может потребоваться назначить другого контроллера домена для объекта групповой Политики:  
  
-   **Обслуживание контроллеров домена и время простоя**-когда контроллер домена, который управляет объект групповой Политики не станет доступным, он может потребоваться объект групповой политики на другом контроллере домена.  
  
-   **Оптимизация конфигурации распространения**-после изменения сетевой инфраструктуры, он может потребоваться для управления групповой Политики сервера точки входа на контроллере домена, в том же сайте Active Directory, как точка входа.   
  
## <a name="bkmk_2_4_DNS"></a>2.4. Планирование DNS  
При планировании DNS для многосайтового развертывания Обратите внимание на следующее:  
  
1.  Клиентские компьютеры используют адрес ConnectTo для подключения к серверу удаленного доступа. Каждой точки входа в вашем развертывании требуется другой адрес ConnectTo. Каждый адрес ConnectTo точки входа должны быть доступны в общедоступной службе DNS и адрес, который выбран должно совпадать с именем субъекта сертификата IP-HTTPS, который развертывается для подключения IP-HTTPS.   
  
2.  Кроме того удаленного доступа обеспечивает симметричной маршрутизации; Таким образом при включении многосайтового развертывания, могут подключаться только клиенты Teredo и IP-HTTPS. Чтобы разрешить собственных клиентов IPv6 для подключения, адрес подключения (URL-адрес IP-HTTPS) должно разрешаться как внешний (Интернет) адреса IPv6 и IPv4 на сервере удаленного доступа.  
  
3.  Служба удаленного доступа создает веб-проба по умолчанию, которая используется клиентскими компьютерами DirectAccess для проверки подключения к внутренней сети. Во время настройки одиночного сервера следующие имена должны были зарегистрированы в DNS:  
  
    1.  DirectAccess-WebProbeHost — должно устранить на внутренний адрес IPv4 сервера удаленного доступа или IPv6-адресом в среде IPv6.  
  
    2.  DirectAccess-CorpConnectivityHost — должно устранить адрес localhost (замыкание на себя) (либо:: 1 или 127.0.0.1, в зависимости от того, развернут ли IPv6 в корпоративной сети).  
  
    При многосайтовом развертывании дополнительных запись DNS для directaccess-WebProbeHost должны создаваться для каждой точки входа. При добавлении точки входа, для автоматического создания этой записи дополнительных directaccess-WebProbeHost пытается удаленного доступа. Тем не менее если он не будет отображаться предупреждение, и необходимо создать запись вручную.  
  
    > [!NOTE]  
    > Если очистку DNS в инфраструктуре DNS включена, рекомендуется отключить очистки записи DNS, автоматически созданные удаленного доступа.
