# <a name="step-3-plan-the-multisite-deployment"></a>Шаг 3 план многосайтового развертывания

>Область применения. Windows Server (полугодовой канал), Windows Server 2016

После планирования инфраструктуры многосайтового, Планирование требований любой дополнительный сертификат, каким образом клиентские компьютеры выбирать точки входа и IPv6-адресов, назначенных в развертывании.  

Следующие разделы предоставляют подробные сведения о планировании.
  
## <a name="bkmk_3_1_IPHTTPS"></a>3.1 сертификатов IP-HTTPS, планирование  
При настройке точки входа, настройкой каждой точки входа с определенным адресом подключения. Сертификат IP-HTTPS для каждой точки входа должно соответствовать адресу ConnectTo. При получении сертификата, обратите внимание на следующее:  
  
-   Самозаверяющие сертификаты нельзя использовать при многосайтовом развертывании.  
  
-   Чтобы список отзыва сертификатов был легко доступен, рекомендуется использовать публичный ЦС.  
  
-   В соответствующем поле укажите IPv4-адрес внешнего адаптера сервера удаленного доступа (если адрес подключения задан как IP-адресом, а не DNS-имя), или полное ДОМЕННОЕ имя URL-адрес IP-HTTPS.  
  
-   Общее имя сертификата должно соответствовать имени веб-узла, IP-HTTPS. Также поддерживается использование подстановочным знаком URL-адрес, соответствующий подключения DNS-имя.  
  
-   Сертификаты IP-HTTPS можно использовать подстановочные знаки в имени субъекта. Один и тот же сертификат подстановочный знак можно использовать для всех точек входа.  
  
-   В поле "Улучшенное использование ключа" используйте идентификатор объекта проверки подлинности сервера.  
  
-   Если вы пользуетесь клиентские компьютеры под управлением Windows 7 в многосайтовое развертывание, в поле точки распространения списков отзыва Сертификатов укажите точку распространения списка отзыва Сертификатов, доступную клиентам DirectAccess, подключенным к Интернету. Это не является обязательным для клиентов под управлением Windows 8 (по умолчанию проверка отзыва CRL отключена для IP-HTTPS на этих клиентах).  
  
-   У сертификата IP-HTTPS должен быть закрытый ключ.  
  
-   Сертификат IP-HTTPS необходимо импортировать непосредственно в личном хранилище компьютера, а не пользователя.  
  
## <a name="bkmk_3_2_NLS"></a>3.2. Планирование сервера сетевых расположений  
Веб-сайта сервера сетевых расположение может размещаться на сервере удаленного доступа или на другом сервере в вашей организации. Если вы размещаете сервер сетевых расположений на сервере удаленного доступа, веб-сайт создается автоматически при развертывании удаленного доступа. Если вы размещаете сервер сетевых расположений на другом сервере под управлением ОС Windows в вашей организации, необходимо убедиться в том, установку Internet Information Services (IIS) для создания веб-сайта.  
  
### <a name="321-certificate-requirements-for-the-network-location-server"></a>3.2.1 требования к сертификатам для сервера сетевых расположений  
Убедитесь, что веб-сайта сервера сетевых расположение соответствует следующим требованиям для развертывания сертификата:  
  
-   Требуется сертификат сервера HTTPS.  
  
-   Если сервер сетевых расположений находится на сервере удаленного доступа, и вы выберите использование самозаверяющего сертификата при развертывании одного сервера удаленного доступа, необходимо перенастроить развертывание одного сервера, чтобы использовать сертификат, выданный внутренним ЦС.  
  
-   клиентские компьютеры DirectAccess должны доверять ЦС, который выдал сертификат сервера для веб-сайта сервера сетевых расположений;  
  
-   Клиентские компьютеры DirectAccess во внутренней сети должен быть способен разрешить имя веб-сайта сервера сетевых расположение.  
  
-   Веб-сайта сервера сетевых расположение необходим высокий уровень доступности, к компьютерам во внутренней сети.  
  
-   сервер сетевых расположений не должен быть доступен клиентским компьютерам DirectAccess в Интернете;  
  
-   Сертификат сервера должен быть проверен для списка отзыва сертификатов (CRL).  
  
-   Групповые сертификаты не поддерживаются, если сервер сетевых расположений размещен на сервере удаленного доступа.  
  
При получении сертификата веб-сайта для сервера сетевых расположений, обратите внимание на следующее:  
  
1.  В соответствующее поле следует ввести IP-адрес интерфейса интрасети сервера сетевых расположений или полное доменное имя для URL сетевого расположения. Обратите внимание на то, что не следует указывать IP-адрес Если сервер сетевых расположений размещен на сервере удаленного доступа. Это так, как сервер сетевых расположений необходимо использовать одно и то же имя субъекта для всех точек входа, а не все точки входа и тот же адрес.  
  
2.  В поле "Улучшенный ключ" используйте идентификатор объекта проверки подлинности сервера.  
  
3.  Точки распространения списков отзыва Сертификатов в поле используйте точку распространения списка отзыва Сертификатов, доступную клиентам DirectAccess, подключенным к интрасети.  
  
### <a name="322dns-for-the-network-location-server"></a>3.2.2DNS для сервера сетевых расположений  
Если вы размещаете сервер сетевых расположений на сервере удаленного доступа, необходимо добавить запись DNS для сетевого расположения веб-сайта сервера для каждой точки входа в вашем развертывании. Обратите внимание на следующее условия:  
  
-   Имя субъекта первого сертификата сервера сетевых расположений многосайтового развертывания используется в качестве URL-адрес сервера сетевого расположения для всех точек входа, поэтому имя субъекта и URL-адрес сетевого расположения сервера нельзя же, как имя компьютера Первый сервер удаленного доступа в развертывании. Он должен быть выделенной для сервера сетевых расположений полное доменное имя.  
  
-   Служба, предоставляемая расположение сервера сетевой трафик распределяется среди точек входа, с помощью DNS и поэтому должен быть DNS-запись с тем же URL-адрес для каждой точки входа, внутренний IP-адрес точки входа.  
  
-   Все точки входа должны быть настроены сертификата сервера сетевых расположений с тем же именем субъекта (которое совпадает с URL-адрес сервера сетевого расположения).  
  
-   Прежде чем добавлять точку входа должны создаваться сети расположение серверной инфраструктуры (DNS и параметры сертификата) для точки входа.  
  
## <a name="bkmk_3_3_IPsec"></a>3.3 Планирование корневой сертификат IPsec для всех серверов удаленного доступа  
При планировании для проверки подлинности IPsec клиента при многосайтовом развертывании, обратите внимание на следующее:  
  
1.  Если вы решили использовать встроенный прокси-сервер Kerberos для проверки подлинности компьютера при установке одного сервера удаленного доступа, необходимо изменить параметр, чтобы использовать сертификаты компьютеров, выданный внутренним ЦС, так как прокси-сервер Kerberos не поддерживается для установку Мультисайт развертывание.  
  
2.  Если используется самозаверяющий сертификат, необходимо перенастроить развертывание одного сервера, чтобы использовать сертификат, выданный внутренним ЦС.  
  
3.  Для проверки подлинности IPsec для успешного выполнения во время проверки подлинности клиента все серверы удаленного доступа должны иметь сертификат, выданный IPsec корневым или промежуточным центром сертификации и идентификатор Объекта проверки подлинности клиента для расширенного использования ключа.  
  
4.  Таким же корнем IPsec или промежуточный сертификат устанавливается на всех серверах удаленного доступа в многосайтового развертывания.  
  
## <a name="bkmk_3_4_GSLB"></a>3.4 Планирование балансировки нагрузки глобального сервера  
При многосайтовом развертывании можно дополнительно настроить подсистему балансировки нагрузки глобального сервера. Подсистемы балансировки нагрузки глобального сервера можно использовать для вашей организации, если развертывание охватывает больших географического распределения, так как он может распределять нагрузку трафика между точками входа.  Подсистема балансировки нагрузки глобального сервера можно настроить для предоставления клиентам DirectAccess сведения о точке входа ближайшей точки входа. Процесс работает следующим образом:  
  
1.  Клиентские компьютеры под управлением Windows 10 или Windows 8 имеет список IP-адресов подсистемы балансировки нагрузки глобального серверов, каждый из которых связан с точки входа.  
  
2.  Клиентский компьютер Windows 10 или Windows 8 пытается разрешить полное ДОМЕННОЕ имя подсистемы балансировки нагрузки глобального сервера в публичном DNS в IP-адрес. Если разрешенный IP-адрес указан как подсистемы балансировки нагрузки глобального сервера IP-адрес точки входа, клиентский компьютер автоматически выбирает этой точки входа и подключении к его IP-HTTPS URL-адрес (адрес подключения), или его IP-адрес сервера Teredo. Обратите внимание, что IP-адрес подсистемы балансировки нагрузки глобального сервера совпадать адрес подключения или адрес сервера Teredo точки входа, так как клиентские компьютеры никогда не пытаться подключиться к IP-адрес балансировщика нагрузки глобального сервера не требуется.  
  
3.  Если клиентский компьютер находится за веб-прокси (и не может использовать разрешение DNS) или полного доменного ИМЕНИ балансировщика нагрузки глобального сервера не соответствует настроенные IP-адрес балансировщика нагрузки глобального сервера, а затем точку входа будет выбран автоматически при использовании зонда HTTPS для IP-HTTPS URL-адреса всех точек входа. Клиент будет подключаться к серверу, который сначала отвечает.  
  
Список устройств, поддерживающих удаленный доступ с балансировкой нагрузки глобального server, см. в статье Find страницы партнеров [Microsoft Server и облачным платформам](https://www.microsoft.com/server-cloud/).  
  
## <a name="bkmk_3_5_EP_Selection"></a>3.5 Планирование выбирать точки входа клиента DirectAccess  
При настройке многосайтового развертывания, по умолчанию Windows 10 с Windows 8 клиентские компьютеры настроены с информацией, необходимые для подключения к всех точек входа в развертывании и автоматически подключаться к единой точки входа на основе выбора алгоритм. Можно также настроить вашего развертывания, чтобы разрешить Windows 10 и Windows 8 клиентские компьютеры могли вручную выберите запись укажите которого они будут подключены. Если клиентский компьютер Windows 10 или Windows 8 в настоящее время подключен к точке входа Соединенных Штатов и включен автоматически выбирать точки входа, если точка входа Соединенных Штатах, становится недоступной, через несколько минут клиентский компьютер попытается подключиться через точку входа Europe. Рекомендуется использовать автоматически выбирать точки входа; Тем не менее позволяя точки входа вручную, что выбор позволяет конечным пользователям для подключения к точке различные операции, в зависимости от текущего состояния сети. Например если компьютер подключен к точка входа Соединенных Штатов и подключения к внутренней сети становится гораздо медленнее, чем ожидалось. В этом случае конечному пользователю вручную можно выбрать для подключения к точке входа Европе для улучшения подключения к внутренней сети.  
  
> [!NOTE]  
> После того как пользователь выберет точку входа вручную, клиентский компьютер не вернется к автоматически выбирать точки входа. То есть если точку вручную выбранную запись становится недоступной, конечному пользователю необходимо вернуться к автоматически выбирать точки входа или вручную выберите другую точку входа.  
  
 Настройки клиентских компьютеров Windows 7 со сведениями, необходимыми для подключения к одной точкой входа многосайтового развертывания. Они не удается сохранить сведения для нескольких точек входа одновременно. Например можно настроить клиентский компьютер Windows 7 соединиться с точкой входа Соединенных Штатах, но не точка входа Европе. Точку входа США недоступен, клиентский компьютер Windows 7 потеряет подключение к внутренней сети до точки входа доступен. Конечный пользователь не может внести изменения в пытаться подключиться к точке входа Europe.  
  
## <a name="bkmk_3_6_IPv6"></a>3.6 планирование префиксов и маршрутизации  
  
### <a name="internal-ipv6-prefix"></a>Внутреннего префикса IPv6  
Во время развертывания одного удаленного доступа, сервера, вы запланировали префиксы IPv6 внутренней сети, обратите внимание на следующее при многосайтовом развертывании:  
  
1.  Если бы вы добавили всех подразделений Active Directory, когда вы настроили единый сервер развертывания удаленного доступа, префиксы IPv6 внутренней сети уже определены в консоли управления удаленным доступом.  
  
2.  При создании дополнительных сайтов Active Directory для многосайтового развертывания, необходимо планирование новых префиксов IPv6 для дополнительных сайтов и их определения в режиме удаленного доступа. Обратите внимание на то, что префиксы IPv6 можно настроить только с помощью консоли управления удаленным доступом или командлеты PowerShell, если IPv6 развертывается во внутренней корпоративной сети.  
  
### <a name="ipv6-prefix-for-directaccess-client-computers-ip-https-prefix"></a>Префикс IPv6 для DirectAccess клиентские компьютеры (IP-HTTPS-префикс)  
  
1.  Если IPv6 развертывается во внутренней корпоративной сети, необходимо спланировать префикс IPv6, который назначен клиентским компьютерам DirectAccess в любые дополнительные точки входа в вашем развертывании.  
  
2.  Убедитесь, что префиксы IPv6 для назначения клиентов DirectAccess в каждой точки входа отличаются, и нет перекрывают друг друга в префиксы IPv6.  
  
3.  Если IPv6 не развертывается в корпоративной сети, IP-HTTPS-префикс для каждой точки входа выбирается автоматически при добавлении точки входа.  
  
### <a name="ipv6-prefix-for-vpn-clients"></a>Префикс IPv6 для VPN-клиентов  
Если вы развернули VPN на одном сервере удаленного доступа, обратите внимание на следующее:  
  
1.  Добавление VPN IPv6 префикс на точку входа — только требуется, если вы хотите разрешить VPN-клиента IPv6-подключения к корпоративной сети.  
  
2.  Префикс VPN можно настроить только на точку входа, с помощью консоли управления удаленным доступом или командлет PowerShell, если IPv6 развертывается во внутренней корпоративной сети, а также если включена VPN, в точке входа.  
  
3.  Префикс VPN должны быть уникальными в каждой точки входа и не должны перекрываться с другими префиксами VPN или IP-HTTPS.  
  
4.  Если IPv6 не развертывается в корпоративной сети, VPN-клиенты, подключающиеся к точке входа не назначен адрес IPv6.  
  
### <a name="routing"></a>Маршрутизация  
При многосайтовом развертывании симметричной маршрутизации обеспечивается с использованием Teredo или IP-HTTPS. Если в корпоративной сети развернут IPv6 Обратите внимание на следующее:  
  
1. Префиксы Teredo или IP-HTTPS из каждой точки входа должны маршрутизироваться в корпоративной сети на связанный сервер удаленного доступа.  
  
2. Маршруты должны настраиваться в инфраструктуре маршрутизации корпоративной сети.  
  
3. Для каждой точки входа во внутренней сети должны быть одного до трех маршруты:  
  
   1. Этот префикс IP-HTTPS-префикс выбирается администратором в окне добавления точки входа мастера.  
  
   2. Префикс VPN IPv6 (необязательно). Этот префикс можно выбрать после включения VPN для точки входа  
  
   3. Префикс Teredo (необязательно). Этот префикс используется, только в том случае, если настроены два последовательных общедоступных IPv4-адреса внешнего адаптера сервера удаленного доступа. Префикс основан на первый общедоступный адрес IPv4 пары адресов. Например, если внешние адреса:  
  
      1. www.xxx.yyy.zzz  
  
      2. www.xxx.yyy.zzz+1  
  
      Затем используется префикс Teredo для настройки 2001:0:WWXX:YYZZ:: / 64, где WWXX: YYZZ — шестнадцатеричное представление www.xxx.yyy.zzz адрес IPv4.  
  
      Обратите внимание, что можно использовать следующий сценарий для вычисления префикс Teredo.  
  
      ```  
      $TeredoIPv4 = (Get-NetTeredoConfiguration).ServerName # Use for a Remote Access server that is already configured  
      $TeredoIPv4 = "20.0.0.1" # Use for an IPv4 address  
  
          [Byte[]] $TeredoServerAddressBytes = `  
          [System.Net.IPAddress]::Parse("2001::").GetAddressBytes()[0..3] + `  
          [System.Net.IPAddress]::Parse($TeredoIPv4).GetAddressBytes() + `  
          [System.Net.IPAddress]::Parse("::").GetAddressBytes()[0..7]  
  
      Write-Host "The server's Teredo prefix is $([System.Net.IPAddress]$TeredoServerAddressBytes)/64"  
      ```  
  
   4. Все маршруты, выше должно быть направлено в IPv6-адрес внутреннего адаптера сервера удаленного доступа (или внутренний виртуальный IP-адрес (VIP) адрес для загрузки с балансировкой точки входа).  
  
> [!NOTE]  
> Если IPv6 развертывается в корпоративной сети и администрирование сервера удаленного доступа выполняется удаленно с помощью DirectAccess, маршруты для Teredo и префиксы IP-HTTPS из других точек входа должны добавляться к каждому серверу удаленного доступа, так, чтобы трафик перенаправляются к внутренней сети.  
  
### <a name="active-directory-site-specific-ipv6-prefixes"></a>Связанные с сайтом Active Directory, префиксов IPv6  
При подключении клиентский компьютер под управлением Windows 10 или Windows 8 на точку входа, клиентский компьютер находится непосредственно связанной с сайтом Active Directory точки входа и настраивается с помощью IPv6 префиксы, связанные с точкой входа. Приоритет — для клиентских компьютеров, для подключения к ресурсам с помощью эти префиксы IPv6, так как они настроены динамически в таблице политик префикс IPv6 с более высоким приоритетом при подключении к точке входа.  
  
Если ваша организация использует топологию Active Directory с узелом префиксы IPv6 (например app.corp.com полное доменное имя внутреннего ресурса размещен в Северной Америке и Европе конкретного узла IP-адрес в каждом расположении) не настраивается по по умолчанию, используя консоль удаленного доступа и отдельных сайтов префиксы IPv6 не настраиваются для каждой точки входа. Если вы хотите реализовать этот сценарий необязательно, необходимо настроить каждой точки входа с конкретными префиксами IPv6, которые должны быть предпочитаются клиентских компьютеров, подключающихся к конкретной точкой входа. Сделайте это следующим образом:  
  
1.  Для каждого объекта групповой Политики, используемый для Windows 10 или клиентских компьютеров Windows 8 выполните командлет PowerShell Set-DAEntryPointTableItem  
  
2.  Задайте для параметра EntryPointRange командлета с узелом префиксы IPv6. Например, для добавления отдельных сайтов префиксы 2001:db8:1:1:: / 64 и 2001:db:1:2:: / 64, на точку входа с именем Europe, выполните следующую команду,  
  
    ```  
    $entryPointName = "Europe"  
    $prefixesToAdd = @("2001:db8:1:1::/64", "2001:db8:1:2::/64")  
    $clientGpos = (Get-DAClient).GpoName  
    $clientGpos | % { Get-DAEntryPointTableItem -EntryPointName $entryPointName -PolicyStore $_ | %{ Set-DAEntryPointTableItem -PolicyStore $_.PolicyStore -EntryPointName $_.EntryPointName -EntryPointRange ($_.EntryPointRange) + $prefixesToAdd}}  
    ```  
  
3.  При изменении параметра EntryPointRange, убедитесь, что вы не удалите существующие, 128 бит префиксов, которые принадлежат конечные точки туннеля IPsec и адрес DNS64.  
  
## <a name="bkmk_3_7_TransitionIPv6"></a>3.7 Планирование перехода на IPv6 при развертывании многосайтового удаленного доступа  
Многие организации используют протокол IPv4 в корпоративной сети. С исчерпание доступные префиксы IPv4 во многих организациях осуществляющие переход с только с IPv4 на IPv6 облачные сети.  
  
Этот переход, скорее всего, должно выполняться в два этапа:  
  
1.  С IPv4 только для чтения к корпоративной сети IPv6 + IPv4.  
  
2.  Из IPv6 + IPv4 к корпоративной сети только протокол IPv6.  
  
В каждой части переход может выполняться поэтапно. На каждом этапе можно изменить только одну подсеть сети для новой конфигурации сети. Таким образом многосайтового развертывания DirectAccess является обязательным для поддержки гибридного развертывания, где, например, некоторые из точек входа входит в подсеть только с IPv4 и другие входит в подсеть IPv6 + IPv4. Кроме того изменения конфигурации во время перехода не должно отключать возможность подключения клиентов через DirectAccess.  
  
### <a name="TransitionIPv4toMixed"></a>Переход с IPv4 только для чтения к корпоративной сети IPv6 + IPv4  
При добавлении IPv6-адресов в корпоративной сети IPv4, может потребоваться добавить IPv6-адрес уже развернутых серверу DirectAccess. Кроме того вы можете добавить точки входа или узел для нагрузки кластера с балансировкой нагрузки с адресами IPv4 и IPv6 для развертывания DirectAccess.  
  
Удаленный доступ позволяет добавлять серверы с адресами IPv4 и IPv6 для развертывания, который изначально был настроен на поддержку только адресов IPv4. Эти серверы добавляются в качестве серверов только для IPv4 и IPv6-адресов, будут игнорироваться DirectAccess; Следовательно организации не может использовать преимущества этой исходной связи IPv6 на этих новых серверах.  
  
Для преобразования развертывания в развертывание IPv6 + IPv4 и воспользоваться возможностями собственного IPv6, необходимо переустановить DirectAccess. Для поддержания подключения клиентов по всему повторной установки см. в разделе перехода с IPv4 только для чтения к развертыванию с помощью двух развертываний DirectAccess только протокол IPv6.  
  
> [!NOTE]  
> Как и в случае с сетью только с IPv4 в смешанной сети IPv4 + IPv6, адреса DNS-сервера, который используется для разрешения запросов DNS клиента необходимо настроить DNS64, который развертывается на самих серверов удаленного доступа, а не в корпоративную службу DNS.  
  
### <a name="TransitionMixedtoIPv6"></a>Переход из IPv6 + IPv4 к корпоративной сети только для IPv6  
DirectAccess позволяет добавлять точки входа только протокол IPv6, только в том случае, если первый сервер удаленного доступа в развертывании изначально было обоих IPv4 и IPv6-адресов или только IPv6-адрес. То есть вам не удается выполнить переход из сети только с IPv4 к сети только IPv6 за один шаг без повторной установки DirectAccess. Для перехода непосредственно из сети только с IPv4 к сети только протокол IPv6, см. в разделе перехода с IPv4 только для чтения к развертыванию с помощью двух развертываний DirectAccess только протокол IPv6.  
  
После завершения перехода с IPv4-только для развертывания в развертывание IPv6 + IPv4, вы можете переходить к сети только протокол IPv6. Во время и после перехода Обратите внимание на следующее:  
  
-   Если любой IPv4-только для внутренних серверов остаются в корпоративной сети, они будут доступны клиентам, которые подключаются через точки входа только протокол IPv6.  
  
-   При добавлении точек входа только протокол IPv6 в IPv4 + IPv6 развертывание, DNS64 и NAT64 не будет включен на новых серверах. Клиенты, подключающиеся к этих точек входа будет автоматически настроен для использования корпоративных серверов DNS.  
  
-   Если вам нужно удалить IPv4-адреса с развернутой сервера, необходимо удалить сервер из развертывания DirectAccess, удалить его IPv4-адресу корпоративной сети и снова добавить его в развертывание.  
  
Для поддержки возможности подключения клиента к корпоративной сети, необходимо убедиться, что сервер сетевых расположений можно устранить, корпоративную службу DNS для IPv6-адрес. Дополнительный адрес IPv4 можно также задать, но не требуется.  
  
### <a name="DualDeployment"></a>Переход из только на IPv4 в среде только протокол IPv6, с помощью двух развертываний DirectAccess  
Переход от только IPv4 к корпоративной сети только IPv6 не может выполняться без переустановки развертывания DirectAccess. Чтобы обеспечить подключение клиента во время перехода, можно использовать другое развертывание DirectAccess. Два развертывания является обязательным, при (сеть только с IPv4 до IPv4 + IPv6) завершает первый этап перехода, и вы собираетесь Подготовка для будущих перехода только IPv6 корпоративной сети orto использовать преимущества преимуществ собственного подключения IPv6. Два развертывания описан в следующее:  
  
1.  Установите второе развертывание DirectAccess. Можно установить DirectAccess на новых серверах, или удалите серверы из первого развертывания и использовать их для второго.  
  
    > [!NOTE]  
    > При установке дополнительное развертывание DirectAccess вместе с текущей, убедитесь, что не двух точек входа имеют одинаковый префикс клиента.  
    >   
    > При установке с помощью мастера начальной настройки DirectAccess или с помощью командлета `Install-RemoteAccess`, удаленного доступа автоматически задает префикс первую точку входа для клиента в развертывание, чтобы значение по умолчанию < подсеть IPv6\_префикс >: 1000:: / 64. При необходимости, необходимо изменить префикс.  
  
2.  Удалите выбранный клиентских групп безопасности из первого развертывания.  
  
3.  Добавление клиентских групп безопасности для второго развертывания.  
  
    > [!IMPORTANT]  
    > Чтобы обеспечить возможность подключения клиентов на протяжении всего процесса, необходимо добавить в группы безопасности для второе развертывание сразу после их удаления из первого. Это гарантирует, что клиенты не обновляются с помощью объектов групповой политики DirectAccess или два, или ноль. Клиенты будут начинаться с использованием второе развертывание, после их извлечения и обновления их объекты групповой Политики клиента.  
  
4.  Дополнительно Удалите точек входа DirectAccess из первого развертывания и добавить эти серверы в качестве новой точки входа за секунду.  
  
После завершения перехода, можно удалить первое развертывание DirectAccess. При удалении, могут возникнуть следующие проблемы:  
  
-   Если развертывание был настроен для поддержки клиентов только на мобильных компьютерах, будут удалены фильтра WMI. Если клиент безопасности группы второе развертывание настольных компьютеров, GPO клиента DirectAccess не будет отфильтровывать настольных компьютеров и может вызвать проблемы на них. Если требуется фильтр мобильных компьютеров, создайте ее, следуя инструкциям [создать фильтры WMI для объекта групповой Политики](https://technet.microsoft.com/library/cc947846.aspx).  
  
-   Если оба развертывания были созданы в одном домене Active Directory, DNS проверки записи, которое указывает на localhost, будут удалены и может вызвать клиент, проблемы с подключением. Например клиенты могут подключаться с помощью IP-HTTPS, а не Teredo или переключение между точек входа DirectAccess. В этом случае необходимо добавить следующую запись DNS корпоративную службу DNS:  
  
    -   Зоны: имя домена  
  
    -   Имя: directaccess-corpConnectivityHost  
  
    -   IP-адрес::: 1  
  
    -   Введите команду: AAAA
