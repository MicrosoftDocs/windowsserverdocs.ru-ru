# <a name="step-3-plan-a-load-balanced-cluster-deployment"></a>Шаг 3 Планирование развертывания в кластере с балансировкой нагрузки

>Область применения. Windows Server (полугодовой канал), Windows Server 2016

Следующим шагом является планирование конфигурации балансировки нагрузки и развертывания кластера.  
  
|Задача|Описание|  
|----|--------|  
|3.1 планирование балансировки нагрузки|Решите, следует ли использовать Windows сетевой нагрузки балансировки (NLB), или внешний балансировщик нагрузки (ELB).|  
|3.2 план IP-HTTPS|Если не используется самозаверяющий сертификат, сервер удаленного доступа требуется SSL-сертификат на каждом сервере в кластере, для проверки подлинности подключений IP-HTTPS.|  
|3.3 Планирование клиентские VPN-подключения|Обратите внимание на требования для клиентских подключений VPN.|  
|3.4. Планирование сервера сетевых расположений|Если веб-сайт сети расположение сервера размещается на сервере удаленного доступа, а не используется самозаверяющий сертификат, убедитесь, что каждый сервер в кластере имеет сертификат сервера для проверки подлинности подключения на веб-сайт.|  
  
## <a name="bkmk_2_1_Plan_LB"></a>3.1 планирование балансировки нагрузки  
Удаленного доступа можно развернуть на одном сервере или в кластере серверов удаленного доступа. Трафик кластера может быть сбалансирована для обеспечения высокого уровня доступности и масштабируемости для клиентов DirectAccess. Существует два варианта балансировки нагрузки:  
  
-   **Балансировка сетевой Нагрузки Windows**-балансировки сетевой Нагрузки Windows — это компонент Windows server. Для его использования не требуется дополнительное оборудование, так как все серверы в кластере отвечают за управление нагрузки трафика. Балансировка сетевой Нагрузки Windows поддерживает не более восьми серверов кластера удаленного доступа.  
  
-   **Внешняя Подсистема балансировки нагрузки**-с помощью подсистемы балансировки нагрузки требуется внешний оборудование для управления нагрузки трафика между серверами кластера удаленного доступа. Кроме того с помощью подсистемы балансировки нагрузки поддерживает не более 32 серверов удаленного доступа в кластере. Ниже приведены некоторые моменты, которые следует помнить при настройке балансировки нагрузки.  
  
    -   Администратору необходимо убедиться, что виртуальные IP-адреса, настроенные с помощью мастера балансировки нагрузки удаленного доступа используются на внешние подсистемы балансировки нагрузки (например, системы F5 Big-Ip Local Traffic Manager). Если включена внешняя Балансировка нагрузки, IP-адреса внешних и внутренних интерфейсов будет повышен до виртуальные IP-адреса и нужно действуют в подсистеме балансировки нагрузки. Это делается, чтобы администратор изменять записи DNS для общедоступного имени развертывания кластера не требуется. Кроме того конечные точки туннеля IPsec являются производными от IP-адресами серверов. Если администратор предоставляет отдельные виртуальные IP-адреса, клиент не сможете подключиться к серверу. См. пример для настройки DirectAccess с внешней балансировкой нагрузки в 3.1.1 пример конфигурации внешней подсистемы балансировки нагрузки.  
  
    -   Многие внешние подсистемы балансировки нагрузки (включая F5) не поддерживают 6to4 и ISATAP балансировки нагрузки. Если сервер удаленного доступа является маршрутизатора ISATAP, функция ISATAP должны быть перенесены на другой компьютер. Кроме того Если функция ISATAP на другом компьютере, серверы DirectAccess должны иметь собственное подключение IPv6 с маршрутизатором ISATAP. Обратите внимание, что это подключение должно присутствовать перед настройкой DirectAccess.  
  
    -   Для балансировки нагрузки, если Teredo для использования, все серверы удаленного доступа необходимо иметь два последовательных открытых адреса IPv4 как выделенный IP-адреса. Виртуальные IP-адреса кластера также должен иметь два последовательных открытых адреса IPv4. Это не так для балансировки сетевой Нагрузки Windows, когда только виртуальные IP-адреса кластера должен иметь два последовательных открытых адреса IPv4. Если не используется Teredo, два последовательных IP-адреса не являются обязательными.  
  
    -   Администратор может перейти с балансировки сетевой Нагрузки Windows внешней подсистемы балансировки нагрузки и наоборот. Обратите внимание на то, что администратор не допускается переход из внешней подсистемы балансировки нагрузки для балансировки сетевой Нагрузки Windows, если он обладает более чем 8 серверов в развертывание подсистемы балансировки нагрузки.  
  
### <a name="ELBConfigEx"></a>3.1.1 пример конфигурации внешней подсистемы балансировки нагрузки  
В этом разделе описываются шаги настройки для включения подсистемы балансировки нагрузки на новое развертывание удаленного доступа. При использовании подсистемы балансировки нагрузки, кластера удаленного доступа может выглядеть как на рисунке ниже, где подключения серверов удаленного доступа к корпоративной сети через подсистему балансировки нагрузки во внутренней сети и к Интернету через балансировщик нагрузки подключено к внешней сети:

##### <a name="planning-information"></a>Информация о планировании  
  
1.  Внешние виртуальные IP-адреса (IP-адреса, клиент будет использовать для подключения к удаленному доступу) были решил поискать 131.107.0.102, 131.107.0.103  
  
2.  Подсистемы балансировки нагрузки для внешней сети self-IP-адреса — 131.107.0.245 (Интернет), 131.107.1.245  
  
    Сети периметра (также известный как демилитаризованная зона DMZ) — между балансировщиком нагрузки во внешней сети и сервера удаленного доступа.  
  
3.  IP-адреса для сервера удаленного доступа в сети периметра - 131.107.1.102, 131.107.1.103  
  
4.  IP-адреса для сервера удаленного доступа в сети ELB (т. е. между сервером удаленного доступа и подсистемы балансировки нагрузки во внутренней сети) - 30.11.1.101 2006:2005:11:1::101  
  
5.  Подсистемы балансировки нагрузки для внутренней сети self-IP-адреса — 30.11.1.245 2006:2005:11:1::245 (ELB), 30.1.1.245 2006:2005:1:1::245 (Corpnet)  
  
6.  Было решено внутренних виртуальных IP-адресов (и IP-адреса для веб-клиента-пробы для сервера сетевых расположений, если установлены на серверах удаленного доступа), быть 30.1.1.10 2006:2005:1:1::10  
  
##### <a name="steps"></a>Действия  
  
1.  Настройка внешнего сетевого адаптера сервера удаленного доступа (который подключен к сети периметра) с адресами 131.107.0.102, 131.107.0.103. Этот шаг является обязательным для настройки DirectAccess для обнаружения нужных конечных точек туннеля IPsec.  
  
2.  Настройка сервера удаленного доступа внутренний сетевой адаптер (который подключен к сети ELB) с IP-адреса сервера веб пробы и сетевого расположения (30.1.1.10 2006:2005:1:1::10). Этот шаг является обязательным, позволяющие клиентам получать доступ к IP-адрес веб-пробы, чтобы правильно показывает состояние подключения к DirectAccess, помощник по подключению к сети. Этот шаг также предоставляет доступ к серверу сетевых расположений, если он настроен на сервере DirectAccess.  
  
    > [!NOTE]  
    > Убедитесь, что контроллер домена доступен с сервера удаленного доступа с этой конфигурацией.  
  
3.  Настройте сервер DirectAccess на сервере удаленного доступа.  
  
4.  Включение внешней балансировки нагрузки в конфигурации DirectAccess. Использовать в качестве внешних выделенный IP-адрес (DIP) 131.107.1.102 (131.107.1.103 будет выбран автоматически), используйте 30.11.1.101 2006:2005:11:1::101 как внутренняя частные интерфейсы.  
  
5.  Настройте внешний виртуальный IP-адресов (VIP) внешней подсистемы балансировки нагрузки с адресами 131.107.0.102 и 131.107.0.103. Кроме того настройте внутренних виртуальных IP-адресов в подсистеме балансировки нагрузки с адресами 30.1.1.10 и 2006:2005:1:1::10.  
  
6.  Сервер удаленного доступа, теперь можно настроить Плановое IP-адреса и внешних и внутренних IP-адреса для кластера будут настроены в соответствии с плановой IP-адреса.  
  
## <a name="bkmk_2_2_NLB"></a>3.2 план IP-HTTPS  
  
1.  **Требований к сертификатам**-во время развертывания одиночного сервера удаленного доступа, вы решили использовать сертификат IP-HTTPS, выданный центром сертификации public или internal (ЦС), или самозаверяющий сертификат. Для развертывания кластера необходимо использовать одинаковый тип сертификата на каждом члене кластера удаленного доступа. То есть если используется сертификат, выданный общедоступным центром сертификации (рекомендуется), необходимо установить сертификат, выданный общедоступным центром сертификации на каждом члене кластера. Имя субъекта сертификата должно совпадать имя субъекта сертификата IP-HTTPS, используемых в развертывании. Обратите внимание, что если вы используете самозаверяющие сертификаты они будут настроены автоматически на каждом сервере во время развертывания кластера.  
  
2.  **Префикс требования**-удаленного доступа обеспечивает балансировку нагрузки трафика на основе SSL и трафика DirectAccess. Для балансировки нагрузки все IPv6 на основе трафика DirectAccess, удаленный доступ для изучения туннелирование IPv4 для всех технологий перехода. Так как IP-HTTPS-трафик шифруется, проверка содержимого туннеля IPv4 не поддерживается. Чтобы включить IP-HTTPS-трафика для балансировки нагрузки, необходимо выделить достаточно широким, префикс IPv6, чтобы другой префикс IPv6 /64 можно назначить для каждого члена кластера. Можно настроить более 32 серверов в кластере с балансировкой нагрузки; Таким образом, необходимо указать /59 префикс. Этот префикс должен поддерживать маршрутизацию к внутренний IPv6-адрес кластера удаленного доступа и настройки в мастере установки сервера удаленного доступа.  
  
    > [!NOTE]  
    > Префикс требований, относящихся только во внутренней сети протокол IPv6 (только для IPv6 или IPV4 + IPv6). В IPv4-только корпоративной сети префикс для клиента настраивается автоматически, и администратор не может изменить его.  
  
## <a name="BKMK_3.3"></a>3.3 Планирование клиентские VPN-подключения  
Существует ряд аспектов для клиентских подключений VPN:  
  
-   Трафик клиента VPN не может быть сбалансирован Если выделяются адреса клиента VPN с помощью DHCP. Пул статических адресов является обязательным.  
  
-   RRAS можно включить в кластере с балансировкой нагрузки, развернутую для DirectAccess только с помощью **включить VPN** в области задачи консоли управления удаленным доступом.  
  
-   Любые изменения VPN, заполнять консоль маршрутизации и удаленного доступа (rrasmgmt.msc) будет реплицироваться на всех серверах удаленного доступа в кластере вручную.  
  
-   Чтобы включить трафик клиента VPN IPv6 для балансировки нагрузки, необходимо указать префикс IPv6 59 бит.  
  
## <a name="BKMK_nls"></a>3.4. Планирование сервера сетевых расположений  
Если вы используете веб-сайт сети расположение сервера на сервер удаленного доступа, во время развертывания вы выбрали для использования сертификата, выданного внутренним центром сертификации (ЦС), или самозаверяющий сертификат.  Обратите внимание на следующее условия:  
  
1.  Каждый член кластера удаленного доступа должен иметь сертификат для сервера сетевых расположений, соответствующий записи DNS для веб-сайт сети расположение сервера.  
  
2.  Таким же образом, что и сертификат на один удаленный доступ сервера текущего сертификата сервера сетевых расположений должен быть выдан сертификат для каждого сервера кластера. Например если используется сертификат, выданный внутренним ЦС, необходимо установить сертификат, выданный внутреннего центра сертификации на каждом члене кластера.  
  
3.  Если используется самозаверяющий сертификат, самозаверяющий сертификат будет настроен автоматически для каждого сервера во время развертывания кластера.  
  
4.  Имя субъекта сертификата не должны быть идентичны имени любого сервера в развертывании удаленного доступа.
