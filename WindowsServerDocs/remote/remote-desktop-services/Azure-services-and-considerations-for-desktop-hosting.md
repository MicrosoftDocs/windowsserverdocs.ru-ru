# <a name="azure-services-and-considerations-for-desktop-hosting"></a>Службы и рекомендации Azure для размещения рабочих столов

>Относится к: Windows Server (полугодовой канал), Windows Server 2016

В следующих разделах описаны службы инфраструктуры Azure.
  
## <a name="azure-portal"></a>Портал Azure

После того как поставщик создаст подписки Azure, портала Azure можно вручную создать среду для каждого клиента. Этот процесс можно автоматизировать с помощью скриптов PowerShell.  

Дополнительные сведения см. в статье [Microsoft Azure](https://www.azure.microsoft.com) веб-сайта.
  
## <a name="azure-load-balancer"></a>Azure Load Balancer

Компоненты клиента запустить на виртуальных машинах, которые взаимодействуют друг с другом в изолированной сети. Во время развертывания извне доступны эти виртуальные машины через подсистему балансировки нагрузки Azure с помощью конечных точек протокола удаленного рабочего стола или конечную точку удаленного использования PowerShell. После завершения развертывания этих конечных точек обычно удаляется чтобы уменьшить контактную зону для атак. Только конечные точки будут конечные точки HTTPS и UDP, созданный для виртуальной машины, работающей в Интернете удаленных рабочих Столов и шлюза удаленных рабочих Столов компонентов. Это позволяет клиентам в Интернете для подключения к сеансов, запущенных в служба размещения рабочих столов клиента. Если пользователь откроет приложение, которое подключается к Интернету, таких как веб-браузер, соединения будут переданы через подсистему балансировки нагрузки Azure.  
  
Дополнительные сведения см. в разделе [что такое Подсистема балансировки нагрузки Azure?](https://azure.microsoft.com/documentation/articles/virtual-machines-linux-load-balance/)
  
## <a name="security-considerations"></a>Замечания по безопасности

Этот рабочий стол размещения ссылки руководства по архитектуре Azure предназначен для предоставления среде с высоким уровнем безопасности и изолированной для каждого клиента. Безопасность системы также зависит от меры, предпринятые поставщиком во время развертывания и эксплуатации размещенной службы. Ниже перечислены некоторые соображения, которые должен выполнить поставщик для сохранения их рабочего стола размещения решение, основанное на схеме этой эталонной архитектуры безопасности.

- Все административные пароли должны быть строгим, в идеале случайным образом создается, часто изменяются и сохраняется в безопасном центральном местоположении, доступна только администраторам выберите несколько поставщика.  
- При репликации среды клиента для новых клиентов, избегайте, используя те же или простой административные пароли.
- URL-адрес сайта веб-доступа к удаленным рабочим Столам, имя и сертификатов должен быть уникальным и характерный для каждого клиента, для предотвращения атак путем подделки.  
- Во время нормальной работы служба размещения рабочих столов следует удалить все общедоступные IP-адреса для всех виртуальных машин, за исключением веб-удаленных рабочих Столов и шлюза удаленных рабочих Столов виртуальной машины, которая позволяет пользователям безопасно подключаться к клиента рабочего стола размещения облачной службы. Общедоступные IP-адреса могут быть временно добавлены, когда это необходимо для выполнения задач управления, но они всегда должна удаляться сразу.

## <a name="design-considerations"></a>Вопросы разработки

Это важно при разработке многопользовательских рабочего стола службы необходимо учитывать ограничения служб инфраструктуры Microsoft Azure. В следующем списке приведены соображения, поставщик должен выполнить для достижения функционального и экономичного рабочего стола размещения решения, основанного на схеме этой эталонной архитектуры.  
  
- Подписка Azure имеет максимальное количество виртуальных сетей, ядра виртуальных Машин и облачных служб, который может использоваться. Если поставщик требует больше ресурсов, чем это, может потребоваться создать несколько подписок.
- В облачной службе Azure имеет максимальное количество виртуальных машин, которые могут использоваться. Поставщик может потребоваться создать несколько облачных служб для больших клиентов, превышает максимальное.  
- Затраты на развертывание Azure частично основаны на количество и размер виртуальных машин. Поставщик следует оптимизировать число и размер виртуальных машин для каждого клиента предоставляют функциональное и безопасной среды размещения рабочих столов по минимальной цене.  
- Ресурсы физического компьютера в центре обработки данных Azure являются виртуальными с помощью Hyper-V. Узлы Hyper-V не настроены в кластерах узлов, поэтому доступность виртуальных машин зависит от доступности отдельных серверов, используемых в инфраструктуре Azure. Чтобы обеспечить более высокий уровень доступности, несколько экземпляров каждой роли службы виртуальной машины могут создаваться в группе доступности, то кластеризация гостевых систем можно реализовать в виртуальных машинах.  
- В конфигурации типичный хранилища поставщик услуг будет иметь одной учетной записью с несколько контейнеров (например, по одному для каждого клиента) и несколько дисков в каждом контейнере. Однако есть ограничение на общий объем хранилища и производительности, которого можно достигнуть для одной учетной записью. Для поставщиков услуг, которые поддерживают большое количество клиентов или клиентов с требованиями к емкости и производительности, большой емкости хранилища поставщик услуг может потребоваться создать несколько учетных записей хранения.

## <a name="azure-active-directory-application-proxy"></a>Прокси приложения Azure Active Directory

Прокси приложения Azure Active Directory (AD) — это служба, предоставляемая в платных номеров SKU Azure AD, позволяют пользователям подключаться к внутренним приложениям через собственные обратного прокси-сервера служба Azure. Это позволяет конечные точки веб-удаленных рабочих Столов и шлюза удаленных рабочих Столов будут скрыты внутри виртуальной сети, что устраняет необходимость быть подключены к Интернету, общедоступный IP-адрес. Уменьшать количество виртуальных машин в среде клиента, сохраняя при этом полное развертывание, поставщиков услуг размещения можно использовать прокси приложения Azure AD. Прокси приложения Azure AD также включает многие из преимуществ, предоставляемых Azure AD, таких как условный доступ и многофакторную проверку подлинности.

Дополнительные сведения см. в разделе [приступить к работе с прокси приложения и установка соединителя](https://docs.microsoft.com/azure/active-directory/manage-apps/application-proxy-enable).
