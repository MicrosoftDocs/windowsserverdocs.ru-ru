# <a name="remote-desktop-services-architecture"></a>Архитектура служб удаленных рабочих столов

>Область применения. Windows Server (полугодовой канал), Windows Server 2016

Ниже приведены различные конфигурации для развертывания служб удаленных рабочих столов для размещения приложений Windows и настольных компьютеров для конечных пользователей.

>[!NOTE]
> Архитектура рисунках ниже показано использование служб удаленных рабочих СТОЛОВ в Azure. Тем не менее, можно развернуть службы удаленных рабочих столов в локальной и в других облаках. Эти схемы первую очередь предназначены для демонстрации как совместного размещения и использования других служб ролей служб удаленных рабочих СТОЛОВ.

## <a name="standard-rds-deployment-architectures"></a>Стандартные архитектуры развертывания служб удаленных рабочих СТОЛОВ

Службы удаленных рабочих столов состоит из двух стандартных архитектурах.
-   Базовое развертывание — это содержит минимальное количество серверов для создания эффективной среды служб удаленных рабочих СТОЛОВ
-   Высокодоступное развертывание — этот файл содержит все необходимые компоненты для имеют наивысший гарантирована времени для среды служб удаленных рабочих СТОЛОВ

## <a name="rds-architectures-with-unique-azure-paas-roles"></a>Архитектуры служб удаленных рабочих СТОЛОВ с уникальным ролей PaaS в Azure

Хотя стандартные архитектуры развертывания служб удаленных рабочих СТОЛОВ помещается большинство сценариев, Azure продолжает развивать в основном решения PaaS, приносит клиентам пользу. Ниже приведены некоторые архитектуры, показывающий, как они могут использовать с RDS.

### <a name="rds-deployment-with-azure-ad-domain-services"></a>Развертывание служб удаленных рабочих СТОЛОВ с помощью доменных служб Azure AD

Два стандартных архитектура диаграммах, показанных выше основаны на традиционной Active Directory (AD) развернуты на виртуальную Машину Windows Server. Тем не менее если у вас нет традиционных AD и только у вас есть клиент Azure AD — через службы, такие как Office 365 — но по-прежнему нужно использовать служб удаленных рабочих СТОЛОВ, можно использовать [доменных служб Azure AD](https://docs.microsoft.com/azure/active-directory-domain-services/active-directory-ds-overview) для создания полностью управляемого домена в Azure IaaS. Среда, которая использует тех же пользователей, которые существуют в вашем клиенте Azure AD. Это избавляет от сложностей вручную синхронизацию пользователей и управления ими большего числа виртуальных машин. Доменные службы Azure AD может работать в любом развертывании: основные или высокой доступности.

![Azure AD и развертывания служб удаленных рабочих СТОЛОВ](./media/aadds-rds.png)

### <a name="rds-deployment-with-azure-ad-application-proxy"></a>Развертывание служб удаленных рабочих СТОЛОВ с помощью прокси приложения Azure AD

Два стандартных архитектуры в вышеприведенных диаграммах использования Web/шлюза удаленных рабочих Столов серверов как Интернет-точку входа в систему служб удаленных рабочих СТОЛОВ. В некоторых средах администраторы предпочли бы удалить свои собственные серверы периметра и вместо этого использовать технологии, которые также обеспечивают дополнительную безопасность за счет использования технологий обратного прокси-сервера. [Прокси приложения Azure AD](https://docs.microsoft.com/azure/active-directory/active-directory-application-proxy-get-started) роль PaaS хорошо вписывается в этом сценарии.

Поддерживаемые конфигурации и способы создания этой установки, см. в разделе Практическое [публикация удаленного рабочего стола с помощью прокси приложения Azure AD](/azure/active-directory/application-proxy-publish-remote-desktop).

![RDS с прокси приложения Azure AD](./media/aadappproxy-rds.png)
