# <a name="use-dns-policy-for-split-brain-dns-in-active-directory"></a>Использование политики DNS для Split-Brain DNS-сервер в Active Directory

>Область применения: Windows Server (канал точками годовой), Windows Server 2016

В этом разделе можно использовать для эффективного использования возможности управления трафика DNS политик для развертывания вычислительных мощностей split\ с Active Directory интегрированного зоны в Windows Server 2016.

В Windows Server 2016 расширена поддержка политик DNS в Active Directory интегрированы зон DNS. Интеграция с Active Directory предоставляет возможности высокая доступность хозяина используется DNS-сервер. 

Ранее этот сценарий требуется, что администраторы DNS поддерживать два различных DNS-сервера, каждой обслуживает пользователей, внутренние и внешние. Если имеется только несколько записей в зоне были split\ brained или обоих экземпляров зоны (внутренний и внешний) были делегированы же родительский домен, это стало conundrum управления.

>[!NOTE]
> - DNS-сервера развертывания — вычислительных мощностей split\, при наличии двух версий одной зоне, один для внутренних пользователей в интрасети организации и версии для внешних пользователей —, которые, как правило, пользователям в Интернете.
> - Раздел [использование политики DNS для Split-Brain развертывания DNS](split-brain-DNS-deployment.md) объясняется, как использовать политик DNS-сервера и области зоны для развертывания системы вычислительных мощностей split\ DNS-сервера на одном сервере DNS-сервера Windows Server 2016.



##  <a name="example-split-brain-dns-in-active-directory"></a>Пример вычислительных мощностей Split\ DNS в Active Directory

В этом примере используется один вымышленной компании Contoso, которая поддерживает веб-сайта на www.career.contoso.com.

Сайт имеет две версии, один для внутренних пользователей, где доступны учет внутренней работ. Внутренний сайт доступен в локальный IP-адрес 10.0.0.39. 

Вторая версия является открытой версией том же сайте, который доступен на странице общедоступный IP-адрес 65.55.39.10.

В отсутствие политики DNS администратору необходимо разместить эти две зоны на отдельных серверах Windows Server DNS и управлять ими отдельно. 

С помощью политики DNS этих зон может теперь размещаться на том же сервере DNS-сервера.

Если DNS-сервер для contoso.com интегрировано с Active Directory и выполняет прослушивание на двух сетевых интерфейсов, администратор DNS Contoso выполнить действия, описанные в этом разделе, чтобы добиться развертывания вычислительных мощностей split\.

Администратор DNS настраивает следующие IP-адреса интерфейсов сервера DNS.

- Общий IP-адрес 208.84.0.53 для внешних запросов оснащен сетевого адаптера сети Интернет.
- Частный IP-адрес 10.0.0.56 внутренних запросов оснащен интрасети сетевом адаптере.

## <a name="how-dns-policy-for-split-brain-dns-in-active-directory-works"></a>Как работает политика DNS-сервера для вычислительных мощностей Split\ DNS-сервер в Active Directory

Когда DNS-сервер настроен необходимые политики DNS, каждый запрос на разрешение имен оценивается для политик на DNS-сервере.

Сервер интерфейс используется в этом примере в качестве условия для различения внутренних и внешних клиентов.

Если интерфейс сервера, на котором выдается запрос соответствует какой-либо политики, область связанные зоны используется для ответа на запрос. 

Таким образом в нашем примере для www.career.contoso.com DNS-запросов, полученных частный IP-адресов (10.0.0.56) получать ответ DNS, который содержит внутренний IP-адрес; и запросы DNS, которые поступают на общем сетевом интерфейсе получать ответ DNS, который содержит общий IP-адрес в области зоны по умолчанию (это так же, как обычный запрос разрешения).  

Поддержка обновления \(DDNS\) динамического DNS и уборку поддерживается только на область зоны по умолчанию. Из-за внутренних клиентов обслуживаемых области по умолчанию зона, Администраторы DNS Contoso можно продолжать использовать существующие механизмы (динамического DNS или статического) для обновления записей в contoso.com. Для областей, отличных от заданных по умолчанию зона \ (например, внешних область в этом example\), DDNS или очистки поддержки недоступна.

### <a name="high-availability-of-policies"></a>Высокая доступность политик

Политики DNS, не являются интегрировано с Active Directory. По этой причине политики DNS, не реплицируются на другие DNS-серверы, которых размещается же интегрированной зоне Active Directory. 

Политики DNS, хранятся на локальном DNS-сервере. Легко можно экспортировать политики DNS с одного сервера на другой с помощью команды Windows PowerShell в следующем примере.

    $policies = Get-DnsServerQueryResolutionPolicy -ZoneName "contoso.com" -ComputerName Server01
    
    $policies |  Add-DnsServerQueryResolutionPolicy -ZoneName "contoso.com" -ComputerName Server02

Дополнительные сведения см. в следующих разделах Справочник по Windows PowerShell.

- [Get-DnsServerQueryResolutionPolicy](https://technet.microsoft.com/itpro/powershell/windows/dns-server/get-dnsserverqueryresolutionpolicy)
- [Add-DnsServerQueryResolutionPolicy](https://technet.microsoft.com/itpro/powershell/windows/dns-server/add-dnsserverqueryresolutionpolicy)


## <a name="how-to-configure-dns-policy-for-split-brain-dns-in-active-directory"></a>Настройка политики DNS для вычислительных мощностей Split\ DNS-сервер в Active Directory

Настройка DNS Split-Brain развертывания с помощью политики DNS, необходимо использовать следующие разделы, в которых предоставления подробных инструкций по настройке.

### <a name="add-the-active-directory-integrated-zone"></a>Добавление зоны, интегрированной в Active Directory

Следующий пример команды можно использовать для добавления contoso.com интегрированной зоне Active Directory на DNS-сервер.

    Add-DnsServerPrimaryZone -Name "contoso.com" -ReplicationScope "Domain" -PassThru

Дополнительные сведения см. в разделе [Add-DnsServerPrimaryZone](https://technet.microsoft.com/library/jj649876.aspx).

### <a name="create-the-scopes-of-the-zone"></a>Создание областей зоны

В этом разделе можно использовать для создания разделов зоны contoso.com Создание областью внешних зоны.

Область зоны представляет уникальный экземпляр зоны. Зона DNS может иметь несколько областей зоны, с каждой областью зону, содержащую собственный набор DNS-записей. Ту же запись может быть представлен в нескольких областях, различные IP-адреса или одинаковые IP-адреса. 

Так как этой новой области зоны добавляются в интегрированной зоне Active Directory, область зоны и записи в него будет реплицировать через Active Directory на другие серверы-реплики в домене.

По умолчанию область зоны существует в каждой зоне DNS. Эта область зоны имеет то же имя, что зона и устаревших операции DNS-сервера работают в этой области. Эта область зоны по умолчанию и будет размещаться внутренней версии www.career.contoso.com.

Следующий пример команды можно использовать для создания области зоны на DNS-сервере.

    Add-DnsServerZoneScope -ZoneName "contoso.com" -Name "external"

Дополнительные сведения см. в разделе [добавить DnsServerZoneScope](https://technet.microsoft.com/library/mt126267.aspx).

### <a name="add-records-to-the-zone-scopes"></a>Добавить записи в области зоны

Следующим шагом является добавление записей, представляющий веб-сервера в два зоны внешние областей и по умолчанию \(for internal clients\). 

По умолчанию внутренней зоны, в области добавления записи www.career.contoso.com с IP-адресом 10.0.0.39, который является частный IP-адрес; и внешних зоны, в области же записи \(www.career.contoso.com\) добавляются с общедоступный IP-адрес 65.55.39.10. 

Записи \ (как в области внутренней зоны по умолчанию и внешних зоны scope\) автоматически выполнит репликацию данных между доменами с помощью областей их соответствующие зоны.

Следующий пример команды можно использовать для добавления записей области зоны на DNS-сервер.

    Add-DnsServerResourceRecord -ZoneName "contoso.com" -A -Name "www.career" -IPv4Address "65.55.39.10" -ZoneScope "external"
    
    Add-DnsServerResourceRecord -ZoneName "contoso.com" -A -Name "www.career" -IPv4Address "10.0.0.39”

>[!NOTE]
>**— Зонаобласть** параметр не включен, при добавлении записи зоны области по умолчанию. Это действие выполняется такой же, как добавить записи для обычных зоны.

Дополнительные сведения см. в разделе [добавить DnsServerResourceRecord](https://technet.microsoft.com/library/jj649925.aspx).

### <a name="create-the-dns-policies"></a>Создание политик DNS

После того как вы определили интерфейсов сервера для внешней сети и внутренней сети и созданный области зоны, необходимо создать политики DNS, которые связывают области внутренних и внешних зоны.

>[!NOTE]
>В этом примере используется интерфейс сервера \ (параметр - ServerInterface below\ команда примере) в качестве условия для различения внутренних и внешних клиентов. Другой способ различать внешних и внутренних клиентов — с помощью как критерий для подсети клиента. Если можно определить подсети, к которым принадлежат внутренних клиентов, можно настроить политики DNS для различения на основе подсети клиента. Сведения о том, как настроить трафик управления, с помощью критериев подсети клиента см. в разделе [использование политики DNS для управления трафиком на основе географического расположения, основных серверах с](primary-geo-location.md).

После настройки политики, при получении запроса DNS на общедоступный интерфейс, ответ возвращается из внешних области зоны. 

>[!NOTE]
>Политики не требуются для сопоставления области внутренней зоны по умолчанию. 

    Add-DnsServerQueryResolutionPolicy -Name "SplitBrainZonePolicy" -Action ALLOW -ServerInterface "eq,208.84.0.53" -ZoneScope "external,1" -ZoneName contoso.com

>[!NOTE]
>208.84.0.53 — IP-адрес, на общий сетевой интерфейс.

Дополнительные сведения см. в разделе [добавить DnsServerQueryResolutionPolicy](https://technet.microsoft.com/library/mt126273.aspx).

Теперь DNS-сервера настроены необходимые политики DNS для сервера разделенной имен с помощью Active Directory интегрированного зоны.

Можно создать тысячи политик DNS в соответствии с трафик требования к управлению, и все новые политики применяются динамически — без перезагрузки DNS-сервер — на входящие запросы.
