# <a name="whats-new-in-dns-server-in-windows-server"></a>Новые возможности DNS-сервера в Windows Server

>Относится к: Windows Server (полугодовой канал), Windows Server 2016

В этом разделе описываются возможности сервера доменных имен (DNS), которые являются новыми или измененными в Windows Server 2016.  
  
В Windows Server 2016 DNS-сервер обеспечивает расширенную поддержку в следующих областях.  
  
|Функция|Новый или улучшенный|Описание|  
|-----------------|-------------------|---------------|  
|Политики DNS|Оператор new|Можно настроить политики DNS, чтобы указать, как DNS-сервер отвечает на запросы DNS. Ответы DNS, которые могут быть основаны на IP-адрес клиента (расположение), время дня и несколько других параметров. Политики DNS включить DNS, привязанный к местонахождению, управление трафиком, балансировка нагрузки, разделенная система DNS и других сценариев.|  
|Скорость ответа, ограничивая (RRL)|Оператор new|Вы можете включить ограничение скорости ответа на DNS-серверы. Таким образом, вы защититься от вредоносных систем, с помощью DNS-серверы для инициирования атаки отказа в обслуживании на DNS-клиента.|  
|Проверка подлинности на основе DNS именованных сущностей (DANE)|Оператор new|Записи TLSA (проверка подлинности безопасности уровня транспорта) можно использовать для предоставления информации о DNS-клиенты, сообщающие некий какие ЦС, они должны ожидать сертификат для доменного имени. Это предотвращает атаки man-in--middle где кто-то поврежденный кэш DNS для указания собственных веб-сайтов и укажите сертификат, который они выданных разные ЦС.|  
|Поддержка неизвестных записи|Оператор new|Вы можете добавить записи, которые явно не поддерживаются Windows DNS-сервером, используя функцию неизвестной записи.|  
|IPv6 корневых ссылок|Оператор new|Можно использовать собственный протокол IPV6, поддерживают корневые ссылки для разрешения имен Интернета, с помощью корневые серверы IPV6.|  
|Поддержка Windows PowerShell|Улучшенная|Новые командлеты Windows PowerShell доступны для DNS-сервера.|  
  
## <a name="dns-policies"></a>Политики DNS

Можно использовать политику DNS для географического расположения на основе управление трафиком, интеллектуальных ответов DNS, на основе времени дня, чтобы управлять один DNS-сервер, настроенный для разбиения\-развертывания вычислительных мощностей, применения фильтров к DNS-запросы и многое другое. Ниже приведены дополнительные сведения об этих возможностях.

-   **Балансировка нагрузки приложения.** Если вы развернули несколько экземпляров приложения в разных местах, можно использовать политики DNS для балансировки нагрузки трафика между экземплярами приложений с различными, динамического выделения нагрузки трафика для приложения.

-   **Географическая\-Location Based управление трафиком.** Можно использовать политики DNS позволяет основного и дополнительного DNS-серверы реагировать на запросы DNS-клиента на основе географического расположения клиента и ресурса, к которому клиент пытается подключиться, предоставляя клиент с IP-адрес ближайшего ресурс. 

-   **Разделение вычислительных мощностей DNS.** С разделением\-мозг DNS, записи DNS были разбиты на различные области зоны на одном сервере DNS и DNS-клиентам получить ответ, основанного на клиенты внутренних и внешних клиентов. Можно настроить разделение\-мозг DNS для зон, интегрированных в Active Directory или для зон на автономных серверах DNS.

-   **Фильтрация.** Можно настроить политики DNS для создания фильтров запросов, основанных на введенных признаков. Фильтры запросов в политике DNS позволяют настроить DNS-сервер отвечает специальным образом на основе запроса DNS и DNS-клиента, который отправляет запрос DNS. 
-   **Расследований.** Можно использовать политики DNS для перенаправления вредоносные клиенты DNS не\-существующий IP-адрес вместо направления их на компьютер, они удается связаться.

-   **Время суток, на основе перенаправления.** Политики DNS можно использовать для распределения трафика приложения для различных экземпляров географически распределенного приложения с помощью политики DNS, которые основаны на время суток. 
  
Можно также использовать политики DNS для интеграции с Active Directory DNS зоны.

Дополнительные сведения см. в разделе [руководство по сценарию политики DNS](deploy/DNS-Policies-Overview.md).

## <a name="response-rate-limiting"></a>Ограничение скорости ответа

Можно настроить параметры RRL для управления как отвечать на запросы DNS-клиента, когда сервер получает несколько запросов, предназначенных для одного клиента. Таким образом, можно запретить пользователю отправки атаки Отказ в обслуживании (Dos), с помощью DNS-серверы. Например программы-робота, net может отправлять запросы на DNS-сервер, с помощью IP-адрес третий компьютер с указанным в запросе. Без RRL DNS-серверов может отвечать на все запросы, перегрузки на третьем компьютере. При использовании RRL, можно настроить следующие параметры:  
  
-   **Filetable в секунду**. Это максимальное количество раз, которое получит одинаковый ответ на клиенте в течение одной секунды.  
  
-   **Ошибок в секунду**. Это максимальное количество раз, когда сообщение об ошибке будет отправляться на одном клиенте в течение одной секунды.  
  
-   **Окно**. Это количество секунд, для которых будет приостановлен ответов для клиента, если слишком много запросов.  
  
-   **Скорость утечки**. Это, как часто DNS-сервер ответит на запрос во время приостановки ответов. Например если сервер приостанавливает ответов для клиента на 10 секунд, а скорость утечки равно 5, сервер по-прежнему будет отвечать на один запрос для каждые 5 запросов, отправленных. Это позволяет клиентам, допустимые для получения ответов, даже в том случае, если DNS-сервер применяет ответивших, ограничивая их подсети или полное доменное имя.  
  
-   **TC скорость**. Это позволяет сообщить клиенту, чтобы попытаться подключиться с помощью TCP Приостановка ответа для клиента. К примеру Если скорость TC — 3, а сервер приостанавливает ответов на заданном клиенте, сервер будет выдавать запрос для TCP-подключение для каждые 3 запросы, полученные. Убедитесь, что значение для скорости TC меньше, чем скорость утечки для предоставления клиенту возможность подключения по протоколу TCP перед утечка ответов.  
  
-   **Максимальное ответы**. Это максимальное количество ответов, сервер будет выдавать клиенту приостановленная ответов.  
  
-   **Домены в список разрешенных**. Это список доменов должны быть исключены из RRL параметры.  
  
-   **Утверженный список подсетей**. Это список подсетей должны быть исключены из RRL параметры.  
  
-   **Список разрешений сервера интерфейсы**. Это список интерфейсов сервера DNS должны быть исключены из RRL параметры.  
  
## <a name="dane-support"></a>Поддержка DANE

Можно использовать поддержку DANE \(RFC 6394 и 6698\) для указания клиентов DNS, какие ЦС, они должны ожидать сертификатов должен быть выдан для имен доменов, размещенных на DNS-сервере. Это предотвращает возникновение формы man-in--middle атаки, когда кто-то может привести к повреждению кэш DNS и указывать свои собственные IP-адрес DNS-имя.  
  
Например представьте себе узла безопасный веб-сайт, использующий SSL на www.contoso.com с помощью сертификата из известного центра, с именем CA1. Кто-то по-прежнему сможете получить сертификат для www.contoso.com из разных, не так известной, центр с именем CA2 сертификата. Затем размещение на веб-сайт www.contoso.com фиктивные сущности можно повредить кэш клиента или сервера, чтобы он указывал www.contoto.com их поддельный узел. Конечный пользователь откроется сертификат из CA2 и может просто подтвердить его и подключитесь к этот поддельный узел. С помощью DANE клиент будет сделать запрос на DNS-сервер для contoso.com, запрашивая запись TLSA и узнайте, что сертификат для www.contoso.com был проблемы по CA1. Если предъявляется сертификат от другого ЦС, подключение будет прервано.  
  
## <a name="unknown-record-support"></a>Поддержка неизвестных записи

«Неизвестная запись» — это записи Ресурса, формат которого RDATA неизвестен на DNS-сервер. Недавно добавлена поддержка типов записей неизвестно (RFC 3597) означает, что можно добавить неподдерживаемых типов записей в зоне Windows DNS server в двоичном формате на лету. Windows кэширующий Сопоставитель уже имеет возможность обрабатывать типы неизвестной записи. Windows DNS-сервер не будет выполнять обработку определенных записей за неизвестными записями, но отправит его обратно в ответов Если для него будет получено запросов.  
  
## <a name="ipv6-root-hints"></a>IPv6 корневых ссылок

Корневые ссылки IPV6, как опубликованный IANA, были добавлены на DNS-сервер windows. Запрос на разрешение имени Интернет теперь можно использовать IPv6 корневые серверы для выполнения разрешения имен.

## <a name="windows-powershell-support"></a>Поддержка Windows PowerShell

В Windows Server 2016 представлены следующие новые командлеты Windows PowerShell и параметры.
  
-   **Добавить DnsServerRecursionScope**. Этот командлет создает новую область рекурсию на DNS-сервере. Рекурсия области используются политики DNS для указания списка серверов пересылки для использования в запрос DNS.  
  
-   **Remove-DnsServerRecursionScope**. Этот командлет удаляет существующих областей видимости рекурсии.  
  
-   **SET-DnsServerRecursionScope**. Этот командлет изменяет параметры существующей области рекурсии.  
  
-   **Get-DnsServerRecursionScope**. Этот командлет извлекает сведения о существующих областей видимости рекурсии.  
  
-   **Добавить DnsServerClientSubnet**. Этот командлет создает новую подсеть клиента DNS. Подсети используются политики DNS для определения, где находится DNS-клиента.  
  
-   **Remove-DnsServerClientSubnet**. Этот командлет удаляет существующие подсети клиента DNS.  
  
-   **SET-DnsServerClientSubnet**. Этот командлет изменяет параметры существующей подсети клиента DNS.  
  
-   **Get-DnsServerClientSubnet**. Этот командлет извлекает сведения о существующих подсетях клиента DNS.  
  
-   **Добавить DnsServerQueryResolutionPolicy**. Этот командлет создает новую политику разрешения запросов DNS. Политики разрешения запросов DNS используются для указания как это сделать, или если запрос ответил, на основе различных критериев.  
  
-   **Remove-DnsServerQueryResolutionPolicy**. Этот командлет удаляет существующие политики DNS.  
  
-   **SET-DnsServerQueryResolutionPolicy**. Этот командлет изменяет параметры существующей политики DNS.  
  
-   **Get-DnsServerQueryResolutionPolicy**. Этот командлет извлекает сведения о существующих политик DNS.  
  
-   **Enable-DnsServerPolicy**. Этот командлет включает существующих политик DNS.  
  
-   **Disable-DnsServerPolicy**. Этот командлет отключает существующих политик DNS.  
  
-   **Добавить DnsServerZoneTransferPolicy**. Этот командлет создает новую политику передачи зоны DNS server. Политики передачи зоны DNS укажите, следует ли запретить или игнорировать передачу зоны на основе различных критериев.  
  
-   **Remove-DnsServerZoneTransferPolicy**. Этот командлет удаляет существующие политики передачи зоны DNS server.  
  
-   **SET-DnsServerZoneTransferPolicy**. Этот командлет изменяет параметры существующей политики передачи зоны DNS server.  
  
-   **Get-DnsServerResponseRateLimiting**. Этот командлет извлекает параметры RRL.  
  
-   **SET-DnsServerResponseRateLimiting**. Этот командлет изменяет RRL установки.  
  
-   **Добавить DnsServerResponseRateLimitingExceptionlist**. Этот командлет создает список исключений RRL на DNS-сервере.  
  
-   **Get-DnsServerResponseRateLimitingExceptionlist**. Этот командлет извлекает списки excception RRL.  
  
-   **Remove-DnsServerResponseRateLimitingExceptionlist**. Этот командлет удаляет существующий список RRL исключение.  
  
-   **SET-DnsServerResponseRateLimitingExceptionlist**. Этот командлет изменяет RRL списка исключений.  
  
-   **Добавить DnsServerResourceRecord**. Этот командлет теперь поддерживает запись Неизвестный тип.  
  
-   **Get-DnsServerResourceRecord**. Этот командлет теперь поддерживает запись Неизвестный тип.  
  
-   **Remove-DnsServerResourceRecord**. Этот командлет теперь поддерживает запись Неизвестный тип.  
  
-   **SET-DnsServerResourceRecord**. Этот командлет теперь поддерживает тип записи в неизвестный
