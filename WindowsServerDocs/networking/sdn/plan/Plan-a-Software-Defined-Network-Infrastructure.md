# <a name="plan-a-software-defined-network-infrastructure"></a>Планирование инфраструктуры программно-конфигурируемой сети

>Относится к: Windows Server (полугодовой канал), Windows Server 2016

Дополнительные сведения о развертывании, планирование инфраструктуры программно-определяемой сети, в том числе к оборудованию и программному обеспечению. 


## <a name="prerequisites"></a>предварительные требования
В этом разделе описываются некоторые оборудованию и программному обеспечению, включая:

-   **Настроить группы безопасности, расположение файлов журнала и динамическая регистрация DNS** центра обработки данных необходимо подготовить для развертывания сетевого контроллера, который требует один или несколько компьютеров или виртуальных машин и один компьютер или виртуальную Машину. Перед развертыванием сетевого контроллера, необходимо настроить группы безопасности, расположение файлов журнала (при необходимости) и динамическая регистрация DNS.  Если в вашем центре обработки данных не подготовлен для развертывания сетевого контроллера, см. в разделе [установки и требования к подготовке для развертывания сетевого контроллера](Installation-and-Preparation-Requirements-for-Deploying-Network-Controller.md) подробные сведения.

-   **Физический сетевой** нужен доступ к устройствам физической сети для настройки виртуальных локальных сетей, маршрутизация, BGP, Data Center мост (ETS) при использовании технологии RDMA, и Data Center мост (PFC) при использовании RoCE на основе технологии RDMA. В этом разделе показаны конфигурации коммутатора вручную, а также Пиринг BGP на коммутаторах уровня 3 / маршрутизаторы или виртуальную машину, маршрутизации и удаленного доступа (RRAS).   

-   **Физические вычислительные узлы** эти узлы Hyper-V выполняется и необходимы для размещения SDN инфраструктуры и клиентского виртуальных машин.  Требуется определенного сетевого оборудования на этих узлах для лучшей производительности, который описан далее в **сетевое оборудование** раздел.

Уровень-2 [виртуальной ЛС](https://en.wikipedia.org/wiki/Virtual_LAN) поддерживает сети разделены на несколько сегментов логической сети.

## <a name="routing-infrastructure"></a>Инфраструктура маршрутизации  

Если вы развертываете в своей инфраструктуре SDN с помощью скриптов, управление, поставщика HNV, пути, и подсети виртуального IP-адреса должны маршрутизироваться друг с другом в физической сети.     

Сведения о маршрутизации \(например следующего прыжка\) для виртуальных IP-адресов подсети объявляется SLB/MUX и шлюзов RAS в физической сети с помощью внутреннего пиринга BGP. Логические сети виртуальных IP-адресов, не имеют назначение виртуальной локальной и не предварительно настроен в коммутатор уровня 2 (например, Top-of-Rack switch).  

Вам нужно создать узла BGP на маршрутизаторе, используемого в своей инфраструктуре SDN для получения маршруты для логических сетей виртуальных IP-адресов, объявленными Мультиплексорами SLB/и шлюзов RAS. Пиринга BGP необходимо пройти только один из способов (из SLB/MUX или внешнего узла BGP шлюза RAS).  Выше первый слой маршрутизации вы можно использовать статические маршруты или другой протокол динамической маршрутизации например OSPF, тем не менее, как уже говорилось ранее, префикс подсети IP-адреса для логических сетей виртуальный IP-адрес обязательно должны маршрутизироваться из физической сети для внешних узлов BGP.   

Пиринг BGP, обычно настраивается в управляемых коммутатор или маршрутизатор, как часть сетевой инфраструктуры. Узел BGP может также настроить на Windows Server с ролью сервера удаленного доступа (RAS) устанавливаются в режиме только маршрутизацию. Этот узел маршрутизатора BGP в инфраструктуре сети должен быть настроен на свой собственный ASN и предусматривают пиринга ASN, присвоенный компонентов SDN \(SLB/MUX и шлюзов RAS\). Из физического маршрутизатор или администратору сети в элементе управления маршрутизатора, необходимо получить следующие сведения:

- Router ASN  
- IP-адрес маршрутизатора  
- ASN для использования компонентов SDN (может быть любое число AS из частного диапазона ASN)

>[!NOTE]
>4-байтовым ASN не поддерживаются SLB/MUX. Необходимо выделить два байтовых ASN для SLB/MUX и маршрутизатором wo, которой он подключается. 4-байтовым ASN можно использовать в другом месте в вашей среде.  

Вам или вашему администратору сети необходимо настроить узел маршрутизатора BGP для приема подключений от ASN и IP-адрес или адрес подсети транзитной логической сети, в которых используется шлюз RAS и Мультиплексорами SLB /.

Дополнительные сведения см. в разделе [протокола пограничного шлюза (BGP)](../../../remote/remote-access/bgp/Border-Gateway-Protocol-BGP.md).

## <a name="default-gateways"></a>Шлюзы по умолчанию
Компьютеры, настроенные для подключения к нескольким сетям, такие как физические узлы и виртуальные машины шлюза должен иметь только один настроен шлюз по умолчанию. Настройка шлюза по умолчанию на адаптер, используемый для доступа в Интернет.

Для виртуальных машин выполните эти правила, чтобы решить, какие сети для использования в качестве шлюза по умолчанию.

1. Используйте транзитной логической сети как шлюз по умолчанию, если виртуальная машина подключается к транзитной сети, или в том случае, если это многосетевой транзитной сети или любой другой сети.
2. Используйте сеть управления как шлюз по умолчанию, если виртуальная машина подключается только к сети управления. 
3. Используйте сеть поставщика HNV для Мультиплексорами SLB/и шлюзов RAS. Не используйте сеть поставщика HNV в качестве шлюза по умолчанию. 
4. Не подключать виртуальные машины напрямую к сети Storage1, Storage2, общедоступного виртуального IP-адреса или частного виртуального IP-адреса.

Для узлов Hyper-V и узлы хранилища используйте сети как шлюз по умолчанию.  Сети хранения данных никогда не должно быть назначен шлюз по умолчанию.


## <a name="network-hardware"></a>Сетевое оборудование

Можно использовать следующие разделы для планирования развертывания сетевого оборудования.

### <a name="network-interface-cards-nics"></a>Сетевые платы

Сетевые адаптеры интерфейс (NIC), используемые в узлах Hyper-V и узлов хранения не требуются конкретные возможности для обеспечения лучшей производительности. 

Удаленный доступ к памяти (RDMA) — это способ, который дает возможность передачи больших объемов данных без использования ЦП, который позволяет освободить ЦП для выполнения прочей работы узла обхода ядра. 

Коммутатор внедренных коммутаторов (SET) является альтернативой решения, объединение Сетевых карт, который можно использовать в средах Hyper-V и стек программно определяемой сети (SDN) в Windows Server 2016. НАБОР объединяет некоторые функции объединения Сетевых карт в виртуальный коммутатор Hyper-V. 

Дополнительные сведения см. в разделе [удаленный доступ к памяти (RDMA) и коммутатора внедренных коммутаторов (SET)](../../../virtualization//hyper-v-virtual-switch/RDMA-and-Switch-Embedded-Teaming.md).   

Чтобы учесть дополнительные затраты из-за VXLAN или NVGRE заголовков инкапсуляции трафика виртуальной сети клиента, MTU сети fabric второго уровня (переключатели и узлов) необходимо задать больше или равным 1674 байт \(включая Ethernet уровня 2 заголовки\). 

Сетевые адаптеры, поддерживающие новое *EncapOverhead* ключевое слово расширенные адаптера задает MTU автоматически через агент узла сетевого контроллера. Сетевые адаптеры, которые не поддерживают новый *EncapOverhead* необходимо вручную установить значение MTU на каждом физическом узле, с помощью ключевого слова *JumboPacket* \(или аналогичный\) ключевое слово. 


### <a name="switches"></a>Коммутаторы

При выборе физическому коммутатору и маршрутизатор для вашей среды, убедитесь, что он поддерживает следующий набор возможностей:  

- Параметры Switchport MTU \(требуется\)  
- Значение MTU > = 1674 байт \(включая заголовок кадра протокола L2 Ethernet\)  
- Протоколы L3 \(требуется\)  
- ECMP  
- BGP \(IETF RFC 4271\)\-основе ECMP

Реализации должны удовлетворять операторы должны в следующие стандарты IETF.

- RFC 2545: «Расширения Multiprotocol BGP 4 для маршрутизации IPv6 между доменами»  
- RFC 4760: «Многопротокольный Extensions для BGP-4»  
- RFC 4893: «Поддержка BGP AS четыре октета номер места»  
- RFC 4456: «Отражения маршрут BGP: Вместо полной сетки внутренней BGP (IBGP)»  
- RFC 4724: «Корректного перезапуска механизм для BGP»  

Требуются следующие протоколы тегов.

- VLAN - изоляции различных типов трафика
- 802.1Q магистрали

Ниже приведены ссылки элемента управления.

- Качество обслуживания \(PFC, требуется только при использовании RoCE\)
- Расширенный выбор трафика \(802.1Qaz\)
- Управление потоком на основании приоритета \(802.1 p и Q и 802.1Qbb\)

Следующие элементы обеспечивают доступность и избыточность.

- Доступность коммутатора (обязательно)
- Маршрутизатор высокой доступности необходимо выполнять функции шлюза. Это можно сделать с помощью маршрутизатора switch\ несколькими шасси или технологии, такие как VRRP.

Ниже приведены возможности управления.

**Мониторинг**

- SNMP v1 или v2 SNMP (требуется, если с помощью сетевого контроллера для мониторинга физического коммутатора)  
- SNMP MIB \(требуется, если вы используете сетевого контроллера для мониторинга физического коммутатора\)  
- MIB-II (RFC 1213) об LLDP, интерфейс MIB \(RFC 2863\), IF-MIB, MIB IP-адрес, IP-ПЕРЕСЫЛКИ MIB, Q-МОСТ MIB, МОСТ MIB, LLDB MIB, сущности MIB, IEEE8023 LAG-MIB

## <a name="extending-your-infrastructure"></a>Расширение инфраструктуры  
Изменения размера и требования к ресурсам инфраструктуры, зависят от виртуальных машин клиента рабочей нагрузки, которые вы планируете размещать. ЦП, памяти и дискового для виртуальных машин инфраструктуры (например: сетевой контроллер, SLB, шлюз, и т.д.), перечислены в приведенной выше таблице. Можно добавить несколько этих виртуальных машин инфраструктуры для масштабирования по мере необходимости. Тем не менее все виртуальные машины клиента, работающих на узлах Hyper-V имеют собственные ЦП, памяти и на диске, которые необходимо учитывать.   

Когда рабочей нагрузки виртуальных машин клиента начать потреблять слишком много ресурсов на физических узлах Hyper-V, можно расширить инфраструктуру, добавив дополнительные физические узлы. Это можно сделать с помощью Virtual Machine Manager или с помощью сценариев PowerShell (в зависимости от способа развертывания изначально инфраструктуры) для создания новых ресурсов сервера через сетевой контроллер. Если необходимо добавить дополнительные IP-адреса для сети поставщика HNV, можно создать новый логических подсетей (с соответствующие пулы IP-адресов), которые могут использовать узлы.
