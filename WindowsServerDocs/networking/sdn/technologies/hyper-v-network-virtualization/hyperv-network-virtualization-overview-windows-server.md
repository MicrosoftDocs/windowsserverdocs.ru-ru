# <a name="hyper-v-network-virtualization-overview-in-windows-server-2016"></a>Обзор виртуализации сети Hyper-V в Windows Server 2016

>Область применения: Windows Server (канал точками годовой), Windows Server 2016

В Windows Server 2016 и диспетчер виртуальных машин Майкрософт предоставляет решение виртуализации сети в сквозном режиме.  Существует пять основных компонентов, составляющих решение Майкрософт по виртуализации сети.  
  
-   **Windows Azure Pack для Windows Server** предоставляет клиентский портал для создания виртуальных сетей и административные портал для управления виртуальными сетями.  
  
-   **Диспетчер виртуальных машин** (VMM) обеспечивает централизованное управление структуре сети.  
  
-   **Сетевой контроллер Microsoft** обеспечивает централизованную программируемую точку автоматизации для управления, настройки, мониторинга и устранения неисправностей в виртуальной и физической сетевой инфраструктуре центра обработки данных.  
  
-   **Виртуализация сети Hyper-V** предоставляет инфраструктуру для виртуализации сетевого трафика.  
  
-   **Шлюзы виртуализации сети Hyper-V** обеспечивают связь между виртуальными и физическими сетями.  
  
В этом разделе вводятся основные понятия и описываются основные преимущества и возможности виртуализации сети Hyper-V (одного из компонентов решения виртуализации сети) в Windows Server 2016. Здесь также описаны преимущества применения виртуализации сети обоих частных облаков, предназначенных для консолидации корпоративной рабочей нагрузки и поставщиков услуг общедоступных облаков инфраструктуру как услугу (IaaS).

## <a name="BKMK_OVER"></a>Описание компонентов  
Виртуализация сети Hyper-V предоставляет (сеть ВМ) «виртуальные сети» для виртуальных машин, подобно тому как виртуализация сервера (низкоуровневая оболочка) предоставляет «виртуальные машины» для операционной системы. Виртуализация сети разрывает связь виртуальных сетей с инфраструктурой физической сети и позволяет устранить ограничения виртуальной ЛС и иерархического назначения IP-адресов из подготовки виртуальной машины. Такая гибкость позволяет легко для клиентов для перемещения в облака IaaS и эффективно для поставщиков услуг размещения и администраторам центров обработки данных, управления их инфраструктурой, сохраняя необходимую многопользовательскую изоляцию, требований к безопасности, и вспомогательные перекрывающихся виртуальной машины IP-адресов.  
  
Клиентам необходим способ органичного расширения своих центров обработки данных в облако. Сегодня существует техническими трудностями в создание такой цельной гибридной облачной архитектуры. Один из важнейших стоящих перед клиентами является повторное использование собственных сетевых топологий (подсети, IP адреса, сетевые службы и т. д.) в облаке и Создание моста между собственными ресурсами локальными и облачными ресурсами.  Виртуализация сети Hyper-V реализует концепцию сети виртуальной Машины, которая не зависит от базовой физической сети. В данной концепции сети виртуальной Машины, состоящей из одного или нескольких виртуальных подсетей точное расположение физической сети виртуальных машин, подключенных к виртуальной сети связано с топологией виртуальной сети. В результате клиентам легко переносить свои виртуальные подсети в облако сохраняя существующие IP-адреса и топологию в облаке, чтобы существующие службы продолжают работать независимо от физического расположения подсетей. Виртуализация сети Hyper-V позволяет создать цельное гибридное облако.  
  
В дополнение к гибридному облаку многие организации объединяют свои центры обработки данных и создают Частные облака, чтобы внутри организации пользоваться преимуществами облачной архитектуры эффективности и масштабируемости. Виртуализация сети Hyper-V обеспечивает большую гибкость и эффективность за частного облака за счет разделения в подразделении сетевой топологии (которая становится виртуальной) из фактической топологии физической сети. Таким образом бизнес-подразделениям можно легко общий доступ к внутреннему частному облаку, при этом изолированными друг от друга и сохраняя существующие сетевые топологии. Группа эксплуатации центра обработки данных получает возможность разворачивать и динамически перемещать рабочие нагрузки в любом месте центра обработки данных без перерывов сервера, предоставляя работы и общую эффективность центра обработки данных.  
  
Для владельцев рабочей нагрузки основное преимущество заключается в том, что теперь они могут перемещать «топологии» своей рабочей нагрузки в облаке не меняя свои IP-адреса и не переписывая свои приложения. Например обычное БИЗНЕС-приложение трехуровневой состоит из уровня переднего плана, уровня бизнес логики и уровень базы данных. Через политику виртуализация сети Hyper-V позволяет клиенту подключения все или часть три уровня в облако, сохраняя при этом топологию маршрутизации и IP-адреса служб (т. е. виртуальной машины IP-адреса), не требуя изменения приложений должен быть изменен.  
  
Для владельцев инфраструктуры дополнительная свобода в размещении виртуальных машин означает возможность перемещения рабочих нагрузок в любом месте в центрах обработки данных без смены виртуальных машин и перенастройки сетей. Например виртуализации сети Hyper-V обеспечивает динамическую миграцию между подсетями, чтобы виртуальная машина может выполните динамическую миграцию в любом месте в центре обработки данных без приостановки на обслуживание. Ранее Динамическая миграция была ограничена той же подсети, ограничения, где может находиться виртуальных машин. Между подсетями динамической миграции позволяет администраторам объединять рабочие нагрузки на основании динамических требований к ресурсам, эффективности энергопотребления и можно также обеспечивает возможность проведения обслуживания инфраструктуры без приостановки функционирования рабочей нагрузки клиентов.  
  
## <a name="BKMK_APP"></a>Практическое применение  
Благодаря успеху виртуализированных центров обработки данных ИТ-организации и поставщики услуг размещения (поставщики, которые предоставляют сдачу или физических серверов) начали предлагать более гибкие виртуализированные инфраструктуры, которые упрощают предлагать своим клиентам экземпляров сервера по запросу. Этот новый класс услуг называется инфраструктура как услуга (IaaS). Windows Server 2016 предоставляет все возможности платформы, необходимые для включения корпоративным клиентам создавать частные облака и переходить в ИТ, как рабочую модель службы. Windows Server 2016 2016 также позволяет поставщикам услуг размещения создавать общедоступные облака и предлагать своим клиентам решения IaaS. В сочетании с Virtual Machine Manager и Windows Azure Pack для управления политикой виртуализации сети Hyper-V, Майкрософт предоставляет мощное Облачное решение.  
  
Виртуализация Windows Server 2016 Hyper-V обеспечивает виртуализацию сети на основе политики, программного обеспечения, позволяющую снизить расходы сталкиваются предприятия при развертывании выделенных облаков IaaS, и предоставляет поставщикам услуг размещения в облаке лучше гибкость и масштабируемость в управлении виртуальными машинами для более эффективного использования ресурсов на управление.  
  
Сценарий IaaS, включающий виртуальные машины из разных отделов предприятия (выделенное облако) или разных клиентов (размещенное облако), требует безопасной изоляции. В этом сценарии современных решений, виртуальные локальные сети (VLAN), может вызвать значительные затруднения.  
  
**Виртуальные локальные сети**  
  
В настоящее время виртуальные локальные сети — это механизм, большинство организаций применяет для поддержки многократного использования пространства адресов и изоляции клиентов. Виртуальной локальной сети применяются явные пометки (ИД виртуальной ЛС) в заголовках кадров Ethernet, а коммутаторов Ethernet обеспечение изоляции и ограничение трафика к сетевым узлам с того же ИД виртуальной ЛС. Основные недостатки виртуальных локальных сетей таковы:  
  
-   Повышенный риск случайного отключения по причине громоздкой процедуры перенастройки коммутаторов рабочей всякий раз, когда виртуальные машины или границы изоляции перемещаются в динамическом центре обработки данных.  
  
-   Ограниченная масштабируемость, так как более 4094 виртуальных локальных сетей и обычные коммутаторы поддерживают не более 1000 идентификаторов виртуальных ЛС.  
  
-   Существование в пределах одной IP-подсети, что ограничивает количество узлов в отдельной виртуальной ЛС и размещение виртуальных машин, в зависимости от физического расположения. Хотя виртуальные локальные сети могут быть развернуты между сайтами, вся виртуальная локальная сеть должна находиться в той же подсети.  
  
**Назначение IP-адресов**  
  
В дополнение к недостаткам, связанным с виртуальными локальными сетями назначение IP-адреса виртуальной машины представляется проблем, которые включают в себя:  
  
-   IP-адреса виртуальных машин определяются физическим расположением в сетевой инфраструктуре центра обработки данных. В результате для перемещения в облако обычно требуется менять IP-адреса рабочей нагрузки служб.  
  
-   Политики привязаны к IP-адреса, такие как правила брандмауэра, обнаружение ресурсов и каталоге службы и т. д. Изменение IP-адресов требует обновления всех сопоставленных политик.  
  
-   Развертывание виртуальной машины и изоляция трафика зависят от топологии.  
  
Когда администраторы сети центра обработки данных планируют физическую структуру центра обработки данных, им необходимо принимать решения о будет быть физического расположения подсетей и их маршрутизации. Эти решения основываются на технологии IP и Ethernet, которая влияет на потенциальные IP-адреса, которые разрешены для виртуальных машин, работающих на данном сервере или сверхкомпактном сервере, подключенном к определенной стойке в центре обработки данных. Виртуальная машина подготовленная и размещенная в центре обработки данных, она должна соответствовать этим стратегиям и ограничениям относительно IP-адрес. Таким образом обычно является что администраторам центров обработки данных назначить новые IP-адреса виртуальных машин.  
  
Проблема с этим требованием, что в дополнение к наличию адреса семантическая сведения, сопоставленные с IP-адрес. Например одна подсеть может включать в себя конкретные службы или находиться в определенное физическое расположение. Правила брандмауэра, политики управления доступом и сопоставления безопасности IPsec обычно сопоставлены IP-адресов. Изменение IP-адресов вынуждает владельцев виртуальных машин вносить поправки во все свои политики, которые были основаны на исходный IP-адрес. Дополнительные издержки перенумерацию настолько высоки, что многие предприятия развертывают только новые службы в облаке, не трогают уже приложений прежних версий.  
  
Виртуализация сети Hyper-V разрывает связь виртуальных сетей для виртуальных машин клиента с инфраструктурой физической сети. В результате она позволяет виртуальным машинам клиента сохранять свои исходные IP-адреса, а также дает администраторам центров обработки данных возможность подготавливать виртуальные машины клиентов в любом месте центра обработки данных без перенастройки физических IP-адресов или ИД виртуальных ЛС. Ниже приводится обзор основных функциональных возможностей.  
  
## <a name="BKMK_NEW"></a>Важные функции  
Ниже приведен список основных функций, преимущества и возможности виртуализации сети Hyper-V в Windows Server 2016:  
  
-   **Обеспечивает гибкое размещение рабочих нагрузок - сетевой изоляции и использование IP-адресов повторно без виртуальных ЛС.**  
  
    Виртуализация сети Hyper-V разрывает связь виртуальных сетей клиента с инфраструктурой физических сетей поставщиков услуг размещения, обеспечивая свободу размещения рабочих нагрузок в центрах обработки данных. Размещение нагрузки на виртуальной машине, поскольку оно реализуется в узлах Hyper-V на основании политик виртуализации, которые управляются программным обеспечением- и больше не ограничен назначение IP-адресов или требований в отношении изоляции виртуальной локальной сети физической сети.  
  
    Виртуальные машины разных клиентов с перекрывающимися IP-адресами теперь могут развертываться на одном сервере узла без необходимости громоздкой настройки виртуальной локальной сети или нарушения иерархии IP-адресов. Это может упростить перенос рабочих нагрузок клиентов к поставщикам, поскольку позволит клиентам переносить рабочие нагрузки без изменений, включая сохранение IP-адреса виртуальной машины без изменений размещения IaaS с общим доступом. У поставщика услуг размещения поддержка большого количества клиентов, которые хотят расширить свое существующее пространство сетевых адресов общей центра обработки данных IaaS — сложная задача, требующая настройки и поддержания изолированных виртуальных локальных сетей для каждого клиента, чтобы обеспечить совместное существование потенциально перекрывающихся пространств адресов. Виртуализация сети Hyper-V поддерживать перекрывающиеся адреса становится проще и требует меньше работы перенастройке сети со поставщику услуг размещения.  
  
    Кроме того обслуживание и обновления физической инфраструктуры можно проводить без приостановки функционирования рабочих нагрузок клиентов. Виртуализация сети Hyper-V можно переносить виртуальные машины на конкретного узла, стойки, подсети, виртуальной локальной сети или всего кластера без необходимости физического IP-адрес или проводить масштабную перенастройку.  
  
-   **Обеспечение простоты перемещения рабочих нагрузок в облако IaaS с общим доступом**  
  
    С помощью виртуализации сети Hyper-V IP-адреса и настройки виртуальных машин останется неизменным. Это позволяет ИТ-организациям возможность большей легкостью перемещать рабочие нагрузки со своих центров обработки данных для общего поставщика размещения IaaS с потребуются минимальные изменения параметров рабочей нагрузки или инструментов и инфраструктуры политики. В случаях, где имеется подключение между двумя центрами обработки данных, ИТ-администраторы могут продолжить использование инструментов без перенастройки.  
  
-   **Обеспечение динамической миграции между подсетями**  
  
    Динамическая миграция рабочих нагрузок виртуальных машин традиционно была ограничена одной IP-подсети или виртуальной локальной сети, потому что перемещение между подсетями требовало операционной системы виртуальной машины изменение его IP-адреса. Такое изменение адреса нарушает существующие связи и работу служб, запущенных на виртуальной машине. Виртуализация сети Hyper-V рабочие нагрузки можно выполнить динамическую миграцию с серверов под управлением Windows Server 2016 в одной подсети, на серверы под управлением Windows Server 2016 в другой подсети, без изменения IP-адреса рабочей нагрузки. Виртуализация сети Hyper-V гарантирует, что изменений расположения виртуальных машин, вызванных динамической миграцией обновление и синхронизацию между узлами, которые имеют непрерывную связь с перемещенной виртуальной машиной.  
  
-   **Обеспечение простоты управления несвязанным администрированием сервера и сети**  
  
    Размещение нагрузки на сервере упрощается, так как миграция и размещение рабочих нагрузок не зависят от лежащей в основе физической сети. Администраторы серверов могут сосредоточиться на управлении службами и серверами, и администраторы сети могут сосредоточиться на общей сетевой инфраструктуре и управлении трафиком. Это позволяет администраторам серверов ЦОД развертывать и перемещать виртуальные машины без изменения IP-адресов виртуальных машин. Снижаются затраты, потому что виртуализация сети Hyper-V размещение виртуальных машин может происходить независимо от топологии сети, сокращая потребность в сетевых администраторов со размещениями, которые могут вызвать изменение границ изоляции.  
  
-   **Упрощение сети и повышение эффективности использования ресурсов сервера и сети**  
  
    Отсутствие гибкости у виртуальных локальных сетей и зависимость размещения виртуальных машин от инфраструктуры физической сети выражаются в излишней подготовке ресурсов и недостаточной эффективности их использования. Разрывая эту зависимость, увеличенная гибкость размещения рабочих нагрузок виртуальных машин можно упростить управление сетью и повысить сервера и использование сетевых ресурсов. Обратите внимание, что виртуализация сети Hyper-V поддерживает виртуальные локальные сети в контексте физического центра обработки данных. Например в центре обработки данных может потребоваться весь трафик виртуализации сети Hyper-V на определенную виртуальную локальную сеть.  
  
-   **Совместимо с существующей инфраструктурой и развивающимися технологиями**  
  
    Виртуализация сети Hyper-V можно развернуть в современных центров обработки данных, но она совместима с новые технологии datacenter «однородной сети».  
  
    Например Виртуализация в Windows Server 2016 поддерживает формат инкапсуляция VXLAN и открыть виртуальный коммутатор протокола управления базы данных (OVSDB) как Южный интерфейса (SBI)..  
  
-   **Предоставляет возможности взаимодействия и адаптации к экосистеме**  
  
    Виртуализация сети Hyper-V поддерживает несколько конфигураций для связи с существующими ресурсами, таких как перекрестное подключение в сетях организации, сеть хранения (данных SAN), доступ к ресурсам без виртуализации и т. д. Корпорация Майкрософт стремится к сотрудничеству с партнерами по экосистеме на поддержку и улучшение взаимодействия виртуализации сети Hyper-V в отношении производительности, масштабируемости и управляемости.  
  
-   **Конфигурация на основе политики**  
  
    Политиках сетевой виртуализации в Windows Server 2016, настраиваются через сетевой контроллер корпорации Майкрософт. Сетевой контроллер имеет RESTful Северный API и интерфейс Windows PowerShell для настройки политики. Дополнительные сведения о сетевой контроллер Майкрософт см. в разделе [сетевой контроллер](../../../sdn/technologies/network-controller/../../../sdn/technologies/network-controller/Network-Controller.md).  
  
## <a name="BKMK_SOFT"></a>Требования к программному обеспечению  
Виртуализация сети Hyper-V с помощью сетевого контроллера Microsoft требуется Windows Server 2016 и роль Hyper-V.
