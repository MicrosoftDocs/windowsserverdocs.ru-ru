# <a name="deploy-password-based-8021x-authenticated-wireless-access"></a>Развертывание на основе Password\ с проверкой подлинности беспроводного доступа 802.1 X

>Область применения: Windows Server (канал точками годовой), Windows Server 2016

Это дополнительное руководство по в Windows Server&reg; 2016 руководство по основной сети. Руководство по основной сети содержит инструкции по планированию и развертыванию компонентов, необходимых для создания полнофункциональной сети и нового домена Active Directory® в новом лесу.

В этом руководстве рассматривается построение по основной сети приводятся инструкции о том, как развернуть институтом инженеров по электротехнике и электронике \(IEEE\) 802.1X\-проверку подлинности беспроводной доступ IEEE 802.11 с помощью защищенных расширяемого протокола проверки подлинности — проверка подлинности протоколом MSCHAP версии 2 \ (v2\ PEAP\-MS\-CHAP).

Поскольку PEAP\-MS\-CHAP v2 требует от пользователя ввод учетных данных на основе password\, а не сертификат во время процесса проверки подлинности, это обычно проще и дешевле развертывание чем EAP\ TLS или PEAP\-TLS.

>[!NOTE]
>В этом руководстве IEEE 802.1 X с проверкой подлинности беспроводного доступа с PEAP\-MS\-CHAP v2 сокращенном виде «беспроводной доступ» и «Wi-Fi доступ».

- Сертификаты сервера для компьютеров под управлением NPS.

- Беспроводные клиентские компьютеры под управлением Windows&reg; 10, Windows 8.1 или Windows 8.

### <a name="ieee-8021x"></a>IEEE 802.1 X
Стандарт IEEE 802.1 X определяет управление доступом к сети на основе port\, который используется для предоставления доступа к сети с проверкой подлинности к Ethernet-сетям. Управление доступом на основе port\ сети используются физические характеристики инфраструктуры коммутируемой локальной сети для проверки подлинности устройств, подключенных к Сетевому порту. Доступ к данному порту может быть запрещен в случае непрохождения проверки подлинности. Несмотря на то, что этот стандарт, предназначенный для проводных Ethernet-сетей, был адаптирован и для использования в беспроводных локальных сетях 802.11.

### <a name="8021x-capable-wireless-access-points-aps"></a>802.1X\-\(APs\) точки беспроводного доступа с поддержкой
Этот сценарий предусматривает развертывание одного или нескольких 802.1X\-поддержкой беспроводных точек доступа, совместимых с протоколом удаленного пользователя службы проверки подлинности Dial\-In \(RADIUS\).

802.1 X и RADIUS\-совместимые точки доступа, когда развернуты в инфраструктуре протокола RADIUS с RADIUS-сервером, таких как сервер политики сети, называются *RADIUS-клиенты*.

### <a name="wireless-network-security-methods"></a>Методы безопасности для беспроводной сети
**Беспроводные сети методы безопасности** представляет собой группировку неформальным проверки подлинности беспроводной сети \ (которую иногда называют беспроводной security\) и шифрования безопасности беспроводной сети. Беспроводной проверки подлинности и шифрования используются парами для предотвращения несанкционированного доступа к беспроводной сети и для защиты беспроводной передаче данных. 

При настройке параметров безопасности беспроводной сети в беспроводной сети из групповой политики, существует несколько комбинаций, которые можно выбрать. Тем не менее WPA2\-Enterprise, WPA\ Enterprise и открыть с 802.1 X стандартов проверки подлинности поддерживаются развертываний 802.1 X с проверкой подлинности беспроводной сети.

>[!NOTE]
>При настройке политик беспроводной сети, необходимо выбрать **предприятия WPA2\**, **WPA\ предприятия**, или **открыть с 802.1 X** Чтобы получить доступ к параметрам EAP, необходимых для беспроводных сетей с проверкой подлинности 802.1 X.  

#### <a name="wireless-authentication"></a>Проверка подлинности беспроводной сети
В этом руководстве рекомендует использование следующим стандартам проверки подлинности беспроводной сети 802.1 X с проверкой подлинности беспроводных сред.  

**Доступ к Wi\-Fi Protected — предприятия \(WPA\-Enterprise\)** WPA представляет собой промежуточный стандарт, разработанный Wi-Fi Alliance, в соответствии с протоколом 802.11 безопасности беспроводной сети. Протокол WPA был разработан в ответ на ряд серьезные недостатки, обнаруженные в предыдущей протокола \(WEP\) проводной эквивалентные конфиденциальности.

WPA\ предприятия обеспечивает повышенную безопасность по сравнению WEP по.  

1. Требование проверки подлинности, который использует 802.1 X EAP framework в рамках инфраструктуры, который обеспечивает централизованный взаимную проверку подлинности и динамическое управление ключами  

2. Улучшение \(ICV\) значения проверки целостности с \(MIC\) проверка целостности сообщений, чтобы защитить заголовка и полезных данных  

3. Реализация счетчик кадров, чтобы не допустить атаки с повторением пакетов  

**Доступ Wi\-Fi Protected 2 — \(WPA2\-Enterprise\) предприятия** как WPA\-Enterprise standard, WPA2\ Enterprise использует платформу EAP и 802. 1 X. Корпоративный WPA2\ обеспечивает надежную защиту для нескольких пользователей и большое управляемой сети. Корпоративный WPA2\ является надежным протоколом, который предотвращает несанкционированный доступ, проверяя пользователей посредством проверки подлинности сервера.  

#### <a name="wireless-security-encryption"></a>Шифрование безопасности беспроводной сети
Шифрование безопасности беспроводной сети используется для защиты беспроводной передаче данных, передаваемых между клиентом беспроводной сети и беспроводной точке доступа. Шифрование безопасности беспроводной сети используется в сочетании с методом проверки подлинности безопасности выбранной сети. По умолчанию компьютеры под управлением Windows 10, Windows 8.1 и Windows 8 поддерживает два стандарта шифрования:

1. **Протокол целостности временного ключа** \(TKIP\) — это более старых протокол шифрования, который изначально была разработана для предоставления более надежного шифрования беспроводной чем выполняемой протоколом изначально слабых \(WEP\) проводной эквивалентные конфиденциальности. TKIP был разработан IEEE 802.11i задач группы и Wi\-Fi Alliance для замены WEP без необходимости замены оборудования прежних версий. TKIP — это набор алгоритмов инкапсулирует WEP полезных данных, который позволяет пользователям устаревшее оборудование Wi-Fi для обновления до TKIP без замены оборудования. Как WEP TKIP использует алгоритм шифрования RC4 потока его основе. Новый протокол, однако шифрует всех пакетов данных с уникальный ключ шифрования и ключи, намного надежнее, чем с WEP. Несмотря на то, что TKIP полезен для обновление системы безопасности на более старых устройствах, предназначенных для использования только WEP, не устраняет все проблемы безопасности, обращенной беспроводных локальных сетях и в большинстве случаев не достаточно устойчивы, чтобы защитить конфиденциальные правительственным организациям или передаче корпоративных данных.  

2. **Расширенного стандарта шифрования** \(AES\) предпочтительные шифрования протокол используется для шифрования данных коммерческих и государственные учреждения. AES обеспечивает более высокий уровень безопасности при беспроводной передаче данных, чем TKIP или WEP. В отличие от TKIP и WEP AES требует беспроводного оборудования, который поддерживает стандарт AES. AES является стандарт symmetric\ ключ шифрования, использующий три блочные шифры, AES\-128, AES\ 192 и AES\-256.

В Windows Server 2016 беспроводной шифрования на основе AES\ доступны следующие методы для конфигурации в свойствах профиля беспроводной сети при выборе метода проверки подлинности WPA2\ предприятия, который рекомендуется использовать.

1. **AES\ CCMP**. Счетчик \(CCMP\) режим шифров блок цепочки сообщений протокола проверки подлинности кода реализует 802 стандартных.11i и предназначен для более чем WEP шифрование и использует ключи шифрования AES 128 бит.
2. **AES\ GCMP**. Протокол режим счетчика Galois \(GCMP\) поддерживается стандарт 802.11ac, более эффективно, чем AES\ CCMP и обеспечивает более высокую производительность для беспроводных клиентов. GCMP использует ключи шифрования AES 256 бит.

> [!IMPORTANT]
> Проводной конфиденциальности эквивалентность \(WEP\) был исходного беспроводной стандарт безопасности, был использован для шифрования сетевого трафика. Не следует развертывать WEP в сети, так как хорошо известных уязвимостей в этой форме устаревшие безопасности.

### <a name="active-directory-doman-services-ad-ds"></a>Домен Active Directory служб \(AD DS\)
Службы AD DS предоставляют распределенную базу данных, который сохраняет и управляет ими сведения о сетевых ресурсах и включать данные из приложений с поддержкой directory\. Администраторы могут использовать AD DS для упорядочения в иерархическую вложенную структуру элементов сети, такие как пользователи, компьютеры и другие устройства. Иерархическая вложенная структура включает лес Active Directory, домены в лесу и организационные подразделения \(OUs\) в каждом домене. Сервер, на котором выполняется AD DS, называется *контроллер домена*.  

AD DS содержит учетные записи пользователей, учетные записи компьютеров и свойства учетной записи, которые требуются для проверки подлинности учетных данных пользователя и проверки авторизации для беспроводных подключений IEEE 802.1 X и PEAP\-MS\-CHAP v2.

### <a name="active-directory-users-and-computers"></a>Пользователи и компьютеры Active Directory
Active Directory-пользователи и компьютеры — это компонент службы AD DS, содержащую учетные записи, которые представляют физические объекты, такие как компьютера, пользователя или группу безопасности. A *группы безопасности* — это коллекция учетных записей пользователей или компьютеров, которые администраторы могут управлять как единым целым. Учетные записи пользователей и компьютеров, входящих в определенную группу, называются *членов группы*.  

### <a name="group-policy-management"></a>Управление групповой политикой
Управление групповыми политиками позволяет directory\ изменения и настройки управления на основе параметров пользователя и компьютера, включая сведения о безопасности и пользователей. Используйте групповую политику для определения конфигурации групп пользователей и компьютеров. С помощью групповой политики можно указать параметры для записи реестра, безопасности, установки программного обеспечения, сценарии, перенаправление папок, служб удаленной установки и обслуживания Internet Explorer. Объект групповой политики, которые содержатся параметры, которые вы создаете в групповой политике, \(GPO\). Связав объект групповой Политики с выбранными контейнерами системы Active Directory — сайты, домены и подразделения, можно применить параметры групповой Политики к пользователям и компьютерам в эти контейнеры Active Directory. Для управления объектами групповой политики на предприятии, можно использовать консоль управления групповыми политиками управления \(MMC\).  

Это руководство содержит подробные инструкции о том, как указывать параметры в беспроводной сети \ (IEEE 802.11\) расширения политик для управления групповой политикой. Беспроводной сети \ (IEEE 802.11\) политики настроить член domain\ беспроводных клиентских компьютеров с необходимых подключений и беспроводной доступ с проверкой подлинности параметров беспроводного доступа 802. 1 X.

### <a name="server-certificates"></a>Сертификаты сервера
Этот сценарий развертывания требуются сертификаты каждого NPS-сервера, выполняющего проверку подлинности 802.1 X.  

Сертификат сервера — это цифровой документ, который обычно используется для проверки подлинности и защиты данных в общедоступных сетях. Сертификат безопасно привязывает открытый ключ к объекту, обладающему закрытым ключом. Сертификаты утверждаются цифровой подписью ЦС, и они могут выдаваться для пользователя, компьютера или службы.  

Центр сертификации \(CA\) — организация, ответственная за установление и подтверждает подлинность открытых ключей, принадлежащих субъектам \ (как правило, пользователям или компьютеры\) либо другим ЦС. Центр сертификации может привязывать открытые ключи к различающимся именам посредством подписанных сертификатов, управлять серийными номерами сертификатов и отзыв сертификатов.  

Active Directory службы сертификации \(AD CS\) — это роль сервера, выдаче сертификатов сети ЦС. AD CS сертификатов инфраструктуры, также известные как *инфраструктуры открытых ключей \(PKI\)*, предоставляет настраиваемые службы выдачи и управления сертификатов для предприятия.

### <a name="eap-peap-and-peap-ms-chap-v2"></a>Протокол EAP, PEAP и PEAP\-MS\-CHAP v2
Расширяемый протокол проверки подлинности \(EAP\) расширяет возможности протоколов Point\ заполнением точки \(PPP\), позволяя дополнительные методы проверки подлинности, использующих учетные данные и сведения для обмена произвольной длины. С помощью проверки подлинности EAP, в сети, доступ к клиента и структуры проверки подлинности \ (например, сервере политики сети) должны поддерживать тот же тип EAP, для успешной проверки подлинности и выполнить. Windows Server 2016 включает инфраструктура EAP, поддерживает два типа EAP, а также возможность передачи сообщений EAP для серверов политики сети. С помощью протокола EAP, можно обеспечить поддержку дополнительных схем проверки подлинности, известные как *типы EAP*. Ниже приведены типы EAP, которые поддерживаются в Windows Server 2016:  

- Безопасность транспортного уровня \(TLS\)

- Протокол проверки подлинности MSCHAP версии 2 \ (v2\ MS\ CHAP)

>[!IMPORTANT]
>Строгие типы EAP \ (например, тех, которые основаны на certificates\) обеспечивают лучшую защиту от атак brute\, атак перебором по словарю и пароля атак, чем протоколы проверки подлинности на основе password\ \ (например, CHAP или MS\ CHAP 1\ версии).

Защищенный EAP \(PEAP\) использует протокол TLS для создания зашифрованного канала между PEAP-клиентом, таких как беспроводной компьютера и проверки подлинности PEAP, таких как сервер политики сети или другие серверы RADIUS. PEAP не определяет метод проверки подлинности, но она обеспечивает дополнительную безопасность для других протоколов проверки подлинности EAP \ (например, v2\ EAP\-MS\-CHAP), могут работать через зашифрованный канал TLS, предоставленных в PEAP. PEAP используется как метод проверки подлинности для доступа клиентов, подключающихся к сети вашей организации с помощью следующих типов \(NASs\) серверы доступа к сети:

- 802.1X\-поддержкой беспроводные точки доступа

- 802.1X\-поддержкой коммутаторы с проверкой подлинности

- Компьютеры под управлением Windows Server 2016 и \(RAS\) службы удаленного доступа, настроенных в качестве виртуальной частной сети \(VPN\) серверов и серверов DirectAccess

- Компьютеры под управлением Windows Server 2016 и службы удаленных рабочих столов

Проще развертывать чем EAP\ TLS, так как проверка подлинности пользователей выполняется с помощью учетных данных на основе password\ PEAP\-MS\-CHAP v2 \ (имя пользователя и password\), а не сертификаты или смарт-карты. Только сервер политики сети или другие RADIUS-серверы должны иметь сертификат. Сертификат сервера политики сети используется сервером политики сети во время процесса проверки подлинности для подтверждения его подлинности для PEAP-клиентов.

Это руководство содержит инструкции по настройке беспроводных клиентов и server\(s\) вашего сервера политики сети для использования PEAP\-MS\-CHAP v2 для доступа с проверкой подлинности 802.1 X.

### <a name="network-policy-server"></a>Сервер политики сети
\(NPS\) сервера политики сети позволяет централизованно настраивать и контролировать политики сети с помощью удаленного пользователя службы проверки подлинности Dial\-In \(RADIUS\) сервера и прокси-сервера RADIUS. Сервер политики сети является обязательным, если развертывание беспроводного доступа 802.1 X.

При настройке точки беспроводного доступа 802.1 X в качестве RADIUS-клиентов на сервере политики сети, сервер политики сети обрабатывает запросы на подключение, отправленных точки доступа. Во время обработки запросов на подключение, выполняет проверку подлинности и авторизацию. Проверка подлинности определяет ли клиент представил действительные учетные данные. Если сервер политики сети успешно проходит проверку подлинности клиента, сервер политики сети определяет ли клиент авторизован для запрошенного подключения и либо разрешает или запрещает подключение. Это описано более подробно следующим образом:

#### <a name="authentication"></a>Проверка подлинности

Успешной взаимной PEAP\-MS\-CHAP v2 проверки подлинности имеет два основных частей:

1.  Клиент выполняет проверку подлинности с сервером политики сети.  На этом этапе взаимную проверку подлинности на NPS-сервер отправляет свой сертификат сервера на клиентском компьютере, чтобы клиент может проверять удостоверения сервера политики сети с помощью сертификата. Для успешной проверки подлинности сервера политики сети, клиентский компьютер должен доверять ЦС, выдавшего сертификат сервера политики сети. Клиент доверяет этот ЦС при наличии сертификата ЦС в хранилище сертификатов доверенных корневых центров сертификации на клиентском компьютере.

    При развертывании частного ЦС, сертификат ЦС автоматически устанавливается в хранилище сертификатов доверенных корневых центров сертификации для текущего пользователя и для локального компьютера, при обновлении групповой политики на клиентском компьютере члена домена. Если вы решили развертывание сертификатов сервера из общедоступного центра сертификации, убедитесь, что открытый сертификат ЦС уже находится в хранилище сертификатов доверенных корневых центров сертификации.  

2.  Сервер политики сети выполняет проверку подлинности пользователя. После клиент успешно проходит проверку подлинности с сервером политики сети, клиент отправляет на основе password\ учетные данные пользователя на сервере политики сети, который проверяет учетные данные пользователя с базой данных учетных записей пользователей в доменных службах Active Directory \(AD DS\).

Если учетные данные действительны и проверка подлинности проходит успешно, на NPS-сервер начинает фазы авторизации обработки запроса на подключение. Если учетные данные не являются допустимыми и происходит сбой проверки подлинности, NPS отправляет сообщение отклонить доступ и подключение запрос отклоняется.  

#### <a name="authorization"></a>Авторизация

NPS-сервера выполняет авторизации следующим образом:  

1. Сервер политики сети проверяет ограничения в пользователя или компьютера свойства учетной записи в dial\ в Доменных службах Active Directory. Каждой учетной записи пользователя и компьютера в Active Directory-пользователи и компьютеры включает несколько свойств, в том числе на **Dial\ в** вкладку. На этой вкладке в **права доступа к сети**, если значение равно **разрешить доступ**, пользователя или компьютера разрешено подключаться к сети. Если значение равно **запрещать доступ**, пользователя или компьютера не разрешено подключаться к сети. Если значение равно **управление на основе политики сети**, сервер политики сети оценивает политики сети, настроенные для определения пользователя или компьютера, разрешено ли подключение к сети. 

2. Сервер политики сети обрабатывает его политики сети, чтобы найти политику, которая соответствует запрос на подключение. Если найден соответствующей политики NPS предоставляет или запрещает подключения, в зависимости от настройки этой политики.  

При успешной проверки подлинности и авторизации, если соответствующей сетевой политики предоставил доступ, NPS предоставляет доступ к сети, и пользователь и компьютер может подключаться к сетевым ресурсам, для которых у них есть разрешения.  

>[!NOTE]
>Для развертывания беспроводного доступа, необходимо настроить политики NPS. Это руководство содержит инструкции для использования **мастера настройки 802.1 X** на сервере политики сети, чтобы создать NPS политики беспроводного доступа с проверкой подлинности 802.1 X.  

### <a name="bootstrap-profiles"></a>Загрузочный профилей
В 802.1X\-беспроводных сетей с проверкой подлинности беспроводных клиентов необходимо предоставить учетные данные безопасности, которые выполняют проверку подлинности RADIUS-сервером для подключения к сети. Для защищенного EAP \[PEAP\]\-Microsoft протокола CHAP версии 2 \[MS\-CHAP v2\], учетные данные безопасности, имя пользователя и пароль. Для \[TLS\ EAP\ Transport Layer Security] или PEAP\-TLS, учетные данные безопасности, сертификатов, например, сертификаты пользователей и компьютеров клиентов или смарт-карты.

При подключении к сети, которая по умолчанию настроен для выполнения PEAP\-MS\-CHAP v2, PEAP\ TLS или проверки подлинности EAP\ TLS, беспроводные клиенты Windows необходимо также проверить сертификат компьютера, который отправляется сервером RADIUS. Сертификат компьютера, который отправляется сервером RADIUS для каждого сеанса проверки подлинности часто называют сертификат сервера.

Как упоминалось ранее, можно выдавать серверы RADIUS свой сертификат сервера в одном из двух способов: в коммерческом центре сертификации \ (такие как VeriSign, Inc., \), или в частных ЦС, развертываемыми в вашей сети. Если RADIUS-сервер отправляет сертификат компьютера, выданный в коммерческом центре сертификации, который уже установлен корневой сертификат в хранилище сертификатов доверенных корневых центров сертификации клиента, клиент беспроводной сети может проверять сертификат сервера RADIUS, независимо от того, является ли беспроводной клиент присоединен к домену Active Directory. В этом случае клиент беспроводной сети можно подключиться к беспроводной сети, а затем присоединить компьютер к домену.

>[!NOTE]
>Можно отключить поведение, требуя от клиента для проверки сертификата сервера, но в производственной среде не рекомендуется отключать проверку сертификата сервера.

Загрузочный профилей беспроводной связи, временные профили, которые настроены таким образом, чтобы разрешить пользователям подключаться к 802.1X\ клиент беспроводной сети-проверкой подлинности беспроводной сети, прежде чем компьютер присоединен к домену, and\ или или до пользователь успешно вошел в систему в домен с помощью данного беспроводной компьютера в первый раз.  В этом разделе перечислены какие произошла ошибка при попытке присоединиться к беспроводной сети компьютер к домену, или пользователь может использовать domain\ беспроводной сети компьютерах, объединенных в первый раз для входа в домен.

Для развертываний, в которых пользователь или ИТ-администратор не удается подключить физически компьютер к проводной сети Ethernet, чтобы присоединить компьютер к домену и на компьютере нет от необходимости выдачи корневой сертификат ЦС, установленный в его **доверенные корневые центры сертификации** хранилище сертификатов, можно настроить беспроводные клиенты профиля временный беспроводной связи, называется *загрузки профиля* , чтобы подключиться к беспроводной сети.

A *загрузки профиля* устраняет необходимость проверять сертификат сервера RADIUS. Благодаря такой временный конфигурации пользователя беспроводной сети присоединить компьютер к домену, в какое время беспроводной сети \ (IEEE 802.11\) применяются политики, и соответствующие корневые ЦС сертификат автоматически устанавливается на компьютере.

При применении групповой политики, профили беспроводных подключений, которые сделать обязательным для взаимной проверки подлинности применяются на компьютере. загрузочный профиль больше не требуется и удаляется. После присоединения компьютера к домену и перезагрузки компьютера, пользователь может использовать беспроводного подключения для входа в домен.

Обзор процесса развертывания беспроводного доступа с помощью этих технологий см. в разделе [Обзор развертывания беспроводного доступа](b-wireless-access-deploy-overview.md).
