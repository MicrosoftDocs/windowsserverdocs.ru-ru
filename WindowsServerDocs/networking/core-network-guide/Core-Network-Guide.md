## <a name="BKMK_about"></a>Об этом руководстве
Данное руководство предназначено для администраторов сети и системы, которые устанавливают новую сеть или которым необходимо создать доменной сети, вместо сети, состоящей из рабочих групп. Сценарий развертывания, описанный в этом руководстве, особенно полезен в том случае, если предвидится необходимость добавления служб и компонентов сети в будущем.

Рекомендуется ознакомиться проектированию и руководства по развертыванию для каждой из технологий, используемых в данном сценарии развертывания, чтобы определить, предоставляет ли в этом руководстве по службам и конфигурации, необходимые.

Основная сеть — это набор сетевого оборудования, устройств и программное обеспечение, которое обеспечивает работу основных служб для вашей организации информационных технологий (ИТ).

Основная сеть Windows Server предоставляет много преимуществ, включая следующие.

-   Основные протоколы для подключения к сети между компьютерами и других совместимых устройств протокола (TCP/IP). TCP/IP — это набор стандартных протоколов для соединения компьютеров и построения сетей. TCP/IP — программном обеспечении сетевого протокола с операционными системами Microsoft Windows, реализующий и поддерживает набор протоколов TCP/IP.

-   Динамический протокол настройки узла (DHCP) автоматическое назначение IP-адресов для компьютеров и других устройств, которые настроены как DHCP-клиентов. Ручная настройка IP-адресов на всех компьютерах сети отнимает время и обладает меньшей гибкостью, чем динамическое предоставление компьютерам и другим устройствам конфигурацию IP-адресов с помощью DHCP-сервера.

-   Служба разрешения имен доменных имен (DNS). Служба DNS предоставляет пользователям, компьютерам, приложения и службы найти IP-адресов компьютеров и устройств в сети, используя полное доменное имя компьютера или устройства.

-   Лес, который является один или несколько доменов Active Directory, которые совместно используют же определения классов и атрибутов (схему), сведения о сайте и репликации (конфигурацию) и возможности поиска по всему лесу (глобальный каталог).

-   Корневой домен леса, — это первый домен, созданные в новом лесу. В корневом домене леса находятся группы "Администраторы предприятия" и "Администраторы схемы", что группы администраторов уровня леса. Кроме того, корневой домен леса, как и другие домены, представляет собой набор компьютеров, пользователей и группы объектов, определенных администраторами в доменных службах Active Directory (AD DS). Эти объекты используют общую базу данных и безопасности политики каталога. Они могут также совместно использовать отношения безопасности с другими доменами если последние добавляются по мере роста организации. Служба каталогов также хранит данные каталогов и позволяет авторизованным компьютерам, приложениям и пользователям доступ к данным.

-   Базе учетной записи пользователя и компьютера. Служба каталогов представляет базу данных учетных записей централизованного пользователя, позволяет создавать учетные записи пользователей и компьютеров, пользователей и компьютеров, которым разрешено подключаться к сети и получать доступ к сетевым ресурсам, таким как приложения, базы данных, общие файлы и папки и принтеры.

Основной сети также дает возможность масштабировать сеть по мере роста предприятия и изменения требований к ИТ. Например с помощью основной сети можно добавить домены, IP-подсети, службы удаленного доступа, беспроводные службы и других компонентов и ролей сервера, предоставляемые Windows Server 2016.

### <a name="network-hardware-requirements"></a>Требования к сетевому оборудованию
Чтобы успешно развернуть основную сеть, необходимо развернуть сетевое оборудование, включая следующие:

-   Ethernet, Fast Ethernet или Gigabit Ethernet-кабеля

-   Концентратор, уровня 2 или 3 коммутатора, маршрутизатор или другое устройство, которое выполняет функции ретрансляции сетевого трафика между компьютерами и устройствами.

-   Компьютеры, которые удовлетворяют минимальным требованиям к оборудованию для соответствующих клиентских и серверных операционных систем.

## <a name="technology-overviews"></a>Обзор технологий
Ниже приведен краткий обзор обязательных технологий, которые развертываются для создания основной сети.

### <a name="active-directory-domain-services"></a>Доменные службы Active Directory
Каталог представляет собой иерархическую структуру, содержащую информацию об объектах в сети, таких как пользователи и компьютеры. Служба каталогов, таких как AD DS предоставляет методы для хранения данных каталога и предоставления этих данных пользователям и администраторам сети. Например AD DS хранит сведения об учетных записях пользователей, включая имена, адреса электронной почты, пароли и номеров телефонов и позволяет других авторизованным пользователям в одной сети для доступа к этой информации.

### <a name="dns"></a>DNS-СЕРВЕРА
DNS — это протокол разрешения имен для сетей TCP/IP, таких как Интернет или сети организации. DNS-сервер содержит сведения, позволяющие клиентским компьютерам и легко запоминаемые буквенно-цифровые DNS-имена в IP-адреса, которые компьютеры используют для связи друг с другом.

### <a name="dhcp"></a>DHCP
Протокол DHCP — это стандарт IP для упрощения управления IP-конфигурацией узлов. Стандарт DHCP предусматривает использование DHCP-серверов для управления динамическим выделением IP-адресов и других сопутствующих сведений о настройке DHCP-клиентов в сети.

Протокол DHCP позволяет использовать DHCP-сервера для динамического назначения IP-адрес компьютера или другого устройства, такие как принтер, в вашей локальной сети. Каждый компьютер в сети TCP/IP должен иметь уникальный IP-адрес, IP-адрес и его с маской подсети однозначно определяет главный компьютер и подсеть, к которой подключен компьютер. С помощью DHCP, убедитесь, что все компьютеры, которые настроены как DHCP-клиенты получают IP-адрес, который подходит для данной подсети и сетевого расположения и параметры DHCP, как шлюз по умолчанию и DNS-серверов можно автоматически предоставлять DHCP-клиентов с помощью информации, необходимой для правильной работы в сети.

Для сетей на основе TCP/IP DHCP снижает сложность и количество административной работы, которая требуется для перенастройки компьютеров.

### <a name="tcpip"></a>TCP/IP
TCP/IP в Windows Server 2016 выглядит следующим образом:

-   Сетевые программного обеспечения на основе стандартных сетевых протоколов.

-   Маршрутизируемый протокол корпоративных сетей, обеспечивающий подключение компьютера под управлением Windows к локальной сети (LAN) и сред глобальной сети (WAN).

-   Базовые технологии и служебные программы для соединения компьютеров под управлением Windows с разнородными системами для обмена данными.

-   Основа доступа к глобальным службам Интернета, таких как серверы Интернета и передачи файлов (FTP).

-   Это платформа надежной, масштабируемая кроссплатформенная система клиент сервер.

TCP/IP обеспечивает основные служебные программы TCP/IP, которые позволяют компьютерам под управлением Windows для подключения и обмениваться сведениями с другими корпорации Майкрософт и сторонних систем, включая:

-    Windows Server 2012

-    Windows 7

-    Windows Server 2008

-   Windows Vista

-   Узлы в Интернете

-   Системы Apple Macintosh

-   Мэйнфреймы IBM

-   Системы UNIX и Linux

-   Системы Open VMS

-   Сетевые принтеры

-   Планшеты и телефоны сотовой сети с проводной сети Ethernet и беспроводной сети стандарта 802.11 технологию

### <a name="core-network-components"></a>Компоненты основной сети
Ниже приведено описание компонентов основной сети.

##### <a name="router"></a>Маршрутизатор
Это руководство по развертыванию приведены инструкции по развертыванию основной сети с двумя подсетями, разделенными маршрутизатором, на котором включена пересылка пакетов DHCP. Тем не менее, можно развернуть коммутатор уровня 2, коммутатор уровня 3 или центр, в зависимости от требований и доступных ресурсов. При применении коммутатора последний должен обладать возможностью пересылки пакетов DHCP или DHCP-сервера необходимо поместить в каждой из подсетей. При применении концентратора выполняется развертывание одной подсети и необходимость в пересылке DHCP или использовании второй области на DHCP-сервера.

##### <a name="static-tcpip-configurations"></a>Статические конфигурации TCP/IP
Серверы в этом развертывании настроены статические IPv4-адреса. Клиентские компьютеры настроены по умолчанию для получения IP-адресов в аренду от DHCP-сервера.

##### <a name="active-directory-domain-services-global-catalog-and-dns-server-dc1"></a>Глобальный каталог Active Directory доменных служб и DNS-сервер DC1
Оба установлены на этом сервере с именем DC1, который предоставляет каталог доменных служб Active Directory (AD DS) и доменных имен (DNS) и разрешение имен для всех компьютеров и устройств в сети.

##### <a name="dhcp-server-dhcp1"></a>DHCP-сервер DHCP1
DHCP-сервер, с именем DHCP1 настроена область, предоставляющая в аренду адреса протокола Интернета (IP) для компьютеров в локальной подсети. DHCP-сервер можно также настроить дополнительные области, чтобы обеспечивать аренду IP-адресов для компьютеров в других подсетях, если на маршрутизаторах настроена пересылка пакетов DHCP.

##### <a name="client-computers"></a>Клиентские компьютеры
Компьютеры под управлением клиентских операционных систем Windows по умолчанию настроены как DHCP-клиенты, которые автоматически получают IP-адреса и параметры DHCP от DHCP-сервера.

### <a name="bkmk_NetFndtn_Pln_Subnt"></a>Планирование подсетей
Маршрутизаторы в сети протокола (TCP/IP), используются для соединения оборудования и программного обеспечения, используемого в разных сегментах физической сети называемых подсетями. Маршрутизаторы также используются для пересылки IP-пакетов между различными подсетями. Определите физическую структуру сети, включая число маршрутизаторах и подсетях, прежде чем приступить к выполнению инструкций данного руководства.

Кроме того чтобы настроить серверы в сети статических IP-адресов, необходимо определить диапазон IP-адресов, который вы хотите использовать для подсети, где находятся серверы основной сети. В этом руководстве частных IP-адресов, диапазонов адресов 10.0.0.1 - 10.0.0.254 и 10.0.1.1 - 10.0.1.254 используются в качестве примера, но можно использовать любой закрытый диапазон IP-адресов, который вы предпочитаете.

> [!IMPORTANT]
> После выбора диапазоны IP-адресов, которые вы хотите использовать для каждой подсети убедитесь, настройте маршрутизаторы IP-адрес из того же диапазон IP-адресов, который использовался в подсети, в которой маршрутизатор установлен. Например если маршрутизатор имеет по умолчанию IP-адрес 192.168.1.1 и установлен, но вы устанавливаете маршрутизатор подсети, диапазон IP-адресов 10.0.0.0/24, необходимо перенастроить маршрутизатора, чтобы использовать IP-адрес из диапазона IP-адресов 10.0.0.0/24.

Следующие распознать частных IP-адресов, диапазоны определены Internet запроса для документом RFC 1918.

-   10.0.0.0 - 10.255.255.255

-   172.16.0.0 - 172.31.255.255

-   192.168.0.0 - 192.168.255.255

При использовании диапазонов частных IP-адресов, определенных в документе RFC 1918, не удается подключиться к Интернету, используя частный IP-адрес, поскольку маршрутизаторы поставщика услуг Интернета автоматически отклоняют запросы, поступающие на эти адреса или с напрямую. Более поздней версии, чтобы добавить подключение к Интернету в основной сети необходимо обратиться к поставщика услуг Интернета для получения общего IP-адреса.

> [!IMPORTANT]
> При использовании частных IP-адресов, необходимо использовать какого-либо прокси-сервера или сетевого адресов (NAT) сервер для преобразования диапазонов частных IP-адресов в локальной сети общедоступный IP-адрес, который можно передать в Интернете. Большинство маршрутизаторов предоставляют службу NAT, поэтому выбор маршрутизатора не составит особого труда.

Дополнительные сведения см. в разделе [Планирование развертывания DHCP1](#bkmk_NetFndtn_Pln_DHCP-01).

#### <a name="planning-naming-conventions-for-computers-and-devices"></a>Планирование контекстов именования для компьютеров и устройств
Для единообразия в сети рекомендуется использовать согласованные имена для серверов, принтеров и других устройств. Имена компьютеров можно использовать, чтобы помочь пользователям и администраторам задачу определения назначения и расположения сервера, принтер или другое устройство. Например, при наличии трех DNS-серверов, в Сан-Франциско, в Лос-Анджелесе и в Чикаго можно использовать соглашение об именовании *функции server*-*расположение*-*номер*:

-   DNS-КАБИНЕТ-01. Данное имя представляет DNS-сервер в Денвере, штат Колорадо. Если DNS-серверы будут добавлены в Денвере, можно увеличить числовое значение в имени, например DNS-SF-02 и DNS-SF-03.

-   DNS-SPAS-01. Данное имя представляет DNS-сервер в Южной Пасадене, штат Калифорния.

-   DNS-ORL-01. Данное имя представляет DNS-сервер в Орландо, штат Флорида.

#### <a name="planning-dns-zones"></a>Планирование зон DNS
На DNS-серверах первичной, интегрированный с Active Directory зона прямого просмотра создается по умолчанию во время установки роли DNS-сервера. Зона прямого просмотра позволяет компьютерам и устройствам запрашивать другого компьютера или устройства IP-адрес, на основе имени DNS. В дополнение к зоне прямого просмотра рекомендуется создать зону обратного просмотра DNS. С помощью DNS обратного просмотра, запроса, компьютера или устройства могут узнавать имя другого компьютера или устройства с использованием его IP-адрес. Развертывание зоны обратного просмотра обычно повышает эффективность работы службы DNS и существенно повышает успешность выполнения запросов DNS.

При создании зоны обратного просмотра домен in-addr.arpa, определенный в стандартах DNS и зарезервированный в пространстве имен DNS Интернета, чтобы обеспечить удобный и надежный способ выполнения обратных запросов, настроен в DNS. Чтобы создать обратное пространство имен, поддомены в домене in-addr.arpa формируются с использованием обратного порядка разделяемых точками десятичных чисел, IP-адресов в.

Домен in-addr.arpa применяется ко всем сетям TCP/IP, которые основаны на протокол IP версии 4 (IPv4) адресации. Мастер создания зоны автоматически предполагает, что этот домен используется при создании новой зоны обратного просмотра.

При запуске мастера создания новой зоны рекомендуется использовать следующие варианты:

|Элементы конфигурации|Примеры значений|
|-----------------------|------------------|
|Тип зоны|**Основная зона**, и **хранить зону в Active Directory** выбран|
|Область репликации зоны Active Directory|**Для всех DNS-серверов в этом домене**|
|Страница мастера имени зоны обратного просмотра первого|**Зона обратного просмотра IPv4**|
|Страница мастера имени второго зоны обратного просмотра|Идентификатор сети = 10.0.0.|
|Динамические обновления|**Разрешить только безопасные динамические обновления**|

### <a name="bkmk_NetFndtn_Pln_DomAccess"></a>Планирование доступа к домену
Чтобы войти в домен, то компьютеру необходима компьютера входит в домен и учетная запись пользователя должна создаваться в Доменных службах Active Directory, до попытки входа в систему.

> [!NOTE]
> Отдельные компьютеры под управлением Windows содержат локальный пользователей и групп учетной записи пользователя базы данных, который называется базы данных учетных записей пользователей диспетчера учетных записей безопасности (SAM). При создании учетной записи пользователя на локальном компьютере в базе данных SAM на локальном компьютере можно выполнить вход, но не смогут войти в домен. Учетные записи пользователей домена создаются с помощью Active Directory-пользователи и компьютеры Microsoft Management Console (MMC) на контроллере домена, а не с локальных пользователей и групп на локальном компьютере.

После первого успешного входа с учетными данными для входа в систему домена параметры входа существуют, если компьютер удаляется из домена или параметры входа изменяются вручную.

Прежде чем вы войдите в домен:

-   Создайте учетные записи пользователей в Active Directory-пользователи и компьютеры. Каждый пользователь должен иметь учетную запись пользователя доменных служб Active Directory в Active Directory-пользователи и компьютеры. Дополнительные сведения см. в разделе [Создание учетной записи пользователя в Active Directory-пользователи и компьютеры](#BKMK_createUA).

-   Проверьте правильность конфигурации IP адрес. Чтобы присоединить компьютер к домену, компьютер должен иметь IP-адрес. В данном руководстве у серверов настроены статические IP-адреса и клиентские компьютеры получают IP-адреса в аренду от DHCP-сервера. По этой причине необходимо развернуть DHCP-сервера, перед присоединением клиентов к домену. Дополнительные сведения см. в разделе [развертывание DHCP1](#BKMK_deployDHCP01).

-   Присоедините компьютер к домену. Любой компьютер, предоставляющий или получающий доступ к ресурсам сети должен быть присоединен к домену. Дополнительные сведения см. в разделе [Присоединение серверов к домену и вход](#BKMK_joinlogserver) и [присоединение клиентских компьютеров к домену и вход](#BKMK_joinlogclients).

### <a name="bkmk_NetFndtn_Pln_DHCP-01"></a>Планирование развертывания DHCP1
Ниже приведены ключевые действия по планированию перед установкой роли DHCP-сервера на DHCP1.

#### <a name="planning-dhcp-servers-and-dhcp-forwarding"></a>Планирование DHCP-серверов и пересылки DHCP
Поскольку сообщения DHCP являются широковещательными, не пересылают их между подсетями маршрутизаторами. Если имеется несколько подсетей и службу DHCP для каждой подсети требуется обеспечить, необходимо выполнить одно из следующих:

-   Установка DHCP-сервера в каждой подсети

-   Настройте маршрутизаторы на пересылку широковещательных сообщений DHCP между подсетями, а на DHCP-сервере, один на подсеть настройте несколько областей.

В большинстве случаев настройка маршрутизаторов на пересылку широковещательных сообщений DHCP является более экономически эффективным решением, чем развертывание DHCP-сервера на каждом из физических сегментов сети.

#### <a name="planning-ip-address-ranges"></a>Планирование диапазонов IP-адресов
У каждой подсети должен быть собственный уникальный диапазон IP-адресов. Эти диапазоны представлены на DHCP-сервере с помощью областей.

Область представляет собой группировку IP-адреса для компьютеров подсети, которые используют службу DHCP администрирования. Администратор сначала создает область для каждой физической подсети, а затем использует ее для определения параметров клиентских компьютеров.

Область имеет следующие свойства:

-   Диапазон IP-адресов, из которого следует включить или исключить адресов, используемых в предложениях на выделение адреса службы DHCP.

-   Маска подсети, определяющая префикс подсети для конкретного IP-адреса.

-   Имя области, назначаемое при ее создании.

-   Сроки действия аренды адреса, которые назначаются DHCP-клиентам, получающим динамически выделяемые IP-адреса.

-   Любые параметры области DHCP, настроенные для назначения DHCP-клиентам, такие как IP-адрес сервера DNS и IP-адрес маршрутизатора или основного шлюза.

-   Зарезервированные адреса используются, при необходимости, чтобы гарантировать получение DHCP-клиентом одного IP-адреса.

Перед развертыванием серверов, создайте список подсетей и диапазонов IP-адрес, которые вы хотите использовать для каждой подсети.

#### <a name="planning-subnet-masks"></a>Планирование масок подсетей
С помощью маски подсети различения идентификаторов сетей и узла в IP-адрес. Каждая маска подсети — 32-разрядное число, в котором последовательных единичных битов (1), идентифицирующего сеть, код и группа последовательных нулевых битов (0), для обозначения адреса узла IP-адреса.

Например с IP-адресом 131.107.16.200 обычно используется маска подсети является следующее 32-разрядное двоичное число:

```
11111111 11111111 00000000 00000000
```

Это число маски подсети — 16 единичных битов и 16 нулевых битов, указывающее, что сетевой идентификатор и узла идентификатор разделы этот IP-адрес обе длиной по 16 бит. Обычно маска подсети отображается как 255.255.0.0 в десятичной записи.

В следующей таблице показаны маски подсетей для классов адресов Интернета.

|Класс адреса|Количество битов для маски подсети|Маска подсети|
|-----------------|------------------------|---------------|
|Класса A|11111111 00000000 00000000 00000000|255.0.0.0|
|Класс B|11111111 11111111 00000000 00000000|255.255.0.0|
|Класс C|11111111 11111111 11111111 00000000|255.255.255.0|

При создании области в DHCP и вводе диапазона IP-адресов для области, служба DHCP обеспечивает эти значения масок подсетей по умолчанию. Обычно значения масок подсетей по умолчанию применимы для большинства сетей, не имеющих особых требований, а также там, где каждый сегмент IP-сети соответствует одной физической сети.

В некоторых случаях можно использовать специальные маски подсети для реализации IP-подсети. С помощью IP-подсети, можно разделить, по умолчанию часть узла IP-адрес подсети, которые являются подразделениями исходного идентификатора сети на основе класса.

Настройкой длины маски подсети можно уменьшить количество бит, необходимое для адреса узла.

Чтобы избежать проблем с адресацией и маршрутизацией, должны убедиться, что все компьютеры TCP/IP в сегменте сети, используют одинаковую маску подсети и что каждого устройства имеется уникальный IP-адрес.

#### <a name="planning-exclusion-ranges"></a>Планирование диапазонов исключения
При создании области на DHCP-сервере, можно указать диапазон IP-адресов, включающий все адреса, которые разрешено арендовать DHCP-клиентам, такие как компьютерам и другим устройствам DHCP-сервера. Если перейти и вручную настроить несколько серверов и других устройств с статического IP-адреса из того же IP-адресов, используя DHCP-сервера, можно непреднамеренно создать конфликт IP-адресов, где вы и DHCP-сервер назначите же IP-адрес для различных устройств.

Чтобы решить эту проблему, можно создать диапазон исключения для области DHCP. Диапазон исключения является непрерывным диапазон IP-адресов внутри области диапазон IP-адресов DHCP-сервера не разрешено использовать. Если создать диапазон исключения DHCP-сервер не назначает адреса в диапазоне, что позволяет вручную назначать без создания конфликт IP-адресов.

IP-адреса можно исключить из распределения DHCP-сервером, путем создания диапазона исключения для каждой области. Исключения необходимо использовать для всех устройств, для которых настроены статические IP-адрес. Исключаемые адреса должны включать все IP-адреса, назначенные вручную на другие серверы, не DHCP-клиенты, бездисковым рабочим станциям или клиентам маршрутизации и удаленного доступа и PPP.

Рекомендуется настроить диапазона исключения дополнительные адреса, чтобы предусмотреть рост сети в будущем. В следующей таблице приведены диапазон исключения для области с диапазоном IP-адрес 10.0.0.1 - 10.0.0.254 и маской подсети 255.255.255.0.

|Элементы конфигурации|Примеры значений|
|-----------------------|------------------|
|Запустите IP-адрес диапазона исключения|10.0.0.1|
|Конечный IP-адрес диапазона исключения|10.0.0.25|

#### <a name="planning-tcpip-static-configuration"></a>Планирование статической конфигурации TCP/IP
Некоторые устройства, например маршрутизаторы, серверы DHCP и DNS-серверы должны быть настроены статические IP-адрес. Кроме того возможно, и другим дополнительным устройствам, таким как принтеры, которые вы хотите обеспечить всегда же IP-адрес. Список устройств, которые необходимо настроить статически для каждой подсети, а затем запланируйте диапазон исключения, который вы хотите использовать на DHCP-сервере, чтобы убедиться, что DHCP-сервер не предоставляет в аренду IP-адрес статически настроенных устройств. Диапазон исключения — это ограниченная последовательность IP-адресов внутри области, исключаемых из числа DHCP службы. Диапазонов исключения не назначает адреса из этих диапазонов сервером DHCP-клиентам в сети.

Например если диапазон IP-адресов для подсети используется 192.168.0.1–192.168.0.254 и имеется десять устройств, которые вы хотите настроить статический IP-адрес, можно создать диапазон исключения для области 192.168.0. *x* область десятью и более IP-адресами: 192.168.0.1–192.168.0.15.

В этом примере десять из исключенные IP-адреса используется для настройки статических IP-адресов серверов и других устройств, а пять дополнительных IP-адресов оставлены про запас для статической настройки новых устройств, которые могут быть добавлены в будущем. С помощью этого диапазона исключения DHCP-сервера оставлен пул адресов 192.168.0.16 по 192.168.0.254.

#### <a name="BKMK_ip"></a>Настройка статического IP-адреса
Настройка протокола IP версии 4 (IPv4), можно использовать процедуры, описанные в этом разделе Свойства сетевого подключения статический IP-адрес компьютерам под управлением Windows Server 2016.

> [!NOTE]
> Чтобы выполнить эту процедуру с помощью Windows PowerShell, откройте PowerShell и введите следующие командлеты в отдельных строках и нажмите клавишу ВВОД. Также необходимо переименовать интерфейсы и IP-адреса в этом примере со значениями, которые вы хотите использовать для настройки компьютера.

###### <a name="to-configure-a-static-ip-address-on-computers-running-windows-server-2016-windows-server-2012-r2-and-windows-server-2012"></a>Настройка статического IP-адреса на компьютерах под управлением Windows Server 2016, Windows Server 2012 R2 и Windows Server 2012

1.  На панели задач щелкните правой кнопкой мыши значок сети и нажмите кнопку **открыть управления сетями и общим доступом**.

2.  В **сетями и общим доступом**, нажмите кнопку **изменение параметров адаптера**. **Сетевых подключений** папку открывает и отображает доступные сетевые подключения.

3.  В **сетевых подключений**, щелкните правой кнопкой мыши подключение, которое вы хотите настроить, а затем нажмите кнопку **свойства**. Сетевое подключение **свойства** откроется диалоговое окно.

4.  В сети **свойства** диалогового окна **отмеченные компоненты используются этим подключением**, выберите **Internet IP версии 4 (TCP/IPv4)**, а затем нажмите кнопку **свойства**. **Internet IP версия 4 (TCP/IPv4)** откроется диалоговое окно.

5.  В **Internet IP версия 4 (TCP/IPv4)**на **Общие** щелкните **использовать следующий IP-адрес**. В **IP-адрес**, введите IP-адрес, который вы хотите использовать.

6.  Нажмите клавишу tab, чтобы переместить курсор в **маска подсети**. Значение маски подсети по умолчанию будет введено автоматически. Либо примите маску подсети по умолчанию, либо введите маску подсети, который вы хотите использовать.

7.  В **шлюз по умолчанию**, введите IP-адрес шлюза по умолчанию.

    > [!NOTE]
    > Необходимо настроить **шлюз по умолчанию** же IP-адрес, который использовался в интерфейсе локальной сети (LAN) маршрутизатора. Например, если имеется маршрутизатор, который подключен к глобальной сети (WAN) Интернета, а также к локальной настроить интерфейсе локальной сети с помощью того же IP-адрес, который будет затем определен как **шлюз по умолчанию**. В другом примере если имеется маршрутизатор, который подключен к два локальных сетях, где сети LAN A используется диапазон адресов 10.0.0.0/24, а LAN B использует 192.168.0.0/24 диапазон адресов, настройте IP-адрес маршрутизатора LAN A с адресом из этого диапазона адресов, например 10.0.0.1. Кроме того, в области DHCP для данного диапазона адресов, настройте **шлюз по умолчанию** IP-адрес 10.0.0.1. Для LAN B интерфейса маршрутизатора LAN B с адресом из этого диапазона адресов, например 192.168.0.1, а затем настройте 192.168.0.0/24 области LAN B с **шлюз по умолчанию** IP-адресом 192.168.0.1.

8.  В **предпочитаемый DNS-сервер**, введите IP-адрес вашего DNS-сервера. Если планируется использовать локальный компьютер в качестве предпочтительного DNS-сервера, введите IP-адрес локального компьютера.

9. В **Альтернативный DNS-сервер**, введите IP-адрес альтернативного DNS-сервера, если есть. Если планируется использовать локальный компьютер как альтернативный DNS-сервер, введите IP-адрес локального компьютера.

10. Нажмите кнопку **ОК**, а затем нажмите кнопку **закрыть**.

> [!NOTE]
> Сведения о настройке статического IP-адреса на компьютерах, работающих под управлением других операционных систем Майкрософт см. в разделе [приложение B — Настройка статических IP-адресов](#BKMK_B).

Это упростит для вас в качестве администратора для всех сетевых ресурсов. Кроме того, рекомендуется вход с этой учетной записи только в том случае, если необходимо выполнять административные задачи и создать отдельную учетную запись пользователя для выполнения без ИТ связанных задач.

**Различия между учетными записями пользователей доменов и учетные записи пользователей на локальном компьютере**

Одним из преимуществ доменной инфраструктуры является, что не нужно создавать учетные записи пользователей на каждом компьютере в домене. Это справедливо ли компьютер находится на клиентском компьютере или сервере.

По этой причине не следует создать учетные записи пользователей на каждом компьютере в домене. Создайте все учетные записи пользователей в Active Directory-пользователи и компьютеры и используйте описанные выше процедуры для назначения членства в группах. По умолчанию все учетные записи пользователей являются членами группы пользователей домена.

Все члены группы "Пользователи домена" может входить на любой клиентский компьютер после присоединен к домену.

Вы можете настроить учетные записи пользователей можно определить дни и время, пользователь может войти на компьютер. Можно также указать, какие компьютеры, каждый пользователь может использовать. Чтобы настроить эти параметры, откройте Active Directory — пользователи и компьютеры, найдите учетную запись пользователя, который вы хотите настроить и дважды щелкните учетную запись. В учетной записи пользователя **свойства**, нажмите кнопку **учетной записи** , а затем щелкните одно **часы входа** или **вход в**.

9. В **параметры контроллера домена**в **выберите режим работы нового леса и корневым доменом**, выберите режим работы леса и режим работы домена, который вы хотите использовать. В **укажите возможности контроллера домена**, убедитесь, что **сервера доменных имен (DNS)** и **глобальный каталог (GC)** выбраны. В **пароль** и **подтверждение пароля**, введите пароль режима восстановления служб каталогов (DSRM), который вы хотите использовать. Нажмите кнопку **Далее**.

10. В **параметры DNS**, нажмите кнопку **Далее**.

11. В **Дополнительные параметры**, убедитесь, что имя NetBIOS, назначенное домену и только при необходимости измените его. Нажмите кнопку **Далее**.

12. В **пути**в **указать расположение базы данных AD DS, файлов журналов и SYSVOL**, выполните одно из следующих действий:

    -   Примите значения по умолчанию.

    -   Введите расположения папок, которые вы хотите использовать для **папки базы данных**, **папки файлов журнала**, и **папки SYSVOL**.

13. Нажмите кнопку **Далее**.

Когда появится запрос Windows, щелкните **закрыть**. Для полной установки AD DS и DNS-сервер перезагрузится.

16. Для проверки успешной установки представления консоли диспетчера сервера, после перезагрузки сервера. В левой области, как выделенные элементы на изображении ниже должна появиться AD DS и DNS.

##### <a name="BKMK_createUA"></a>Создание учетной записи пользователя в Active Directory-пользователи и компьютеры
Эту процедуру можно использовать для создания новой учетной записи пользователя домена в Active Directory-пользователи и компьютеры Microsoft Management Console (MMC).

Членство в группе **"Администраторы домена"**, или в эквивалентной минимальным требованием для выполнения этой процедуры.

> [!NOTE]
> Чтобы выполнить эту процедуру с помощью Windows PowerShell, откройте PowerShell и введите следующий командлет в одной строке и нажмите клавишу ВВОД. Кроме того, имя учетной записи пользователя в этом примере необходимо заменить значение, которое вы хотите использовать.
>
> `New-ADUser -SamAccountName User1 -AccountPassword (read-host "Set user password" -assecurestring) -name "User1" -enabled $true -PasswordNeverExpires $true -ChangePasswordAtLogon $false`
>
> После этого нажмите клавишу ВВОД, введите пароль для учетной записи пользователя. Учетная запись создается и по умолчанию будет входить в группу пользователей домена.
>
> С помощью следующего командлета можно назначить дополнительные группах для новой учетной записи пользователя. В этом примере пользователь User1 добавляется в группы "Администраторы домена" и "Администраторы предприятия". Убедитесь, перед выполнением этой команды, изменить имя учетной записи пользователя, имя домена и групп соответствуют вашим требованиям.
>
> `Add-ADPrincipalGroupMembership -Identity "CN=User1,CN=Users,DC=corp,DC=contoso,DC=com" -MemberOf "CN=Enterprise Admins,CN=Users,DC=corp,DC=contoso,DC=com","CN=Domain Admins,CN=Users,DC=corp,DC=contoso,DC=com"`

###### <a name="to-create-a-user-account"></a>Создание учетной записи пользователя

1.  На DC1 в диспетчере серверов щелкните **средства**, а затем нажмите кнопку **Active Directory-пользователи и компьютеры**. Откроется окно MMC Active Directory-пользователи и компьютеры. Если он еще не выбран, щелкните узел домена. Например, если ваш домен называется corp.contoso.com, нажмите кнопку **corp.contoso.com**.

2.  В области сведений щелкните правой кнопкой мыши папку, в которую следует добавить учетную запись пользователя.

    **Где?**

    -   Active Directory-пользователи и компьютеры или*узел домена*/*папки*

3.  Наведите курсор на **New**, а затем нажмите кнопку **пользователя**. **Новый объект — пользователь** откроется диалоговое окно.

4.  В **имя**, введите имя пользователя.

5.  В **инициалы**, введите инициалы пользователя.

6.  В **Фамилия**, введите фамилию пользователя.

7.  Изменение **полное имя** Чтобы добавить инициалы или поменять имя и фамилию местами.

8.  В **имя входа пользователя**, введите имя пользователя для входа. Нажмите кнопку **Далее**.

9. В **новый объект — пользователь**в **пароль** и **подтверждение пароля**, введите пароль пользователя и выберите нужные параметры пароля.

10. Нажмите кнопку **Далее**, проверьте параметры учетной записи нового пользователя и нажмите кнопку **Готово**.

Членство в группе **"Администраторы домена"**, или эквивалентной группе минимальным требованием для выполнения этой процедуры.

Если он еще не выбран, щелкните узел домена. Например, если ваш домен называется corp.contoso.com, нажмите кнопку **corp.contoso.com**.

##### <a name="BKMK_reverse"></a>Настройка зоны обратного просмотра DNS
Эту процедуру можно использовать для настройки зоны обратного просмотра в системе доменных имен (DNS).

Членство в группе **"Администраторы домена"** является минимальным требованием для выполнения этой процедуры.

> [!NOTE]
> -   Для средних и крупных предприятий рекомендуется настроить и использовать группу DNSAdmins в Active Directory-пользователи и компьютеры. Дополнительные сведения см. в разделе [Дополнительные технические источники](#BKMK_resources)
> -   Чтобы выполнить эту процедуру с помощью Windows PowerShell, откройте PowerShell и введите следующий командлет в одной строке и нажмите клавишу ВВОД. Кроме того, DNS обратного просмотра зоны и смените имена в этом примере необходимо заменить значения, которые вы хотите использовать. Убедитесь, что идентификатор сети в имени зоны обратного. Например, если идентификатор сети 192.168.0, создать имя зоны обратного просмотра **0.168.192.in-addr.arpa**.
>
> `Add-DnsServerPrimaryZone 0.0.10.in-addr.arpa -ZoneFile 0.0.10.in-addr.arpa.dns`

###### <a name="to-configure-a-dns-reverse-lookup-zone"></a>Настройка зоны обратного просмотра DNS

1.  На DC1 в диспетчере серверов щелкните **средства**, а затем нажмите кнопку **DNS**. Откроется оснастка консоли Управления DNS.

2.  В DNS если он еще не развернут, дважды щелкните имя сервера, чтобы развернуть дерево. Например, если имя DNS-сервера DC1, дважды щелкните **DC1**.

3.  Выберите **зоны обратного просмотра**, щелкните правой кнопкой мыши **зоны обратного просмотра**, а затем нажмите кнопку **новая зона**. Откроется мастер создания новой зоны.

4.  В **Добро пожаловать в мастер создания новой зоны**, нажмите кнопку **Далее**.

5.  В **тип зоны**выберите **основная зона**.

6.  Если DNS-сервер является контроллером домена с возможностью записи, убедитесь, что **хранить зону в Active Directory** выбран. Нажмите кнопку **Далее**.

7.  В **область репликации зоны в Active Directory**выберите **для всех DNS-серверов, работающих на контроллерах домена в этом домене**, если нет особых причин выбрать другой параметр. Нажмите кнопку **Далее**.

8.  В первом **имя зоны обратного просмотра** выберите **зона обратного просмотра IPv4**. Нажмите кнопку **Далее**.

9. Во второй **имя зоны обратного просмотра** выполните одно из следующих действий:

    -   В **идентификатор сети**, введите идентификатор сети диапазона IP-адресов. Например, если диапазон IP-адресов 10.0.0.1–10.0.0.254, введите **10.0.0**.

    -   В **имя зоны обратного просмотра**, автоматически добавляется имя зоны обратного просмотра IPv4. Нажмите кнопку **Далее**.

10. В **динамическое обновление**, выберите тип динамических обновлений, которые вы хотите разрешить. Нажмите кнопку **Далее**.

11. В **завершение работы мастера создания новой зоны**, проверьте выбранные параметры и нажмите кнопку **Готово**.

#### <a name="BKMK_joinlogserver"></a>Присоединение серверов к домену и вход
После установки доменных служб Active Directory (AD DS) и создания одного или нескольких учетных записей пользователей, имеющие разрешение на присоединение компьютера к домену серверы основной сети можно присоединить к домену и войти на серверы, чтобы установить дополнительные технологии, например DHCP Dynamic Host Configuration Protocol ().

На всех серверах, которые вы развертываете, за исключением сервера, на котором выполняется AD DS выполните следующие действия.

1.  Выполните процедуры, описанные в [Настройка всех серверов](#BKMK_configuringAll).

2.  Чтобы присоединить серверы к домену и войти на серверы для выполнения дополнительных задач развертывания, используйте инструкции из следующих процедур:

> [!NOTE]
> Чтобы выполнить эту процедуру с помощью Windows PowerShell, откройте PowerShell и введите следующий командлет и нажмите клавишу ВВОД. Кроме того, имя домена необходимо заменить имя, которое вы хотите использовать.
>
> `Add-Computer -DomainName corp.contoso.com`
>
> При появлении соответствующего запроса введите имя пользователя и пароль для учетной записи, имеющей разрешение на присоединение компьютера к домену. Чтобы перезагрузить компьютер, введите следующую команду и нажмите клавишу ВВОД.
>
> `Restart-Computer`

###### <a name="to-join-computers-running--windows-server-2016--windows-server-2012-r2--and--windows-server-2012--to-the-domain"></a>Присоединение компьютеров под управлением Windows Server 2016, Windows Server 2012 R2 и Windows Server 2012 в домене

1.  В диспетчере серверов щелкните **локальный сервер**. В области сведений щелкните **рабочей группы**. **Свойства системы** откроется диалоговое окно.

2.  В **свойства системы** диалоговом нажмите кнопку **изменение**. **Изменение имени компьютера или домена** откроется диалоговое окно.

3.  В **имя компьютера**в **членом**, нажмите кнопку **домена**, а затем введите имя домена, который требуется присоединить. Например, если имя домена — corp.contoso.com, введите **corp.contoso.com**.

4.  Нажмите кнопку **ОК**. **Безопасности Windows** откроется диалоговое окно.

5.  В **изменение имени компьютера или домена**в **имя пользователя**, введите имя пользователя и в **пароль**, введите пароль и нажмите кнопку **ОК**. **Изменение имени компьютера или домена** откроется диалоговое окно приветствия домена. Нажмите кнопку **ОК**.

6.  **Изменение имени компьютера или домена** диалоговое окно отображает сообщение о том, что необходимо перезагрузить компьютер, чтобы применить изменения. Нажмите кнопку **ОК**.

7.  На **свойства системы** диалогового **имя компьютера** щелкните **закрыть**. **Microsoft Windows** диалоговое окно открывается и отображается сообщение о том, что необходимо перезагрузить компьютер, чтобы применить изменения. Нажмите кнопку **Перезагрузить сейчас**.

> [!NOTE]
> Сведения о присоединении к домену компьютеров, работающих под управлением других операционных систем Майкрософт к домену см. в разделе [приложение C — присоединение компьютеров к домену](#BKMK_C).

###### <a name="to-log-on-to-the-domain-using-computers-running-windows-server-2016"></a>Для входа в домен с компьютеров под управлением Windows Server 2016

3.  В левом нижнем углу выберите **другой пользователь**.

4.  В **имя пользователя**, введите имя пользователя.

5.  В **пароль**, введите пароль для домена и затем щелкните стрелку или нажмите клавишу ВВОД.

> [!NOTE]

##### <a name="BKMK_installDHCP"></a>Установите протокол DHCP (DHCP)
Эту процедуру можно использовать для установки и настройки роли DHCP-сервера, с помощью добавления ролей и компонентов мастера.

Членство в группе **"Администраторы домена"**, или в эквивалентной минимальным требованием для выполнения этой процедуры.

###### <a name="to-install-dhcp"></a>Установка протокола DHCP

1.  На сервере DHCP1 в диспетчере серверов щелкните **управление**, а затем нажмите кнопку **Добавить роли и компоненты**. Откроется мастер добавления ролей и компонентов.

2.  В **перед началом**, нажмите кнопку **Далее**.

    > [!NOTE]
    > **Перед началом** страница мастера компонентов и добавить роли не отображается, если ранее вы выбрали **пропустить эту страницу по умолчанию** запуска добавления ролей и компонентов мастера.

3.  В **Выбор типа установки**, убедитесь, что **Установка на основе ролей или компонентов** установлен, а затем нажмите кнопку **Далее**.

4.  В **Выбор целевого сервера**, убедитесь, что **выберите сервер из пула серверов** выбран. В **пула серверов**, убедитесь, что выбран локальный компьютер. Нажмите кнопку **Далее**.

5.  В **Выбор ролей сервера**в **ролей**выберите **DHCP-сервера**. В **добавить компоненты, необходимые для DHCP-сервера**, нажмите кнопку **добавить компоненты**. Нажмите кнопку **Далее**.

#### <a name="BKMK_deployNPS1"></a>Развертывание NPS1
Сервер политики сети (NPS) сервер установлен как подготовительным действием для развертывания других технологий доступа к сети, такие как серверы виртуальных частных сетей (VPN) и точки беспроводного доступа, коммутаторы с проверкой подлинности 802.1 X.

Сервер политики сети (NPS) позволяет централизованно настраивать и контролировать политики сети с помощью следующих средств: сервера службы проверки подлинности удаленного пользователя (RADIUS) и прокси-сервера RADIUS.

NPS — это необязательный компонент основной сети, но следует установить сервер политики сети, если истинно одно из следующих:

-   Планируется расширение сети с добавлением серверов удаленного доступа, совместимых с протоколом RADIUS, например компьютер под управлением Windows Server 2016, Windows Server 2012 R2, Windows Server 2012, Windows Server 2008 R2 или Windows Server 2008 и службы маршрутизации и удаленного доступа, шлюза служб терминалов или шлюза удаленных рабочих столов.


-   Планируется развертывание проверку подлинности 802.1 X для проводного или беспроводного доступа.

Прежде чем развернуть эту службу роли, необходимо выполнить следующие действия на компьютере, для которого настраивается как сервер политики сети.

-   Выполните действия в разделе [Настройка всех серверов](#BKMK_configuringAll).

-   Выполните действия в разделе [Присоединение серверов к домену и вход](#BKMK_joinlogserver)

##### <a name="bkmk_NetFndtn_Pln_NPS-01"></a>Планирование развертывания NPS1
Если вы планируете развернуть серверы доступа к сети, таких как точки беспроводного доступа или VPN-серверы после развертывания основной сети, рекомендуется развернуть сервер политики сети.

При использовании сервера политики сети в качестве сервера службы проверки подлинности удаленного пользователя (RADIUS) выполняет проверку подлинности и авторизацию запросов на подключение через серверы доступа к сети. Сервер политики сети также позволяет осуществлять централизованную настройку и управление политиками сети, определяющие, кто может получить доступ к сети, как они могут получить доступ к сети и когда они могут подключаться к сети.

Ниже приведены ключевые действия по планированию перед установкой сервера политики сети.

- Планирование базы данных учетных записей пользователей. По умолчанию если присоединить сервер политики сети в домене Active Directory, сервер политики сети выполняет проверку подлинности и авторизации с помощью базы данных учетных записей пользователей Доменных службах Active Directory. В некоторых случаях например в больших сетях, использующих сервер политики сети как RADIUS-прокси для пересылки запросов подключения другим RADIUS-серверам, вы может потребоваться установка сервера политики сети на компьютере не входящих в домен.

- Планирование учета RADIUS. Сервер политики сети позволяет вести журнал данных учета в базе данных SQL Server или в текстовый файл на локальном компьютере. Если вы хотите использовать ведение журнала SQL Server, запланируйте установку и настройку сервера SQL Server.

##### <a name="BKMK_installNPS"></a>Установка сервера политики сети (NPS)
Эту процедуру можно использовать для установки сервера политики сети (NPS) с помощью добавления ролей и компонентов мастера. NPS — это служба роли в роли сервера служб политики сети и доступа.

> [!NOTE]

Если при установке сервера политики сети, включен брандмауэр Windows в режиме повышенной безопасности, исключения брандмауэра для этих портов автоматически создаются во время установки для протокола IP версии 6 \(IPv6\) и IPv4-трафика. Если серверы доступа к сети настроены на отправку RADIUS-трафика через порты, отличные от значения по умолчанию, удалите исключения, созданные в брандмауэре Windows в режиме повышенной безопасности во время установки сервера политики сети и создать исключения для портов, используемых для RADIUS-трафика.

##### <a name="BKMK_registerNPS"></a>Регистрация сервера политики сети в домене по умолчанию
Эту процедуру можно использовать для регистрации сервера политики сети в домене, где сервер является членом домена.

NPS-серверов необходимо зарегистрировать в Active Directory, чтобы они могли разрешение на чтение свойств удаленного доступа учетных записей пользователей в процессе авторизации. Регистрация NPS-сервер добавляется в **серверы RAS и IAS** группы в Active Directory.

**Учетные данные администратора**

Для выполнения этой процедуры необходимо быть членом **"Администраторы домена"** группы.

> [!NOTE]

> `netsh nps add registeredserver domain=corp.contoso.com server=NPS1.corp.contoso.com`

#### <a name="BKMK_IIS"></a>Развертывание WEB1

Роль веб-сервер (IIS) в Windows Server 2016 предоставляет безопасную, простую в управлении, модульную и расширяемую платформу для надежного размещения веб-сайтов, служб и приложений. С помощью Internet Information Services (IIS), обеспечивает доступ к информации пользователям в Интернете, интрасети или экстрасети. IIS является унифицированной веб-платформой, которая интегрирует IIS, ASP.NET, службы FTP, PHP и Windows Communication Foundation (WCF).

Кроме того опубликовать список отзыва Сертификатов для доступа с компьютерах-членах домена, роль сервера веб-сервер (IIS) позволяет настроить и управлять несколькими веб-сайты, веб-приложений и FTP-узлов. Кроме того, в IIS предоставляет следующие преимущества:

-   Развертывание веб-безопасности благодаря сокращению объема сервера основания печати и автоматической изоляции приложений.

-   Простое развертывание и выполнения ASP.NET, classic ASP и PHP, веб-приложения на том же сервере.

-   Изоляция приложений путем рабочих процессов уникальное удостоверение и запуска в изолированной конфигурации по умолчанию, еще более сокращающие риски безопасности.

-   Легко добавлять, удалять и даже замена встроенных компонентов IIS с настраиваемыми модулями, отвечающими потребностям пользователя.

-   Повышение скорости работы веб-сайта с помощью встроенного динамического кэширования и расширенного сжатия.

Для развертывания WEB1 — это компьютер, на котором выполняется роль сервера веб-сервер (IIS), необходимо выполнить следующие действия.

-   Выполните действия в разделе [Настройка всех серверов](#BKMK_configuringAll).

-   Выполните действия в разделе [Присоединение серверов к домену и вход](#BKMK_joinlogserver)

-   [Установка роли сервера веб-сервер (IIS)](#BKMK_install_IIS)
