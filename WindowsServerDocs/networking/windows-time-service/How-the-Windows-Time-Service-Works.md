Несмотря на то, что служба времени Windows не точное реализация протокола сетевого времени (NTP), он использует сложного набора алгоритмов, определенный в спецификации NTP для обеспечения максимальной точности часы на компьютерах в сети. В идеальном случае все часы компьютера в домене AD DS синхронизируются с учетом времени основного компьютера. Многие факторы могут повлиять на синхронизацию времени в сети. Следующие факторы часто повлиять на точность синхронизации в Доменных службах Active Directory:  
  
-   Состояния сети  
  
-   Точность часов оборудования компьютера  
  
-   Объем ЦП и сетевых ресурсов, доступных для службы времени Windows

В разделе [времени точность 2016 Windows](accurate-time.md) и [поддержки границы, чтобы настроить службу времени Windows для сред высокая точность](https://go.microsoft.com/fwlink/?LinkID=179459) Дополнительные сведения.  
  
По умолчанию для синхронизации с time.windows.com настроены компьютеров, которые часто синхронизации времени меньше или не присоединен к домену. Таким образом невозможно гарантировать точность времени на компьютеры с временными или отсутствие сетевых подключений.  
  
Лес AD DS имеет иерархию синхронизации времени. Служба времени Windows синхронизирует время между компьютерами в иерархии с максимально точным часами ссылки в верхней части. Если на компьютере настроено более чем одного источника времени, времени Windows использует алгоритмы NTP для выбора Лучший источник времени из настроенных источников, в зависимости от компьютера, возможность синхронизации с источником этого времени. Служба времени Windows не поддерживает синхронизацию сети от рассылки или многоадресной рассылки одноранговые элементы. Дополнительные сведения об этих функциях NTP см 1305 RFC в базе данных RFC IETF.  
  
Каждый компьютер, на котором выполняется служба времени Windows использует службу для поддержания наиболее точное время. Компьютеры, которые являются членами домена выступать в качестве клиента времени по умолчанию, таким образом, в большинстве случаев не необходимо настроить службу времени Windows. Тем не менее служба времени Windows можно настроить на время запроса из источника времени назначенного ссылки и могут также предоставлять время клиентам.
  
Степени, в которую точных время компьютера называется первого уровня. Наиболее точные источник времени в сети (например, системных часов) занимает низкий уровень иерархической или иерархической один. Этот источник точное время называется часы ссылки. С сервером NTP-Сервером, который получает время его непосредственно с часами ссылку занимает иерархической, расположен на один уровень выше, чем ссылка часов. Ресурсы, которые получение времени с NTP-сервера состоит из двух этапов вне часов ссылки и таким образом занимают иерархической, который двух выше, чем наиболее точные источник времени и так далее. Как число иерархической компьютера увеличивается, время на часах его системы, может стать менее точным. Таким образом уровень иерархической для любого компьютера является показателем насколько компьютера синхронизируется с максимально точным источника времени.  
  
Диспетчер W32Time получает примеры времени, используются специальные алгоритмов в NTP определяется примеров время подходит для использования. Служба времени также использует другой набор алгоритмов, чтобы определить, какие из источникам времени становится максимально точным. Если служба времени установила лучше какие Образец времени, в зависимости от описанных выше условий ли его регулирует скорость часов локального разрешить сведения к правильное время. Если разница во времени между часов локального и пример выбранного точное время (также называемый время отклонения) слишком велик, чтобы исправить путем регулировку скорости часов локального, служба времени устанавливает часов локального правильное время. Коррекция частоты синхронизации или изменение времени прямой часы называется определенный промежуток времени.

## <a name="w2k3tr_times_how_ekoc"></a>Протоколы времени службы времени Windows  

Протоколы время определить, насколько двух компьютеров Часы синхронизированы. Протокол NTP отвечает за определение сведения о наиболее доступное время и схождение часы, чтобы обеспечить непрерывную согласованное время в отдельных системах.  
  
Служба времени Windows использует протокола сетевого времени (NTP) для синхронизации времени в сети. NTP-Сервером представляет время IP-протокол, включающий Дисциплина алгоритмы, необходимые для синхронизации часов. NTP-сервер — это более точные протокол времени чем простые сети времени протокол SNTP, используемый в некоторых версиях Windows. Тем не менее W32Time продолжает поддерживать SNTP для обеспечения обратной совместимости с компьютерами под управлением времени на основе SNTP служб, таких как Windows 2000.  
  
### <a name="network-time-protocol"></a>Протокол NTP  
Протокола сетевого времени (NTP) по умолчанию используется значение времени синхронизации протоколом, используемым службой времени Windows в операционной системе. NTP-сервер — это протокол отказоустойчивой, высокомасштабируемых время и протоколы, наиболее часто используются для синхронизации часов компьютера с помощью ссылки назначенного времени.  
  
Выполните синхронизацию времени NTP происходит в течение периода времени и предполагает передачу NTP-Сервером пакетов в сети. NTP пакеты содержат отметки времени, включающие образец времени от клиента и сервера, который участвует в во время синхронизации.  
  
NTP-сервер использует часами ссылок для определения наиболее точное время для использования и синхронизирует все часы в сети для этой ссылки часы. NTP-сервер использует Всемирное координированное время (UTC) в качестве универсальный стандарт текущего времени. UTC не зависит от часовых поясов и позволяет NTP для использования в любом месте в мире независимо от того, параметры часового пояса.  
  
#### <a name="ntp-algorithms"></a>Алгоритмы NTP  
NTP включает два алгоритма, алгоритма фильтрации часы и выбора часы алгоритм, чтобы определить наилучший Образец времени службы времени Windows. Алгоритм фильтрации часы предназначен для анализировать примеры времени, полученные из источников запрошенное время и определить лучший время примеров из каждого источника. Алгоритм выбора часы затем определяет наиболее точные сервера времени в сети. Эта информация передается в алгоритм Дисциплина часы, который использует данные, собранные для исправления часов локального компьютера, при компенсация ошибки из-за неточность часы задержки и компьютера сети.  
  
Алгоритмы NTP являются наиболее точные условиях освещения и средних нагрузку сети и сервера. Как и в случае с любой алгоритм, учитывает время передачи по сети, NTP алгоритмы, могут выполнять плохо условиях перегрузки предельно сети. Дополнительные сведения об алгоритмах NTP см. в разделе 1305 RFC в базе данных RFC IETF.  
  
#### <a name="ntp-time-provider"></a>Поставщик времени NTP  
Служба времени Windows является пакет полное время синхронизации, который может поддерживать множество устройств и протоколов, время. Чтобы включить такую поддержку, данная служба использует поставщиков модульного времени. Поставщик времени отвечает за либо Получение отметки точного времени (по сети или от оборудования), а также за предоставление этих отметки времени на другие компьютеры в сети.  
  
Поставщик NTP — поставщик время (зима), входящий в состав операционной системы. Поставщик NTP следующим стандартам, определяется NTP версии 3 для клиента и сервера и могут взаимодействовать с SNTP клиентами и серверами для обеспечения обратной совместимости с Windows 2000 и других клиентов SNTP. Поставщик NTP-сервер в службе времени Windows состоит из следующих двух частей:  
  
-   **NTP-сервер поставщика выходных данных.** Это время сервер, который отвечает на запросы клиентов времени в сети.  
  
-   **Поставщик NTP-клиент.** Это время клиент, который получает сведения о времени из другого источника, устройство или NTP-сервером и может возвращать время примеры, которые можно использовать для синхронизации часов локального.  
  
Несмотря на то, что фактическое выполнение операций из этих двух поставщиков тесно связаны, они отображаются независимых службы времени. Начиная с Windows 2000 Server, когда Windows компьютер подключен к сети, он настраивается как NTP-клиента. Кроме того компьютеры под управлением Windows времени службы только попытки синхронизации времени с контроллером домена или источника времени вручную заданного по умолчанию. Эти поставщики Предпочтительное время поскольку они автоматически доступны, безопасного источников времени.  
  
#### <a name="ntp-security"></a>NTP-сервер безопасности  

Леса AD DS служба времени Windows зависит от стандартного средства безопасности для применения проверки подлинности данных времени. Безопасность NTP пакетов, передаваемых между членом домена и контроллером локального домена, который выступает в качестве сервера времени на основе общего ключа проверки подлинности. Служба времени Windows использует ключ сеанса Kerberos компьютера для создания подписи с проверкой подлинности на NTP пакетов, передаваемых по сети. NTP пакеты не передаются внутри безопасный канал сетевого входа в систему. Вместо этого когда компьютер запрашивает время с контроллера домена в иерархии доменов, службы времени Windows требует проверки подлинности время. Затем контроллер домена возвращает необходимые сведения в форме 64-разрядное значение, который прошел проверку подлинности с помощью ключа сеанса от службы сетевого входа в систему. Если возвращенный NTP пакетов не подписана ключом сеанса на компьютере или неправильно подписан, время отклоняется. Такие сбои проверки подлинности регистрируются в журнале событий. Таким образом служба времени Windows обеспечивает безопасность данных NTP в лесу AD DS.  
  
Как правило клиентов службы времени Windows автоматически получить точного времени для синхронизации с контроллерами домена в том же домене. В лесу контроллеров домена в дочерний домен синхронизации времени с контроллерами домена в их родительских доменов. Когда сервер времени возвращает пакет с проверкой подлинности NTP клиенту, который запрашивает время, пакет подписан с помощью ключа сеанса Kerberos, определенные с использованием учетной записи доверия между доменами. Учетная запись доверия между доменами создается в том случае, когда нового домена AD DS соединяет леса и ключ сеанса управляет служба сетевого входа в систему. Таким образом контроллер домена, которая настроена как надежный, в корневом домене леса становится источником времени с проверкой подлинности для всех контроллеров домена в доменах родительским и дочерним, а также косвенно для всех компьютеров, расположенных в доменном дереве.  
  
Служба времени Windows можно настроить для работы между лесами, но важно отметить, что эта конфигурация не является безопасным. Например NTP-сервером может быть доступен в другом лесу. Тем не менее поскольку этот компьютер находится в другом лесу, отсутствует ключ сеанса Kerberos с помощью которого для входа и проверки подлинности NTP-Сервером пакетов. Чтобы получить точную синхронизацию времени с компьютера в другом лесу, необходимые клиенту доступ к сети для этого компьютера, и служба времени должна быть настроена на использование источника конкретный момент времени, расположенный в другом лесу. Если клиент настроен вручную для доступа с NTP-сервером, за пределами иерархии домена, NTP пакетов, передаваемых между клиентом и сервером времени не прошел проверку подлинности и поэтому не являются надежными. Даже при реализации доверие леса служба времени Windows не является безопасным между лесами. Несмотря на то, что сетевого входа в систему безопасного канала — это механизм проверки подлинности для службы времени Windows, не поддерживается проверка подлинности между лесами.  
  
#### <a name="hardware-devices-that-are-supported-by-the-windows-time-service"></a>Устройства, которые поддерживаются службой времени Windows  
Аппаратные часы, например, GPS или часы радио часто используется в качестве устройства часы высокоточный ссылки. По умолчанию данный поставщик времени NTP службы времени Windows не поддерживает прямое подключение устройства к компьютеру, хотя можно создать поставщик независимых времени на основе программного обеспечения, которая поддерживает этот тип подключения. Этот тип поставщика, в сочетании со службой времени Windows можно предоставить ссылку надежной, стабильной времени.  
  
Устройств, например cesium часы или получатель системы глобального позиционирования (GPS), предоставляют точное текущее время, выполнив standard для получения точных определения времени. Cesium часы высокой стабильностью и не подвержены влиянию факторы, например температуру, давления или влажности, но также очень дорого. GPS-приемник гораздо меньше дорого в эксплуатации, а также часы точные ссылки. GPS-приемник получают значения времени от спутников, которые получают значения времени с часами cesium. Без использования поставщика независимых времени серверы Windows могут приобрести их времени путем подключения к внешним NTP-сервером, подключенный к устройству посредством телефонной связи или Интернета. Организации, например военно-морской обсерватории США предоставляют NTP серверов, подключенных к максимальную надежность синхронизирующие.  
  
Многие GPS-приемник и другие устройства времени могут функционировать как NTP-серверами в сети. Вы можете настроить леса AD DS для синхронизации времени с этих устройств внешнее оборудование только в том случае, если они также выступает NTP-серверами в сети. Чтобы сделать это, настройте контроллер домена работает как основной контроллер домена эмулятора в корне леса для синхронизации с NTP-сервером, предоставляемые GPS-устройства. Чтобы сделать это, в разделе [настроить службу времени Windows на эмулятор основного контроллера домена в корневом домене леса](https://docs.microsoft.com/en-us/previous-versions/windows/it-pro/windows-server-2008-R2-and-2008/cc731191%28v=ws.10%29).  
  
### <a name="simple-network-time-protocol"></a>Простой протокол NTP  
Простой протокол времени сети (SNTP) является протоколом упрощенное времени, предназначенный для серверов и клиентов, которые не требуют степень точности, который предоставляет NTP. SNTP более Минимальная версия NTP, является основной время протокол, который используется в Windows 2000. Поскольку форматы пакетов сети SNTP и NTP идентичны, совместимы двух протоколов. Основное различие между этими двумя является SNTP наличие ошибки управления и сложные фильтрации системы, предоставляемых NTP-Сервером. Дополнительные сведения о простой протокол NTP. в разделе 1769 RFC в базе данных RFC IETF.  
  
### <a name="time-protocol-interoperability"></a>Время протокола взаимодействия  
Служба времени Windows может работать в смешанной среде с компьютеров под управлением Windows 2000, Windows XP и Windows Server 2003, поскольку SNTP протокол, используемый в Windows 2000 поддерживает взаимодействие с протокол NTP-Сервером в Windows XP и Windows Server 2003.  
  
Служба времени Windows NT Server 4.0, называется TimeServ, синхронизирует время в сети Windows NT 4.0. TimeServ — это функция надстройки, доступные как часть *Microsoft Windows NT 4.0 Resource Kit* и не предоставляет степень надежности синхронизации времени, которые требуются для Windows Server 2003.  
  
Служба времени Windows могут взаимодействовать с компьютерами под управлением Windows NT 4.0, так как они синхронизации времени с компьютеров под управлением Windows 2000 или Windows Server 2003; Тем не менее на компьютере под управлением Windows 2000 или Windows Server 2003 автоматически не обнаруживает серверы Windows NT 4.0. Например если ваш домен настроен для синхронизации времени с использованием домена, на основе иерархии метода синхронизации, и вы хотите, чтобы компьютеры в иерархии доменов для синхронизации времени с контроллером домена Windows NT 4.0, необходимо настроить эти компьютеры вручную выполнить синхронизацию с контроллерами домена Windows NT 4.0.  
  
Windows NT 4.0 использует более простой механизм для синхронизации времени, чем использует службы времени Windows. Таким образом чтобы обеспечить синхронизацию точное время в сети, рекомендуется обновить контроллеры домена Windows NT 4.0, Windows 2000 или Windows Server 2003.  
  
## <a name="w2k3tr_times_how_izcr"></a>Windows времени службы процессы и взаимодействия  

Служба времени Windows предназначена для синхронизации часы компьютеров в сети. Процесс синхронизации времени сети, также называемое конвергенции времени происходит во всей сети как каждый раз, доступ к компьютеру с более точное время сервера. Время конвергенции включает процесс, с помощью которого полномочный сервер предоставляет текущего времени на клиентских компьютерах в виде пакетов NTP. Сведения, приведенные в пакет указывает, требуется ли корректировка сценарию текущее время на компьютере, что синхронизируется с сервером более точные.  
  
В рамках процесса конвергенции времени членов домена попытка синхронизации времени с контроллером домена, расположенных в том же домене. Если компьютер является контроллером домена, предпринимается попытка синхронизации с более полномочным контроллера домена.  
  
Компьютеры под управлением Windows XP Home Edition или компьютеров, которые не присоединены к домену не выполнять синхронизацию с иерархии доменов, но по умолчанию для получения времени с сервером time.windows.com.  
  
Чтобы установить компьютер под управлением Windows Server 2003 как заслуживающую доверия, компьютер должен быть настроен для надежного источника времени. По умолчанию первый контроллер домена, установленного на домене Windows Server 2003 автоматически настраивается для надежного источника времени. Так как основного компьютера домена, его необходимо настроить синхронизацию с внешним источником времени, а не с помощью иерархии доменов. Также по умолчанию, все другие члены домена Windows Server 2003 и которые настроены для синхронизации с иерархии доменов.  
  
После создания сети Windows Server 2003 можно настроить службы времени Windows, используйте один из следующих вариантов для синхронизации:  
  
-   Синхронизация на основе иерархии доменов  
  
-   Источник синхронизации вручную  
  
-   Все доступные способы синхронизации  
  
-   Без синхронизации.  
  
Каждая из этих типов синхронизации рассматривается в следующем разделе.  
  
### <a name="domain-hierarchy-based-synchronization"></a>Синхронизация на основе иерархии доменов

На основании иерархии доменов, служба времени Windows определяет точность каждый сервер времени. В лесу Windows Server 2003 компьютер, который содержит основного домена (PDC) контроллеру эмулятора роль хозяина операций, расположенный в корневом домене леса, занимает должность Лучший источник времени, если не настроен другой надежного источника времени. #### <a name="reliable-time-source-configuration"></a>Конфигурация надежного источника времени  
Компьютер, который настроен в качестве надежного источника времени определяется как корень службы времени. Корень службы времени является полномочным сервером для домена и обычно настроен для получения времени от внешнего сервера NTP или устройства. Сервер времени может настроен в качестве надежного источника времени для оптимизации, как время передается во всей иерархии доменов. Если контроллер домена настроен для надежного источника времени, службы сетевого входа в систему объявляет о выпуске этого контроллера домена в качестве надежного источника времени при входе в сеть. Когда другие контроллеры домена будут искать источник времени для синхронизации с, они выберите надежного источника сначала при его наличии.  
  
#### <a name="time-source-selection"></a>Выбор источника времени  
Процесс выбора источника времени можно создать две проблемы в сети.  
  
-   Циклическое переключение дополнительной синхронизации.  
  
-   Повышенная тома входящего сетевого трафика.  
  
Цикл в сети синхронизации возникает, когда время согласованность между группа контроллеров домена и то же время является общим для их постоянно без повторной синхронизации с другим надежным источником времени. Алгоритм выбора источника времени службы времени Windows предназначен для защиты от подобных проблем.  
  
Компьютер использует один из следующих методов для идентификации источник времени для синхронизации с:  
  
-   Если компьютер не является членом домена, его необходимо настроить синхронизацию с выбранным источником времени.  
  
-   Если компьютер является рядовым сервером или рабочей станции в рамках домена, по умолчанию, его соответствует структуре Доменных службах Active Directory и выполняет синхронизацию времени с контроллером домена в его локального домена, на котором в настоящее время работает служба времени Windows.  
  
Если компьютер является контроллером домена, она совершает до шести запросы для поиска другой контроллер домена для синхронизации с. Каждый запрос предназначен для определения источника времени с определенные атрибуты, такие как тип контроллера домена, в определенном расположении, и того, является ли это надежного источника времени. Источник времени также должны соблюдать следующие ограничения:  
  
-   Надежным источником времени только можно синхронизировать с контроллером домена в родительском домене.  
  
-   Эмулятор основного контроллера домена можно синхронизировать с надежным источником времени в собственном домене или любой контроллер домена в родительском домене.  
  
Если контроллер домена не смогут синхронизировать с тип контроллера домена, он запрашивает, запрос не выполняется. Контроллер домена знает, какой тип компьютера, можно получить время, прежде чем запрос. Например локальные эмулятор основного контроллера домена не пытается номера запроса три или шесть так как контроллер домена не будет пытаться синхронизировать с самим собой.

### <a name="manually-specified-synchronization"></a>Синхронизация вручную  
Синхронизация вручную позволяет назначить один или несколько узлов, с которых компьютер получает времени. Если компьютер не является членом домена, его необходимо вручную настроить на синхронизацию с выбранным источником времени. Компьютер, который является членом домена, настройки по умолчанию для синхронизации из иерархии доменов, синхронизации вручную наиболее полезна для корневого домена леса, или для компьютеров, которые не присоединены к домену. Указав внешним NTP-сервером для синхронизации с компьютером, заслуживающих доверия для домена вручную предоставляет надежный источник времени. Тем не менее Настройка основного компьютера домена для синхронизации с системных часов — четкой для предоставления наиболее точные и безопасного время, необходимое для вашего домена.  
  
Источника времени вручную не проходят проверку подлинности, если поставщик конкретный момент времени для их, и поэтому уязвимы к атакам. Кроме того Если компьютер выполняет синхронизацию с источником вручную, а не его проверяющий контроллер домена, эти два компьютера может быть не синхронизировано, вызывающую сбой проверки подлинности Kerberos. Это может привести к других действий, требующих проверки подлинности сети к сбою, например, печати или совместный доступ к файлам. Если только корня леса настроен на синхронизацию с внешним источником, все компьютеры в лесу, будут синхронизированы друг с другом, что усложняет атаки с повторением пакетов.  
  
### <a name="all-available-synchronization-mechanisms"></a>Все доступные способы синхронизации  

Вариант «все доступные способы синхронизации» — это наиболее полезно метод синхронизации для пользователей в сети. Этот метод позволяет синхронизацию с иерархии доменов и также может предоставлять альтернативный источник времени, если иерархии доменов становится недоступным, в зависимости от конфигурации. Если клиент не удалось синхронизировать время с иерархии доменов, источник времени автоматически возвращается к источника времени, указанного **NTP-сервер** параметр. Этот метод синхронизации, скорее всего предоставить клиентам точное время.  

### <a name="stopping-time-synchronization"></a>Остановка во время синхронизации  
Существуют определенные ситуаций, в которых требуется запретить синхронизацию времени. Например если компьютер пытается выполнить синхронизацию из источника времени в Интернете или из другого сайта в глобальной сети с помощью подключения удаленного доступа, он может взиматься плата дорогостоящих телефона. При отключении синхронизации на этом компьютере, компьютер предотвратить попытки доступа к источник времени через подключение удаленного доступа.  
  
Также можно отключить синхронизацию, чтобы предотвратить создание ошибки в журнале событий. Каждый раз, когда компьютер пытается выполнить синхронизацию с источником времени, который недоступен, он создает сообщение об ошибке в журнале событий. Если источник времени берется из сети для запланированного обслуживания, и вы не планируете измените конфигурацию клиента для синхронизации из другого источника, можно отключить синхронизации на стороне клиента, чтобы предотвратить попытка синхронизации, пока недоступен сервер времени.  
  
Это полезно для отключения синхронизации на компьютере, который используется в качестве корневого сети синхронизации. Это означает, что доверие его часов локального компьютера корневой. Если корень иерархии синхронизации не присвоено **NoSync** , и если это не удалось выполнить синхронизацию с другим источником времени, клиенты не принимают пакет, который отправляет этот компьютер из-за его время, не может быть доверенным.  
  
Только один раз серверы, которые являются доверенными для клиентов, даже если они не были синхронизированы с другой источник времени, которые определены как надежный источник времени серверы клиентом.  
  
### <a name="disabling-the-windows-time-service"></a>Отключение службы времени Windows  
Можно полностью отключить службу времени Windows (W32Time). Если вы хотите реализовать продукт синхронизации времени сторонних производителей, который использует NTP-Сервером, необходимо отключить службу времени Windows. Это так, как все серверы NTP требуется доступ к порту протокола передачи датаграмм пользователя (UDP) 123 и до тех пор, пока служба времени Windows запущена в операционной системе Windows Server 2003, порт 123 остается зарезервированным времени Windows.
