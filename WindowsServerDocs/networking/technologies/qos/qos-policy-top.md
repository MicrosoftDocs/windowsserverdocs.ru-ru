# <a name="quality-of-service-qos-policy"></a>Политика качества обслуживания \( QoS \)

>Применяется к: Windows Server (Semi-Annual Channel), Windows Server 2016

Вы можете использовать политику качества обслуживания как централизованное средство управления пропускной способностью сети для всей инфраструктуры Active Directory, создавая профили качества обслуживания, параметры которых распространяются с помощью групповой политики.

>[!NOTE]
>  Помимо этого раздела, доступна следующая документация по политикам QoS.
>
>  - [начало работы с политикой QoS](qos-policy-get-started.md)
>  - [Управление политикой QoS](qos-policy-manage.md)
>  - [Часто задаваемые вопросы о политике качества обслуживания](qos-policy-faq.md)

Политики качества обслуживания применяются к сеансу входа пользователя или к компьютеру в составе объекта \( групповой политики объекта Групповая политика \) , связанного с контейнером Active Directory, например с доменом, сайтом или подразделением подразделения \( \) .

Управление трафиком QoS происходит ниже уровня приложения. Это означает, что существующие приложения не нужно изменять, чтобы воспользоваться преимуществами, обеспечиваемыми политиками QoS.

## <a name="operating-systems-that-support-qos-policy"></a>Операционные системы, поддерживающие политику QoS

Политику качества обслуживания можно использовать для управления пропускной способностью компьютеров или пользователей со следующими операционными системами Майкрософт.

- Windows Server 2016
- Windows 10
- Windows Server 2012 R2
- Windows 8.1
- Windows Server 2012
- Windows 8
- Windows Server 2008 R2
- Windows 7
- Windows Server 2008
- Windows Vista

**Политика домена по умолчанию | Конфигурация компьютера | Политики | Параметры Windows | Служба \- QoS на основе политик**

Этот путь показан на следующем рисунке.

![Расположение политики качества обслуживания в групповая политика](../../media/QoS/QoS-Gp.jpg)

В Windows Server 2016 редактор "Управление групповыми политиками" путь к политике качества обслуживания для пользовательской конфигурации приведен ниже.

**Политика домена по умолчанию | Конфигурация пользователя | Политики | Параметры Windows | Служба \- QoS на основе политик**

По умолчанию политики качества обслуживания не настроены.

## <a name="why-use-qos-policy"></a>Зачем использовать политику QoS?

По мере роста трафика в сети все равно важнее сбалансировать производительность сети с учетом затрат на обслуживание, но сетевой трафик обычно непрост в определении приоритетов и управления.

В вашей сети \- критически важные \- приложения и приложений с задержкой должны конкурировать за пропускную способность сети и трафика с более низким приоритетом. В то же время для некоторых пользователей и компьютеров с определенными требованиями к производительности сети могут потребоваться отличительные уровни обслуживания.

Проблемы, связанные с предоставлением экономичных и предсказуемых уровней производительности сети, часто отображаются по глобальным \( сетевым \) подключениям или с приложениями с учетом задержки, такими как передача голоса по IP-каналам \( VoIP \) и видео. Однако конечная цель предоставления прогнозируемых уровней сетевой службы применяется к любой сетевой среде, например к \( локальной сети предприятия \) , а также к более приложениям VoIP, например к настраиваемым \- \- бизнес-приложениям вашей компании.

Служба QoS на основе политик — это средство управления пропускной способностью сети, обеспечивающее управление сетью на основе приложений, пользователей и компьютеров.

При использовании политики качества обслуживания приложения не нужно писать для конкретных API-интерфейсов прикладных программ \( \) . Это дает возможность использовать QoS с существующими приложениями. Кроме того, Служба QoS на основе политик использует преимущества существующей инфраструктуры управления, так как качество обслуживания на основе политик встроено в групповая политика.

## <a name="define-qos-priority-through-a-differentiated-services-code-point-dscp"></a>Определение приоритета качества обслуживания с помощью приоритизированные службы DSCP кодовая точка \(\)

Можно создать политики QoS, определяющие приоритет сетевого трафика с помощью приоритизированные службы значения DSCP кодовой точки \( \) , назначаемого различным типам сетевого трафика.

DSCP позволяет применять значение \( 0 – 63 \) в поле типа службы \( TOS \) в заголовке пакета IPv4 и в поле Класс трафика в IPv6.

Значение DSCP обеспечивает классификацию сетевого трафика на \( уровне IP-адреса протокола Интернета \) , который маршрутизаторы используют для выбора поведения очереди трафика.

Например, можно настроить маршрутизаторы на размещение пакетов с конкретными значениями DSCP в одну из трех очередей: высокая важность, наиболее интенсивная работа или меньшее количество усилий.

Критически важный сетевой трафик, который находится в очереди с высоким приоритетом, имеет приоритет над другим трафиком.

### <a name="limit-network-bandwidth-use-per-application-with-throttle-rate"></a>Ограничение использования пропускной способности сети на приложение с частотой регулирования

Можно также ограничить исходящий сетевой трафик приложения, указав частоту регулирования в политике QoS.

Политика QoS, определяющая ограничения регулирования, определяет частоту исходящего сетевого трафика. Например, для управления затратами на глобальную сеть ИТ-отдел может реализовать соглашение об уровне обслуживания, которое указывает, что файловый сервер никогда не будет предоставлять загрузку сверх определенной частоты.

### <a name="use-qos-policy-to-apply-dscp-values-and-throttle-rates"></a>Использование политики качества обслуживания для применения значений DSCP и регулирования скорости

Можно также использовать политику качества обслуживания для применения значений DSCP и регулирования скорости для исходящего сетевого трафика следующим образом:

- Отправка пути к приложению и каталогу

- Исходные и конечные адреса IPv4 или IPv6 или префиксы адресов

- Протокол — протокол управления передачей \( TCP \) и UDP по протоколу датаграмм \(\)

- Порты источника и назначения, а также диапазоны портов \( TCP или UDP\)

- Определенные группы пользователей или компьютеров путем развертывания в групповая политика

С помощью этих элементов управления можно указать политику качества обслуживания со значением DSCP 46 для приложения VoIP, что позволяет маршрутизаторам размещать пакеты VoIP в очереди с низкой задержкой или использовать политику качества обслуживания для регулирования объема исходящего трафика серверов до 512 килобайт в секунду \( \) при отправке с TCP-порта 443.

Можно также применить политику качества обслуживания к конкретному приложению, которое имеет специальные требования к пропускной способности. Дополнительные сведения см. в разделе [сценарии политики качества обслуживания](qos-policy-scenarios.md).

## <a name="advantages-of-qos-policy"></a>Преимущества политики качества обслуживания

С помощью политики качества обслуживания можно настраивать и применять политики QoS, которые не могут быть настроены на маршрутизаторах и коммутаторах. Политика QoS предоставляет следующие преимущества.

1. **Уровень детализации:** Создавать политики качества обслуживания на уровне пользователя на маршрутизаторах или коммутаторах очень сложно, особенно если компьютер пользователя настроен с помощью динамического назначения IP-адресов или если компьютер не подключен к фиксированным портам коммутатора или маршрутизатора, как правило, в случае с переносными компьютерами. В отличие от этого, политика QoS упрощает настройку \- политики QoS на уровне пользователя на контроллере домена и распространяет политику на компьютер пользователя.
2. **Гибкость.** Независимо от того, где или как компьютер подключается к сети, применяется политика QoS — компьютер может подключаться с помощью WiFi или Ethernet из любого расположения. Для \- политик QoS на уровне пользователя политика QoS применяется на любом совместимом устройстве в любом месте, где пользователь входит в систему.
3. **Безопасность:** Если ИТ-отдел шифрует трафик пользователей от конца к концу с помощью протокола IP-безопасности \( IPSec \) , вы не можете классифицировать трафик на маршрутизаторах на основе какой-либо информации, находящегося на сервере в пакете \( (например, порт TCP) \) . Однако с помощью политики качества обслуживания можно классифицировать пакеты на конечном устройстве, чтобы указать приоритет пакетов в IP-заголовке до того, как будут зашифрованы полезные данные IP и отправлены пакеты.
4. **Производительность:** Некоторые функции QoS, такие как регулирование, лучше выполняются, если они находятся ближе к источнику. Политика QoS перемещает такие функции качества обслуживания, самые близкие к источнику.
5. **Управляемость:** Политика QoS повышает управляемость сети двумя способами:

    **а**. Так как он основан на групповая политика, можно использовать политику QoS для настройки набора политик качества обслуживания "пользователь-компьютер" и управления им при необходимости, а также на одном центральном компьютере контроллера домена.

    **б**. Политика QoS упрощает настройку пользователей и компьютеров, предоставляя механизм для указания политик по \( URL-адресу универсального указателя ресурсов \) вместо политик, основанных на IP-адресах каждого сервера, на котором должны применяться политики QoS. Например, предположим, что в сети есть кластер серверов с общим URL-адресом. С помощью политики QoS можно создать одну политику на основе общего URL-адреса, а не создавать одну политику для каждого сервера в кластере, где каждая политика основана на IP-адресе каждого сервера.
