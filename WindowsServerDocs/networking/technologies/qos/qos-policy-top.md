# <a name="quality-of-service-qos-policy"></a>\(QoS\) политики QoS

>Область применения: Windows Server (канал точками годовой), Windows Server 2016

Политики качества обслуживания можно использовать как централизованное управление пропускной способностью сети для всей инфраструктуры Active Directory путем создания профилей QoS, параметры которого распространяются с помощью групповой политики.

>[!NOTE]
>  Дополнение к данному разделу доступна следующая документация политики QoS.  
>   
>  - [Приступая к работе с политики QoS](qos-policy-get-started.md)
>  - [Управление политики QoS](qos-policy-manage.md)
>  - [Вопросы и ответы по политики QoS](qos-policy-faq.md)

Политики QoS применяются к сеансу входа пользователя или компьютера как часть объекта групповой политики, связанный с контейнером Active Directory, например домена, сайта или подразделения \(GPO\) \(OU\).

Управление трафиком QoS происходит ниже уровня приложения, это означает, что существующие приложения не требуется изменить, чтобы воспользоваться преимуществами преимущества, предоставляемые с помощью политик QoS.

## <a name="operating-systems-that-support-qos-policy"></a>Операционные системы, поддерживающие политики QoS

Политики качества обслуживания можно использовать для управления пропускной способностью для компьютеров или пользователей с помощью следующих операционных систем корпорации Майкрософт.

- Windows Server 2012
- Windows 8
- Windows Server 2008 R2
- Windows 7
- Windows Server 2008
- Windows Vista

### <a name="location-of-qos-policy-in-group-policy"></a>Расположение политики качества обслуживания в групповой политике

В Windows Server 2016 редактор управления групповыми политиками путь к политики качества обслуживания для конфигурации компьютера выглядит следующим образом.

**Политика домена по умолчанию | Конфигурация компьютера | Политики | Параметры Windows | QoS на основе вступил**

Этот путь проиллюстрирован на следующем изображении.

![Расположение политики качества обслуживания в групповой политике](../../media/QoS/QoS-Gp.jpg)

В Windows Server 2016 редактор управления групповыми политиками путь к политики качества обслуживания для конфигурации пользователя выглядит следующим образом.

**Политика домена по умолчанию | Конфигурация пользователя | Политики | Параметры Windows | QoS на основе вступил**

По умолчанию политики QoS не настроены.

## <a name="why-use-qos-policy"></a>Зачем использовать политики качества обслуживания?
  
Мере увеличения трафика в сети, все чаще важно для вас для балансировки сетевой производительности вследствие отсутствия стоимость службы -, но сетевой трафик не обычно легко определять их приоритеты и управлять им.

В сети критически важных mission\ latency\ с учетом и приложения должны конкурируют пропускной способности сети с более низким приоритетом трафика. В то же время некоторые пользователи и компьютеры с производительности конкретных сетевых требований может потребоваться разнообразных уровней обслуживания.

Проблемы предоставления уровни производительности экономически эффективная, предсказуемую сети часто сначала отображаются по глобальной сети \(WAN\) или с приложениями, чувствительных к задержкам, например голос через IP \(VoIP\) и потоковая передача видео. Тем не менее, конечная цель предоставления уровней обслуживания предсказуемую сети применяется к любой сети среде \ (например, предприятий локальной сетевой) и более чем VoIP приложений, таких как вашей компании пользовательских режимы of\ бизнес приложений.
  
QoS на основе политики — средство управления пропускной способности сети, обеспечивающий сети управления — в зависимости от приложений, пользователей и компьютеров. 

При использовании политики качества обслуживания приложений не требуется быть предназначены для конкретного приложения программирования \(APIs\) интерфейсов. Это дает возможность использовать QoS с существующими приложениями. Кроме того на основе политики QoS пользуется существующей инфраструктуры управления, поскольку QoS на основе политики встроено в групповую политику.

## <a name="define-qos-priority-through-a-differentiated-services-code-point-dscp"></a>Определение приоритетов QoS через точку кода дифференцированных служб \(DSCP\)
  
Можно создать политики QoS, которые определения приоритета трафика сети со значением \(DSCP\) точка кода Differentiated служб, которые назначены различных типов сетевого трафика. 

DSCP позволяет применять \(0–63\) значение в поле типа службы \(TOS\) в заголовке пакета IPv4, а в поле класса трафика Traffic Class в IPv6. 

Значение DSCP предоставляет классификации трафика сети на уровне протокола \(IP\) использовать маршрутизаторы, чтобы решить очереди поведение трафика. 

Например, можно настроить маршрутизаторы для перевода пакеты с определенными значениями DSCP в одной из трех очередей: высокий приоритет, наилучшим возможным образом или ниже все усилия. 

Критически важные сетевой трафик, который находится в очереди высокий приоритет, имеет предпочтения другой трафик.

### <a name="limit-network-bandwidth-use-per-application-with-throttle-rate"></a>Ограничение использования полосы пропускания сети каждого приложения с помощью ограничения скорости

Вы также можете ограничить приложения исходящего сетевого трафика, указав ограничение скорости в политике QoS.

Политику качества обслуживания, который определяет ограничения регулирования определяет скорость исходящего сетевого трафика. Например чтобы управлять затраты на глобальную сеть, ИТ-отдел может реализовать соглашение об уровне обслуживания, указывает, что файловый сервер можно никогда не предоставляют загрузки за рамки определенного скорость.  

### <a name="use-qos-policy-to-apply-dscp-values-and-throttle-rates"></a>Использование политики качества обслуживания для применения DSCP значения и регулирования скорости

Можно также использовать политики качества обслуживания для применения значений DSCP и регулирования скорости для исходящего сетевого трафика для следующих:

- Отправка приложения и путь к каталогу

- Исходный и конечный IPv4- или IPv6 адреса или префиксы адресов

- Протокол: управление передачей протокола \(TCP\) и \(UDP\) протокола передачи датаграмм пользователя

- Исходные и целевые порты и диапазоны портов \(TCP or UDP\)

- Определенным группам пользователей или компьютеров с помощью развертывания в групповой политике

Используя эти элементы управления, можно задать политику QoS со значением DSCP 46 приложения по протоколу VoIP, включение маршрутизаторы для размещения пакетов по протоколу VoIP в очереди низкой задержкой или можно использовать политики QoS регулирования набор серверов исходящего трафика 512 Кбайт второй \(KBps\) при отправке из TCP-порт 443.

Можно также применить политику качества обслуживания для конкретного приложения, имеющего специальные пропускную способность. Дополнительные сведения см. в разделе [сценарии политик QoS](qos-policy-scenarios.md).
  
## <a name="advantages-of-qos-policy"></a>Преимущества политики QoS

С помощью политики QoS можно настроить и применять политики качества обслуживания, которые нельзя настроить на маршрутизаторах и коммутаторах. Политика QoS предоставляет следующие преимущества.
  
1. **Уровень детализации:** сложно создать политик QoS на уровне пользователя на маршрутизаторы и коммутаторы, особенно в том случае, если на компьютере, либо настроить с помощью динамических IP-адресов адрес назначения или если компьютер не подключен фиксированный ключ или маршрутизатор порты, как часто в случае с портативные компьютеры. Напротив политики QoS упрощает настроить политику QoS уровня определяемые на контроллере домена и распространять политики для компьютера пользователя.
2. **Гибкость**. Независимо от того, где и как компьютер подключается к сети применяется политика QoS - компьютер может подключаться с помощью Wi-Fi или Ethernet из любого места. Для политики QoS на уровне определяемые политики QoS применяется на любом совместимом устройстве в любом месте, где входе пользователя в систему.
3. **Безопасность:** Если ИТ-отдел шифрование трафика пользователей от начала до конца с помощью протокола безопасности \(IPsec\), нельзя классифицировать трафика на маршрутизаторах, исходя из любой выше уровень IP в пакете \ (например, TCP port\). Однако с помощью политики QoS, можно классифицировать пакетов циклом устройства, чтобы указать приоритет пакетов в IP-заголовке перед IP полезные данные шифруются и пакеты отправляются.
4. **Производительность:** некоторые функции качества обслуживания, например, регулирование, лучше выполняются, когда они находятся ближе к источнику. Политика QoS перемещает такие функции QoS, наиболее близкое к источнику.
5. **Управляемость:** политики QoS улучшает управляемость сети двумя способами:

    **a**. Так как он работает на основе групповой политики, можно использовать политики качества обслуживания для настройки и управления набор политик QoS пользователя или компьютера, на компьютере один центральный контроллер домена и при необходимости.

    **b**. Политика QoS облегчает конфигурации пользователя или компьютера, предоставляя механизм определения политик, \(URL\) URL-адрес вместо указания политики на основе IP-адресов каждого из серверов, где необходимо применить политики QoS. Например предположим, что сети имеется кластер серверов, которые совместно используют общие URL-адрес. С помощью политики QoS, можно создать одну политику, исходя из распространенных URL-адреса, вместо создания одной политики для каждого сервера в кластере, с каждой политики на основании IP-адрес каждого сервера.
