# <a name="qos-policy-scenarios"></a>Сценарии политик QoS

>Область применения: Windows Server (канал точками годовой), Windows Server 2016

В этом разделе можно использовать для просмотра гипотетических сценариев, демонстрирующих как, когда и почему нужно использовать политики качества обслуживания.

Описаны две сценария в этом разделе.

1. Приоритет сетевого трафика для бизнес-приложения
2. Приоритет сетевого трафика для HTTP-сервер приложения

>[!NOTE]
>Некоторые разделы в этом разделе содержат общие действия, которые можно предпринять для выполнения описанных действий. Более подробные инструкции по управлению политики качества обслуживания см. в разделе [управление политики QoS](qos-policy-manage.md).

## <a name="scenario-1-prioritize-network-traffic-for-a-line-of-business-application"></a>Сценарий 1: Определение приоритетов сетевого трафика для бизнес-приложения

В этом случае ИТ-отдел имеет несколько целей, их можно выполнить с помощью политики качества обслуживания:

- Обеспечить лучшую производительность сети для mission\ важных приложений.
- Обеспечить лучшую производительность сети для набор ключей пользователей, когда пользователи работают конкретного приложения.
- Убедитесь, что данные company\ всего приложения резервного копирования не снизить производительность сети с помощью слишком большого объема пропускной способности за один раз.

ИТ-отдел решает настроить политику QoS, чтобы определять их приоритеты конкретных приложений с помощью значений \(DSCP\) точкой кода дифференцированных службы для классификации сетевого трафика и настройте его маршрутизаторы для предоставления раздельное обслуживание с более высоким приоритетом трафика. 

>[!NOTE]
>Дополнительные сведения о DSCP см. раздел **определить QoS приоритет через разнообразных кода точки служб** раздела [политики качества обслуживания (QoS)](qos-policy-top.md).

Помимо значения DSCP политики качества обслуживания можно указать ограничения скорости. Регулирование действует ограничение скорость отправки совпадений политики QoS к определенному всего исходящего трафика.

### <a name="qos-policy-configuration"></a>Конфигурация политики QoS

С три отдельные цели для достижения ИТ-администратор решает создать три различных политик QoS.

#### <a name="qos-policy-for-lob-app-servers"></a>Политики качества обслуживания для серверов приложений бизнес-ПРИЛОЖЕНИЙ

Первого mission\ критические приложения, для которых ИТ-отдел создается политика QoS является ресурсом company\ всего предприятия, планирование \(ERP\) приложения. Приложение ERP размещено на нескольких компьютерах под управлением Windows Server 2016. В доменных службах Active Directory, эти компьютеры являются членами подразделения организации, созданные для серверов приложений \(LOB\) бизнес-\(OU\). Компонент приложения ERP в стороне отведенный период устанавливается на компьютерах, работающих под управлением Windows 10 и Windows 8.1.

В групповой политике ИТ-администратор выбирает объекта групповой политики \(GPO\) по которому будет применена политика QoS. С помощью мастера политики QoS, ИТ-администратор создает политику качества обслуживания под названием «бизнес-ПРИЛОЖЕНИЙ сервера политики», задает значение DSCP сценарии высокой приоритета 44 для всех приложений, любой IP-адрес, TCP и UDP, а также номер порта.

Политики качества обслуживания применяется только к серверам бизнес-ПРИЛОЖЕНИЙ путем связывания объекта групповой Политики для Подразделения, содержащего только эти серверы, через консоль управления групповыми политиками \(GPMC\) средство. Этот исходный сервер политики бизнес-ПРИЛОЖЕНИЙ применяет значение DSCP сценарии высокой приоритет всякий раз, когда компьютер отправляет сетевой трафик. Эта политика качества обслуживания можно изменить позже \ (в tool\ редактор объектов групповой политики) для включения номеров портов ERP приложения, который ограничивает политики вступили в силу только при использовании указанного номер порта.

#### <a name="qos-policy-for-the-finance-group"></a>Политики качества обслуживания для группы "Финансы"

Хотя многие группы в организации доступ к приложения ERP, группы "Финансы" зависит от этого приложения при работе с клиентами, и группы требуется постоянно высокой производительности из приложения.

Чтобы убедиться, что группы "Финансы" поддерживает своих клиентов, политики QoS необходимо классифицировать эти пользователи трафика высокий приоритет. Члены группы "Финансы" использовать приложения, отличные от приложения ERP политики применяются не. 

Из-за этого, ИТ-отдел определяет второй политики QoS под названием «клиента бизнес-ПРИЛОЖЕНИЙ политики» в средстве редактор объектов групповой политики, применяется значение DSCP 60, когда пользователь группы "Финансы" запускает приложение ERP-системы.

#### <a name="qos-policy-for-a-backup-app"></a>Политики качества обслуживания для резервного копирования приложений

Отдельное приложение резервного копирования работает на всех компьютерах. Чтобы убедиться, что приложение резервного копирования трафик не использует всех доступных сетевых ресурсов, ИТ-отдел создает политику резервного копирования данных. Указывает этой политики архивации, значением DSCP, равным 1, в зависимости от имя исполняемого файла для резервного копирования приложение, которое является **backup.exe **. 

Третий объект групповой Политики, создания и развертывания для всех клиентских компьютеров в домене. Каждый раз, когда приложение резервного копирования отправляет данные, применяется значение DSCP низким приоритетом, даже если он исходит от компьютеров в финансовом отделе.
  
>[!NOTE]
>Сетевой трафик без политики QoS отправляет со значением DSCP, равным 0.

### <a name="scenario-policies"></a>Сценарии политик

В следующей таблице представлены политики качества обслуживания для данного сценария.
  
|Имя политики|Значение DSCP|Ограничение скорости|Применять к подразделениям|Описание|  
|-----------------|----------------|-------------------|-----------------------------------|-----------------|
|[Политика не]|0|Нет|[Развертывание]|Лучшие обработки усилия (по умолчанию) для Неклассифицированные трафика.|  
|Данные архивации|1|Нет|Все клиенты|Применяется значение DSCP низкого приоритета для массового данных.|  
|Сервер бизнес-ПРИЛОЖЕНИЙ|44|Нет|Компьютер Подразделение для серверов ERP|Применяет DSCP высоким уровнем приоритета для трафика ERP сервера|  
|Клиент бизнес-ПРИЛОЖЕНИЙ|60|Нет|Группа пользователей|Применяет DSCP высоким уровнем приоритета для трафика клиента ERP|  

>[!NOTE]
>Значения DSCP будут представлены в десятичной форме.

С помощью политик QoS определены и применены с помощью групповой политики исходящий сетевой трафик получает значение DSCP указан политики. Затем маршрутизаторов предоставляют разностного обработки, основанные на эти значения DSCP с помощью очереди. Для этого ИТ-отдела, маршрутизаторы настроены четыре очереди: высоким приоритетом, приоритет среднего, усилия и низким уровнем приоритета.

Когда трафик поступает в маршрутизаторе значениями DSCP из» сервера бизнес-ПРИЛОЖЕНИЙ политики» и «клиента бизнес-ПРИЛОЖЕНИЙ политики, «данных помещается в очереди с высоким приоритетом. Трафик со значением DSCP, равным 0 получает усилия уровень обслуживания. Пакеты с значением DSCP, равным 1 (в программе резервного копирования) имеют обработку низким приоритетом.  
  
### <a name="prerequisites-for-prioritizing-a-line-of-business-application"></a>Необходимые условия для определения приоритета бизнес приложения

Чтобы выполнить эту задачу, убедитесь, что соблюдены следующие требования:

- Вовлеченных компьютеров под управлением операционных систем QoS\-compatible.

- Вовлеченных компьютеров являются членами \(AD DS\) домена доменных служб Active Directory, так что они могут быть настроены с помощью групповой политики.

- С маршрутизаторами, настроенные для DSCP \(RFC 2474\) настроены сети TCP/IP. Дополнительные сведения см. в разделе [RFC 2474](http://www.ietf.org/rfc/rfc2474.txt).

- Учетные данные администратора требований.

#### <a name="administrative-credentials"></a>Учетные данные администратора

Чтобы выполнить эту задачу, необходимо иметь возможность создания и развертывания объектов групповой политики.
  
#### <a name="setting-up-the-test-environment-for-prioritizing-a-line-of-business-application"></a>Настройка тестовой среды для определения приоритета бизнес приложения

Чтобы настроить тестовой среде, выполните следующие задачи.

- Создайте домен AD DS с клиентов и пользователей, которые сгруппированы в подразделениях. Инструкции по развертыванию AD DS см. в разделе [руководство по основной сети ](https://docs.microsoft.com/windows-server/networking/core-network-guide/core-network-guide).

- Настройте маршрутизаторы на differentially очереди на основе значений DSCP. Например значение DSCP 44 вводит очереди «Платина», и все остальные взвешенные справедливой-в очереди.

>[!NOTE]
> Значения DSCP можно просмотреть с помощью сетевого мониторинга с помощью таких средств, как сетевого монитора. После выполнения захвата сети, вы можете увидеть поля TOS в собранных данных.

#### <a name="steps-for-prioritizing-a-line-of-business-application"></a>Действия для определения приоритета бизнес приложения

Для установки приоритетов бизнес приложение, выполните следующие задачи.

1. Создать и связать \(GPO\) объекта групповой политики с помощью политики QoS.

2. Настройте маршрутизаторы differentially обрабатывать из бизнес-приложений (с помощью очереди) на основании клиент, выбранными значениями DSCP. Процедуры этой задачи будет зависеть от типа маршрутизаторов, у вас.

## <a name="scenario-2-prioritize-network-traffic-for-an-http-server-application"></a>Сценарий 2: Определение приоритетов сетевого трафика для HTTP-сервер приложения

В Windows Server 2016 на основе политики QoS включает компонент политик на основе URL-адрес. URL-адрес политики позволяют управлять пропускной способностью для HTTP-серверам.

Многие предприятия и приложений, разработанных для на веб-серверах \(IIS\) Internet Information Services, веб-приложений осуществляется из веб-браузерами на клиентских компьютерах.

В этом сценарии предполагается управлять набор серверов IIS, обучающие видео узла для сотрудников вашей организации. Целью является убедитесь, что трафик, передаваемый из этих серверов видео не перегрузки сети и убедитесь, что видео трафика отличается от голосовой и передачи данных в сети. 

Задача похож на задачу в сценарии 1. Будет проектировать и настраивать параметры управления трафиком, такие как значение DSCP для трафика, видео, и регулирование скорости так же, как для бизнес приложений. Но при указании трафика, вместо имени приложения, предоставляя только введите URL-адрес, на который будет отвечать серверное приложение HTTP: например,https://hrweb/training.
  
> [!NOTE]
>Политики QoS на основе URL-адреса нельзя использовать для определения приоритета трафика сети для компьютеров под управлением операционных систем Windows, выпущенных до Windows 7 и Windows Server 2008 R2.

### <a name="precedence-rules-for-url-based-policies"></a>Правила приоритета для политик на основе URL-адрес

Все следующие URL-адреса являются допустимыми и может быть указано в политике QoS и одновременно применять для компьютера или пользователя:

- http://video

- https://training.hr.mycompany.com

- http://10.1.10.249:8080/tech  

- https://*/ebooks

Но какой из них получат приоритет? Правила просты в использовании. Политики на основе URL-адрес более высокий приоритет в порядке чтения слева направо. Таким образом с самым высоким приоритетом самый низкий приоритет, URL-адреса имеются следующие поля:
  
[1. Схема URL-адрес](#bkmk_QoS_UrlScheme)

[2. URL-адрес узла](#bkmk_QoS_UrlHost)

[3. порт URL-адрес](#bkmk_QoS_UrlPort)

[4. путь URL-адреса](#bkmk_QoS_UrlPath)

Имеются следующие сведения:

####  <a name="bkmk_QoS_UrlScheme"></a>1. Схема URL-адрес

 `https://` обладает более высоким приоритетом, чем `http://`.

####  <a name="bkmk_QoS_UrlHost"></a>2. URL-адрес узла

 С самым высоким приоритетом до самого низкого они являются:

1. Имя узла

2. IPv6-адрес

3. IPv4-адрес

4. Подстановочный знак

В случае имени узла имя узла, с помощью более точечно элементов (подробнее) обладает более высоким приоритетом, чем узла с меньшим числом элементов, разделенных точками. Например между следующие имена:

- video.internal.training.hr.mycompany.com (глубины = 6)
  
- selfguide.training.mycompany.com (глубины = 4)
  
- Обучение (глубины = 1)
  
- Библиотека (глубины = 1)
  
 **video.internal.training.hr.mycompany.com** имеет наивысший приоритет и **selfguide.training.mycompany.com** имеет следующий высокий приоритет. **Обучение** и **библиотеки** имеют одинаковый низкий приоритет.  
  
####  <a name="bkmk_QoS_UrlPort"></a>3. порт URL-адрес

Конкретный или неявный порт номер обладает более высоким приоритетом, чем порт подстановочный знак.

####  <a name="bkmk_QoS_UrlPath"></a>4. путь URL-адреса

Как имя узла URL-путь может состоять из нескольких элементов. С более элементов всегда имеет более высокий приоритет, чем с минимальным. Например следующие пути перечисляются в порядке приоритета:  

1.  /ebooks/Tech/Windows/Networking/QoS

2.  / электронных книг и Технический/windows /

3.  /ebooks

4.  /

Если пользователь выбирает включаются все вложенные папки и файлы пути URL-адрес, этот путь URL-адрес будет иметь более низким приоритетом, чем если выбор не были внесены.

Пользователь также может выбрать указать IP-адрес назначения в политику на основе URL-адрес. IP-адрес назначения имеет более низким приоритетом, чем любой из четырех полей URL-адрес, описанные ранее.
  
### <a name="quintuple-policy"></a>Пятиэлементный политики

Пятиэлементный политики определяется код протокола, IP-адрес источника, порт источника, конечный IP-адрес и порт назначения. Пятиэлементный политики всегда имеет более высокий приоритет, чем любой политики на основе URL-адрес. 

Если уже пятиэлементный политики применяется для пользователя, новую политику на основе URL-адрес не приведут к конфликты на любом из этого пользователя клиента компьютеров.
