# <a name="manage-qos-policy"></a>Управление политики QoS

>Область применения: Windows Server (канал точками годовой), Windows Server 2016

Дополнительные сведения об использовании мастера политики качества обслуживания на создание, изменение или удаление политики QoS, можно использовать в этом разделе.

>[!NOTE]
>  Дополнение к данному разделу доступна следующая документация управления политики QoS.
> 
>  - [Политика QoS событий и ошибок](qos-policy-errors.md)

В операционных системах Windows политики QoS сочетает функциональность качества обслуживания на основе стандартов с управляемость групповой политики. Конфигурации такое сочетание позволяет упростить применение политик качества обслуживания для объектов групповой политики. В состав Windows входит мастер политики QoS, помогающие выполнять следующие задачи.

-  [Создание политики QoS](#bkmk_createpolicy)

-  [Просмотр, изменение или удаление политики QoS](#bkmk_editpolicy)

##  <a name="bkmk_createpolicy"></a>Создание политики QoS

Перед созданием политики QoS, важно понимать два ключа QoS элементы управления, используемые для управления сетевым трафиком.

- Значение DSCP

-   Ограничение скорости

### <a name="prioritizing-traffic-with-dscp"></a>Распределение приоритета для трафика с помощью DSCP

Как упоминалось в предыдущем примере бизнес-приложения, можно определить приоритет для исходящего сетевого трафика с помощью **укажите значение DSCP** настроить политику QoS с конкретное значение DSCP. 

Как описано в стандарте RFC 2474, DSCP позволяет значения от 0 до 63 быть указан в поля TOS пакетов IPv4, находящиеся в поле класса трафика Traffic Class в IPv6. Сетевые маршрутизаторы используют значение DSCP для классификации сетевых пакетов и их в очередь соответствующим образом.
  
> [!NOTE]
>  По умолчанию трафик Windows имеет значением DSCP, равным 0.
  
Количество очередей и поведение приоритетов должно разрабатываться как часть стратегии обеспечения качества обслуживания вашей организации. Например, организации могут предпочесть 5 очередям: трафик для чувствительных к задержкам, управляющий трафик, критически трафика, усилия и передачи данных массового трафика.  
  
### <a name="throttling-traffic"></a>Регулируемый трафик

Вместе с значения DSCP регулирования является другой элемент управления ключами, для управления пропускной способностью сети. Как упоминалось ранее, можно использовать **укажите частоту глушителя** параметр политики позволяет настроить политики QoS определенных ограничение скорости для исходящего трафика. Путем регулирования использования, политики QoS ограничивает исходящего сетевого трафика для заданного ограничения скорости. Пометки DSCP и регулирование можно использовать вместе для эффективного управления трафиком.

>[!NOTE]
>По умолчанию **укажите частоту глушителя** флажок не установлен.

Чтобы создать политику QoS, необходимо измените параметры из объект групповой политики (GPO) из в средство управления групповыми политиками (GPMC). Затем, в консоли управления групповыми Политиками открывается редактор объектов групповой политики.

Имя политики QoS должно быть уникальным. Применение политик к серверам и конечных пользователей зависит от того, где сохранена политика QoS в редакторе объектов групповой политики:

- Политика QoS в Конфигурация компьютера\Параметры Settings\QoS политика применяется к компьютерам независимо от пользователя, вошедшего в систему. Обычно используется компьютер политики QoS для компьютеров сервера.

- Политика QoS в политику Settings\QoS Конфигурация пользователя применяется к пользователям, вошедшим в систему, независимо от того, какой компьютер они вошли в систему.

#### <a name="to-create-a-new-qos-policy-with-the-qos-policy-wizard"></a>Для создания новой политики QoS, с помощью мастера политики QoS

-   В редакторе объектов групповой политики щелкните правой кнопкой мыши любой из **политики QoS** узлов, а затем щелкните **создать новую политику **.

### <a name="wizard-page-1---policy-profile"></a>Страница мастера 1 — "Профиль политики"

На первой странице мастера QoS политики можно указать имя политики и настроить как QoS элементы управления исходящего сетевого трафика.

#### <a name="to-configure-the-policy-profile-page-of-the-qos-based-policy-wizard"></a>Чтобы настроить на странице "Профиль политики" мастера QoS на основе политики

1. В **имя политики**, введите имя для политики QoS. Имя должно однозначно идентифицировать политику.

2. При необходимости используйте **укажите значение DSCP** включить назначение DSCP и затем задать значение DSCP в диапазоне от 0 до 63.

3. При необходимости используйте **укажите частоту глушителя** для включения регулировки трафика и настройки ограничения скорости. Ограничение скорости должно быть больше 1, и можно указать единицы либо килобайт в второй \(KBps\) либо мегабайт в второй \(MBps\).

4. Нажмите кнопку **Далее**.

### <a name="wizard-page-2---application-name"></a>Страница мастера 2 - имя приложения

На второй странице мастера QoS политики может применить политику ко всем приложениям, для определенного приложения, как его имя исполняемого файла, путь и имя приложения, или к идентифицируемому по HTTP серверных приложений, которые обрабатывают запросы для определенного URL-адреса.

- **Все приложения** указывает, что параметры управления трафиком на первой странице мастера QoS политики применяются ко всем приложениям.

- **Только приложения с именем исполняемого файла** указывает, что параметры управления трафиком на первой странице мастера QoS политики для конкретного приложения. Исполняемый файл имя и должен заканчиваться с расширением .exe.

- **Только HTTP-сервера приложений отвечает на запросы для этого URL-адреса** указывает, что параметры управления трафиком на первой странице мастера QoS политики применяются для определенных серверов приложений HTTP только.

Дополнительно можно указать путь к приложению. Чтобы указать путь приложения, будет включите путь с имени приложения. Путь может содержать переменные среды. Пример: моему Path\MyApp.exe %ProgramFiles%\My или c:\program files\my application path\myapp.exe.

>[!NOTE]
>Путь к приложению, не могут включать путь, который разрешает символические ссылки.

URL-адрес должен соответствовать [приведено](http://tools.ietf.org/html/rfc1738), в виде `http[s]://<hostname\>:<port\>/<url-path>`. Можно использовать подстановочные знаки, `‘*’`, для `<hostname>`и/или `<port>`, например `http://training.\*/, https://\*.\*`, но подстановочные знаки не может указать подстрокой `<hostname>`или `<port>`.

Другими словами, ни `http://my\*site/`ни `https://\*training\*/`является допустимым. 

Кроме того, вы можете проверить **включают подкаталоги и файлы** выполнять сравнение на все подкаталоги и файлы следующий URL-адрес. Например, если этот флажок установлен, и URL-адрес `http://training`, политики QoS будет считать, что запросы` http://training/video` хорошо подходит.

#### <a name="to-configure-the-application-name-page-of-the-qos-policy-wizard"></a>Настройка имени приложения на странице мастера политики QoS

1. В **применение политики к**, выберите пункт **все приложения** или **только к приложениям с именем исполняемого файла **.

2. При выборе **только к приложениям с именем исполняемого файла**, укажите имя исполняемого файла с расширением .exe.

3. Нажмите кнопку **Далее**.

### <a name="wizard-page-3---ip-addresses"></a>Страница мастера 3 - IP-адресов

На третьей странице мастера QoS политики можно указать условия IP адрес для политики QoS, включая следующие:

- Ко всем исходным IPv4- или IPv6-адреса или адреса определенным исходным IPv4- или IPv6

- Все назначения адресов IPv4 или IPv6 или конкретных целевых IPv4 или IPv6-адресов

При выборе **только для следующих IP-адрес источника** или **только для IP-адрес назначения следующие**, необходимо указать одно из следующих действий:

- Адрес IPv4, таких как `192.168.1.1`

- Префикс IPv4-адреса, с использованием нотации длины префикса сети, таких как `192.168.1.0/24`

- Адреса IPv6, таких как `3ffe:ffff::1`

- IPv6-адресов префикс, таких как `3ffe:ffff::/48`

При выборе вариантов **только для следующих IP-адрес источника** и **только для IP-адрес назначения следующие**, адреса или префикса адресов должны быть либо IPv4 - или IPv6-на основе.

Если указан URL-адрес HTTP-сервера приложений на предыдущей странице мастера, вы заметите, что исходный IP-адрес для политики QoS на этой странице мастера отображается серым. 

Это справедливо, так как адрес сервера HTTP IP-адрес источника, и он не настраивается здесь. С другой стороны можно настроить политики, указав IP-адрес назначения. Это позволяет создавать различные политики для разных клиентов с помощью того же приложения сервера HTTP.

#### <a name="to-configure-the-ip-addresses-page-of-the-qos-policy-wizard"></a>Настройка IP-адреса на странице мастера политики QoS

1. В **применение политики к** (исходные) выберите **любой исходный IP-адрес** или **только для следующих IP-адрес источника **.

2. Если вы выбрали **только следующие IP-адреса источника**, укажите IPv4- или IPv6-адрес либо префикс.

3. В **применение политики к** (конечные) выберите **любым конечным адресам** или **только для следующих IP-адрес назначения.**

4. Если вы выбрали **только для следующих IP-адрес назначения**, укажите IPv4 или IPv6-адрес либо префикс, соответствующий типу адреса или префикса, указанного для исходного адреса.

5.  Нажмите кнопку **Далее**.  

### <a name="wizard-page-4---protocols-and-ports"></a>Страница мастера 4 - протоколы и порты

На четвертой странице мастера политики QoS можно указать типы трафика и порты, управление которыми осуществляется с помощью параметров на первой странице мастера. Можно указать:  
-   Трафик TCP, UDP или оба типа трафика  

-   Все исходные порты, диапазон исходных портов или определенный исходный порт

-   Все конечные порты, диапазон конечные порты или определенный конечный порт.  

#### <a name="to-configure-the-protocols-and-ports-page-of-the-qos-policy-wizard"></a>Чтобы настроить протоколы и порты страницу мастера политики QoS

1. В **выберите протокол, применяется политика QoS в**выберите **TCP**, **UDP**, или **TCP и UDP **.

2. В **укажите номер исходного порта**выберите **любой исходный порт** или **номер исходного порта **.

3. Если вы выбрали **номер исходного порта**, введите номер порта от 1 до 65535.

     При необходимости можно указать диапазон портов в формате «*низкий*:*высокой*, «где *низкий* и *высокой* представляют нижней и верхней границам диапазона портов, включительно. *Низкая* и *высокой* каждого должны представлять собой числа от 1 до 65535. Не должно быть пробела между символом двоеточия (:) и числами.

4. В **укажите номер порта назначения**выберите **любой порт назначения** или **номер целевого порта**.

5. Если вы выбрали **номер целевого порта** на предыдущем шаге, введите номер порта от 1 до 65535.

Чтобы завершить создание новой политики QoS, нажмите кнопку **Готово** на **протоколы и порты** мастера политики QoS. После завершения новая политика QoS появится в списке в области сведений редактора объектов групповой политики.  
  
Чтобы применить параметры политики QoS к пользователям или компьютерам, свяжите объект групповой Политики, в котором размещены политики QoS на контейнеры доменных служб Active Directory, например домена, сайта или подразделения (OU).  
  
##  <a name="bkmk_editpolicy"></a>Просмотр, изменение или удаление политики QoS

Политики QoS, мастер описанные ранее соответствуют страницам свойств, которые отображаются при Просмотр и изменение свойств политики.  
  
### <a name="to-view-the-properties-of-a-qos-policy"></a>Просмотр свойств политики QoS  
  
-   Щелкните правой кнопкой мыши имя политики в области сведений редактора объектов групповой политики, а затем нажмите кнопку **свойства**.  
  
     Редактор объектов групповой политики отображает страницу свойств со следующими вкладками:  
  
    -   "Профиль политики"  
  
    -   Имя приложения  
  
    -   IP-адресов  
  
    -   Протоколы и порты  
  
### <a name="to-edit-a-qos-policy"></a>Изменение политики QoS  
  
-   Щелкните правой кнопкой мыши имя политики в области сведений редактора объектов групповой политики, а затем нажмите кнопку **Правка существующей политики**.  
  
     Отображает редактор объектов групповой политики **изменение существующей политики QoS** диалоговое окно.  
  
### <a name="to-delete-a-qos-policy"></a>Удаление политики QoS  
  
-   Щелкните правой кнопкой мыши имя политики в области сведений редактора объектов групповой политики, а затем нажмите кнопку **удалить политику**.  
  
### <a name="qos-policy-gpmc-reporting"></a>Политика QoS отчетов консоли управления групповыми Политиками 

После применения различные политики QoS в организации может быть полезным, или необходимо периодически проверять применение политики. Обзор политик QoS для конкретного пользователя или компьютера можно просмотреть с помощью отчетов консоли управления групповыми Политиками.  
  
#### <a name="to-run-the-group-policy-results-wizard-for-a-report-of-qos-policies"></a>Чтобы запустить мастер результатов групповой политики для отчетов политик качества обслуживания  
  
-   В консоли управления групповыми Политиками, щелкните правой кнопкой мыши **результатов групповой политики** и выберите узел для параметра меню **мастер результатов групповой политики.**  
  
После создания результатов групповой политики щелкните **параметры** вкладку. На **параметры** вкладку, политики качества обслуживания можно найти в узлы «Политики Settings\QoS Конфигурация компьютера» и «Политики Settings\QoS Конфигурация пользователя».  
  
На **параметры** вкладку, политики QoS перечисляются по их именам политики QoS, с их значение DSCP, регулируемую скорость, условия политики и результирующего объекта групповой Политики, перечисленные в той же строке.  
  
Просмотр результатов групповой политики однозначно определяет результирующий объект групповой Политики. Несколько объектов групповой политики, пока политики QoS, с тем же именем политики QoS, применяется объект групповой Политики с более высоким приоритетом объекта групповой Политики. Это результирующий объект групповой Политики. Конфликтующие политики качества обслуживания (определенный по имя политики), подключенные к более низким приоритетом, объект групповой Политики не применяются. Обратите внимание, что определить приоритеты GPO политики QoS, развернутые в сайта, домена или Подразделения, соответствующим образом. После развертывания на уровне пользователя или компьютера [приоритете политики QoS](#BKMK_precedencerules) определить, какой трафик разрешено, блокируются.  
  
Политики QoS значение DSCP регулируемую скорость и условия политики, также отображаются в редакторе объектов групповой политики (GPOE)  
  
### <a name="advanced-settings-for-roaming-and-remote-users"></a>Дополнительные параметры для роуминга и удаленным пользователям  
С помощью политики QoS цель — для управления трафиком в корпоративной сети. В мобильных сценариях пользователей может отправлять трафик или отключить корпоративной сети. Поскольку политики QoS не играют роли во время за пределами корпоративной сети, политики QoS включены только в сетевые интерфейсы, которые подключены к предприятия для Windows 8, Windows 7 или Windows Vista.  
  
Например пользователь может подключаться свой портативный компьютер своей корпоративной сети через виртуальную частную сеть (VPN) из кафе. Для VPN физический сетевой интерфейс (таких как беспроводной) не будет применять политики качества обслуживания. Однако интерфейс VPN будет применено, так как он подключается к предприятия политики QoS. Если пользователь введет позже сети другой организации, которая не имеет отношение доверия Доменные службы Active Directory, политики качества обслуживания, не включается.  
  
Обратите внимание, что эти мобильных сценариях не применяются для рабочих нагрузок сервера. Например сервер с несколькими сетевыми адаптерами может находиться на границе сети предприятия. ИТ-отдел может заставить трафика регулирования политики QoS, egresses предприятия; Тем не менее. Этот сетевой адаптер, которое отправляет исходящий трафик не обязательно подключиться к корпоративной сети. По этой причине политики QoS всегда включена на всех сетевых интерфейсов на компьютере под управлением Windows Server 2012.  
  
> [!NOTE]
>  Выборочное включение применяется только для политики QoS, а не к QoS Дополнительные параметры, описанные далее в этом документе.  
  
### <a name="advanced-qos-settings"></a>Дополнительные параметры качества обслуживания

Дополнительные параметры качества обслуживания предоставляют дополнительные элементы управления для ИТ-администраторам управлять маркировка DSCP и потребления сетевого компьютера. Дополнительные параметры качества обслуживания применяются только на уровне компьютера, тогда как политики QoS можно применять на уровне компьютера и пользователя.

#### <a name="to-configure-advanced-qos-settings"></a>Чтобы настроить дополнительные параметры качества обслуживания

1.  Нажмите кнопку **Конфигурация компьютера**, а затем нажмите кнопку **параметры Windows в групповой политике**.
  
2.  Щелкните правой кнопкой мыши **политики QoS**, а затем нажмите кнопку **Дополнительные параметры качества обслуживания**.

     На следующем рисунке показаны два дополнительных вкладки Параметры качества обслуживания: **входящий трафик TCP** и **Переопределение пометки DSCP**.
  
> [!NOTE]
>  Дополнительные параметры качества обслуживания: параметры групповой политики на уровне компьютеров.
  
#### <a name="advanced-qos-settings-inbound-tcp-traffic"></a>Дополнительные параметры качества обслуживания: входящий трафик TCP

**Входящий трафик TCP** элементы управления пропускной способности TCP на стороне получателя, а политики QoS, влияют на исходящий трафик TCP и UDP. 

Задав пропускной способности на нижнем уровне на **входящий трафик TCP** вкладке TCP будет ограничивать объем объявлена окна получения TCP. Влияние этого параметра будут увеличивается пропускная и использование канала для подключений TCP с более высокой пропускной способности и задержки (задержку передачи). По умолчанию компьютеры под управлением Windows Server 2012, Windows 8, Windows Server 2008 R2, Windows Server 2008 и Windows Vista устанавливаются на уровень максимальной пропускной способности.
  
Приема TCP окно было изменено в Windows Server 2012, Windows 8, Windows Server 2008 R2, Windows Server 2008 и Windows Vista с предыдущих версий Windows. Предыдущие версии Windows ограниченные автонастройке окно TCP максимум 64 килобайт (КБ), в то время как Windows Server 2012, Windows 8, Windows Server 2008 R2, Windows Server 2008 и Windows Vista динамически размер автонастройке окно до 16 МБ. В элементе управления входящего трафика TCP можно управлять уровнем пропускной способности входящего установкой максимальное значение, до которого может увеличиться TCP-окно приема. Уровни соответствуют следующие максимальные значения. 
  
|Уровень пропускной способности входящего трафика|Максимум|  
|------------------------|-------|  
|0|64 КБ|
|1|256 КБ|
|2|1 МБ|
|3|16 МБ|

Размер окна может быть меньше или равно максимальному в зависимости от условий работы сети.

###### <a name="to-set-the-tcp-receive-side-window"></a>Настройка окна на стороне приема TCP

1. В редакторе объектов групповой политики, нажмите кнопку **политика локального компьютера**, нажмите кнопку **параметры Windows**, щелкните правой кнопкой мыши **политики QoS**, а затем нажмите кнопку **Дополнительные параметры качества обслуживания**.
  
2. В **пропускной способности получения TCP**выберите **настроить пропускную способность TCP получение**и выберите уровень пропускной способности, который вы хотите.

3.  Свяжите объект групповой Политики для Подразделения.

#### <a name="advanced-qos-settings-dscp-marking-override"></a>Дополнительные параметры качества обслуживания: переопределение пометки DSCP

Переопределение пометки DSCP ограничивает возможности приложения, чтобы указать — или «пометить» — DSCP значения, отличные от указанных в политике QoS. Указав, что приложения могут устанавливать значения DSCP, приложения могут устанавливать значения DSCP отличное от нуля. 

Указав **игнорировать**, приложения, использующие API-интерфейсы QoS будет иметь значениями DSCP, равным нулю, а только политики QoS могут устанавливать значения DSCP. 

По умолчанию компьютеры под управлением Windows Server 2016, Windows 10, Windows Server 2012 R2, Windows 8.1, Windows Server 2012, Windows 8, Windows Server 2008 R2, Windows Server 2008 и Windows Vista позволяют приложениям указывать значения DSCP; не переопределяются приложений и устройств, не использующих API-интерфейсы QoS.

##### <a name="wireless-multimedia-and-dscp-values"></a>Значения DSCP и беспроводной мультимедиа

[Wi-Fi Alliance](https://go.microsoft.com/fwlink/?LinkId=160769) установил сертификации для беспроводного мультимедиа \(WMM\), определяет четыре категории доступа, переданных \(WMM_AC\) приоритетов сетевого трафика на беспроводной сети Wi\-Fi. Категории доступа включают \ (в порядке наивысший к наименьшее priority\): голоса, видео, усилия лучший и фон; соответственно сокращение VO, VI, BE и черной печати. Спецификация WMM определяет, какие DSCP значения соответствуют с каждой из категорий четыре доступа:
  
|Значение DSCP|Категории доступа к WMM|
|----------|-------------------|
|48-63|Голосовых команд (VO)|
|32-47|Видео (VI)|
|24-31, 0-7|Все усилия (BE)|
|8-23|Фон (КБ)|

Можно создать политики QoS, использующих эти значения DSCP, чтобы гарантировать получение портативные компьютеры с Wi\-Fi Certified™ для беспроводных адаптеров WMM приоритета обработки, если они связаны с Wi\-Fi Certified WMM точек доступа.
  
### <a name="BKMK_precedencerules"></a>Правила приоритета политики QoS

Аналогично приоритеты объектов групповой Политики, политики QoS должны приоритете устранить конфликты при несколько политик QoS применяются к определенному набору трафика. Для исходящего трафика TCP или UDP можно применять только одна политика QoS одновременно, это означает, что политики QoS не воздействие, например, где бы суммировать регулировки скорости.

Как правило политики QoS, с наиболее соответствия условиям wins. Когда применены несколько политик качества обслуживания, правила делятся на три категории: уровне пользователя и уровне компьютера; приложения и quintuple сети; и среди сети quintuple.

По *сети пятиэлементный*, мы означают исходный IP-адрес, конечный IP-адрес, порт источника, порт назначения и протокола \(TCP/UDP\).  

 **Политика QoS на уровне пользователя имеет приоритет над политики QoS на уровне компьютера**

Это правило существенно облегчает управление сетевым администраторам объектов групповой политики QoS, особенно для пользователей на основе групповой политики. Например если сетевой администратор хочет определить политики качества обслуживания для группы пользователей, их можно просто создать и распространить объекта групповой Политики для этой группы. Они не нужно беспокоиться о компьютеры, которые эти пользователи вошли в систему и ли эти компьютеры могут иметь конфликтующие политики QoS, которые определены, потому, что при возникновении конфликта политики на уровне пользователя всегда имеет приоритет.

> [!NOTE]
>  Политики QoS уровень пользователя применяется только к трафику, который создан с помощью этого пользователя. Другие пользователи конкретного компьютера, и компьютер, сам по себе не будут распространяются все политики QoS, определенные для этого пользователя.

 **Спецификой приложения и имеют более высокий приоритет над quintuple сети**

Когда конкретного трафика соответствует несколько политик QoS, применяются конкретная политика. Среди политик, определите приложения политики, которая содержит путь к файлу отправляющего приложения считается более частное, чем другой политики, определяющий только имя приложения (не путь). Если несколько политик с приложениями по-прежнему применяются, правила очередности используйте пятиэлементный сети, чтобы найти лучше всего подходит.

Кроме того несколько политик QoS могут относиться к тот же трафик, указав неперекрывающихся условия. Между условия приложений и quintuple сети политики, который определяет приложение считается более конкретные и применяется. 

Например policy_A только указывает (app.exe) имя приложения и policy_B указывает конечный IP адрес 192.168.1.0/24. При этих политик QoS конфликтов \ (app.exe пересылает трафик в IP-адрес в диапазоне 192.168.4.0 или 24\), policy_A применяется.

 **Дополнительные спецификой имеет более высокий приоритет, в пределах сети пятиэлементный**

Для конфликтов политик в quintuple сети политики с условиями, наиболее соответствующего имеет приоритет. Например предположим, что policy_C указывает исходный IP-адресов, «все», конечный IP-адрес, адрес 10.0.0.1, порт источника «any», порт назначения «все», а также протокол «TCP». 

Далее предполагается, что policy_D указывает исходный IP-адресов, «все», конечный IP-адрес, адрес 10.0.0.1, порт источника «any», конечный порт 80, а также протокол «TCP». Затем policy_C и policy_D соответствует подключения к 10.0.0.1:80 назначения. Поскольку политики QoS применяет политику с наиболее конкретных условий соответствия, policy_D преимущество в этом примере.  
  
Однако политики качества обслуживания может быть одинаковое число условий. Например, несколько политик каждого указать только один (но не совпадают) часть quintuple сети. Среди quintuple сети указанном ниже порядке — из более высокого уровня, чтобы уменьшить приоритет

- Исходный IP-адрес

- IP-адрес назначения

- Порт источника

- Порт назначения

- Протокол (TCP или UDP)

В определенных условиях, например IP-адрес рассматривается более конкретные IP-адрес с более высоким приоритетом; например IP-адрес 192.168.4.1 относится более чем 192.168.4.0/24.

Разработка политик QoS как можно точнее для упрощения законодательным требованиям в организации, чтобы понять, какие политики вступили в силу.
