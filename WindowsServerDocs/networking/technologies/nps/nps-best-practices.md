# <a name="network-policy-server-best-practices"></a>Лучшие методики для сервера политики сети

>Область применения: Windows Server (канал точками годовой), Windows Server 2016

В этом разделе можно использовать для изучения советы и рекомендации по развертыванию и управлению \(NPS\) сервера политики сети.

В следующих разделах приводятся рекомендации по различные аспекты развертывания NPS.

## <a name="accounting"></a>Бухгалтерский учет

Ниже приведены рекомендации для ведения журнала сервера политики сети.

Существует два типа учета или ведение журнала на сервере политики сети.

- Ведение журнала событий для сервера политики сети. Ведение журнала событий для записи событий сервера политики сети можно использовать в журналы событий системы и безопасности. Используется в основном для аудита и устранения неполадок попытки подключения.

- Ведение журнала запросов учета и проверки подлинности пользователя. Можно записать в журнал запросов проверки подлинности и учета пользователей, в файлы журналов в формате текста или базы данных или вы можете войти хранимая процедура в базе данных SQL Server 2000. Ведение журнала запросов в основном используются для целей анализа и выставления счетов подключения, а также можно использовать в качестве средства анализа системы безопасности, предоставляя отслеживание несанкционированных действий злоумышленников.

Для создания наиболее эффективного использования NPS ведения журнала:

- Включите \(initially\) ведения журнала для проверки подлинности и учетных записей. Измените эти параметры, после вы определили, что является более подходящим для вашей среды.

- Убедитесь, что ведение журнала событий, настроен с емкости, которая является достаточным для ведения журналов.

- Создайте резервную копию всех файлов журнала на регулярной основе, так как они не могут быть заново после повреждения или удаления.

- Используйте атрибут класса RADIUS для отслеживания использования и упростить определение отделения или пользователя, с которых будет взиматься плата. Несмотря на то, что автоматически созданный класс атрибута является уникальным для каждого запроса, повторяющиеся записи может существовать в случаях, когда ответ на сервер доступа теряются и запрос снова. Возможно, потребуется удалить дубликатов запросов из журналов, чтобы точно отслеживать использование.

- Если серверы доступа к сети и прокси-серверы RADIUS периодически отправлять сообщения запроса вымышленные подключения для сервера политики сети, убедитесь, что сервер политики сети находится в сети, используйте **связь имени пользователя** параметр реестра. Этот параметр позволяет настроить сервер политики сети, чтобы автоматически отклонять такие запросы на подключение false без их обработки. Кроме того сервер политики сети не будет записывать транзакции с участием вымышленное имя пользователя в все файлы журналов, что облегчает интерпретировать журнала событий.

- Отключение отправки уведомлений NAS. Вы можете отключить переадресации кнопки "Пуск" и stop сообщения серверы доступа к сети (NAS), чтобы члены удаленных RADIUS-сервера группы, настроен на сервере политики сети. Дополнительные сведения см. в разделе [отключение переадресации уведомлений NAS](nps-disable-nas-notifications.md).

Дополнительные сведения см. в разделе [Настройка сети политики сервера учета](nps-accounting-configure.md).

- Для обеспечения отказоустойчивости и избыточности с помощью ведения журнала SQL Server, поместите два компьютера под управлением SQL Server в разных подсетях. Использовать SQL Server **мастер создания публикаций** для настройки базы данных репликации между двумя серверами. Дополнительные сведения см. в разделе [технической документации SQL Server](https://msdn.microsoft.com/library/ms130214.aspx) и [репликации SQL Server](https://msdn.microsoft.com/library/ms151198.aspx).

## <a name="authentication"></a>Проверка подлинности

Ниже приведены рекомендации для проверки подлинности.

- Используйте методы проверки подлинности на основе сертификатов, например \(PEAP\) защищенный расширяемый протокол проверки подлинности и \(EAP\) расширяемого протокола проверки подлинности для строгой проверки подлинности. Не следует использовать методы проверки подлинности только для пароля, так как они уязвимы для различных атак и не являются надежными. Для безопасной проверки подлинности беспроводной PEAP\-MS\-CHAP v2 рекомендуется использовать, так как сервер политики сети подтверждает свою подлинность для беспроводных клиентов с помощью сертификата сервера, когда пользователи предоставлять свои удостоверения свое имя пользователя и пароль.  Дополнительные сведения об использовании сервера политики сети в развертывании беспроводной сети см. в разделе [развертывание пароль 802.1 X с проверкой подлинности беспроводного доступа на основе](https://technet.microsoft.com/windows-server-docs/networking/core-network-guide/cncg/wireless/a-deploy-8021x-wireless-access).
- Развернуть собственный центр сертификации \(CA\) с Active Directory&reg; \(AD CS\) службы сертификации при использовании методов строгой проверки подлинности на основе сертификатов, например PEAP и EAP, которые требуется использовать сертификат сервера на серверах политики сети. Также можно использовать ЦС для регистрации сертификатов компьютера и сертификатов пользователей. Дополнительные сведения о развертывании сертификатов сервера для серверов политики сети и удаленного доступа см. в разделе [развертывание сертификатов сервера для проводных сетей 802.1 X и беспроводных развертываний](https://technet.microsoft.com/windows-server-docs/networking/core-network-guide/cncg/server-certs/deploy-server-certificates-for-802.1x-wired-and-wireless-deployments).

## <a name="installation-suggestions"></a>Советы по установке

Ниже приведены рекомендации для установки сервера политики сети.

- Перед установкой сервера политики сети, установите и проверьте все серверы доступа к сети, с помощью локальных методов проверки подлинности, прежде чем настраивать их как RADIUS-клиенты на сервере политики сети.

- После установки и настройки сервера политики сети, сохранить конфигурацию с помощью команды Windows PowerShell [экспорта NpsConfiguration](https://technet.microsoft.com/en-us/library/jj872749.aspx). Сохранение конфигурации сервера политики сети с помощью следующей команды, каждый раз при повторной настройке сервера политики сети.

>[!CAUTION]
>- Экспортированный файл конфигурации NPS содержит незашифрованные общие секреты клиентов RADIUS и членов удаленных групп серверов RADIUS. Таким образом Убедитесь, что файл сохранен в безопасном расположении.
>- Процесс экспорта не включает в экспортированном файле параметров ведения журнала для Microsoft SQL Server. При импорте экспортированный файл на другой сервер политики сети, необходимо вручную настроить ведения журнала SQL Server на новом сервере.

## <a name="performance-tuning-nps"></a>Настройка сервера политики сети производительности

Ниже приведены рекомендации по повышению производительности, настройки сервера политики сети.

- Для оптимизации времени отклика проверки подлинности и авторизации сервера политики сети и свести к минимуму сетевой трафик, установка сервера политики сети на контроллере домена.

- При использовании \(UPNs\) универсальных имена участников или доменов Windows Server 2008 и Windows Server 2003, сервер политики сети использует глобальный каталог для проверки подлинности пользователей. Чтобы минимизировать время, необходимое для этого, установите NPS на сервер глобального каталога или сервер, который находится в той же подсети, что и сервер глобального каталога.

- Если настроенные группы внешних RADIUS-серверов и, в NPS политики запросов на подключение, снимите **записи учетных данных на серверах в следующие группы внешних серверов RADIUS** флажок, эти группы по-прежнему отправляются запуск \(NAS\) сервера доступа к сети и остановите сообщения уведомления. Это создает излишнего сетевого трафика. Чтобы исключить этот трафик, отключите уведомлений NAS для отдельных серверов в каждой группы внешних RADIUS-серверов, сняв **пересылать сети начала и остановки сервера** флажок.

## <a name="using-nps-in-large-organizations"></a>С помощью сервера политики сети в крупных организациях

Ниже приведены рекомендации по использованию NPS в крупных организациях.

- При использовании политики сети для ограничения доступа для всех, но некоторые группы, создайте универсальную группу для всех пользователей, которым вы хотите разрешить доступ и создайте сетевой политики, которая предоставляет доступ для этой универсальной группы. Не помещайте всех пользователей непосредственно в универсальную группу, особенно в том случае, если у вас есть большое количество их в сети. Вместо этого создайте отдельные группы, которые являются членами универсальные группы и добавить пользователей в эти группы.

- Для ссылки на пользователей, по возможности используйте имя участника-пользователя. У пользователя может быть одно основное имя пользователя независимо от того, членства в домене. Это обеспечивает масштабируемость, что может быть необходимым в организациях с большим числом доменов.

- Если вы установили \(NPS\) сервера политики сети на компьютере, отличном от контроллера домена и NPS-сервер принимает большое количество запросов проверки подлинности в секунду, может повысить производительность сервера политики сети, увеличив количество одновременных проверки подлинности между сервером политики сети и контроллера домена. ## <a name="security-issues"></a>Проблемы безопасности

Ниже приведены рекомендации для сокращения вопросов безопасности.

При удаленном администрировании сервера политики сети не передают важные или конфиденциальные данные (например, общие секреты или пароли) через сеть в виде обычного текста. Существует два рекомендуемые методы для удаленного администрирования серверов политики сети.

- Использование служб удаленных рабочих столов для доступа к серверу политики сети. При использовании служб удаленных рабочих столов, данные не отправляются между клиентом и сервером. Только пользовательский интерфейс сервера (например, рабочий стол операционной системы и образа консоли NPS) отправляются на клиенте служб удаленных рабочих столов, названный подключение к удаленному рабочему столу в Windows&reg; 10. Клиент отправляет клавиатуры и мыши ввода, который обрабатывается локально, у которых включены службы удаленных рабочих столов. При входе пользователей служб удаленных рабочих столов, они могут просматривать только их отдельных клиентских сеансов, которые управляются сервера и не зависят друг от друга. Кроме того подключения к удаленному рабочему столу предоставляет 128-битовое шифрование между клиентом и сервером.

- Используйте безопасность протокола IP (IPsec) для шифрования конфиденциальных данных. Можно использовать протокол IPsec для шифрования обмена данными между сервером политики сети и удаленного клиентского компьютера, используемая для администрирования сервера политики сети. Для удаленного администрирования сервера можно установить [удаленного средства администрирования сервера для Windows 10](https://www.microsoft.com/download/details.aspx?id=45520) на клиентском компьютере. После установки используйте Microsoft Management Console (MMC), чтобы добавить оснастку сервера политики сети в консоли.

>[!IMPORTANT]
>Удаленный средства администрирования сервера для Windows 10 можно установить только на полного выпуска Windows 10 Professional и Windows 10 Корпоративная.

Дополнительные сведения о сервере политики сети см. в разделе [сервера политики сети (NPS)](nps-top.md).
