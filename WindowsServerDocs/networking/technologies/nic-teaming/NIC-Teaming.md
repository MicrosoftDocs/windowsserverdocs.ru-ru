## <a name="bkmk_over"></a>Обзор объединения сетевых КАРТ  
Объединение сетевых КАРТ позволяет сгруппировать между одной и двумя тридцать физические сетевые адаптеры Ethernet в один или несколько программных виртуальных сетевых адаптеров. Эти виртуальные сетевые адаптеры обеспечивают высокую производительность и отказоустойчивость в случае сбоя сетевого адаптера.  
  
Объединение Сетевых карт члена сетевых адаптеров необходимо установить в том же компьютере, необходимо поместить в группу.  
  
> [!NOTE]  
> Объединение Сетевых карт, который содержит только один сетевой адаптер не может предоставить Балансировка нагрузки и отказоустойчивость; Однако с помощью одного сетевого адаптера, можно использовать объединение сетевых КАРТ для разделения трафика сети при использовании также виртуальные локальные сети (VLAN).  
  
При настройке сетевых адаптеров в группу сетевых КАРТ, они подключены NIC объединения решения распространенных основных, который предоставляет один или несколько виртуальных адаптеров (также называется группы сетевых адаптеров [tNICs] интерфейсы группы) в операционную систему. Windows Server 2016 поддерживает до 32 интерфейсы команды каждой команды. Существуют разнообразные алгоритмов распределения исходящего трафика между сетевыми (загрузки).  
  
Объединение Сетевых карт с несколькими tNICs показана на следующем рисунке.

Кроме того можно подключиться к объединенных сетевых адаптеров, к тому же коммутатору, или к разным коммутаторам. Если сетевых адаптеров подключаться к разным коммутаторам, как коммутаторы должны поддерживать в той же подсети.  
  
## <a name="bkmk_avail"></a>Доступность объединение сетевых КАРТ  
Объединение сетевых КАРТ — доступны во всех версиях Windows Server 2016. Кроме того можно использовать команды Windows PowerShell, удаленного рабочего стола и средств удаленного администрирования сервера для управления объединение сетевых КАРТ с компьютеров под управлением клиентской операционной системы, на которой средства поддерживаются.  
  
## <a name="bkmk_nics"></a>Поддерживаемые и неподдерживаемые сетевых адаптеров для объединения сетевых КАРТ  
Можно использовать любой сетевой Адаптер Ethernet, прошло проверку квалификации оборудования для Windows и логотип (WHQL тесты) в группу Сетевых карт в Windows Server 2016.  
  
Нельзя разместить следующих сетевых адаптеров в группу Сетевых карт.  
  
-   Hyper-V виртуальных сетевых адаптеров, которые портов виртуального коммутатора Hyper-V, сетевых адаптеров в виде раздела узла.  
  
    > [!IMPORTANT]  
    > Виртуальных сетевых адаптеров Hyper-V, которые предоставляются раздела узла (виртуальные сетевые карты), не должны находиться в группе. Объединение из виртуальную сетевую карту, в разделе узла не поддерживается в любой конфигурации или в сочетании. При возникновении сбоя сети попытки виртуальные сетевые карты группы может привести к полной потере связи.  
  
-   Сетевой адаптер с отладкой ядра (KDNIC).  
  
-   Сетевые адаптеры, которые используются для сетевой загрузки.  
  
-   Сетевые адаптеры, использующие технологии, отличных от Ethernet, например беспроводной глобальной сети, WLAN, Wi-Fi, Bluetooth и Infiniband, включая протокол через сетевые адаптеры Infiniband (IPoIB).  
  
## <a name="bkmk_compat"></a>Совместимость объединение сетевых КАРТ  
Объединение сетевых КАРТ совместимо со всех сетевых технологий в Windows Server 2016 со следующими исключениями.  
  
-   **Единой корневой виртуализации ввода-вывода (SR-IOV)**. Для SR-IOV данные доставляются непосредственно на сетевой Адаптер без передачи через стек сети (в главной операционной системой, в случае виртуализации). Таким образом не может проверять или перенаправлять данные на другой маршрут в объединении Сетевых карт.  
  
-   **Основной узла качества обслуживания (QoS)**. При задании политики качества обслуживания собственного или системы сервера виртуальных машин и этих политик вызова ограничения минимальной пропускной способности, общую пропускную способность для Сетевых карт будет меньше, чем было бы без политики пропускной способности на месте.  
  
-   **TCP Chimney**. TCP Chimney не поддерживается при объединении сетевых КАРТ, поскольку разгрузку TCP Chimney весь сетевой стек непосредственно на сетевой адаптер.  
  
-   **802. 1 проверка подлинности x**. 802.1 проверка подлинности X не должен использоваться с Объединением сетевых карт. Некоторые ключи не допускают настройку проверки подлинности 802.1 X и объединение сетевых КАРТ на тот же порт.  
  
Сведения об использовании объединения сетевых КАРТ в виртуальных машин (ВМ), работающих на узле Hyper-V см. в разделе [объединение сетевых КАРТ в виртуальных машин & #40; Виртуальные машины & #41;](../../technologies/nic-teaming/../../technologies/nic-teaming/NIC-Teaming-in-Virtual-Machines--VMs-.md).  
  
## <a name="bkmk_vmq"></a>Объединение сетевых КАРТ и очереди виртуальных машин (Vmq)  
VMQ и объединение сетевых КАРТ работают вместе; VMQ следует включить в любое время включить Hyper-V. В зависимости от режима конфигурации коммутатора и алгоритм распределения нагрузки объединение сетевых КАРТ будет либо предоставлять возможности VMQ к коммутатору Hyper-V, содержащие число очередей, необходимо наименьшее количество очередей, поддерживаемых любой адаптер, в группе (режим очередей мин.) или общее количество очередей, которые доступны для всех членов группы (режим сумма из очереди).  
  
В частности Если сотрудникам в независимых от коммутатора объединение режим и распределения нагрузки установлен режим порта Hyper-V или динамического режима, а затем числа очередей отчет представляет собой сумму всех очередей, доступных от членов группы (режима сумма из очереди). в противном случае число очередей, о которых сообщает является наименьшее количество очередей, поддерживаемых любого члена группы (режим очередей мин.).  
  
Вот почему:  
  
-   Когда команда независимых коммутатора в режиме порта Hyper-V или динамического входящего трафика для порта коммутатора Hyper-V (VM) всегда будут поступать в один и тот же элемент группы. Узел можно прогнозирования и управление которым приема трафика для определенной виртуальной Машины, объединение сетевых КАРТ может быть более продумайте, какие очереди VMQ, чтобы распределить на члена определенной группы. Объединение сетевых КАРТ, работа с коммутатором Hyper-V, включен только один член группы VMQ для виртуальной Машины и знать, что входящий трафик будет достигнуть данной очереди.  
  
-   Когда команда работает в любой коммутатора зависимые режиме (статическая поддержка групп или объединение LACP), переключатель, который подключен команда управляет распространением входящий трафик. Программное обеспечение объединение сетевых КАРТ узла невозможно предсказать, какие группы члена получит входящего трафика для виртуальной Машины и, возможно, коммутатор распределяет трафик для виртуальной Машины для всех членов группы. Как результат объединения программного обеспечения, работа с коммутатором Hyper-V, программы очередь для виртуальных Машин на каждый член группы, не только один член группы.  
  
-   Если команда находится в режиме независимые от коммутатора и использует нагрузки распространения адрес хэш-алгоритм, входящий трафик будет всегда иметь на одной сетевой КАРТОЙ (на основной член группы) -, все из них на только что одному члену группы. Поскольку других сотрудников, не входящий трафик, который они получить программировать с помощью же очереди в основном члене, чтобы в случае сбоя основного члена любой другой участник команды можно использовать для забрать входящего трафика и очереди уже на своем месте.  
  
Большинство сетевые адаптеры имеют очередей, которые можно использовать для получения масштабирование на стороне (RSS) или VMQ, но не оба одновременно. Некоторые параметры VMQ кажутся параметры для очередей RSS, но действительно параметры на универсальный очередей, которые RSS и VMQ в зависимости от того, какая из этих функций используется в настоящее время используется. Каждый сетевой Адаптер имеет, в нем дополнительные свойства, значения * RssBaseProcNumber и \*MaxRssProcessors. Ниже приведены несколько параметров VMQ, которые обеспечивают более высокую производительность системы.  
  
-   В идеале каждый сетевой Адаптер должен иметь * RssBaseProcNumber равным четном числе больше или равен 2 (два). Это вызвано первого физического процессора Core 0 (логических процессоров 0 и 1), обычно делает большую часть обработка системы, поэтому сетевой обработки следует steered от этого физического процессора. (Некоторые архитектур не имеют два логических процессоров на физическом процессоре, поэтому для таких машин базового процессор должен быть больше или равно 1. Если в сомнительных случаях предполагается на узле используется 2 логического процессора по архитектурам физического процессора.)  
  
-   Если команда находится в режиме сумма из очередей процессоров участников группы необходимо определить, насколько это возможно, без перекрытия. Например в узле 4-ядерный (8 логических процессоров) с группой двумя сетевыми адаптерами 10 Гбит / с, можно задать первый из них для использования базового процессора 2 и 4 ядра; второй будет установлено в используйте базового процессора 6 и 2 ядра.  
  
-   Если команда находится в режиме мин очереди процессора наборов, используемых для участников группы должны совпадать.  
  
## <a name="bkmk_hnv"></a>Объединение сетевых КАРТ и Hyper-V виртуализация сети (HNV)  
Объединение сетевых КАРТ полностью совместима с виртуализацией сети Hyper-V (HNV).  Система управления HNV сведения объединение сетевых КАРТ драйвер, который позволяет распределять нагрузку способом, который оптимизирован для трафика HNV объединение сетевых КАРТ.  
  
## <a name="bkmk_live"></a>Объединение сетевых КАРТ и динамической миграции  
Объединение сетевых КАРТ в виртуальных машинах не влияет на динамическую миграцию. Те же правила существует для динамической миграции ли объединение сетевых КАРТ настроен на виртуальной машине.
