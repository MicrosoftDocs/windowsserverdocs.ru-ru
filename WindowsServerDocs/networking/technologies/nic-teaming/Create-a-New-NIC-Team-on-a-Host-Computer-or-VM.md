# <a name="create-a-new-nic-team-on-a-host-computer-or-vm"></a>Создание группы сетевых КАРТ на главном компьютере или виртуальной Машины

>Область применения: Windows Server (канал точками годовой), Windows Server 2016

В этом разделе сведения о конфигурации объединения сетевых КАРТ, чтобы понять параметры, необходимые при настройке новой группы сетевых КАРТ. Этот раздел содержит следующие разделы.  
  
-   [Выбрав режим объединения](#bkmk_teaming)  
  
-   [Выбор режима балансировки нагрузки](#bkmk_lb)  
  
-   [Выбор параметра ожидания адаптера](#bkmk_standby)  
  
-   [С помощью свойства интерфейса основной группы](#bkmk_primary)  
  
> [!NOTE]  
> Если вы уже понять такие элементы конфигурации, можно использовать следующие процедуры Настройка объединения сетевых КАРТ.  
>   
> -   [Создание группы сетевых КАРТ в виртуальной Машине](../../technologies/nic-teaming/Create-a-New-NIC-Team-in-a-VM.md)  
> -   [Создание группы сетевых КАРТ](../../technologies/nic-teaming/Create-a-New-NIC-Team.md)  
  
При создании новой группы сетевых КАРТ, необходимо настроить следующие свойства объединения сетевых Адаптеров.  
  
-   Имя группы  
  
-   Адаптеры группы  
  
-   Режим объединения  
  
-   Режима балансировки нагрузки  
  
-   Резервный адаптер  
  
Можно также при необходимости Настройка интерфейса основной группы и настройки виртуальной локальной сети (VLAN) номера.  
  
Эти свойства объединения сетевых Адаптеров, отображаются на следующем рисунке, который содержит примеры значений для некоторые свойства объединения сетевых Адаптеров.

## <a name="bkmk_teaming"></a>Выбрав режим объединения  
Возможные режим поддержки групп **зависит от коммутатора**, **статическая поддержка групп**, и **агрегирования управления протокола агрегированием каналов LACP**. Статическая поддержка групп и LACP есть зависимые коммутатора режима. Для достижения наивысшей производительности группы сетевых КАРТ, все три режима карт рекомендуется использовать режим балансировки нагрузки динамического распределения.  
  
**Независимые коммутатора**  
  
Переключение независимом режиме, коммутатора или коммутаторы, к которым подключены членов группы сетевых КАРТ не уведомлены о присутствии группы сетевых КАРТ, не определение способа распределения сетевого трафика для членов группы сетевых КАРТ — вместо этого группы сетевых КАРТ распределяет входящий сетевой трафик между членами группы сетевых КАРТ.  
  
При использовании зависит от коммутатора режима с помощью динамического распределения нагрузки трафика сети распространяется на основании хэш адрес TCP-порты, как изменено динамического алгоритм балансировки нагрузки. Алгоритм балансировки нагрузки, динамических перераспределяет потоков для оптимизации использования пропускной способности член группы, чтобы передачи отдельных потока можно перемещать из одного члена группы active в другое. Алгоритм учитывает небольшая вероятность, что распространение трафика может привести к вне очереди доставки пакетов, поэтому он выполняет действия, чтобы свести к минимуму эту возможность.  
  
**Зависимые от коммутатора**  
  
Переключение зависимые режимах, коммутатора членов группы сетевых КАРТ подключенный определяет, как распределять входящего сетевого трафика между членами группы сетевых КАРТ. Коммутатор имеет полный независимость для определения способа распределения сетевого трафика между членами группы сетевых КАРТ.  
  
> [!IMPORTANT]  
> Зависимые объединения коммутатора необходимо, чтобы все члены группы подключены к одному физическому коммутатору или несколькими корпуса переключения, предоставляет идентификатор коммутатора среди нескольких корпуса.  
  
Статические карт необходимо вручную настроить коммутатора и узла, чтобы определить, какие ссылки формируют группу. Так как это статически настроенных решений, существует нет никакого дополнительного протокола, чтобы помочь коммутатора и узла в обнаружении неправильно подключенных кабелей или других ошибок, которые могут привести к сбою группы адаптеров для выполнения. Этот режим обычно поддерживается коммутаторами серверного класса.  
  
В отличие от статических объединение режим объединения LACP динамически определяет ссылки, которые соединены между узлом и коммутатора. Динамические подключения позволяет автоматически команды, а в теории редко, но на практике также увеличивать и уменьшать группу путем передачи или приема пакетов LACP от однорангового объекта. Все коммутаторы серверного класса поддерживают LACP и всех требуется оператор сети для административно включения поддержки протокола LACP на порт коммутатора. При настройке режим поддержки групп данных по протоколу LACP объединение сетевых КАРТ всегда работает в активный режим LACP с короткие таймера.  Не доступен в настоящее время для изменения таймера или изменить режим LACP.  
  
При использовании режимов, зависимых коммутатора с динамическое распределение сетевой нагрузки трафика распространяется на основании хэш адрес TransportPorts как изменено динамического алгоритм балансировки нагрузки.  Алгоритм балансировки нагрузки динамического перераспределяет потоков для оптимизации использования пропускной способности члена группы. Передачи отдельных потока можно перемещать из одного активного члена группы в другой части динамического распределения. Алгоритм учитывает небольшая вероятность, что распространение трафика может привести к вне очереди доставки пакетов, поэтому он выполняет действия, чтобы свести к минимуму эту возможность.  
  
Как все зависимые конфигурации коммутатора, параметр определяет, как распределения входящего трафика между участниками группы.  Коммутатор ожидает выполнения разумные задания распределения трафика между членами группы, но его завершения независимость, чтобы определить, как это сделано.  
  
## <a name="bkmk_lb"></a>Выбор режима балансировки нагрузки  
Параметры для режима распределения нагрузки, являются **хэш адрес**, **порта Hyper-V**, и **динамическое**.  
  
**"Хэш адрес"**  
  
Этот режим балансировки нагрузки создает хэш на основе адрес компонентов пакета. Затем назначает пакеты, имеющие это значение хэша, одному из доступных адаптеров. Обычно этого механизма достаточно для создания разумного баланса между доступных адаптеров.  
  
Windows PowerShell можно использовать для указания значений для следующих компонентов функции хеширования.  
  
-   TCP-порты источника и назначения исходный и конечный IP-адреса серверов. Это значение по умолчанию при выборе **хэш адрес** как режим балансировки нагрузки.  
  
-   Исходный и конечный IP-адреса только.  
  
-   Конечный MAC-адреса источника и только.  
  
Хэш порты TCP создает самый детальный распределение потоков трафика, из-за которой меньше потоков, которые могут независимо перемещаться между членами группы сетевых КАРТ. Тем не менее, нельзя использовать для трафика, не TCP хэш портов TCP или UDP, или где TCP и UDP-порты скрыты от стека — например, с трафика, защищенного с помощью IPsec. В этих случаях хэш автоматически использует хэш IP адрес или, если трафик идет не по IP-трафика, используется хэш адрес MAC.  
  
**Порт Hyper-V**  
  
С помощью режима порта Hyper-V для группы сетевых Адаптеров, которые настроены на узлах Hyper-V обеспечивает определенное преимущество. Поскольку виртуальные машины имеют независимые MAC-адреса, MAC-адрес виртуальной Машины - или порт, к которому подключен ВМ на виртуальном коммутаторе Hyper-V - может быть основой которых для разделения сетевого трафика между членами группы сетевых КАРТ.  
  
> [!IMPORTANT]  
> Нельзя настроить группы сетевых Адаптеров, можно создать в виртуальные машины с помощью режима балансировки нагрузки порта Hyper-V. Используйте хэш адрес режима балансировки нагрузки вместо.  
  
Поскольку соседний коммутатор всегда видит определенный MAC-адреса на один порт, коммутатор распределяет нагрузку входящих (трафик с коммутатора на узел) по нескольким каналам, исходя из MAC (виртуальных Машин MAC) адреса. Это особенно полезно при использовании очереди виртуальной машины (Vmq), так как очередь может размещаться на определенных сетевых КАРТ, ожидающей прием трафика.  
  
Тем не менее, если узел имеет только несколько виртуальных машин, этот режим может быть детальное, чтобы получить хорошее распределение нагрузки. Этот режим также всегда будет ограничивать отдельную виртуальную Машину (т. е. трафика через порт одним переключателем) пропускной способности, которая доступна на одном интерфейсе. Объединение сетевых КАРТ использует порт коммутатора Hyper-V виртуальной как идентификатор вместо MAC-адрес источника, поскольку в некоторых случаях виртуальная машина может быть настроен с более одного MAC-адреса на порте коммутатора.  
  
**Динамические**  
  
Этот режим балансировки использует преимущества каждой из двух режимов и объединяет их в один режим:  
  
-   Исходящий загружает распространяются на основании хэша TCP-портов и IP-адресов.  Динамический режим также балансировку нагрузки в режиме реального времени, заданного исходящего потока может перемещаться между членами группы.  
  
-   Входящий загружает распространяются в так же, как режим порта Hyper-V.  
  
Динамически распределяются на основе на концепции flowlets исходящих загружает в этом режиме. Аналогично человеческой речи естественным вставок концов словами и фразами, потоки TCP (потоки связи TCP) также разрывами встречающихся естественным образом. Часть потока TCP между двумя вставок называется flowlet.  
  
Когда алгоритм динамического режима обнаруживает, что границы flowlet обнаружена - например при произошла разрыв достаточно длины потока TCP - алгоритм автоматически балансировку потока на другой член группы, если это возможно.  В некоторых случаях алгоритм может также периодически заново сбалансировать потоки, которые не содержат все flowlets. По этой причине сопоставления между члена группы и потока TCP можно изменить в любое время, как работает динамического алгоритм балансировки нагрузки для балансировки сетевой нагрузки членов группы.  
  
Ли команда настроен зависит от коммутатора или один из зависимых коммутатора режимов, рекомендуется использовать режим динамического распределения для повышения производительности.  
  
Существует исключение, это правило при имеет только два члена группы Сетевых карт, настроен в режиме зависит от коммутатора и имеет активный и режим ожидания активном режиме, с одной сетевой КАРТОЙ active, а другой — настроен для ждущего режима. При такой конфигурации группы сетевых КАРТ распространения хеша адрес обеспечивает производительность, чем динамическое распространения.  
  
## <a name="bkmk_standby"></a>Выбор параметра ожидания адаптера  
Параметры адаптера режим ожидания: None (все адаптеры Active) или Выбор конкретного сетевого адаптера в группе Сетевых карт, выступающий в качестве к адаптеру ждущий режим, если при этом другие члены группы не выбрано Active. При настройке сетевой КАРТЫ как адаптер ждущий режим, отправляется или обрабатываемых адаптера, если не сетевой трафик и только в момент, при сбое активных сетевых КАРТ. После сбоя активных Сетевых NIC ждущий режим становится активным и процессы сетевой трафик. При службы восстановлены все члены группы, ожидания члену возвращается состояние ожидания.  
  
При наличии двух Сетевых карт и настройку одного сетевого Адаптера, как адаптер ждущий режим, вы потеряете пропускной способности агрегирования преимущества, которые существуют с двумя сетевыми адаптерами active.  
  
> [!IMPORTANT]  
> Необходимо назначить адаптер ждущий режим, чтобы достичь отказоустойчивости. отказоустойчивость всегда присутствует, при наличии по крайней мере два сетевых адаптеров в группу Сетевых карт.
