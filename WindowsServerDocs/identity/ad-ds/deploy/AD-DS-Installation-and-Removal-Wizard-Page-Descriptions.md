## <a name="BKMK_DepConfigPage"></a>Конфигурация развертывания  
Диспетчер серверов начинает установку всех контроллеров домена с **конфигурации развертывания** страницы. Оставшиеся параметры и обязательные поля меняются на этой странице и последующих страницах в зависимости от того, какая операция развертывания выбрана. Например, при создании нового леса **параметры подготовки** страница не отображается, но не при установке первого контроллера домена под управлением Windows Server 2012 в существующем лесу или домене.  
  
Проверочные тесты выполняются на этой странице, а затем часть проверок предварительных требований. Например при попытке установить первый контроллер домена Windows Server 2012 в лесу с режимом работы Windows 2000, на этой странице отображается сообщение об ошибке.  
  
При создании нового леса отображаются следующие параметры.  
  
![Установка AD DS](media/AD-DS-Installation-and-Removal-Wizard-Page-Descriptions/ADDS_SMI_DeploymentConfiguration_Forest.gif)  
  
-   При создании нового леса необходимо указать имя корневого домена леса. Имя корневого домена леса не может быть однокомпонентным (например, он должен быть «contoso.com» вместо «contoso»). Он должен использовать разрешенные контексты именования домена DNS. Можно указать международное доменное имя (IDN). Дополнительные сведения о контекстах именования домена DNS см. в разделе [КБ 909264](https://support.microsoft.com/kb/909264).  
  
-   Не создавайте лесов Active Directory с тем же именем, как внешними именами DNS. Например если URL-адрес DNS Интернета — http://contoso.com, необходимо выбрать другое имя для своего внутреннего леса избежать проблем с совместимостью в дальнейшем. Данное имя должно быть уникальным и достаточно редким для веб-трафика, например corp.contoso.com.  
  
-   Необходимо быть членом группы администраторов на сервере, где вы хотите создать новый лес.  
  
Дополнительные сведения о создании леса см. в разделе [Установка нового леса Windows Server 2012 Active Directory & #40; Уровень 200 & #41;](../../ad-ds/deploy/Install-a-New-Windows-Server-2012-Active-Directory-Forest--Level-200-.md).  
  
При создании нового домена отображаются следующие параметры.  
  
![Установка AD DS](media/AD-DS-Installation-and-Removal-Wizard-Page-Descriptions/ADDS_SMI_DeploymentConfiguration_ChildDomain.gif)  
  
> [!NOTE]  
> При создании нового домена дерева, необходимо указать имя корневого домена леса, вместо имени родительского домена причем остальные страницы мастера и параметры остаются без изменений.  
  
-   Нажмите кнопку **выберите** для перехода к родительскому домену или дереву Active Directory, или введите имя допустимого родительского домена или дерева. Затем введите имя нового домена в **новое имя домена**.  
  
-   Домен дерева: укажите имя допустимое полное имя корневого домена; Имя не может быть однокомпонентным и должно отвечать требованиям имя домена DNS.  
  
-   Дочерний домен: Укажите допустимое однокомпонентное дочернего домена имя; Имя должно соответствовать требованиям DNS-имени домена.  
  
-   Мастер настройки доменных служб Active Directory запросит учетные данные домена, если текущие учетные данные не из домена. Нажмите кнопку **изменение** для предоставления учетных данных домена.  
  
Дополнительные сведения о создании домена см. в разделе [Установка Windows Server 2012 Active Directory дочерний или домена дерева & #40; Уровень 200 & #41;](../../ad-ds/deploy/Install-a-New-Windows-Server-2012-Active-Directory-Child-or-Tree-Domain--Level-200-.md).  
  
При добавлении нового контроллера домена к существующему домену отображаются следующие параметры.  
  
![Установка AD DS](./media/AD-DS-Installation-and-Removal-Wizard-Page-Descriptions/ADDS_SMI_DeploymentConfiguration_Replica.gif)  
  
-   Нажмите кнопку **выберите** перейти к домену, или введите допустимое имя домена.  
  
-   Диспетчер серверов запрашивает действующие учетные данные при необходимости. Установка дополнительного контроллера домена требуется членство в группе администраторов домена.  
  
    Кроме того Установка первого контроллера домена под управлением Windows Server 2012 в лесу требуются учетные данные, включающие членство в группах "Администраторы предприятия" и "Администраторы схемы". Мастер настройки доменных служб Active Directory предлагает позже, если у текущих учетных данных нет соответствующих разрешений или членства в группах.  
  
Дополнительные сведения о том, как добавить контроллер домена в существующий домен см. в разделе [установки контроллера домена реплики Windows Server 2012 в существующем домене & #40; Уровень 200 & #41;](../../ad-ds/deploy/Install-a-Replica-Windows-Server-2012-Domain-Controller-in-an-Existing-Domain--Level-200-.md).  
  
## <a name="BKMK_DCOptionsPage"></a>Параметры контроллера домена  
При создании нового леса на странице параметров контроллера домена имеет следующие параметры:  
  
![Установка AD DS](media/AD-DS-Installation-and-Removal-Wizard-Page-Descriptions/ADDS_SMI_DCOptions_Forest.gif)  
  
-   В Windows Server 2012 по умолчанию заданы режимы работы леса и домена.  
  
    Доступна одна новая возможность в режиме работы домена Windows Server 2012: поддержка динамического контроля доступа и защиты политики административных шаблонов центра распространения КЛЮЧЕЙ Kerberos имеет два параметра (всегда предоставлять утверждения и отказывать незащищенным проверкам подлинности), требующие режима работы домена Windows Server 2012. Дополнительные сведения см. в разделе «Поддержка утверждений, комплексной проверки подлинности и защиты Kerberos» в [новые возможности проверки подлинности Kerberos](https://technet.microsoft.com/library/hh831747.aspx).    
    Режим работы леса Windows Server 2012 не предоставляет новых возможностей, но это гарантирует, что любой домен, созданный в лесу, будут автоматически работать в режиме работы домена Windows Server 2012. Режим работы домена Windows Server 2012 не предоставляет какие-либо других новых возможностей рядом с поддержка динамического контроля доступа и защиты Kerberos, но это гарантирует, что любой контроллер домена в домене работает под управлением Windows Server 2012. Дополнительные сведения о других возможностях, доступных при разных режимах работы см. в разделе [режимах работы доменных служб общее представление о Active Directory (AD DS)](../active-directory-functional-levels.md).  
  
    Кроме режимов работы контроллер домена под управлением Windows Server 2012 предоставляет дополнительные возможности, недоступные на контроллере домена под управлением более ранней версии Windows Server. Например контроллер домена под управлением Windows Server 2012 можно использовать для клонирования виртуального контроллера домена, а контроллер домена под управлением более ранней версии Windows Server — нельзя.  
  
-   DNS-сервер выбран по умолчанию при создании нового леса. Первый контроллер домена в лесу должен быть сервером глобального каталога (GC), и не может быть единственным контроллером домена чтения (RODC).  
  
-   Пароль режима восстановления служб каталогов (DSRM) необходим для входа контроллере домена, на котором не запущены AD DS. Указать пароль должен соответствовать политике паролей, действующей для сервера, которая по умолчанию не требует надежного пароля — только непустой пароль. Всегда выбирайте надежный, сложный пароль, предпочтительно парольную фразу. Сведения о том, как синхронизировать Пароль DSRM с паролем учетной записи пользователя домена см. в разделе [КБ 961320](https://support.microsoft.com/kb/961320).  
  
Дополнительные сведения о создании леса см. в разделе [Установка нового леса Windows Server 2012 Active Directory & #40; Уровень 200 & #41;](../../ad-ds/deploy/../../ad-ds/deploy/Install-a-New-Windows-Server-2012-Active-Directory-Forest--Level-200-.md).  
  
При создании дочернего домена на странице параметров контроллера домена имеет следующие параметры:  
  
![Установка AD DS](media/AD-DS-Installation-and-Removal-Wizard-Page-Descriptions/ADDS_SMI_DCOptions_Child.gif)  
  
-   В Windows Server 2012 по умолчанию задан режим работы домена. Можно указать любое другое значение не меньше значения из режима работы леса или более поздней версии.  
  
-   Настраиваемые параметры контроллера домена включают **DNS-сервер** и **глобального каталога**; как первый контроллер домена в новом домене, нельзя назначить контроллер домена только для чтения.

    > [!IMPORTANT]  
    > Если сервер не входит в подсеть Active Directory и имеется несколько сайтов, выбор не производится и **Далее** кнопка будет недоступна, пока не будет выбран сайт из списка.  
  
Дополнительные сведения о создании домена см. в разделе [Установка Windows Server 2012 Active Directory дочерний или домена дерева & #40; Уровень 200 & #41;](../../ad-ds/deploy/../../ad-ds/deploy/../../ad-ds/deploy/Install-a-New-Windows-Server-2012-Active-Directory-Child-or-Tree-Domain--Level-200-.md).  
  
При добавлении контроллера домена в домен на странице параметров контроллера домена имеет следующие параметры:  
  
![Установка AD DS](media/AD-DS-Installation-and-Removal-Wizard-Page-Descriptions/ADDS_SMI_DCOptions_Replica.gif)

## <a name="BKMK_DNSOptionsPage"></a>Параметры DNS  
При установке DNS-сервера, следующие **параметры DNS** откроется страница:  
  
![Установка AD DS](media/AD-DS-Installation-and-Removal-Wizard-Page-Descriptions/ADDS_SMI_DNSOptions_Replica.gif)  
  
При установке DNS-сервера записи делегирования, которые указывают на DNS-сервер в качестве разрешенного для зоны, должны создаваться в родительской зоне доменных имен (DNS). Записи делегирования передают орган разрешения имен и обеспечивают правильные ссылки на другие DNS-серверы и на клиентов новых серверов, которые были сделаны полномочными для новой зоны. Эти записи ресурсов содержат следующие сведения:  
  
-   Запись сервера имен (NS) ресурсов для реализации делегирования. Эта запись ресурса сообщает, что сервер с именем ns1.na.example.microsoft.com является полномочным сервером для делегированного субдомена.  
  
-   Запись ресурса узла (A или AAAA) также называется связанной записью должен присутствовать для разрешения имени сервера, указанный в записи ресурса сервера (NS) имя в IP-адрес. Процесс разрешения имени узла в этой записи ресурса в делегированный DNS-сервер в записи ресурса для имени сервера (NS) иногда называется «связывающим преследованием».  
  
Может иметь Active Directory домена мастер настройки служб автоматически. Мастер проверяет наличие соответствующих записей в родительской зоне DNS после нажатия кнопки **Далее** на **параметры контроллера домена** страницы. Если мастер не может проверить наличие записей в родительском домене, мастер предоставляет возможность создать новое делегирование DNS для нового домена (или обновить существующее делегирование) автоматически и продолжить установку нового контроллера домена.  
  
Кроме того можно создать эти записи нового делегирования перед установкой DNS-сервера. Чтобы создать делегирование зоны, откройте **диспетчера DNS**, щелкните правой кнопкой мыши родительский домен и нажмите кнопку **новое делегирование**. Следуйте указаниям мастера делегирования, создания делегирования.  
  
Процесс установки пытается создать делегирование, чтобы убедиться, что компьютеры в других доменах разрешать DNS-запросы для узлов в, включая контроллеры домена и компьютерами-членами, поддомена DNS. Обратите внимание, что записи делегирования могут создаваться автоматически только на DNS-серверах Майкрософт. Если зона родительского DNS домена расположена на DNS-серверы сторонних производителей, например BIND, на странице проверки предварительных требований отображается предупреждение о неудачной попытке создания записей DNS-делегирования. Дополнительные сведения об этом предупреждении см. в разделе [известные проблемы установки AD DS](https://technet.microsoft.com/library/cc754463(v=WS.10).aspx).  
  
Делегирование между родительским доменом и субдоменом, роль которого повышается можно создать и проверить до или после установки. Не следует откладывать установку нового контроллера домена, так как не удается создать или обновить делегирование DNS.  
  
Дополнительные сведения о делегировании см. в разделе [общее представление о делегировании зоны](https://go.microsoft.com/fwlink/?LinkId=164773) (https://go.microsoft.com/fwlink/?LinkId=164773). Если делегирование зоны не представляется возможным, в данной ситуации, можно рассмотреть другие способы обеспечения разрешения имен из других доменов для узлов в вашем домене. Например, администратор DNS другого домена может настроить условную пересылку, зоны заглушки или дополнительные зоны для разрешения имен в вашем домене. Дополнительные сведения см. в следующих разделах:  
  
-   [Общее представление о типах зон](https://go.microsoft.com/fwlink/?LinkID=157399) (https://go.microsoft.com/fwlink/?LinkID=157399)  
  
-   [Общее представление о зонах-заглушках](https://go.microsoft.com/fwlink/?LinkId=164776) (https://go.microsoft.com/fwlink/?LinkId=164776)  
  
-   [Общее представление о пересылке](https://go.microsoft.com/fwlink/?LinkId=164778) (https://go.microsoft.com/fwlink/?LinkId=164778)  
  
## <a name="BKMK_RODCOptionsPage"></a>Параметры RODC  
При установке контроллера домена только для чтения (RODC) отображаются следующие параметры.  
  
![Установка AD DS](media/AD-DS-Installation-and-Removal-Wizard-Page-Descriptions/ADDS_SMI_RODCOptions.gif)  
  
-   Делегированные учетные записи администратора получают локальные права администратора для RODC. Эти пользователи имеют привилегии группы администраторов локального компьютера. Они не являются членами группы администраторов домена или встроенной группы администраторов домена. Этот параметр полезен при делегировании администрирования филиалом без выдачи разрешения на администрирование домена. Настройка делегирования прав администратора не требуется. Дополнительные сведения см. в разделе [делегирование прав администратора](https://technet.microsoft.com/library/cc753170(v=WS.10).aspx).  
  
-   Политика репликации паролей действует как список управления доступом (ACL). Она определяет, разрешается ли RODC кэширование пароля. После как RODC получает запрос на вход прошедшего проверку подлинности пользователя или компьютера, он обращается к политике репликации паролей, чтобы определить, нужно ли кэшировать пароль учетной записи. Ту же учетную запись можно этого следующие входы под более эффективными.  
  
    Репликация политики Паролей список учетных записей, пароли которых разрешено кэшировать и учетные записи, пароли от которых кэшировать явным образом запрещено. Список учетных записей пользователей и компьютеров, которые разрешено кэшировать не подразумевает, что RODC обязательно кэширует пароли этих учетных записей. Администратор может например, указать заранее учетные записи, которые RODC будет кэшировать. Таким образом, RODC может проверить подлинность этих учетных записей, даже если ссылка глобальной сети на узловой сайт неактивна.  
  
    Пользователям или компьютерам не разрешено (в том числе неявно) или Отказано не кэшируют свои пароли. Если вышеупомянутые пользователи или компьютеры не имеют доступа к контроллеру домена с возможностью записи, они не имеют доступа к AD DS-ресурсам и функциональным возможностям. Дополнительные сведения о политике репликации Паролей см. в разделе [политики репликации паролей](https://technet.microsoft.com/library/cc730883(v=ws.10).aspx). Дополнительные сведения об управлении политикой репликации Паролей см. в разделе [Администрирование политики репликации паролей](https://technet.microsoft.com/library/rodc-guidance-for-administering-the-password-replication-policy(v=ws.10).aspx).  
  
Дополнительные сведения об установке RODC см. в разделе [Установка Windows Server 2012 Active Directory Read-Only контроллера домена & #40; RODC & #41; & #40; Уровень 200 & #41;](../../ad-ds/deploy/RODC/Install-a-Windows-Server-2012-Active-Directory-Read-Only-Domain-Controller--RODC---Level-200-.md).

## <a name="BKMK_AdprepCreds"></a>Параметры подготовки

Adprep необходим для выполнения, чтобы добавить первый контроллер домена под управлением Windows Server 2012 в существующий домен или лес. В частности:

Выполнения этой команды необходимо быть членом группы администраторов домена в домене, где устанавливается контроллер домена под управлением Windows Server 2012. ## <a name="BKMK_RemovalCredsPage"></a>Учетные данные для удаления роли  
![Установка AD DS](media/AD-DS-Installation-and-Removal-Wizard-Page-Descriptions/ADDS_RRW_Credentials.gif)  
  
Параметры понижения уровня настраиваются на **учетные данные** страницы. Учетные данные, необходимые для понижения уровня из следующего списка:  
  
-   Для понижения уровня дополнительного контроллера домена требуются учетные данные администратора домена. Выбор **Принудительное удаление контроллера домена** можно понизить уровень контроллера домена без удаления метаданных объекта контроллера домена из Active Directory.  
  
    > [!IMPORTANT]  
    > Не выбирайте этот параметр, если контроллер домена не удается связаться с другими контроллерами домена и *нет другого способа* чтобы разрешить эту сетевую проблему. Принудительное понижение уровня оставляет потерянные метаданные в Active Directory на оставшихся контроллерах домена в лесу. Кроме того все нереплицированные изменения на этом контроллере домена, например пароли и новые учетные записи пользователей, навсегда теряются. Потерянные метаданные — это основная причина обращения в значительный процент обращений в службу поддержки пользователей Майкрософт AD DS, Exchange, SQL и другого программного обеспечения. При принудительном понижении уровня контроллера домена, вы *необходимо* вручную немедленно почистить метаданные. Действия, просмотрите [Очистка метаданных сервера](https://technet.microsoft.com/library/cc816907(WS.10).aspx).  
  
-   Понижение роли последнего контроллера домена в домене требуется членство в группе администраторов предприятия, так как при этом удаляется сам домен (если это последний домен леса, при этом удаляется лес). Диспетчер сервера сообщает, если текущий контроллер домена является последний контроллер домена в домене. Выберите **последний контроллер домена в домене** для подтверждения того, что контроллер является последний контроллер домена в домене.

## <a name="BKMK_RemovalOptionsPage"></a>Параметры удаления AD DS и предупреждения  
Если требуется помощь со страницей просмотреть параметры, см. раздел Просмотр параметров  
  
Если на контроллере домена размещены дополнительные роли, такие как роль DNS-сервера или сервера глобального каталога, появляется следующая страница с предупреждением:  
  
![Установка AD DS](media/AD-DS-Installation-and-Removal-Wizard-Page-Descriptions/ADDS_RRW_Warnings.gif)  
  
Необходимо щелкнуть **продолжить удаление** чтобы подтвердить, что дополнительные роли больше не будут доступны до нажатия **Далее** для продолжения.  
  
Если вы принудительно удалить контроллер домена, будут потеряны все изменения объектов Active Directory, которые не были реплицированы на другие контроллеры домена в домене. Кроме того, если на контроллере домена размещены роли хозяина операций, глобального каталога или роль DNS-сервера, критические операции в домене и лесе могут быть изменены следующим образом. Перед удалением контроллера домена, на котором находится роль хозяина операций, необходимо попытаться переместить роль на другой контроллер домена. Если нет возможности передать роль, сначала удалите доменных служб Active Directory с этого компьютера и затем использовать Ntdsutil.exe для захвата роли. Используйте программу Ntdsutil на контроллере домена, который планируется захватить роль; по возможности используйте последний партнер репликации на том же сайте, что и данный контроллер домена. Дополнительные сведения о переносе и захвате ролей хозяина операций см. в разделе [статья 255504](https://go.microsoft.com/fwlink/?LinkId=80395) базы знаний Майкрософт. Если мастер не может определить, если контроллер домена размещены роли хозяина операций, выполните команду netdom.exe, чтобы проверить, выполняет ли данный контроллер домена роли хозяина операций.  
  
-   Глобальный каталог: пользователи могут испытывать проблемы со входом домены в лесу. Перед удалением сервера глобального каталога, убедитесь в наличии достаточно серверов глобального каталога в этом лесу и узел для обслуживания входа в систему пользователя. При необходимости назначить другой сервер глобального каталога и обновить информацию клиентов и приложений.  
  
-   DNS-сервер: все данные DNS, хранящиеся в зоны, интегрированные с Active Directory будут потеряны. После удаления AD DS этот DNS-сервер не будет выполнять разрешение имен для зон DNS, которые были интегрированы в Active Directory. Следовательно рекомендуется обновить DNS-конфигурацию всех компьютеров, которые в настоящий момент обращаются к IP-адрес этого DNS-сервера для разрешения имен с IP-адресом нового DNS-сервера.  
  
-   Хозяин инфраструктуры: клиенты в домене могут испытывать трудности с определением местоположения объектов в других доменах. Перед продолжением перенесите роль хозяина инфраструктуры на контроллер домена, не является сервером глобального каталога.  
  
-   Хозяин RID: возможны проблемы при создании новых учетных записей пользователей, учетные записи компьютеров и групп безопасности. Перед продолжением перенесите роль хозяина RID на контроллер домена в том же домене, что и данный контроллер домена.  
  
-   Эмулятор основного домена (PDC) контроллеру: операции эмулятора основного контроллера домена, такие как обновление групповой политики и сброс паролей для учетных записей не относящихся к AD DS, не будут правильно работать. Перед продолжением перенесите роль хозяина эмулятора основного контроллера домена к контроллеру домена, который находится в том же домене, что и данный контроллер домена.  
  
-   Хозяин схемы: вы больше не сможете изменять схему этого леса. Перед продолжением перенесите роль хозяина схемы на контроллер домена в корневом домене леса.  
  
-   Хозяин именования доменов: больше не будет возможность добавлять или удалять домены в этом лесу. Перед продолжением перенесите роль контроллера домена в корневом домене леса хозяина именования доменов.  
  
-   Будут удалены все разделы каталога приложений на этом контроллере домена Active Directory. Если контроллер домена содержит последний Репликат одного или нескольких разделов каталога приложений, по завершении операции удаления, эти разделы перестанут существовать.  
  
Имейте в виду, что домена больше не будет существовать после удаления доменных служб Active Directory с последнего контроллера домена в домене.  
  
Если контроллер домена является DNS-сервер, которому делегированы права на размещение зоны DNS, на следующей странице будут представлены возможность удалить DNS-сервер из делегирования зоны DNS.
