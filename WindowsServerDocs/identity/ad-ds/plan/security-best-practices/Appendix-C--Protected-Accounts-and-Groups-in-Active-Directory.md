# <a name="appendix-c-protected-accounts-and-groups-in-active-directory"></a>Приложение в. защищенные учетные записи и группы в Active Directory

>Область применения: Windows Server 2016, Windows Server 2012 R2, Windows Server 2012


## <a name="appendix-c-protected-accounts-and-groups-in-active-directory"></a>Приложение в. защищенные учетные записи и группы в Active Directory  
В Active Directory набор по умолчанию привилегированных учетных записей и групп, считаются защищенных учетных записей и групп. Для большинства объектов в Active Directory полномочные администраторы (пользователей, которые были предоставлены разрешения для управления объектами Active Directory) можно изменить разрешения для объектов, включая изменение разрешения, позволяющие сами изменение членства групп, например.  

Однако с помощью защищенных учетных записей и групп объектов разрешения задать и принудительно через автоматический процесс, который гарантирует, что разрешения для объектов остается согласованной, даже если объекты переместить каталог. Даже если кто-то вручную изменяет разрешения защищенных объектов, этот процесс гарантирует, что разрешения возвращаются к значениям по умолчанию, быстро.

#### <a name="adminsdholder"></a>AdminSDHolder  
Объект AdminSDHolder предназначено для предоставления разрешений «шаблон» для защищенных учетных записей и групп в домене. AdminSDHolder автоматически создается как объект в системном контейнере каждого домена Active Directory. Указан путь: **CN = AdminSDHolder, CN = System, DC = < domain_component > контроллера домена = < domain_component >?.**  

В отличие от большинства объектов в домене Active Directory, которые принадлежат группы «Администраторы», AdminSDHolder владельцем группы администраторов домена. По умолчанию EAs внести изменения объект AdminSDHolder любого домена, а также группы администраторов домена и администраторов домена. Кроме того несмотря на то что по умолчанию владелец AdminSDHolder группы администраторов домена в домен, членами администраторов или администраторов предприятия может стать владельцем объекта.  

#### <a name="sdprop"></a>SDProp  
SDProp — это процесс, который выполняется каждые 60 минут (по умолчанию) на контроллер домена, содержащий эмулятора основного контроллера домена в домене (PDCE). SDProp сравнивает разрешений на объект AdminSDHolder домена с разрешениями на защищенных учетных записей и групп в домене. Если разрешения на всех защищенных учетных записей и групп разрешений на объект AdminSDHolder не совпадают, должны совпадать с объект AdminSDHolder домена сбрасываются разрешений на защищенных учетных записей и групп.  

Кроме того наследование разрешений отключен на защищенных групп и учетных записей, это означает, что даже если учетные записи и группы перемещаются в другие расположения в каталоге, они не наследуют разрешения от родительских объектов новый. Наследование отключается на объект AdminSDHolder, чтобы изменения разрешений для родительских объектов, не изменяйте разрешения AdminSDHolder.  

##### <a name="changing-sdprop-interval"></a>Изменение интервала SDProp  
Как правило нет необходимости изменить интервал, в которой работает SDProp, за исключением тестирования. Если вам нужно будет изменить интервал SDProp в pdce для домена, используйте regedit, чтобы добавить или изменить значение AdminSDProtectFrequency DWORD в HKLM\SYSTEM\CurrentControlSet\Services\NTDS\Parameters.  

Диапазон значений — в секундах, с 60 7200 секунд (одна минута на два часа). Чтобы отменить изменения, удалите AdminSDProtectFrequency ключ, который вызовет SDProp вернуться к 60 минут. Обычно не уменьшайте этот интервал в доменах производства как повысить LSASS затратам ресурсов на обработку на контроллере домена. Влияние этого увеличения зависит от количества защищенных объектов в домене.  

##### <a name="running-sdprop-manually"></a>Запуск SDProp вручную  
Лучший подход к тестированию изменения AdminSDHolder — запускать SDProp вручную, который запускает задание немедленно запустить, но не влияет на выполнение запланированных. Запуск SDProp вручную выполнять немного по-разному в контроллерах домена под управлением Windows Server 2008 и более ранних версий, чем на контроллерах домена под управлением Windows Server 2012 или Windows Server 2008 R2.  

В приведены процедуры запуск SDProp вручную в старых операционных системах [статья службы поддержки Майкрософт 251343](https://support.microsoft.com/kb/251343), и ниже приведены пошаговые инструкции для более старых и более поздних операционных систем. В любом случае необходимо подключиться к объекту rootDSE в Active Directory и выполнять операцию изменения с null DN объекта rootDSE указание как атрибут, чтобы изменить имя операции. Дополнительные сведения об операциях изменяемой на объекте rootDSE см. в разделе [операций изменения rootDSE](https://msdn.microsoft.com/library/cc223297.aspx) на веб-сайте MSDN.  

###### <a name="running-sdprop-manually-in-windows-server-2008-or-earlier"></a>Под управлением SDProp вручную в Windows Server 2008 или более ранней версии  
Вы можете принудительно SDProp выполнять с помощью Ldp.exe или выполнив сценарий изменения LDAP. Чтобы запустить с помощью Ldp.exe SDProp, выполните следующие действия после внесения изменений в объект AdminSDHolder в домен:  

1.  Запуск **Ldp.exe**.  

2.  Нажмите кнопку **подключения** на Ldp диалоговое окно и нажмите кнопку **Connect**.  

    ![защищенные учетные записи и группы](media/Appendix-C--Protected-Accounts-and-Groups-in-Active-Directory/SAD_9.gif)  

3.  В **Connect** диалогового окна введите имя контроллера домена для домена, которому поручена роль эмулятора PDC (PDCE) и нажмите кнопку **ОК**.  

    ![защищенные учетные записи и группы](media/Appendix-C--Protected-Accounts-and-Groups-in-Active-Directory/SAD_10.png)  

4.  Убедитесь, что вы подключены успешно, указываемый **Dn: (RootDSE)** на следующем снимке экрана, нажмите кнопку **подключения** и нажмите кнопку **привязки**.  

    ![защищенные учетные записи и группы](media/Appendix-C--Protected-Accounts-and-Groups-in-Active-Directory/SAD_11.png)  

5.  В **привязки** диалогового окна введите учетные данные учетной записи пользователя, имеющую разрешение на изменение объекта rootDSE. (Если вы вошли на от имени этого пользователя, можно выбрать **привязки в качестве** вошедшего пользователя.) Нажмите кнопку **ОК**.  

    ![защищенные учетные записи и группы](media/Appendix-C--Protected-Accounts-and-Groups-in-Active-Directory/SAD_12.png)  

6.  После завершения операции привязки щелкните **Обзор**и нажмите кнопку **изменить**.  

    ![защищенные учетные записи и группы](media/Appendix-C--Protected-Accounts-and-Groups-in-Active-Directory/SAD_13.png)  

7.  В **изменить** диалоговом оставить **DN** пустым. В **изменить атрибут записи** введите **FixUpInheritance**и в **значения** введите **Да**. Нажмите кнопку **ввод** для заполнения **список записей** как показано на следующем снимке экрана.  

    ![защищенные учетные записи и группы](media/Appendix-C--Protected-Accounts-and-Groups-in-Active-Directory/SAD_14.gif)  

8.  В диалоговом окне указывается изменить щелкните "Выполнить" и убедитесь, что изменения, внесенные в объект AdminSDHolder проявляться на этот объект.

> Сведения об изменении AdminSDHolder, чтобы разрешить назначенным непривилегированным учетным записям изменить членство защищенных групп, см. в разделе [приложение I: создание управления учетными записями для защищенных учетных записей и групп в Active Directory](../../../ad-ds/manage/component-updates/Appendix-I--Creating-Management-Accounts-for-Protected-Accounts-and-Groups-in-Active-Directory.md).
