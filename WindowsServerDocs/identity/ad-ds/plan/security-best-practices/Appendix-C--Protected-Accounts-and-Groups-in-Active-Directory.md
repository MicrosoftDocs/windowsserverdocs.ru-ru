# <a name="appendix-c-protected-accounts-and-groups-in-active-directory"></a>Приложение В. Защищенные учетные записи и группы в Active Directory

>Область применения. Windows Server 2016, Windows Server 2012 R2, Windows Server 2012

## <a name="appendix-c-protected-accounts-and-groups-in-active-directory"></a>Приложение В. Защищенные учетные записи и группы в Active Directory

В Active Directory по умолчанию набор высокопривилегированные учетные записи и группы, считаются защищенных учетных записей и групп. Для большинства объектов в Active Directory полномочные администраторы (пользователи, которые были предоставлены разрешения для управления объектами Active Directory) можно изменить разрешения для объектов, включая изменение разрешения, позволяющие сами изменение членства группы, например.  

Тем не менее защищенные учетные записи и группы, разрешения объектов задаются и применяются с помощью автоматического процесса гарантирует, что разрешения на эти объекты остаются согласованный, даже если объекты перемещены в каталоге. Даже если кто-то вручную изменяет разрешения защищенных объектов, этот процесс гарантирует, что возвращаются разрешения по умолчанию быстро.

#### <a name="adminsdholder"></a>AdminSDHolder

Объект AdminSDHolder предназначена для предоставления разрешения «шаблон» для защищенных учетных записей и групп в домене. AdminSDHolder автоматически создается как объект в системном контейнере каждого домена Active Directory. Указан путь: **CN = AdminSDHolder, CN = System, DC = < domain_component >, DC = < domain_component >?.**  

В отличие от большинства объектов в домене Active Directory, которые принадлежат группы "Администраторы", AdminSDHolder принадлежит группе "Администраторы домена". По умолчанию EAs можно изменить объект AdminSDHolder любого домена, как и группы "Администраторы домена" и администраторов домена. Кроме того несмотря на то, что по умолчанию владельцем AdminSDHolder — группа "Администраторы домена" домена, членами администраторов или администраторов предприятия может стать владельцем объекта.  

#### <a name="sdprop"></a>SDProp

SDProp — это процесс, который выполняется каждые 60 минут (по умолчанию) на контроллере домена, который содержит эмулятор домена PDC (PDCE). SDProp сравнивает разрешения на объект AdminSDHolder домена с разрешениями на защищенных учетных записей и групп в домене. Если разрешения на любой из защищенных учетных записей и групп не соответствуют разрешения на объект AdminSDHolder, разрешения на защищенных учетных записей и групп будут заменены для получения соответствия строкам AdminSDHolder объекта домена.  

Кроме того наследование разрешений отключена на защищенных групп и учетных записей, это означает, что даже если учетные записи и группы перемещаются в разные местоположения в каталоге, они не наследуют разрешения от новых родительских объектов. Наследование отключено в объекте AdminSDHolder таким образом, чтобы изменения разрешений к родительским объектам не изменяйте разрешения AdminSDHolder.  

##### <a name="changing-sdprop-interval"></a>Изменение интервала SDProp

Как правило нет необходимости этот интервал, в которой выполняется SDProp, за исключением в целях тестирования. Если вам нужно изменить интервал SDProp на PDCE для домена, используйте regedit, чтобы добавить или изменить значение AdminSDProtectFrequency DWORD в HKLM\SYSTEM\CurrentControlSet\Services\NTDS\Parameters.  

Диапазон значений — в от 60 до 7200 об/мин (одна минута до двух часов) секунд. Отмена изменений, удалите раздел AdminSDProtectFrequency, что вызовет SDProp вернуться к 60 минут. Обычно следует не уменьшить этот интервал в рабочей среде домены как его можно увеличить LSASS затратам ресурсов на обработку на контроллере домена. Влияние это увеличение зависит количество защищенных объектов в домене.  

##### <a name="running-sdprop-manually"></a>Ручной запуск SDProp

Лучший подход к тестированию AdminSDHolder изменений — это выполнить SDProp вручную, который запускает задание для немедленного запуска, но не влияет на запланированные выполнения. Ручной запуск SDProp немного по-разному в контроллерах домена под управлением Windows Server 2008 и более ранних версий, а не на контроллерах домена под управлением Windows Server 2012 или Windows Server 2008 R2.  

Процедуры для ручной запуск SDProp на старых операционных системах приведены в [статье службы поддержки Майкрософт 251343](https://support.microsoft.com/kb/251343), и ниже приведены пошаговые инструкции для старых и новых операционных систем. В любом случае необходимо подключиться к объекту rootDSE в Active Directory и выполнить операцию по изменению с null DN объекта rootDSE, указывать имя операции в качестве атрибута для изменения. Дополнительные сведения об операциях изменения в объекте rootDSE см. в разделе [операции изменения rootDSE](https://msdn.microsoft.com/library/cc223297.aspx) на веб-сайте MSDN.  

###### <a name="running-sdprop-manually-in-windows-server-2008-or-earlier"></a>SDProp ручной запуск в Windows Server 2008 или более ранней версии

Вы можете принудительно SDProp для запуска с помощью Ldp.exe или, запустив скрипт изменения LDAP. Чтобы запустить с помощью Ldp.exe SDProp, выполните следующие действия, после внесения изменений в объект AdminSDHolder в домене:  

1. Запустите **Ldp.exe**.  
2. Нажмите кнопку **подключения** в диалоговое окно Ldp, а затем нажмите **Connect**.  

   ![защищенные учетные записи и группы](media/Appendix-C--Protected-Accounts-and-Groups-in-Active-Directory/SAD_9.gif)  

3. В **Connect** диалогового окна введите имя контроллера домена для домена, играющий роль эмулятора PDC (PDCE) и нажмите кнопку **ОК**.  

   ![защищенные учетные записи и группы](media/Appendix-C--Protected-Accounts-and-Groups-in-Active-Directory/SAD_10.png)  

4. Убедитесь, что вы подключились успешно, обозначенный **Dn: (RootDSE)**  на следующем снимке экрана, нажмите кнопку **подключения** и нажмите кнопку **привязать**.  

   ![защищенные учетные записи и группы](media/Appendix-C--Protected-Accounts-and-Groups-in-Active-Directory/SAD_11.png)  

5. В **привязать** диалоговом окне введите учетные данные учетной записи пользователя, имеющую разрешение на изменение объекта rootDSE. (Если вы вошли от имени этого пользователя, можно выбрать **привязки в качестве** вошедшего в систему пользователя.) Нажмите кнопку **ОК**.  

   ![защищенные учетные записи и группы](media/Appendix-C--Protected-Accounts-and-Groups-in-Active-Directory/SAD_12.png)  

6. После завершения операции привязки, нажмите кнопку **Обзор**и нажмите кнопку **изменить**.  

   ![защищенные учетные записи и группы](media/Appendix-C--Protected-Accounts-and-Groups-in-Active-Directory/SAD_13.png)  

7. В **изменить** диалоговом окне оставьте **DN** пустым. В **изменить атрибут записи** введите **FixUpInheritance**, а затем в **значения** введите **Да**. Нажмите кнопку **ввод** для заполнения **список записей** как показано на следующем снимке экрана.  

   ![защищенные учетные записи и группы](media/Appendix-C--Protected-Accounts-and-Groups-in-Active-Directory/SAD_14.gif)  

8. В диалоговом окне "Изменение заполненный" щелкните "Выполнить" и убедитесь, что изменения, внесенные в объект AdminSDHolder проявляться на этот объект.  

> [!NOTE]  
> Сведения об изменении AdminSDHolder, чтобы разрешить назначенным непривилегированным учетным записям для изменения членства в защищенных группах см. в разделе [I: приложение Создание учетных записей управления защищенных учетных записей и групп в Active Directory](../../../ad-ds/manage/component-updates/Appendix-I--Creating-Management-Accounts-for-Protected-Accounts-and-Groups-in-Active-Directory.md).  

Если вы предпочитаете выполнять SDProp вручную через LDIFDE или сценарий, можно создать запись "Изменить", как показано ниже:  

![защищенные учетные записи и группы](media/Appendix-C--Protected-Accounts-and-Groups-in-Active-Directory/SAD_15.gif)
