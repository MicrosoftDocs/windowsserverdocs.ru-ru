>Область применения: Windows Server 2016, Windows Server 2012 R2, Windows Server 2012


## <a name="appendix-b-privileged-accounts-and-groups-in-active-directory"></a>Приложение б. привилегированные учетные записи и группы в Active Directory  
«Привилегированных» учетных записей и групп в Active Directory являются те, к которым мощные права, права и разрешения предоставляются, позволяющие им выполнять практически любые действия, в Active Directory и на компьютерах, присоединенных к домену. В этом приложении начинается с обсуждения права, права, и разрешения, за которым следует сведения о «Максимальная прав» учетных записей и групп в Active Directory, то есть, самые мощные учетных записей и групп.  

Также предоставлены сведения о встроенных и по умолчанию учетных записей и групп в Active Directory в дополнение к их права. Несмотря на то, что определенной конфигурации рекомендаций по защите максимальной прав учетных записей и групп предоставляются как отдельные приложения, это приложение содержит общие сведения, который поможет вам определить, пользователей и групп, которые следует сосредоточиться на защите. Это необходимо сделать так, как они могут использоваться злоумышленники могут взломать и даже уничтожение установки Active Directory.

#### <a name="permissions"></a>Разрешения  
Разрешения, элементы управления доступом, которые применяются к защищаемым объектам, например файловой системы, реестра, службы и объекты Active Directory. Каждый защищаемый объект имеет список управления связанные доступом (ACL), который содержит элементы управления доступом (ACE), разрешить или запретить субъектов безопасности (пользователей, служб, компьютеров или групп), возможность выполнять различные операции с объектом. Например списки управления доступом много объектов в Active Directory содержат записи ACE, разрешить проверку подлинности пользователям читать Общие сведения об объектах, но не предоставляйте им возможность чтения конфиденциальных данных или изменять объекты.
За исключением каждого домена встроенная учетная запись гостя каждого субъекта безопасности, которое входит в систему и проверки подлинности на контроллере домена в лесу Active Directory или доверенного леса имеет с проверкой подлинности пользователей идентификатор безопасности (SID) добавляются в свой маркер доступа по умолчанию. Таким образом является ли пользователь, служба или учетную запись компьютера пытается прочитать общих свойств объектов-пользователей в домене, операция чтения прошла успешно.  

Если определены попыток субъекта безопасности, доступ к объекту, для которых нет элементов управления доступом и которые содержат SID, который присутствует в маркер доступа субъекта, участника не имеют доступа к объекта. Кроме того Если элемент управления ДОСТУПОМ в ACL объекта содержит запись deny для SID, который соответствует маркер доступа пользователя, ACE «отклонить» будет обычно переопределение конфликтующие «разрешить» ACE. Дополнительные сведения о контроле доступа в Windows см. в разделе [контроля доступа](https://msdn.microsoft.com/library/aa374860(v=VS.85).aspx) на веб-сайте MSDN.  

В настоящем документе разрешения ссылается возможности, которые разрешен или запрещен субъектам безопасности на защищаемых объектов. При наличии конфликта между пользователю права и разрешения, право пользователя обычно имеет приоритет. Например если объект в Active Directory имеет ACL, который запрещает администраторы все доступ на чтение и запись к объекту, пользователя, являющегося членом группы администраторов домена будет невозможно просмотреть больше сведений об объекте. Тем не менее так как группы администраторов предоставляется пользователю право «Смена владельцев файлов и других объектов», пользователь может просто стать владельцем объекта в вопросе, а затем перепишите ACL объекта на предоставление администраторам — полный доступ объекта.  

Именно по этой причине, что в этом документе настоятельно рекомендует избежать с помощью мощных учетных записей и групп для повседневного администрирования, а не для ограничения возможности учетных записей и групп. Не эффективно можно остановить определенные пользователи, кто имеет доступ к мощные учетные данные из используя эти учетные данные для получения доступа к любому ресурсу защищаемого объекта.  

### <a name="built-in-privileged-accounts-and-groups"></a>Встроенные привилегированных учетных записей и групп  
Active Directory предназначена для упрощения делегирования прав администратора и принцип наименьших привилегий в назначения прав и разрешений. «Обычный» пользователей, имеющих учетные записи в домене Active Directory, по умолчанию, смогут прочитать большая часть что хранится в каталоге, но которые могут изменять только ограниченный набор данных в каталоге. Пользователи, которым требуется дополнительная привилегия может иметь предоставленное членство в различных привилегированных групп, которые встроены в каталоге, где они могут выполнять определенные задачи, связанные с их ролями, но не может выполнять задачи, которые не относятся к своих обязанностей.  

В Active Directory существует три встроенные группы, которые составляют наивысший группы привилегий в каталоге: группы администраторов предприятия (EA), в группу администраторов домена (DA) и встроенная группа администраторов (BA).  

Четвертый группу, в группу администраторов схемы (SA), имеет права доступа, которые могут повредить или уничтожить весь лес Active Directory, если неправильно, но эта группа является более ограничены возможности его чем EA, DA и BA групп.  

Помимо этих четырех групп существует ряд дополнительных встроенные учетные записи по умолчанию и групп в Active Directory, каждый из которых предоставляются права и разрешения, позволяющие определенные административные задачи, выполняемые. Несмотря на то, что это приложение не предоставляет подробное описание каждого встроенные или по умолчанию группы в Active Directory, он предоставляет таблицы групп и учетных записей, вы скорее всего узнать, при установке.  

Например если вы устанавливаете Microsoft Exchange Server в лесу Active Directory, дополнительных учетных записей и групп может быть создан в контейнерах встроенные и пользователи в доменах. В этом приложении описываются только групп и учетных записей, созданных в контейнерах встроенные и пользователей в Active Directory, на основе собственного ролей и компонентов. Учетные записи и группы, созданные программой установки корпоративного программного обеспечения не включаются.  

#### <a name="enterprise-admins"></a>Администраторы предприятия  
Группы администраторов предприятия (EA) находится в корневом домене леса, и по умолчанию она является членом встроенной группы администраторов в каждом домене леса. Встроенной учетной записи администратора в корневом домене леса входит только по умолчанию EA группы. EAs предоставлены права и разрешения, позволяющие их для изменения уровня леса. Это изменения, которые влияют на все домены в лесу, таких как добавление или удаление доменов, установлении доверия лесов или Повышение режима работы леса. В модель делегирования правильно разработанный и внедрении EA членства является обязательным только в том случае, когда сначала построении леса, или при внесении изменений определенных леса, такие как установка исходящем доверии леса.  

По умолчанию в контейнере пользователей в корневом домене леса находятся группы EA, и именно универсальную группу безопасности, если корневого домена леса не работает в смешанном режиме Windows 2000 Server, в котором случае группы является глобальной группой безопасности. Несмотря на то, что некоторые права предоставляются группам EA, многие из этой группе права фактически наследуются группы EA, так как он является членом группы администраторов в каждом домене леса. Администраторы предприятия права не по умолчанию на рабочих станциях или рядовых серверах.  

#### <a name="domain-admins"></a>Администраторы домена  
Каждый домен в лесу имеет свою собственную группы администраторов домена (DA), которая является членом встроенной группы администраторов (BA) этого домена помимо является членом локальной группы администраторов на каждом компьютере, который присоединен к домену. По умолчанию единственным членом группы DA для домена является встроенной учетной записи администратора для этого домена.  

DAs, всемогущими в своих доменах, пока EAs привилегии уровня леса. В модели делегирования правильно разработанный и внедрении DA членства должны только в сценариях «break стекло», которые являются ситуации, в которых требуется учетная запись с высоким уровнем привилегий на каждом компьютере в домене или при необходимо внести определенные изменения широкий домена. Несмотря на то, что собственные механизмы делегирования Active Directory разрешить делегирование, при условии, что его можно использовать только в сценариях аварийного DA учетные записи, создав модель эффективных делегирования может занять много времени и многие организации используют сторонние приложения, чтобы ускорить процесс.  

DA группа является глобальной группой безопасности в контейнере пользователей домена. Имеется одна группа DA для каждого домена в лесу, и по умолчанию DA группы входит только учетная запись встроенной учетной записи администратора домена. Так как группы домена DA вложена в группе BA домена и группы локальных администраторов системы каждые присоединенных к домену, DAs не только разрешения, предоставляемые специально для администраторов домена, но они также наследуют все права и разрешения для группы администраторов домена и локальной группы администраторов на всех компьютерах, присоединенных к домену.  

#### <a name="administrators"></a>Администраторы  
Встроенная группа администраторов (BA) является локальной группой домена в контейнере встроенных домена, в который вложены DAs и EAs, и именно этой группы, который предоставляется многие прямой права и разрешения в каталоге, а также на контроллерах домена. Тем не менее в группу администраторов домена нет никаких прав на рядовых серверах или на рабочих станциях. Членство в локальной группе администраторов компьютеров к домену, где предоставляется локальных привилегий; и описано групп только DAs являются членами локальной группы администраторов всех присоединенных к домену компьютеров по умолчанию.  

Группы «Администраторы» — это группа локального домена в контейнере встроенных домена. По умолчанию группа BA каждый домен содержит локального домена встроенной учетной записи администратора, группа DA локального домена и группы EA корневого домена леса. Специально для группы администраторов, не EAs или DAs предоставляются много прав пользователя в Active Directory, а также на контроллерах домена. Группа домена BA предоставляются разрешения на полный доступ для большинства объектов каталога и могут становиться владельцами объектов каталога. Несмотря на то, что группы EA и DA предоставлены определенные разрешения конкретным объектом лесу и домене, большая часть питания групп фактически» наследуется» от их членства в группах BA.  

> [!NOTE]  
> Несмотря на то, что их конфигурацию по умолчанию эти привилегированных групп, членом одной из трех групп можно управлять каталога для получения членства в любых других групп. В некоторых случаях это просто достижения в других более сложной, но с точки зрения потенциальных привилегий всех трех групп следует учитывать эффективно эквивалент.  

#### <a name="schema-admins"></a>Администраторы схемы  
В группу администраторов схемы (SA) является универсальной группой в корневом домене леса и имеет только этого домена встроенной учетной записи администратора, что элемент по умолчанию группе EA. Несмотря на то, что членство в группе SA может дать злоумышленнику нарушить схему Active Directory, которая — это платформа для всего леса Active Directory, SAs имеют несколько по умолчанию права и разрешения за пределы схемы.  

Следует тщательно управлять и проводить мониторинг членства в группе сопоставления безопасности, но в ряде аспектов эта группа является «меньшими привилегиями» более трех наивысший привилегированных групп, описанных выше, так как область привилегий узкой; Иными словами SAs иметь административные права в любом месте, отличном от схемы.
