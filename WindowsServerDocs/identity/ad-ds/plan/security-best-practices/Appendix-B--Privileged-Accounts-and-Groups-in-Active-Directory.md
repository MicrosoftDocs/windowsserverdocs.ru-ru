# <a name="appendix-b-privileged-accounts-and-groups-in-active-directory"></a>Приложение B. Привилегированные учетные записи и группы в Active Directory

>Область применения. Windows Server 2016, Windows Server 2012 R2, Windows Server 2012


## <a name="appendix-b-privileged-accounts-and-groups-in-active-directory"></a>Приложение B. Привилегированные учетные записи и группы в Active Directory  
«Привилегированными» учетных записей и групп в Active Directory являются те, к которым эффективные права, права и разрешения предоставляются, позволяющие им выполнять практически любые действия в Active Directory и в системах, присоединенных к домену. В этом приложении начинается с описания права, права доступа, и разрешения, следуют сведения о «максимальные привилегия» учетных записей и групп в Active Directory, то есть самые мощные учетных записей и групп.  

Также предоставляется информация о встроенных и по умолчанию учетных записей и групп в Active Directory, а также их права. Несмотря на то, что рекомендуемые конфигурации для защиты максимальной привилегированных учетных записей и групп предоставляются в виде отдельных приложений, в этом приложении сведения общего характера, позволяющая определить пользователей и группы, которые следует сосредоточиться на обеспечение безопасности . Это необходимо делать так, как они могут использоваться злоумышленниками для компрометации и даже удалить установки Active Directory.  

### <a name="rights-privileges-and-permissions-in-active-directory"></a>Права, права и разрешения в Active Directory  
Различия между правами, разрешений и прав может оказаться запутанным и противоречащих друг другу, даже в пределах документации корпорации Майкрософт. В этом разделе описываются некоторые характеристик каждого, так как они используются в этом документе. Эти описания не принимать во внимание следует для другой документации Майкрософт, так как он может использовать эти термины по-разному.  

#### <a name="rights-and-privileges"></a>Права и привилегии  
Права и привилегии действуют те же возможности для всей системы, которые предоставляются субъектов безопасности, например пользователей, служб, компьютеров или групп. В интерфейсы, обычно используемые ИТ-специалистов они обычно называются «права» или «права пользователей», и часто они назначены, объекты групповой политики. На следующем рисунке показаны некоторые из наиболее употребительные права пользователя, которые могут быть назначены субъекты безопасности (он представляет по умолчанию GPO контроллеров домена в домене Windows Server 2012). Некоторые из этих прав применимы к Active Directory, таким как **включить учетные записи компьютеров и пользователей доверенным для делегирования** право, хотя другие права применимы к операционной системе Windows, таким как **изменения системы время**.

#### <a name="permissions"></a>Разрешения  
Разрешения являются элементы управления доступом, которые применяются к защищаемым объектам, например файловой системы, реестра, службы и объекты Active Directory. Каждый защищаемый объект имеет связанные управления доступом (ACL), содержащий записи управления доступом (ACE), предоставляющие или запрещающие возможность выполнять различные операции объекта субъектов безопасности (пользователей, служб, компьютеры или группы). Например списки управления доступом для нескольких объектов в Active Directory содержат записи ACE, разрешить проверку подлинности пользователям читать Общие сведения об объектах, но не предоставить им возможность считывать конфиденциальные сведения или изменять объекты.
За исключением каждого домена встроенная учетная запись гостя каждого участника безопасности, который входит в систему и проходит проверку подлинности с помощью контроллера домена в лесу Active Directory или в доверенном лесу есть прошел проверку подлинности пользователей идентификатор безопасности (SID) добавить к ней доступ токен по умолчанию. Таким образом ли пользователя, службы или учетной записи компьютера пытается прочитать общие свойства в объектах user в домене, чтения операция выполнена успешно.  

Если попытки участника безопасности, доступа к объекту, для которого без записей ACE определены и которые содержат SID, который присутствует в маркере доступа участника, участник не может доступа к объекту. Кроме того Если элемент управления ДОСТУПОМ в список управления Доступом объекта содержит запись deny для безопасности, соответствующий маркер доступа пользователя, элемент управления ДОСТУПОМ «deny» будет обычно конфликтующее переопределение «разрешить» ACE. Дополнительные сведения об управлении доступом в Windows см. в разделе [контроля доступа](https://msdn.microsoft.com/library/aa374860(v=VS.85).aspx) на веб-сайте MSDN.  

В этом документе разрешения относится к возможностям, которые предоставлены или запрещены для субъектов безопасности на защищаемые объекты. Каждый раз, когда возникает конфликт между пользователю права и разрешения, право пользователя обычно имеет приоритет. Например если объект в Active Directory был настроен с помощью ACL, запрещает администраторам все чтение и запись к объекту, пользователь, являющийся членом группы администраторов домена будет не удается просмотреть много информации об объекте. Тем не менее так как группы "Администраторы" предоставляется пользователем вправо «Смена владельца файлов или других объектов», пользователь можно просто стать владельцем объекта, рассматриваемый, а затем переписать ACL данного объекта, для предоставления администраторам полного объекта.  

Именно по этой причине, что в этом документе рекомендует, чтобы избежать с помощью мощных учетных записей и групп для повседневного администрирования, вместо того чтобы ограничить возможности учетных записей и групп. Это не эффективно возможность остановить пользовательской контролировать доступ к мощным учетные данные с помощью этих учетных данных для получения доступа к любому ресурсу защищаемого объекта.  

### <a name="built-in-privileged-accounts-and-groups"></a>Встроенные привилегированные учетные записи и группы  
Active Directory предназначен для упрощения делегирования, администрирования и принципа наименьших прав доступа в назначении прав и разрешений. «Regular» пользователи с учетными записями в домене Active Directory, по умолчанию, может считывать большую часть того, что содержится в каталоге, но могут изменять только ограниченный набор данных в каталоге. Пользователям, которым требуется дополнительная привилегия можно предоставить членство в различных привилегированных групп, которые встроены в каталоге, что они могут выполнять определенные задачи, связанные с их ролями, но не могут выполнять задачи, которые не относятся к их обязанностей.  

В Active Directory имеются три встроенные группы, которые составляют наивысший привилегий группам в каталоге: в группу "Администраторы" Enterprise (EA), в группу "Администраторы" домена (DA) и встроенная группа администраторов (BA).  

Четвертый группе, в группу "Администраторы" схемы (SA), имеет права, которые могут повредить или уничтожить весь лес Active Directory, если злоупотребляли, но эта группа является более ограниченные свои возможности, чем EA, DA и BA групп.  

Помимо этих четырех групп существует ряд дополнительных встроенные учетные записи по умолчанию и групп в Active Directory, каждый из которых предоставляется права и разрешения, которые позволяют выполнять определенные административные задачи. Несмотря на то, что в этом приложении не предоставляет подробное описание того, каждая группа встроенной или значение по умолчанию в Active Directory, он предоставляет таблицу групп и учетных записей, которые вы скорее всего, см. в разделе, при установке.  

Например при установке сервера Exchange в лесу Active Directory, дополнительные учетные записи и группы могут создаваться в контейнерах встроенные и пользователей в ваших доменов. В этом приложении описываются только группы и учетные записи, созданные в контейнерах встроенные и пользователей в Active Directory, на основе собственного ролей и компонентов. Учетные записи и группы, которые создаются для установки корпоративного программного обеспечения, не включаются.  

#### <a name="enterprise-admins"></a>Администраторы предприятия  
В группу "Администраторы" Enterprise (EA) находится в корневом домене леса, и по умолчанию, он является членом встроенной группы администраторов в каждом домене в лесу. Встроенной учетной записи администратора в корневом домене леса является единственным членом по умолчанию группе EA. EAs предоставляются права и разрешения, которые позволяют им для изменения уровня леса. Это изменения, влияющие на все домены в лесу, таких как добавление или удаление доменов, установлении доверия лесов или Повышение режима работы леса. В модели делегирования правильно спроектированная и реализованная EA членства является обязательным только в том случае, если сначала создав леса, или при внесении определенных изменений уровня леса, такими как определение при исходящем доверии леса.  

По умолчанию в контейнере пользователей в корневом домене леса расположен группе EA и это универсальная группа безопасности, если корневого домена леса не работает в смешанном режиме Windows 2000 Server, в котором случае группа является глобальной группой безопасности. Несмотря на то, что некоторые права предоставляются непосредственно группе EA, многие из этой группы права, фактически наследуются группе EA, так как он является членом группы администраторов в каждом домене в лесу. "Администраторы предприятия" никаких прав по умолчанию на рабочих станциях и серверах.  

#### <a name="domain-admins"></a>Администраторы домена  
Каждый домен в лесу имеет собственную группу "Администраторы" домена (DA), которая является членом домена по встроенной группы администраторов (BA) помимо является членом локальной группы администраторов на каждом компьютере, который присоединен к домену. По умолчанию единственным членом группы DA для домена учетная запись встроенной учетной записи администратора для этого домена.  

DAs, обладающую всеми возможными правами в свои домены, в том случае, когда EAs привилегии уровня леса. В модели делегирования правильно спроектированная и реализованная, DA членства должны только в сценариях «аварийная», которые являются ситуации, в которой необходима учетная запись с высоким уровнем привилегий на каждом компьютере в домене или при определенных домене Google необходимо внести изменения. Несмотря на то, что собственные механизмы делегирования Active Directory разрешить делегирование, при условии, что его можно использовать только в сценариях аварийного DA учетные записи, создав модель действующие делегирования может занять много времени, и многие организации используют сторонние приложения, чтобы ускорить процесс.  

DA группа является глобальной группой безопасности размещаются в контейнере пользователей домена. Есть одна группа DA для каждого домена в лесу и по умолчанию единственным членом группы DA запись встроенной учетной записи администратора домена. Так как группы домена DA вложен в группе BA домена и локальную группу администраторов системы каждого присоединенного к домену, DAs не только иметь разрешения, предоставляемые специально для "Администраторы домена", но они также наследуют все права и разрешения, предоставляемые группы администраторов домена и локальной группы администраторов на всех компьютерах, присоединенных к домену.  

#### <a name="administrators"></a>Администраторы  
Встроенная группа администраторов (BA) является локальной группой домена в домене встроенный контейнер, куда вложены DAs и EAs, и это этой группе, имеющей многие права прямого доступа и разрешения в каталоге, а также на контроллерах домена. Тем не менее в группу администраторов домена имеет какие-либо права на рядовых серверах или на рабочих станциях. Является членство в локальной группы администраторов компьютеров домена, где предоставляется локальных привилегий; они обсуждаются группы только DAs членами локальных групп администраторов всех присоединенных к домену компьютеров по умолчанию.  

Группы "Администраторы" — это группа Локальная в домене в встроенный контейнер домена. По умолчанию каждый домен BA группу локального домена встроенной учетной записи администратора, DA группы локального домена и группы EA корневого домена леса; Многие права пользователя в Active Directory, а также на контроллерах домена предоставляются специально для группы администраторов, чтобы не EAs или DAs. BA группы домена предоставляется полный доступ для большинства объектов каталога, а также могут стать владельцами объектов каталога. Несмотря на то, что группы EA и DA предоставлены определенные разрешения связанных с объектами в лес и домены, значительную часть мощности групп фактически «наследует» их членства в группах бизнес-Аналитики.  

> [!NOTE]  
> Несмотря на то, что они являются конфигурациями по умолчанию эти привилегированных групп, членом любой из трех групп можно управлять каталога для получения членства в любых других групп. В некоторых случаях это просто достичь, в других случаях это является более сложной, но с точки зрения возможных привилегий, все три группы должно считаться аналогично оператору.  

#### <a name="schema-admins"></a>Администраторы схемы  
В группу "Администраторы" схемы (SA) является универсальной группой в корневом домене леса и имеет только учетная запись встроенной учетной записи администратора домена, что элемент по умолчанию, аналогичную группе EA. Несмотря на то, что членство в группе SA может дать злоумышленнику злоумышленнику схемы Active Directory, которая предназначена для всего леса Active Directory, SAs имеют несколько права по умолчанию и разрешений за пределы схемы.  

Следует тщательно управлять и отслеживать членство в группе SA, но в некоторых отношениях, эта группа является «менее привилегированные «чем трех наивысший привилегированных групп, описанных ранее, так как область действия его прав очень мала; то есть SAs имеет административные права в любом месте, отличной от схемы.
