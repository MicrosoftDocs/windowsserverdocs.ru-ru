# <a name="audit-policy-recommendations"></a>Рекомендации по политике аудита

>Область применения: Windows Server 2016, Windows Server 2012 R2, Windows Server 2012

В этом разделе рассматриваются параметры политики аудита Windows по умолчанию, базовых рекомендуется использовать параметры политики аудита, а также более активную рекомендации от корпорации Майкрософт, для продуктов для рабочей станции и сервера.  

Базовые рекомендации SCM, показанный здесь, вместе с параметрами, которые мы рекомендуем для обнаружения нарушений безопасности, предназначены только для начала руководства базовых показателей для администраторов. Каждая организация должна принятия решений относительно угрозы, которые могут возникнуть, их отклонения приемлемым риском и какой категории политики аудита или подкатегории, их следует включить. Сведения об угрозах называть [по угрозам и мерам](https://technet.microsoft.com/library/hh125921(v=ws.10).aspx). Администраторы без политики аудита тщательные на месте, рекомендуется начать с параметрами, рекомендуется использовать, а затем для изменения и проверки перед их реализацией в рабочей среде.  

Рекомендации предназначены для компьютеров корпоративного уровня, которые корпорация Майкрософт определяет, как компьютеры, требованиями к безопасности, средний и высокий уровень оперативную функциональность требуется. Объекты, требуются более высокого уровня защиты, которые следует учитывать более активную требования политики аудита.  

> [!NOTE]  
> По умолчанию Microsoft Windows и Базовые рекомендации были выполнены из [средство Microsoft Security Compliance Manager](https://technet.microsoft.com/library/cc677002.aspx).  

Рекомендуется использовать следующие параметры политики аудита базовых показателей для обычной безопасности компьютеров, которые не известно должно находиться в активных, успешных атак злоумышленников или вредоносных программ.

## <a name="set-audit-policy-on-workstations-and-servers"></a>Настроить политику аудита на рабочих станциях и серверах  
Все планы управления журнала событий следует отслеживать рабочих станциях и серверах. Очень часто встречается только мониторинг серверов и контроллеров домена. Поскольку вредоносных программ взлома часто изначально выполняется на рабочих станциях, мониторинг рабочие станции не пропускает лучшие и ранняя источник информации.  

Администраторы продуманным следует просмотреть и проверить все политики аудита перед реализацией в своей рабочей среде.  

## <a name="events-to-monitor"></a>События для мониторинга  
Идентификатор события идеальным для создания оповещения системы безопасности, должны иметь следующие атрибуты:  

-   Высокая вероятность того что возникновение указывает на несанкционированные действия.  

-   Небольшое число ложных срабатываний  

-   Возникновение должно приводить к расследование или экспертиза.  

Необходимо оценить и оповещения два типа событий:  

1.  Эти события, даже однократное возникновение которых указывает на несанкционированные действия  

2.  Накопление количества событий выше ожидаемых и допустимых базовых показателей  

Ниже показан пример первого события.  

Если Администраторы домена (DAs) запрещен вход в систему на компьютерах, которые не являются контроллерами домена, единственный член DA, войдя в систему рабочей станции для конечных пользователей следует создавать оповещения и требует изучения. Оповещение этого типа можно легко создать с помощью аудит специального входа событие 4964 (специальным группам назначен новый сеанс входа). Другие примеры предупреждения одного экземпляра.  

-   Если сервер A никогда не следует подключить на сервер B, предупреждение, если они подключаются друг к другу.  

-   Предупреждение, если учетная запись обычного пользователя неожиданно добавляется в группу безопасности конфиденциальных.  

-   Если сотрудники в расположении фабрики A никогда не работают ночью, оповещения при входе пользователя в систему в полночь.  

-   Предупреждение, если несанкционированного служба установлена на контроллере домена.  

-   Исследуйте попытка регулярных для конечных пользователей напрямую войдите в систему SQL Server, для которого у них есть очистить причины этого.  

-   Если вы не имеют членов в группы DA, а другой человек добавляет сами существует, проверьте его сразу.  

Ниже показан пример второго события.  

Ошибочное число неудачных попыток входа может указывать пароль, атаки. В организации для обеспечения оповещение необычно большое число неудачных попыток входа их необходимо понимать нормальном уровне неудачных попыток входа в их среде до события вредоносных безопасности.  

Полный список событий, которые следует включить при мониторинге для обнаружения признаков компрометации см. в разделе [приложение м. события для мониторинга](../../../ad-ds/plan/Appendix-L--Events-to-Monitor.md).  

## <a name="active-directory-objects-and-attributes-to-monitor"></a>Объекты Active Directory и атрибуты для монитора  
Ниже приведены учетных записей, групп и атрибуты, которые необходимо отслеживать для обнаружения попытки создать угрозу установки доменных служб Active Directory.  

-   Системы для отключения или удаления программ антивирусных программ и защиты от вредоносных программ (автоматически перезапуске защиты вручную отключенного)  

-   Учетные записи администраторов несанкционированные изменения  

-   Действия, которые выполняются с использованием привилегированных учетных записей (автоматически удалить учетную запись при завершении подозрительных действий или отведенное время истечения срока действия)  

-   Привилегированных и ВАЖНЫХ учетных записей в Доменных службах Active Directory. Отслеживайте изменения, особенно изменения в атрибутах на вкладке "учетная запись" (например, cn, имя, sAMAccountName, userPrincipalName или userAccountControl). Помимо наблюдения за учетных записей, ограничения, кто может изменять учетные записи в качестве небольшой набор административных пользователей, насколько это возможно.  

Обратитесь к [приложение м. события для мониторинга](../../../ad-ds/plan/Appendix-L--Events-to-Monitor.md) список рекомендуемых события монитор, их категории важности и Сводка событий сообщений.  

-   Группа серверов по классификации рабочих нагрузок, который позволяет быстро определять серверов, которые должны быть наиболее точно отслеживаемых и наиболее строго, настроенных  

-   Изменения свойств и членство в следующие группы в Доменных службах Active Directory: администраторов предприятия (EA), администраторов домена (DA), Администраторы (BA) и администраторов схемы (SA)  

-   Отключенные привилегированных учетных записей (например, встроенных учетных записей администраторов в Active Directory, а также в системах члена) для включения учетные записи  

-   Управление учетными записями в журнал все операции записи к учетной записи  

-   Встроенные мастер настройки безопасности для настройки службы, реестра, аудита и параметры брандмауэра для уменьшения поверхности атаки на сервере. Используйте этот мастер, если вы реализуете серверами перехода в рамках стратегии администратора узла.

## <a name="general-list-of-security-event-id-recommendation-criticalities"></a>Общий список Criticalities рекомендация ИД событий безопасности  
Все рекомендации событие с Идентификатором сопровождаются важности, оценка следующим образом:  

**Высокий:** идентификаторы событий с высоким уровнем важности оценку следует всегда и немедленно оповещения и изучения.  

**Средний:** идентификатор события с оценкой средний уровень важности может указывать на вредоносную деятельность, но оно должно сопровождаться некоторые другие abnormality (например, необычных номер в определенный период времени, Непредвиденная вхождений или вхождений на компьютере, который обычно не ожидается событий в.). Событие средний важности может также r заноситься в качестве метрики и по сравнению с течением времени.  

**Низкий:** и событие с кодом события низкий уровень важности не привлекают внимание или вызвать предупреждения, если не связаны с событиями, средним или высоким уровнем важности.  

Эти рекомендации предназначены для предоставления руководстве базовых показателей для администратора. Все рекомендации должны тщательно проверяться перед реализацией в рабочей среде.  

Обратитесь к [приложение м. события для мониторинга](../../../ad-ds/plan/Appendix-L--Events-to-Monitor.md) список рекомендуемых события монитор, их категории важности и Сводка событий сообщений.
