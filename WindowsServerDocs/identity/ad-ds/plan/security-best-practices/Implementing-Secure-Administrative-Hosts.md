# <a name="implementing-secure-administrative-hosts"></a>Реализация безопасного администрирования узлов

>Область применения. Windows Server 2016, Windows Server 2012 R2, Windows Server 2012

Защищенные административные узлы являются рабочих станциях и серверах, которые были настроены специально для целей создания безопасной платформы, из которых привилегированных учетных записей может выполнять административные задачи в Active Directory или на контроллерах домена, присоединенных к домену систем и приложений, работающих на компьютерах, присоединенных к домену. В этом случае «привилегированных учетных записей» относится не только к учетным записям, которые являются членами наиболее привилегированных групп в Active Directory, но и ко всем учетным записям, которые были делегированные права и разрешения, которые позволяют выполнять административные задачи.  
  
Возможно, эти учетные записи службы технической поддержки учетных записей, которые имеют возможность сбрасывать пароли для большинства пользователей домена, учетные записи, используемые для администрирования зон и записей DNS, или учетными записями, используемыми для управления конфигурацией. Защищенные административные узлы выделены для административные функции, и они не работают под управлением программного обеспечения, такие как почтовые приложения, веб-браузеров или офисных приложений для таких как Microsoft Office.  
  
Несмотря на то, что «наибольшие права» учетные записи и группы должен быть соответствующим образом наиболее компиляторам защищенный, это не устраняет необходимость защищать учетные записи и группы, к полномочия выше стандартный пользователь предоставил учетные записи.  
  
Надежного администратора узла можно выделенной рабочей станции, который используется только для административных задач, рядовой сервер, на котором выполняется роль сервера шлюза удаленных рабочих столов и для пользователей, которые ИТ подключитесь к администрировать узлы назначения или сервер под управлением роль Hyper-V и предоставляет уникальное виртуальную машину для каждого пользователя ИТ для задачи администрирования. Во многих средах могут быть реализованы сочетание всех трех методов.  
  
Реализация безопасного администрирования узлов требуется планирование и конфигурации, которая согласуется с размером, методов администрирования, аппетит риска и бюджета организации. Вопросы и вариантов реализации защищенные административные узлы, указанные здесь для использования в разработке стратегии администрирования подходит для вашей организации.  
  
## <a name="principles-for-creating-secure-administrative-hosts"></a>Принципы защищенные административные узлы  
Эффективно защищать системы от атак, следует иметь в виду несколько общих принципов:  
  
1.  Вы никогда не должны администрировать доверенной системы (то есть безопасном сервере как контроллер домена) из менее доверенного узла (то есть, рабочая станция, не защищены в такой же степени, что системы, которыми она управляет).  
  
2.  Не следует полагаться на одного фактора проверки подлинности при выполнении привилегированных действий; то есть сочетания имени и пароля пользователя не должен считаться проверки подлинности так, как представляется однофакторных (то, что вы знаете). Следует учитывать, где созданный и кэширования или хранятся учетные данные в сценариях администрирования.  
  
3.  Несмотря на то, что большинство атак в современных угроз используют вредоносного программного обеспечения и вредоносных взлома, не Опускайте физической безопасности при разработке и реализации защищенные административные узлы.  
  
### <a name="account-configuration"></a>Конфигурация учетной записи  
Даже если ваша организация не использует смарт-карты в настоящее время, следует реализовать их для привилегированных учетных записей и защищенные административные узлы. Административные узлы должны требуют входа с помощью смарт-карты для всех учетных записей, изменив следующий параметр в объекте групповой Политики, связанного с подразделений, содержащих узлов администрирования:  
  
**Вход в систему компьютера компьютера\Политики\Параметры безопасности\Локальные политики\Параметры Options\Interactive: Требуется смарт-карта**  
  
Этот параметр требует все интерактивный вход в сеть для использования смарт-карты, независимо от конфигурации на отдельную учетную запись в Active Directory.  
  
Можно также настроить защищенные административные узлы, чтобы разрешить вход в систему только с авторизованных учетных записей, которые могут быть настроены в:  
  
**Конфигурация компьютера\Политики\Параметры Параметры безопасности\Локальные политики\Параметры безопасности\Локальные Политики\назначение прав**  
  
Это предоставляет интерактивный (и, где уместно, службы удаленных рабочих столов) права на вход в систему только авторизованным пользователям безопасного администрирования узла.  
  
### <a name="physical-security"></a>Физическая безопасность  
Для узлов администрирования, чтобы считаться trustworthy их следует настраивать и защищенных в такой же степени, что системы, которыми они управляют. Почти все действия, описанные в рекомендации [защита контроллеры домена от атак](../../../ad-ds/plan/security-best-practices/Securing-Domain-Controllers-Against-Attack.md) также применимы к узлам, которые используются для администрирования контроллеров домена и базы данных AD DS. Одной из проблем реализации систем безопасности администратора в большинстве сред является физической безопасности, что может быть более сложно, так как эти компьютеры часто находятся в областях, которые не так безопасен, как серверы, размещенные в центрах обработки данных, таких как административные пользователи настольных компьютеров.  
  
Физическая безопасность включает в себя контроля физического доступа к узлах администрирования. В небольшой организации это может означать вести заблокированной выделенный станции администратора, который хранится в office или ящике неиспользуемых. Или может означать, когда вам нужно выполнять администрирование Active Directory или контроллеров домена, выполнять вход в контроллер домена напрямую.  
  
В организациях среднего размера, можно реализовать безопасный административные «серверы перехода», находятся в безопасном месте в офисе и используются при необходимости управления Active Directory и контроллерами доменов. Кроме того, могут также реализовывать административных рабочих станций, которые заблокированы в безопасных местах не во время работы, независимо от серверы перехода.  
  
В крупных организациях можно развернуть серверы перехода, размещенное в центре обработки данных, обеспечивающие строго контролируемый доступ к Active Directory; контроллеры домена; и файлов, печати или серверов приложений. Реализация архитектуру сервера перехода, скорее всего, для включения сочетание безопасности рабочих станций и серверов в крупных средах.  
  
Независимо от размера вашей организации и структуры администрирования узлов необходимо обеспечить защиту физических компьютеров от несанкционированного доступа или кража и следует использовать шифрование диска BitLocker для шифрования и защиты дисков на узлах администрирования . Путем реализации BitLocker на узлах администрирования, даже кражи узла или его диски будут удалены, убедитесь, что данные на диске является недоступной для неавторизованных пользователей.  
  
### <a name="operating-system-versions-and-configuration"></a>Версии операционной системы и конфигурации  
Все административные узлы, серверы или рабочие станции, следует ли запускать последние версии ОС используется в вашей организации по причинам, описанным ранее в этом документе. Запустив текущими операционными системами, административному персоналу преимущества новых функций безопасности, поддержку полного поставщика и дополнительные функциональные возможности, появившиеся в операционной системе. Кроме того при оценке новой операционной системы, развернув узлы первой для администратора, необходимо ознакомиться с новыми функциями, параметры и механизмы управления, предлагаемыми ею, которые впоследствии могут использоваться в планировании широкого развертывания операционной системы. Потом самым опытным пользователям в вашей организации также будет пользователей, которые знакомы с новой операционной системы и лучше всего позиционированные для его поддержки.  
  
### <a name="microsoft-security-configuration-wizard"></a>Мастер настройки безопасности Майкрософт  
Если вы реализуете серверы перехода, как часть стратегии администратора узла, следует использовать встроенные мастера настройки безопасности для настройки службы, реестра, аудита и параметры брандмауэра, чтобы уменьшить уязвимость сервера. После сбора и настройки параметров конфигурации мастер настройки безопасности параметров могут преобразовываться к объекту групповой Политики, который используется для принудительного применения согласованного базовой конфигурации на всех серверах перехода. Затем можно редактировать объект групповой Политики для реализации параметров безопасности, относящиеся к серверы перехода и всех параметров с помощью дополнительных базовых параметров, извлеченных из Microsoft Security Compliance Manager.  
  
### <a name="microsoft-security-compliance-manager"></a>Microsoft Security Compliance Manager  
[Microsoft Security Compliance Manager](https://technet.microsoft.com/library/cc677002.aspx) — можно бесплатно средство, которое интегрирует конфигурации безопасности, которые рекомендуются корпорацией Майкрософт, в зависимости от версии и роль конфигурации операционной системы и собирает их в единое средство и пользовательский Интерфейс, который может использоваться для создания и настройки базовых показателей конфигурации безопасности для контроллеров домена. Шаблоны Microsoft Security Compliance Manager может сочетаться с параметры мастера настройки безопасности для создания комплексное конфигурационных для серверы перехода, которые развертываются и принудительно объектами групповой политики, развернутых в подразделения, в какие перехода, серверы найти в Active Directory.  
  
> [!NOTE]  
> На момент написания этой статьи Microsoft Security Compliance Manager не поддерживает параметры, относящиеся к серверы перехода или другие защищенные административные узлы, но по-прежнему Security Compliance Manager (SCM) можно использовать для создания начальной базовых показателей для администрирования узлы. Тем не менее, для защиты узлов, необходимо применить дополнительные параметры для высокой степенью безопасности рабочих станций и серверов.  
  
### <a name="applocker"></a>AppLocker  
Административные узлы и виртуальные machinesshould настроить с помощью сценария, средства и разрешенных приложений с помощью AppLocker или ограничений приложения стороннего программного обеспечения. Административные приложения или служебные программы, которые не соответствуют параметры безопасности должны обновления или замены с инструментарием, который соответствует безопасной разработки и администрирования. Когда на узле администрирования требуется новое или дополнительное оборудование, приложений и служебных программ должны тщательно тестироваться, а если инструментарий подходит для развертывания на узлах администрирования, его можно добавить разрешенных систем.  
  
### <a name="rdp-restrictions"></a>Ограничения протокола удаленного рабочего СТОЛА  
Несмотря на то, что конкретная конфигурация зависит от архитектуру администрирования систем, следует включить ограничения, на которых учетные записи и компьютеров может использоваться для подключения удаленного рабочего стола (RDP) на управляемых системах Например, с помощью шлюза удаленных рабочих столов (шлюз RD) выполняется переход серверы для управления доступ к контроллерам домена и других управляемых систем из авторизованных пользователей и систем.  
  
Следует разрешить интерактивный вход авторизованными пользователями и следует удалить или заблокировать другие типы входа, которые не требуются для доступа к серверу.  
  
### <a name="patch-and-configuration-management"></a>Исправления и управления конфигурацией  
Небольших организаций может полагаться на предложения, такие как Windows Update или [Windows Server Update Services](https://technet.microsoft.com/windowsserver/bb332157) (WSUS) по управлению развертывание обновлений для системы Windows, в том случае, когда в более крупных организациях может быть реализована enterprise patch и программы управления конфигурацией например System Center Configuration Manager. Независимо от того, механизмы, используемой для развертывания обновлений для общего сервера и заполнение рабочей станции следует рассмотреть отдельные развертывания для высокий уровень безопасности систем, таких как контроллеры домена, центров сертификации и узлах администрирования. По возможности размещая эти системы из-за общее управление инфраструктуры, компрометация учетных записей по или службы управления компромисс не так легко расширять наиболее обеспечить безопасность систем в вашей инфраструктуре.  
  
Несмотря на то, что не следует реализовывать процессы обновления вручную для безопасных систем, следует настроить инфраструктуру, отдельные обновления систем безопасности. Даже в очень крупных организаций эту инфраструктуру можно обычно реализовать с помощью выделенные серверы WSUS и объекты групповой политики для защищенных систем.  
  
### <a name="blocking-internet-access"></a>Блокировать доступ к Интернету  
Административные узлы не должен быть разрешен доступ к Интернету, а также должны быть возможность перейти интрасети организации. Веб-браузеров и других подобных приложений не должен быть разрешен на узлах администрирования. Вы можете заблокировать доступ к Интернету для защищенные узлы с помощью сочетания параметров брандмауэра периметра WFAS и настройки конфигурации прокси-сервера «черных дыр» на защищенные узлы. Также можно применять список разрешений для предотвращения использования на узлах администрирования веб-браузеров.  
  
### <a name="virtualization"></a>Виртуализация  
Если это возможно, рассмотрите возможность реализации виртуальных машин в качестве узлов администрирования. С помощью виртуализации, можно создать пользователя администратора системы, централизованно хранятся и управляются и который можно легко отключить работу, если в нем нет гарантируя, что учетные данные не остаются active на администрирования систем. Можно также указать, что виртуальных узлов администрирования сбрасываются до исходного моментального снимка после каждого использования, гарантируя, что виртуальные машины остаются в первоначальном состоянии. Дополнительные сведения о параметрах для виртуализации узлам администрирования предоставляется в следующем разделе.  
  
## <a name="sample-approaches-to-implementing-secure-administrative-hosts"></a>Пример подхода к реализации защиты узлов администрирования  
Независимо от того, проектирование и развертывание инфраструктуры администратора узла следует Имейте в виду рекомендациями из «Принципы для создания защиты узлов администрирования» ранее в этом разделе. Каждый из подходов, описанных здесь предоставляет общие сведения о том, как можно отделить «административная» и «производительность» системы, используемые ИТ-специалисты. Производительность системы, компьютеры, ИТ-администраторы используют для проверки электронной почты, в Интернете и использовать общие офисных приложений, например Microsoft Office. Административные систем – компьютеры, которые будут записаны на диск и выделенной для использования для повседневного администрирования среды ИТ.  
  
— Это самый простой способ реализации защищенные административные узлы для предоставления ИТ-специалисты с помощью защищенных рабочих станций, из которых они могут выполнять задачи администрирования. При реализации только для рабочей станции каждая рабочая станция администратора используется для запуска средства управления и подключения удаленного рабочего СТОЛА для управления серверами и другой инфраструктуры. Реализации только для рабочей станции может быть эффективным в небольших организациях, несмотря на то, что более крупных и сложных инфраструктур выгоден распределенной архитектуры для узлов администрирования, в котором выделенных административных серверов и рабочих станциях используются, как в «Реализация защиты административных рабочих станций и переходов серверы» описываются далее в этом разделе.  
  
### <a name="implementing-separate-physical-workstations"></a>Реализация отдельных физических рабочих станций  
Один из способов, что вы можете реализовать узлов администрирования — Отправка каждого пользователя ИТ два рабочих станций. Одной рабочей станции используется с учетной записью «regular» для выполнения действий, таких как проверка электронной почты и с помощью приложений, а вторая рабочая станция будет выделен для работы исключительно к административным функциям.  
  
Для повышения производительности рабочей станции ИТ-специалисты могут получить учетные записи обычных пользователей, а не с помощью привилегированных учетных записей для входа на незащищенных компьютерах. Рабочая станция администратора должны быть настроены компиляторам управляемой конфигурации и ИТ-специалистам следует использовать другую учетную запись для входа на станции администратора.  
  
Если вы реализовали смарт-карт, рабочие станции администраторов должно быть настроено обязательное забытых смарт-карт и ИТ-специалистам следует задавать отдельные учетные записи для администратора, также настроено обязательное использование смарт-карты для интерактивного входа в систему. Администратора узла должен быть жесткой как описано выше, а только определенным пользователям ИТ должны локальный вход на станции администратора.  
  
#### <a name="pros"></a>Плюсы  
Путем реализации отдельных физических системах, можно убедиться, что каждый компьютер настроен соответствующим образом его роль и что ИТ-пользователи не могут случайно предоставлять администратора системы риску.  
  
#### <a name="cons"></a>Минусы  
  
-   Реализация отдельных физических компьютерах увеличит расходы на оборудование.  
  
-   Вход в систему на физический компьютер с учетными данными, которые используются для администрирования удаленных систем кэширует учетные данные в памяти.  
  
-   Если административных рабочих станций не хранятся безопасно, они может оказаться уязвим для атак с помощью механизмов, например нажатий клавиш физического оборудования или других физических атак.  
  
### <a name="implementing-a-secure-physical-workstation-with-a-virtualized-productivity-workstation"></a>Реализация безопасной физической рабочей станции с рабочей станции виртуализированных повышения производительности  
При таком подходе пользователи ИТ, получают доступ к защищенным рабочая станция администратора, из которого они могут выполнять повседневные административные функции, с помощью средств удаленного администрирования сервера (RSAT) или RDP-подключения к серверам в своей области ответственности. Когда ИТ-пользователи должны выполнять рабочие задачи, они могут подключаться по протоколу RDP для повышения производительности удаленной рабочей станции, выполняющийся как виртуальная машина. Следует использовать отдельные учетные данные для каждой рабочей станции, и элементы управления, таких как смарт-карты должен быть реализован.  
  
#### <a name="pros"></a>Плюсы  
  
-   Административных рабочих станций и рабочих станциях разделены.  
  
-   ИТ-специалистов, с использованием безопасных рабочих станций для подключения к рабочих станциях можно использовать отдельные учетные данные и смарт-карты и привилегированные учетные данные не выводятся на компьютере менее безопасных.  
  
#### <a name="cons"></a>Минусы  
  
-   Для реализации решения требуется и работ по реализации вариантов структуры и надежные виртуализации.  
  
-   Если не безопасное хранение физических рабочих станций, возможно, уязвимы для физических атак проникнуть оборудования или операционной системы и сделать их уязвимо для перехвата связи.  
  
### <a name="implementing-a-single-secure-workstation-with-connections-to-separate-productivity-and-administrative-virtual-machines"></a>Реализация одной безопасной рабочей станции с подключениями для разделения «Производительность» и «Административная» виртуальные машины  
При таком подходе можно выдать ИТ-пользователей одной физической рабочей станции, заблокирована как описано выше, и на котором ИТ-пользователи не имеют привилегированный доступ. Можно предоставить подключения служб удаленных рабочих столов для виртуальных машин, размещенных на выделенных серверах, предоставляя ИТ-специалистов с одной виртуальной машины, работающей по электронной почте и других повышающих эффективность приложений, а вторую виртуальную машину, настроенный в качестве пользователя выделенный узел администрирования.  
  
Необходимо требовать смарт-карты или других многофакторного входа в систему для виртуальных машин, используя отдельные учетные записи, отличной от учетной записи, который используется для входа на физическом компьютере. После ИТ пользователя в систему физического компьютера, их можно использовать их производительность смарт-карты для подключения к их производительность удаленного компьютера и отдельную учетную запись и смарт-карты для подключения к их удаленного администрирования компьютера.  
  
#### <a name="pros"></a>Плюсы  
  
-   ИТ-пользователей можно использовать в одной физической рабочей станции.  
  
-   Требующих отдельные учетные записи для виртуальными узлами и с помощью подключений служб удаленных рабочих столов для виртуальных машин учетных записей пользователей ИТ не кэшируются в памяти на локальном компьютере.  
  
-   В такой же степени, как административные узлы, уменьшая вероятность компрометации локального компьютера можно защитить физический узел.  
  
-   В случаях, в которых пользователя ИТ производительности виртуальной машины или их администратора виртуальной машины мог быть скомпрометирован виртуальную машину можно легко сбросить в состояние «известному работающему».  
  
-   Если физический компьютер скомпрометирован, привилегированные учетные данные не будут кэшироваться в памяти, и использование смарт-карт можно предотвращения нарушений безопасности учетных данных, средств регистрации нажатия клавиш.  
  
#### <a name="cons"></a>Минусы  
  
-   Для реализации решения требуется и работ по реализации вариантов структуры и надежные виртуализации.  
  
-   Если не безопасное хранение физических рабочих станций, возможно, уязвимы для физических атак проникнуть оборудования или операционной системы и сделать их уязвимо для перехвата связи.  
  
### <a name="implementing-secure-administrative-workstations-and-jump-servers"></a>Реализация безопасности административных рабочих станций и серверы перехода  
В качестве альтернативы для защиты административных рабочих станций, или в сочетании с ними можно реализовать серверы безопасного перехода, и пользователи с правами администратора могут подключаться к серверам перехода, с помощью удаленного рабочего СТОЛА и смарт-карт для выполнения административных задач.  
  
Серверы перехода следует настроить для выполнения роли шлюза удаленных рабочих столов, чтобы можно было реализовать ограничения на подключения к серверу перехода и на конечных серверах, управление которыми будет осуществляться из него. Если это возможно, следует также установить роль Hyper-V и создать [личные виртуальные рабочие столы](https://technet.microsoft.com/library/dd759174.aspx) или других виртуальных машин на пользователя для администратора пользователи могут использовать свои задачи на серверы перехода.  
  
Предоставляя пользователей с правами администратора виртуальных машин на пользователя на сервере перехода, обеспечения физической безопасности административных рабочих станций и пользователей с правами администратора может сбросить или завершить работу их виртуальных машин не во время работы. Если вы не хотите установить роль Hyper-V и роли шлюза удаленных рабочих столов на одном узле администрирования, их можно установить на разных компьютерах.  
  
По возможности средств удаленного администрирования можно использовать для управления серверами. Средства удаленного администрирования сервера (RSAT) компонент должен устанавливаться на виртуальных машинах пользователей (или сервер перехода, если вы не реализуете виртуальных машин на пользователя для администрирования), и административного персонала должны подключаться по протоколу RDP к их виртуальные машины для выполнения административных задач.  
  
В случаях, когда пользователь с правами администратора должен подключение по протоколу RDP на конечный сервер, чтобы управлять им напрямую, шлюз удаленных рабочих Столов должен разрешить подключение только в том случае, если соответствующий пользователь и компьютер используются для установления подключения к назначению сервер. Выполнение средств удаленного администрирования сервера (или схожий) средств должен быть запрещен в системах, не предназначенные системами управления, такие как общего использования рабочих станций и рядовых серверах, которые являются не серверы перехода.  
  
#### <a name="pros"></a>Плюсы  
  
-   Создание серверы перехода позволяет сопоставить отдельных серверов для «зоны» (коллекций систем со сходными параметрами конфигурации, подключения и безопасности) в сети и требуют, что Администрирование каждой зоны достигается за счет административного персонала подключение из защищенные административные узлы к серверу указанного «zone».  
  
-   Путем сопоставления серверы перехода с зонами, можно реализовать детальные элементы управления для свойства соединения и требования к конфигурации и можно легко определить подключиться через несанкционированного систем.  
  
-   Реализовав прав администратора виртуальных машин на серверы перехода, можно принудительно применить завершение работы и сброс виртуальных машин в известных исходное состояние после выполнения административных задач. Применяя завершение работы или перезапуска виртуальных машин после выполнения административных задач, виртуальные машины не может быть целью злоумышленников, а также не представляется возможным атак кражи учетных данных так, как учетные данные, кэшированных в памяти не сохраняются после перезагрузки.  
  
#### <a name="cons"></a>Минусы  
  
-   Выделенные серверы являются обязательными для серверы перехода, физические и виртуальные.  
  
-   Реализация назначенные серверы перехода и административных рабочих станций требует тщательного планирования и конфигурации, который сопоставляется любой зоны безопасности, настроены в среде.
