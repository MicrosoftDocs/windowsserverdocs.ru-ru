# <a name="implementing-secure-administrative-hosts"></a>Реализация безопасного администрирования узлов

>Область применения: Windows Server 2016, Windows Server 2012 R2, Windows Server 2012

Защищенные административные узлы, рабочие станции или серверы, настроенными специально для целей создания безопасного платформ, из которых привилегированных учетных записей могут выполнять задачи администрирования в Active Directory или на контроллерах домена, присоединенных к домену систем и приложений, работающих на компьютерах, присоединенных к домену. В этом случае «привилегированные учетные записи» относится не только к учетным записям, которые являются членами наиболее привилегированных групп в Active Directory, но все учетные записи, которые были делегированы права и разрешения, позволяющие выполнения административных задач.  
  
Эти учетные записи могут быть учетные записи службы технической поддержки, которые имеют возможность сбрасывать пароли для большинства пользователей в домене, учетные записи, используемые для администрирования DNS-записи и зоны или учетные записи, используемые для управления конфигурацией. Защищенные административные узлы стремимся административные функции, и они не могут выполняться программного обеспечения, например, почтовые приложения, веб-браузеры или программного обеспечения производительности таких как Microsoft Office.  
  
Несмотря на то, что «наиболее привилегированном» учетных записей и групп соответствующим образом должен быть строго наиболее защищенные, это устраняете необходимость в защитить учетные записи и какие права выше обычного пользователя предоставлены учетные записи групп.  
  
Безопасного администрирования узлов могут быть выделенной рабочей станции, который используется только для выполнения задач администрирования, рядовой сервер под управлением роли сервера шлюза удаленных рабочих столов, к которому подключаются пользователи ИТ осуществлять администрирование узлы назначения или сервер, который выполняет роль Hyper-V и предоставляет уникальные виртуальную машину для каждого пользователя ИТ для задачи администрирования. Во многих средах комбинации все три подхода может быть реализован.  
  
Реализация безопасного администрирования узлов требует планирования и настройки, совместимое с размер вашей организации, методов администрирования, позиции в отношении рисков и бюджета. Вопросы и варианты реализация безопасного администрирования узлов приведены ниже для использования в разработке стратегии администрирования подходит для вашей организации.  
  
## <a name="principles-for-creating-secure-administrative-hosts"></a>Принципы безопасного администрирования узлов  
Для эффективной защиты системы от атак, следует иметь в виду несколько общие принципы:  
  
1.  Никогда не следует администрирования доверенных system (то есть безопасный сервер, такой как контроллер домена) с менее доверенного узла (то есть рабочей станции, не защищены на тот же угол систем, которым он управляет).  
  
2.  Не следует полагаться на одного фактора проверки подлинности при выполнении привилегированных действий; то есть сочетаний имени пользователя и пароля не подразумевает проверки подлинности так, как представляется только один фактор (что-то известно). Следует учитывать, где создаваемые и кэшировать или хранятся учетные данные в сценариях с правами администратора.  
  
3.  Несмотря на то, что большинство атак в текущем угроз используют вредоносных программ и вредоносных программ взлома, не пропустите физической безопасности при разработке и реализация безопасного администрирования узлов.  
  
### <a name="account-configuration"></a>Настройка учетной записи  
Даже если ваша организация в настоящее время не использовать смарт-карты, вы должны учитывать при реализации для привилегированных учетных записей и безопасного администрирования узлов. Узлы администрирования должен быть настроен требуют входа с помощью смарт-карт для всех учетных записей, изменив следующий параметр в объекте групповой Политики, привязанный к подразделений, содержащих администрирования узлов:  
  
**Компьютер Конфигурация компьютера\Политики\Параметры безопасности\Локальные политики\Параметры Options\Interactive входа в систему: требовать смарт-карту**  
  
Этот параметр требует все интерактивные входы в систему для использования смарт-карты, независимо от конфигурации на отдельной учетной записью в Active Directory.  
  
Также следует настроить безопасного администрирования узлов, чтобы разрешить вход в систему только в учетных записях авторизованных, которые могут быть настроены в:  
  
**Конфигурация компьютера\Политики\Параметры безопасности\Локальные политики\Параметры безопасности\Локальные Политики\назначение прав**  
  
Это предоставляет интерактивный (и, при необходимости служб удаленных рабочих столов) права на вход в систему только авторизованным пользователям безопасного администрирования узла.  
  
### <a name="physical-security"></a>Физическая безопасность  
Административные узлы считается trustworthy их необходимо настроить и защищены на том же уровне, как системы, которыми они управляют. Большая часть рекомендации, приведенные в [Защита контроллеров домена от атак](../../../ad-ds/plan/security-best-practices/Securing-Domain-Controllers-Against-Attack.md) применяются также для узлов, которые используются для администрирования контроллеров домена и базы данных AD DS. Одна из сложностей реализации безопасной администратора системы в большинстве сред является то, что физической безопасности можно более сложно реализовать, поскольку эти компьютеры часто находятся в области, которые не так безопасны, как серверы, размещенные в центрах обработки данных, например настольные административных пользователей.  
  
Физической безопасности включает в себя управление физический доступ к узлам администрирования. В небольшой организации это может означать поддерживать блокировки выделенных административных рабочих станций, которая хранится в офисе или ящике в случае использования. Или может означать, что при необходимо выполнить администрирования Active Directory или контроллерах домена, вход контроллер домена непосредственно.  
  
В организациях среднего размера, можно попробовать реализация безопасного администрирования «серверы перехода», находятся в безопасном месте в приложении Microsoft office и используются, когда требуется управления Active Directory и контроллерами доменов. Может также реализовать рабочие станции администраторов, которые заблокированы в безопасном расположении в случае использования с или без серверами перехода.  
  
В больших организациях можно развернуть серверы переходов, размещенные в центре обработки данных, предоставляющих строго управляемый доступ к Active Directory; контроллеры домена; и файл, печати или серверов приложений. Реализация архитектуру сервера перехода, скорее всего, для включения в больших средах сочетание безопасных рабочих станциях и серверах.  
  
Независимо от размера организации и структуры администрирования узлов следует защитить физических компьютеров от несанкционированного доступа или кража и следует использовать шифрование диска BitLocker для шифрования и защиты диски на узлах администрирования. Путем реализации BitLocker на узлах администрирования, даже в том случае, если узел будет украден или его диски удалены, вы обеспечите что данные на диске будут недоступными для неавторизованных пользователей.  
  
### <a name="operating-system-versions-and-configuration"></a>Версии операционной системы и конфигурации  
Все административные узлы ли серверы или рабочие станции, следует запустить самую последнюю операционную систему, используемых в вашей организации по причинам, описанным выше в данном документе. Запустив текущей операционной системы, административные персонал преимущества новых функций безопасности, поддержки полный поставщиков и дополнительные функциональные возможности, появившиеся в операционной системе. Кроме того при новой операционной системы и при оценке, развертывая ее сначала администрирования узлов, необходимо ознакомиться с новые компоненты, параметры и механизмы управления, предоставляемые, что впоследствии может использоваться в планировании широкого развертывания операционной системы. Потом самых изощренных пользователи в вашей организации также будет пользователей, которые знакомы с новой операционной системы и лучше всего находится для поддержки этой функции.  
  
### <a name="microsoft-security-configuration-wizard"></a>Мастер настройки безопасности Майкрософт  
Если вы реализуете серверами перехода в рамках стратегии администрирования узлов, следует использовать встроенные мастер настройки безопасности для настройки службы, реестра, аудита и параметры брандмауэра для уменьшения поверхности атаки на сервере. После сбора и настройки параметров конфигурации мастер настройки безопасности параметры можно преобразовать в объект групповой Политики, который используется для обеспечения согласованного базовую конфигурацию на всех серверах переходов. Затем можно редактировать объект групповой Политики для реализации специальные параметры безопасности для серверы перехода и можно объединить все параметры, дополнительные базовые параметры, извлеченные из Microsoft Security Compliance Manager.  
  
### <a name="microsoft-security-compliance-manager"></a>Microsoft Security Compliance Manager  
[Microsoft Security Compliance Manager](https://technet.microsoft.com/library/cc677002.aspx) — бесплатно получить средство, которое интегрируется конфигурации безопасности, которые рекомендуется корпорацией Майкрософт, в зависимости от конфигурации роли и версии операционной системы, и собирает данные о них в единое средство и пользовательский Интерфейс, который можно использовать для создания и настройки базовых параметров безопасности для контроллеров домена. Microsoft Security Compliance Manager шаблоны можно объединить с параметрами мастер настройки безопасности для создания шаблонов базовой конфигурации полный для серверами перехода, которые развертываются и применяемые объекты групповой политики, развернутый на подразделения, в которых переходов серверы находятся в Active Directory.  
  
> [!NOTE]  
> На момент написания этой статьи Microsoft Security Compliance Manager не включает параметры, относящиеся к серверами перехода или других безопасного администрирования узлов, но можно использовать диспетчер соответствия требованиям безопасности (SCM) для создания начальной базовые показатели для администрирования узлов. Чтобы обеспечить безопасность узлов, следует применять дополнительные параметры безопасности для надежной рабочих станциях и серверах.  
  
### <a name="applocker"></a>AppLocker  
Административные узлы и виртуальные machinesshould настраиваться с помощью сценария, средства и приложения whitelists через AppLocker или программного обеспечения ограниченного использования приложений сторонних разработчиков. Административных приложений и служебных программ, которые не соответствуют параметры безопасности следует обновить или заменить оборудование, отвечающее безопасной разработки и методов администрирования. Следует при необходимости новый или дополнительный инструментарий на узле с правами администратора тщательно тестировать приложения и служебные программы, и если инструментарий подходит для развертывания на узлах администрирования, его можно добавить whitelists системы.  
  
### <a name="rdp-restrictions"></a>Ограничения протокола удаленного рабочего СТОЛА  
Несмотря на то, что определенной конфигурации и будет зависеть от архитектуры администрирования систем, следует включить ограничения, на которых учетных записей и компьютеров могут использоваться для установления подключения удаленного рабочего стола (RDP) к управляемыми системами, например, с помощью шлюза удаленных рабочих столов (шлюз RD) переходов серверов для управления доступом к контроллерам домена и другие управляемые системы из авторизованных пользователей и систем.  
  
Вы должны разрешить интерактивные входы в систему авторизованными пользователями и необходимо удалить либо даже блокировать другие типы входа в систему, которые не требуются для доступа к серверу.  
  
### <a name="patch-and-configuration-management"></a>Исправление и управления конфигурациями  
Организации меньшего размера могут зависеть от предложения, такие как Центр обновления Windows или [Windows Server Update Services](https://technet.microsoft.com/windowsserver/bb332157) (WSUS) для управления развертыванием обновлений для системы Windows, хотя в более крупных организациях может реализовать исправления и конфигурации программное обеспечение управления как System Center Configuration Manager. Независимо от механизмы, используемые для развертывания обновлений для общего сервера и остальными рабочими станциями следует учитывать отдельных развертываний для высокую безопасность систем, таких как контроллеры домена, центров сертификации и узлах администрирования. Путем разделения этих систем от инфраструктуры общего управления, компрометация учетные записи программного обеспечения или службы управления компрометации невозможно расширить легко наиболее безопасный систем в вашей инфраструктуре.  
  
Несмотря на то, что не следует реализовывать процессы ручного обновления для системы безопасности, необходимо настроить отдельные инфраструктуры для обновления системы безопасности. Даже в очень крупных организациях этой инфраструктуры можно обычно реализовать через выделенные серверы WSUS и объекты групповой политики для защищенной систем.  
  
### <a name="blocking-internet-access"></a>Блокирование доступа к Интернету  
Узлы администрирования не разрешается доступ к Интернету, а также должны быть возможность найти внутренней сети организации. Веб-браузеры и аналогичных приложений не разрешается на узлах администрирования. Вы можете заблокировать доступ к Интернету для безопасного узлов через сочетание конфигурацию прокси-сервера «черная дыра» защищенные узлы, WFAS конфигурации и параметры брандмауэра периметра. Также можно использовать список утвержденных приложений, для предотвращения использования на узлах администрирования веб-браузеры.  
  
### <a name="virtualization"></a>Виртуализация  
Если это возможно, рассмотрите возможность использования виртуальных машин как административные узлы. С помощью виртуализации, можно создать пользователя администрирования систем, централизованного хранения и управления и который можно легко завершить при не заняты, гарантируя, что учетные данные не остаются active на администрирования систем. Можно также указать, что виртуальный администрирования узлов сбрасываются в исходный моментальный снимок после каждого использования убедившись, что виртуальные машины останутся без изменений. Дополнительные сведения о параметрах виртуализации узлам администрирования предоставляется в следующем разделе.  
  
## <a name="sample-approaches-to-implementing-secure-administrative-hosts"></a>Пример подхода к реализации безопасного администрирования узлов  
Независимо от того, разработка и развертывание инфраструктуры администрирования узлов следует Имейте в виду рекомендации, приведенные в «Принципы для создания безопасного администрирования узлов» ранее в этом разделе. Каждый из методов, описанных здесь содержит общие сведения о том, как можно разделить «администрирования» и «производительность» системы, используемые ИТ-специалистов. Производительность системы — это компьютеры, которые ИТ-администраторам использовать для проверки электронной почты, в Интернете и использовать общие производительности программного обеспечения, таких как Microsoft Office. Административные системы — это компьютеры, которые защищенный режим и выделенных для использования для повседневного администрирования ИТ-среде.  
  
Реализация безопасного администрирования узлов проще всего предоставить ИТ-специалисты с помощью защищенных рабочих станций, из которых они могут выполнять задачи администрирования. При реализации только для рабочих станций каждая рабочая станция администратора используется для запуска средства управления и RDP-подключения для управления серверами и другой инфраструктурой. Только для рабочих станций реализации может эффективно небольших организаций, несмотря на то, что больше, более сложной инфраструктуры полезным может оказаться распределенных проектирования для администрирования узлов, в которых используются выделенных административных серверов и рабочих станций, как описано в «Реализации безопасной административных рабочих станций и переходов серверы» далее в этом разделе.  
  
### <a name="implementing-separate-physical-workstations"></a>Реализация отдельные физические рабочие станции  
Один из способов, вы можете реализовать администрирования узлов — предоставление каждого пользователя ИТ два рабочих станций. Одна рабочая станция используется с учетной записью «обычный» для выполнения операции, такие как проверка электронной почты и используя рабочие приложения, когда второй рабочей станции строго предназначенный для выполнения административных функций.  
  
Для повышения производительности рабочей станции ИТ-специалисты могут предоставляться учетные записи обычных пользователей, а не с использованием привилегированных учетных записей для входа на незащищенным компьютерам. Рабочую станцию администрирования должны быть настроены строго управляемой конфигурации и ИТ-специалистам следует использовать другую учетную запись для входа на рабочую станцию администрирования.  
  
Если вы реализовали смарт-карты, рабочие станции администраторов должен быть настроен требовать входов смарт-карт и ИТ-специалистам следует предоставлять отдельные учетные записи для администрирования использования, также настроить для ввода смарт-карты для интерактивного входа в систему. Административные узла должен быть защищен, как описано выше, и только назначенный ИТ-пользователей, должны выполнять локальный вход на рабочую станцию администрирования.  
  
#### <a name="pros"></a>Специалисты  
Реализуя отдельных физических системах, можно убедиться, что каждый компьютер настроен правильно для его роль и что ИТ-пользователей, не могут непреднамеренно предоставить администрирования систем риску.  
  
#### <a name="cons"></a>Недостатки  
  
-   Реализация отдельных физических компьютерах увеличивает затраты на оборудование.  
  
-   Вход в систему на физический компьютер с учетными данными, которые используются для администрирования удаленных систем кэширует учетные данные в памяти.  
  
-   Если не безопасно хранятся рабочие станции администраторов, они, возможно, уязвимым для атак через механизмы, такие как клавиатурных физического оборудования или другие физические атаки.  
  
### <a name="implementing-a-secure-physical-workstation-with-a-virtualized-productivity-workstation"></a>Реализация безопасного физических рабочей станции с виртуализированных производительность рабочей станции  
При использовании этого подхода ИТ-пользователей, получают доступ к защищенной административных рабочих станций, из которого они могут выполнять повседневные административные функции, с помощью средств удаленного администрирования сервера (RSAT) или RDP-подключения к серверам в пределах их области ответственности. Когда ИТ-пользователей должны выполнять задачи производительность, они могут подключаться через протокол удаленного рабочего СТОЛА удаленного производительность рабочей станции, под управлением в качестве виртуальной машины. Следует использовать отдельные учетные данные для каждой рабочей станции и элементы управления, например смарт-карты должен быть реализован.  
  
#### <a name="pros"></a>Специалисты  
  
-   Рабочие станции администраторов и рабочих станциях разделены.  
  
-   ИТ-специалисты с помощью безопасных рабочих станциях для подключения к рабочих станциях можно использовать отдельные учетные данные и смарт-карт и привилегированных учетных данных не выводятся на компьютере менее безопасным.  
  
#### <a name="cons"></a>Недостатки  
  
-   Реализация решения требуется надежное виртуализации вариантов и проектирования и реализации рабочих.  
  
-   Если не безопасно хранятся физических рабочих станций, они, возможно, уязвимы для физических атак, которые риску оборудования и операционной системы и сделать их уязвимо для перехвата связи.  
  
### <a name="implementing-a-single-secure-workstation-with-connections-to-separate-productivity-and-administrative-virtual-machines"></a>Реализация одной рабочей станции безопасных подключений для разделения «Производительность» и «Администратора» виртуальных машин  
При использовании этого подхода может выдавать ИТ-пользователей одной физической рабочей станции, заблокирована, как описано выше, и на котором ИТ-пользователей могут не привилегированного доступа. Вы можете предоставить подключения служб удаленных рабочих столов для виртуальных машин, размещенных на выделенных серверах, предоставляя ИТ-специалисты с одной виртуальной машины под управлением электронной почты и других приложений, а второй виртуальной машины, которая настроена как выделенный узел администрирования пользователя.  
  
Рекомендуется требовать смарт-карты или других многофакторной входа для виртуальных машин с помощью отдельных учетных записей, отличных от учетную запись, используемая для входа физическом компьютере. После ИТ входе в систему на физический компьютер, они могут использовать смарт-карту производительности подключаться к их эффективность работы удаленного компьютера и отдельную учетную запись и смарт-карты для подключения к их удаленного администрирования компьютера.  
  
#### <a name="pros"></a>Специалисты  
  
-   ИТ-пользователей можно использовать один физический рабочей станции.  
  
-   Требующих отдельные учетные записи для виртуальных узлов и с помощью служб удаленных рабочих столов подключения к виртуальным машинам ИТ-пользователей учетные данные не кэшируются в памяти на локальном компьютере.  
  
-   Физического узла могут быть защищены на тот же угол администрирования узлов, уменьшая вероятность того, компрометации локального компьютера.  
  
-   В случаях, когда пользователь с ИТ производительность виртуальной машины или их административная виртуальная машина был взломан виртуальной машины можно сбросить легко с состоянием «хорошо известных».  
  
-   Если физический компьютер был скомпрометирован, привилегированные учетные данные не будут кэшироваться в памяти и использование смарт-карт может помешать учетных данных с средства регистрации нажатия клавиш.  
  
#### <a name="cons"></a>Недостатки  
  
-   Реализация решения требуется надежное виртуализации вариантов и проектирования и реализации рабочих.  
  
-   Если не безопасно хранятся физических рабочих станций, они, возможно, уязвимы для физических атак, которые риску оборудования и операционной системы и сделать их уязвимо для перехвата связи.  
  
### <a name="implementing-secure-administrative-workstations-and-jump-servers"></a>Реализация безопасного администрирования рабочих станциях и серверах переходов  
В качестве альтернативы для защиты административных рабочих станций или в сочетании с их вы можете реализовать серверами перехода безопасного и пользователи могут подключаться к серверам перехода, с помощью удаленного рабочего СТОЛА и смарт-карты для выполнения административных задач.  
  
Серверы переходов должны быть настроены для выполнения роли шлюза удаленных рабочих столов, чтобы иметь возможность реализации ограничений в подключения к серверу переходов и конечные серверы, которые будут управляться из него. Если это возможно, следует установить роль Hyper-V и создать [личные виртуальные рабочие столы](https://technet.microsoft.com/library/dd759174.aspx) или других виртуальных машин на уровне пользователей для административных пользователей для выполнения своих задач на серверами перехода.  
  
Предоставляя пользователи с правами администратора на пользователя виртуальных машин на сервере перехода, обеспечения физической безопасности административных рабочих станций, а пользователи с правами администратора может сбросить или завершение работы виртуальных машин в случае использования. Если вы не хотите установить роль Hyper-V и роли шлюза удаленных рабочих столов на одном узле администрирования, их можно установить на отдельных компьютерах.  
  
Когда это возможно, средств удаленного администрирования следует использовать для управления серверами. Средства удаленного администрирования сервера (RSAT) компонент должен устанавливаться на пользователей виртуальных машин (или сервер перехода, если не реализуется на уровне пользователей виртуальных машин для администрирования), и администраторов должны подключаться с помощью протокола удаленного рабочего СТОЛА к виртуальным машинам для выполнения административных задач.  
  
В случаях, когда пользователь с правами администратора должен подключиться к конечному серверу и управлять им напрямую, через протокола удаленного рабочего СТОЛА шлюза удаленных рабочих Столов должен быть разрешено подключение выполнялось только в том случае, если соответствующие пользователя и компьютера используются для установления подключения к серверу назначения. Выполнение средств удаленного администрирования сервера (или сходные) средства следует запретить в системах, не выбранные системы управления, такие как общего использования рабочих станций и рядовых серверах, которые не серверы перехода.  
  
#### <a name="pros"></a>Специалисты  
  
-   Создание серверами перехода позволяет сопоставить конкретных серверов на «зоны» (коллекции систем со сходными параметрами безопасности, настройки и подключения) в вашей сети и требование, что достигается за счет административного персонала, подключение из безопасного администрирования узлов на выделенном «зоны» сервер администрирования каждой зоны.  
  
-   Сопоставив серверами перехода зон, можно реализовать детальное элементы управления для свойства подключения и требования к конфигурации и можно легко определить пытается установить подключение с неавторизованными системами.  
  
-   Реализация каждого администратора виртуальных машин на серверах переходов, применять завершение работы и сброс виртуальных машин известных исходное состояние после выполнения административных задач. Применяя завершение работы (или перезапуск) виртуальных машин по завершении задачи администрирования, виртуальные машины не может быть целью злоумышленников, а также являются атак кражи учетных данных может быть выполнено, так как кэшированных в памяти учетные данные не сохраняются после перезагрузки.  
  
#### <a name="cons"></a>Недостатки  
  
-   Выделенные серверы необходимы для серверами перехода ли физический или виртуальный.  
  
-   Реализация назначенные серверы перехода и рабочих станциях администраторов требует тщательного планирования и конфигурацию, которая сопоставляет все зоны безопасности, настроенных в среде.
