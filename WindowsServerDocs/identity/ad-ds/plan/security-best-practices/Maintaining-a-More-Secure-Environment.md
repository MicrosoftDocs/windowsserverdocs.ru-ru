# <a name="maintaining-a-more-secure-environment"></a>Повышение уровня безопасности среды

>Область применения. Windows Server 2016, Windows Server 2012 R2, Windows Server 2012

*Номер закона 10: технология не является панацеей.* - [10 неизменяемых законов по администрированию безопасности](/previous-versions//cc722488(v=technet.10))  
  
После создания управляемой, безопасной среды для критически важных бизнес-ресурсов фокус должен быть перемещен, чтобы гарантировать его безопасное обслуживание. Хотя вы указали определенные технические элементы управления для повышения безопасности установок AD DS, сама по себе технологии не будет защищать среду, в которой она не работает в связи с бизнесом, чтобы поддерживать безопасную и удобную инфраструктуру. Рекомендации высокого уровня в этом разделе предназначены для использования в качестве руководств, которые можно использовать для разработки не только эффективного обеспечения безопасности, но и эффективного управления жизненным циклом.  
  
В некоторых случаях у ИТ-организации уже может быть тесное взаимодействие с подразделениями, что упростит реализацию этих рекомендаций. В организациях, в которых ИТ-подразделения не тесно связаны друг с другу, вам может потребоваться сначала получить Руководящий спонсор, чтобы подделывать более тесную связь между ИТ-подразделениями. Эта [Сводка](../../../ad-ds/manage/component-updates/Executive-Summary.md) предназначена для использования в качестве автономного документа для руководящего обзора и может распространяться на руководителей, принимающих решения в Организации.  
  
## <a name="creating-business-centric-security-practices-for-active-directory"></a>Создание ориентированных на бизнес методов обеспечения безопасности для Active Directory  
В прошлом информационные технологии во многих организациях были просмотрены как структура поддержки и центр затрат. ИТ-отделы часто были в значительной степени разделены от деловых пользователей, а взаимодействия ограничиваются моделью «запрос-ответ», в которой бизнес-ресурсы и ответы на них ответили.  
  
По мере развития и распространения технологий концепция «компьютера на каждом рабочем столе» фактически передается в большую часть мира и даже была создана с помощью широкого спектра легко доступных общедоступных технологий. Информационная технология больше не является функцией поддержки, она является основной бизнес-функцией. Если ваша организация не может продолжать работу, если все ИТ-службы были недоступны, предприятие Организации — по крайней мере часть информационных технологий.  
  
Чтобы создать эффективные планы восстановления, ИТ-службы должны тесно работать с подразделениями в вашей организации, чтобы вычислить не только наиболее важные компоненты ИТ-структуры, но и критически важные функции, необходимые для бизнеса. Определив, что важно для вашей организации в целом, вы можете сосредоточиться на обеспечении безопасности компонентов, имеющих наибольшее значение. Это не является рекомендацией шерком безопасность систем и данных с низкими значениями. Вместо того, как вы определили уровни обслуживания для бесперебойной работы системы, следует рассмотреть возможность определения уровней контроля и мониторинга безопасности на основе важности ресурса.  
  
Когда вы вкладываетесь в создание текущей, защищенной, управляемой среды, вы можете сосредоточиться на эффективном управлении ИТ и обеспечить эффективное управление жизненным циклом, не определяемое ИТ-отделами, но в бизнесе. Для этого вам нужно не только сотрудничать с бизнесом, но и вкладывать бизнес в "владение" данными и системами в Active Directory.  
  
Когда данные и системы вводятся в Active Directory без назначенных владельцев, владельцев бизнеса и владельцев ИТ, нет четкой цепи ответственности за подготовку, управление, мониторинг, обновление и постепенное списание системы. Это приводит к инфраструктуре, в которых системы могут снизить риск, но нельзя списать, так как владение неясно. Для эффективного управления жизненным циклом пользователей, данных, приложений и систем, управляемых Active Directory установкой, необходимо следовать принципам, описанным в этом разделе.  
  
### <a name="assign-a-business-owner-to-active-directory-data"></a>Назначение бизнес-владельца для Active Directory данных  
Данные в Active Directory должны иметь идентифицированного владельца, т. е. указанного отдела или пользователя, который является точкой контакта для принятия решений о жизненном цикле ресурса. В некоторых случаях ответственный за компонент Active Directory будет являться ИТ-отделом или пользователем. Такие компоненты инфраструктуры, как контроллеры домена, DHCP-и DNS-серверы, а также Active Directory, скорее всего, будут "принадлежат". Для данных, добавляемых в AD DS для поддержки бизнеса (например, новые сотрудники, новые приложения и новые репозитории данных), необходимо связать с данными указанное подразделение или пользователя.  
  
Независимо от того, используется ли Active Directory для записи владельца данных в каталоге или реализации отдельной базы данных для отслеживания ИТ-ресурсов, не нужно создавать учетную запись пользователя, устанавливать сервер или рабочую станцию не нужно, и приложение не должно развертываться без назначенного владельца записи. Попытка установить владение системами после развертывания в рабочей среде может быть сложной и невозможной в некоторых случаях. Таким образом, владение должно быть установлено во время, когда данные вводятся в Active Directory.  
  
### <a name="implement-business-driven-lifecycle-management"></a>Реализация управления жизненным циклом на основе бизнес-процессов  
Управление жизненным циклом должно быть реализовано для всех данных в Active Directory. Например, если новое приложение введено в домен Active Directory, то, по крайней мере, необходимо, чтобы владелец приложения должен был подтвердить продолжение использования приложения. При выпуске новой версии приложения необходимо информировать владельца приложения и решить, когда будет реализована новая версия.  
  
Если бизнес-владелец решил не утверждать развертывание новой версии приложения, этот владелец должен также получать уведомления о дате, когда текущая версия больше не будет поддерживаться, и должна отвечать за определение того, будет ли приложение списано или заменено. Хранение устаревших приложений, которые работают и не поддерживаются, не должно быть параметром.  
  
Когда учетные записи пользователей создаются в Active Directory, их диспетчеры записей должны получать уведомления при создании объекта и должны подтверждать срок действия учетной записи через равные промежутки времени. Благодаря реализации жизненного цикла на основе бизнес-процессов и обычной аттестации достоверности данных люди, которые лучше всего наявляют аномалии в данных, являются людьми, которые просматривают данные.  
  
Например, злоумышленники могут создать учетные записи пользователей, которые являются действительными учетными записями, следуя соглашениям об именовании и размещению объектов в вашей организации. Для обнаружения этих операций создания учетных записей можно реализовать ежедневную задачу, которая возвращает все пользовательские объекты без назначенного владельца Организации, чтобы можно было исследовать учетные записи. Если злоумышленники создают учетные записи и назначают владельца компании, реализуя задачу, которая сообщает о создании нового объекта указанному владельцу бизнеса, владелец компании может быстро определить, является ли учетная запись законной.  
  
Необходимо реализовать аналогичные подходы для групп безопасности и распространения. Хотя некоторые группы могут быть созданы функциональными группами, создавая каждую группу с назначенным владельцем, вы можете получить все группы, принадлежащие назначенному пользователю, и потребовать от пользователя подтвердить допустимость их членства. Как и в случае с созданием учетной записи пользователя, можно активировать изменения группы отчетов для указанного владельца Организации. Чем больше подставится бизнес-владельцу для аттестации допустимости или недействительности данных в Active Directory, тем более известно, что вы назначите аномалии, которые могут указывать на сбои процессов или фактический компромисс.  
  
### <a name="classify-all-active-directory-data"></a>Классификация всех Active Directory данных  
Помимо записи владельца компании для всех Active Directory данных во время добавления в каталог, необходимо также требовать, чтобы владельцы бизнес-процессов предписывали классификацию данных. Например, если приложение хранит критически важные для бизнеса данные, владелец компании должен пометить приложение как таковое в соответствии с инфраструктурой классификации вашей организации.  
  
Некоторые организации реализуют политики классификации данных, которые помечают данные в соответствии с повреждением данных, если бы они были украдены или предоставлены. Другие организации реализуют классификацию данных, которая помечает данные по критически важности, по требованиям доступа и хранению. Независимо от модели классификации данных, используемой в Организации, следует убедиться, что вы можете применить классификацию к Active Directoryным данным, а не только к файлам данных. Если учетная запись пользователя является виртуальной IP-адресом, она должна быть определена в базе данных классификации активов (независимо от того, реализуется ли это с помощью атрибутов объектов в AD DS или при развертывании отдельных баз данных классификации ресурсов).

### <a name="systems"></a>системы  
Не только классифицировать данные, но и их заполнение серверов. Для каждого сервера должно быть известно, какая операционная система установлена, какие общие роли предоставляются сервером, какие приложения выполняются на сервере, владелец ИТ-владельца и владелец записи, где это применимо. Для всех данных или приложений, выполняемых на сервере, необходимо указать классификацию, и сервер должен быть защищен в соответствии с требованиями для поддерживаемых рабочих нагрузок и классификаций, применяемых к системе и данным. Можно также сгруппировать серверы по классификации рабочих нагрузок, что позволяет быстро определять серверы, которые должны быть наиболее тщательно отслеживаемыми и наиболее строго настроенными.  
  
### <a name="applications"></a>Приложения  
Вы должны классифицировать приложения по функциональным возможностям (то, что они делают), базе пользователей (кто использует приложения) и операционной системе, в которой они выполняются. Необходимо сохранить записи, содержащие сведения о версии, состояние исправления и другие необходимые сведения. Вы также должны классифицировать приложения по типам обрабатываемых данных, как описано выше.  
  
### <a name="users"></a>Пользователи  
Независимо от того, вызываются ли им пользователи виртуальных IP-адресов, критические учетные записи или используют другую метку, учетные записи в Active Directoryных установках, которые, скорее всего, будут использоваться злоумышленниками, должны быть помечены и отслеживаться. В большинстве организаций просто нецелесообразно отслеживать все действия всех пользователей. Однако, если вы можете найти критические учетные записи в Active Directory установки, вы можете отслеживать изменения, как описано выше в этом документе.  
  
Вы также можете приступить к созданию базы данных "ожидаемых поведений" для этих учетных записей при аудите учетных записей. Например, если вы обнаружите, что данный руководитель использует защищенную рабочую станцию для доступа к важным для бизнеса данным из своего офиса и из дома, но редко из других расположений, если вы видите попытки получить доступ к данным с помощью своей учетной записи с неавторизованного компьютера или расположении около планеты, где известно, что руководитель не находится в настоящее время, вы можете быстро определить и исследовать это аномаль  
  
Интегрируя бизнес-информацию в инфраструктуру, вы можете использовать эту бизнес-информацию для выявления ложных срабатываний. Например, если выезд руководителя записывается в календарь, доступный ИТ-специалистам, ответственным за мониторинг среды, можно сопоставить попытки подключения с известными расположениями руководителей.  
  
Предположим, что «руководитель» обычно находится в Чикаго и использует защищенную рабочую станцию для доступа к важным для бизнеса данным со своего рабочего места, а событие активируется неудачной попыткой получить доступ к данным с незащищенной рабочей станции, расположенной в Атланта. Если вы можете проверить, что в настоящее время находится в Атланта, вы можете решить это событие, обратившись к руководителю или помощнику руководителя, чтобы определить, является ли сбой доступа результатом руководителя, который забыли использовать защищенную рабочую станцию для доступа к данным. Создавая программу, которая использует подходы, описанные в статье [планирование компромиссов](../../../ad-ds/plan/security-best-practices/Planning-for-Compromise.md), можно приступить к созданию базы данных ожидаемого поведения для наиболее важных учетных записей в Active Directoryной установке, которые могут помочь вам быстрее обнаружить атаки и реагировать на них.
