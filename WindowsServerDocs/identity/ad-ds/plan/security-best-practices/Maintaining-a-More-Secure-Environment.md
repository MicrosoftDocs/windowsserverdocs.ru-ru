## <a name="creating-business-centric-security-practices-for-active-directory"></a>Создание ориентированных бизнес безопаности для Active Directory  
В прошлом информационных технологий, многие организации смотрели как структуры поддержки и центр затрат. ИТ-отделы часто главным образом были выделены из бизнес-пользователей и взаимодействий, ограничивается модели запрос ответ, в котором бизнес запрошенных ресурсов и ответил ИТ.  
  
Как развитие технологий и, вследствие роста, концепция «компьютера на каждом рабочем столе» имеет эффективно поставляются для передачи для стран мира и даже были число с широкого спектра легкодоступном технологий современных. Информационные технологии больше не является функцией поддержки, он является основной функцией бизнеса. Если ваша организация не продолжают работать, если не были доступны все ИТ-услуг, вашей организации, по крайней мере в части, информационные технологии.  
  
Чтобы создать эффективный компромисс планы восстановления, ИТ-услуг необходимо проконсультироваться с филиалов в вашей организации определить не только наиболее важные компоненты альбомная ИТ, но критически важных функций, необходимых в бизнес. Определяя, что важно для организации в целом, могут сосредоточиться на защите компоненты, имеющие большинство значение. Это не рекомендация по shirk безопасности низкое значение систем и данных. Вместо этого, как определить уровни обслуживания для время работы системы, рассмотрите возможность определения уровня безопасности управления и мониторинга на основе важности активов.  
  
Когда вы вложили в создании текущего безопасные, управляемые среды, может перемещения фокуса эффективно управлять им и убедитесь, что действующие жизненного цикла управления процессы, которые не определено только ИТ, но с помощью бизнес. Для этого необходимо не только сотрудничать с бизнеса, но инвестиций бизнеса в «владение» данных и систем в Active Directory.  
  
При данным и системам добавляются в среду Active Directory без назначенного владельцев, владельцы предприятия и владельцы ИТ, существует не очистить цепочки ответственности для подготовки, управления, мониторинга, обновления и в конечном итоге списание системы. В результате инфраструктур, в которых предоставлять организации риску системы, но не можно вывести из эксплуатации из-за плохой владельца. Для эффективного управления жизненным циклом пользователей, данные, приложений и систем, под управлением установки Active Directory, должно соответствовать принципам, описанным в данном разделе.  
  
### <a name="assign-a-business-owner-to-active-directory-data"></a>Назначьте владельца для бизнеса данных Active Directory  
Данные в Active Directory должны иметь владельца определенных бизнес, то есть указанное подразделение или пользователя, являющегося точку контакта для принятия решений о жизненном цикле средства. В некоторых случаях владелец бизнеса компонентом Active Directory будет для ИТ-отдел или пользователя. Компоненты инфраструктуры, такие как контроллеры домена, серверы DHCP и DNS и Active Directory будет скорее «владельцем» ИТ. Для данных, который добавляется в AD DS для поддержки бизнес (например, новые сотрудники, новые приложения и новые сведения о репозитории), назначенный бизнес подразделение или пользователь должен быть связан с данными.  
  
Ли использовать Active Directory для записи владения данными, в каталоге, или реализовать отдельные базы данных для отслеживания ИТ-ресурсов, должны создаваться без учетной записи пользователя, не сервера или рабочей станции должен быть установлен и приложение не следует развертывать без уполномоченный владелец записи. Устанавливается владения систем после они были развернуты в производственной среде может быть сложной задачей в лучшем и невозможно в некоторых случаях. Таким образом владение следует установить во время данных впервые появился в Active Directory.  
  
### <a name="implement-business-driven-lifecycle-management"></a>Реализация управления жизненным циклом управляемой бизнеса  
Управление жизненным циклом должно быть реализовано для всех данных в Active Directory. Например когда новое приложение впервые появился в домен Active Directory, владельца бизнеса приложения через регулярные интервалы, ожидаемое для подтверждения того, для дальнейшего использования приложения. При выпуске новой версии приложения владельца бизнеса приложения должны быть в курсе и следует решить, когда новая версия будет реализовано.  
  
Если владелец бизнеса принимает решение не утвердить развертывания новой версии приложения, владельца этого бизнеса также уведомляться об дату, когда текущая версия больше не будет поддерживаться и должен быть отвечает за определение, будет ли приложение вывести из эксплуатации или заменить. Сохранение приложений прежних версий, работает и не поддерживается не должен быть параметр.  
  
При создании учетных записей пользователей в Active Directory своих менеджеров записи следует уведомление при создании объекта и требуется аттестацию на срок действия учетной записи через регулярные интервалы. По реализации бизнес управляемых жизненный цикл и регулярных аттестации достоверности данных, люди, которые являются наиболее широкие возможности для определения ошибок в данных, тех, кто просмотреть данные.  
  
К примеру злоумышленники могут создать учетные записи пользователей, которые кажутся действительные учетные записи соглашения об именовании вашей организации и размещения объекта. Чтобы обнаружить эти операции создания учетных записей, может реализовать ежедневного задания, который возвращает все объекты пользователя без владельца назначенного предприятия, позволяя просматривать учетные записи. Если злоумышленники Создание учетных записей и назначить владельцем бизнеса путем реализации задачу, которая выдает сообщение, создания новых объектов владельцу назначенного бизнес, владелец бизнеса можно быстро определить ли обосновано учетной записи.  
  
Вы должны реализовать подобные методы для групп безопасности и распространения. Несмотря на то, что некоторые группы может быть функциональные группы, созданные ИТ, создав с уполномоченный владелец каждой группы, вы сможете получить все группы, принадлежащие указанного пользователя и пользователю необходимо аттестацию на допустимость их членства. Аналогично подход, сделанных с помощью создания учетной записи пользователя, вы можете активировать отчетов изменения группы владельца назначенного бизнеса. Дополнительные процедуры становится для бизнес-владельцу аттестацию на срок действия или недопустимости данных в Active Directory, более широкие, что используется для идентификации аномалий, которые могут указывать сбоев процесса или фактические компрометацию.

### <a name="users"></a>Пользователи  
Ли их вызывать пользователей «Виртуальных IP-адресов», критически важных учетных записей, или использовать другое название, учетные записи в Active Directory установки, наиболее подверженные атакам злоумышленников следует с тегами и отслеживаться. В большинстве организаций не просто подойдет для наблюдения за все действия для всех пользователей. Тем не менее если в вашей установке Active Directory, можно определить критически важных учетных записей, можно отслеживать эти учетные записи для изменения, как описано ранее в этом документе.  
  
Кроме того, можно также начинать для создания базы данных «ожидаемого поведения» для этих учетных записей аудита учетные записи. Например если вы нашли конкретного руководителя для доступа к критически важные данные из своего офиса и из него дома, но редко из других расположений, если вы видите пытается получить доступ к данным с помощью своей учетной записи, от неавторизованного компьютера или в расположение среднюю вокруг планете где известно, что и руководитель находится не в настоящее время использует его защищенной рабочей станции, можно более быстро обнаружить и изучить этот Аномальное поведение.  
  
Внедряя бизнес-информации в инфраструктуру, можно использовать бизнес-информации для выявления ложных положительных результатов. Например если executive путешествия записывается в календаре, доступном для ИТ-специалистов, выполняет мониторинг в среде, может помочь вам попыток подключения с использованием известных расположений руководителей.  
  
Предположим, что исполнитель A обычно находится в Чикаго и использует защищенной рабочей станции для доступа к критически важных данных из своей службы поддержки и событие инициируется неудачной попытке доступа к данным с незащищенным рабочей станции, расположенных в Атланте. Если вы уверены, что и руководитель в настоящий момент в Атланте, события можно разрешить, обратившись в службу и руководитель или и руководитель помощника, чтобы определить, было ли ошибки доступа результат и руководитель забыть использование защищенных рабочих станций для доступа к данным. С помощью программы, использующей подходов, описанных в [планирование при компрометации](../../../ad-ds/plan/security-best-practices/Planning-for-Compromise.md), вы сможете создать базу данных ожидаемого поведения для «важнейших» учетные записи в установке Active Directory, который потенциально может помочь вам быстрее обнаруживать и реагировать на атаки.
