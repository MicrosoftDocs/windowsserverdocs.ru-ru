## <a name="privileged-accounts-and-groups-in-active-directory"></a>Привилегированные учетные записи и группы в Active Directory  
Этот раздел содержит справочные сведения о привилегированных учетных записей и групп в Active Directory предназначен для объяснения универсальность и различия между привилегированных учетных записей и групп в Active Directory. Понимание эти различия, следует ли вы выполните рекомендации в [реализация административных моделей наименьших прав](../../../ad-ds/plan/security-best-practices/../../../ad-ds/plan/security-best-practices/Implementing-Least-Privilege-Administrative-Models.md) точные или настроить их для вашей организации, у вас есть средства, необходимые для каждой группы защиты и учетной записи соответствующим образом.  
  
### <a name="built-in-privileged-accounts-and-groups"></a>Встроенные привилегированных учетных записей и групп  
Active Directory упрощает делегирования прав администратора и поддерживает принципа наименьших прав доступа в назначения прав и разрешений. «Обычный» пользователей, имеющих учетные записи в домене, по умолчанию, смогут прочитать большая часть что хранится в каталоге, но которые могут изменять только ограниченный набор данных в каталоге. Пользователи, которым требуется дополнительная привилегия может быть предоставлено участия в различных группах «привилегированные», которые встроены в каталоге, где они могут выполнять определенные задачи, связанные с их ролями, но не может выполнять задачи, которые не относятся к своих обязанностей. Кроме того, организации могут создавать группы, которые предназначены для определенных должностных обязанностей и предоставлены детальное права и разрешения, позволяющие ИТ-специалистам выполнять повседневные административные функции без предоставления прав и разрешений, чем требуется для этих функций.  
  
В Active Directory три встроенные группы, которые наивысший группы привилегий в каталоге: "Администраторы предприятия", "Администраторы домена" и администраторам. Конфигурация по умолчанию и возможности каждого из этих групп описаны в следующих разделах:  
  
#### <a name="highest-privilege-groups-in-active-directory"></a>Высокий привилегий групп в Active Directory  
  
##### <a name="enterprise-admins"></a>Администраторы предприятия  
Администраторов предприятия (EA) — это группа, который существует только в корневом домене леса, и по умолчанию, он является членом группы "Администраторы" во всех доменах леса. Встроенная учетная запись администратора для корневого домена леса входит только по умолчанию EA группы. EAs предоставлены права и разрешения, которые позволяют им для реализации изменений на уровне леса (то есть изменения, которые влияют на все домены в лесу), таких как добавление или удаление доменов, установлении доверия лесов или Повышение режима работы леса. В модель делегирования правильно разработанный и внедрении EA членства является обязательным только в том случае, когда сначала построении леса, или при внесении изменений определенных леса, такие как установка исходящем доверии леса. Большая часть права и разрешения, предоставленные группе EA можно делегировать меньше привилегий пользователей и групп.  
  
##### <a name="domain-admins"></a>Администраторы домена  

Каждый домен в лесу имеет собственную группу администраторов домена (DA), которая является членом группы администраторов домена и является членом локальной группы администраторов на каждом компьютере, который присоединен к домену. По умолчанию группе DA для домена входит только встроенную учетную запись администратора для этого домена. DAs, «всемогущими» в своих доменах, пока EAs привилегии уровня леса. В модели делегирования правильно разработанный и внедрении членства Администраторы домена должны только в сценариях «break стекло» (например, ситуации, в которых требуется учетная запись с высоким уровнем привилегий на каждом компьютере в домене). Несмотря на то, что собственные механизмы делегирования Active Directory разрешить делегирование, при условии, что его можно использовать только в сценариях аварийного DA учетные записи, создав модель эффективных делегирования может занять много времени и многих организаций используют инструментов сторонних разработчиков, чтобы ускорить процесс.  
  
##### <a name="administrators"></a>Администраторы  
Третья является встроенной домена локальных администраторов (BA) группой куда вложенные DAs и EAs. Эта группа предоставляется многие прямой права и разрешения в каталоге, а также на контроллерах домена. Тем не менее в группу администраторов домена имеет не права на рядовых серверах или на рабочих станциях. Это посредством членства компьютеров локальной группы администраторов, которому предоставлено право локального.  
  
> [!NOTE]  
> Несмотря на то, что их конфигурацию по умолчанию эти привилегированных групп, членом любой из трех групп можно управлять каталога для получения членства в любых других групп. В некоторых случаях — непростая для получения членства в группах, в других более сложной, но с точки зрения потенциальных привилегий всех трех групп следует учитывать эффективно эквивалент.  
  
##### <a name="schema-admins"></a>Администраторы схемы  

Четвертый привилегированные группы администраторов схемы (SA), существует только в корневом домене леса и имеет только этого домена встроенной учетной записи администратора, что элемент по умолчанию группе администраторов предприятия. Группа "Администраторы схемы" предназначен для заполняться только временно и иногда (если требуется внесение изменений в схему AD DS).  
  
Несмотря на то, что группа SA являются только группы, который можно изменять схему Active Directory (этот о-ва, каталог базовых структур данных, таких как объекты и атрибуты), область действия группы SA прав и разрешений ограничено по сравнению групп описано выше. — Также частая, чтобы найти что организации разработали соответствующие рекомендации по управлению членства в группе сопоставления безопасности, так как обычно редко требуется членство в группе и только на короткое время. Технически это справедливо EA, DA и BA групп в Active Directory, а также, но это намного реже, чтобы найти что организаций были реализованы аналогичные рекомендации для этих групп, что и для сопоставления безопасности группы.  
  
#### <a name="protected-accounts-and-groups-in-active-directory"></a>Защищенные учетные записи и группы в Active Directory  
В Active Directory набор привилегированных учетных записей и групп по умолчанию называется «защищенные» учетные записи и группы являются безопасными по-другому, чем другие объекты в каталоге. Любую учетную запись, имеющую прямой, либо транзитивными членство в любую группу защищенных (вне зависимости от ли членство извлекается из групп безопасности или распространения) наследует ограниченные безопасности.  

  
Например если пользователь является членом группы распространения то есть, в свою очередь, члена группы защищенных в Active Directory данного объекта-пользователя помечается как защищенной учетной записи. Если учетную запись помечается как защищенной учетной записи, значение атрибута adminCount объекта имеет значение 1.  
  
> [!NOTE]
> Несмотря на то, что транзитивное членство в группе защищенных включает вложенных распространения и группы вложенные безопасности, учетных записей, которые являются членами вложенные группы рассылки, не будут получать SID группы защищенных в их маркерам доступа. Тем не менее группы рассылки можно преобразовать в группы безопасности в Active Directory, именно поэтому группы распространения включаются в группы защищенных член перечисления. Следует это защищенные вложенные группы рассылки постоянно преобразовать в группу безопасности, учетных записей, являющихся членами группы распространения бывший впоследствии будет получать родительского защищены SID группы в их маркерам доступа при следующем входе.

##### <a name="adminsdholder-and-sdprop"></a>AdminSDHolder и SDProp  
Объект, называемый AdminSDHolder в системном контейнере каждого домена Active Directory, автоматически создается. Объект AdminSDHolder предназначено для того, чтобы убедиться, что разрешения на защищенных учетных записей и групп согласованно применяются, независимо от расположения защищенных групп и учетных записей в домене.  

Каждые 60 минут (по умолчанию), этот процесс известен как работа распространителя дескрипторов безопасности (SDProp) выполняется на контроллере домена, в котором находится роль эмулятора основного контроллера домена. SDProp сравнивает разрешений на объект AdminSDHolder домена с разрешениями на защищенных учетных записей и групп в домене. Если разрешения на всех защищенных учетных записей и групп разрешений на объект AdminSDHolder не совпадают, должны совпадать с объект AdminSDHolder домена сбрасываются разрешений на защищенных учетных записей и групп.  
  
Наследование разрешений отключен на защищенных групп и учетных записей, это означает, что даже если учетные записи или группы перемещаются в другие расположения в каталоге, они не наследуют разрешения от родительских объектов новый. Наследование также отключается на объект AdminSDHolder, чтобы изменения разрешений для родительских объектов не изменяет разрешения AdminSDHolder.  
  
> [!NOTE]
> При удалении учетной записи из группы защищенных больше не считается защищенной учетной записи, но его остается атрибут adminCount, равным 1, если не изменен вручную. Эта конфигурация образом ACL объекта больше не обновляются с помощью SDProp, что объект по-прежнему не наследует разрешения от родительского объекта. Таким образом объект может находиться в подразделение (OU) для которой были делегированы разрешения, но ранее защищенных объектов не будет наследовать эти делегированные разрешения. Сценарий для обнаружения и восстановить ранее защищенных объектов в домене можно найти в [статья службы поддержки Майкрософт 817433](https://support.microsoft.com/?id=817433).  
  
###### <a name="adminsdholder-ownership"></a>Владение AdminSDHolder  
Большинство объектов в Active Directory, принадлежат BA группы домена. Тем не менее объект AdminSDHolder по умолчанию владельцем группы DA домена. (Это случай, в котором DAs не являются производными свои права и разрешения через членство в группе администраторов домена).  
  
В версиях Windows, предшествующих Windows Server 2008 владельцы объекта можно изменить разрешения объекта, включая предоставление сами разрешения, которые изначально они не было. Таким образом разрешения на объект AdminSDHolder домена по умолчанию пользователи, являющиеся членами групп BA или EA на изменение разрешений для доменного объекта AdminSDHolder. Однако члены группы администраторов домена может стать владельцем объекта и предоставлять дополнительные разрешения, который означает, что эта защита ограниченны и только позволяет защитить объект от случайного изменения пользователями, которые не являются членами группы DA в домене. Кроме того BA и EA (если применимо) группы имеют разрешение на изменение атрибутов объект AdminSDHolder в локальном домене (корневой домен EA).  
  
> [!NOTE]  
> Атрибут объекта AdminSDHolder dSHeuristics, позволяет ограниченный (удаления) настройки групп, которые считаются защищенных групп и затронутые AdminSDHolder и SDProp. Эта настройка следует тщательно исключать реализуется, несмотря на то, что существует действительный ситуации, в которых полезно изменение dSHeuristics на AdminSDHolder. Дополнительные сведения об изменении атрибута dSHeuristics на объект AdminSDHolder можно найти в статьях технической поддержки Майкрософт [817433](https://support.microsoft.com/?id=817433) и [973840](https://support.microsoft.com/kb/973840)и в [C: приложение защищенных учетных записей и групп в Active Directory](Appendix-C--Protected-Accounts-and-Groups-in-Active-Directory.md).  
  
Несмотря на то, что здесь описываются наиболее привилегированных групп в Active Directory, существует ряд других групп, которые были предоставлены повышенных уровня привилегий. Дополнительные сведения обо всех по умолчанию и встроенные группы в Active Directory и права пользователя, назначенные каждому см. в разделе [приложение б. привилегированные учетные записи и группы в Active Directory](../../../ad-ds/plan/security-best-practices/Appendix-B--Privileged-Accounts-and-Groups-in-Active-Directory.md).
