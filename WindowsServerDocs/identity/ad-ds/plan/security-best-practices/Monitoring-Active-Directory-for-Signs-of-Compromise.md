### <a name="windows-audit-categories"></a>Категорий аудита Windows  
До Windows Vista и Windows Server 2008 Windows был только девяти категориям политики аудита в журнал событий:  
  
-   События входа учетной записи  
  
-   Управление учетной записью  
  
-   Доступ к службе каталогов  
  
-   События входа в систему  
  
-   Доступ к объектам  
  
-   Изменение политики  
  
-   Использование привилегий  
  
-   Процесс отслеживания  
  
-   Системные события  
  
Эти девять категорий аудита традиционных составляют политику аудита. Каждой категории политики аудита можно включить успехов, отказов или Успех и отказ события. Их описания приведены в следующем разделе.  
  
#### <a name="audit-policy-category-descriptions"></a>Описание категории политики аудита  
Категории политики аудита позволяют указанные ниже типы сообщений журнала событий.  
  
##### <a name="audit-account-logon-events"></a>Аудит событий входа учетной записи  
Отчеты каждый экземпляр участника безопасности (например, пользователя, компьютера или учетная запись службы), войти в систему или выхода из одного компьютера, в котором используется другой компьютер для проверки учетной записи. Событий входа в систему создаются при проверке подлинности учетную запись участника безопасности домена на контроллере домена. Проверка подлинности локального пользователя на локальном компьютере создает событие входа в систему, которое регистрируется в локальном журнале безопасности. События выхода из системы не регистрируются.  
  
Эта категория создает большой «шума» Windows постоянно возникли учетные записи, вход и выход из системы на локальных и удаленных компьютеров в ходе обычной бизнеса. Тем не менее любого плана по обеспечению безопасности должны включать Успех и отказ этой категории аудита.  
  
##### <a name="audit-account-management"></a>Управление учетной записью аудита  
Эта политика аудита определяет, следует ли отслеживать действия управления для пользователей и групп. Например пользователей и групп должны отслеживаться при создании, изменении или удален; пользователя или учетная запись компьютера, группы безопасности или группы рассылки Если переименовать, отключать и включать; учетной записи пользователя или компьютера или при изменении пароля пользователя или компьютера. События могут быть созданы для пользователей или групп, которым добавлены или удалены из других групп.  
  
##### <a name="audit-directory-service-access"></a>Аудит доступа к службе каталогов  

Этот параметр политики определяет, аудит безопасности участника доступа к объекту Active Directory, имеет ли собственный указанного системный список управления доступом (SACL). Как правило эта категория следует включить только на контроллерах домена. Если параметр включен, этот параметр создает большой «шума».  
  
##### <a name="audit-logon-events"></a>Аудит входа в систему  
События входа в систему создаются при проверке подлинности субъекта безопасности на локальном компьютере. События входа в систему записывает входов в домен, на локальном компьютере. События выхода из системы не формируются. При включении событий входа в систему создает большой «шума», но они должны быть включены по умолчанию в любого плана аудита безопасности.  
  
##### <a name="audit-object-access"></a>Аудит доступа к объектам  
Доступ к объектам могла создавать события, когда впоследствии определенные объекты с включенным аудитом доступа (для примера, Opened, чтения, переименован, удаленные или закрыто). После включения основной категории аудита администратор по отдельности необходимо определить, какие объекты будут аудит включен. Многие объекты системы Windows поставляется с аудита включена, поэтому включение этой категории обычно начинает создавать события, прежде чем какие-либо определенные администратором.  
  
Эта категория слишком «шумными» и будет создавать события от пяти до десяти каждого доступа к объекту. Он может быть трудно администраторам новый аудит получить полезные сведения. Его следует включать только при необходимости.  
  
##### <a name="auditing-policy-change"></a>Аудит изменения политики  
Этот параметр политики определяет, подлежит ли аудиту каждый факт изменения политик назначения прав пользователей, политик брандмауэра Windows, политик доверия или изменения политики аудита. На всех компьютерах, следует включить в эту категорию. Он создает очень мало шум.  
  
##### <a name="audit-privilege-use"></a>Аудит использования привилегий  

Существует множество права и разрешения в Windows (например, вход в качестве пакетного задания и Act как часть операционной системы). Этот параметр политики определяет, подлежит ли аудиту каждый экземпляр субъект безопасности, проверяя право пользователя или привилегий. Включение этой категории приводит к повышению «шум», но могут быть полезны при отслеживании безопасности участника, учетные записи с повышенными привилегиями.  
  
##### <a name="audit-process-tracking"></a>Аудит отслеживания процессов  
Этот параметр политики определяет, подлежит ли аудиту подробное описание процесса, сведения об отслеживании событий, таких как активация программы, завершение процесса, дублирование дескрипторов и косвенный доступ к объекту. Это полезно для пользователей-злоумышленников и программы, которые они используют.  
  
Включение аудита процесс отслеживания создает большое количество событий, обычно он имеет значение **нет аудита**. Тем не менее этот параметр может предоставить крупное преимущество во время реагирования на события из подробный журнал процессам, запущенным и временем, которое они были запущены. Для контроллеров домена и другие серверы инфраструктуры одной роли в эту категорию можно безопасно включить все время. Серверы в единую роль не создают процесс значительно отслеживания трафика в ходе обычной своих обязанностей. Таким образом они станут доступны для записи несанкционированного событий, если они происходят.  
  
##### <a name="system-events-audit"></a>События аудита системы  

Системные события — почти универсального перехвата всех категория, регистрации различных событий, которые влияют на компьютер, его безопасность системы или журнал безопасности. Он включает события завершения работы компьютера и перезагрузки, сбоев питания, изменения системы во время, инициализации пакет проверки подлинности, clearings журнала аудита, проблем олицетворения и целый ряд других общих событий. Как правило Включение этой категории аудита создает много «шума», но создает достаточно очень полезны события, что сложно никогда не рекомендуем не включить его.  
  
#### <a name="advanced-audit-policies"></a>Расширенные политики аудита  
Начиная с Windows Vista и Windows Server 2008, Майкрософт улучшила способа выбора категории журнала событий можно сделать путем создания подкатегорий в каждой основной категории аудита. Подкатегории обеспечивают дополнительную детализацию, чем в противном случае с использованием основных категорий аудита. С помощью подкатегорий, можно включить только в части определенной основной категории и пропустить создания события, для которых имеется не используется. Каждую подкатегорию политики аудита можно включить успехов, отказов или Успех и отказ события.  
  
Чтобы вывести список всех доступных подкатегорий аудита, просмотрите контейнера расширенной политики аудита в объекте групповой политики, или введите следующую команду в командной строке на любом компьютере под управлением Windows Server 2012, Windows Server 2008 R2 или Windows Server 2008, Windows 8, Windows 7 или Windows Vista:  
  
**/subcategory/List Auditpol: \ ***  
  
Чтобы получить список настроенных подкатегории аудита на компьютере под управлением Windows Server 2012, Windows Server 2008 R2 или Windows 2008, введите следующую команду:  
  
**/ Category/Get Auditpol: \ ***

### <a name="auditing-subcategories-descriptions"></a>Аудит описания подкатегории  
Подкатегорий политики аудита включить указанные ниже типы сообщений журнала событий:  
  
#### <a name="account-logon"></a>Вход учетной записи  
  
##### <a name="credential-validation"></a>Проверка учетных данных  
Эта Подкатегория формирующий проверочных тестов на учетные данные, отправленные для запрос на вход учетной записи пользователя. Эти события происходят на компьютере, который не является полномочным для учетных данных. Для учетных записей домена контроллер домена не является полномочным, тогда как для локальных учетных записей локального компьютера является полномочным.  
  
В доменных средах большинство событий входа учетной записи заносятся в журнал безопасности контроллеров домена, которые являются полномочными для доменных учетных записей. Однако при локальных учетных записей используются для входа эти события могут возникать на других компьютерах в организации.  
  
##### <a name="kerberos-service-ticket-operations"></a>Операции с билетами службы Kerberos  
Эта Подкатегория отчеты события, создаваемые процессы запроса билета Kerberos на контроллере домена, который является заслуживающим доверия для учетной записи домена.  
  
##### <a name="kerberos-authentication-service"></a>Служба проверки подлинности Kerberos  
Эта Подкатегория отчеты события, создаваемые службой проверки подлинности Kerberos. Эти события происходят на компьютере, который не является полномочным для учетных данных.  
  
##### <a name="other-account-logon-events"></a>Другие события входа учетной записи  
Эта Подкатегория отчеты события, происходящие в ответ на учетные данные для запрос на вход учетной записи пользователя, не связанные с проверки учетных данных или билеты Kerberos. Эти события происходят на компьютере, который не является полномочным для учетных данных. Для учетных записей домена контроллер домена не является полномочным, тогда как для локальных учетных записей локального компьютера является полномочным.  
  
В доменных средах большинство событий входа в систему регистрируются в журнале безопасности контроллеров домена, которые являются полномочными для доменных учетных записей. Однако при локальных учетных записей используются для входа эти события могут возникать на других компьютерах в организации. Примеры может включать следующее:  
  
-   Отключение удаленного рабочего стола служб сеанса.  
  
-   Новые сеансы служб удаленных рабочих столов  
  
-   Блокировка и разблокировка рабочей станции.  
  
-   Вызов заставки.  
  
-   Закрытие заставки.  
  
-   Обнаружение Kerberos атаки с повторением, в которой запрос Kerberos с идентичным сведениями получен дважды.  
  
-   Доступ к беспроводной сети предоставлен учетной записи пользователя или компьютера  
  
-   Доступ к проводной сети 802.1 x сети предоставлен учетной записи пользователя или компьютера  
  
#### <a name="account-management"></a>Управление учетной записью  
  
##### <a name="user-account-management"></a>Управление учетными записями пользователей  
Эта Подкатегория сообщает каждого события управления учетными записями пользователей, например при создании, изменении или удален; учетной записи пользователя Учетная запись пользователя переименован, отключен или включено; или установка или изменение пароля. Если включен этот параметр политики аудита, администраторы могут отслеживать события для обнаружения вредоносных случайного и авторизованным создания учетных записей пользователей.  
  
##### <a name="computer-account-management"></a>Управление учетной записью компьютера  
Эта Подкатегория отчеты каждого события управления учетными записями компьютера, например при учетной записи компьютера создан, изменены, удалены, переименованы, отключен или включен.  
  
##### <a name="security-group-management"></a>Управление группой безопасности  
Эта Подкатегория отчеты каждого события управления группой безопасности, например при создании, изменении или удалить группу безопасности или при добавлении или удаления из группы безопасности члена. Если включен этот параметр политики аудита, администраторы могут отслеживать события для обнаружения вредоносных случайного и авторизованным создания учетных записей групп безопасности.  
  
##### <a name="distribution-group-management"></a>Управление группой распространения  
Эта Подкатегория отчеты каждого события Управление группами рассылок, например при создании, изменении или удалить группу рассылки или при добавлении или удаления из группы распространения члена. #### <a name="object-access"></a>Доступ к объектам  
  
##### <a name="file-system"></a>Файловая система  
Эта Подкатегория отчеты при доступе к объектам файловой системы. Только объектов файловой системы с SACL вызывают создание событий аудита для созданного и только при доступе к способом, соответствующие их записи системный список управления ДОСТУПОМ. Сам по себе этот параметр политики не приведут к аудит событий. Определяет, подлежит ли аудиту событие пользователя, подключающегося к объект файловой системы, с указанной системный список управления доступом (SACL), эффективно Включение аудита вступили в силу.  
  
Если аудит доступа к объектам параметр настроен на **успех**, то запись аудита генерируется каждый раз, успешном доступе пользователя к объект с указанный список SACL. Если этот параметр политики настроен для **сбой**, то запись аудита генерируется каждый раз, которое завершается ошибкой пользователя при попытке получить доступ к объекту с указанный список SACL.  
  
##### <a name="registry"></a>Реестра  
Эта Подкатегория отчеты при доступе к объектам реестра. Только объекты с системных списков управления доступом реестра вызывают создание событий аудита для созданного и только при доступе к способом, соответствующие их записи системный список управления ДОСТУПОМ. Сам по себе этот параметр политики не приведут к аудит событий.

### <a name="configuring-windows-audit-policy"></a>Настройка политики аудита Windows  
Политики аудита Windows можно задать с помощью групповой политики, auditpol.exe, API-интерфейсами или реестра изменения. Рекомендуемые методы для настройки политики аудита в большинстве компаний, групповую политику или auditpol.exe. Настройка политики аудита системы требует разрешений уровня администратора учетной записи или делегированы соответствующие разрешения.  
  
> [!NOTE]  
> **Управление аудитом и журналом безопасности** право предоставлять субъектов безопасности (Администраторы имеют его по умолчанию) позволяет изменять параметры для отдельных ресурсов, таких как файлы, объекты Active Directory и разделы реестра аудит доступа к объектам.

Каждой категории политики аудита можно включить для **успех**, **сбой**, или **успех** и событий сбоя.

Чтобы настроить расширенной политики аудита, настройте соответствующие подкатегории, расположенные в папке **компьютера Конфигурация компьютера\Параметры Windows\Параметры безопасности\расширенная аудита политика** (см. следующий пример из редактор локальных групповых политик (gpedit.msc) снимок экрана). Каждую подкатегорию политики аудита можно включить для **успех**, **сбой**, или **успех** и **сбой** события.

#### <a name="setting-windows-audit-policy-using-auditpolexe"></a>Настройка политики аудита Windows с помощью Auditpol.exe  
Auditpol.exe (для параметра политики аудита Windows) впервые появился в Windows Server 2008 и Windows Vista. Изначально только auditpol.exe может использоваться для задания расширенной политики аудита, но групповой политики можно использовать в Windows Server 2012, Windows Server 2008 R2 или Windows Server 2008, Windows 8 и Windows 7.  
  
Auditpol.exe является служебной программы командной строки. Синтаксис выглядит следующим образом:  
  
**Auditpol/Set / < категории | Подкатегория >:<audit category> / < успех | сбоя: > или < включить | отключить >**  
  
Примеры синтаксиса Auditpol.exe.  
  
**Auditpol/set /subcategory: /failure:enable /success:enable «Управление учетной записью пользователя»**  
  
**Auditpol/set /subcategory: «входа в систему» /success:enable /failure:enable**  
  
**Auditpol/set /subcategory: /failure:enable «Основного режима IPSEC»**  
  
> [!NOTE]  
> Auditpol.exe задает расширенной политики аудита локально. Если локальная политика конфликтует с Active Directory или локальная групповая политика, параметры групповой политики обычно преобладают над параметрами auditpol.exe. Если имеется несколько групповой или локальной политики конфликты, только одна политика будут применяться (который замените). Политики аудита не будут объединены.  
  
#### <a name="scripting-auditpol"></a>Сценарии Auditpol  
Корпорация Майкрософт предоставляет [пример сценария](https://support.microsoft.com/kb/921469) для администраторов, которые хотят настроить дополнительные политику аудита с помощью сценария вместо вручную вводить в каждой команде auditpol.exe.  
  
**Примечание** групповой политики не сообщает о всегда точно состояние все включенные политики аудита, тогда как auditpol.exe. В разделе [получение действующей политики аудита в Windows 7 и Windows 2008 R2](http://blogs.technet.com/b/askds/archive/2011/03/11/getting-the-effective-audit-policy-in-windows-7-and-2008-r2.aspx) для получения дополнительных сведений.  
  
#### <a name="other-auditpol-commands"></a>Другие команды Auditpol  
Auditpol.exe можно использовать для сохранения и восстановления локальную политику аудита и для просмотра других аудита связанных команд. Вот другой **auditpol** команды.  
  
**auditpol/clear** — используется для очистки и сброс параметров политики аудита на локальном  
  
**/ BACKUP Auditpol/file:<filename> ** — используется для архивации текущего локальную политику аудита в двоичный файл  
  
**/ File/Restore Auditpol:<filename> ** — используется для импорта файл политики аудита, сохраненные ранее на локальную политику аудита  
  
**Auditpol / < get/set >/параметр:<CrashOnAuditFail> / < Включение и отключение >** -вызывает при включении этого параметра политики аудита системы, чтобы немедленно прекратить (с STOP: C0000244 {аудита Сбой} сообщения) Если по какой-либо причине невозможно вести журнал аудита безопасности. Как правило, событие не удается войти в том случае, если журнал аудита безопасности заполнен и хранения метод, указанный в журнале безопасности **не перезаписывать события** или **Затирать старые события по дням**. Обычно он включается только в средах, требующих более гарантию того, что вход в журнал безопасности. Если параметр включен, администраторы должны точно смотрите размер журнала безопасности и повернуть журналы при необходимости. Его можно также настраивать с помощью групповой политики, изменив параметр безопасности **Аудит: немедленное отключение системы, если невозможно внести в журнал записи об аудите безопасности** (по умолчанию = отключено).  
  
**Auditpol / < get/set >/параметр:<AuditBaseObjects> / < Включение и отключение >** -этот параметр политики аудита определяет, следует ли проводить аудит доступа глобальных системных объектов. Если эта политика включена, то системных объектов, таких как мьютексы, события, семафоры и устройства DOS будут создаваться с по умолчанию системный список управления доступом (SACL). Большинство администраторов рекомендуем аудит глобальных системных объектов, чтобы быть слишком «шумными», и они будут только включить Если есть подозрения, вредоносных программ взлома. Только именованные объекты, получают список SACL. Если также включен аудит объект доступа аудита политики (или подкатегория аудита объектов ядра), осуществляется аудит доступа к таким системным объектам. При настройке этого параметра безопасности, изменения не вступят в силу только после перезапуска Windows. Эту политику можно также задать с помощью групповой политики, изменив параметр безопасности Аудит доступа глобальных системных объектов (по умолчанию = отключено).  
  
**Auditpol / < get/set >/параметр:<AuditBaseDirectories> / < Включение и отключение >** -этот параметр политики аудита указывает, что именованный ядра объектов (таких как мьютексы и семафоры) необходимо предоставить системных списков управления доступом, когда они создаются. AuditBaseDirectories влияет на объекты контейнера во время AuditBaseObjects влияет на объекты, которые не могут содержать другие объекты.  
  
**Auditpol / < get/set >/параметр:<FullPrivilegeAuditing> / < Включение и отключение >** -  
  
Этот параметр политики аудита, указывает ли клиент создает событие, когда один или несколько из этих привилегий, назначаются маркер безопасности пользователя: AssignPrimaryTokenPrivilege, AuditPrivilege, BackupPrivilege, CreateTokenPrivilege, DebugPrivilege, EnableDelegationPrivilege, ImpersonatePrivilege, LoadDriverPrivilege, RestorePrivilege, SecurityPrivilege, SystemEnvironmentPrivilege, TakeOwnershipPrivilege и TcbPrivilege. Если этот параметр не включен (по умолчанию = отключено), привилегии BackupPrivilege и RestorePrivilege не регистрируются. Включение этого параметра можно сделать в журнал безопасности слишком шумный (иногда сотни событий в секунду) во время операции резервного копирования. Эту политику можно также задать с помощью групповой политики, изменив параметр безопасности **Аудит: аудит использования привилегии на архивацию и восстановление**.

### <a name="enforcing-traditional-auditing-or-advanced-auditing"></a>Применение традиционных аудита или расширенного аудита  
В Windows Server 2012, Windows Server 2008 R2, Windows Server 2008, Windows 8, Windows 7 и Windows Vista администраторы могут выбрать включение девяти категориям традиционных или использовать подкатегории. Это двоичного выбора, должны быть внесены в каждой системе Windows. Основные категории можно включить либо subcategoriesit не может быть одновременно.  
  
Для предотвращения перезаписи подкатегорий политики аудита политики устаревших традиционных категории, необходимо включить **параметрами подкатегории политики аудита Force (Windows Vista или более поздней версии) переопределяет параметры категории политики аудита** параметр политики, расположенные в папке **Конфигурация компьютера\Параметры Windows\Параметры безопасности\Локальные Политики\параметры безопасности**.  
  
Мы рекомендуем включить и настроить вместо девять основных категорий подкатегории. Для этого требуется включить параметр групповой политики (чтобы разрешить подкатегории переопределить категорий аудита) вместе с настройке различных подкатегории, которые поддерживают политики аудита.  
  
Подкатегорий можно настроить с помощью нескольких методов, в том числе групповой политики и программы командной строки, auditpol.exe.
