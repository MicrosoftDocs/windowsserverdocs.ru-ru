# Планирование раскрытия

>Область применения: Windows Server 2016, Windows Server 2012 R2, Windows Server 2012

*Номер один закон: Никто не считает ничего плохих может произойти на них, пока получит.* - [10 неизменяемые законов администрирования безопасности](https://technet.microsoft.com/library/cc722488.aspx)  
  
Планы аварийного восстановления во многих организациях сосредоточиться на восстановление из региональных сбоев или сбоев, которые могут привести к потере вычислений служб. Тем не менее, при работе с клиентами скомпрометированных часто выясняется, что восстановление из преднамеренное компромиссов отсутствует в свои планы аварийного восстановления. Это особенно значение true, если компромиссов приводит к хищения личных данных интеллектуальная собственность или преднамеренное удаления, использующая логические границы (например, удаления всех Active Directory — домены и всех серверов) (например, вместо того чтобы физический границ удаления центра обработки данных). Несмотря на то, что в организации может быть реагирование планы, определяющие начальные действия необходимо выполнить, когда обнаружено компромиссов, эти планы часто опустить действия, чтобы восстановить из компромиссов, влияет на весь вычислительных инфраструктуры.  
  
Поскольку Active Directory предоставляет широкий возможности управления доступом и учетными данными для пользователей, серверов, станции и приложений, неизменно предназначено злоумышленников. Если он получает пользователь с привилегированными доступ к Active Directory домена или контроллера домена, что доступ можно использовать с помощью доступ, управлять или даже destroy всего леса Active Directory.  
  
Этот документ обсудить некоторые из наиболее распространенных атак Windows и Active Directory и следует контрмерам, которые можно реализовать для уменьшения уязвимой зоны, но только что способ восстановить в случае завершения компромиссов службы каталогов Active Directory Подготовка раскрытия, это происходит. В этом разделе рассматривается меньше на странице сведения о технической реализации, чем в предыдущих разделах этого документа и более высокого уровня рекомендациям, которые можно использовать для создания комплексный неполный планом для обеспечения безопасности и управление вашей организации критических бизнес и ИТ-ресурсами.  
  
Ли инфраструктуры никогда не было атаки, resisted предпринимается нарушения, или имеет succumbed атак и были полностью ненадежен, следует учитывать, что вам будет иметь атаки дальнейшем избежать реальности. Невозможно для предотвращения атак, но возможно в самом деле предотвратить существенной нарушений или оптовый компромиссов. Все организации должен тесно оценить их существующие программы управления рисками и внесите необходимые изменения, чтобы уменьшить уровень общей уязвимость путем изменения неравномерные инвестиций в предотвращение, обнаружение, контейнеры и восстановление.  
  
Для создания эффективных способах защиты обеспечивая служб для пользователей и организаций, которые зависят от инфраструктуры и приложений, можно необходимо учитывать праздник способы избавиться, обнаружить и содержат компромиссов в вашей среде и нажмите Восстановить из нарушение безопасности. Подходы и рекомендации в этом документе не позволяют восстановить скомпрометированных установку Active Directory, но может помочь защитить к следующему.  
  
Рекомендации для восстановления леса Active Directory представлены в [Windows Server 2012: Планирование восстановление леса Active Directory](https://www.microsoft.com/download/details.aspx?id=16506). Вы сможете избежать среды новые из полностью несанкционированного доступа, но даже если вы не можете будет иметь средства для восстановления и возврата контроля среды.

Эталонного леса является просто установленную леса Active Directory, обычно ограничен размер и область и которой построен с помощью текущего операционных систем, приложений и с принципами описано в [Уменьшение атаки Active Directory Поверхностная диаграмма](../../../ad-ds/plan/security-best-practices/../../../ad-ds/plan/security-best-practices/Reducing-the-Active-Directory-Attack-Surface.md).

Подробные инструкции по проектирование и реализация эталонного установки AD DS относятся за пределами области этот документ, выполните некоторые общие принципы и рекомендации для создания в котором можно размещать наиболее важных «безопасного ячейки» средства. 6.  Упрощение безопасности для конечных пользователей.  
  
### Определение принципов для разделения и защита важных активов  

Характеристики эталонного среды, созданный house критических ресурсов могут различаться. Например можно создать эталонного леса, в которую можно перенести только VIP пользователей и конфиденциальные данные, можно получить доступ только те пользователи. Вы можете создавать эталонного леса, в котором переноса не только VIP пользователей, но реализовать как административные леса реализации принципы, описанные в [Сокращение атак Active Directory](../../../ad-ds/plan/security-best-practices/../../../ad-ds/plan/security-best-practices/Reducing-the-Active-Directory-Attack-Surface.md) для создания безопасного учетных записей администраторов и hosts, которые могут использоваться для управления вашей устаревшие леса из эталонного леса. Можно реализовать «специальные» леса, в которой хранятся VIP учетные записи, привилегированных и требуя дополнительных безопасности, например серверы сертификат службы Active Directory (AD CS) с целью исключительную разделение их из менее безопасный систем леса. Наконец можно реализовать эталонного леса, которая становится фактически место для всех новых пользователей, систем, приложений и данных, чтобы в итоге списать устаревшие леса через attrition.  
  
Вне зависимости от ли эталонного леса содержит несколько пользователей и систем или она является основанием более сложные миграции, необходимо выполнить эти принципы планирование:  
  
1.  Предположим, что ваш устаревшие леса стал ненадежен.  
  
2.  Не настроен эталонного среды о доверии устаревшего леса, несмотря на то, что можно настроить устаревшие среды о доверии эталонного леса.  
  
3.  Не переносятся учетные записи пользователей или групп из устаревшего леса к исходному среде Если существует вероятность того, что учетные записи пользователей в группах, журнал ИД безопасности и другие атрибуты могут намеренно изменены. Вместо этого используйте «nonmigratory» подходы для заполнения эталонного леса. (Nonmigratory подходы описаны далее в этом разделе.)  
  
4.  Не переносятся компьютеров с подключением к устаревшим леса эталонного леса. Реализации исходное серверы эталонного леса, установка приложений на только что установленных серверов и перенос данных приложения в установленную системы. Для файловых серверов скопировать данные в только что установленных серверов, задать списки управления доступом с помощью пользователей и групп в нового леса и создайте серверов печати аналогичным образом.  
  
5.  Не разрешать установку устаревших операционных систем и приложений в эталонного леса. Если приложение не удается обновить и только что установлена, оставить на устаревшие леса и рассмотрим creative удаления заменить функциональные возможности приложения.  
  
### Определение плана ограниченный, основанная на риск миграции  
Создание ограниченный плана миграции на основе риск просто означает, что при выборе какие пользователей, приложений и данных для переноса в эталонного леса, необходимо определить цели миграции с учетом степенью риска, к которому предоставляется вашей организации, если один из снижена пользователей или систем. VIP пользователей, учетные записи которых, скорее всего, злоумышленников определенным следует устанавливают эталонного леса. Приложения, которые предоставляют функции важную деловую должен быть установлен на только что встроенных серверы эталонного леса и конфиденциальные данные нужно переместить защищать серверы эталонного леса.  
  
Если очистить рисунка наиболее важных пользователей уже нет систем, приложений и данных в вашей среде Active Directory работать с бизнес-единицы для определения их. Любое приложение, необходимое для бизнеса для работы должны подтвердить свою личность, как должно быть любой серверах, на которых запуска важных приложений или важные данные хранятся. Определяя пользователями и ресурсами, которые необходимы для вашей организации должен работать по-прежнему, создайте естественно упорядоченный коллекции эксплуатации, на которые следует обратить внимание усилий.  
  
### Использование «Nonmigratory» миграции  
Будет ли вы знаете, что среды стал ненадежен, подозреваете он стал ненадежен, или просто не хотите перенести старые данные и объекты из старую установку Active Directory в новый, рассмотрите возможность подходы к переносу, которые технически не выполнять объекты «перенести».  
  
### "Ученые записи пользователей"  
В традиционных Active Directory миграции из одного леса в другой атрибут SIDHistory (идентификатор безопасности журнал) на объекты пользователя используется для хранения ИД безопасности пользователей и идентификаторы безопасности групп, что пользователи были членами устаревшие леса. Если в доступе к ресурсам устаревшие леса миграции учетных записей пользователей для нового леса, ИД безопасности в журнале ИД безопасности используются для создания маркер доступа, который пользователи могут получить доступ к ресурсам, к которым у них были до учетные записи были перенесены.  
  
Сохранение истории ИД безопасности, однако оказалось проблемы в некоторых средах из-за заполнение маркеры доступа пользователей с текущим и статистических ИД безопасности может привести Раздувание маркера. Маркер Раздувание использует количество идентификаторы безопасности, которые должны храниться в маркер доступа пользователя вопросов или превышает интервал, доступные в маркер.  
  
Несмотря на то, что можно увеличить размер маркеров в некоторой степени, максимум для маркеров Раздувание рекомендуется сократить количество идентификаторов безопасности, связанных с учетными записями пользователей ли рационализации членство в группах, исключая журнал ИД безопасности или сочетание этих. Дополнительные сведения о маркера Раздувание читайте [MaxTokenSize и маркер размеров Kerberos](http://blogs.technet.com/b/shanecothran/archive/2010/07/16/maxtokensize-and-kerberos-token-bloat.aspx).  
  
Вместо переноса пользователей из устаревших среды (особенно по одному в какие членство в группах и ИД безопасности могут выдать историю) с помощью ИД безопасности журнала, рассмотрите возможность усиление метакаталог приложениям «перенести» пользователи, без хранение историю ИД безопасности в нового леса. При создании учетных записей пользователей в нового леса метакаталог приложения можно использовать для сопоставления соответствующих учетных записей в устаревшего леса учетные записи.  
  
Для предоставления новые учетные записи пользователю доступ к ресурсам устаревшие леса метакаталог оборудование для идентификации группы ресурсов, в которые устаревших учетных записей пользователей был предоставлен доступ и нажмите Добавить новые учетные записи пользователей в эти группы. В зависимости от стратегии группы в устаревшего леса может потребоваться создать локальные группы домена для доступа к ресурсам или преобразование существующей группы в локальные группы домена разрешить новые учетные записи для добавления группы ресурсов. В центре внимания сначала наиболее важных приложений и данных и перенос их в новой среды (независимо от ИД безопасности журнала), можно ограничить увеличиваются в среде устаревшие трудозатрат.  
  

  
### Серверы и рабочие станции  
В традиционных миграции из одного Active Directory леса на другой, миграция компьютеров часто относительно простой по сравнению с переноса пользователей, групп и приложений. В зависимости от роли компьютера миграции нового леса может состоять исключив старый домен и присоединения к новый. Тем не менее миграция учетные записи компьютера без изменений в сравнениях эталонного леса цель создания новой среды. Вместо миграции (потенциально скомпрометированные, неправильно настроенные или устаревшие) учетных записей компьютеров для нового леса, только что устанавливайте серверы и рабочие станции новой среды. Вы можете переместить данные из системы в устаревшего леса к системам эталонного леса, но не системы, дом данные.  
  
### Приложения  

Приложений можно представить значащий проблема в любой миграции из одного леса, но в случае «nonmigratory» миграции, один из самых основные принципы, которые следует применять не приложений эталонного леса следует текущий, поддерживаются, а только что установлена. Можно перенести данные из экземпляров приложения старого леса возможности. В ситуациях, в которых приложения «восстановить» эталонного леса следует учитывать подходы, например creative удаления или изоляции устаревших приложений, как описано в следующем разделе.  
  
### Реализация творческого удаления  
Creative удаления — экономики термин, описывающий экономического развития созданные удаления предшествующего заказа. За последние годы в области информационных технологий был применен термин. Это обычно подразумевается механизмы какие старый инфраструктура исключается, не обновив его, но его замена другое имя полностью. 2011 [ITXPO симпозиуме Gartner](http://www.gartner.com/technology/symposium/orlando/) отделов и руководителей старший ИТ-отделов представлены creative удаления к одному из его ключа темы для увеличения и сокращения затрат в эффективности. Возможны усовершенствования системы безопасности в качестве natural outgrowth процесса.  

Например организации может состоять из несколько подразделений, использующих разных приложение, выполняющих похожие функции, с различными уровнями поддержки modernity и поставщика. Как правило ИТ возможно, ответственного за ведение каждого филиала приложения отдельно и мероприятиях консолидации будет состоять из попытке выяснить, какое приложение предлагаются наиболее функциональные возможности и затем переноса данных в, приложение в других представлениях.  
  
В creative удаления вместо обслуживания устаревшие или избыточные приложений реализации совершенно новых приложений для замены старого, перенос данных в новых приложений и списать старых приложений и систем, на которых они запущены. В некоторых случаях можно реализовать creative удаления старых приложений с помощью нового приложения в свой собственный инфраструктуры развертывания, но возможно, следует учитывать вместо перенос приложения решение на основе облака.  
  
Путем развертывания облачных приложений для замены устаревших внутренние приложений, не только уменьшить объем работ по обслуживанию и затраты, но уменьшить атак вашей организации, удалив старые системы и приложения, которые представляют уязвимостей для злоумышленников влиял. Этот подход обеспечивает быстрый способ для организации для получения нужного функциональных возможностей во время одновременно исключая устаревших целевых в инфраструктуре. Несмотря на то, что принцип creative удаления не применяется ко всем ИНФОРМАЦИОННЫМ ресурсам, он предоставляет возможность часто приемлемым исключая устаревших систем и приложений при развертывании надежных, безопасных и облачных приложений одновременно.  
  
### Изоляции устаревших систем и приложений  
Естественные outgrowth перенос важных пользователей и систем к исходному, безопасной среды — что устаревшие леса будет содержать менее важной информации и систем. Несмотря на то, что устаревших систем и приложений, которые осталось менее безопасен среды могут отображаться с повышенными правами риск раскрытия, эти значения также представляют ограниченной серьезность компромиссов. Rehoming и модернизация важных активов, можно сосредоточиться на развертывание эффективного управления и контроля во время без необходимости вместить устаревшие настройки и протоколов.  
  
Когда переместить важных данных к исходному леса, можно вычислить параметры для дальнейшей изоляции устаревших систем и приложений в вашей «основной» леса AD DS. Несмотря на то, что можно реализовать творческие удаления, чтобы заменить одно приложение и серверы, для которых необходимо запустить, в других случаях может потребоваться дополнительные изоляции наименее безопасный систем и приложений. Например приложение используется несколько пользователей, но что предполагает устаревших учетных данных как хэш-коды диспетчер локальной сети можно перенести в небольших доменов можно создать для поддержки систем, для которых у вас нет вариантов замены.  
  
При удалении этих систем из доменов, где они принудительно реализации устаревшие параметры, впоследствии можно увеличить безопасности домены, настроив их поддерживает только текущий операционных систем и приложений. Хотя предпочтительнее списать устаревших систем и приложений, когда это возможно. Если списать просто не подходящим для небольшой сегмент устаревшие генеральной совокупности., разделения в отдельном домена (или леса) позволяет выполнять добавочные усовершенствования в остальной части старую установку.  
  
### Упрощение безопасности для конечных пользователей  
В большинстве организаций у пользователей, имеющих доступ к наиболее важных данных из-за природы их ролей в организации часто наименьшее количество времени, чтобы выделять для изучения сложные access ограничениями и элементов управления. Несмотря на то, что должны иметь программы образования эффективной защиты для всех пользователей в вашей организации, следует также сосредоточиться на выполнение безопасности как простой с помощью максимально реализации элементов управления, которые являются прозрачным и упрощения процесса принципов пользователей, которые придерживаться.  
  
Например может определять политики, в котором руководители и другие виртуальные IP требуются для использования станции безопасный доступ к важным данным и систем, позволяя использовать их других устройств для доступа к менее конфиденциальные данные. Это простой принцип для пользователей для запоминания, но можно реализовать большим количеством элементов управления серверной помогут обеспечение подхода.  

Можно использовать [Надежность механизма проверки подлинности](https://technet.microsoft.com/library/dd391847(v=WS.10).aspx) , чтобы разрешить пользователям доступ к конфиденциальные данные только в том случае, если они войти их безопасных систем с помощью смарт-карты и можно использовать для управления систем, из которых можно ограничения прав IPsec и пользователя подключение к репозиториев конфиденциальных данных. [Набор средств классификации данных Microsoft](https://www.microsoft.com/download/details.aspx?id=27123) можно использовать для построения инфраструктура классификации надежные файлов, а можно реализовать [Динамического контроля доступа](http://blogs.technet.com/b/windowsserver/archive/2012/05/22/introduction-to-windows-server-2012-dynamic-access-control.aspx) , чтобы ограничить доступ к данным на основе характеристик попытке доступа перевода бизнес-правила в технические элементы управления.  
  
С точки зрения пользователя, для доступа к конфиденциальные данные из защищенных системы «просто works» и попытка сделать это из небезопасного системы «просто не». Тем не менее с точки зрения мониторинга и управления ими среды помогает создать явные закономерности в способ доступа пользователей к важным данным и систем, упрощая обнаружить попытки непредвиденных доступа.  
  
В средах, в которых пользователь устойчивость к длительный и сложный пароли привела к недостаточно политик, особенно для пользователей VIP рассмотрим альтернативные способы проверки подлинности, ли с помощью смарт-карт (которой бывают нескольких факторов формы и с дополнительными возможностями для повышения проверку подлинности) биометрические элементов управления, например, читатели проведите пальцем или даже проверку подлинности данные, которые защищены с доверенных платформы модуля TPM микросхемы на компьютерах пользователей. Несмотря на то, что многофакторной проверки подлинности не предотвратить атак кражи учетных данных, если компьютер уже нарушена, указав пользователей проверка подлинности с помощью элементов управления, вы можете назначать более надежные пароли учетные записи пользователей которого традиционное пользователя элементы управления имя и пароль, громоздкими.
