# <a name="planning-for-compromise"></a>Планирование при компрометации

>Область применения. Windows Server 2016, Windows Server 2012 R2, Windows Server 2012

*Закон часть первая: Никто не верит, что что-либо неверный может произойти, пока получит.* - [10 непреложных законов безопасности администрирования](https://technet.microsoft.com/library/cc722488.aspx)  
  
Планы аварийного восстановления во многих организациях сосредоточиться на восстановление после аварий и сбоев, которые приводят к потере вычислительных служб. Тем не менее, при работе с клиентами скомпрометированных часто обнаружим, что восстановление после преднамеренного компрометации отсутствует в свои планы аварийного восстановления. Это особенно верно при компрометации результатов в случае кражи интеллектуальной собственности или намеренно деструктора, который использует логические границы (например, удаление всех доменов Active Directory или все серверы) вместо физических границ (такие как уничтожение центра обработки данных). Несмотря на то, что организация может иметь планы реагирования на инциденты, определяющих начальные действия, выполняемые при обнаружении нарушения безопасности, эти планы часто опустить действия по восстановлению от взлома, которые влияют на всю вычислительную инфраструктуру.  
  
Поскольку Active Directory предоставляет широкие возможности управления удостоверениями и доступом для пользователей, серверов, рабочих станций и приложений, неизменно целью злоумышленников. Если злоумышленник получит доступ высоким уровнем привилегий в домене Active Directory или контроллер домена, что доступ можно использовать доступ, управления или даже уничтожить весь лес Active Directory.  
  
В этом документе описывается, некоторые из наиболее распространенных атак против Windows и Active Directory и меры противодействия, которые можно реализовать, чтобы уменьшить поверхность атаки, но единственный надежный способ восстановления в случае полной компрометации Active Directory должен быть Подготовка к компрометации, до ее выполнения. В этом разделе рассматриваются, не отвлекаясь на сведения о технической реализации, чем в предыдущих разделах этого документа, и более на общие рекомендации, которые можно использовать для создания целостного комплексный подход для защиты и управления вашей организации критические бизнеса и ИТ-активами.  
  
Ли инфраструктуры никогда не подверглись атаке, resisted попытки брешей в системе, или succumbed к атакам и был полностью скомпрометирован, необходимо запланировать неизбежно реальность, что вам будут атаковано снова и снова. Невозможно для предотвращения атак, но он действительно может быть невозможно предотвратить значительное нарушений или оптовой компрометации. Каждая организация должна точно оценить их программ управления рисками и внести необходимые изменения с целью сокращения их общий уровень уязвимость с сбалансированной инвестирует в предотвращения, обнаружения, вложения и восстановления.  
  
Для создания эффективных средств защиты, по-прежнему предоставляя такие службы для пользователей и предприятий, которые зависят от инфраструктуры и приложений, вы может должны рассмотреть novel способов предотвращения, обнаружения и содержать компрометации в вашей среде и восстанавливать из нарушения системы безопасности. Подходы и рекомендации в этом документе может затруднить восстановление скомпрометированных установки Active Directory, но может помочь вам защитить ваши следующим.  
  
Рекомендации для восстановления леса Active Directory, можно найти в [Windows Server 2012: Планирование Active Directory Forest Recovery](https://www.microsoft.com/download/details.aspx?id=16506). Можно предотвратить новую среду полностью предотвратить несанкционированный доступ, но даже если это невозможно, вы получите средства для восстановления и восстановить контроль над вашей среды.  
  
## <a name="rethinking-the-approach"></a>Переосмысление подход  
*Закон цифра восемь: Защита сети сложности прямо пропорционален ее сложность.* - [10 непреложных законов безопасности администрирования](https://technet.microsoft.com/library/cc722488.aspx)  
  
Это хорошо общепринятые что если злоумышленник получил системы, администратор, root или доступа к компьютеру, независимо от операционной системы, этот компьютер может больше не быть считаются надежными, независимо от того, сколько усилий вносятся «очистить» система. Active Directory ничем не отличается. Если злоумышленник получил привилегированный доступ к контроллером домена или высоким уровнем привилегий учетной записи в Active Directory, если у вас нет записи каждого изменения, он создает или проверенной рабочей резервной копии, можно восстановить каталог никогда не полностью состояние уровня доверия.  
  
При компрометации и изменении с помощью которых злоумышленники рядового сервера или рабочей станции компьютер больше не является доверенной, но по-прежнему можно доверять соседних устойчивости серверов и рабочих станций. Компрометация на одном компьютере не означает, что все компьютеры были скомпрометированы.  
  
Тем не менее в домене Active Directory, все контроллеры домена размещения реплики одной и той же базы данных AD DS. Если один контроллер домена нарушена и злоумышленник изменяет базу данных AD DS, для каждого контроллера домена в домене и в зависимости от раздела, в котором изменения вносятся, леса реплицировать эти изменения. Даже если необходимо переустановить каждый контроллер домена в лесу, просто переустановке узлы, на которых расположена база данных AD DS. Вредоносных изменений в Active Directory будут реплицированы на контроллеры домена вновь установленной так же легко, как они будут реплицированы на контроллеры домена, которые использовались в лет.  
  
В оценке скомпрометированных средах, часто обнаружим, что должен быть первый нарушения «event» был фактически запуск после недели, месяцы или годы после злоумышленники изначально компрометации среды. Злоумышленники обычно получается учетные данные для длинных высокопривилегированные учетные записи, до того, как было обнаружено нарушение безопасности они используются эти учетные записи для компрометации каталога, контроллеры домена, рядовые серверы, рабочие станции и даже подключены отличных от Windows системы.  
  
Эти результаты соответствуют несколько выводов в Verizon 2012 данных нарушения исследования отчет, который указывает, что:  
  
-   98 процентов нарушения конфиденциальности данных, происходит из-за внешних агентов  
  
-   85 процентов утечки данных заняло недели или другие возможности  
  
-   92 процента инцидентов обнаруженных сторонними разработчиками, и  
  
-   97 процентов брешей в системе были рода, то, что это простой или средний уровень элементов управления.  
  
Компрометация в степени, описанные ранее эффективно после неустранимого и стандартная рекомендация «упростить и перестроить» каждой поврежденной системе не просто возможные допустимого или даже если Active Directory компрометации или уничтожения. Даже восстановление до известного рабочего состояния не устраняет недостатки, которые разрешены среды будут скомпрометированы в первую очередь.  
  
Несмотря на то, что необходимо защищать каждым аспектом вашей инфраструктуры, злоумышленнику достаточно найти достаточно недостатки ваши средства защиты, чтобы перейти к их нужной цели. Если ваша среда — относительно простой и эталонного и исторически хорошо управляемой, применяя рекомендации, приведенные ранее в этом документе может быть просто предложение.  
  
Тем не менее, мы пришли к выводу, более старые, более крупными и сложными является среде, тем выше вероятность что рекомендации, приведенные в этом документе будут выбираться или даже невозможно реализовать. Это гораздо сложнее защитить инфраструктуру за прошедшие периоды, чем начать с нуля и создать среду, которая находится устойчивы к атакам и создать угрозу. Но как было отмечено выше, это не небольшой задачей, для перестроения всего леса Active Directory. По этим причинам рекомендуется более узкий, направленный подход для защиты ваших лесов Active Directory.  
  
Вместо того чтобы сосредоточиться на и исправить все, что, которые являются «неисправный», рекомендуется подход, в котором можно определять их приоритеты в зависимости от наиболее важных для бизнеса и в вашей инфраструктуре. Вместо того, чтобы исправлять среде, заполненный устаревшая, неправильно настроенных систем и приложений, рекомендуется создать новый небольшой, безопасной среды, в котором можно безопасно перенести пользователей, систем и сведения, которые наиболее важны для вашего бизнеса.  
  
В этом разделе мы рассмотрим подход, по которому можно создать эталонный леса AD DS, который служит в качестве «жизнь яхту» или «безопасный» ячейки для основных бизнес-инфраструктурой. Эталонный лес — это просто в только что установленного лес Active Directory, обычно ограничен по размеру и области, поэтому построены с использованием текущего операционных систем, приложений и с принципы, описанные в [уменьшение активного Каталог возможных направлений атак](../../../ad-ds/plan/security-best-practices/../../../ad-ds/plan/security-best-practices/Reducing-the-Active-Directory-Attack-Surface.md).  
  
Путем реализации рекомендуемые параметры конфигурации в лесу, созданный, можно создать установку AD DS, которая построена с нуля с помощью параметров безопасности и рекомендации, и уменьшить проблемы, которые сопровождают поддержки устаревших систем и приложения. Хотя подробные инструкции по проектированию и реализации установки AD DS в первоначальном состоянии выходит за рамки данного документа, необходимо выполнить некоторые общие принципы и рекомендации для создания «безопасного ячейки», в который можно разместить самые важные ресурсы. Эти рекомендации являются следующим образом:  
  
1.  Определите принципов для разделения и защиты важных активов.  
  
2.  Определение плана миграции ограниченный, на основе рисков.  
  
3.  Используйте «nonmigratory» миграции, при необходимости.  
  
4.  Реализуйте «creative уничтожения».  
  
5.  Изолируйте устаревшие системы и приложений.  
  
6.  Упрощенная защита для конечных пользователей.  
  
### <a name="identifying-principles-for-segregating-and-securing-critical-assets"></a>Определение принципов для разделения и защиты важных активов  

Характеристики образует чистую среду, созданной для дома критически важные ресурсы могут существенно различаться. Например можно создать эталонный лес, в который переносом только пользователям виртуальных IP-адресов и конфиденциальные данные, которые доступны только тем пользователям. Вы можете создать эталонный лес, в который выполняется миграция не только пользователям виртуальных IP-адресов, но который реализовать как административного леса, применению принципов описано в разделе [значительно снизить уязвимость Active Directory](../../../ad-ds/plan/security-best-practices/../../../ad-ds/plan/security-best-practices/Reducing-the-Active-Directory-Attack-Surface.md) для создания безопасного административные учетные записи и узлы, которые можно использовать для управления ваших устаревших лесов в первоначальном состоянии в лесу. Можно реализовать «специально» леса, в котором находятся учетные записи, привилегированные учетные записи и систем, требующих дополнительной защиты, такие как серверы с запущенными службами сертификатов Active Directory (AD CS) с единственной целью отделяя их от менее безопасных виртуальных IP-адресов леса. Наконец можно реализовать эталонный лес, который становится де-факто расположение для всех новых пользователей, систем, приложений и данных, позволяя со временем списания устаревших леса с помощью путем замены устаревшего оборудования.  
  
Вне зависимости от ли эталонный лес содержит небольшое количество пользователей и систем, или он формирует основу для более интенсивное миграции, должны следовать следующим принципам в планировании:  
  
1.  Предположим, взлом ваших лесов прежних версий.  
  
2.  Не настраивайте образует чистую среду на доверие лесу, прежних версий, несмотря на то, что устаревшие среде доверять эталонный лес можно настроить.  
  
3.  Не переносятся учетным записям или группам из прежних версий леса в первоначальном состоянии в среде, если есть вероятность, что учетные записи членства в группах, журнал идентификаторов безопасности и других атрибутов может были изменены злоумышленником. Вместо этого используйте «nonmigratory» подходы для заполнения эталонный лес. (Nonmigratory подходы описаны далее в этом разделе.)  
  
4.  Не Миграция компьютеров из прежних версий лесов в первоначальном состоянии в лесах. Реализовать вновь установленной серверы в первоначальном состоянии в лесу, установка приложений на вновь установленной серверах и перенести данные приложения в только что установленных систем. Для файловых серверов скопировать данные в только что установленных серверов, Настройка списков управления доступом с помощью пользователей и групп в новом лесу и создайте серверов печати таким же образом.  
  
5.  Не допускают установку устаревших операционных систем или приложений в первоначальном состоянии в лесу. Если приложение не может обновить и заново установить, оставьте в лесу прежних версий и рассмотрим creative уничтожения замену функциональных возможностей приложения.  
  
### <a name="defining-a-limited-risk-based-migration-plan"></a>Определение плана миграции ограниченный, на основе рисков  
Создание ограниченный, план миграции на основе риска просто означает, что при определении того, какие пользователи, приложения и данные для миграции в первоначальном состоянии в лесу, необходимо определить цели миграции, в зависимости от степени риска, к которому предоставляется вашей организации, если один из нарушена пользователей или систем. В первоначальном состоянии в лесу должен иметь различное ВАЖНЫХ пользователей, чьи учетные записи являются скорее всего, стали целью злоумышленников. Приложения, предоставляющие важные бизнес-функции должен быть установлен на только что созданных серверов в первоначальном состоянии в лесу, а конфиденциальные данные, которые следует перемещать в защищенных серверов в первоначальном состоянии в лесу.  
  
Если у вас еще нет четкое представление о наиболее важных бизнес-пользователей, систем, приложений и данных в среде Active Directory, работаете с бизнес-подразделения для их идентификации. Любое приложение, необходимых для бизнес-операций должна быть определена так же как и все серверы, на которых запуска критически важных приложений или хранятся важные данные. Определив пользователей и ресурсы, которые требуются для вашей организации, чтобы продолжать функционировать, создается коллекция естественным образом приоритетных ресурсов, на котором следует сосредоточить усилия.  
  
### <a name="leveraging-nonmigratory-migrations"></a>Использование «Nonmigratory» миграции  
Ли вы знаете, что среды был скомпрометирован, подозреваете, он был скомпрометирован, или просто не хотите перенести данных из прежних версий, а также объекты из предыдущих версий установки Active Directory в новую, рассмотрите возможность миграции подходы, которые технически и не выполнять «Миграция» объектов.  
  
### <a name="user-accounts"></a>Ученые записи пользователей  
В традиционной Active Directory миграции из одного леса в другой атрибут SIDHistory (журнал идентификаторов безопасности) объектов-пользователей используется для хранения идентификатора безопасности пользователей и идентификаторы безопасности групп, что пользователи были членами леса прежних версий. Если учетные записи пользователей, переносятся в новый лес, и при доступе к ресурсам в лесу прежних версий, идентификаторы безопасности в журнал идентификаторов безопасности используются для создания маркера доступа, которое позволяет пользователям получать доступ к ресурсам, к которым они имели доступ до учетных записей были перенесены.  
  
Сохранение журнала SID, тем не менее, если все пройдет проблемы в некоторых средах поскольку заполнении маркеры доступа пользователей с идентификаторами безопасности текущие и прошлые может привести Раздувание токена. Токен Раздувание является проблемой, в котором используется номер SID, должны храниться в маркере доступа пользователя или превышает объем свободного пространства в маркере.  
  
Несмотря на то, что размерам маркеров может быть увеличен до некоторой степени, гарантированное решение маркера Раздувание — сократить число идентификаторов безопасности, связанных с учетными записями пользователей, по рационализации членства в группах, устраняя журнал идентификаторов безопасности или их сочетание. Дополнительные сведения о маркера Раздувание, см. в разделе [MaxTokenSize и маркеров Раздувание Kerberos](http://blogs.technet.com/b/shanecothran/archive/2010/07/16/maxtokensize-and-kerberos-token-bloat.aspx).  
  
Вместо перенос пользователей из предыдущих версий среды (особенно при наличии в каких группах и ИД безопасности могут быть скомпрометированы журналы) с помощью журнала SID, рассмотрите использование метакаталог приложения «перевести» пользователей, без выполнения истории SID в новом лесу. При создании учетных записей пользователей в новом лесу метакаталог приложения можно использовать для сопоставления учетных записей их соответствующие учетные записи в лесу прежних версий.  
  
Для предоставления новых учетных записей пользователю доступ к ресурсам в лесу прежних версий, можно использовать метакаталог инструментарий для идентификации группы ресурсов, в которых учетные записи пользователей в устаревшие получит доступ и затем добавлять новые учетные записи пользователей в эти группы. В зависимости от стратегии группы в лесу прежних версий может потребоваться создать локальные группы домена для доступа к ресурсам или преобразовать существующие группы в локальные группы домена, чтобы новые учетные записи могли быть добавлены в группы ресурсов. Сначала сосредоточиться на наиболее важных приложений и данных, и перенести их в новой среде (с или без журнала SID), можно ограничить объем усилий, Потребленная в устаревших среде.  
  

  
### <a name="servers-and-workstations"></a>Серверы и рабочие станции  
В традиционной миграции из одной службы Active Directory леса в другой, миграция компьютеров часто является относительно простой по сравнению с Миграция пользователей, групп и приложений. В зависимости от роли компьютера Миграция в новый лес может быть сложнее, чем исключения из старого домена и присоединение его. Тем не менее Миграция учетных записей компьютеров без изменений в сравнениях эталонный лес целью создания новой среды. Вместо миграции (потенциально скомпрометированные, неправильно настроенные или устаревшей) учетных записей компьютеров в новый лес, необходимо заново установить серверов и рабочих станций в новой среде. Можно перенести данные из системы, в лесу устаревших систем в первоначальном состоянии в лесу, но не системы которых размещаются данные.  
  
### <a name="applications"></a>Приложения  

Приложения могут обеспечивать наиболее сложная задача при любом переходе из одного леса в другой, но в случае с «nonmigratory» миграции, одним из основных принципов, которым должна применяться выполняется, что приложений в первоначальном состоянии в лесу должен быть текущий, поддерживается и только что установлен. Данные могут быть перенесены из экземпляров приложения в старых лесу, где это возможно. В ситуациях, в которых приложение нельзя «воссоздать» в первоначальном состоянии в лесу следует подходы, например creative уничтожения или изоляции старых приложений, как описано в следующем разделе.  
  
### <a name="implementing-creative-destruction"></a>Реализация Creative уничтожения  
Уничтожение Creative — это термин экономики, описанием экономических разработки, созданные деструкция предыдущий порядок. В последние годы термина, примененное к информационных технологий. Обычно он указывает на механизмы, исключается какие старой инфраструктуры, не обновив его, но, заменив его на что-то совершенно новое. 2011 [ITXPO симпозиуме Gartner](http://www.gartner.com/technology/symposium/orlando/) для директоров по информационным технологиям и старшим ИТ-руководители представлены creative уничтожения как один из его ключей темы для снижение затрат и увеличивая эффективность. Усовершенствования в безопасности возможны как естественное outgrowth процесса.  

Например организация может состоять из нескольких филиалов, использующих разных приложение, которое выполняет похожие функции, с различной степенью modernity и поставщика поддержки. Исторически сложилось так, что ИТ нести ответственность за обслуживание каждого филиала компании приложения отдельно, и попытки консолидации заключается в попытке выяснить, какое приложение Предлагаемые рекомендации функциональные возможности и последующим переносом данных, в который приложение от остальных.  
  
В creative уничтожения, а не на обслуживании устаревшие или дублирующих приложений реализации совершенно новых приложений для замены старого, перенесите данные в новых приложениях и списать старые приложения и системы, на которых они запущены. В некоторых случаях вы можете реализовать creative деструкция устаревшие приложения, развертывание нового приложения в вашей собственной инфраструктуры, но везде, где это возможно, следует вместо этого перенос приложения в Облачное решение.  
  
Путем развертывания облачных приложений для замены устаревшего приложения собственной разработки, вы не только сократить объем работ по обслуживанию и затраты, но вы уменьшение уязвимой зоны вашей организации, устраняя устаревших систем и приложений, которые предлагают уязвимостей Злоумышленники могут воспользоваться. Такой подход позволяет быстрее организации для получения нужной функции одновременно устраняя устаревшие целевые объекты в инфраструктуре. Несмотря на то, что принцип creative уничтожения не применяются ко всем ресурсам ИТ, он предоставляет часто приемлемым вариантом в устранение устаревших систем и приложений при одновременном развертывании надежных, безопасных и облачных приложений.  
  
### <a name="isolating-legacy-systems-and-applications"></a>Изоляция прежних версий систем и приложений  
Естественное outgrowth перенос критически важных бизнес-пользователям и компьютерам в первоначальном состоянии, безопасной среде — наличие лесу прежних версий менее ценную информацию и систем. Несмотря на то, что устаревшие системы и приложений, которые остаются в менее защищенной среде могут присутствовать повышенный риск компрометации, они также представляют ограниченной серьезность компрометации. Rehoming и модернизации важных ресурсов, можно сосредоточиться на развертывание эффективное управление и мониторинг при отсутствии необходимости в соответствии с предыдущих версий параметров и протоколов.  
  
Когда переместить важные данные в первоначальном состоянии в лес, вы можете оценить параметры для дальнейшей изоляции устаревших систем и приложений в «main» леса AD DS. Несмотря на то, что можно реализовать creative уничтожения для замены одного приложения и серверы, на которых выполняется, в других случаях может потребоваться дополнительная изоляция наименее безопасный систем и приложений. Например приложение, используемое небольшое число пользователей, но что требует, что устаревших учетных данных, например хеш-значения LAN Manager могут быть перенесены в небольшой домен создается для поддержки систем, для которых у вас есть без параметров замены.  
  
При удалении этих систем из доменов, где они принудительно реализация устаревшие параметры, можно впоследствии повысить безопасность доменов, настраивая их для поддержки только текущего операционных систем и приложений. Хотя желательно списания устаревших систем и приложений, когда это возможно. Если списание это просто невозможно в небольшой сегмент вашей заполнение прежних версий, отделяя его в отдельный домен (или леса) позволяет выполнять несколько дополнительных усовершенствований в остальной части старую установку.  
  
### <a name="simplifying-security-for-end-users"></a>Упрощение безопасности для конечных пользователей  
В большинстве организаций пользователи, имеющие доступ к наиболее важных данных из-за характера их роли в организации часто имеют как минимум количество времени уделить обучения ограничения сложных доступа и элементы управления. Несмотря на то, что следует установлена программа для образовательных учреждений полнофункциональных средств безопасности для всех пользователей в вашей организации, вам также следует сосредоточиться на безопасности, как просто использовать как можно путем реализации элементов управления, которые являются субъектами прозрачным и упрощения процесса, к которой пользователи соблюдать.  
  
Например можно определить политику, в котором руководителей и других виртуальных IP-адресов требуются использовать безопасных рабочих станций для доступа к конфиденциальным данным и системам, позволяя им использовать другие устройства для доступа к менее конфиденциальным данным. Это простой принцип запомнить, но вы можете реализовать ряд элементов управления серверной части, чтобы применить этот подход.  

Можно использовать [Контроль механизма проверки подлинности](https://technet.microsoft.com/library/dd391847(v=WS.10).aspx) чтобы разрешить пользователям доступ к конфиденциальным данным только в том случае, если они вошел их безопасных систем, с помощью смарт-карты и можно использовать протокол IPsec и ограничения для управления правами пользователей системы, из которых они могут подключаться к репозиториям конфиденциальных данных. Можно использовать [Microsoft Data Classification Toolkit](https://www.microsoft.com/download/details.aspx?id=27123) создавать инфраструктуру классификации надежный файл и можно реализовать [динамического контроля доступа](http://blogs.technet.com/b/windowsserver/archive/2012/05/22/introduction-to-windows-server-2012-dynamic-access-control.aspx) для ограничения доступа к базе данных характеристики для попытки доступа, переводя бизнес-правила в технических средств.  
  
С точки зрения пользователя, к конфиденциальным данным из защищенной системе «просто работает» и попытке сделать это из незащищенных системы «просто не». Однако с точки зрения мониторинга и управления средой, вы поможете создать явные закономерности в доступе пользователей к конфиденциальным данным и системам, упрощая обнаружение подозрительной активности в доступе.  
  
В средах, в которых пользователь устойчивости к длинные сложные пароли привела к политики паролей недостаточно, особенно для пользователей виртуальных IP-адресов, рассмотрим альтернативные подходы к проверке подлинности, с помощью смарт-карт (который поставляется в нескольких форм-факторов и с дополнительными возможностями по повышению эффективности системы проверки подлинности) биометрические элементы управления, такие как читатели проведите пальцем или даже к проверке подлинности данных, защиту которого выполняет доверенных микросхемы платформенный модуль (TPM) на компьютерах пользователей. Несмотря на то, что многофакторная проверка подлинности не препятствует атак кражи учетных данных компьютера при уже компрометации, предоставляя пользователям элементы управления для использования проверки подлинности, можно назначить более надежные пароли учетных записей пользователей, для которого традиционных пользователя элементы управления имя и пароль являются громоздким.
