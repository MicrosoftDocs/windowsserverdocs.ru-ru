# <a name="identifying-the-deployment-project-participants"></a>Определение участников проекта развертывания

> Область применения. Windows Server 2016, Windows Server 2012 R2, Windows Server 2012

Первым шагом в установке проекта развертывания для службы домен Active Directory (AD DS) является создание групп проектов разработки и развертывания, которые будут отвечать за управление этапом проектирования и этапом развертывания Active Directory цикла проекта. Кроме того, необходимо указать пользователей и группы, которые будут отвечать за владельца и обслуживание каталога после завершения развертывания.

- [Определение ролей для конкретного проекта](#BKMK_1)

- [Установка владельцев и администраторов](#BKMK_2)

- [Создание проектных групп](#BKMK_3)

## <a name="defining-project-specific-roles"></a><a name="BKMK_1"></a>Определение ролей для конкретного проекта
Важным шагом при создании групп проекта является определение отдельных лиц, которые должны содержать роли конкретного проекта. К ним относятся спонсор руководителя, архитектор проекта и руководитель проекта. Эти лица отвечают за запуск проекта развертывания Active Directory.

После даете право архитектора проекта и руководителя проекта эти пользователи устанавливают каналы связи в Организации, создают расписания проектов и выдают пользователей, которые будут членами групп проекта, начиная с разных владельцев.

### <a name="executive-sponsor"></a>Спонсор, ответственный за
Развертывание инфраструктуры, например AD DS, может повлиять на работу Организации в широком спектре. По этой причине важно иметь спонсора руководителя, который понимает ценность развертывания, поддерживает проект на уровне руководителя и может помочь в устранении конфликтов в Организации.

### <a name="project-architect"></a>Архитектор проекта
Каждый проект развертывания Active Directory требует архитектора проектов для управления Active Directory процесса принятия решений по проектированию и развертыванию. Архитектор предоставляет технический опыт для помощи с процессом проектирования и развертывания AD DS.

> [!NOTE]
> Если ни один из сотрудников вашей организации не имеет опыта работы с каталогами, вы можете нанять внешнего консультанта, который является экспертом в Active Directory проектировании и развертывании.

Функциональные обязанности архитектора проекта Active Directory включают следующее:

- Владелец Active Directory проектирования

- Понимание и запись основания для ключевых решений по проектированию

- Обеспечение соответствия проекта потребностям бизнес-потребностей Организации

- Организация взаимоотношений между группами проектирования, развертывания и эксплуатации

- Основные сведения о потребностях приложений, интегрированных с AD DS

Окончательная Active Directoryная структура должна отражать сочетание бизнес-целей и технических решений. Таким образом, архитектор проекта должен проверить проектные решения, чтобы обеспечить соответствие бизнес-целям.

### <a name="project-manager"></a>руководитель проекта;
Руководитель проекта упрощает взаимодействие между бизнес-подразделениями и группами управления технологическими технологиями. В идеале Диспетчер проектов Active Directoryного развертывания — это кто-то из Организации, который знаком с операционными политиками ИТ-группы и требованиями к проектированию для групп, подготавливающих развертывание AD DS. Руководитель проекта видит весь проект развертывания, начиная с разработки и заканчивая реализацией, и гарантирует, что проект остается в расписании и в пределах бюджета. К руководителям проектов относятся следующие обязанности:

- Базовое планирование проекта, например планирование и бюджетирование

- Ход выполнения проекта Active Directory проект проектирования и развертывания

- Обеспечение участия соответствующих лиц в каждой части процесса разработки

- Обслуживание в качестве единой точки контакта для проекта развертывания Active Directory

- Установка связи между группами проектирования, развертывания и эксплуатации

- Установка и обслуживание взаимодействия с руководителем в рамках проекта развертывания

## <a name="establishing-owners-and-administrators"></a><a name="BKMK_2"></a>Установка владельцев и администраторов
В проекте развертывания Active Directory сотрудники, являющиеся владельцами, удерживаются в управлении, чтобы убедиться в том, что задачи развертывания завершены и Active Directory спецификации проектирования соответствуют потребностям Организации. Владельцы не обязательно имеют прямой доступ к инфраструктуре каталогов и не управляют ею. Администраторы являются сотрудниками, ответственными за выполнение необходимых задач развертывания. Администраторы имеют доступ к сети и разрешения, необходимые для работы с каталогом и его инфраструктурой.

Роль владельца является стратегическим и руководителем. Владельцы ответственны за взаимодействие с администраторами задач, необходимых для реализации Active Directory проектирования, например создания новых контроллеров домена в пределах леса. Администраторы несут ответственность за реализацию проекта в сети в соответствии с требованиями к проектированию.

В больших организациях разные пользователи заполняют роли владельца и администратора. Однако в некоторых небольших организациях один и тот же человек может действовать как владелец, так и администратор.

### <a name="service-and-data-owners"></a>Службы и владельцы данных
Управление AD DS на ежедневной основе состоит из двух типов владельцев:

- Владельцы служб, ответственные за планирование и долгосрочное обслуживание Active Directory инфраструктуры, а также гарантируют, что каталог продолжит функционировать, и что цели, установленные в соглашениях об уровне обслуживания, сохраняются.
- Владельцы данных, ответственные за обслуживание информации, хранящейся в каталоге. Сюда входят Управление учетными записями пользователей и компьютеров, а также управление локальными ресурсами, такими как рядовые серверы и рабочие станции.

Важно заранее определить службу Active Directory и владельцев данных, чтобы они могли участвовать в максимальном процессе разработки. Так как служба и владельцы данных отвечают за долгосрочное обслуживание каталога после завершения проекта развертывания, важно, чтобы эти пользователи предоставили входные данные в соответствии с потребностями Организации и знакомы с тем, как и почему выполняются определенные решения по проектированию. Владельцы служб включают в себя владельца леса, владельца системы именования домен Active Directory (DNS) и владельца топологии сайта. Владельцы данных включают в себя владельцы подразделения.

### <a name="service-and-data-administrators"></a>Администраторы служб и данных
Операция AD DS включает в себя двух типов администраторов: администраторов служб и администраторов данных. Администраторы служб реализуют решения по политике, принятые владельцами служб, и выполняют повседневные задачи, связанные с обслуживанием службы каталогов и инфраструктуры. Сюда входит управление контроллерами домена, на которых размещена служба каталогов, управление другими сетевыми службами, такими как DNS, которые необходимы для AD DS, контроль конфигурации параметров на уровне леса и обеспечение постоянного доступа к каталогу.

Администраторы служб также отвечают за выполнение текущих задач Active Directory развертывания, которые необходимы после завершения первоначального процесса развертывания Windows Server 2008 Active Directory. Например, при увеличении требований к каталогу администраторы служб создают дополнительные контроллеры домена и устанавливают или удаляют отношения доверия между доменами по мере необходимости. По этой причине Группа развертывания Active Directory должна включать администраторов служб.

Необходимо соблюдать осторожность при назначении ролей администратора службы только доверенным лицам в Организации. Поскольку эти пользователи имеют возможность изменять системные файлы на контроллерах домена, они могут изменить поведение AD DS. Необходимо убедиться, что администраторы служб в вашей организации являются людьми, знакомыми с политиками эксплуатации и безопасности в сети и которые понимают необходимость применения этих политик.

Администраторы данных — это пользователи в домене, которые отвечают за поддержание данных, хранящихся в AD DS, таких как учетные записи пользователей и групп, а также для обслуживания компьютеров, входящих в их домен. Администраторы данных управляют подмножествами объектов в каталоге и не контролируют установку или настройку службы каталогов.

Учетные записи администратора данных по умолчанию не предоставляются. После того как группа разработки определит, как ресурсы будут управляться для Организации, владельцы домена должны создать учетные записи администратора данных и делегировать им соответствующие разрешения на основе набора объектов, которым должны отвечать Администраторы.

Лучше ограничить количество администраторов служб в Организации минимальным числом, необходимым для обеспечения работы инфраструктуры. Большинство административных задач может быть выполнено администраторами данных. Администраторам служб требуется более широкий набор навыков, так как они отвечают за поддержание каталога и инфраструктуры, которая его поддерживает. Администраторам данных требуются только навыки, необходимые для управления их частью каталога. Деление рабочих назначений таким образом приводит к экономии средств для Организации, так как только небольшое количество администраторов должно быть обработано и обслуживать весь каталог и его инфраструктуру.

Например, администратор службы должен понять, как добавить домен в лес. Сюда входит установка программного обеспечения для преобразования сервера в контроллер домена и управления средой DNS, чтобы контроллер домена можно было легко объединить в среду Active Directory. Администратору данных необходимо только уметь управлять конкретными данными, за которые они отвечают, например созданием новых учетных записей пользователей для новых сотрудников в отделе.

Для развертывания AD DS требуется координация и взаимодействие между многими различными группами, вовлеченными в работу сетевой инфраструктуры. Эти группы должны даете право службы и владельцы данных, ответственные за представление различных групп в процессе проектирования и развертывания.

После завершения проекта развертывания эти владельцы служб и данных будут по-прежнему отвечать за часть инфраструктуры, управляемой их группой. В среде Active Directory эти владельцы являются владельцами леса, DNS для AD DS владельца, владельцем топологии сайта и владельцем подразделения. Роли этих служб и владельцев данных описаны в следующих разделах.

#### <a name="forest-owner"></a>Владелец леса
Владелец леса, как правило, является старшим ИТ-менеджером в Организации, ответственным за процесс развертывания Active Directory и который в конечном счете отвечает за поддержание предоставления услуг в лесу после завершения развертывания. Владелец леса назначает пользователей для заполнения других ролей владения, определяя в организации ключевые сотрудники, которые могут принимать необходимые сведения о сетевой инфраструктуре и административных нуждах. Владелец леса несет ответственность за следующее:

- Развертывание корневого домена леса для создания леса

- Развертывание первого контроллера домена в каждом домене для создания доменов, необходимых для леса;

- Членство в группах администраторов службы во всех доменах леса

- Создание структуры подразделений для каждого домена в лесу

- Делегирование административного центра владельцам подразделения

- Изменения схемы

- Изменения в параметрах конфигурации на уровне леса

- Реализация определенных параметров политики групповая политика, включая политики учетных записей пользователей домена, такие как детально детализированные пароли и политики блокировки учетных записей.

- Параметры бизнес-политики, применяемые к контроллерам домена

- Любые другие параметры групповая политика, применяемые на уровне домена

Владелец леса имеет полномочия на весь лес. Ответственность за установку групповая политика и бизнес-политик и выбор лиц, являющихся администраторами служб, несет владелец леса. Владелец леса является владельцем службы.

#### <a name="dns-for-ad-ds-owner"></a>DNS для владельца AD DS
DNS для владельца AD DS — это человек, который хорошо знаком с существующей инфраструктурой DNS и существующим пространством имен организации.

Владелец DNS для AD DS несет ответственность за следующее:

- В качестве связи между группой проектирования и ИТ-группой, которая в настоящее время владеет инфраструктурой DNS

- Предоставление сведений о существующем пространстве имен DNS организации для помощи в создании нового пространства имен Active Directory

- Работа с группой развертывания, чтобы убедиться, что новая инфраструктура DNS развернута в соответствии с техническими характеристиками группы проектирования и правильно работает.

- Управление DNS для инфраструктуры AD DS, включая службу DNS-сервера и данные DNS

DNS-владелец AD DS является владельцем службы.

#### <a name="site-topology-owner"></a>Владелец топологии сайта
Владелец топологии сайта знаком с физической структурой сети организации, включая сопоставление отдельных подсетей, маршрутизаторов и сетевых областей, подключенных с помощью медленных каналов. Владелец топологии сайта несет ответственность за следующее:

- Основные сведения о топологии физической сети и ее влиянии на AD DS

- Понимание того, как развертывание Active Directory влияет на сеть

- Определение Active Directory логических сайтов, которые необходимо создать

- Обновление объектов сайта для контроллеров домена при добавлении, изменении или удалении подсети

- Создание связей сайтов, мостов связей сайтов и объектов подключения вручную

Владелец топологии сайта является владельцем службы.

#### <a name="ou-owner"></a>Владелец подразделения
Владелец подразделения отвечает за управление данными, хранящимися в каталоге. Этот человек должен быть знаком с политиками эксплуатации и безопасности, которые используются в сети. Владельцы подразделений могут выполнять только те задачи, которые были делегированы им администраторы служб, и могут выполнять только те задачи в подразделениях, к которым они назначены. К задачам, которые могут быть назначены владельцу подразделения, относятся следующие:

- Выполнение всех задач управления учетными записями в назначенном подразделении

- Управление рабочими станциями и рядовыми серверами, входящими в назначенное подразделение

- Делегирование полномочий локальным администраторам в назначенном подразделении

Владелец подразделения является владельцем данных.

## <a name="building-project-teams"></a><a name="BKMK_3"></a>Создание проектных групп
Active Directory группы проектов являются временными группами, ответственными за выполнение Active Directory задач проектирования и развертывания. Когда проект развертывания Active Directory будет завершен, владельцы должны быть ответственными за каталог, и группы проектов могут дисбанд.

Размер групп проекта зависит от размера организации. В небольших организациях один человек может охватывать несколько областей ответственности в группе проекта и участвовать в нескольких стадиях развертывания. Крупные организации могут потребовать более крупных команд с разными людьми или даже разными группами, охватывающими различные области ответственности. Размер команд не имеет значения при условии, что назначены все области ответственности и выполняются цели разработки Организации.

### <a name="identifying-potential-forest-owners"></a>Определение потенциальных владельцев леса
Укажите группы в Организации, которые владеют ресурсами и контролируют ресурсы, необходимые для предоставления служб каталога пользователям в сети. Эти группы считаются потенциальными владельцами леса.

Разделение служб и администрирования данных в AD DS делает возможным ИТ-группу (или группы) инфраструктуры организации управлять службой каталогов, а локальные администраторы в каждой группе — управлять данными, принадлежащими их собственным группам. Потенциальные владельцы леса имеют необходимые полномочия для развертывания и поддержки AD DS в сетевой инфраструктуре.

Для организаций, имеющих одну централизованную ИТ-группу инфраструктуры, Группа ИТ обычно является владельцем леса и, следовательно, потенциальным владельцем леса для любых будущих развертываний. Организации, включающие несколько независимых ИТ-групп, имеют несколько потенциальных владельцев леса. Если в вашей организации уже есть инфраструктура Active Directory, все владельцы текущего леса также могут быть владельцами леса для новых развертываний.

Выберите одного из потенциальных владельцев леса, которые будут использоваться в качестве владельца леса для каждого леса, который вы собираетесь развернуть. Эти потенциальные владельцы леса отвечают за работу с группой проектирования, чтобы определить, будет ли их лес развернут или если вы еще не развернетесь в другом варианте действий (например, присоединиться к другому существующему лесу). Владелец леса (или владельцы) в вашей организации является членом группы разработки Active Directory.

### <a name="establishing-a-design-team"></a>Создание группы разработки
Группа разработки Active Directory отвечает за сбор всех сведений, необходимых для принятия решений о структуре Active Directory логической структуры.

Обязанности группы проектирования включают следующее:

- Определение требуемого количества лесов и доменов и отношений между лесами и доменами

- Работа с владельцами данных для обеспечения соответствия проекта требованиям к безопасности и администрированию

- Работа с текущими сетевыми администраторами позволяет гарантировать, что текущая сетевая инфраструктура поддерживает структуру и что проект не будет негативно влиять на существующие приложения, развернутые в сети.

- Работа с представителями группы безопасности Организации, чтобы убедиться, что проект соответствует установленным политикам безопасности.

- Проектирование структур подразделений, которые допускают соответствующие уровни защиты и право делегирования полномочий владельцам данных.

- Работа с группой развертывания для тестирования проекта в лабораторной среде, чтобы убедиться, что она работает как запланированная и изменяет проект по мере необходимости для устранения всех возникающих проблем.

- Создание схемы топологии сайта, удовлетворяющей требованиям репликации леса, и предотвращение перегрузки доступной пропускной способности. Дополнительные сведения о проектировании топологии сайтов см. [в статье Разработка топологии сайтов для Windows Server 2008 AD DS](/previous-versions/windows/it-pro/windows-server-2008-R2-and-2008/cc772013(v=ws.10)).

- Работа с группой развертывания, чтобы обеспечить правильную реализацию проекта

Группа разработки включает следующие члены:

- Потенциальные владельцы леса

- Архитектор проекта

- руководитель проекта;

- Пользователи, ответственные за установку и обслуживание политик безопасности в сети

В процессе проектирования логической структуры Группа разработки определяет других владельцев. Эти лица должны приступить к участию в процессе разработки, как только они будут идентифицированы. После того как проект развертывания передается группе развертывания, группа разработчиков несет ответственность за процесс развертывания, чтобы обеспечить правильную реализацию проекта. Группа разработки также вносит изменения в проект на основе отзывов о тестировании.

### <a name="establishing-a-deployment-team"></a>Установка группы развертывания
Группа развертывания Active Directory отвечает за тестирование и реализацию структуры Active Directory логической структуры. Это включает следующие задачи:

- Установка тестовой среды, которая достаточно эмулирует рабочую среду

- Тестирование проекта путем реализации предлагаемой структуры леса и домена в лабораторной среде, чтобы убедиться, что она соответствует целям каждого владельца роли.

- Разработка и тестирование сценариев миграции, предлагаемых конструкцией в лабораторной среде

- Проверка того, что каждый владелец подключается к процессу тестирования, чтобы убедиться в тестировании правильных функций проектирования.

- Тестирование операции развертывания в пилотной среде

По завершении задач проектирования и тестирования группа развертывания выполняет следующие задачи:

- Создает леса и домены в соответствии с Active Directory структурой логической структуры.

- Создает сайты и объекты связи сайтов по мере необходимости в зависимости от структуры топологии сайта.

- Гарантирует, что инфраструктура DNS настроена для поддержки AD DS и все новые пространства имен будут интегрированы в существующее пространство имен организации.

Группа развертывания Active Directory включает следующие члены:

- Владелец леса

- DNS для владельца AD DS

- Владелец топологии сайта

- Владельцы подразделения

Группа развертывания работает с администраторами служб и данных на этапе развертывания, чтобы члены Рабочей группы были знакомы с новой структурой. Это помогает обеспечить плавный переход владения по завершении операции развертывания. По завершении процесса развертывания ответственность за обслуживание новой Active Directoryной среды передается группе операций.

### <a name="documenting-the-design-and-deployment-teams"></a>Документирование команд проектирования и развертывания
Задокументируйте имена и контактные данные для людей, которые будут участвовать в проектировании и развертывании AD DS. Выясните, кто будет отвечать за каждую роль в группах проектирования и развертывания. Изначально в этот список входят потенциальные владельцы леса, руководитель проекта и архитектор проекта. При определении количества лесов, которые планируется развернуть, может потребоваться создать новые группы разработки для дополнительных лесов. Обратите внимание, что вам потребуется обновить документацию по мере изменения членства в группе и при определении различных владельцев Active Directory в процессе разработки. Чтобы помочь вам в документировании групп разработки и развертывания для каждого леса, скачайте Job_Aids_Designing_and_Deploying_Directory_and_Security_Services.zip из [комплекта вспомогательных заданий для пакета развертывания Windows Server 2003](https://microsoft.com/download/details.aspx?id=9608) и откройте «сведения о команде проектирования и развертывания» (DSSLOGI_1.doc).
