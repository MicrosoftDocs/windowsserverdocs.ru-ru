# <a name="disjoint-namespace"></a>Несвязанное пространство имен

>Область применения: Windows Server 2016, Windows Server 2012 R2, Windows Server 2012

Несвязанное пространство имен возникает, когда один или несколько компьютеров-членов домена имеют основной суффикс службы доменных имен (DNS), не совпадающий с DNS-именем домена Active Directory, членами которого являются компьютеры. Например, компьютер-член, использующий первичный суффикс DNS corp.fabrikam.com в домене Active Directory с именем na.corp.fabrikam.com, использует несвязанное пространство имен.  
  
Несвязанное пространство имен более сложно администрировать, обслуживать и устранять неполадки, чем непрерывное пространство имен. В смежном пространстве имен основной DNS-суффикс соответствует Active Directory доменному имени. Сетевые приложения, написанные для предположения, что пространство имен Active Directory идентично основному DNS-суффиксу для всех компьютеров, являющихся членами домена, не функционирует надлежащим образом в несвязанном пространстве имен.  
  
## <a name="support-for-disjoint-namespaces"></a>Поддержка несоединений пространств имен  
Компьютеры-члены домена, включая контроллеры домена, могут работать в несвязанном пространстве имен. Компьютеры, являющиеся членами домена, могут регистрировать свои записи ресурсов узла (A) и узла IP версии 6 (IPv6) в несвязанном пространстве имен DNS. Когда компьютеры-члены домена регистрируют свои записи ресурсов таким образом, контроллеры домена продолжают регистрировать глобальные и специфические записи ресурсов (SRV) в зоне DNS, идентичной доменному имени Active Directory.  
  
Например, предположим, что контроллер домена для Active Directory домена с именем na.corp.fabrikam.com, который использует основной DNS-суффикс corp.fabrikam.com, регистрирует записи ресурсов узла (A) и IPv6 (AAAA) в зоне corp.fabrikam.com DNS. Контроллер домена в зонах DNS _msdcs. Na. Corp. fabrikam. com и na.corp.fabrikam.com может регистрировать глобальные записи ресурсов и службы конкретных сайтов (SRV), что делает возможным размещение служб.  
  
> [!IMPORTANT]  
> Хотя операционные системы Windows могут поддерживать несвязанное пространство имен, приложения, которые записываются, предполагают, что основной DNS-суффикс совпадает с суффиксом домена Active Directory может не функционировать в такой среде. По этой причине следует тщательно протестировать все приложения и соответствующие операционные системы перед развертыванием несвязанного пространства имен.  
  
Несвязанное пространство имен должно работать (и поддерживается) в следующих ситуациях:  
  
-   Если лес с несколькими Active Directory доменами использует одно пространство имен DNS, которое также называется зоной DNS  
  
    Например, компания использует региональные домены с такими именами, как na.corp.fabrikam.com, sa.corp.fabrikam.com и asia.corp.fabrikam.com, и использует одно пространство имен DNS, например corp.fabrikam.com.  
  
-   При разбиении одного Active Directory домена на отдельные пространства имен DNS  
  
    Примером этого является организация с доменом Active Directory corp.contoso.com, в котором используются зоны DNS, такие как hr.corp.contoso.com, production.corp.contoso.com и it.corp.contoso.com.  
  
Несвязанное пространство имен работает неправильно (и не поддерживается) в следующих ситуациях:  
  
-   Несвязанный суффикс, используемый членами домена, соответствует доменному имени Active Directory в этом или другом лесу. Это нарушает маршрутизацию имен Kerberos.  
  
-   В другом лесу используется один и тот же несоединенный суффикс. Это предотвращает уникальную маршрутизацию этих суффиксов между лесами.  
  
-   Когда сервер центра сертификации (ЦС) изменяет полное доменное имя (FQDN), чтобы оно больше не использовало тот же основной DNS-суффикс, который используется контроллерами домена домена, членом которого является сервер ЦС. В этом случае могут возникнуть проблемы с проверкой сертификатов, выданных сервером ЦС, в зависимости от того, какие DNS-имена используются в точках распространения списков отзыва сертификатов. Но если сервер ЦС размещен в стабильном несвязанном пространстве имен, он работает правильно и поддерживается.  
  
## <a name="considerations-for-disjoint-namespaces"></a>Рекомендации по несвязанным пространствам имен  
Следующие рекомендации могут помочь решить, следует ли использовать несвязанное пространство имен.  
  
### <a name="application-compatibility"></a>Совместимость приложений  
Как уже упоминалось, несвязанное пространство имен может вызвать проблемы для любых приложений и служб, которые записываются, чтобы предположить, что основной DNS-суффикс компьютера совпадает с именем домена, членом которого он является. Перед развертыванием несвязанного пространства имен необходимо проверить приложения на наличие проблем совместимости. Кроме того, обязательно проверьте совместимость всех приложений, используемых при выполнении анализа. Сюда входят приложения корпорации Майкрософт и других разработчиков программного обеспечения.  
  
### <a name="advantages-of-disjoint-namespaces"></a>Преимущества присоединяемых пространств имен  
Использование несвязанного пространства имен может иметь следующие преимущества.  
  
-   Поскольку основной DNS-суффикс компьютера может указывать на разные сведения, можно управлять пространством имен DNS отдельно от Active Directory доменного имени.  
  
-   Пространство имен DNS можно разделить на основе бизнес-структуры или географического расположения. Например, можно разделить пространство имен на основе имен бизнес-единиц или физического расположения, например континента, страны или региона или здания.  
  
### <a name="disadvantages-of-disjoint-namespaces"></a>Недостатки взаимосвязей пространств имен  
Использование несвязанного пространства имен может иметь следующие недостатки.  
  
-   Необходимо создать отдельные зоны DNS и управлять ими для каждого Active Directory домена в лесу, который имеет компьютеры-члены, использующие несвязанное пространство имен. (То есть требуется дополнительная и более сложная конфигурация.)  
  
-   Необходимо выполнить действия вручную, чтобы изменить атрибут Active Directory, который позволяет членам домена использовать указанные основные DNS-суффиксы, и управлять им.  
  
-   Чтобы оптимизировать разрешение имен, необходимо вручную выполнить действия по изменению и обслуживанию групповая политика для настройки рядовых компьютеров с дополнительными основными DNS-суффиксами.  
  
    > [!NOTE]  
    > Службу Windows Internet Name Service (WINS) можно использовать для смещения этого недостатка путем разрешения имен с одной меткой. Дополнительные сведения о WINS см. в техническом справочнике по WINS ([https://go.microsoft.com/fwlink/?LinkId=102303](https://go.microsoft.com/fwlink/?LinkId=102303)).  
  
-   Если в среде требуется несколько основных DNS-суффиксов, необходимо соответствующим образом настроить порядок поиска DNS-суффиксов для всех Active Directory доменов в лесу.  
  
    Чтобы задать порядок поиска DNS-суффикса, можно использовать групповая политика объекты или параметры службы DHCP-сервера. Также можно изменить реестр.  
  
-   Необходимо тщательно протестировать все приложения на наличие проблем совместимости.  
  
Дополнительные сведения о действиях, которые можно предпринять для устранения этих недостатков, см. в разделе Создание несвязанного пространства имен ([https://go.microsoft.com/fwlink/?LinkId=106638](https://go.microsoft.com/fwlink/?LinkId=106638)).  
  
### <a name="planning-a-namespace-transition"></a>Планирование перехода на пространство имен  
Перед изменением пространства имен ознакомьтесь со следующими соображениями, которые применяются к переходам из смежных пространств имен в несвязанные пространства имен (или обратно):  
  
-   После изменения пространства имен имена участников-служб, настроенные вручную, могут больше не совпадать с именами DNS. Это может вызвать сбои проверки подлинности.  
  
    Дополнительные сведения см. в статье сбой входа в службу из-за неправильной установки имени участника-службы ([https://go.microsoft.com/fwlink/?LinkId=102304](https://go.microsoft.com/fwlink/?LinkId=102304)).  
  
    -   При использовании компьютеров на базе Windows Server 2003 с ограниченным делегированием эти компьютеры могут потребовать дополнительной настройки для изменения имен участников-служб. Дополнительные сведения см. в статье 936628 базы знаний Майкрософт ([https://go.microsoft.com/fwlink/?LinkId=102306](https://go.microsoft.com/fwlink/?LinkId=102306)).  
  
    -   Если вы хотите делегировать разрешения на изменение имен участников-служб в подчиненных администраторах, см. раздел делегирование полномочий для изменения имен SPN ([https://go.microsoft.com/fwlink/?LinkId=106639](https://go.microsoft.com/fwlink/?LinkId=106639)).  
  
-   Если вы используете протокол LDAP по SSL (т. е. LDAPs) с центром сертификации в развертывании с контроллерами домена, настроенными в несвязанном пространстве имен, при настройке сертификатов LDAPs необходимо использовать соответствующие доменные имена Active Directory и основной DNS-суффикс.  
  
    Дополнительные сведения о требованиях к сертификатам контроллера домена см. в статье 321051 базы знаний Майкрософт ([https://go.microsoft.com/fwlink/?LinkId=102307](https://go.microsoft.com/fwlink/?LinkId=102307)).  
  
    > [!NOTE]  
    > Контроллеры домена, использующие сертификаты для LDAPs, могут потребовать повторного развертывания сертификатов. После этого контроллеры домена не смогут выбрать соответствующий сертификат, пока они не будут перезапущены. Дополнительные сведения о проверке подлинности LDAPs и соответствующем обновлении для Windows Server 2003 см. в статье 932834 в базе знаний Майкрософт ([https://go.microsoft.com/fwlink/?LinkId=102308](https://go.microsoft.com/fwlink/?LinkId=102308)).  
  
### <a name="planning-for-disjoint-namespace-deployments"></a>Планирование развертываний несвязанного пространства имен  
При развертывании компьютеров в среде с несвязанным пространством имен примите следующие меры предосторожности:  
  
1.  Уведомлять всех поставщиков программного обеспечения с кем-то бизнес, что они должны тестировать и поддерживать несвязанное пространство имен. Попросите их убедиться, что они поддерживают приложения в средах, использующих несвязанные пространства имен.  
  
2.  Протестируйте все версии операционных систем и приложений в лабораторных средах несвязанного пространства имен. При этом следуйте приведенным ниже рекомендациям.  
  
    1.  Устраните все проблемы с программным обеспечением перед развертыванием программного обеспечения в своей среде.  
  
    2.  По возможности примите участие в тестировании бета-версий операционных систем и приложений, которые планируется развернуть в несвязанных пространствах имен.  
  
3.  Убедитесь, что администраторы и сотрудники службы поддержки знают о несвязанном пространстве имен и его влиянии.  
  
4.  Создайте план, позволяющий переходить от несвязанного пространства имен к смежному пространству имен, если это необходимо.  
  
5.  Пропаганды важность поддержки несвязанного пространства имен с поставщиками операционной системы и приложений.
