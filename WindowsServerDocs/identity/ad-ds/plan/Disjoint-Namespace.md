# <a name="disjoint-namespace"></a>Несвязанное пространство имен.

>Область применения: Windows Server 2016, Windows Server 2012 R2, Windows Server 2012

Несвязанное пространство имен возникает, когда один или несколько члены домена имеют основной суффикс службы доменных имен (DNS), которое не соответствует DNS-имя которого компьютеры являются членами домена Active Directory. Например компьютер, использующий основной DNS-суффикс corp.fabrikam.com в домене Active Directory, с именем na.corp.fabrikam.com использует несвязанного пространства имен.  
  
Несвязанное пространство имен является более сложной для администрирования, обслуживания и устранение неполадок чем непрерывного пространства имен. В непрерывного пространства имен основной DNS-суффикс совпадает с именем домена Active Directory. Сетевых приложений, которые записываются предположить, что пространство имен Active Directory идентична основного суффикса DNS для всех компьютеров домена неправильно работают в несвязанное пространство имен.  
  
## <a name="support-for-disjoint-namespaces"></a>Поддержка несвязанного пространства имен  
Член домена, включая контроллеры домена могут работать в несвязанное пространство имен. Член домена можно зарегистрировать их узла (A) записи ресурса и протокола IP версии 6 (IPv6) узла (AAAA) в записи ресурса в несвязанное пространство имен DNS. Когда компьютерах-членах домена регистрируют свои записи ресурсов таким образом, контроллеры домена продолжают регистрации записей ресурсов глобального и отдельных сайтов службы (SRV) в зоне DNS, которая соответствует имени домена Active Directory.  
  
Например предположим, что контроллер домена для домена Active Directory, с именем na.corp.fabrikam.com, использующий основной DNS-суффикс corp.fabrikam.com регистрирует узла (A) и IPv6 записи ресурсов узла (AAAA) в зоне DNS corp.fabrikam.com. Контроллер домена продолжает регистрации записей ресурсов глобального и отдельных сайтов службы (SRV) в _msdcs. na.corp.fabrikam.com и na.corp.fabrikam.com зонах DNS, которая делает возможным размещение службы.  
  
> [!IMPORTANT]  
> Несмотря на то, что операционных систем Windows может поддерживать несвязанного пространства имен, приложения, написанные предположить, что основной DNS-суффикс совпадает с суффиксом домена Active Directory не может работать в такой среде. По этой причине необходимо проверить все приложения и соответствующих операционных систем тщательно перед развертыванием несвязанного пространства имен.  
  
Несвязанное пространство имен должны работать (и поддерживается) в следующих ситуациях:  
  
-   Если в лесу с несколькими доменами Active Directory используется одно пространство имен DNS-сервера, который также называется зоны DNS  
  
    Примером этого — это компания, которое использует региональных доменов с именами, такие как na.corp.fabrikam.com и sa.corp.fabrikam.com, asia.corp.fabrikam.com и использует одно пространство имен DNS, например corp.fabrikam.com.  
  
-   Когда один домен Active Directory разделяется на отдельные пространства имен DNS  
  
    Пример этого — это компания, в домене Active Directory, использующий зон DNS, такие как hr.corp.contoso.com и production.corp.contoso.com, it.corp.contoso.com corp.contoso.com.  
  
Несвязанное пространство имен не работает (и не поддерживается) в следующих ситуациях:  
  
-   Несвязанное суффикс, используемый членов домена совпадает с именем домена Active Directory в этом или другом лесу. Это нарушает Kerberos Маршрутизация суффикса имени.  
  
-   Один и тот же несвязанного суффикс используется в другом лесу. Это предотвращает маршрутизации этих суффиксов однозначно между лесами.  
  
-   При сертификации (ЦС) члена домена сервера изменений, которые его полное доменное имя (FQDN), чтобы он больше не использовать же основного суффикса DNS, используемые контроллерами домена, к которому сервер ЦС. В этом случае могут возникнуть проблемы, проверке сертификатов сервера ЦС, выдавшего, в зависимости от того, какой DNS-имена используются в точки распространения списков отзыва Сертификатов. Но при установке сервера ЦС стабильной несвязанного пространства имен, он работает правильно и поддерживается.  
  
## <a name="considerations-for-disjoint-namespaces"></a>Рекомендации по несвязанного пространства имен  
Следующие рекомендации помогут вам решить, если необходимо использовать несвязанное пространство имен.  
  
### <a name="application-compatibility"></a>Совместимость приложений  
Как было сказано ранее несвязанного пространства имен может вызвать проблемы для всех приложений и служб, которые записываются в предположить, что основной DNS-суффикс компьютера совпадает с именем имя домена, членом которого он является. Перед развертыванием несвязанное пространство имен, необходимо проверить приложениях проблемы совместимости. Кроме того не забудьте проверить совместимость всех приложений, которые можно использовать для выполнения анализа. Сюда входят приложения Майкрософт и других разработчиков программного обеспечения.  
  
### <a name="advantages-of-disjoint-namespaces"></a>Преимущества несвязанного пространства имен  
С помощью несвязанного пространства имен может иметь следующие преимущества:  
  
-   Так как основной DNS-суффикс компьютера можно указать различные сведения, вы можете управлять пространством имен DNS отдельно от имени домена Active Directory.  
  
-   Вы можете отделить пространства имен DNS на основе бизнес-структуру или географическом расположении. Например можно разделить на основе бизнес-единицы имена или физическое расположение, например континент, Страна или создание пространства имен.  
  
### <a name="disadvantages-of-disjoint-namespaces"></a>Недостатки несвязанного пространства имен  
С помощью несвязанного пространства имен может иметь следующие недостатки:  
  
-   Необходимо создать и управлять отдельные зоны DNS для каждого домена Active Directory в лесу, в которой член компьютеров, использующих несвязанного пространства имен. (То есть его применение требует дополнительных и более сложные конфигурации.)  
  
-   Необходимо выполнить вручную, для изменения и управления Active Directory атрибут, который позволяет членам домена для использования указанного, основной DNS-суффиксы.  
  
-   Чтобы оптимизировать разрешение имен, необходимо выполнить вручную действия для изменения и ведение групповой политики для настройки компьютеров-членов с альтернативного основного DNS-суффикса.  
  
    > [!NOTE]  
    > Имя службы Windows Internet (WINS) может использоваться для смещения этот недостаток, разрешив однокомпонентные имена. Дополнительные сведения о WINS см. в разделе в техническом справочнике по WINS ([https://go.microsoft.com/fwlink/?LinkId=102303](https://go.microsoft.com/fwlink/?LinkId=102303)).  
  
-   Если ваша среда требует несколько суффиксов DNS, необходимо настроить порядок поиска суффикса DNS для всех доменов Active Directory в лесу соответствующим образом.  
  
    Чтобы задать порядок поиска суффикса DNS, можно использовать объекты групповой политики или параметры службы сервера протокол динамической настройки узла (DHCP). Можно также изменить реестр.  
  
-   Необходимо тщательно протестировать все приложения, для решения проблем совместимости.  
  
Дополнительные сведения о действиях, которые можно предпринять, чтобы устранить эти недостатки, см. Создание несвязанного пространства имен ([https://go.microsoft.com/fwlink/?LinkId=106638](https://go.microsoft.com/fwlink/?LinkId=106638)).  
  
### <a name="planning-a-namespace-transition"></a>Планирование перехода пространства имен  
Прежде чем изменять пространства имен, ознакомьтесь со следующими рекомендациями, применимые переходы непрерывного пространства имен для несвязанные пространства имен (или наоборот):  
  
-   Вручную настроить имена участника-службы (SPN) больше не соответствует DNS-имена, после внесения изменений в пространство имен. Это может привести к сбоям проверки подлинности.  
  
    Дополнительные сведения см. в разделе службы входов сбоем из-за неправильно задать имена субъектов-служб ([https://go.microsoft.com/fwlink/?LinkId=102304](https://go.microsoft.com/fwlink/?LinkId=102304)).  
  
    -   При использовании компьютеров на базе Windows Server 2003 с помощью ограниченного делегирования, эти компьютеры могут потребовать дополнительной настройки для изменения имен субъектов-служб. Дополнительные сведения см. в статье 936628 базы знаний Майкрософт ([https://go.microsoft.com/fwlink/?LinkId=102306](https://go.microsoft.com/fwlink/?LinkId=102306)).  
  
    -   Если требуется делегировать разрешения на изменение имена участников-служб на вложенные Администраторы, см. раздел Делегирование право изменять имена участников-служб ([https://go.microsoft.com/fwlink/?LinkId=106639](https://go.microsoft.com/fwlink/?LinkId=106639)).  
  
-   Если вы используете протокол доступа Directory облегченного доступа к каталогам (LDAP) через Secure Sockets Layer (SSL) (известных как LDAPS) с ЦС в развертывании с контроллеров домена, которые настроены в несвязанное пространство имен, необходимо использовать соответствующее имя домена Active Directory и DNS-суффикс при настройке LDAPS сертификаты.  
  
    Дополнительные сведения о требованиях к сертификатам контроллера домена см. в статье 321051 базы знаний Майкрософт ([https://go.microsoft.com/fwlink/?LinkId=102307](https://go.microsoft.com/fwlink/?LinkId=102307)).  
  
    > [!NOTE]  
    > Контроллеры домена, которые используют сертификаты для LDAPS может потребоваться повторное развертывание свои сертификаты. В этом случае контроллеры домена не может выбрать соответствующий сертификат, пока они перезапускаются. Дополнительные сведения о проверке подлинности LDAPS и соответствующие обновления для Windows Server 2003, см. в статье 932834 базы знаний Майкрософт ([https://go.microsoft.com/fwlink/?LinkId=102308](https://go.microsoft.com/fwlink/?LinkId=102308)).  
  
### <a name="planning-for-disjoint-namespace-deployments"></a>Планирование развертываний несвязанное пространство имен.  
Примите следующие меры предосторожности, при развертывании компьютеров в среде несвязанным пространством имен.  
  
1.  Уведомите все производители программного обеспечения, с которым вы имеете дело, необходимо протестировать и поддерживает использование несвязанного пространства имен. Попросите их убедитесь, что они поддерживают своих приложений в средах, использующих несвязанного пространства имен.  
  
2.  Протестируйте всех версий операционных систем и приложений в лабораторных средах, несвязанное пространство имен. В этом случае выполните следующие рекомендации.  
  
    1.  Устраните все проблемы программного обеспечения, перед развертыванием программного обеспечения в вашей среде.  
  
    2.  Когда это возможно, принять участие в бета-тестирования, операционных систем и приложений, которые планируется развернуть в несвязанного пространства имен.  
  
3.  Убедитесь, что администраторов и сотрудников службы технической поддержки извещены несвязанного пространства имен и его влияние.  
  
4.  Создайте план, который делает возможной вы переход из несвязанное пространство имен для непрерывного пространства имен, при необходимости.  
  
5.  Смещение акцентов важность поддержка несвязанное пространство имен с помощью операционной системы и приложений в аренду.
