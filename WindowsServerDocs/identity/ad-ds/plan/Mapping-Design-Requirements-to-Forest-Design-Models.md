# <a name="mapping-design-requirements-to-forest-design-models"></a>Сопоставление требований к модели архитектуры леса разработке

>Область применения: Windows Server 2016, Windows Server 2012 R2, Windows Server 2012

Большинство групп в организации могут совместно использовать один лес организации управляется одной сведения группы технологий (ИТ), на котором размещается учетные записи пользователей и ресурсы для всех групп, которые совместно используют леса. Этот общий леса с именем начальной леса в организации, является основой модель разработки леса в организации.  
  
Так как начальное леса в организации может размещаться несколько групп в организации, владелец леса необходимо установить соглашения об уровне обслуживания с каждой группой, чтобы всех сторон понять, что от них ожидается. Это обеспечивает защиту отдельных групп и владелец леса, устанавливая службы согласованного ожиданиям.  
  
Если не все группы в организации могут использовать одного леса в организации, необходимо развернуть проекта леса для удовлетворения потребностей для разных групп. Это включает в себя идентифицировать требования разработки, которые применяются в группы на основе своих требований для автономность и изоляция и ли они имеют ограниченный подключения сети и затем модель леса, который можно использовать, чтобы адаптировать эти требования. В следующей таблице перечислены сценарии модель разработки леса, на основе автономность, изоляции и факторы подключения. Определив сценарии проектирования леса, который наилучшим образом соответствует вашим требованиям, определите, если необходимо внести какие-либо дополнительные решения в соответствии с требованиями разработки.

## <a name="BKMK_1"></a>Сценарий 1: Присоединение существующий лес для автономность данных  
Требования для данных автономность может соответствовать путем размещения в группу в организационные подразделения (OU) в существующем лесу организации. Делегирование управления подразделениях администраторам данных из этой группы для достижения автономность данных. Дополнительные сведения о делегировании управления с помощью подразделений см. в разделе [организационной единицы проектирование](../../ad-ds/plan/Creating-an-Organizational-Unit-Design.md).  
  
## <a name="BKMK_2"></a>Сценарий 2: Используйте для службы автономность леса в организации или домен  
Если в группу в вашей организации идентифицирует автономность службы как требование, рекомендуется сначала пересмотреть это требование. Достижение автономность служба создает дополнительные расходы на управление и дополнительные затраты для организации. Убедитесь, что требование автономность службы не просто для удобства и что можно выровнять затраты, связанные с конференц-это требование.  
  
Требования для службы автономность удовлетворяет требованиям, выполнив одно из следующих:  
  
-   Создание леса в организации. Поместите пользователей, групп и компьютеров для группы, которая требует автономность службы в отдельный леса в организации. Назначьте отдельного из этой группы в лесу владельца. Если группы должно ресурсов доступа или общей папки с остальные леса в организации, они могут установить отношение доверия между их леса в организации и в других лесах.  
  
-   С помощью организации доменов. Поместите пользователей, групп и компьютеров в отдельный домен в существующем лесу организации. Эта модель предоставляет только автономность службы на уровне домена и не автономность полный службы службы изоляции или изоляции данных.  
  
Дополнительные сведения об использовании организации доменов см. в разделе [использование организационной модели домена леса](../../ad-ds/plan/../../ad-ds/plan/Using-the-Organizational-Domain-Forest-Model.md).  
  
## <a name="BKMK_3"></a>Сценарий 3: Используйте леса в организации или леса ресурсов для изоляции службы  
Требования для изоляции службы удовлетворяет требованиям, выполнив одно из следующих:  
  
-   С помощью леса в организации. Поместите пользователей, групп и компьютеров для группы, которая требует изоляции службы в отдельный леса в организации. Назначьте отдельного из этой группы в лесу владельца. Если группы должно ресурсов доступа или общей папки с остальные леса в организации, они могут установить отношение доверия между их леса в организации и в других лесах. Однако не рекомендуется этот подход, так как доступ к ресурсам в универсальных группах во многом ограничено только в сценариях с доверием леса.  
  
-   С помощью леса ресурсов. Поместить в лес отдельный ресурса хранение учетных записей пользователей в существующий лес организации ресурсам и учетные записи служб. При необходимости альтернативные учетные записи могут создаваться в лесу ресурсов, для доступа к ресурсам в лесу ресурсов, если леса в организации становится недоступным. Альтернативные учетные записи необходимо иметь центр, необходимо войти в лесу ресурсов и сохранить контроль над ресурсами, пока не будет восстановлена леса в организации.  
  
    Установите отношение доверия между ресурсов и организации лесов пользователей на доступ к ресурсам в лесу, используя свои учетные записи обычных пользователей. Эта конфигурация позволяет централизованное управление учетными записями пользователей, одновременно позволяя пользователям, если леса в организации становится недоступным переключится на альтернативные учетные записи в лесу ресурсов.  
  
Рекомендации для изоляции службы включают в себя следующее:  
  
-   Лесов, созданные для изоляции службы можно доверять доменов из других лесов, но не должен включать пользователей из других лесов в их групп администраторов службы. Если в административной группы в изолированной лесу входят пользователи из других лесов, безопасности изолированной леса потенциально может быть нарушена из-за администраторов служб в лесу нет монопольное управление.  
  
-   Как контроллеры домена доступны в сети, они подвержены атакам (например, атаки типа "отказ в обслуживании") от вредоносного программного обеспечения в этой сети. Можно сделать следующее, чтобы защитить атаки.  
  
    -   Контроллеры домена узла только в сетях, которые считаются безопасными.  
  
    -   Ограничение доступа к сети или сетей, размещения контроллеров домена.  
  
-   Служба изоляции требуется создавать дополнительный лес. Проверять затраты на поддержку инфраструктуры для поддержки дополнительных леса перевешивают затраты, связанные с потеря доступа к ресурсам из-за недоступность леса в организации.  
  
## <a name="BKMK_4"></a>Сценарий 4: Используйте леса в организации или леса ограниченный доступ для изоляции данных  
Изоляции данных можно получить, выполнив одно из следующих:  
  
-   С помощью леса в организации. Поместите пользователей, групп и компьютеров для группы, которая требует изоляции данных в отдельные леса в организации. Назначьте отдельного из этой группы в лесу владельца. Если группы должно ресурсов доступа или общей папки с остальные леса в организации, необходимо установите отношение доверия между леса в организации и в других лесах. В нового леса в организации существует только пользователей, которым требуется доступ к конфиденциальной информации. Пользователи имеют одну учетную запись, они используют для доступа классификации данных, их леса и Неклассифицированные данных в других лесах посредством отношения доверия.  
  
-   С помощью леса ограниченного доступа. Это отдельный лес, содержащий защищенных данных и учетные записи пользователей, которые используются для доступа к этим данным. Отдельные учетные записи пользователей сохраняются в существующих лесах организации, используемые для доступа к ресурсам без ограничений в сети. Создается без отношений доверия между лесом ограниченный доступ и другими лесами на предприятии. Можете еще сильнее ограничить леса, развернув леса на отдельную физическую сеть, чтобы он не может подключиться к других лесах. Если вы развертываете леса в отдельной сети, пользователи должны иметь два рабочих станциях: один для доступа к леса с ограниченным и одна для доступа к nonrestricted областях сети.  
  
Следующие рекомендации по созданию лесов для изоляции данных:  
  
-   Организации лесов, созданный для изоляции данных можно доверять доменов из других лесов, но пользователи из других лесов не должно включаться в любом из следующих:  
  
    -   Группы, отвечающие за управление службами или групп, которыми можно управлять членства в группах администратора службы  
  
    -   Группы, которые имеют административный контроль над управлением компьютерами, на которых хранятся защищенные данные  
  
    -   Группы, которые имеют доступ к защищенным данным или группы, которые отвечают за управление пользовательских объектов или группы объектов, которые имеют доступ к защищенным данным  
  
    Если в любой из этих групп входят пользователи из другого леса, компрометации другого леса может привести для компрометации изолированной леса и раскрытия данных на защищенном.  
  
-   Другие леса можно настроить доверие леса в организации, созданный для изоляции данных, чтобы пользователи изолированной леса могут обращаться к ресурсам в других лесах. Тем не менее пользователи из изолированного леса должны входить никогда не в интерактивном режиме на рабочих станциях в доверяющем лесу. Компьютер в доверяющем лесу могут быть раскрыты потенциально вредоносного программного обеспечения и может использоваться для записи учетные данные для входа пользователя.  
  
    > [!NOTE]  
    > Чтобы предотвратить олицетворение пользователей из изолированного леса, а затем доступ к ресурсам в изолированном лесу серверами в доверяющем лесу, владелец леса можно отключить делегирование проверки подлинности или использовать функцию ограниченного делегирования. Дополнительные сведения о делегирование проверки подлинности и ограниченного делегирования см. в разделе делегирование проверки подлинности ([https://go.microsoft.com/fwlink/?LinkId=106614](https://go.microsoft.com/fwlink/?LinkId=106614)).  
  
-   Может потребоваться установить брандмауэра между леса в организации и в других лесах в организации, чтобы ограничить доступ пользователей к данным вне своего леса.  
  
-   Несмотря на то, что создание отдельного леса позволяет изоляции данных, пока контроллеры домена в лесу изолированной и компьютеры, сведения о защищенных узлов доступны в сети, то они подвержены атаки, запущенные с компьютеров в этой сети. Организациям, которые решить риск атаки слишком велика или вследствие нарушение атак или безопасности слишком велика необходимы для ограничения доступа к сети или сетей, в которых размещены контроллеры домена и компьютеры, на которых размещается защищенных данных. Ограничение доступа можно сделать с помощью технологий, например брандмауэры и безопасность протокола IP (IPsec). В некоторых случаях организации могут выбрать для поддержания защищенных данных на независимые сети, которая не имеет физического подключения к другой сети в организации.  
  
    > [!NOTE]  
    > Если любого сетевого подключения между лесом ограниченный доступ и другой сетью, для данных в области ограниченных для передачи в другие сети существует возможность.  
  
## <a name="BKMK_5"></a>Сценарий 5: Используйте леса в организации, или повторной настройки брандмауэра для ограниченные возможности подключения  
В соответствии с требованием ограниченные возможности подключения, выполните одно из следующих действий.  
  
-   Поместить пользователей в существующий лес организации, а затем откройте брандмауэр для разрешения Active Directory прохождение трафика через достаточно.  
  
-   Используйте леса в организации. Переведите пользователей, групп и компьютеров для группы, для которого ограничены подключения в отдельном леса в организации. Назначьте отдельного из этой группы в лесу владельца. Леса в организации обеспечивает отдельную среду на другой стороне брандмауэра. Лес содержит учетные записи пользователей и ресурсы, которые управляются внутри леса, чтобы пользователям не требуется переходить брандмауэра для выполнения своих повседневных задач. Конкретных пользователей или приложений может иметь особые потребности, которые требуется возможность для передачи через брандмауэр для связи с другими лесами. Можно по отдельности соблюдения требований, открыв соответствующие интерфейсы в брандмауэре, включая те, которые необходимы для любой отношений доверия для работы.  
  
Дополнительные сведения о настройке брандмауэров для использования с доменными службами Active Directory (AD DS) см. в разделе Active Directory в сегментированных сетей с брандмауэрами ([https://go.microsoft.com/fwlink/?LinkId=37928](https://go.microsoft.com/fwlink/?LinkId=37928)).  
  
## <a name="BKMK_6"></a>Сценарий 6: Использование домена или леса в организации и перенастроить брандмауэра для службы автономность с ограниченные возможности подключения  
Если в группу в вашей организации идентифицирует автономность службы как требование, рекомендуется сначала пересмотреть это требование. Достижение автономность служба создает дополнительные расходы на управление и дополнительные затраты для организации. Убедитесь, что требование автономность службы не просто для удобства и что можно выровнять затраты, связанные с конференц-это требование.  
  
Ограниченные возможности подключения является проблемой, и требуется автономность службы, необходимо выполнить одно из следующих:  
  
-   Используйте леса в организации. Поместите пользователей, групп и компьютеров для группы, которая требует автономность службы в отдельный леса в организации. Назначьте отдельного из этой группы в лесу владельца. Леса в организации обеспечивает отдельную среду на другой стороне брандмауэра. Лес содержит учетные записи пользователей и ресурсы, которые управляются внутри леса, чтобы пользователям не требуется переходить брандмауэра для выполнения своих повседневных задач. Конкретных пользователей или приложений может иметь особые потребности, которые требуется возможность для передачи через брандмауэр для связи с другими лесами. Можно по отдельности соблюдения требований, открыв соответствующие интерфейсы в брандмауэре, включая те, которые необходимы для любой отношений доверия для работы.  
  
-   Поместите пользователей, групп и компьютеров в отдельный домен в существующем лесу организации. Эта модель предоставляет только автономность службы на уровне домена и не автономность полный службы службы изоляции или изоляции данных. Другие группы в лесу должен доверять администраторов служб нового домена в том же уровне, что они доверяют Владелец леса. По этой причине не рекомендуется использовать этот подход. Дополнительные сведения об использовании организации доменов см. в разделе [использование организационной модели домена леса](../../ad-ds/plan/../../ad-ds/plan/Using-the-Organizational-Domain-Forest-Model.md).  
  
Также необходимо открыть брандмауэр для разрешения Active Directory прохождение трафика через достаточно. Дополнительные сведения о настройке брандмауэров для использования с AD DS см. в разделе Active Directory в сегментированных сетей с брандмауэрами ([https://go.microsoft.com/fwlink/?LinkId=37928](https://go.microsoft.com/fwlink/?LinkId=37928)).  
  
## <a name="BKMK_7"></a>Сценарий 7: Используйте леса ресурсов и перенастроить брандмауэра для изоляции службы с ограниченные возможности подключения  
Если проблема ограниченные возможности подключения, и у вас есть требования для изоляции службы, выполните одно из следующих:  
  
-   Используйте леса в организации. Поместите пользователей, групп и компьютеров для группы, которая требует изоляции службы в отдельный леса в организации. Назначьте отдельного из этой группы в лесу владельца. Леса в организации обеспечивает отдельную среду на другой стороне брандмауэра. Лес содержит учетные записи пользователей и ресурсы, которые управляются внутри леса, чтобы пользователям не требуется переходить брандмауэра для выполнения своих повседневных задач. Конкретных пользователей или приложений может иметь особые потребности, которые требуется возможность для передачи через брандмауэр для связи с другими лесами. Можно по отдельности соблюдения требований, открыв соответствующие интерфейсы в брандмауэре, включая те, которые необходимы для любой отношений доверия для работы.  
  
-   Используйте леса ресурсов. Поместить в лес отдельный ресурса хранение учетных записей пользователей в существующий лес организации ресурсам и учетные записи служб. Может понадобиться создать некоторые альтернативные учетные записи в лесу ресурсов, чтобы обеспечить доступ к лесу ресурсов, если леса в организации становится недоступным. Альтернативные учетные записи необходимо иметь центр, необходимо войти в лесу ресурсов и сохранить контроль над ресурсами, пока не будет восстановлена леса в организации.  
  
    Установите отношение доверия между ресурсов и организации лесов пользователей на доступ к ресурсам в лесу, используя свои учетные записи обычных пользователей. Эта конфигурация позволяет централизованное управление учетными записями пользователей, одновременно позволяя пользователям, если леса в организации становится недоступным переключится на альтернативные учетные записи в лесу ресурсов.  
  
Рекомендации для изоляции службы включают в себя следующее:  
  
-   Лесов, созданные для изоляции службы можно доверять доменов из других лесов, но не должен включать пользователей из других лесов в их групп администраторов службы. Если в административной группы в изолированной лесу входят пользователи из других лесов, безопасности изолированной леса потенциально может быть нарушена из-за администраторов служб в лесу нет монопольное управление.  
  
-   Поскольку контроллеры домена доступны в сети, они подвержены атакам (например, атаки типа "отказ в обслуживании") от компьютеров в этой сети. Можно сделать следующее, чтобы защитить атаки.  
  
    -   Контроллеры домена узла только в сетях, которые считаются безопасными.  
  
    -   Ограничение доступа к сети или сетей, размещения контроллеров домена.  
  
-   Служба изоляции требуется создавать дополнительный лес. Проверять затраты на поддержку инфраструктуры для поддержки дополнительных леса перевешивают затраты, связанные с потеря доступа к ресурсам из-за недоступность леса в организации.  
  
    Конкретных пользователей или приложений может иметь особые потребности, которые требуется возможность для передачи через брандмауэр для связи с другими лесами. Можно по отдельности соблюдения требований, открыв соответствующие интерфейсы в брандмауэре, включая те, которые необходимы для любой отношений доверия для работы.  
  
Дополнительные сведения о настройке брандмауэров для использования с AD DS см. в разделе Active Directory в сегментированных сетей с брандмауэрами ([https://go.microsoft.com/fwlink/?LinkId=37928](https://go.microsoft.com/fwlink/?LinkId=37928)).
