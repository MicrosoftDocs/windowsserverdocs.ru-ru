# <a name="mapping-design-requirements-to-forest-design-models"></a>Сопоставление требований к модели архитектуры леса разработке

>Область применения. Windows Server 2016, Windows Server 2012 R2, Windows Server 2012

Большинство групп в вашей организации могут совместно использовать один лес организации под управлением ИТ-группы сведения о едином, содержащий учетные записи пользователей и ресурсы для всех групп, которые совместно используют леса. Этот общий леса, с именем начальной леса в организации, лежит в основе модели разработки леса в организации.  

Так как начальная леса в организации может содержать несколько групп в организации, владелец леса необходимо установить соглашения об уровне обслуживания с каждой группой, что всем сторонам понять, что от них ожидается. Это обеспечивает защиту отдельных групп и владелец леса, устанавливая службы согласованного ожиданиям.  

Если не все группы в организации могут совместно использовать одного леса в организации, необходимо развернуть проект леса для удовлетворения потребностей различных групп. Это включает определение требований, которые применяются к группам, в зависимости от потребностей автономность и изоляции и ли они имеют limited-подключения к сети и затем определения модели леса, можно использовать в соответствии с их требования. Ниже перечислены сценарии модели разработки леса, основанные на автономность, изоляции и факторов, подключение. После определения сценария разработки леса, который наилучшим образом соответствует вашим требованиям, необходимо определите сделать какие-либо дополнительные решения в соответствии с требованиями проектирования.

## <a name="BKMK_1"></a>Сценарий 1. Присоединяйтесь к существующему лесу для независимости данных  

Позволяет выполнить все требования для независимости данных путем размещения группы в подразделениях (OU) в существующем лесу организации. Делегирование контроля над подразделения администраторам данных из этой группы, для достижения независимости данных. Дополнительные сведения о делегировании управления с помощью подразделений см. в разделе [проектирование организационные единицы](../../ad-ds/plan/Creating-an-Organizational-Unit-Design.md).  
  
## <a name="BKMK_2"></a>Сценарий 2. Использовать леса в организации или домена для службы независимости  

Если в группу в вашей организации определяет автономность службы как требование, мы рекомендуем сначала пересмотреть это требование. Добиться автономности службы создает дополнительные расходы на управление и дополнительные затраты для организации. Убедитесь, что требования службы автономность не просто для удобства и что вы можете оправдать затраты, задействованные в соответствие этому требованию.  
  
Позволяет выполнить все требования для службы автономность, выполнив одно из следующих:  

- Создание леса в организации. Поместите пользователей, групп и компьютеров для группы, которому требуется автономность службы в отдельные леса в организации. Назначьте отдельного из этой группы в лесу владельца. Если группа должна доступа или общей папки ресурсов с другим лесам в организации, они могут устанавливать отношения доверия между их леса в организации и другими лесами.  

- Использование доменов организации. Поместите пользователей, групп и компьютеров в отдельный домен в существующем лесу организации. Эта модель обеспечивает независимость уровня домена службы только и не автономность полноценных службы изоляции, или с изоляцией данных.  

Дополнительные сведения об использовании доменов организации, см. в разделе [с помощью организационной модели леса домена](../../ad-ds/plan/../../ad-ds/plan/Using-the-Organizational-Domain-Forest-Model.md).  

## <a name="BKMK_3"></a>Сценарий 3. Использовать для изоляции служб леса в организации или леса ресурсов  

Позволяет выполнить все требования для изоляции служб, выполнив одно из следующих:  

- С помощью леса в организации. Поместите пользователей, групп и компьютеров для группы, которому требуется изоляции служб в отдельные леса в организации. Назначьте отдельного из этой группы в лесу владельца. Если группа должна доступа или общей папки ресурсов с другим лесам в организации, они могут устанавливать отношения доверия между их леса в организации и другими лесами. Тем не менее не рекомендуется этот подход, так как в сценариях с доверием леса сильно ограничен доступ к ресурсам с помощью универсальных групп.  

- С помощью леса ресурсов. Поместите ресурсы и учетные записи служб в лесу отдельный ресурс, сохранение учетных записей пользователей в существующем лесу организации. При необходимости альтернативные учетные записи могут создаваться в лесу ресурсов для доступа к ресурсам в лесу ресурсов, если леса в организации становится недоступным. Альтернативные учетные записи должен иметь полномочия, необходимые для входа лесу ресурсов и поддерживать контроль над ресурсами, пока не будет восстановлена леса в организации.  

   Доверие между ресурсов и леса в организации, позволяя пользователям получать доступ к ресурсам в лесу при работе с учетной записью обычного пользователя. Эта конфигурация обеспечивает централизованное управление учетными записями пользователей дает возможность пользователям переключиться на альтернативные учетные записи в лесу ресурсов Если леса в организации становится недоступным.  

Ниже приведены рекомендации по изоляции служб:

- Лесов, созданные для изоляции служб могут доверять домены из других лесов, но не должен включать пользователей из других лесов в их группах администраторов службы. Если пользователи из других лесов в административные группы в лесу, изолированной, безопасность изолированного леса потенциально могут быть раскрыты так, как Администраторы службы в лесу не имеют в монопольном режиме.  

- До тех пор, пока контроллеры домена доступны в сети, они подвергаются атакам (таких как атаки типа "отказ в обслуживании") от вредоносных программ в этой сети. Необходимо выполнить следующую команду, чтобы предотвратить атаки действий.  

   - Контроллеров домена только в сетях, которые считаются безопасными.  

   - Ограничение доступа к сети или сетям, размещение контроллеров домена.  

- Изоляция служб необходимо создать дополнительный лес. Оцените ли затраты на поддержку инфраструктуры для поддержки дополнительных леса перевешивает расходы, связанные с потерей доступа к ресурсам из-за недоступности леса в организации.  

## <a name="BKMK_4"></a>Сценарий 4. Использовать для изоляции данных леса в организации или леса с ограниченным доступом  

Изоляция данных можно получить, выполнив одно из следующих:  

- С помощью леса в организации. Поместите пользователей, групп и компьютеров для группы, которому требуется изоляция данных в отдельные леса в организации. Назначьте отдельного из этой группы в лесу владельца. Если группа должна доступа или общей папки ресурсов с другим лесам в организации, необходимо установите отношение доверия между леса в организации и другими лесами. Только пользователи, которым требуется доступ к конфиденциальной информации существует в нового леса в организации. Пользователи имеют одну учетную запись, они используют для доступа классифицировать данные в свои собственные лес и Несекретные данные в других лесах, посредством отношения доверия.  

- С помощью леса ограниченный доступ. Это отдельный лес, содержащий защищенные данные и учетные записи пользователей, которые используются для доступа к этим данным. Отдельные учетные записи сохраняются в существующих лесах организации, которые используются для доступа к неограниченным ресурсам в сети. Нет отношения доверия создается между лесом ограниченного доступа и другим лесам в организации. Можно дополнительно ограничить лес путем развертывания леса в отдельную физическую сеть, таким образом, чтобы его не удается подключиться к других лесах. При развертывании леса в отдельной сети, пользователи должны иметь два рабочих станций: одна для доступа к ограниченным леса и одна для доступа к nonrestricted областях сети.  

Ниже приведены рекомендации по созданию лесов для изоляции данных:  

- Организации лесов, созданный для изоляции данных могут доверять домены из других лесов, но пользователи из других лесов не должен включаться в любой из следующих:  

   - Группы, отвечающие за управление службами или групп, которыми можно управлять членства в группах администратора службы  

   - Групп, имеющих административный контроль над компьютеров, в которых хранятся защищенные данные  

   - Группы, имеющие доступ к защищенным данным или группы, которые отвечают за управление объектов пользователей или группы объектов, имеющих доступ к защищенным данным  

   Если в любой из этих групп пользователей из другого леса, компрометация другого леса может привести компрометацию изолированный лес, а также к раскрытию защищенных данных.  

- Другими лесами может быть настроен на доверие леса в организации, созданные для изоляции данных пользователей в лесу, изолированной можно получить доступ к ресурсам в других лесах. Тем не менее пользователи из изолированного леса должны иметь никогда не в интерактивном режиме на рабочих станциях в доверяющем лесу. Компьютер в доверяющем лесу потенциально может быть нарушена вредоносным программным обеспечением и может использоваться для захвата учетных данных пользователя.  

   > [!NOTE]
   > Чтобы предотвратить серверы в доверяющем лесу олицетворение пользователей из изолированного леса и затем осуществлять доступ к ресурсам в изолированный лес, владелец леса можно отключить делегирование проверки подлинности или использовать функции ограниченного делегирования. Дополнительные сведения о делегирование проверки подлинности и ограниченным делегированием см. в разделе [делегирование проверки подлинности](https://go.microsoft.com/fwlink/?LinkId=106614).  

- Может потребоваться установить брандмауэр между леса в организации и в других лесах в организации, чтобы ограничить доступ пользователей к за пределами своего леса.  

- Несмотря на то, что создание отдельный лес позволяет изоляции данных, до тех пор, пока контроллеров домена в лесу, изолированной и компьютеров, защитой ведущих сведения доступны в сети, они подвержены атакам, запускаемых из компьютеров в этой сети. Организациям, которые решают, что риск атаки, слишком велико или слишком велико, вследствие нарушения атаки и безопасности необходимо, чтобы ограничить доступ к сети или сети, на которых размещены контроллеры домена и компьютеры, на которых размещаются защищенных данных . Ограничение доступа можно сделать с помощью технологий, таких как брандмауэры и безопасность протокола IP (IPsec). В исключительных случаях организации могут выбрать для сохранения защищенных данных в независимой сети, которая не имеет физического подключения к любой другой сети в организации.  

   > [!NOTE]  
   > Если сетевых подключений между лесом ограниченного доступа и другой сетью, существует возможность для данных в области ограниченного для передачи в другой сети.  

## <a name="BKMK_5"></a>Сценарий 5. Используйте леса в организации, или изменить настройки брандмауэра для ограниченным подключением  

Чтобы выполнить требование ограниченным подключением, выполните одно из следующих действий.  

- Поместите пользователей в существующем лесу организации, а затем откройте брандмауэр для разрешения трафика Active Directory проходить через достаточно.  

- С помощью леса в организации. Поместите пользователей, групп и компьютеров для группы, для которого ограничивается подключением в отдельные леса в организации. Назначьте отдельного из этой группы в лесу владельца. Леса в организации предоставляет отдельную среду на другой стороне брандмауэра. Лес содержит учетные записи пользователей и ресурсы, управление которыми осуществляется в пределах леса, таким образом, чтобы пользователям не обязательно должны проходить через брандмауэр так, чтобы выполнять свои повседневные задачи. Определенные пользователи или приложения может иметь особые задачи, которые требуют наличия возможности для передачи через брандмауэр для связи с другими лесами. Может разрешить эти проблемы по отдельности, открыв соответствующие интерфейсы в брандмауэре, включая те, которые необходимы для отношений доверия функционировать.  

Дополнительные сведения о настройке брандмауэров для использования с доменными службами Active Directory (AD DS), см. в разделе [Active Directory в сетях с брандмауэрами](https://go.microsoft.com/fwlink/?LinkId=37928).  

## <a name="BKMK_6"></a>Сценарий 6. Использование домена или леса в организации и изменить настройки брандмауэра для службы автономность с ограниченным подключением

Если отсутствуют требования для службы автономность ограниченным подключением является проблемой, необходимо выполнить одно из следующих:  

- С помощью леса в организации. Поместите пользователей, групп и компьютеров для группы, которому требуется автономность службы в отдельные леса в организации. Назначьте отдельного из этой группы в лесу владельца. Леса в организации предоставляет отдельную среду на другой стороне брандмауэра. Лес содержит учетные записи пользователей и ресурсы, управление которыми осуществляется в пределах леса, таким образом, чтобы пользователям не обязательно должны проходить через брандмауэр так, чтобы выполнять свои повседневные задачи. Другие группы в лесу должен доверять администраторами служб для нового домена в такой же степени, что они доверяют Владелец леса. По этой причине не рекомендуется использовать этот подход. Дополнительные сведения об использовании доменов организации, см. в разделе [с помощью организационной модели леса домена](../../ad-ds/plan/../../ad-ds/plan/Using-the-Organizational-Domain-Forest-Model.md).  

Также необходимо открыть брандмауэр для разрешения трафика Active Directory проходить через достаточно. Дополнительные сведения о настройке брандмауэров для использования с AD DS, см. в разделе [Active Directory в сетях с брандмауэрами](https://go.microsoft.com/fwlink/?LinkId=37928).  

## <a name="BKMK_7"></a>Сценарий 7. Используйте леса ресурсов и изменить настройки брандмауэра для изоляции служб с ограниченным подключением  

Если существует требование для изоляции служб ограниченным подключением является проблемой, необходимо выполнить одно из следующих:  

- С помощью леса в организации. Поместите пользователей, групп и компьютеров для группы, которому требуется изоляции служб в отдельные леса в организации. Назначьте отдельного из этой группы в лесу владельца. Леса в организации предоставляет отдельную среду на другой стороне брандмауэра. Лес содержит учетные записи пользователей и ресурсы, управление которыми осуществляется в пределах леса, таким образом, чтобы пользователям не обязательно должны проходить через брандмауэр так, чтобы выполнять свои повседневные задачи. Определенные пользователи или приложения может иметь особые задачи, которые требуют наличия возможности для передачи через брандмауэр для связи с другими лесами. Может разрешить эти проблемы по отдельности, открыв соответствующие интерфейсы в брандмауэре, включая те, которые необходимы для отношений доверия функционировать.  

- С помощью леса ресурсов. Поместите ресурсы и учетные записи служб в лесу отдельный ресурс, сохранение учетных записей пользователей в существующем лесу организации. Возможно, потребуется создать некоторые учетные записи другого пользователя в лесу ресурсов для сохранения доступа к лесу ресурсов, если леса в организации становится недоступным. Альтернативные учетные записи должен иметь полномочия, необходимые для входа лесу ресурсов и поддерживать контроль над ресурсами, пока не будет восстановлена леса в организации.  

   Доверие между ресурсов и леса в организации, позволяя пользователям получать доступ к ресурсам в лесу при работе с учетной записью обычного пользователя. Эта конфигурация обеспечивает централизованное управление учетными записями пользователей дает возможность пользователям переключиться на альтернативные учетные записи в лесу ресурсов Если леса в организации становится недоступным.  

Ниже приведены рекомендации по изоляции служб:  

- Лесов, созданные для изоляции служб могут доверять домены из других лесов, но не должен включать пользователей из других лесов в их группах администраторов службы. Если пользователи из других лесов в административные группы в лесу, изолированной, безопасность изолированного леса потенциально могут быть раскрыты так, как Администраторы службы в лесу не имеют в монопольном режиме.  

- До тех пор, пока контроллеры домена доступны в сети, они подвергаются атакам (таких как атаки типа "отказ в обслуживании") с компьютеров в этой сети. Необходимо выполнить следующую команду, чтобы предотвратить атаки действий.  

   - Контроллеров домена только в сетях, которые считаются безопасными.  

   - Ограничение доступа к сети или сетям, размещение контроллеров домена.  

- Изоляция служб необходимо создать дополнительный лес. Оцените ли затраты на поддержку инфраструктуры для поддержки дополнительных леса перевешивает расходы, связанные с потерей доступа к ресурсам из-за недоступности леса в организации.  

   Определенные пользователи или приложения может иметь особые задачи, которые требуют наличия возможности для передачи через брандмауэр для связи с другими лесами. Может разрешить эти проблемы по отдельности, открыв соответствующие интерфейсы в брандмауэре, включая те, которые необходимы для отношений доверия функционировать.  

Дополнительные сведения о настройке брандмауэров для использования с AD DS, см. в разделе [Active Directory в сетях с брандмауэрами](https://go.microsoft.com/fwlink/?LinkId=37928).
