# <a name="planning-regional-domain-controller-placement"></a>Планирование размещения регионального контроллера домена

>Область применения: Windows Server 2016, Windows Server 2012 R2, Windows Server 2012

Чтобы рентабельность, спланируйте разместить как можно меньше регионального домена контроллеры максимально. Во-первых, просмотрите лист (DSSTOPO_1.doc) «Географического расположения и каналы связи», используемые в [сбор сведений о сети](../../ad-ds/plan/Collecting-Network-Information.md) для определения расположения концентратора.  
  
Планирование разместить контроллеры регионального домена для каждого домена, который представлен в каждом расположения концентратора. После размещения регионального домена контроллеры во всех расположениях концентратора, проанализируйте необходимость обеспечения размещение регионального домена контроллеры во вспомогательных расположениях. Исключая контроллеры ненужные регионального домена из расположения вспомогательных снижает затраты на поддержку, необходимые для поддержки инфраструктуры удаленного сервера.  
  
Кроме того, физической безопасности контроллеров домена в основной и вспомогательной расположения, чтобы несанкционированную не имеют доступа к их. Не помещайте контроллеров домена в основной и вспомогательной расположения, в которых не может гарантировать физической безопасности контроллера домена. Пользователь, имеющий физический доступ к контроллеру домена с возможностью записи могут атаковать системе путем:  
  
-   Доступ к физических дисков, запустив другую операционную систему на контроллере домена.  
  
-   Удаление (и возможно замена) физические диски на контроллере домена.  
  
-   Получение и обработка копию архивацию состояния системы контроллера домена.  
  
Добавьте контроллеры домена с возможностью региональные только в расположения, в которых можно обеспечить их физической безопасности.  
  
В расположениях с недостатком физической безопасности развертывания контроллера домена только для чтения (RODC) является рекомендуемым решением. За исключением паролей учетных записей RODC содержит все объекты Active Directory и атрибуты, которые содержит контроллер домена с возможностью записи. Тем не менее нельзя было внести изменения в базу данных, хранящийся в RODC. Изменения необходимо на контроллере домена с возможностью записи и затем снова реплицировать на RODC.  
  
Для проверки подлинности клиентских входов и доступ к Локальные файловые серверы, в большинстве организаций разместить контроллеры регионального домена для всех региональных доменов, которые представлены в указанное расположение. Тем не менее большое количество переменных необходимо учитывать при оценке расположение бизнеса требует наличия у локальной проверки подлинности ее клиентов или клиенты, могут положиться на проверку подлинности и запрос через связи глобальной сети (WAN). Ниже показано, как определить, следует ли разместить контроллеры домена на вспомогательных расположения.

## <a name="onsite-technical-expertise-availability"></a>Доступность техническую поддержку на месте  
Контроллеры домена должны управляться постоянно по разным причинам. Поместите контроллера регионального домена только в расположениях, которые включают сотрудники, которые можно администрировать контроллер домена, или убедитесь, что контроллер домена можно управлять удаленно.  
  
В средах филиалов обычно снижению физической безопасности и персонал с небольшой знаний сведения о технологии развертывания контроллера RODC часто является рекомендуемым решением. Локальные административные разрешения для RODC можно делегировать любого пользователя домена, не предоставляя пользователю никаких прав для домена или другие контроллеры домена этого пользователя. Это позволяет локальный пользователь филиала для входа RODC и выполнения работ по обслуживанию на сервере, таких как обновления драйвера. Тем не менее пользователь филиала не удается войти любой другой контроллер домена или выполнять другие задачи администрирования в домене. Таким образом, пользователь филиала может быть делегирована возможность эффективно управлять RODC в филиале без ущерба для безопасности остальной части домена или леса.  
  
## <a name="wan-link-availability"></a>Доступность глобальной  
Подключения к ГЛОБАЛЬНОЙ сети, возникают частых сбоев может привести к потере производительности для пользователей, если расположение не включает контроллер домена, может пройти проверку подлинности пользователей. Если состояния в глобальной сети не 100 процентов, и на удаленных сайтах не допускает сбой в работе службы, разместите контроллера регионального домена в местах, где пользователям требуется возможность войти в систему или exchange server доступа глобальной не работает.  
  
## <a name="authentication-availability"></a>Доступность проверки подлинности  
Некоторые организации, например, банки, требуют проверки подлинности пользователей в любое время. Поместите контроллера регионального домена в расположение, где доступности глобальной не 100 процентов, а пользователи требуют проверки подлинности в любое время.  
  
## <a name="logon-performance-over-wan-links"></a>Производительность входа в систему через ГЛОБАЛЬНУЮ сеть  
Что надежность, доступность глобальной размещения контроллера домена в расположении зависит от требований к производительности входа каналу WAN. Факторы, влияющие на производительность входа в систему через глобальную сеть включают скорость связи и доступная пропускная способность, количество пользователей и использование профилей и объем входа сетевой трафик и трафик репликации.  
  
### <a name="wan-link-speed-and-bandwidth-utilization"></a>Скорость и пропускной способности загруженность каналов глобальной сети  
Сведения о действиях один пользователь может congest медленное WAN. Установите контроллер домена в расположении, если недопустима производительности входа каналу WAN.  
  
Средний процент использования пропускной способности указывает, как перегруженных сетевого подключения. Если сетевое соединение использования средней пропускной способности, который превышает допустимое значение, установите контроллер домена в этом расположении.  
  
### <a name="number-of-users-and-usage-profiles"></a>Число пользователей и использование профилей  
Число пользователей и их использование профилей в заданном расположении помогают определить, требуется ли разместить контроллеры регионального домена в этом расположении. Во избежание снижения производительности в случае сбоя связи WAN, разместите контроллера регионального домена на месте, которое 100 или нескольких пользователей.  
  
Использование профилей указывают, как пользователи применяют сетевым ресурсам. Вам не нужно помещать контроллера домена в расположении, которое содержит только несколько пользователей, которые не осуществляют частый доступ к сетевым ресурсам.  
  
### <a name="logon-network-traffic-vs-replication-traffic"></a>Вход в систему сетевой трафик и трафик репликации  
Если контроллер домена в одном расположении, как клиент Active Directory недоступен, клиент создает входа трафика в сети. Объем сетевого трафика входа, который создается в физической сети зависит от нескольких факторов, включая членства в группах; число и размер объектов групповой политики (GPO); сценарии входа в систему; и компоненты, такие как автономные папки, перенаправление папок и перемещаемых профилей.  
  
С другой стороны контроллер домена, который помещается в указанное расположение создает трафика репликации в сети. Частота и количество обновлений, внесенные на разделов, размещенных на контроллерах домена влиять на объем трафика репликации, которая создается в сети. Добавление или изменение пользователей и атрибутов пользователей, изменение паролей и добавления или глобальные группы, принтеры или томов, включают различные типы обновлений, которые могут быть выполнены на разделов, размещенных на контроллерах домена.  
  
Чтобы определить, если вам нужно будет помещать контроллера регионального домена в расположении, сравните стоимость входа трафик, создаваемый по расположению без контроллер домена или стоимость трафика репликации, созданные размещения контроллера домена в расположении.  
  
Например рассмотрим сети, имеющей филиалов, подключенных по медленному каналу связи в главном офисе и в какие контроллеры домена можно легко добавить. Если дневной трафик просмотра входа и каталогов в несколько пользователей на удаленный сайт вызывает дополнительные сетевого трафика, чем репликацию всех данных компании на ветвь, подумайте о добавлении контроллера домена в ветви.  
  
Что сокращает затраты на обслуживание контроллеров домена важнее, чем сетевой трафик, централизовать контроллеров домена для этого домена и не размещать с контроллеров регионального домена или рассмотрите возможность разместить RODC в расположении.
