# <a name="planning-operations-master-role-placement"></a>Планирование размещения роли хозяина операций

>Область применения: Windows Server 2016, Windows Server 2012 R2, Windows Server 2012

Доменных служб Active Directory (AD DS) поддерживает с несколькими хозяевами репликации данных каталога, что означает любой контроллер домена может принимать directory изменения и реплицировать изменения на другие контроллеры домена. Тем не менее некоторые изменения, например изменения схемы, не рекомендуется выполнять в данном режиме. По этой причине некоторые контроллеры домена, называется Хозяева операций ролями ответственность за принимать запросы на определенные определенные изменения.  
  
> [!NOTE]  
> Владельцев роли хозяина операций должен иметь возможность записи некоторые сведения в базу данных Active Directory. Вследствие особенностей устройства только для чтения базы данных Active Directory на контроллере домена только для чтения (RODC) RODC нельзя использовать в качестве владельцев роли хозяина операций.  
  
Существует три этих ролей (известные также как гибкие операции с единым хозяином или FSMO) в каждом домене:  
  
-   Хозяином операций эмулятора основного домена (PDC) контроллеру обрабатывает все обновления пароля.  
  
-   Хозяина операций относительных Идентификаторов (RID) поддерживает глобальный пул RID для домена и выделяет локального пулы идентификаторов RID, чтобы все контроллеры домена, убедитесь, что все субъекты безопасности, созданные в домене уникальный идентификатор.  
  
-   Хозяин операций инфраструктуры для данного домена ведет список субъекты безопасности из других доменов, которые являются членами групп в рамках своего домена.  
  
Помимо на роли хозяина операций три уровня домена роли хозяина операций два существует в каждом лесу:  
  
-   Хозяин операций схемы регулирует изменений в схеме.  
  
-   Хозяин именования доменов добавляет и удаляет доменов и другие разделы каталога (например, разделы приложений доменных имен (DNS)) и из леса.  
  
Разместить контроллеры домена, размещения эти роли хозяина операций в областях высокой надежности сети и убедитесь, что эмулятор основного контроллера домена и хозяином RID доступны постоянно.  
  
Владельцев роли хозяина операций автоматически назначаются при создании первого контроллера домена в данном домене. Две роли уровня леса (хозяина схемы и хозяина именования доменов), назначаются первого контроллера домена, созданный в лесу. Кроме того первый контроллер домена, можно создать в домене назначаются три роли уровня домена (хозяина RID, хозяин инфраструктуры и эмулятор основного контроллера домена).  
  
> [!NOTE]  
> Назначения владельца роли хозяина автоматического операций вносятся только при создании нового домена после понижения роли текущим владельцем роли. Другие изменения для владельцев ролей требуется запускать с правами администратора.  
  
Эти назначения роли хозяина операций автоматического может привести к очень высокая загрузка ЦП на первом контроллере домена в лесу или домене. Чтобы избежать этого, назначьте операции (передачи) роли хозяина различными контроллерами домена в лесу или домене. Установите контроллеры домена, что узел роли хозяина операций в областях, где сети надежный и где Хозяева операций может быть доступно все остальные контроллеры домена в лесу.  
  
Следует также указать ждущий режим (альтернативный) операций роли хозяина образцы для всех операций. Режим ожидания хозяина операций, контроллеры домена, к которым может переноса роли хозяина операций в случае отказа исходного владельцев ролей. Убедитесь, что режим ожидания хозяина операций прямых партнеров репликации хозяев операций фактические.  
  
## <a name="planning-the-pdc-emulator-placement"></a>Планирование размещения эмулятора основного контроллера домена  
Эмулятор PDC обрабатывает изменения пароля клиентов. Только один контроллер домена выступает в роли эмулятора основного контроллера домена в каждом домене леса.  
  
Даже в том случае, если все контроллеры домена будут обновлены до Windows 2000, Windows Server 2003 и Windows Server 2008, а домен работает в основном режим работы Windows 2000, эмулятор PDC получает предпочтительную репликацию изменений паролей, выполняемые на другие контроллеры домена в домене. Если пароль был недавно изменен, это изменение времени для репликации на каждом контроллере домена в домене. При сбое проверки подлинности входа в систему на контроллере домена, из-за неправильного пароля, этот контроллер домена пересылает запрос на проверку подлинности в эмуляторе PDC перед тем, как принять или отклонить попытки входа в систему.  
  
Поместите эмулятор основного контроллера домена в расположении, которое содержит большое число пользователей из этого домена для перенаправления операций при необходимости пароль. Кроме того, что местоположение сетей с качественным подключением в других местах, чтобы свести к минимуму задержку репликации.  
  
Лист для документирования сведения о которых вы планируете разместить эмуляторы PDC и число пользователей для каждого домена, который представлен в каждом расположении, в разделе задания вспомогательные средства для Windows Server 2003 Deployment Kit ([https://go.microsoft.com/fwlink/?LinkID=102558](https://go.microsoft.com/fwlink/?LinkID=102558) ), загрузите Job_Aids_Designing_and_Deploying_Directory_and_Security_Services.zip и откройте (DSSTOPO_4.doc) размещения контроллера домена.  
  
Вам потребуется обратиться к информации о местах, в которых вам нужно будет помещать эмуляторы PDC, когда развертывание региональных доменов. Дополнительные сведения о развертывании региональных доменов см. в разделе [развертывание Windows Server 2008 региональных доменов](https://technet.microsoft.com/library/cc755118.aspx).  
  
## <a name="requirements-for-infrastructure-master-placement"></a>Требования для размещения хозяина инфраструктуры  

Хозяин инфраструктуры обновляет имена субъектов безопасности из других доменов, которые добавляются к группам в собственном домене. Например если пользователей из одного домена является членом группы в второй домен и имя пользователя можно изменить в первый домен, второй домен не уведомляется, имя пользователя должны быть обновлены в список членов группы. Так как контроллеры домена в одном домене, не реплицируются субъекты безопасности на контроллеры домена в другом домене, второй домен никогда не будет знать об изменениях в отсутствие хозяина инфраструктуры.  
  
Хозяин инфраструктуры постоянно отслеживает членство в группе, поиск субъекты безопасности из других доменов. Если найден, он проверяет с доменом субъекта безопасности, чтобы убедиться, что информация обновляется. Если данные устарели, хозяин инфраструктуры выполняет обновление и затем реплицирует изменения на другие контроллеры домена в своем домене.  
  
Существует два исключения применяются к данному правилу. Во-первых Если все контроллеры домена, серверы глобального каталога контроллера домена, на котором размещена роль хозяина инфраструктуры незначительные, поскольку глобальные каталоги реплицируют обновленные сведения независимо от домена, к которому они принадлежат. Во-вторых Если лес содержит только один домен, контроллер домена, на котором размещена роль хозяина инфраструктуры незначительные, так как субъекты безопасности из других доменов не существуют.  
  
Не следует размещать хозяина инфраструктуры на контроллер домена, который также является сервером глобального каталога. Если хозяин инфраструктуры и глобальный каталог на одном контроллере домена, хозяин инфраструктуры не будет работать. Хозяин инфраструктуры никогда не будет найти данные, истек срок действия; Таким образом его никогда не будет реплицировать любые изменения на другие контроллеры домена в домене.  
  
## <a name="operations-master-placement-for-networks-with-limited-connectivity"></a>Размещение для сетей хозяев операций с ограниченные возможности подключения  
Имейте в виду, что если у вашей среде есть центральное расположение или узловому сайту, в которой можно разместить владельцев роли хозяина операций, некоторых операций контроллера домена, которые зависят от доступности этих операций хозяина роли, которую владельцы может быть нарушена.  
  
Предположим, что организация создает сайты A, B, C, и связях сайтов D. существует между A и B, между B и C, а также между C и D. подключение к сети точно отражает сетевое подключение связи сайтов. В этом примере все операции роли хозяина помещаются в сайта и возможность **установить мост для всех связей сайтов** не выбран.  
  
Несмотря на то, что этой настройки приводит к успешного выполнения репликации между всеми сайтами, функции роли хозяина операций имеют следующие ограничения:  
  
-   Контроллеры домена в сайтах C и D не удается получить доступ к эмулятор основного контроллера домена в сайте A для обновления пароля или на наличие пароль, который был недавно обновлен.  
  
-   Контроллеры домена в сайтах C и D не удается получить доступ к хозяина RID в узле A для получения начального пула относительных ИДЕНТИФИКАТОРОВ после установки Active Directory и обновить пулы RID, как они становятся исчерпаны.  
  
-   Контроллеры домена в сайтах C и D невозможно добавить или удалить каталога, DNS или разделы пользовательских приложений.  
  
-   Контроллеры домена в сайтах C и D нельзя вносить изменения схемы.
