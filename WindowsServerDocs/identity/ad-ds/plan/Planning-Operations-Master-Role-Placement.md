# <a name="planning-operations-master-role-placement"></a>Планирование размещения роли хозяина операций

>Область применения. Windows Server 2016, Windows Server 2012 R2, Windows Server 2012

Доменных служб Active Directory (AD DS) поддерживает с несколькими хозяевами репликацией данных каталогов, который означает, что любой контроллер домена может принимать изменения каталога и реплицирует изменения на все остальные контроллеры домена. Тем не менее некоторые изменения, такие как изменение схемы, не рекомендуется выполнять в данном режиме. По этой причине некоторые контроллеры домена, известный как Хозяева операций ролями ответственность за принятие запросов для некоторых определенных изменений.  
  
> [!NOTE]  
> Владельцев роли хозяина операций должны иметь возможность записать некоторые сведения в базу данных Active Directory. Из-за характера только для чтения базы данных Active Directory на контроллере домена только для чтения (RODC) **RODC не может выступать в качестве владельцев роли хозяина операций**.  
  
В каждом домене существует три к ролям хозяина операций (известные также как гибкие операции с единым хозяином или FSMO):  
  
- Хозяин операций эмулятора основного контроллера основного домена обрабатывает все обновления пароля.  

- Хозяине операций относительного идентификатора (RID) поддерживает глобальный пул RID для домена и выделяет локального пулы RID на все контроллеры домена, обеспечить уникальный идентификатор для всех субъектов безопасности, созданная в домене.  
- Хозяином операций инфраструктуры для данного домена ведет список субъектов безопасности из других доменов, которые являются членами групп в границах своего домена.  

В дополнение к роли хозяина операций три уровня домена два этих ролей существует в каждом лесу:  
  
- Хозяин операций схемы определяет изменения схемы.  
- Хозяин именования доменов, добавляет и удаляет доменов и других разделов каталога (например, разделы приложений доменных имен (DNS)) и из леса.  
  
Разместить контроллеры домена, размещение этих ролей хозяина операций в областях, где высокой надежности сети и убедитесь, что эмулятор основного контроллера домена и хозяином RID доступны постоянно.  
  
При создании первого контроллера домена в заданном домене владельцев роли хозяина операций назначаются автоматически. Две роли уровня леса (хозяина схемы и хозяина именования доменов) назначаются создан в лесу первого контроллера домена. Кроме того для первого контроллера домена в домене назначаются три роли уровня домена (хозяина RID, хозяин инфраструктуры и эмулятор основного контроллера домена).  
  
> [!NOTE]  
> Назначения владельца роли хозяина автоматической операции выполняются только в том случае, когда создается новый домен, и в процессе понижения роли текущего владельца роли. Все другие изменения владельцами роли нужно инициированной администратором.  
  
Эти назначения роли хозяина операций автоматического может привести к очень высокая загрузка ЦП на первом контроллере домена в лесу или домену. Чтобы избежать этого, назначьте операций (передачи) роли хозяина различными контроллерами домена в лесу или домене. Разместите контроллеры домена, что узел роли хозяина операций в областях, где сети является надежным и где хозяев операций может быть доступно все контроллеры домена в лесу.  
  
Следует также указать резервный (сопроводительный) операции роли хозяина образцов для всех операций. Резервный хозяин являются контроллерами домена, к которым удалось перенести роли хозяина операций, в случае если сбой исходной владельцев ролей. Убедитесь, что режим ожидания хозяина операций партнеров прямого репликационного хозяев фактических операций.  
  
## <a name="planning-the-pdc-emulator-placement"></a>Планирование размещения эмулятора основного контроллера домена

Эмулятор PDC обрабатывает изменения пароля клиента. Только один контроллер домена выступает в роли эмулятора основного контроллера домена в каждом домене в лесу.  
  
Даже в том случае, если все контроллеры домена будут обновлены до Windows 2000, Windows Server 2003 и Windows Server 2008, а домен работает в собственном режим работы Windows 2000, эмулятор основного контроллера домена получает предпочтительную репликацию, сведений о смене паролей выполняется считываются другими контроллерами домена в домене. Если пароль был недавно изменен, это изменение занимает время для репликации на каждом контроллере домена в домене. При сбое проверки подлинности на контроллере домена, из-за неправильного пароля, этот контроллер домена перенаправляет запрос на проверку подлинности в эмуляторе PDC перед тем, как принимать или отклонять попытки входа в систему.  
  
Установите эмулятор основного контроллера домена в расположение, которое содержит большое количество пользователей из этого домена для пересылки операций, при необходимости пароль. Кроме того убедитесь в том, что расположение хорошо подключены в другие расположения, чтобы свести к минимуму задержки репликации.  
  
Лист, чтобы помочь в документировании сведения о которых вы планируете размещать эмуляторы PDC и число пользователей для каждого домена, которые представлены в каждом расположении, см. в разделе Задание вспомогательные средства для Windows Server 2003 Deployment Kit ([ https://go.microsoft.com/fwlink/?LinkID=102558 ](https://go.microsoft.com/fwlink/?LinkID=102558)), загрузите Job_Aids_Designing_and_Deploying_Directory_and_Security_Services.zip и открыть размещение контроллера домена (DSSTOPO_4.doc).  
  
Вам потребуется обратиться к сведениям о расположениях, в которых вам потребуется разместить эмуляторы PDC, когда развертывание региональных доменов. Дополнительные сведения о развертывании региональных доменов см. в разделе [развертывание Windows Server 2008 региональных доменов](https://technet.microsoft.com/library/cc755118.aspx).  
  
## <a name="requirements-for-infrastructure-master-placement"></a>Требования для размещения основной инфраструктуры  

Хозяин инфраструктуры обновляет имена участников безопасности из других доменов, которые будут добавлены в группы в своем собственном домене. Например если пользователя из одного домена является членом группы в второй домен и имя пользователя изменится в первый домен, второй домен не уведомляется, необходимо обновить имя пользователя в список членства в группе. Так как контроллеры домена в одном домене субъекты безопасности не реплицируются на контроллеры домена в другом домене, второй домен никогда не узнает о изменения при отсутствии хозяина инфраструктуры.  
  
Хозяин инфраструктуры постоянно отслеживает членство в группе, ищете субъекты безопасности из других доменов. Если таковая найдена, проверяется с помощью субъекта безопасности домена, чтобы убедиться, что данные обновляются. Если данные устарели, хозяин инфраструктуры выполняет обновление и затем реплицирует изменения на другие контроллеры домена в своем домене.  
  
Существует два исключения применяются к этому правилу. Во-первых Если все контроллеры домена, серверы глобального каталога контроллера домена, на котором размещена роль хозяина инфраструктуры не играет роли, так как глобальные каталоги реплицировать обновленную информацию независимо от домена, к которому они принадлежат. Во-вторых Если в лесу есть только один домен, контроллер домена, на котором размещена роль хозяина инфраструктуры не играет роли, так как субъекты безопасности из других доменов не существует.  
  
Не размещайте хозяина инфраструктуры на контроллере домена, который также является сервером глобального каталога. Если хозяин инфраструктуры и глобального каталога находятся на одном контроллере домена, хозяине инфраструктуры не будет работать. Хозяин инфраструктуры никогда не будет найти данные, которые устарели; Таким образом он никогда не будет реплицировать любые изменения на другие контроллеры домена в домене.  
  
## <a name="operations-master-placement-for-networks-with-limited-connectivity"></a>Хозяина операций размещения для сетей с ограниченным подключением

Имейте в виду, что у среды центрального расположения или сайта центра, в котором можно разместить владельцев роли хозяина операций, определенных операций контроллера домена, которые зависят от доступности этих операций главные роли владельцев может снизиться.  
  
Например, предположим, что организация создает сайтами A, B, C, и связей сайтов D. существует между A и B, B и C, а также между C и D. подключения к сети точно отражает сетевое подключение связей сайтов. В этом примере все операции, роли хозяина помещаются в сайта и возможность **мостами все ссылки на узел** не выбран.  
  
Несмотря на то, что эта конфигурация приведет к успешной репликации между всеми сайтами, функции роль хозяина операций имеют следующие ограничения:  
  
- Контроллеры домена на сайтах, C и D не может получить доступ к эмулятора основного контроллера домена на сайте А обновлении пароля или на наличие пароль, который был недавно обновлен.  
- Контроллеры домена на сайтах, C и D не может получить доступ к хозяина RID в веб-узел A для получения начального пула относительных ИДЕНТИФИКАТОРОВ после установки Active Directory и обновить пулы RID, так как они наступить в результате.  
- Контроллеры домена на сайтах, C и D невозможно добавить или удалить directory, DNS или разделов пользовательских приложений.  
- Контроллеры домена на сайтах, C и D не может вносить изменения схемы.  
  
Рабочий лист, который поможет в планировании размещения роли хозяина операций, см. в разделе [задания вспомогательные средства для Windows Server 2003 Deployment Kit](https://go.microsoft.com/fwlink/?LinkID=102558), скачайте Job_Aids_Designing_and_Deploying_Directory_and_Security_Services.zip и открыть Размещение контроллера домена (DSSTOPO_4.doc).  
  
Необходимо будет использовать эту информацию при создании корневого домена леса и региональных доменов. Дополнительные сведения о развертывании корневого домена леса, см. в разделе Развертывание [развертывание корневого домена леса Windows Server 2008](https://technet.microsoft.com/library/cc731174.aspx). Дополнительные сведения о развертывании региональных доменов см. в разделе [развертывание Windows Server 2008 региональных доменов](https://technet.microsoft.com/library/cc755118.aspx).
