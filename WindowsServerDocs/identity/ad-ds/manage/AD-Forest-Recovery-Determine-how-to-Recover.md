# <a name="determine-how-to-recover-the-forest"></a>Определите, как восстановить леса  

>Область применения: Windows Server 2016, Windows Server 2012 и 2012 R2, Windows Server 2008 и 2008 R2

 Восстановление всего леса Active Directory включает в себя восстановления из резервной копии или повторной установки доменных служб Active Directory (AD DS) на каждом контроллере домена (DC) в лесу. Восстановление леса каждого домена в лесу восстанавливает состояние во время последнего резервного копирования доверенным. Следовательно операцию восстановления приведет к потере по крайней мере следующие данные Active Directory:  
  
-   Все объекты (например, пользователи и компьютеры), добавленные после последнего резервного копирования доверенным  
  
-   Все обновления, которые были внесены в существующих объектах с момента последнего доверенного резервного копирования  
  
-   Все изменения, внесенные в раздел конфигурации или раздела схемы в Доменных службах Active Directory (например, изменения схемы) с момента последнего резервного копирования доверенным  
  
 Для каждого домена в лесу должен быть известен пароль учетной записи администратора домена. Предпочтительно это пароль встроенной учетной записи администратора, которая не должны отключаться. Кроме того, необходимо знать пароль DSRM, чтобы выполнить восстановление состояния системы контроллера домена. В общем случае рекомендуется архивировать учетной записи администратора и журнал пароля DSRM в безопасном месте для, а резервные копии действительны, то есть в пределах срока жизни захоронения или удаленного объекта срока жизни, если включена Корзина Active Directory. Можно также синхронизировать Пароль DSRM с учетной записью пользователя домена для упрощения запомнить. Дополнительные сведения см. в статье БАЗЫ знаний [961320](https://support.microsoft.com/kb/961320). Синхронизация учетной записи DSRM должна выполняться до восстановления леса, в рамках подготовки.  
  
> [!NOTE]
>  Учетная запись администратора является членом встроенной группы администраторов по умолчанию, как являются группы "Администраторы домена" и "Администраторы предприятия". Эта группа имеет полный контроль над все контроллеры домена в домене.  
  
## <a name="determining-which-backups-to-use"></a>Определить, какие операций резервного копирования  
 Регулярно выполнять резервное копирование по крайней мере два доступного для записи контроллеров домена для каждого домена, поэтому у вас есть несколько резервных копий на выбор. Обратите внимание, что резервная копия контроллера домена только для чтения (RODC) нельзя использовать для восстановления для записи контроллера домена. Мы рекомендуем восстановление контроллеров домена с помощью резервных копий, которые были выполнены несколько дней до возникновения сбоя. Как правило необходимо определить компромисс между recentness и safeness восстановленных данных. Выбор более актуальную версию резервной копии восстанавливает полезных данных, но может увеличить риск повторного добавления опасных данных в восстановленный лес.  
  
 Восстановление резервных копий состояния системы зависит от исходной операционной системы и сервера резервной копии. Например не следует восстанавливать резервной копии состояния системы на другой сервер. В этом случае может появиться следующее предупреждение:  
  
 «Указанной резервной копии является отличается от текущего сервера. Не рекомендуется выполнять восстановления состояния системы из резервной копии на другом сервере, так как сервер может стать непригодным. Уверены, что вы хотите использовать эту резервную копию для восстановления текущего сервера?»  
  
 Если необходимо восстановить Active Directory на другом оборудовании, создайте резервные копии полного сервера и вы планируете выполнить полное восстановление сервера.  
  
> [!IMPORTANT]
>  Начиная с Windows Server 2008, не поддерживается восстановление резервной копии состояния системы в новую установку Windows Server, на новое оборудование или на том же оборудовании. В случае переустановки Windows Server на том же оборудовании, как это рекомендуется делать этого руководства, можно восстановить контроллер домена в следующем порядке:  
>   
>  1.  Восстановление полного сервера для восстановления операционной системы и всех файлов и приложений.  
> 2.  Выполните восстановление состояния системы с помощью wbadmin.exe, чтобы пометить SYSVOL как заслуживающие доверия.  
>   
>  Дополнительные сведения см в статье базы Знаний Майкрософт [249694](https://support.microsoft.com/kb/249694).  
  
 Если время возникновения ошибки остается неизвестной, исследуйте для идентификации резервных копий, которые удерживайте последнее состояние безопасном леса. Этот подход является менее предпочтительным. Таким образом настоятельно рекомендуется сохранить подробные журналы о состоянии работоспособности службы AD DS ежедневно, чтобы в случае сбоя всего леса, можно определить примерное время сбоя. Вы также должны поддерживать локальную копию резервные копии для восстановления быстрее.  
  
 Если Корзина Active Directory включена, резервного копирования времени жизни равно **deletedObjectLifetime** значение или **tombstoneLifetime** значение, меньшее. Дополнительные сведения см. в разделе [Корзина Active Directory Step-by-Step руководство](https://go.microsoft.com/fwlink/?LinkId=178657) (https://go.microsoft.com/fwlink/?LinkId=178657).  
  
 В качестве альтернативы можно также использовать подключение средство Active Directory базы данных (Dsamain.exe) и средство облегченного доступа к каталогам LDAP Directory Access Protocol (), например Ldp.exe или Active Directory-пользователи и компьютеры, для идентификации какую архивную копию имеет последнее состояние безопасном леса. Средство подключение базы данных Active Directory, которое входит в состав Windows Server 2008 и более поздних операционных системах Windows Server, предоставляет данные Active Directory, которые хранятся в резервное копирование или создание моментальных снимков как LDAP-серверу. Затем можно использовать инструмент LDAP для просмотра данных. Данный подход имеет преимущество не требует перезагрузки любым контроллером домена в каталоге служб восстановления режим (DSRM) для просмотра содержимого резервной копии AD DS.  
  
 Дополнительные сведения об использовании подключение средство базы данных Active Directory см. в разделе [средства Подключение базы данных Active Directory Step-by-Step руководство](https://technet.microsoft.com/library/cc753609\(WS.10\).aspx).  
  
 Можно также использовать **моментальных снимков ntdsutil** команду для создания моментальных снимков базы данных Active Directory. С помощью планирования задач периодически создавать моментальные снимки, можно получить дополнительные копии базы данных Active Directory, со временем. Чтобы точнее определить, когда произошел сбой леса, а затем выберите наиболее архив для восстановления можно использовать эти экземпляры. Создание моментальных снимков, необходимо использовать версию **ntdsutil**, поставляемая с Windows Server 2008 или удаленного администрирования сервера средства (RSAT) для Windows Vista или более поздней версии. Целевой контроллер домена можно запустить в любой версии Windows Server. Дополнительные сведения об использовании **моментальных снимков ntdsutil** см. в разделе [моментального снимка](https://technet.microsoft.com/library/cc731620\(WS.10\).aspx).  
  
  
## <a name="determining-which-domain-controllers-to-restore"></a>Определение контроллеров домена для восстановления  
 Простота процесс восстановления является важным фактором при выборе какого контроллера домена для восстановления. Рекомендуется иметь выделенный контроллера домена для каждого домена, основной контроллер домена для восстановления. Выделенный восстановления, контроллер домена упрощает надежно спланировать и выполнить восстановления леса, так как вы используете ту же конфигурацию источника, который использовался для выполнения восстановления тестов. Сценарии восстановления и не будут конкурировать с различными конфигурациями, такие как ли контроллер домена являющегося хозяином операций или не или ли он является сервером глобального Каталога или DNS или нет.  
  
> [!NOTE]
>  Во время восстановления владельца роли хозяина операций простоты, не рекомендуется, некоторые организации могут выбрать восстановления одного другие преимущества. Например восстановление хозяином RID может предотвратить проблемы с управлением идентификаторов RID во время восстановления.  
  
 Выберите контроллер домена, которое наилучшим образом удовлетворяет следующим критериям:  
  
-   Контроллер домена, который доступен для записи. Является обязательным.  
  
-   Контроллер домена под управлением Windows Server 2012 в качестве виртуальной машины на гипервизоре, поддерживающем VM-GenerationID. Этот контроллер домена можно использовать как источник для клонирования.  
  
-   Контроллер домена доступен, физической или виртуальной сети, который лучше находятся в центре обработки данных. Таким образом можно легко изолировать его от сети во время восстановления леса.  
  
-   Контроллер домена, который имеет хорошее полного резервного копирования сервера. Удачной резервной копии является резервной копией успешно восстановлены, переводится в течение нескольких дней до сбоя, который содержит как можно значительно полезных данных.  
  
-   Контроллер домена, который был сервером доменных имен (DNS) до сбоя. Это экономит время, необходимое для переустановки DNS-сервера.  
  
-   При использовании служб развертывания Windows, выберите контроллер домена, который не настроен для использования сетевой разблокировки BitLocker. В этом случае сетевой разблокировки BitLocker не поддерживается для первого контроллера домена, восстановление из резервной копии во время восстановления леса.  
  
     Сетевая разблокировка BitLocker как *только* предохранитель ключа *нельзя* использоваться на контроллерах домена с развернутым служб развертывания Windows (WDS) тем, что при этом таким образом в случае которых требует первого контроллера домена Active Directory и службы развертывания Windows работают для разблокировки. Но перед восстановлением первого контроллера домена Active Directory еще не доступен для служб развертывания Windows, поэтому он не может разблокировать.  
  
     Чтобы определить, если контроллер домена настроен для использования сетевой разблокировки BitLocker, проверьте, что сертификат сетевой разблокировки определяется в следующем разделе реестра:  
  
     HKEY_LOCAL_MACHINESoftwarePoliciesMicrosoftSystemCertificatesFVE_NKP  
  
 Ведение меры безопасности при обработке или восстановлении резервной копии файлов, включающих Active Directory. Срочности, сопровождающее восстановления леса непреднамеренно может привести к пропуску рекомендации по обеспечению безопасности. Дополнительные сведения см. раздел «Установление домена контроллера резервного копирования и восстановления стратегии» в [рекомендации по защите установок Active Directory и операций Day-to-Day: часть II](https://technet.microsoft.com/library/bb727066.aspx).

 Определите структуре текущего леса, определяя все домены в лесу. Для создания списка всех контроллеров домена в каждом домене, особенно контроллеры домена, имеющие резервные копии и виртуализированных контроллеров домена, которые могут быть источником клонирования. Список контроллеров домена для корневого домена леса будут наиболее важным, поскольку сначала будут восстановить этот домен. После восстановления корневого домена леса могут получить список из других доменов, контроллеры доменов и сайтов в лесу, с помощью оснастки Active Directory.

После восстановления роль эмулятора основного контроллера домена будет захвачен на что сервера и могут быть добавлены в группу клонируемые контроллеры домена для домена.

Но ли контроллер домена является также, что сервер глобального каталога не является определяющим фактором, что начиная с Windows Server 2012, все контроллеры домена, серверы глобального каталога по умолчанию и удаление и добавление глобального каталога, после восстановления рекомендуется как часть процесса восстановления леса, в любом случае.  
  
## <a name="recover-the-forest-in-isolation"></a>Восстановление леса в изоляции  
 Оптимальный сценарий — завершение работы всех доступного для записи контроллеров домена до первого восстановленного контроллера домена переводится обратно в рабочую среду. Это гарантирует, что любой опасных данных, не реплицированы обратно на восстановленные леса. Это особенно важно завершить работу всех владельцев роли хозяина операций.  
  
> [!NOTE]
>  Возможны ситуации, когда перемещать первого контроллера домена, который вы планируете восстановить для каждого домена в изолированной сети, обеспечивая при этом другие контроллеры домена, чтобы оставаться в оперативном режиме, чтобы свести к минимуму время простоя. Например если выполняется восстановление в случае сбоя схемы обновления, вы можете сохранить контроллеры домена под управлением в рабочей сети во время выполнения действия по восстановлению в изоляции.  
  
 Если вы используете виртуализированных контроллеров домена, вы их можно переместить виртуальную сеть, которая изолирована от рабочей сети место выполнения восстановления. Перемещение виртуализированных контроллеров домена к отдельной сети, предоставляет два преимущества.  
  
-   Восстановленные контроллеры домена не смогут повторения неполадку, вызвавшую восстановления леса, поскольку они изолированы.  
  
-   Клонирование виртуализированного контроллера домена можно выполнять на отдельной сети, чтобы ряд критически важных контроллеров домена могут работать под управлением и тестирования, прежде чем они будут вернуть рабочей сети.  
  
 Если контроллеры домена работают под управлением на физическом оборудовании, отсоединяйте сетевой кабель первого контроллера домена, который вы планируете восстановить в корневом домене леса. Если это возможно также отключите сетевые кабели другие контроллеры домена. Это предотвращает контроллеры домена репликации, если они запущены случайно во время восстановления леса.  
  
 Большой леса, который распределяется в нескольких местах может быть трудно гарантировать, что все контроллеры домена, доступного для записи, завершают работу. По этой причине действия по восстановлению — например Сброс учетной записи компьютера и учетной записи krbtgt, в дополнение к очистку метаданных — позволяет убедиться, что восстановленный доступного для записи контроллеров домена не выполнять репликацию с опасность для записи контроллеров домена (в случае некоторых подключены по-прежнему в лесу).  
  
 Тем не менее только с отключением доступного для записи контроллеры домена можно вы гарантирует, что репликация не возникает? Таким образом когда это возможно, следует развертывать технологии удаленного управления, которые могут помочь на завершение работы и физически изолировать доступного для записи контроллеров домена во время восстановления леса.  
  
 RODC может продолжать работать при доступного для записи контроллеров домена в автономном режиме. Других контроллерах домена будет реплицировать любые изменения напрямую из любой RODC — особенно, схемы и конфигурации контейнера изменения не, поэтому они не представлять риск же как для записи контроллеров домена во время восстановления. После для записи контроллеров домена, восстановленные и сети, следует перестроить RODC.  
  
 Чтобы разрешить доступ к локальным ресурсам, которые хранятся на своих сайтах во время операции восстановления выполняются параллельно продолжит RODC. Локальные ресурсы, которые не кэшированы на RODC будет запросы проверки подлинности, перенаправляются доступного для записи контроллера домена. Эти запросы будут отклонены, так как для записи контроллеры домена находятся в автономном режиме. Некоторые операции, например об изменении пароля также не работают, пока восстановить доступного для записи контроллеров домена.  
  
 При использовании архитектуры сети и звезда сосредоточить внимание сначала на восстановление для записи контроллеров домена на сайтах центра. Более поздней версии можно перестройте RODC в удаленных сайтах.
