Этот раздел включает следующие действия:  

- [Восстановление первого контроллера домена с возможностью записи в каждом домене](#Restore-the-first-writeable-domain-controller-in-each-domain)  
- [Повторное подключение каждого контроллера домена с возможностью записи восстановленной к сети](#Reconnect-each-restored-writeable-domain-controller-to-a-common-network)  
- [Добавление глобального каталога на контроллер домена в корневом домене леса](#Add-the-global-catalog-to-a-domain-controller-in-the-forest-root-domain)  

## <a name="restore-the-first-writeable-domain-controller-in-each-domain"></a>Восстановление первого контроллера домена с возможностью записи в каждом домене  

Начиная с возможностью записи контроллера домена в корневом домене леса, выполните действия, описанные в этом разделе для восстановления первого контроллера домена. Корневого домена леса важно, поскольку в нем хранятся в группы "Администраторы схемы" и "Администраторы предприятия". Она также помогает поддерживать Иерархия доверия в лесу. Кроме того корневого домена леса, обычно содержит корневой сервер DNS для пространств имен DNS леса. Следовательно зону DNS, интегрированной в Active Directory для этого домена содержит записи ресурсов псевдонима (CNAME) для всех других контроллеров домена в лесу (который необходимы для репликации) и записи ресурсов DNS глобального каталога. 
  
После восстановления корневого домена леса, повторите эти шаги, чтобы восстановить остальные домены в лесу. Можно восстановить более одного домена одновременно; Тем не менее всегда восстановить родительского домена, перед восстановлением дочерних, чтобы предотвратить прерывание Иерархия доверия или разрешение DNS-имен. 
  
Для каждого домена, можно восстановить восстановление только один доступный для записи контроллер домена из резервной копии. Это самая важная часть восстановления, так как контроллер домена должен иметь базу данных, не затрагивается, причины леса в переход на другой. Очень важно иметь доверенный резервную копию, тщательно тестируется, прежде чем он появился в рабочей среде. 
  
Затем выполните следующие действия. Процедуры для выполнения определенных шагов [восстановление леса AD - процедуры](AD-Forest-Recovery-Procedures.md). 
  
1. Если вы планируете восстановить физический сервер, убедитесь, что сетевой кабель целевого контроллера домена не подключен и таким образом не подключен к рабочей сети. Для виртуальной машины можно удалить сетевой адаптер или использовать сетевой адаптер, подключенный к другой сети, который позволяет проверить процесс восстановления, а изолированной от рабочей сети. 
  
2. Так как это первый доступный для записи контроллер домена в домене, необходимо выполнить принудительное восстановление AD DS и SYSVOL Полномочное восстановление. Операция восстановления должна быть завершена до использованием учитывал Active Directory резервной копии и восстановление приложения, такие как системы архивации данных Windows Server (то есть не следует восстанавливать контроллера домена с помощью неподдерживаемые методы, такие как восстановление моментального снимка виртуальной Машины). 
   - Полномочное восстановление SYSVOL является обязательным, поскольку репликации для репликации SYSVOL, что папка должна быть запущена после восстановления после аварии. Все последующие контроллеры домена, которые добавляются в домене необходимо повторно синхронизировать свои папки SYSVOL с копию папки, который был выбран, достоверными, прежде чем папке может быть объявлена. 

   > [!CAUTION]
   > Выполните операцию восстановления, заслуживающие доверия (или первичный) SYSVOL только для первого контроллера домена для восстановления в корневом домене леса. Неправильно операций основное восстановление каталога SYSVOL на другие контроллеры домена приводит к конфликтам репликацию данных SYSVOL. 

   - Существует два варианта выполните непринудительное восстановление AD DS и SYSVOL Полномочное восстановление:  
   - Выполняется восстановление всего сервера и затем принудительно заслуживающую доверия синхронизацию SYSVOL. Подробные инструкции см. в разделе [выполнение восстановления всего сервера](AD-Forest-Recovery-Perform-a-Full-Recovery.md) и [выполнения заслуживающую доверия синхронизацию SYSVOL с репликацией DFSR](AD-Forest-Recovery-Authoritative-Recovery-SYSVOL.md). 
   - Выполните восстановление всего сервера, следуют восстановление состояния системы. Этот параметр необходимо заранее создать оба типа резервных копий: полного сервера резервной копии и резервной копии состояния системы. Подробные инструкции см. в разделе [выполнение восстановления всего сервера](AD-Forest-Recovery-Perform-a-Full-Recovery.md) и [выполнение непринудительное восстановление доменных служб Active Directory](AD-Forest-Recovery-Nonauthoritative-Restore.md). 
  
3. После восстановления и перезапуска для записи контроллер домена, убедитесь, что сбой не обработала данные на контроллере домена. При повреждении данных контроллера домена, затем повторите шаг 2 из другой резервной копии. 
   - Если восстановленного контроллера домена размещены роли хозяина операций, может потребоваться добавить следующий параметр реестра во избежание недоступностью только после завершения репликации раздела каталога с возможностью записи AD DS:  

      **HKLM\System\CurrentControlSet\Services\NTDS\Parameters\Repl выполнение начальной синхронизации**  
  
      Создайте запись с типом данных **REG_DWORD** и значение **0**. После восстановления леса полностью, вы можете сбросить значение этого параметра **1**, что требует перезапуска контроллера домена и входящего трафика содержит роли хозяина операций для успешного AD DS и исходящую репликацию с его Известные партнеров репликации, прежде чем его объявляющий себя в качестве контроллера домена и начинает предоставлять услуги клиентам. Дополнительные сведения о требованиях к начальной синхронизации, см. в статье базы Знаний [305476](https://support.microsoft.com/kb/305476). 
  
      Только в том случае, после восстановления и проверки правильности данных, и прежде чем присоединить этот компьютер к рабочей сети по-прежнему к дальнейшим действиям. 
  
4. Если вы подозреваете, что сбой леса было связано с сетевых вторжений и атак злоумышленников, сбросьте пароли учетных записей для всех административных учетных записей, включая и членов группы "Администраторы предприятия", "Администраторы домена", "Администраторы схемы", операторов сервера, учетной записи Группы операторов и т. д. Сброс пароля учетной записи администратора должно быть завершено до дополнительного домена контроллеры могут быть установлены на следующем этапе восстановление леса. 
5. Первый восстановленного контроллера домена в корневом домене леса захватить все роли хозяина операций домена и леса. Чтобы присвоить роли хозяина операций леса необходимы учетные данные "Администраторы предприятия" и "Администраторы схемы". 
  
     В каждом дочернем домене Присвоение ролей хозяина операций домена. Несмотря на то, что может сохранить роли хозяина операций на восстановленном контроллере домена только временно захвате этих ролей обеспечивает какую контроллере домена размещается на этом этапе их в процессе восстановления леса. Как часть процесса после восстановления при необходимости можно распространять на роли хозяина операций. Дополнительные сведения о захвате ролей хозяина операций, см. в разделе [захват роли хозяина операций](AD-forest-recovery-seizing-operations-master-role.md). Рекомендации о том, где для размещения роли хозяина операций см. в разделе [Каковы Хозяева операций?](https://technet.microsoft.com/library/cc779716.aspx). 
  
6. Очистка метаданных все другие доступные для записи контроллеров домена в корневом домене леса, восстановление из резервной копии (все доступные для записи контроллеры доменов, за исключением этого первого контроллера домена). Если вы используете версию Active Directory — пользователи и компьютеры Active Directory — сайты и службы, которые входят в состав Windows Server 2008 или более поздней версии или средства удаленного администрирования сервера для Windows Vista или более поздней версии, Очистка метаданных выполняется автоматически при удалении объекта контроллера домена. Кроме того серверный объект и объект-компьютер для удаленного контроллера домена, также удаляются автоматически. Дополнительные сведения см. в разделе [Очистка метаданных удалены контроллерам домена](AD-Forest-Recovery-Cleaning-Metadata.md). 
  
     Очистка метаданных предотвращает возможно дублирование объектов параметров NTDS, если в Доменных службах Active Directory устанавливается на Контроллере домена в другом сайте. Потенциально это может также сохранить проверки согласованности знаний (KCC) процесс создания связи репликации, когда сами контроллеры домена не может присутствовать. Более того, как часть очистку метаданных будут удалены записи ресурсов DNS локатора контроллера домена для всех других контроллеров домена в домене с DNS-сервера. 
  
     Пока не будет удалена метаданные другие контроллеры домена в домене, этот контроллер домена, если бы он был хозяина RID до восстановления, не будет считать роль хозяина RID и таким образом не будет выдавать новые идентификаторы RID. Может появиться событие с кодом 16650 в системном журнале в средстве просмотра событий, указывающих этот ошибку, но вы увидите событие с Идентификатором 16648, означающий успешное выполнение на некоторое время после были обнаружены метаданные. 
  
7. Если у вас есть зон DNS, которые хранятся в Доменных службах Active Directory, убедитесь, что локальной службе DNS-сервер установлен и запущен на контроллере домена, которая была восстановлена. Если этот контроллер домена не был до сбоя леса DNS-сервер, необходимо установить и настроить DNS-сервер. 
  
    > [!NOTE]
    > Если восстановленный Контроллер работает Windows Server 2008, необходимо установить исправление, описанное в статье базы Знаний [975654](https://support.microsoft.com/kb/975654) или временно подключите сервер к изолированной сети, чтобы установить DNS-сервер. Исправление не требуется для других версий Windows Server. 
  
     В корневом домене леса Настройка восстановленный Контроллер с собственным IP-адресом (или петлевой адрес, например 127.0.0.1) в качестве его основной DNS-сервер. Этот параметр можно настроить в свойствах TCP/IP адаптер локальной сети (LAN). Это первый DNS-сервер в лесу. Дополнительные сведения см. в разделе [Настройка TCP/IP на использование DNS](https://technet.microsoft.com/library/cc779282\(WS.10\).aspx). 
  
     В каждом дочернем домене настройте восстановленный Контроллер IP-адрес первого DNS-сервера в корневом домене леса, как его основной DNS-сервер. Этот параметр можно настроить в свойствах TCP/IP сетевого адаптера. Дополнительные сведения см. в разделе [Настройка TCP/IP на использование DNS](https://technet.microsoft.com/library/cc779282\(WS.10\).aspx). 
  
     В зонах DNS _msdcs и домен удалите записи NS контроллеров домена, которые больше не существуют после очистки метаданных. Проверьте, была ли удалена записи SRV Очищенная контроллеров домена. Чтобы помочь ускорить удаление записей DNS SRV, выполните следующую команду:  
  
    ```  
    nltest.exe /dsderegdns:server.domain.tld  
    ```  
  
8. Увеличить значение доступного пула RID на 100 000. Дополнительные сведения см. в разделе [возведение значения доступных пулов RID](AD-Forest-Recovery-Raise-RID-Pool.md). Если у вас есть основания предполагать, что вызов пула RID на 100 000 недостаточно для конкретной ситуации, необходимо определить наименьшее увеличение, по-прежнему безопасно использовать. RID являются ограниченный ресурс, который не должен использоваться вверх без необходимости. 
  
     Если в домене в спустя период времени, используемом для восстановления резервной копии были созданы новые субъекты безопасности, эти субъекты безопасности могут права доступа для определенных объектов. Эти субъекты безопасности не существует после восстановления, так как восстановление, обращается к резервной копии; Тем не менее их права доступа может по-прежнему существуют. Если доступный пул относительных ИДЕНТИФИКАТОРОВ после восстановления не возникает, объекты, созданные после восстановления леса может получать одинаковые безопасности (SID) нового пользователя и сможет получать доступ к этим объектам, который не был изначально предназначен. 
  
     Чтобы проиллюстрировать это, рассмотрим пример нового сотрудника с именем Amy, которое было упомянуто во введении. Объект пользователя для Amy больше не существует после операции восстановления, так как он был создан после резервного копирования, который был использован для восстановления домена. Тем не менее прав доступа, которые были назначены для конкретного объекта-пользователя может сохраняться после операции восстановления. Если идентификатор безопасности для конкретного объекта-пользователя назначается новый объект после операции восстановления, новый объект получит эти права доступа. 
  
9. Сделать недействительным текущий пул относительных ИДЕНТИФИКАТОРОВ. Текущий пул относительных ИДЕНТИФИКАТОРОВ становится недействительным после восстановления состояния системы. Но если восстановление состояния системы не выполнено, текущий пул относительных ИДЕНТИФИКАТОРОВ необходимо сделать недействительным для предотвращения повторной выдачи идентификаторов RID из пула относительных ИДЕНТИФИКАТОРОВ, назначенный во время создания резервной копии восстановленного контроллера домена. Дополнительные сведения см. в разделе [аннулирования текущий пул относительных ИДЕНТИФИКАТОРОВ](AD-Forest-Recovery-Invaildate-RID-Pool.md). 
  
    > [!NOTE]
    > Первый раз, при попытке создать объект с ИД безопасности после недействительным пул относительных ИДЕНТИФИКАТОРОВ, вы получите ошибку. Попытка создания объекта активирует запрос нового пула относительных ИДЕНТИФИКАТОРОВ. Повторите попытку операции завершается успешно, поскольку будет выделен нового пула относительных ИДЕНТИФИКАТОРОВ. 
  
10. Сброс пароля учетной записи компьютера этот контроллер домена дважды. Дополнительные сведения см. в разделе [сброс пароля учетной записи компьютера контроллера домена](AD-Forest-Recovery-Reset-Computer-Account-DC.md). 
  
11. Сброс пароля krbtgt дважды. Дополнительные сведения см. в разделе [сброс пароля krbtgt](AD-Forest-Recovery-Resetting-the-krbtgt-password.md). 
  
     Так как журнал паролей krbtgt два пароля, сбросьте пароли дважды, чтобы исходный пароль (prefailure) из журнал паролей. 
  
    > [!NOTE]
    > Если восстановление леса в ответ на нарушения безопасности, можно сбросить пароли доверия. Дополнительные сведения см. в разделе [сброс пароля доверия на одной стороне отношения доверия](AD-Forest-Recovery-Reset-Trust.md). 
  
12. Если в лесу есть несколько доменов и восстановленный контроллер домена был сервером глобального каталога до сбоя, снимите **глобального каталога** "флажок" в свойствах параметров NTDS, чтобы удалить глобальный каталог с контроллера домена. Исключением из этого правила — это распространенный случай лес, в котором только один домен. В этом случае он не требуется удалить глобальный каталог. Дополнительные сведения см. в разделе [удаление глобального каталога](AD-Forest-Recovery-Remove-GC.md). 
  
     Путем восстановления глобального каталога из резервной копии, более свежие, чем другие резервные копии, которые используются для восстановления контроллеров домена в других доменах, может привести устаревших объектов. Рассмотрим следующий пример. В домене A DC1 восстанавливается из резервной копии, сделанной в момент времени T1. В домене Б DC2 восстанавливается из резервной копии глобального каталога, выполнялась в момент времени T2. Предположим, что T2 более поздней, чем T1, некоторые объекты были созданы T1 и T2. После восстановления эти контроллеры домена, DC2, которая является глобальным каталогом, содержит более новые данные для домена A частичные реплики, чем домен A содержит сам. DC2, таким образом, содержит устаревшие объекты, так как эти объекты не существуют на DC1. 
  
     Наличие устаревших объектов может привести к проблемам. Например сообщений электронной почты не может быть доставлено пользователю, объект пользователя был перемещен между доменами. После подключения устаревшие контроллера домена или сервер глобального каталога обратно в оперативный режим, оба экземпляра объекта пользователя отображаются в глобальном каталоге. Оба объекта имеют тот же адрес электронной почты; Таким образом не удалось доставить сообщения электронной почты. 
  
     Вторая проблема в том, что учетную запись пользователя, который больше не существует, по-прежнему могут отображаться в глобальном списке адресов. Третья проблема в том, что универсальная группа, которая больше не существует, по-прежнему могут отображаться в маркере доступа пользователя. 
  
     Если восстановление контроллера домена, который является глобальным каталогом, либо случайно или потому, что это было одиночные резервную копию, которая доверии — рекомендуется исключить возникновение замедление объектов, отключив глобального каталога, вскоре после операции восстановления Завершите. Снятия флага глобального каталога приведет к компьютеру потери всех его частичных реплик (секции) и сам relegating регулярных состояния контроллера домена. 
  
13. Служба времени Windows. В корневом домене леса настроите эмулятор основного контроллера домена для синхронизации времени из внешнего источника времени. Дополнительные сведения см. в разделе [настроить службу времени Windows в эмуляторе основного контроллера домена в корневом домене леса](https://docs.microsoft.com/previous-versions/windows/it-pro/windows-server-2008-R2-and-2008/cc731191%28v=ws.10%29). 
  
## <a name="reconnect-each-restored-writeable-domain-controller-to-a-common-network"></a>Повторное подключение каждого контроллера домена с возможностью записи восстановленной общей сети

На этом этапе необходимо восстановить один контроллер домена (и выполнить действия по восстановлению) в корневом домене леса и в каждом из оставшихся доменов. Присоединение к общей сети, которая изолирована от остальной части среды эти контроллеры домена и выполните следующие действия, чтобы проверить работоспособность леса и репликации.

> [!NOTE]
> При объединении физических контроллеров домена в изолированной сети, может потребоваться изменить их IP-адреса. Таким образом IP-адреса DNS-записей будет работать неправильно. Так как сервер глобального каталога недоступен, безопасных динамических обновлений DNS для завершится ошибкой. Так как их можно будет присоединить к новой виртуальной сети без изменения их IP-адреса виртуальных контроллеров домена в данном случае являются более выгодным. Это одна из причин, почему виртуальных контроллеров домена рекомендуется в качестве первого контроллера домена для восстановления во время восстановления леса. 
  
После проверки присоединение к рабочей сети контроллеры домена и выполните действия, чтобы проверить работоспособность репликации леса.

- Чтобы исправить разрешения имен, создать записи делегирования DNS и настройку DNS пересылки и корневых ссылок при необходимости. Запустите **repadmin/REPLSUM** проверяемый репликации между контроллерами домена. 
- Если восстановленный Контроллер не прямых партнеров репликации, восстановление репликации будет гораздо быстрее путем создания объектов временное подключение между ними. 
- Чтобы проверить очистку метаданных, запустите **Repadmin /viewlist \***  для получения списка всех контроллеров домена в лесу. Запустите **Nltest /DCList:** *< домена\>*  для получения списка всех контроллеров домена в домене. 
- Чтобы проверить работоспособность контроллера домена и DNS, выполните DCDiag /v для сообщения об ошибках на всех контроллерах домена в лесу. 

## <a name="add-the-global-catalog-to-a-domain-controller-in-the-forest-root-domain"></a>Добавление глобального каталога на контроллер домена в корневом домене леса

Глобальный каталог является обязательным для этой и другим причинам:  
  
- Чтобы включить вход для пользователей. 
- Чтобы включить службу сетевого входа в систему на контроллеры домена в каждом дочернем домене для регистрации и удаления записей на DNS-сервера в корневом домене. 
  
Несмотря на то, что он предпочтителен, корень леса DC становятся глобального каталога, можно выбрать любой из восстановленного контроллера домена, чтобы стать глобального каталога. 
  
> [!NOTE]
> Контроллер домена не будет объявлен сервером глобального каталога, только после завершения полной синхронизации всех разделов каталога в лесу. Таким образом контроллер домена должно обязательно репликации с каждым из восстановленной контроллеров домена в лесу. 
>
> Службы каталогов в журнале событий в средстве просмотра событий для события 1119 идентификатор, который указывает, что этот контроллер домена является сервером глобального каталога, или убедитесь, что следующий раздел реестра имеет значение 1:  
>
> **Полный повышения роли каталога HKLM\System\CurrentControlSet\Services\NTDS\Parameters\Global**  
  
Дополнительные сведения см. в разделе [Добавление глобального каталога](AD-Forest-Recovery-Add-GC.md). 
  
На этом этапе необходимо стабильной лес, с одного контроллера домена для каждого домена и один глобальный каталог в лесу. Следует новую резервную копию всех контроллеров домена, только что была восстановлена. Теперь можно приступать к повторному развертыванию другие контроллеры домена в лесу с установкой AD DS.
