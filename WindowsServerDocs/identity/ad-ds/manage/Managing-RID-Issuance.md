## <a name="BKMK_Manage"></a>Управление выдачей RID  
По умолчанию домен имеет емкость составляет приблизительно один миллиард субъектов безопасности, таких как пользователи, группы и компьютеры. Естественно не существует доменов с таким количеством активно используемых объектов. Однако служба поддержки пользователей Майкрософт с указанными ниже ситуациями где:  
  
-   Для подготовки программного обеспечения или сценариев администрирования случайно создавалось большое количество создания пользователей, групп и компьютеров.  
  
-   Делегированные пользователи создавали множество неиспользуемых безопасности и групп рассылки  
  
-   Большое количество контроллеров домена выполнялось понижение роли, восстановления или очистить метаданные  
  
-   Выполнялось восстановление лесов  
  
-   Часто выполнялась операция invalidateridpool.  
  
-   Значение реестра для размера блока RID, увеличивался некорректно.  
  
Во всех этих ситуациях используются относительные идентификаторы лишние, часто по ошибке. Через много лет в некоторых средах вышел за пределы идентификаторов RID и приходилось выполнять миграцию в новый домен или восстанавливать лес.  
  
Windows Server 2012 решаются проблемы с выделением идентификаторов RID, которые только стали актуальны возраст и распространенности службы каталогов Active Directory. К ним относятся улучшает ведение журнала событий, более подходящие предельные значения и возможность в экстренном случае - вдвое увеличить общий размер глобального пространства RID для домена.  
  
### <a name="periodic-consumption-warnings"></a>Периодические предупреждения об использовании  
Windows Server 2012 добавляет глобального пространства RID события, отслеживания, позволяет своевременно получать предупреждения при достижении основных контрольных точек. Модель рассчитывает десяти (10) процентов использования глобального пула и регистрирует событие при его достижении. Затем рассчитывается Десятипроцентный уровень использования оставшегося пространства и цикл продолжается. По мере исчерпания глобального пространства RID события будут регистрироваться все чаще как 10% попаданий быстрее в пуле убывания (журнала событий каждый Однако будет более чем одной записью в час). Журнал системных событий на каждом контроллере домена записывается событие предупреждения Directory-Services-SAM с кодом 16658.  
  
При использовании по умолчанию 30-разрядного глобального пространства RID, первое событие регистрируется при выделении пул, содержащий 107,374,182<sup>nd</sup> RID. Частота событий увеличивается до последней контрольной точки 100 000, будет зарегистрировано 110 событий всего. Аналогичен для разблокирован 31-разрядного глобального пространства RID: начиная с 214,748,365 и 117 событий.  
  
> [!IMPORTANT]  
> Это событие не ожидается; Проверка пользователей, компьютеров и процессы создания группы в домене. Создание более 100 млн объектов AD DS — довольно соответствует обычным.

### <a name="BKMK_RIDBlockMaxSize"></a>Ограничение на размер блока RID  
Как правило контроллер домена запрашивает RID выделяются блоками по 500 идентификаторов за один раз. Можно переопределить это значение по умолчанию, с помощью следующего параметра реестра значение REG_DWORD в контроллере домена:  
  
```  
HKEY_LOCAL_MACHINE\SYSTEM\CurrentControlSet\Services\NTDS\RID Values  
RID Block Size  
  
```  
  
До выпуска Windows Server 2012 было ограничено максимальное значение этого раздела реестра, за исключением его типом DWORD (которая имеет значение 0xffffffff или 4294967295). Это значение значительно больше общего размера глобального пространства RID. Администраторы иногда ошибке или незнанию настраивали размера блока RID со значениями, которые исчерпания глобального пространства RID массовых скоростью.  
  
В Windows Server 2012 нельзя присвоить это значение реестра, больше 15 000 в десятичной (системе 0x3A98 в шестнадцатеричном). Это предотвращает непреднамеренное выделение большого количества идентификаторов RID.  
  
Если вы задаете значение *выше* 15 000, оно считается равным 15 000, и контроллер домена регистрирует событие 16653 в журнал событий служб каталогов, при каждой загрузке, пока значение не будет исправлено.  
  
### <a name="BKMK_GlobalRidSpaceUnlock"></a>Разблокирование размера глобального пространства RID  
До выпуска Windows Server 2012 глобального пространства RID был ограничен 2<sup>30</sup> (или 1 073 741 823) общий идентификаторов RID. По достижении только миграции домена или леса восстановления есть разрешена создание новых идентификаторов безопасности - аварийное восстановление в любой меры. Начиная с Windows Server 2012 2<sup>31</sup> бит можно разблокировать, чтобы увеличить глобальный пул до 2 147 483 648 идентификаторов RID.  
  
Этот параметр хранится в AD DS в специальном скрытом атрибуте с именем **SidCompatibilityVersion** в контексте RootDSE каждого контроллера домена. Этот атрибут недоступен для чтения с помощью ADSIEdit, LDP или других средств. Чтобы узнать увеличение глобального пространства RID, в журнале системных событий событие предупреждения с кодом 16655 от Directory-Services-SAM или используйте следующую команду Dcdiag:  
  
```  
Dcdiag.exe /TEST:RidManager /v | find /i "Available RID Pool for the Domain"  
  
```  
  
Если глобальный пул RID увеличен, доступный пул изменится на 2 147 483 647 вместо 1 073 741 823 по умолчанию. Например:  
  
![Выдачей RID](media/Managing-RID-Issuance/ADDS_RID_TR_Dcdiag.png)  
  
> [!WARNING]  
> Это снятие ограничения предназначено *только* для предотвращения исчерпания пула RID и должна использоваться *только* в сочетании с принудительной установкой верхнего порога RID (см. следующий раздел). Задайте не» заблаговременно» в средах, имеющих миллионы идентификаторов RID и невысокий темп роста, так как с идентификаторами безопасности, созданные на основе разблокированного пула RID могут вызывать проблемы совместимости приложений.  
>   
> Операцию снятия ограничения нельзя отменить или откатить, кроме как полным восстановлением леса более ранних резервных копий.  
  
#### <a name="important-caveats"></a>Важные предостережения  
Windows Server 2003 и контроллеры домена Windows Server 2008 не могут выдавать идентификаторы RID, если в глобальном пуле RID 31<sup>st</sup> бит разблокирован. Контроллеры домена Windows Server 2008 R2 *можно* использовать 31<sup>st</sup> бит идентификаторов RID *, но только если* исправление [KB 2642658](https://support.microsoft.com/kb/2642658) установлен. Неподдерживаемые незакрытых контроллеры домена воспринимать глобальный пул RID как исчерпанный.  
  
Эта функция не применяется принудительно ни одним режимом работы домена; Следите за который существует только в Windows Server 2012 или обновленные контроллеры домена Windows Server 2008 R2 в домене.

4.  Нажмите кнопку **Обзор** меню и нажмите кнопку **изменить**.  
  
5.  Убедитесь, что **DN** пуст.  
  
6.  В **изменить атрибут записи**, введите:

  
7.  В **значения**, введите:  
  
    ```

    ```  
  
8.  Убедитесь, что **добавить** выбран **операции** и нажмите кнопку **ввод**. Эта команда обновляет **список записей**.  
  
9. Выберите **синхронный** и **Extended** , после чего нажмите кнопку **запуска**.

### <a name="rid-ceiling-enforcement"></a>Принудительная установка верхнего порога RID  
С целью обеспечения защиты и оповещения администраторов, Windows Server 2012 введен искусственный верхний порог для глобального RID на уровне десяти (10) процентов от оставшегося количества идентификаторов RID в глобальном пространстве. При одно (1) до этого искусственного порога процент, контроллеры домена, запрашивающие пулы RID, записывают событие предупреждения Directory-Services-SAM с кодом 16656 в свои журналы системных событий. При достижении десятипроцентного верхнего порога на FSMO хозяина RID, он записывает событие Directory-Services-SAM с кодом 16657 в свой журнал системных событий и перестает не выделять пулы RID до переопределен. Это вынужден оценить состояние хозяина RID в домене и устранить избыточное выделение беглый RID; Это также позволяет защитить домены от полного исчерпания пространства RID.  
  
Пороговое значение жестко запрограммировано на уровне десяти процентов от оставшегося доступного пространства RID. То есть порог считается достигнутым, когда хозяин RID выделяет пула, который содержит соответствующее девяносто 90 процентам глобального пространства относительных ИДЕНТИФИКАТОРОВ RID.  
  
-   Для доменов по умолчанию первое срабатывание — 2<sup>30</sup>-1 * 0,90 = 966 367 640 (или 107,374,183 идентификаторов RID оставшихся).  
  
-   Для доменов с разблокированным 31-разрядным пространством RID срабатывание происходит 2<sup>31</sup>-1 * 0,90 = 1,932,735,282 относительных идентификаторов (или 214,748,365 идентификаторов RID оставшихся).  
  
При активации, хозяин RID присваивает атрибуту Active Directory **msDS-RIDPoolAllocationEnabled** (общее имя **ms-DS-RID-Pool-Allocation-Enabled**) значение FALSE для объекта:  
  
CN = RID Manager$, CN = System, DC =*<domain>*  
  
Это записывает события с кодом 16657 и прекращению выдачи блоков RID на все контроллеры домена. Контроллеры домена продолжают использовать все оставшиеся пулы RID, которые уже выданы им.  
  
Чтобы снять блокировку и разрешить дальнейшее выделение пулов RID, установите значение атрибута равным TRUE. При следующем выделении RID хозяином RID атрибут вернется к значению по умолчанию NOT SET. После этого существует без дальнейшей истощается и в конечном итоге глобальное пространство RID будет исчерпано, чего потребуется восстановление леса или домена миграция.  
  
#### <a name="removing-the-ceiling-block"></a>Снятие блокировки при достижении верхнего порога  
Чтобы снять блокировку при достижении искусственного верхнего порога, выполните следующие действия:  
  
1.  Убедитесь, что хозяин RID роли работает на контроллере домена Windows Server 2012. Если это не так, перенесите его на контроллер домена Windows Server 2012.  
  
2.  Запустите LDP.exe.  
  
3.  Нажмите кнопку **подключения** меню и нажмите кнопку *Connect* для Windows Server 2012 хозяина RID на порт 389, а затем нажмите кнопку **привязки** с правами администратора домена.  
  
4.  Нажмите кнопку **представление** меню и нажмите кнопку **дерева**, затем в **базовое DN** выберите контекст именования домена хозяина RID самостоятельно. Нажмите кнопку **ОК**.  
  
5.  В области навигации детализировать **CN = System** контейнера и нажмите кнопку **CN = RID Manager$** объекта. Щелкните его правой кнопкой мыши и выберите **изменить**.  
  
6.  В изменить атрибут записи введите:  
  
    ```  
    MsDS-RidPoolAllocationEnabled  
    ```  
  
7.  В **значения**, типа (в верхний регистр):  
  
    ```  
    TRUE  
    ```  
  
8.  Выберите **заменить** в **операции** и нажмите кнопку **ввод**. Эта команда обновляет **список записей**.  
  
9. Включить **синхронный** и **Extended** , после чего нажмите кнопку **запуска**:  
  
    ![Выдачей RID](media/Managing-RID-Issuance/ADDS_RID_TR_LDPRaiseCeiling.png)  
  
10. В случае успешного выполнения LDP выходных данных указывается, окно:  
  
    ```  
    ***Call Modify...  
    ldap_modify_ext_s(ld, 'CN=RID Manager$,CN=System,DC=<domain>',[1] attrs, SvrCtrls, ClntCtrls);  
    Modified "CN=RID Manager$,CN=System,DC=<domain>".  
  
    ```  
  
    ![Выдачей RID](media/Managing-RID-Issuance/ADDS_RID_TR_LDPRaiseCeilingSuccess.png)

### <a name="unfixed-rid-issues"></a>Проблемы с нефиксированных RID  
Существует утечка идентификаторов RID на сбой при создании учетной записи; При создании учетной записи, ошибки все равно используется RID. Распространенным примером является создание пользователя с паролем, который не отвечает требованиям к сложности.  
  
### <a name="rid-fixes-for-earlier-versions-of-windows-server"></a>Исправления относительных ИДЕНТИФИКАТОРОВ для более ранних версий Windows Server  
Все исправления и изменения выше имеют исправлениях, выпущенных для Windows Server 2008 R2. Отсутствуют новые исправления для Windows Server 2008 не планируются к выпуску.  
  
## <a name="BKMK_Tshoot"></a>Устранение неполадок выдачи RID  
  
### <a name="introduction-to-troubleshooting"></a>Общие сведения об устранении неполадок  
Устранение неполадок с выдачей RID требует логического линейного подхода. Если вы не проверяете журналы событий тщательно на наличие ошибок и предупреждений относительными, то первым признаком проблемы, скорее всего будут создании неудачных учетных записей. Ключ при устранении неполадок с выдачей RID является понимание того, когда Симптом является ожидаемым, а не; Многие проблемы с выдачей RID могут затрагивать только один контроллер домена и не имеют отношения к улучшению компонентов. Приведенная ниже Простая схема может помочь прояснить процесс принятия решений более понятным.  
  
![Выдачей RID](media/Managing-RID-Issuance/adds_rid_issuance_troubleshooting.png)  
  
### <a name="troubleshooting-options"></a>Способы устранения неполадок  
  
#### <a name="logging-options"></a>Параметры ведения журнала  
Все записи с выдачей RID, регистрируются в журнале системных событий, а источником их является Directory-Services-SAM. Ведение журнала включено и настроена максимальная степень подробности по умолчанию. Если не будут указаны записи при изменении компонентов в Windows Server 2012, проблемы следует решать как классические (то есть, предшествующих Windows Server 2012) неполадки с выдачей RID в Windows 2008 R2 или более старыми операционными системами.  
  
#### <a name="utilities-and-commands-for-troubleshooting"></a>Служебные программы и команды для устранения неполадок  
Для устранения неполадок, объяснения которым нет упомянутых выше журналах - особенно старые проблемы с выдачей RID - использовать средства из следующего списка в качестве отправной точки:  
  
-   Dcdiag.exe  
  
-   Repadmin.exe  
  
-   Сетевой монитор 3.4  
  
### <a name="general-methodology-for-troubleshooting-domain-controller-configuration"></a>Общая методология диагностики конфигурации контроллера домена  
  
1.  Ошибка, из-за простого разрешения или проблема доступности контроллера домена?  
  
    1.  Пытаетесь ли вы создать субъект без необходимых разрешений безопасности? Проверьте в выходных данных наличие ошибок отказа в доступе.  
  
    2.  Доступен ли контроллер домена? Изучите полученную ошибка или LDAP или домена контроллера доступности сообщения.  
  
2.  Ошибка, возвращенная специально упоминаются идентификаторов RID и определенные, чтобы использовать в качестве инструкции? Если Да, следуйте указаниям.  
  
3.  Ошибка, возвращенная специально упоминаются идентификаторов RID, но является недостаточно конкретным? Например «Windows не удается создать объект, так как служба каталогов не удалось выделить относительный идентификатор.»  
  
    1.  Просмотрите журнал системных событий контроллера домена «традиционные» (до Windows Server 2012) события RID, описанные в [запрос пула RID](https://technet.microsoft.com/en-us/library/ee406152(WS.10).aspx) (16642, 16643, 16644, 16645, 16656).  
  
    2.  Проверьте журнал системных событий на контроллере домена и хозяине RID на наличие новых блок-событий, которые описаны ниже в этом разделе (16655, 16656, 16657).  
  
    3.  Проверьте работоспособность репликации Active Directory с помощью Repadmin.exe и доступность хозяина RID с **Dcdiag.exe/Test: RidManager /v**. Включите двухстороннего сетевого мониторинга между контроллером домена и хозяином RID, если этих проверок оказалось недостаточно.
