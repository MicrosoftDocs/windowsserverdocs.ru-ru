# <a name="appendix-i-creating-management-accounts-for-protected-accounts-and-groups-in-active-directory"></a>Приложение И. Создание учетных записей управления для защищенных учетных записей и групп в Active Directory

>Область применения: Windows Server 2016, Windows Server 2012 R2, Windows Server 2012

Одной из трудностей при реализации модели Active Directory, которая не полагается на постоянное членство в группах с высоким уровнем привилегий, является то, что должен существовать механизм заполнения этих групп, если требуется временное членство в группах. Для некоторых решений по управлению привилегированными пользователями требуется, чтобы учетные записи службы программного обеспечения предоставляли постоянное членство в группах, таких как DA или Администраторы, в каждом домене леса. Однако технически не требуется, чтобы решения управление привилегированными пользователями (PIM) выполняли свои службы в таких контекстах с высоким уровнем привилегий.  
  
В этом приложении содержатся сведения, которые можно использовать для самостоятельно реализованных или сторонних решений PIM для создания учетных записей с ограниченными привилегиями, которые можно жестко контролировать, но могут использоваться для заполнения привилегированных групп в Active Directory, если требуется временное повышение прав. Если вы реализуете PIM в качестве собственного решения, эти учетные записи могут использоваться административными сотрудниками для выполнения временных заполнений групп, и если вы реализуете PIM с помощью стороннего программного обеспечения, вы можете адаптировать эти учетные записи для работы в качестве учетных записей служб.  
  
> [!NOTE]  
> Процедуры, описанные в этом приложении, предоставляют один подход к управлению группами с высоким уровнем привилегий в Active Directory. Эти процедуры можно адаптировать в соответствии с вашими потребностями, добавить дополнительные ограничения или опустить некоторые из описанных здесь ограничений.  
  
## <a name="creating-management-accounts-for-protected-accounts-and-groups-in-active-directory"></a>Создание учетных записей управления для защищенных учетных записей и групп в Active Directory

Создание учетных записей, которые могут использоваться для управления членством привилегированных групп без необходимости предоставления учетным записям управления лишних прав и разрешений, состоит из четырех общих действий, описанных в следующих пошаговых инструкциях.  
  
1.  Во-первых, следует создать группу, которая будет управлять учетными записями, так как управление этими учетными записями должно осуществляться ограниченным набором доверенных пользователей. Если у вас еще нет структуры подразделений, которая поддерживает разделение привилегированных и защищенных учетных записей и систем из общего заполнения в домене, следует создать его. Хотя в этом приложении не предоставлены конкретные инструкции, снимки экрана демонстрируют пример такой иерархии подразделений.  
  
2.  Создайте учетные записи управления. Эти учетные записи должны быть созданы как обычные учетные записи пользователей и не получают прав пользователей, кроме тех, которые уже предоставлены пользователям по умолчанию.  
  
3.  Реализуйте ограничения для учетных записей управления, которые делают их доступными только для специализированной цели, для которой они были созданы, а также для того, кто может включать и использовать учетные записи (группы, созданные на первом шаге).  
  
4.  Настройте разрешения для объекта AdminSDHolder в каждом домене, чтобы разрешить учетным записям управления изменять членство привилегированных групп в домене.  
  
Следует тщательно протестировать все эти процедуры и изменить их в соответствии с потребностями среды перед их реализацией в рабочей среде. Также следует убедиться, что все параметры работают должным образом (некоторые процедуры тестирования предоставляются в этом приложении), и следует протестировать сценарий аварийного восстановления, в котором учетные записи управления недоступны для заполнения защищенных групп в целях восстановления. Дополнительные сведения о резервном копировании и восстановлении Active Directory см. в статье Пошаговое описание [резервного копирования и восстановления AD DS](https://technet.microsoft.com/library/cc771290(v=ws.10).aspx).  
  
> [!NOTE]  
> Выполняя действия, описанные в этом приложении, вы создадите учетные записи, которые смогут управлять членством в всех защищенных группах в каждом домене, а не только с наивысшими правами доступа Active Directory группами, такими как EAs, DAs и BAs. Дополнительные сведения о защищенных группах в Active Directory см. [в разделе приложение C: защищенные учетные записи и группы в Active Directory](../../../ad-ds/plan/security-best-practices/Appendix-C--Protected-Accounts-and-Groups-in-Active-Directory.md).  
  
### <a name="step-by-step-instructions-for-creating-management-accounts-for-protected-groups"></a>Пошаговые инструкции по созданию учетных записей управления для защищенных групп  
  
#### <a name="creating-a-group-to-enable-and-disable-management-accounts"></a>Создание группы для включения и отключения учетных записей управления

При каждом использовании учетных записей управления должны быть сброшены пароли, и их следует отключать при завершении действий, которым они требуются. Несмотря на то, что для этих учетных записей можно также реализовать требования к входу с помощью смарт-карты, это необязательная конфигурация. в этих инструкциях предполагается, что в учетных записях управления в качестве минимальных элементов управления будут настроены имя пользователя и длинный пароль. На этом шаге вы создадите группу с разрешениями на сброс пароля для учетных записей управления, а также включите и отключите учетные записи.  
  
Чтобы создать группу для включения и отключения учетных записей управления, выполните следующие действия.  
  
1.  В структуре подразделений, в которой будут разноситься учетные записи управления, щелкните правой кнопкой мыши подразделение, в котором нужно создать группу, щелкните **создать** и выберите **Группа**.  
  
    ![Создание учетных записей управления](media/Appendix-I--Creating-Management-Accounts-for-Protected-Accounts-and-Groups-in-Active-Directory/SAD_115.png)  
  
2.  В диалоговом окне **Создание группы объектов** введите имя группы. Если вы планируете использовать эту группу для включения всех учетных записей управления в лесу, сделайте ее универсальной группой безопасности. Если у вас есть лес с одним доменом или если вы планируете создать группу в каждом домене, можно создать глобальную группу безопасности. Нажмите кнопку **ОК** , чтобы создать группу.  
  
    ![Создание учетных записей управления](media/Appendix-I--Creating-Management-Accounts-for-Protected-Accounts-and-Groups-in-Active-Directory/SAD_116.png)  
  
3.  Щелкните правой кнопкой мыши только что созданную группу, выберите пункт **Свойства**и перейдите на вкладку **объект** . В диалоговом окне **свойства объекта** группы выберите **защитить объект от случайного удаления**, который не только запрещает полномочным пользователям удалять группу, но и перемещает ее в другое подразделение, если только этот атрибут не был выбран первым.  
  
    ![Создание учетных записей управления](media/Appendix-I--Creating-Management-Accounts-for-Protected-Accounts-and-Groups-in-Active-Directory/SAD_117.png)  
  
    > [!NOTE]  
    > Если вы уже настроили разрешения на родительские подразделения группы, чтобы ограничить администрирование ограниченным набором пользователей, вам может не потребоваться выполнить следующие действия. Они предоставляются здесь, поэтому даже если вы еще не реализовали ограниченный административный контроль над структурой подразделений, в которой вы создали эту группу, вы можете защитить группу от изменений неавторизованными пользователями.  
  
4.  Перейдите на вкладку **члены** и добавьте учетные записи для членов вашей команды, которые будут отвечать за включение учетных записей управления или заполнение защищенных групп при необходимости.  
  
    ![Создание учетных записей управления](media/Appendix-I--Creating-Management-Accounts-for-Protected-Accounts-and-Groups-in-Active-Directory/SAD_118.png)  
  
5.  Если вы еще не сделали это, в консоли **Active Directory пользователи и компьютеры** щелкните **Просмотреть** и выберите **Дополнительные функции**. Щелкните правой кнопкой мыши только что созданную группу, выберите пункт **Свойства**и перейдите на вкладку **Безопасность** . На вкладке **Безопасность** нажмите кнопку **Дополнительно**.  
  
    ![Создание учетных записей управления](media/Appendix-I--Creating-Management-Accounts-for-Protected-Accounts-and-Groups-in-Active-Directory/SAD_119.png)  
  
6.  В диалоговом окне **Дополнительные параметры безопасности для [Группа]** щелкните **отключить наследование**. При появлении запроса нажмите кнопку **преобразовать унаследованные разрешения в явные разрешения для этого объекта**и нажмите кнопку **ОК** , чтобы вернуться в диалоговое окно **Безопасность** группы.  
  
    ![Создание учетных записей управления](media/Appendix-I--Creating-Management-Accounts-for-Protected-Accounts-and-Groups-in-Active-Directory/SAD_120.png)  
  
7.  На вкладке **Безопасность** удалите группы, которым не должен быть разрешен доступ к этой группе. Например, если вы не хотите, чтобы пользователи, прошедшие проверку подлинности, могли считывать имя группы и общие свойства, вы можете удалить эту запись ACE. Кроме того, можно удалить записи ACE, например, для операторов учетных записей и для пред-Windows 2000 доступа, совместимых с сервером. Однако следует оставить минимальный набор разрешений объекта на месте. Оставить следующие элементы ACE без изменений:  
  
    -   САМООБСЛУЖИВАНИЯ  
  
    -   СИСТЕМОЙ  
  
    -   Администраторы домена  
  
    -   Администраторы предприятия  
  
    -   Администраторы  
  
    -   Группа доступа к авторизации Windows (если применимо)  
  
    -   КОНТРОЛЛЕРЫ ДОМЕНА ПРЕДПРИЯТИЯ  
  
    Несмотря на то, что может показаться нелогичным разрешить группам с наибольшим уровнем прав в Active Directory для управления этой группой, ваша цель реализации этих параметров — не предотвращать внесение изменений членами этих групп. Задача заключается в том, чтобы убедиться в том, что если у вас есть возможность потребовать очень высокого уровня привилегий, разрешенные изменения будут выполнены успешно. По этой причине изменение вложенности привилегированных групп по умолчанию, прав и разрешений не рекомендуется в рамках этого документа. Если не сохранить структуры по умолчанию и очистить членство в группе с самыми высокими правами доступа в каталоге, можно создать более безопасную среду, которая работает должным образом.  
  
    ![Создание учетных записей управления](media/Appendix-I--Creating-Management-Accounts-for-Protected-Accounts-and-Groups-in-Active-Directory/SAD_121.png)  
  
    > [!NOTE]  
    > Если вы еще не настроили политики аудита для объектов в структуре подразделений, в которой была создана эта группа, следует настроить аудит на запись изменений в журнал этой группы.  
  
8.  Вы завершили настройку группы, которая будет использоваться для "извлечения" учетных записей управления, когда они понадобятся, и "возвращать" учетные записи после завершения их действий.  
  
#### <a name="creating-the-management-accounts"></a>Создание учетных записей управления

Необходимо создать по крайней мере одну учетную запись, которая будет использоваться для управления членством в привилегированных группах при установке Active Directory и, желательно, второй учетной записью для резервного копирования. Если вы решили создать учетные записи управления в одном домене в лесу и предоставите возможности управления для всех защищенных групп домена, а также хотите ли вы выбрать реализацию учетных записей управления в каждом домене леса, процедуры будут одинаковыми.  
  
> [!NOTE]  
> В этом документе предполагается, что вы еще не реализовали элементы управления доступом на основе ролей и управление привилегированными пользователями для Active Directory. Поэтому некоторые процедуры должны выполняться пользователем, учетная запись которого является членом группы "Администраторы домена" для рассматриваемого домена.  
>   
> При использовании учетной записи с привилегиями DA можно войти в контроллер домена для выполнения действий по настройке. Действия, не требующие привилегий DA, могут выполняться с учетными записями с меньшими привилегиями, вошедшими на рабочие станции администрирования. Снимки экрана, на которых отображаются диалоговые окна с светлым синим цветом, представляют действия, которые могут быть выполнены на контроллере домена. Снимки экрана, отображающие диалоговые окна в темно-синем цвете, представляют действия, которые можно выполнять на административных рабочих станциях с учетными записями с ограниченными правами.  
  
Чтобы создать учетные записи управления, выполните следующие действия.  
  
1. Войдите в контроллер домена, используя учетную запись, которая является членом группы DA домена.  

2. Запустите **Active Directory пользователей и компьютеры** и перейдите к подразделению, в котором будет создаваться учетная запись управления.  

3. Щелкните правой кнопкой мыши подразделение и выберите **создать** и щелкните **пользователь**.  

4. В диалоговом окне **новый объект — пользователь** введите необходимые сведения об именовании для учетной записи и нажмите кнопку **Далее**.  

   ![Создание учетных записей управления](media/Appendix-I--Creating-Management-Accounts-for-Protected-Accounts-and-Groups-in-Active-Directory/SAD_122.png)  
  
5. Укажите первоначальный пароль для учетной записи пользователя, снимите флажок **пользователь должен сменить пароль при следующем входе в систему**, выберите параметр **пользователь не может изменить пароль** и **учетная запись отключена**и нажмите кнопку **Далее**.  

   ![Создание учетных записей управления](media/Appendix-I--Creating-Management-Accounts-for-Protected-Accounts-and-Groups-in-Active-Directory/SAD_123.png)  

6. Убедитесь, что сведения об учетной записи указаны правильно, и нажмите кнопку **"Готово"** .  

7. Щелкните правой кнопкой мыши только что созданный объект пользователя и выберите пункт **Свойства**.  

8. Перейдите на вкладку **учетная запись** .  

9. В поле **Параметры учетной записи** выберите **учетную запись конфиденциальная и не может быть делегирована** , установите флажок **Эта учетная запись поддерживает протокол Kerberos AES 128 разрядное шифрование** и/или **Эта учетная запись поддерживает флаг шифрования Kerberos AES 256** и нажмите кнопку **ОК**.  

   ![Создание учетных записей управления](media/Appendix-I--Creating-Management-Accounts-for-Protected-Accounts-and-Groups-in-Active-Directory/SAD_124.png)  

   > [!NOTE]  
   > Так как эта учетная запись, как и другие учетные записи, будет иметь ограниченную, но мощную функцию, учетную запись следует использовать только на защищенных узлах администрирования. Для всех безопасных административных узлов в среде следует рассмотреть возможность реализации групповая политика настройки **сетевой безопасности. Настройте типы шифрования, разрешенные для Kerberos** , чтобы разрешить только наиболее безопасные типы шифрования, которые можно реализовать для защищенных узлов.  
   >
   > Несмотря на то, что реализация более безопасных типов шифрования для узлов не позволяет предотвратить атаки методом кражи учетных данных, необходимо использовать и настраивать защищенные узлы. Установка более надежных типов шифрования для узлов, которые используются привилегированными учетными записями, просто сокращает общую область атак компьютеров.  
   >
   > Дополнительные сведения о настройке типов шифрования в системах и учетных записях см. в статье [конфигурации Windows для типа шифрования, поддерживаемого протоколом Kerberos](https://blogs.msdn.com/b/openspecification/archive/2011/05/31/windows-configurations-for-kerberos-supported-encryption-type.aspx).  
   >
   > Эти параметры поддерживаются только на компьютерах под управлением Windows Server 2012, Windows Server 2008 R2, Windows 8 или Windows 7.  
  
10. На вкладке **объект** выберите **защитить объект от случайного удаления**. Это не только предотвратит удаление объекта (даже с помощью полномочных пользователей), но сделает его недоступным для перемещения в другое подразделение в иерархии AD DS, если только флажок не будет впервые снят пользователем с разрешением на изменение атрибута.  

    ![Создание учетных записей управления](media/Appendix-I--Creating-Management-Accounts-for-Protected-Accounts-and-Groups-in-Active-Directory/SAD_125.png)  

11. Перейдите на вкладку **Удаленное управление** .  

12. Снимите флажок **включить флаг удаленного управления** . Он никогда не должен быть нужен для того, чтобы сотрудники службы поддержки подключались к сеансам этой учетной записи для реализации исправлений.  

    ![Создание учетных записей управления](media/Appendix-I--Creating-Management-Accounts-for-Protected-Accounts-and-Groups-in-Active-Directory/SAD_126.png)  

    > [!NOTE]

##### <a name="creating-additional-management-accounts"></a>Создание дополнительных учетных записей управления

Можно создать дополнительные учетные записи управления, повторив предыдущие шаги, скопировав только что созданную учетную запись или создав скрипт для создания учетных записей с нужными параметрами конфигурации. Однако обратите внимание, что если вы скопировали только что созданную учетную запись, многие из настроенных параметров и ACL не будут скопированы в новую учетную запись, и вам придется повторить большинство этапов настройки.  
  
Вместо этого можно создать группу, которой делегируются права на заполнение и незаполнение защищенных групп, но необходимо защитить группу и учетные записи, которые вы поместите в нее. Поскольку в каталоге должно быть очень мало учетных записей, которым предоставляется возможность управлять членством в защищенных группах, создание отдельных учетных записей может быть самым простым подходом.  
  
Независимо от того, как вы решили создать группу, в которую помещаются учетные записи управления, следует убедиться, что каждая учетная запись защищена, как описано выше. Также следует рассмотреть возможность реализации ограничений объектов групповой политики, аналогичных описанным в [приложении г: защита встроенных учетных записей администраторов в Active Directory](../../../ad-ds/plan/security-best-practices/Appendix-D--Securing-Built-In-Administrator-Accounts-in-Active-Directory.md).  
  
##### <a name="auditing-management-accounts"></a>Учетные записи управления аудитом

Необходимо настроить аудит для учетной записи, как минимум, всех операций записи в учетную запись. Это позволит не только правильно включить учетную запись и сбросить ее пароль во время авторизованного использования, но также выявляет попытки неавторизованными пользователями управлять учетной записью. Неудачные операции записи в учетную запись должны быть записаны в системе безопасности и мониторинга событий (если применимо) и должны активировать оповещения, которые предоставляют сотрудникам, ответственным за исследование потенциальных угроз.  
  
SIEM решения принимают данные о событиях из связанных источников безопасности (например, журналов событий, данных приложений, сетевых потоков, продуктов для защиты от вредоносных программ и источников обнаружения вторжений), выполняет сортировку данных и пытается сделать интеллектуальные представления и упреждающие действия. Существует множество коммерческих решений SIEM, и многие предприятия создают частные реализации. Хорошо спроектированная и надлежащим образом реализованная SIEM может значительно улучшить возможности мониторинга безопасности и реагирования на инциденты. Однако возможности и точность различаются невероятно между решениями. Решения Siem выходит за рамки этого документа, но конкретные рекомендации по событиям должны рассматриваться любыми SIEM.  
  
Дополнительные сведения о рекомендуемых параметрах конфигурации аудита для контроллеров домена см. в разделе [monitoring Active Directory to признаки компрометации](../../../ad-ds/plan/security-best-practices/Monitoring-Active-Directory-for-Signs-of-Compromise.md). Параметры конфигурации, относящиеся к контроллеру домена, задаются в Active Directory мониторинга, что дает [признаки компрометации](../../../ad-ds/plan/security-best-practices/Monitoring-Active-Directory-for-Signs-of-Compromise.md).  
  
#### <a name="enabling-management-accounts-to-modify-the-membership-of-protected-groups"></a>Включение учетных записей управления для изменения членства в защищенных группах

В этой процедуре вы настроите разрешения для объекта AdminSDHolder домена, чтобы разрешить вновь созданным учетным записям управления изменять членство в защищенных группах в домене. Эта процедура не может быть выполнена через графический пользовательский интерфейс (GUI).  
  
Как обсуждалось в [приложении C: защищенные учетные записи и группы в Active Directory](../../../ad-ds/plan/security-best-practices/Appendix-C--Protected-Accounts-and-Groups-in-Active-Directory.md), список ACL в объекте AdminSDHolder домена фактически копируется в защищенные объекты при выполнении задачи SDProp. Защищенные группы и учетные записи не наследуют разрешения от объекта AdminSDHolder; их разрешения явным образом устанавливаются в соответствии с параметрами объекта AdminSDHolder. Таким образом, при изменении разрешений на объект AdminSDHolder необходимо изменить их для атрибутов, соответствующих типу защищаемого объекта.  
  
В этом случае вновь созданные учетные записи управления будут предоставлены для чтения и записи атрибута Members в объектах группы. Однако объект AdminSDHolder не является объектом группы, и атрибуты группы не отображаются в графическом редакторе ACL. По этой причине изменения разрешений будут реализованы с помощью служебной программы командной строки dsacls. Чтобы предоставить учетным записям управления (отключенным) разрешения на изменение членства в защищенных группах, выполните следующие действия.  
  
1. Войдите на контроллер домена, предпочтительнее контроллера домена, содержащего роль эмулятора PDC (ПДЦЕ), с учетными данными учетной записи пользователя, которая была сделана членом группы DA в домене.  
  
   ![Создание учетных записей управления](media/Appendix-I--Creating-Management-Accounts-for-Protected-Accounts-and-Groups-in-Active-Directory/SAD_136.png)  
  
2. Откройте командную строку с повышенными привилегиями, щелкнув правой кнопкой мыши пункт **Командная строка** и выбрав команду **Запуск от имени администратора**.  
  
   ![Создание учетных записей управления](media/Appendix-I--Creating-Management-Accounts-for-Protected-Accounts-and-Groups-in-Active-Directory/SAD_137.gif)  
  
3. При появлении запроса на подтверждение повышения прав нажмите кнопку **Да**.  
  
   ![Создание учетных записей управления](media/Appendix-I--Creating-Management-Accounts-for-Protected-Accounts-and-Groups-in-Active-Directory/SAD_138.gif)  
  
   > [!NOTE]  
   > Дополнительные сведения о повышении прав и контроле учетных записей (UAC) в Windows см. в разделе [процессы и взаимодействия UAC](https://technet.microsoft.com/library/dd835561(v=WS.10).aspx) на веб-сайте TechNet.  
  
4. В командной строке введите команду **DSACLS [различающееся имя объекта adminSDHolder в домене]/g [домен учетной записи управления UPN]: рпвп; Member**.  
  
   ![Создание учетных записей управления](media/Appendix-I--Creating-Management-Accounts-for-Protected-Accounts-and-Groups-in-Active-Directory/SAD_139.gif)  
  
   Предыдущая команда (без учета регистра) работает следующим образом:  
  
   - DSACLS устанавливает или отображает записи ACE для объектов каталога  
  
   - CN = AdminSDHolder, CN = System, DC = Таилспинтойс, DC = MSFT определяет изменяемый объект  
  
   - /G указывает, что настраивается предоставление ACE  
  
   - PIM001@tailspintoys.msft — имя участника-пользователя (UPN) субъекта безопасности, которому будут предоставлены ACE  
  
   - РПВП предоставляет разрешения на чтение и запись свойства  
  
   - Member — это имя свойства (атрибута), для которого будут установлены разрешения.  
  
   Для получения дополнительных сведений об использовании команды **DSACLS**введите команду DSACLS без параметров в командной строке.  
  
   Если вы создали несколько учетных записей управления для домена, выполните команду DSACLS для каждой учетной записи. После завершения настройки ACL для объекта AdminSDHolder следует принудительно запустить SDProp или подождать завершения запланированного выполнения. Сведения о принудительном выполнении SDProp см. в разделе "Запуск SDProp вручную" в [приложении C: защищенные учетные записи и группы в Active Directory](../../../ad-ds/plan/security-best-practices/Appendix-C--Protected-Accounts-and-Groups-in-Active-Directory.md).  
  
   После выполнения SDProp можно убедиться, что изменения, внесенные в объект AdminSDHolder, применены к защищенным группам в домене. Это невозможно проверить, просмотрев список ACL объекта AdminSDHolder по описанным выше причинам, но можно проверить, что разрешения были применены, просмотрев списки управления доступом в защищенных группах.  
  
5. В **Active Directory пользователи и компьютеры**убедитесь, что включены **Дополнительные функции**. Для этого щелкните **Просмотр**, выберите группу **"Администраторы домена** ", щелкните правой кнопкой мыши группу и выберите пункт " **Свойства**".  
  
6. Перейдите на вкладку **Безопасность** и нажмите кнопку **Дополнительно** , чтобы открыть диалоговое окно **Дополнительные параметры безопасности для администраторов домена** .  
  
   ![Создание учетных записей управления](media/Appendix-I--Creating-Management-Accounts-for-Protected-Accounts-and-Groups-in-Active-Directory/SAD_140.gif)  
  
7. Установите флажок **Разрешить запись ACE для учетной записи управления** и нажмите кнопку **изменить**. Убедитесь, что учетной записи предоставлены разрешения только **Чтение участников** и **запись членов** в группе Da, и нажмите кнопку **ОК**.  
  
8. Нажмите кнопку **ОК** в диалоговом окне **Дополнительные параметры безопасности** и снова нажмите кнопку **ОК** , чтобы закрыть диалоговое окно Свойства для группы Da.  
  
   ![Создание учетных записей управления](media/Appendix-I--Creating-Management-Accounts-for-Protected-Accounts-and-Groups-in-Active-Directory/SAD_141.gif)  
  
9. Вы можете повторить предыдущие шаги для других защищенных групп в домене. разрешения должны быть одинаковыми для всех защищенных групп. Теперь вы завершили создание и настройку учетных записей управления для защищенных групп в этом домене.  
  
    > [!NOTE]  
    > Любая учетная запись, имеющая разрешение на запись членства в группе в Active Directory, также может быть добавлена в группу. Это поведение характерно и не может быть отключено. По этой причине следует всегда отключать учетные записи управления, если они не используются, и тщательно отслеживать учетные записи, когда они отключены и когда они используются.  
  
#### <a name="verifying-group-and-account-configuration-settings"></a>Проверка параметров конфигурации группы и учетной записи

Теперь, когда вы создали и настроили учетные записи управления, которые могут изменять членство в защищенных группах в домене (которая включает в себя самые высокопривилегированные группы EA, DA и BA), необходимо убедиться, что учетные записи и их группа управления созданы правильно. Проверка состоит из следующих общих задач.  
  
1.  Протестируйте группу, которая может включать и отключать учетные записи управления, чтобы убедиться, что члены группы могут включать и отключать учетные записи и сбрасывать пароли, но не могут выполнять другие административные действия с учетными записями управления.  
  
2.  Протестируйте учетные записи управления, чтобы убедиться, что они могут добавлять или удалять членов защищенных групп в домене, но не могут изменять другие свойства защищенных учетных записей и групп.  
  
##### <a name="test-the-group-that-will-enable-and-disable-management-accounts"></a>Тестирование группы, которая будет включать и отключать учетные записи управления
  
1.  Чтобы проверить включение учетной записи управления и сброс пароля, войдите на защищенную рабочую станцию администрирования с учетной записью, которая является членом группы, созданной в [приложении I. Создание учетных записей управления для защищенных учетных записей и групп в Active Directory](../../../ad-ds/manage/component-updates/Appendix-I--Creating-Management-Accounts-for-Protected-Accounts-and-Groups-in-Active-Directory.md).  
  
    ![Создание учетных записей управления](media/Appendix-I--Creating-Management-Accounts-for-Protected-Accounts-and-Groups-in-Active-Directory/SAD_142.gif)  
  
2.  Откройте **Active Directory пользователи и компьютеры**, щелкните правой кнопкой мыши учетную запись управления и выберите пункт **включить учетную запись**.  
  
    ![Создание учетных записей управления](media/Appendix-I--Creating-Management-Accounts-for-Protected-Accounts-and-Groups-in-Active-Directory/SAD_143.gif)  
  
3.  Должно отобразиться диалоговое окно с подтверждением включения учетной записи.  
  
    ![Создание учетных записей управления](media/Appendix-I--Creating-Management-Accounts-for-Protected-Accounts-and-Groups-in-Active-Directory/SAD_144.gif)  
  
4.  Затем сбросьте пароль учетной записи управления. Для этого еще раз щелкните учетную запись правой кнопкой мыши и выберите команду **сбросить пароль**.  
  
    ![Создание учетных записей управления](media/Appendix-I--Creating-Management-Accounts-for-Protected-Accounts-and-Groups-in-Active-Directory/SAD_145.gif)  
  
5.  Введите новый пароль для учетной записи в полях **новый пароль** и **Подтверждение пароля** и нажмите кнопку **ОК**.  
  
    ![Создание учетных записей управления](media/Appendix-I--Creating-Management-Accounts-for-Protected-Accounts-and-Groups-in-Active-Directory/SAD_146.gif)  
  
6.  Появится диалоговое окно с подтверждением сброса пароля учетной записи.  
  
    ![Создание учетных записей управления](media/Appendix-I--Creating-Management-Accounts-for-Protected-Accounts-and-Groups-in-Active-Directory/SAD_147.gif)  
  
7.  Теперь попытайтесь изменить дополнительные свойства учетной записи управления. Щелкните учетную запись правой кнопкой мыши и выберите пункт **Свойства**и перейдите на вкладку **Удаленное управление** .  
  
8.  Выберите **включить удаленное управление** и нажмите кнопку **Применить**. Операция должна завершиться ошибкой, и должно отобразиться сообщение об ошибке **отказа в доступе** .  
  
    ![Создание учетных записей управления](media/Appendix-I--Creating-Management-Accounts-for-Protected-Accounts-and-Groups-in-Active-Directory/SAD_148.gif)  
  
9. Перейдите на вкладку **Учетная** запись для учетной записи и попытайтесь изменить имя учетной записи, время входа или рабочие станции входа. Все должно завершиться ошибкой, и параметры учетной записи, не контролируемые атрибутом **userAccountControl** , должны быть неактивны и недоступны для изменения.  
  
    ![Создание учетных записей управления](media/Appendix-I--Creating-Management-Accounts-for-Protected-Accounts-and-Groups-in-Active-Directory/SAD_149.gif)  
  
10. Попытка добавить группу управления в защищенную группу, например в группу DA. При нажатии кнопки **ОК**появится сообщение, информирующее о том, что у вас нет разрешений на изменение группы.  
  
    ![Создание учетных записей управления](media/Appendix-I--Creating-Management-Accounts-for-Protected-Accounts-and-Groups-in-Active-Directory/SAD_150.gif)  
  
11. Выполните дополнительные тесты, чтобы убедиться, что в учетной записи управления не удается настроить что-либо, кроме параметров **userAccountControl** и сбросов паролей.  
  
    > [!NOTE]  
    > Атрибут **userAccountControl** управляет параметрами конфигурации нескольких учетных записей. Нельзя предоставлять разрешение на изменение только некоторых параметров конфигурации при предоставлении разрешения на запись для атрибута.  
  
##### <a name="test-the-management-accounts"></a>Тестирование учетных записей управления

Теперь, когда вы включили одну или несколько учетных записей, которые могут изменять членство в защищенных группах, можно проверить учетные записи, чтобы убедиться, что они могут изменять членство в защищенной группе, но не могут выполнять другие изменения в защищенных учетных записях и группах.  
  
1.  Войдите на защищенный административный узел в качестве первой учетной записи управления.  
  
    ![Создание учетных записей управления](media/Appendix-I--Creating-Management-Accounts-for-Protected-Accounts-and-Groups-in-Active-Directory/SAD_151.gif)  
  
2.  Запустите **Active Directory "пользователи и компьютеры** " и откройте **группу "Администраторы домена**".  
  
3.  Щелкните правой кнопкой мыши группу **Администраторы домена** и выберите пункт **свойства**.  
  
    ![Создание учетных записей управления](media/Appendix-I--Creating-Management-Accounts-for-Protected-Accounts-and-Groups-in-Active-Directory/SAD_152.gif)  
  
4.  В **свойствах администраторов домена**перейдите на вкладку **члены** и **нажмите кнопку** добавить. Введите имя учетной записи, которой будут предоставлены права доступа администраторов домена, и нажмите кнопку **Проверить имена**. Если имя учетной записи подчеркнуто, нажмите кнопку **ОК** , чтобы вернуться на вкладку **члены** .  
  
    ![Создание учетных записей управления](media/Appendix-I--Creating-Management-Accounts-for-Protected-Accounts-and-Groups-in-Active-Directory/SAD_153.gif)  
  
5.  На вкладке **члены** диалогового окна **Свойства: Администраторы домена** нажмите кнопку **Применить**. После нажатия кнопки **Применить**учетная запись должна оставаться членом группы Da, и сообщения об ошибках отображаться не будут.  
  
    ![Создание учетных записей управления](media/Appendix-I--Creating-Management-Accounts-for-Protected-Accounts-and-Groups-in-Active-Directory/SAD_154.gif)  
  
6.  Перейдите на вкладку **управляется** в диалоговом окне **Свойства: Администраторы домена** и убедитесь, что текст не может быть введен в поля, а все кнопки неактивны.  
  
    ![Создание учетных записей управления](media/Appendix-I--Creating-Management-Accounts-for-Protected-Accounts-and-Groups-in-Active-Directory/SAD_155.gif)  
  
7.  Перейдите на вкладку **Общие** в диалоговом окне **Свойства: Администраторы домена** и убедитесь, что вы не можете изменить сведения об этой вкладке.  
  
    ![Создание учетных записей управления](media/Appendix-I--Creating-Management-Accounts-for-Protected-Accounts-and-Groups-in-Active-Directory/SAD_156.gif)  
  
8.  При необходимости повторите эти действия для дополнительных защищенных групп. По завершении Войдите на защищенный административный узел с помощью учетной записи, которая является членом группы, созданной для включения и отключения учетных записей управления. Затем сбросьте пароль для только что проверенной учетной записи управления и отключите учетную запись. Вы завершили настройку учетных записей управления и группу, которая будет отвечать за включение и отключение учетных записей.
