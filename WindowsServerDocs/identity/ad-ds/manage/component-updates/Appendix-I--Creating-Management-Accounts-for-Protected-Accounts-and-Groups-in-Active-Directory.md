# <a name="appendix-i-creating-management-accounts-for-protected-accounts-and-groups-in-active-directory"></a>Приложение и: Создание управления учетных записей для защищенных учетных записей и групп в Active Directory

>Область применения. Windows Server 2016, Windows Server 2012 R2, Windows Server 2012

Одной из сложностей при реализации модели Active Directory, не зависит от постоянного членства в группах с высоким уровнем привилегий является, что необходим механизм для заполнения этих групп, когда требуется временного членства в группах. Некоторые решения по управлению привилегированными пользователями требуют, что учетные записи служб для программного обеспечения, предоставляется постоянного членства в группах как DA или Администраторы в каждом домене в лесу. Тем не менее это технически не необходимо для решения Privileged Identity Management (PIM) для работы своих служб в таких контекстах высоким уровнем привилегий.  
  
Данное приложение содержит сведения, которые можно использовать для скомпилированных в собственном коде реализованного или сторонних решений PIM для создания учетных записей, которые ограниченные привилегии можно жестко контролировать, но может использоваться для заполнения привилегированных групп в Active Directory при временное повышение уровня прав является обязательным. При реализации PIM решения, эти учетные записи могут использоваться административного персонала выполнить заполнение временного членства, и если вы реализуете PIM с помощью стороннего программного обеспечения, можно адаптировать эти учетные записи в качестве службы учетные записи.  
  
> [!NOTE]  
> Процедуры, описанные в этом приложении предоставляют один из подходов к управлению высокопривилегированных групп в Active Directory. Можно немного изменить эти процедуры, в соответствии со своими потребностями, добавьте дополнительные ограничения, или пропустить некоторые ограничения, описанные здесь.  
  
## <a name="creating-management-accounts-for-protected-accounts-and-groups-in-active-directory"></a>Создание управления учетных записей для защищенных учетных записей и групп в Active Directory

Создание учетных записей, который может использоваться для управления членством в привилегированных групп, без необходимости управления учетные записи должны быть предоставлены избыточные права и разрешения состоит из четырех общие действия, описанные в пошаговые инструкции, действия.  
  
1.  Во-первых следует создать группу, которая будет управлять учетными записями, так как эти учетные записи должны управляться ограниченный набор доверенных пользователей. Если у вас еще нет структуру Подразделений, который допустим для разделения привилегированным и защищенные учетным записям и системам генеральной совокупности в домене, его следует создать. Несмотря на то, что инструкции не указаны в настоящем приложении, снимки экрана показан пример такой иерархии Подразделений.  
  
2.  Создание учетных записей управления. Эти учетные записи должны создаются как учетные записи пользователей «regular» и без права пользователя, помимо тех, которые уже предоставляются пользователям по умолчанию.  
  
3.  Реализуйте ограничения для учетных записей управления, которые делают их можно использовать только для специализированных целей, для которого они были созданы, помимо управления, можно включить и использовать учетные записи (группы, созданной на первом шаге).  
  
4.  Настройка разрешений на объект AdminSDHolder в каждом домене, чтобы учетные записи управления изменить членство привилегированных групп в домене.  
  
Следует тщательно протестировать все эти процедуры и изменить их при необходимости для вашей среды перед их реализацией в рабочей среде. Необходимо также проверить, что все параметры работают должным образом (некоторых процедур тестирования приведены в этом приложении), и проверить сценарии аварийного восстановления, в котором учетных записей управления недоступны для использования для заполнения защищенных групп для восстановления в целях. Дополнительные сведения о резервном копировании и восстановлении Active Directory, см. в разделе [AD DS архивации и восстановления Пошаговое руководство по использованию](https://technet.microsoft.com/library/cc771290(v=ws.10).aspx).  
  
> [!NOTE]  
> Путем реализации действия, описанные в этом приложении, будут созданы учетные записи, которые будут иметь возможность управлять членством всех защищенных группах в каждом домене, не только группы Active Directory с высоким уровнем привилегий EAs, DAs и BAs. Дополнительные сведения о защищенных группах в Active Directory, см. в разделе [приложение в: Защищенные учетные записи и группы в Active Directory](../../../ad-ds/plan/security-best-practices/Appendix-C--Protected-Accounts-and-Groups-in-Active-Directory.md).  
  
### <a name="step-by-step-instructions-for-creating-management-accounts-for-protected-groups"></a>Пошаговые инструкции по созданию учетных записей управления для защищенных групп  
  
#### <a name="creating-a-group-to-enable-and-disable-management-accounts"></a>Создание группы для включения и отключения учетных записей управления

Учетные записи управления должны иметь свои пароли, сброса параметров каждого использования и должен быть отключен после выполнения действий, требующих их. Несмотря на то, что может также потребоваться реализации требований входа смарт-карты для этих учетных записей, он не является обязательным, и эти инструкции предполагают, что учетные записи управления будет настроен с именем пользователя и длинный сложный пароль как минимум элементы управления. На этом шаге вы создадите группу, имеющую разрешения для сброса пароля на учетные записи управления, а также включать и отключать учетные записи.  
  
Чтобы создать группу для включения и отключения учетных записей управления, выполните следующие действия.  
  
1.  В структуре Подразделений, где будут размещены учетных записей управления, щелкните правой кнопкой мыши Подразделение, где вы хотите создать группу, нажмите кнопку **New** и нажмите кнопку **группы**.  
  
    ![Создание учетных записей управления](media/Appendix-I--Creating-Management-Accounts-for-Protected-Accounts-and-Groups-in-Active-Directory/SAD_115.png)  
  
2.  В **новый объект - группа** диалогового окна введите имя для группы. Если вы планируете использовать эту группу для «активировать» всем учетным записям управления в вашем лесу, сделать универсальную группу безопасности. Если у вас есть лес с одним доменом, или если планируется создать группу в каждом домене, можно создать глобальную группу безопасности. Нажмите кнопку **ОК** , чтобы создать группу.  
  
    ![Создание учетных записей управления](media/Appendix-I--Creating-Management-Accounts-for-Protected-Accounts-and-Groups-in-Active-Directory/SAD_116.png)  
  
3.  Щелкните только что созданную группу правой кнопкой мыши, выберите пункт **Свойства** и перейдите на вкладку **Объект**. В группе **свойство объекта** выберите **защитить объект от случайного удаления**, что позволит блокировать не только иным пользователям удалять группы, но также от перемещения другое Подразделение если атрибута отличен от первого отменен.  
  
    ![Создание учетных записей управления](media/Appendix-I--Creating-Management-Accounts-for-Protected-Accounts-and-Groups-in-Active-Directory/SAD_117.png)  
  
    > [!NOTE]  
    > Если вы уже настроили разрешения на родительскую группу подразделения, чтобы ограничить администрирования к ограниченному кругу пользователей, может не потребоваться выполните следующие действия. Они получают здесь таким образом, даже если вы еще не реализован ограниченный административный контроль над структурой Подразделения, в котором вы создали эту группу, вы можете защитить группу от изменения неавторизованными пользователями.  
  
4.  Нажмите кнопку **члены** вкладке и добавьте учетные записи членами вашей команды, которые будет отвечать за включение учетных записей управления или заполнение защищенных групп при необходимости.  
  
    ![Создание учетных записей управления](media/Appendix-I--Creating-Management-Accounts-for-Protected-Accounts-and-Groups-in-Active-Directory/SAD_118.png)  
  
5.  Если вы еще не сделано, в **Active Directory — пользователи и компьютеры** последовательно щелкните **представление** и выберите **дополнительные функции**. Щелкните правой кнопкой мыши только что созданную группу, нажмите кнопку **свойства**и нажмите кнопку **безопасности** вкладки. На вкладке **Безопасность** нажмите кнопку **Дополнительно**.  
  
    ![Создание учетных записей управления](media/Appendix-I--Creating-Management-Accounts-for-Protected-Accounts-and-Groups-in-Active-Directory/SAD_119.png)  
  
6.  В **Дополнительные параметры безопасности для [Группа]** диалоговом окне щелкните **отключить наследование**. При появлении запроса, щелкните **преобразовать наследуемые разрешения в явных разрешений на этот объект**и нажмите кнопку **ОК** для возврата в группу **безопасности** диалоговое окно.  
  
    ![Создание учетных записей управления](media/Appendix-I--Creating-Management-Accounts-for-Protected-Accounts-and-Groups-in-Active-Directory/SAD_120.png)  
  
7.  На **безопасности** вкладке, удалите группы, которые не должен быть разрешен доступ к этой группе. Например если вы не хотите Authenticated Users, чтобы иметь возможность считывать имя группы и общие свойства, можно удалить ACE. Можно также удалить элементы управления доступом, например, для учетной записи операторов и пред-Windows 2000 Server доступ. Следует однако оставить минимальный набор разрешений объекта на месте. Следующие элементы управления доступом оставьте без изменений:  
  
    -   SELF  
  
    -   СИСТЕМЫ  
  
    -   Администраторы домена  
  
    -   Администраторы предприятия  
  
    -   Администраторы  
  
    -   Группа авторизации доступа Windows (если применимо)  
  
    -   КОНТРОЛЛЕРЫ ДОМЕНА ПРЕДПРИЯТИЯ  
  
    Несмотря на то, что может показаться нелогичным разрешать наивысший привилегированных групп в Active Directory для управления этой группы, задача в реализации этих параметров — не для того, чтобы предотвратить внесение изменений авторизованным членам этих групп. Вместо этого призван обеспечить при наличии требуют очень высокий уровень привилегий в редких случаях изменения авторизованных успешное завершение. Именно по этой причине, что изменение по умолчанию привилегированный вложенные группы, права и разрешения не рекомендуются в этом документе. Остаются без изменений структуры по умолчанию и очистка членство наивысший привилегий группам в каталоге, можно создать более безопасную среду, по-прежнему работает ожидаемым образом.  
  
    ![Создание учетных записей управления](media/Appendix-I--Creating-Management-Accounts-for-Protected-Accounts-and-Groups-in-Active-Directory/SAD_121.png)  
  
    > [!NOTE]  
    > Если вы еще не настроили политики аудита для объектов в структуре Подразделений, которой вы создали эту группу, следует настроить аудит для регистрации изменений этой группы.  
  
8.  Настройка группы, которая будет использоваться для «проверьте» завершена управления учетными записями, когда они нужны и «возврат» учетные записи после завершения их действия.  
  
#### <a name="creating-the-management-accounts"></a>Создание учетных записей управления

Необходимо создать по крайней мере одной учетной записи, который будет использоваться для управления членством в привилегированных группах в установке Active Directory, а лучше — вторую учетную запись для использования в качестве резервной копии. Выбор для создания учетных записей управления в одном домене в лесу и предоставить им возможности управления для всех доменов защищенных групп или информацию о том, являются ли вы решили реализовать учетным записям управления в каждом домене в лесу, процедуры фактически то же самое.  
  
> [!NOTE]  
> В этом документе предполагается, что вы еще не использовали элементы управления доступом на основе ролей и управление привилегированными пользователями для Active Directory. Таким образом некоторые процедуры должны выполняться пользователем, чья учетная запись является членом группы администраторов домена для домена в вопросе.  
>   
> При использовании учетной записи с правами DA, можно вход на контроллер домена для выполнения действий конфигурации. Шаги, которые не требуют прав DA может выполняться с меньшими правами доступа учетных записей, которые выполнили вход в административных рабочих станций. Снимки экрана, которые показывают диалоговым окнам Обычная рамка в более светлым голубым цветом представляют действия, которые могут выполняться на контроллере домена. Снимки экрана, которые показывают диалоговым окнам в темнее синий цвет представляют действия, которые могут выполняться на рабочих станциях администраторов с учетными записями, обладающих ограниченными правами.  
  
Чтобы создать учетные записи управления, выполните следующие действия.  
  
1. Войдите на контроллер домена с учетной записью, которая является членом группы домена DA.  

2. Запустите **Active Directory — пользователи и компьютеры** и перейдите к Подразделению, где вы создадите учетную запись управления.  

3. Щелкните правой кнопкой мыши Подразделение и нажмите кнопку **New** и нажмите кнопку **пользователя**.  

4. В **новый объект — пользователь** диалогового окна введите ваш требуемую информацию об именах для учетной записи и нажмите кнопку **Далее**.  

   ![Создание учетных записей управления](media/Appendix-I--Creating-Management-Accounts-for-Protected-Accounts-and-Groups-in-Active-Directory/SAD_122.png)  
  
5. Предоставить начального пароля для учетной записи пользователя, **пользователь должен сменить пароль при следующем входе в систему**выберите **Запретить смену пароля пользователем** и **учетная запись отключена**, и Нажмите кнопку **Далее**.  

   ![Создание учетных записей управления](media/Appendix-I--Creating-Management-Accounts-for-Protected-Accounts-and-Groups-in-Active-Directory/SAD_123.png)  

6. Убедитесь, что учетные данные указаны правильно и нажмите кнопку **Готово**.  

7. Щелкните правой кнопкой мыши только что созданный объект пользователя и нажмите кнопку **свойства**.  

8. Нажмите кнопку **учетной записи** вкладки.  

9. В **параметры учетной записи** выберите **учетная запись важна и не может быть делегирована** флаг, выберите **эта учетная запись поддерживает шифрование Kerberos AES 128 бит** и (или) **эта учетная запись поддерживает шифрование Kerberos AES 256** и нажать кнопку **ОК**.  

   ![Создание учетных записей управления](media/Appendix-I--Creating-Management-Accounts-for-Protected-Accounts-and-Groups-in-Active-Directory/SAD_124.png)  

   > [!NOTE]  
   > Поскольку эта учетная запись, как и другие учетные записи, будет иметь ограниченный, но мощная функция, учетной записи следует использовать только на защищенные административные узлы. Для всех защищенные административные узлы в вашей среде, следует реализовать параметр групповой политики **Сетевая безопасность: Настройка типов шифрования, разрешенных Kerberos** для разрешения только наиболее безопасный типов шифрования можно реализовать для защищенные узлы.  
   >
   > Несмотря на то, что реализация более безопасные типы шифрования для узлов не снижает количество атак кражи учетных данных, выполняет соответствующую использование и конфигурацию защищенные узлы. Задание более надежные типы шифрования для узлов, которые используются только привилегированным учетным записям просто уменьшает общую возможных направлений для атаки компьютеров.  
   >
   > Дополнительные сведения о настройке типов шифрования в системах и учетных записей, см. в разделе [конфигурации Windows для типа шифрования поддерживается Kerberos](http://blogs.msdn.com/b/openspecification/archive/2011/05/31/windows-configurations-for-kerberos-supported-encryption-type.aspx).  
   >
   > Эти параметры поддерживаются только на компьютерах под управлением Windows Server 2012, Windows Server 2008 R2, Windows 8 или Windows 7.  
  
10. На **объект** выберите **защитить объект от случайного удаления**. Это позволит блокировать не только объект удалилась (даже авторизованными пользователями), но могут помешать его перемещения в другое подразделение в иерархии Доменных службах Active Directory, если флажок снят сначала пользователем, имеющим разрешения на изменение атрибута.  

    ![Создание учетных записей управления](media/Appendix-I--Creating-Management-Accounts-for-Protected-Accounts-and-Groups-in-Active-Directory/SAD_125.png)  

11. Нажмите кнопку **удаленного управления** вкладки.  

12. Очистить **включить удаленное управление** флаг. Он никогда не требуется для подключения к этой учетной записи сеансов для реализации исправления сотрудникам службы поддержки.  

    ![Создание учетных записей управления](media/Appendix-I--Creating-Management-Accounts-for-Protected-Accounts-and-Groups-in-Active-Directory/SAD_126.png)  

    > [!NOTE]  
    > Каждый объект в Active Directory должны иметь уполномоченный владелец ИТ и владелец указанного предприятия, как описано в разделе [планирование при компрометации](../../../ad-ds/plan/security-best-practices/Planning-for-Compromise.md). При отслеживании владельца Доменных объектов в Active Directory (в отличие от внешней базы данных), следует ввести информацию о владельце соответствующие в свойствах этого объекта.

##### <a name="creating-additional-management-accounts"></a>Создание учетных записей для дополнительного управления

Можно создать дополнительное управление учетными записями, повторите предыдущие шаги, копируя учетной записи, которую вы только что создали или путем создания скрипта для создания учетных записей с требуемой конфигурации. Обратите внимание, что при копировании учетной записи, которую вы только что создали, различные способы настройки и списки управления доступом не копируются в новую учетную запись, и вам будет необходимо будет повторить большая часть действия по настройке.  
  
Вместо этого можно создать группу, к которому вы делегировать права для заполнения и unpopulate защищенных групп, но вам будет нужно обеспечить группы и учетные записи, которые вы поместите в него. Поскольку должен быть очень мало учетных записей в каталоге, которым предоставляется возможность управления членством в защищенных группах, создание отдельных учетных записей может оказаться самым простым подходом.  
  
Независимо от того, как вы решили создать группу, в который вы поместите учетные записи управления следует убедиться, что каждая учетная запись защищена, как описано выше. Также следует реализовать объект групповой Политики ограничения, похожие на описанные в [приложение г Обеспечение безопасности встроенной учетной записи администратора учетных записей в Active Directory](../../../ad-ds/plan/security-best-practices/Appendix-D--Securing-Built-In-Administrator-Accounts-in-Active-Directory.md).  
  
##### <a name="auditing-management-accounts"></a>Аудит управления учетными записями

Необходимо настроить аудит для учетной записи для входа, как минимум, все операции записи к учетной записи. Это позволит вам не только определить, успешно Включение учетной записи и сброса его пароля во время разрешенных целей, но также определить попытках неавторизованных пользователей для управления учетной записи. Сбой операции записи на учетную запись следует регистрировать в системе сведения о безопасности и наблюдения за событиями (SIEM) (если применимо) и активируемых предупреждений, которые уведомляют персонал, ответственность за анализ потенциальные уязвимости.  
  
Решения SIEM принимающим сведения о событии из источников безопасности (например, журналы событий, данные приложения, сетевые потоки, продукты защиты от вредоносных программ и источников обнаружения вторжений), разбор по копиям данных и попробуйте сделать intelligent представления и упреждающие действия . Многие коммерческие решения SIEM и на многих предприятиях создайте частные реализации. Хорошо спроектированные и соответствующим образом реализованные SIEM может существенно улучшить наблюдения за безопасностью и возможности реагирования на инциденты. Тем не менее возможности и точность отличаются очень решения. Siem выходят за рамки настоящего документа, но с любой исполнитель SIEM можно рассматривать рекомендаций определенное событие.  
  
Дополнительные сведения о параметрах конфигурации рекомендуемые аудита для контроллеров домена см. в разделе [мониторинг Active Directory для знаки нарушения](../../../ad-ds/plan/security-best-practices/Monitoring-Active-Directory-for-Signs-of-Compromise.md). Параметры конфигурации для контроллера домена, приведены в [мониторинг Active Directory для знаки нарушения](../../../ad-ds/plan/security-best-practices/Monitoring-Active-Directory-for-Signs-of-Compromise.md).  
  
#### <a name="enabling-management-accounts-to-modify-the-membership-of-protected-groups"></a>Включение управления учетных записей для изменения членства в защищенных группах

В рамках этой процедуры вы настроите разрешения на объект AdminSDHolder домена для только что созданный управления учетные записи могли изменять членство в защищенных группах в домене. Эта процедура не может выполняться через графический пользовательский интерфейс (GUI).  
  
Как уже говорилось в [приложение в: Защищенные учетные записи и группы в Active Directory](../../../ad-ds/plan/security-best-practices/Appendix-C--Protected-Accounts-and-Groups-in-Active-Directory.md), ACL для домена AdminSDHolder, объект фактически «копируется» для защищенных объектов, при выполнении задачи SDProp. Защищенных групп и учетных записей не наследуют свои разрешения на объект AdminSDHolder. их разрешения заданы явно, для получения соответствия строкам в объекте AdminSDHolder. Таким образом при изменении разрешения на объект AdminSDHolder, необходимо изменить их для атрибутов, которые подходят для типа, из защищенных объектов, который вы используете.  
  
В этом случае вам будут предоставлять учетные записи только что созданный управления, чтобы разрешить их для чтения и записи, элементы атрибута, от объектов групповой. Тем не менее объект AdminSDHolder не является объектом группы и атрибуты группы, не представлены в графическом редакторе ACL. Именно по этой причине, что вы реализуете изменения разрешений с помощью программы командной строки команды Dsacls. Для предоставления учетных записей разрешения на изменение членства в защищенных группах управления (отключено), выполните следующие действия:  
  
1. Войдите на контроллер домена, предпочтительно контроллеру домена с ролью эмулятора PDC (PDCE) с учетными данными учетной записи пользователя, стала участником этой группы DA в домене.  
  
   ![Создание учетных записей управления](media/Appendix-I--Creating-Management-Accounts-for-Protected-Accounts-and-Groups-in-Active-Directory/SAD_136.png)  
  
2. Откройте командную строку с повышенными правами щелкните правой кнопкой мыши **командной** и нажмите кнопку **Запуск от имени администратора**.  
  
   ![Создание учетных записей управления](media/Appendix-I--Creating-Management-Accounts-for-Protected-Accounts-and-Groups-in-Active-Directory/SAD_137.gif)  
  
3. При появлении предложения подтвердите повышение прав, щелкните **Да**.  
  
   ![Создание учетных записей управления](media/Appendix-I--Creating-Management-Accounts-for-Protected-Accounts-and-Groups-in-Active-Directory/SAD_138.gif)  
  
   > [!NOTE]  
   > Дополнительные сведения о повышение прав и пользователей контроль учетных записей (UAC) в Windows, см. в разделе [UAC процессы и взаимодействия](https://technet.microsoft.com/library/dd835561(v=WS.10).aspx) на веб-сайте TechNet.  
  
4. В командной строке, тип (заменив вашей информации о домене) **Dsacls [различающееся имя объекта AdminSDHolder в вашем домене] / g: отображение [учетная запись управления имя участника-пользователя]: RPWP; член**.  
  
   ![Создание учетных записей управления](media/Appendix-I--Creating-Management-Accounts-for-Protected-Accounts-and-Groups-in-Active-Directory/SAD_139.gif)

   - Член является имя свойства (атрибута), на которой будет установить разрешения  
  
   Дополнительные сведения о способах применения **Dsacls**, введите команды Dsacls без параметров в командной строке.  
  
   Если вы создали несколько учетных записей управления для домена, следует запустить команды Dsacls для каждой учетной записи. Сведения о принудительном создании SDProp для запуска, см. в разделе «SDProp ручной запуск» в [приложение в: Защищенные учетные записи и группы в Active Directory](../../../ad-ds/plan/security-best-practices/Appendix-C--Protected-Accounts-and-Groups-in-Active-Directory.md).

Вам не удается проверить, просмотрев список управления Доступом в объекте AdminSDHolder по причинам, описанным ранее, но вы можете проверить, что разрешения были применены, просмотрите списки управления доступом на защищенных групп.

#### <a name="verifying-group-and-account-configuration-settings"></a>Проверка того, группы и параметры конфигурации учетной записи

Теперь, когда вы создали и настроили учетные записи управления, которые могут изменять членство в защищенных группах в домене (которая включает самым высоким уровнем привилегий EA, DA и BA группы), следует проверить, что учетные записи и их группы управления были создался неправильно. Проверка состоит из этих общих задач:  
  
1.  Тестовой группы, который можно включать и отключать учетные записи управления, чтобы убедиться, что члены группы могут включить и отключить учетные записи и сбрасывать свои пароли, но не могут выполнять другие административные действия на учетные записи управления.  
  
2.  Проверка учетных записей управления, чтобы убедиться, что они могут добавлять и удалять участников для защищенных групп в домене, но нельзя изменить другие свойства защищенных учетных записей и групп.

1.  Чтобы проверить, включив учетную запись управления и сбросив пароль, войдите на безопасной рабочей станции администрирования с учетной записью, которая входит в группу вы создали в [приложении I: Создание учетных записей управления защищенных учетных записей и групп в Active Directory](../../../ad-ds/manage/component-updates/Appendix-I--Creating-Management-Accounts-for-Protected-Accounts-and-Groups-in-Active-Directory.md).  
  
    ![Создание учетных записей управления](media/Appendix-I--Creating-Management-Accounts-for-Protected-Accounts-and-Groups-in-Active-Directory/SAD_142.gif)  
  
2.  Откройте **Active Directory — пользователи и компьютеры**, щелкните правой кнопкой мыши учетную запись управления и нажмите кнопку **включить учетную запись**.  
  
    ![Создание учетных записей управления](media/Appendix-I--Creating-Management-Accounts-for-Protected-Accounts-and-Groups-in-Active-Directory/SAD_143.gif)  
  
3.  Отображать диалоговое окно, подтверждающее, что учетная запись включена.  
  
    ![Создание учетных записей управления](media/Appendix-I--Creating-Management-Accounts-for-Protected-Accounts-and-Groups-in-Active-Directory/SAD_144.gif)  
  
4.  Затем сбросьте пароль на учетной записи управления. Чтобы сделать это, щелкните правой кнопкой мыши учетную запись и нажмите кнопку **сброс пароля**.  
  
    ![Создание учетных записей управления](media/Appendix-I--Creating-Management-Accounts-for-Protected-Accounts-and-Groups-in-Active-Directory/SAD_145.gif)  
  
5.  Введите новый пароль для учетной записи в **новый пароль** и **подтверждение пароля** поля и нажмите кнопку **ОК**.  
  
    ![Создание учетных записей управления](media/Appendix-I--Creating-Management-Accounts-for-Protected-Accounts-and-Groups-in-Active-Directory/SAD_146.gif)  
  
6.  Появится диалоговое окно, подтверждающее, что был сброшен пароль для учетной записи.  
  
    ![Создание учетных записей управления](media/Appendix-I--Creating-Management-Accounts-for-Protected-Accounts-and-Groups-in-Active-Directory/SAD_147.gif)  
  
7.  Теперь попробуем провести изменения дополнительных свойств учетной записи управления. Щелкните правой кнопкой мыши учетную запись и нажмите кнопку **свойства**и нажмите кнопку **удаленного управления** вкладки.  
  
8.  Выберите **включить удаленное управление** и нажмите кнопку **применить**. Операция должна завершиться ошибкой и **отказано в доступе** должно появиться сообщение об ошибке.  
  
    ![Создание учетных записей управления](media/Appendix-I--Creating-Management-Accounts-for-Protected-Accounts-and-Groups-in-Active-Directory/SAD_148.gif)  
  
9. Нажмите кнопку **учетной записи** вкладку для учетной записи и предпринята попытка изменить имя учетной записи, времени входа или рабочие станции. Все должны завершиться ошибкой и параметры, которые не управляются учетной записи **userAccountControl** атрибут должен быть серым цветом и будет недоступен для изменения.  
  
    ![Создание учетных записей управления](media/Appendix-I--Creating-Management-Accounts-for-Protected-Accounts-and-Groups-in-Active-Directory/SAD_149.gif)  
  
10. Попытка добавить в группу защищенных, например группа DA группы управления. При нажатии кнопки **ОК**, должно появиться сообщение, информирующее о том, что у вас нет разрешения на изменение в группу.  
  
    ![Создание учетных записей управления](media/Appendix-I--Creating-Management-Accounts-for-Protected-Accounts-and-Groups-in-Active-Directory/SAD_150.gif)  
  
11. Выполнить дополнительную проверку, по необходимости, чтобы убедиться, что что-либо невозможно настроить для учетной записи управления, за исключением **userAccountControl** параметры и сброс пароля.  
  
    > [!NOTE]  
    > **UserAccountControl** атрибут контролирует различные варианты настройки учетной записи. Вы не можете предоставить разрешение на изменение только некоторые параметры конфигурации, при предоставлении разрешения на запись для атрибута.  
  
##### <a name="test-the-management-accounts"></a>Проверка учетных записей управления

Теперь, когда вы включили одну или несколько записей, которые можно изменить его членство в защищенных группах, можно проверить учетные записи, чтобы убедиться, что их можно изменить членство в группе защищенных, но невозможно выполнить другие изменения в защищенные учетные записи и группы.  
  
1.  Вход для надежного администратора узла первой учетной записи управления.  
  
    ![Создание учетных записей управления](media/Appendix-I--Creating-Management-Accounts-for-Protected-Accounts-and-Groups-in-Active-Directory/SAD_151.gif)  
  
2.  Запустите **Active Directory — пользователи и компьютеры** и найдите **группы "Администраторы домена"** .  
  
3.  Щелкните правой кнопкой мыши **"Администраторы домена"** группы и нажмите кнопку **свойства**.  
  
    ![Создание учетных записей управления](media/Appendix-I--Creating-Management-Accounts-for-Protected-Accounts-and-Groups-in-Active-Directory/SAD_152.gif)  
  
4.  В **свойства "Администраторы" домена**, нажмите кнопку **члены** вкладку и **щелкните** Add. Введите имя учетной записи, которая будет предоставляться временный привилегии "Администраторы домена" и нажмите кнопку **проверить имена**. Имя учетной записи подчеркнуты, нажмите кнопку **ОК** для возврата **члены** вкладки.  
  
    ![Создание учетных записей управления](media/Appendix-I--Creating-Management-Accounts-for-Protected-Accounts-and-Groups-in-Active-Directory/SAD_153.gif)  
  
5.  На **члены** вкладке **свойства "Администраторы" домена** диалоговом окне щелкните **применить**. После нажатия кнопки **применить**, учетной записи должны оставаться членом группы DA, и вы получите сообщения об ошибке.  
  
    ![Создание учетных записей управления](media/Appendix-I--Creating-Management-Accounts-for-Protected-Accounts-and-Groups-in-Active-Directory/SAD_154.gif)  
  
6.  Нажмите кнопку **управляется** вкладке **свойства "Администраторы" домена** диалоговое окно и убедитесь, что нельзя вводить текст по любым полям, и все кнопки будут недоступны.  
  
    ![Создание учетных записей управления](media/Appendix-I--Creating-Management-Accounts-for-Protected-Accounts-and-Groups-in-Active-Directory/SAD_155.gif)  
  
7.  Нажмите кнопку **Общие** вкладке **свойства "Администраторы" домена** диалоговое окно и убедитесь, что вы не может изменить любые сведения об этой вкладке.  
  
    ![Создание учетных записей управления](media/Appendix-I--Creating-Management-Accounts-for-Protected-Accounts-and-Groups-in-Active-Directory/SAD_156.gif)  
  
8.  Повторите эти шаги для дополнительных защищенных групп, при необходимости. Когда вы закончите, войдите в систему на надежного администратора узла с помощью учетной записи, которая является членом группы, созданные для включения и отключения учетных записей управления. Сбросьте пароль учетной записи управления, вы только что протестированы и отключить учетную запись. Вы завершили настройку учетных записей управления и группы, который будет отвечать за включение и отключение учетных записей.
