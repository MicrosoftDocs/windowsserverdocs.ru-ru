# <a name="executive-summary"></a>Краткий обзор

>Область применения: Windows Server 2012

>[!IMPORTANT] 
>Следующая документация по написано в 2013 и предоставляются исключительно в целях исторических.  В настоящее время мы просмотра этой документации, и это может быть изменено.  Он может не отражать методики.

Нет организаций, инфраструктура информационных технологий (ИТ) которых полностью защищена от атак, но если для защиты сегментов ключей вычислительной инфраструктуры реализованы соответствующие политики, процессы и элементы управления, может быть невозможно предотвратить событий нарушение растет оптовые компрометации вычислительную среду.  
  
Это краткий обзор предназначен для использоваться в качестве изолированного документа суммирование содержимого документа, который содержит рекомендации, которые помогут организациям в усиливают безопасность их установки Active Directory. Реализуя эти рекомендации, организации смогут идентифицировать и определять их приоритеты операции безопасности, защиты сегментов ключей вычислительной инфраструктуры своей организации и создайте элементы управления, которые значительно уменьшить вероятность успешных атак на критически важных компонентов ИТ-среде.  
  
Несмотря на то, что в этом документе рассматриваются наиболее распространенных атак Active Directory и меры противодействия, чтобы снизить вероятность атак злоумышленников, он также содержит рекомендации для восстановления в случае полной компрометации. Только что способ восстановления в случае полной компрометации Active Directory — для подготовки к работе для компрометации это произойдет.

## <a name="avenues-to-compromise"></a>Способы решения  
В этом разделе представлены сведения о некоторых из наиболее часто leveraged уязвимостей, используемых злоумышленниками для компрометации инфраструктуры клиентов. Он содержит общие категории уязвимостей и как они используются для изначально проникновение инфраструктур клиентов, распространяется компрометации на дополнительным системам и в конечном итоге целевой Active Directory и контроллерами домена получить полный контроль над лесами этими организациями. Она предоставляет подробные рекомендации о адресации каждого типа уязвимости, особенно в областях, в которых эти уязвимости не используются для ориентированы Active Directory. Тем не менее для каждого типа уязвимости, мы предоставили ссылки на дополнительную информацию для разработки противодействия и уменьшить уязвимость в организации.  
  
Включены следующие темы:  
  
-   **Начальные целевые объекты в связи с нарушением** -большинство нарушений безопасности сведения стартовым компрометация мелкие части организации инфраструктуры часто одного или двух систем одновременно. Эти события начальной или точек входа в сеть, часто воспользоваться уязвимостями, может исправлена, которые не были. Ниже перечислены часто употребляемые уязвимостей  
  
    -   Пробелы в развертываниях антивирусных программ и защиты от вредоносных программ  
  
    -   Неполная исправления  
  
    -   Устаревшие приложения и операционные системы  
  
    -   Неправильной настройки  
  
    -   Отсутствие рекомендаций по разработке безопасных приложений  
  
-   **Хороший учетные записи для кражи учетных данных** -атак кражи учетных данных являются те, в которых злоумышленник изначально получает привилегированный доступ к компьютеру в сети, а затем использует свободно доступные инструменты для извлечения учетные данные из сеансы другим учетным записям вошедшего в систему.   
    В этот раздел включены следующие:  
  
    -   **Действия, которые повысить вероятность взлома** — из-за целью кражи учетных данных обычно высокой привилегированных учетных записей домена и «очень важно человека» учетных записей (VIP), очень важно для администраторов необходимо осознавать действий, которые повышают вероятность успешных атак кражи учетных данных. Эти действия являются:  
  
        -   Вход в систему к незащищенным компьютерам с привилегированных учетных записей  
  
        -   В Интернете с высоким уровнем привилегий учетной записи  
  
        -   Настройка локальных привилегированных учетных записей с теми же учетными данными в нескольких системах  
  
        -   Overpopulation и чрезмерного использования групп привилегированных домена  
  
        -   Недостаточно управления безопасности контроллеров домена.  
  
    -   **Прав повышение прав и распространения** -определенные учетные записи, серверы и компоненты инфраструктуры обычно являются основных целей атак Active Directory. Эти учетные записи являются:  
  
        -   Окончательно привилегированных учетных записей  
  
        -   Учетные записи виртуальных IP-адресов  
  
        -   «Подключенные привилегий» учетных записей Active Directory  
  
        -   Контроллеры домена  
  
        -   Другие службы инфраструктуры, влияющие на конфигурацию, удостоверений и доступа управления, например серверы инфраструктуры открытых ключей (PKI) и серверы управления системами  
  
## <a name="reducing-the-active-directory-attack-surface"></a>Снижение уязвимостей Active Directory  
В этом разделе основное внимание уделяется технические средства, чтобы снизить вероятность атак злоумышленников установки Active Directory. В этот раздел включены следующие темы:  
  
-   **Привилегированных учетных записей и групп в Active Directory** разделе рассматриваются наивысший привилегированных учетных записей и групп в Active Directory и механизмы, по которым защищены привилегированных учетных записей. В Active Directory три встроенные группы, которые наивысший группы привилегий в каталоге ("Администраторы предприятия", "Администраторы домена" и Администраторы), несмотря на то, что количество дополнительные группы и учетные записи также должны быть защищены.  
  
-   **Реализация административных моделей наименьших прав** разделе основное внимание уделяется определение риск того, что использование высокий уровень привилегий учетные записи для повседневного администрирования представляется, кроме предоставить рекомендации для снижения этого риска.  
  
Чрезмерное привилегий только не найдены в Active Directory в скомпрометированных средах. Если организация разработала правило предоставление больше прав, чем требуется, обычно находится во всей инфраструктуры:  
  
-   В Active Directory  
  
-   На рядовых серверах  
  
-   На рабочих станциях  
  
-   В приложениях  
  
-   В репозитории данных  
  
-   **Реализация безопасного администрирования узлов** приведена безопасного администрирования узлов, компьютеры, которые будут настроены для поддержки администрирования Active Directory и связанных систем. Эти узлы стремимся административные функции и не работают под управлением программного обеспечения, например, почтовые приложения, веб-браузеры или производительности программного обеспечения (например, Microsoft Office).  
  
В этот раздел включены следующие:  
  
-   **Принципы создания безопасного администрирования узлов** -общие принципы помнить являются:  
  
    -   Никогда не Администрирование доверенного системы с менее доверенного узла.  
  
    -   Не полагайтесь на одного фактора проверки подлинности при выполнении привилегированных действий.  
  
    -   Не забывайте физической безопасности при разработке и реализация безопасного администрирования узлов.  
  
-   **Защита контроллеров домена от атак** — Если пользователь-злоумышленник получит привилегированный доступ к контроллеру домена, можно изменить этого пользователя, поврежден и уничтожить базы данных Active Directory и все учетные записи, которые управляются Active Directory и систем.  
  
В этот раздел включены следующие темы:  
  
-   **Физической безопасности для контроллеров домена** -содержатся рекомендации для обеспечения физической безопасности для контроллеров домена в центрах обработки данных, филиалов и удаленных объектах.  
  
-   **Операционные системы контроллера домена** — содержит рекомендации по защите операционные системы контроллера домена.  
  
-   **Безопасной конфигурации контроллеров домена** -конфигурации встроенных и бесплатно получить средства и параметры можно использовать для создания безопасности базовые показатели конфигурации для контроллеров домена, которые впоследствии может применяться объектов групповой политики (GPO).  
  
## <a name="monitoring-active-directory-for-signs-of-compromise"></a>Мониторинг Active Directory для обнаружения признаков компрометации  
Этот раздел посвящен устаревших аудита категорий и подкатегорий политики аудита (которые были представлены в Windows Vista и Windows Server 2008) и расширенной политики аудита (которая была введена в Windows Server 2008 R2). Также предоставляется сведения о событиях и объекты для отслеживания, может указывать на попытки скомпрометировать среде, а также некоторые дополнительные ссылки, которые можно использовать для создания всеохватывающей политики аудита для Active Directory.  
  
В этот раздел включены следующие темы:  
  
-   **Политики аудита Windows** - журналов событий безопасности Windows имеют категории и подкатегории, которые определяют, какие события безопасности отслеживаются и записи.  
  
-   **Рекомендации по политике аудита** — в этом разделе описываются параметры политики аудита Windows по умолчанию, параметры политики аудита, рекомендуется корпорацией Майкрософт и более активную рекомендации для организаций, используйте для аудита критически важных серверов и рабочих станций.  
  
## <a name="planning-for-compromise"></a>Планирование при компрометации  
Этот раздел содержит рекомендации, которые помогут организациям подготовке компрометации это произойдет, внедрите элементы управления, можно обнаружить событие компрометации до полной взлома, которые предоставляют инструкции ответ и восстановления для случаев, в которых полной компрометации каталога достигается за счет злоумышленники. В этот раздел включены следующие темы:  
  
-   **Пересмотр понятия подход** -содержит принципы и рекомендации для создания безопасных сред, в которых организации можно помещать их самые важные ресурсы. Эти рекомендации таковы:  
  
    -   Определение принципов для разделения и защиты критически важных ресурсов  
  
    -   Определение плана миграции ограниченное, основанное на риска  
  
    -   Использование «nonmigratory» миграция, при необходимости  
  
    -   Реализация «creative уничтожения»  
  
    -   Изоляция устаревших систем и приложений  
  
    -   Упрощение безопасности для конечных пользователей  
  
-   **Повышение уровня более безопасной среды** -содержит общие рекомендации предназначены для использования в качестве рекомендации для использования в разработке не только эффективной, но эффективные жизненного цикла управления. В этот раздел включены следующие темы:  
  
    -   **Создание ориентированных бизнеса по обеспечению безопасности для Active Directory** : чтобы эффективно управлять жизненным циклом пользователей, данные, приложений и систем, управляемых с Active Directory, руководствоваться следующими принципами.  
  
        -   **Назначить владельцем бизнеса данных Active Directory** -назначения владения компонентам инфраструктуры ИТ; для данных, который добавляется в доменных службах Active Directory (AD DS) для поддержки бизнес, например, новые сотрудники, новые приложения и новые сведения о репозитории, назначенный подразделения или пользователь должен быть связан с данными.  
  
        -   **Реализация управления жизненным циклом Business-Driven** -жизненного цикла управления должен быть реализован для данных в Active Directory.  
  
        -   **Классифицировать все данные Active Directory** -владельцы предприятия должен предоставить классификации данных в Active Directory. В рамках модели классификации данных классификации для следующих данных Active Directory должны быть включены:  
  
            -   **Систем** -классифицировать заполнений сервера, операционную систему роли, приложения, выполняющиеся на них, а ИТ и бизнеса Владельцы записи.  
  
            -   **Приложения** -классифицировать приложений, функции, базу пользователей и их операционной системы.  
  
            -   **Пользователи** -учетных записей в Active Directory установки, наиболее подверженные атакам злоумышленников следует с тегами и отслеживаться.
