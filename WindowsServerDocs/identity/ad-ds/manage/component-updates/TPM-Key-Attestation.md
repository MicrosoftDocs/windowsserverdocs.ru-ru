## <a name="overview"></a>Обзор  
При поддержке для ключей, защищенных доверенным платформенным Модулем существовала с момента выпуска Windows 8, не были не механизмы для ЦС, чтобы криптографически аттестацию фактически защиту закрытого ключа инициатор запроса сертификата с доверенного платформенного модуля (TPM). Это обновление позволяет ЦС для выполнения этого аттестации и с учетом этого аттестации в выданном сертификате.  
  
> [!NOTE]  
> В этой статье предполагается, что читатель имеет знакомы с понятием шаблон сертификата (Справочник по см. в разделе [шаблонов сертификатов](https://technet.microsoft.com/library/cc730705.aspx)). Также предполагается, что читатель имеет опыт Настройка ЦС предприятия для выдачи сертификатов на основе шаблонов сертификатов (Справочник по см. в разделе [контрольный список: Настройка центров сертификации, выдача и управление сертификатами](https://technet.microsoft.com/library/cc771533.aspx)).  
  
### <a name="terminology"></a>Терминология  
  
|Термин|Определение|  
|--------|--------------|

|EKpub|Указывает на открытый ключ EK.|  
|EKPriv|Указывает на закрытый ключ EK.|  
|EKCert|Сертификат EK. Доверенный платформенный модуль производителем сертификат, выданный для EKPub. Не все доверенные платформенные модули имеют EKCert.|  
|ДОВЕРЕННЫЙ ПЛАТФОРМЕННЫЙ МОДУЛЬ|Доверенный платформенный модуль. Доверенный платформенный модуль предназначен для предоставления аппаратных функций безопасности. Микросхема TPM — это надежный криптографический процессор, которые предназначены для выполнения операций шифрования. Микросхему содержит несколько механизмов физической безопасности, чтобы сделать его подделки и вредоносных программ не удалось незаконно функций безопасности доверенного платформенного МОДУЛЯ.|  
  
### <a name="background"></a>Фон  
Начиная с Windows 8, доверенного платформенного модуля (TPM) можно использовать для защиты закрытого ключа сертификата. Корпорация Майкрософт платформы Crypto поставщика поставщик хранилища ключей (KSP) включено. Было две проблемы с реализацией:  

-   Не было никакой гарантии, что ключ действительно защищен TPM (кто-то могут легко подделать KSP программного обеспечения как в KSP доверенного платформенного МОДУЛЯ с помощью учетной записи локального администратора).

-   Не удалось ограничить список доверенных платформенных модулей, которые разрешены для защиты организации сертификатов, выданных (в случае, если администратору PKI для управления типов устройств, которые могут использоваться для получения сертификатов в среде).  

### <a name="tpm-key-attestation"></a>Аттестация ключа TPM  
Аттестация ключа TPM является возможность субъекта, запрашивающего сертификат криптографически подтвердить ЦС, что ключ RSA в запросе на сертификат защищен «» или «» доверенным платформенным Модулем, которому доверяет центр сертификации. Модель доверия TPM более рассматривается в [Общие сведения о развертывании](../../../ad-ds/manage/component-updates/TPM-Key-Attestation.md#BKMK_DeploymentOverview) далее в этом разделе.  
  
### <a name="why-is-tpm-key-attestation-important"></a>Почему важно аттестации ключей доверенного платформенного МОДУЛЯ  
Сертификата пользователя с помощью ключа подтверждения TPM предоставляет гарантию повышенной безопасности, по запретом экспорта, противодействием подбору паролей и изоляцией ключей, предоставленных TPM.  
  
В аттестации ключей доверенного платформенного МОДУЛЯ, новая концепция управления теперь возможно: администратор может определить набор устройств, которые пользователи могут использовать для доступа к корпоративным ресурсам (таких как VPN или беспроводной точкой доступа) и **надежный** гарантирует, что другие устройства можно использовать для доступа к ним. — Это новая концепция управления доступа **надежный** поскольку привязана к *привязкой оборудования* удостоверение пользователя, которое надежнее, чем учетных данных на основе программного обеспечения.
  
### <a name="how-does-tpm-key-attestation-work"></a>Как работает аттестации ключей доверенного платформенного МОДУЛЯ?  
Как правило Аттестация ключа TPM основан на следующих столбы:  
  
1.  Каждый TPM поставляется с уникальным асимметричного ключа называется *ключ подтверждения* (EK) встроенного производителем. Мы будем называть открытая часть ключ как *EKPub* и связанный закрытый ключ, как *EKPriv*. В некоторых чипах TPM также иметь сертификат EK, выданного производителем для EKPub. Мы будем называть этот сертификат как *EKCert*.  
  
2.  ЦС устанавливает отношения доверия в доверенном платформенном МОДУЛЕ через EKPub или EKCert.  
  
3.  Пользователь предоставляет ЦС, что ключ RSA, для которого запрашивается сертификат криптографически отношение к EKPub и что пользователь является владельцем EKpriv.  
  
4.  Центр сертификации выдает сертификат с помощью политики выдачи специальный OID означают, что ключ теперь аттестацию под защитой доверенного платформенного МОДУЛЯ.  
  
## <a name="BKMK_DeploymentOverview"></a>Обзор развертывания  
В этом сценарии предполагается, настроить ЦС предприятия Windows Server 2012 R2. Кроме того шаблоны сертификатов клиентов, которое (Windows 8.1) настроены для регистрации, используя Центр сертификации предприятия. 

Существует три шага к развертыванию аттестации ключей доверенного платформенного МОДУЛЯ.  
  
1.  **Планирование модель отношений доверия доверенного платформенного МОДУЛЯ:** первый шаг состоит в выборе модели доверия TPM для использования. Существует 3 поддерживаемых способов сделать это:  
  
    -   **Отношения доверия на основании учетных данных пользователя:** ЦС предприятия доверия EKPub пользовательских как часть запроса на сертификат, и проверка не выполняется отличные от учетных данных пользователя домена.  
  
    -   **Отношения доверия на основании EKCert:** ЦС предприятия проверка цепочки сертификатов, который предоставляется как часть запроса на сертификат по списку управляемые администратором *допустимые цепочки сертификатов EK*. Допустимые цепочек являются определенного изготовителя и выражаются через два хранилища настраиваемого сертификата в ЦС (один магазин для промежуточных) и один для сертификатов корневого центра сертификации. Этот режим доверия означает, что **все** TPM конкретного изготовителя являются доверенными. Обратите внимание, что в этом режиме доверенных платформенных модулей, используемых в среде должен содержать EKCerts.
  
    -   **Отношения доверия на основании EKPub:** ЦС предприятия проверяет, что EKPub, предоставляются как часть запроса на сертификат отображается в списке управляемые администратором разрешен EKPubs. Этот список выражается каталог файлов, где имя каждого файла в этот каталог является хэш SHA-2 разрешенных EKPub. Этот параметр обеспечивает высокий уровень надежности, но требует дополнительных усилий с правами администратора, поскольку каждое устройство идентифицируется по отдельности. В этой модели доверия могут только на устройствах, которые были добавлены в список разрешенных для EKPubs EKPub своим доверенным платформенным Модулем, подачи заявки на сертификат, аттестованного для доверенного платформенного МОДУЛЯ.  
  
    В зависимости от того, какой метод проверки ЦС применяются политики выдачи различных OID выданный сертификат. Дополнительные сведения о политики выдачи OID см. в таблице OID политик выдачи в [Настройка шаблона сертификата](../../../ad-ds/manage/component-updates/TPM-Key-Attestation.md#BKMK_ConfigCertTemplate) в этой статье.  
  
    Обратите внимание, что можно выбрать сочетание модели доверия доверенного платформенного МОДУЛЯ. В этом случае ЦС будет принимать все методы аттестации и политики выдачи, OID будут отражены все аттестации методы, которые завершились успешно.  
  
2.  **Настройка шаблона сертификата:** настройки шаблона сертификата описано в [сведения о развертывании](../../../ad-ds/manage/component-updates/TPM-Key-Attestation.md#BKMK_DeploymentDetails) в этой статье. В этой статье рассматриваются способы назначения этого шаблона сертификата ЦС предприятия или не как зарегистрировать доступ предоставляется группе пользователей. Дополнительные сведения см. в разделе [контрольный список: Настройка центров сертификации, выдача и управление сертификатами](https://technet.microsoft.com/library/cc771533.aspx).  
  
3.  **Настройка ЦС для доверенного платформенного МОДУЛЯ модели доверия**  
  
    1.  **Отношения доверия на основании учетных данных пользователя:** никаких особых настроек не требуется.  
  
    2.  **Отношения доверия на основании EKCert:** администратору необходимо получить цепочки сертификатов EKCert производители TPM и импортировать их на двух новых хранилищ сертификатов, созданные администратором, в ЦС, который выполнять аттестацию ключей доверенного платформенного МОДУЛЯ. Дополнительные сведения см. в разделе [конфигурации ЦС](../../../ad-ds/manage/component-updates/TPM-Key-Attestation.md#BKMK_CAConfig) в этой статье.  
  
    3.  **Отношения доверия на основании EKPub:** администратору необходимо получить EKPub для каждого устройства, потребуется сертификаты аттестованного для доверенного платформенного МОДУЛЯ и добавить их в список разрешенных EKPubs. Дополнительные сведения см. в разделе [конфигурации ЦС](../../../ad-ds/manage/component-updates/TPM-Key-Attestation.md#BKMK_CAConfig) в этой статье.  
  
    > [!NOTE]  
    > -   Для этой функции требуется Windows 8.1 или Windows Server 2012 R2.  
    > -   Аттестация ключей TPM для смарт-карты сторонних поставщиков хранилища ключей не поддерживается. Необходимо использовать KSP поставщика шифрования платформы Майкрософт.  
    > -   Аттестация ключа TPM работает только для ключей RSA.  
    > -   Аттестация ключа TPM не поддерживается для изолированного ЦС.  
    > -   Аттестация ключа TPM не поддерживает [обработки временных сертификат](https://technet.microsoft.com/library/ff934598).

### <a name="BKMK_CAConfig"></a>Конфигурации ЦС  
  
1.  **Настройка это EKCA и EKROOT в хранилищах сертификатов выдающий ЦС**  
  
    Если вы выбрали **сертификат подтверждения** параметры шаблона выполните следующие действия по настройке:  
  
    1.  Используйте Windows PowerShell, чтобы создать два новых хранилищ сертификатов на сервер центра сертификации (ЦС), выполняющее аттестации ключей доверенного платформенного МОДУЛЯ.  
  
    2.  Получите промежуточные и корневой ЦС от производителей, которые вы хотите разрешить в среде предприятия. Эти сертификаты необходимо импортировать в магазинах ранее созданный сертификат (это EKCA и EKROOT) соответствующим образом.  
  
    Следующий сценарий Windows PowerShell выполняет оба этих действия. В следующем примере изготовителя доверенного платформенного МОДУЛЯ Fabrikam предоставила корневой сертификат *FabrikamRoot.cer* и сертификат ЦС *Fabrikamca.cer*.  
  
    ```powershell  
    PS C:>\cd cert:  
    PS Cert:\>cd .\\LocalMachine  
    PS Cert:\LocalMachine> new-item EKROOT  
    PS Cert:\ LocalMachine> new-item EKCA  
    PS Cert:\EKCA\copy FabrikamCa.cer .\EKCA  
    PS Cert:\EKROOT\copy FabrikamRoot.cer .\EKROOT  
    ```  
  
2.  **Настраивать список EKPUB список, если с помощью типа EK аттестации**  
  
    Если вы выбрали **ключ подтверждения** в параметрах шаблона далее действия по настройке для создания и настройки папки на ЦС, содержащий файлы с размером 0 байт, каждый с именем хэш SHA-2 разрешенных EK. В этой папке выступает в качестве «список разрешенных» устройств, которые могут получить сертификаты аттестацию ключей доверенного платформенного МОДУЛЯ. Поскольку EKPUB для каждого устройства, которое требует получения сертификата необходимо добавить вручную, он предоставляет предприятия гарантирует устройств, которые авторизованы для получения сертификаты аттестованным ключом TPM. Настройка ЦС в этом режиме состоит из двух этапов:  
  
    1.  **Создает запись реестра EndorsementKeyListDirectories:** использовать средство командной строки Certutil для настройки расположения папок, где устанавливаются доверенных EKpubs, как описано в следующей таблице.  
  
        |Операция|Синтаксис команд|  
        |-------------|------------------|  
        |Добавить расположения папок|Certutil.exe - setreg CA\EndorsementKeyListDirectories + «<folder>»|  
        |Удаление расположения папок|CA\EndorsementKeyListDirectories - setreg certutil.exe-«<folder>»|  
  
        EndorsementKeyListDirectories в команды certutil — это параметр реестра, как описано в следующей таблице.  
  
        |Имя значения|Тип|Данные|  
        |--------------|--------|--------|  
        |EndorsementKeyListDirectories|REG_MULTI_SZ|< ЛОКАЛЬНЫЙ или UNC-путь к EKPUB разрешить списки ><br /><br />Пример:<br /><br />*\\\blueCA.contoso.com\ekpub*<br /><br />*\\\bluecluster1.contoso.com\ekpub*<br /><br />D:\ekpub|  
  
        HKLM\SYSTEM\CurrentControlSet\Services\CertSvc\Configuration\\<CA Sanitized Name>  
  
        *EndorsementKeyListDirectories* будет содержать список UNC-путь или пути к локальной файловой системы, каждый из которых указывает на папку, что ЦС имеет доступ для чтения. Каждая папка может содержать ноль или больше разрешить элементы списка, где каждая запись — это файл с именем, которое является SHA-2 хэш доверенных EKpub, без расширения файла. 
        Создание или изменение данной конфигурации ключа реестра требуется перезапуск центра сертификации, так же, как существующие параметры конфигурации реестра центра сертификации. Тем не менее изменения параметра конфигурации вступят в силу немедленно и не потребует перезапуска ЦС.  
  
        > [!IMPORTANT]  
        > Безопасность папки в списке от взлома и несанкционированного доступа, настроив разрешения, чтобы только авторизованные администраторы чтение и запись. Учетная запись компьютера ЦС требует доступ только для чтения.  
  
    2.  **Заполнить список EKPUB:** использовать следующий командлет Windows PowerShell для получения хэш открытого ключа EK доверенного платформенного МОДУЛЯ с помощью Windows PowerShell на каждом устройстве, а затем отправить этот открытого ключа хэш для ЦС и сохранить ее в папке EKPubList.  
  
        ```powershell  
        PS C:>\$a=Get-TpmEndorsementKeyInfo -hashalgorithm sha256  
        PS C:>$b=new-item $a.PublickKeyHash -ItemType file  
        ```  
  
## <a name="troubleshooting"></a>Устранение неполадок  
  
### <a name="key-attestation-fields-are-unavailable-on-a-certificate-template"></a>Аттестация ключа поля недоступны в шаблоне сертификата  
Аттестация по ключу поля недоступны, если параметры шаблона не соответствуют требованиям для аттестации. Распространенные из них:  
  
1.  Параметры совместимости не настроены правильно. Убедитесь, что они настроены следующим образом:  
  
    1.  **Центр сертификации**: **Windows Server 2012 R2**  
  
    2.  **Получатель сертификата**: **Windows 8.1 или Windows Server 2012 R2**  
  
2.  Параметры шифрования не настроены правильно. Убедитесь, что они настроены следующим образом:  
  
    1.  **Категория поставщика**: **поставщик хранилища ключей**  
  
    2.  **Имя алгоритма**: **RSA**  
  
    3.  **Поставщики**: **поставщик шифрования платформы Майкрософт**  
  
3.  Параметры обработки запроса не настроены правильно. Убедитесь, что они настроены следующим образом:  
  
    1.  **Разрешить экспортировать закрытый ключ** не должен быть выбран вариант.  
  
    2.  **Архивировать закрытый ключ шифрования субъекта** не должен быть выбран вариант.  
  
### <a name="verification-of-tpm-device-for-attestation"></a>Проверка устройства TPM для Аттестация  
Используйте командлет Windows PowerShell, **подтверждение CAEndorsementKeyInfo**, чтобы убедиться, что конкретное устройство доверенного платформенного МОДУЛЯ является доверенным для аттестации ЦС. Существует два варианта: один для проверки EKCert, а другой — для проверки EKPub. Командлет либо запускается локально на ЦС или на удаленном ЦС с помощью удаленного взаимодействия Windows PowerShell.  
  
1.  Для проверки доверия на EKPub, выполните следующие два действия:  
  
    1.  **Извлеките EKPub с клиентского компьютера:** EKPub можно извлечь с клиентского компьютера через **Get-TpmEndorsementKeyInfo**. Из командной строки с повышенными привилегиями выполните следующую команду:  
  
        ```  
        PS C:>\$a=Get-TpmEndorsementKeyInfo -hashalgorithm sha256  
        ```  
  
    2.  **Проверка отношения доверия на EKCert на компьютере ЦС:** скопируйте извлеченные строку (SHA-2 хэш EKPub) к серверу (например, по электронной почте) и передайте его в командлет Confirm-CAEndorsementKeyInfo. Обратите внимание, что этот параметр должен быть 64 символов.  
  
        ```  
        Confirm-CAEndorsementKeyInfo [-PublicKeyHash] <string>  
        ```  
  
2.  Для проверки доверия на EKCert, выполните следующие два действия:  
  
    1.  **Извлеките EKCert с клиентского компьютера:** EKCert можно извлечь с клиентского компьютера через **Get-TpmEndorsementKeyInfo**. Из командной строки с повышенными привилегиями выполните следующую команду:  
  
        ```  
        PS C:>\$a= Get- TpmEndorsementKeyInfo  
        PS C:>\$a.manufacturerCertificates|Export-Certificate c:\myEkcert.cer  
        ```  
  
    2.  **Проверка отношения доверия на KCert на компьютере ЦС:** скопируйте извлеченные (EkCert.cer) EKCert ЦС (например, по электронной почте или xcopy). Например если вы копируете файл сертификата в папку «c:\diagnose» на сервере ЦС, выполните следующую команду для завершения проверки:  
  
        ```  
        PS C:>new-object System.Security.Cryptography.X509Certificates.X509Certificate2 "c:\diagnose\myEKcert.cer" | Confirm-CAEndorsementKeyInfo  
        ```
