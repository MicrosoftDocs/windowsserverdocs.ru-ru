# <a name="securing-privileged-access"></a>Защита привилегированного доступа

>Область применения. Windows Server

Обеспечение безопасности привилегированного доступа — это критически важный первый шаг для защиты бизнес-ресурсов в современных организациях. Безопасность большинства или всех бизнес-активов в организации ИТ зависит от целостности привилегированных учетных записей, используемых для администрирования, управления и разработки. Кибератак-злоумышленники часто используют эти учетные записи и другие элементы привилегированного доступа для получения доступа к данным и системам, используя атаки с кражой учетных данных, такие как [Pass-a-hash и Pass-a-Ticket](https://www.microsoft.com/pth).

Для защиты привилегированного доступа к определенным злоумышленниковам необходимо принять полный и продуманный подход к изоляции этих систем от рисков.

## <a name="what-are-privileged-accounts"></a>Что такое привилегированные учетные записи?

Прежде чем говорить о том, как защитить их, можно определить привилегированные учетные записи.

Привилегированные учетные записи, такие как администраторы служб домен Active Directory, имеют прямой или косвенный доступ к большинству или всем ресурсам в ИТ-организации, что приводит к компрометации этих учетных записей на значительный риск.

## <a name="why-securing-privileged-access-is-important"></a>Зачем важнее безопасность привилегированного доступа?

Кибератак — злоумышленники могут сосредоточиться на привилегированном доступе к таким системам, как Active Directory (AD), чтобы быстро получить доступ ко всем целевым данным Организации. Традиционные подходы к безопасности посвящены сети и брандмауэру в качестве основного периметра безопасности, но эффективность сетевой безопасности значительно снижена за счет двух тенденций:

* Организации размещаются данные и ресурсы за пределами традиционной сетевой границы на мобильных ПК, таких как мобильные телефоны и планшеты, облачные службы, а также для переноса собственных устройств (BYOD).
* Злоумышленники постоянно получают доступ к рабочим станциям в пределах сети с помощью фишинга, атак через электронную почту и других атак через Интернет.

В дополнение к традиционной стратегии периметра сети, эти факторы должны создавать современный периметр безопасности из элементов управления проверки подлинности и авторизации. Периметр безопасности здесь определяется как согласованный набор элементов управления между ресурсами и угрозами для них. Привилегированные учетные записи эффективно управляют этой новой периметром безопасности, поэтому важно защищать привилегированный доступ.

### <a name="1-separate-accounts"></a>1. Отдельные учетные записи

Чтобы отделить Интернет-риски (атаки с фишингом, просмотр в Интернете) из учетных записей привилегированного доступа, создайте специальную учетную запись для всех сотрудников с привилегированным доступом. Администраторы не должны просматривать веб-сайт, проверять их электронную почту и выполнять задачи по повышению производительности с высокопривилегированными учетными записями. Дополнительные сведения об этом можно найти в разделе [отдельные административные учетные записи](securing-privileged-access-reference-material.md#separate-administrative-accounts) справочного документа.

Следуйте указаниям в статье [Управление учетными записями аварийного доступа в Azure AD](/azure/active-directory/users-groups-roles/directory-emergency-access) , чтобы создать по крайней мере две учетные записи для аварийного доступа с постоянными назначенными правами администратора в локальных средах AD и Azure AD. Эти учетные записи используются только в том случае, если традиционные учетные записи администраторов не могут выполнить необходимую задачу, например в случае аварии.

### <a name="2-just-in-time-local-admin-passwords"></a>2. JIT — пароли локальных администраторов

Чтобы снизить риск кражи хэша пароля учетной записи локального администратора из локальной базы данных SAM и злоумышленниками его к атакам других компьютеров, организациям следует убедиться, что каждый компьютер имеет уникальный пароль локального администратора. Средство локального администратора с паролем (Lap) позволяет настроить уникальные случайные пароли на каждой рабочей станции и сервере, которые сохраняют их в Active Directory (AD), защищенном ACL. Только разрешенные пользователи могут читать или запрашивать сброс паролей учетной записи локального администратора. Вы можете получить Lap для использования на рабочих станциях и серверах из [центра загрузки Майкрософт](http://Aka.ms/LAPS).

Дополнительные рекомендации по работе с средой с Lap и лапы можно найти в разделе [эксплуатационные стандарты на основе принципа чистого источника](securing-privileged-access-reference-material.md#operational-standards-based-on-clean-source-principle).

### <a name="3-administrative-workstations"></a>3. Рабочие станции администрирования

В качестве начальной меры безопасности для пользователей с Azure Active Directory и традиционными административными правами Active Directory, убедитесь, что они используют устройства Windows 10, настроенные с использованием [стандартов для высокозащищенного устройства Windows 10. ](/windows-hardware/design/device-experiences/oem-highly-secure). Учетные записи привилегированного администратора не должны быть членами локальной группы администраторов рабочих станций администрирования.  Повышение привилегий через контроль доступа пользователей (UAC) можно использовать, если требуется внести изменения в конфигурацию рабочих станций.  Кроме того, для дополнительной защиты устройства на рабочих станциях необходимо применить базовый план безопасности Windows 10.

### <a name="4-identity-attack-detection"></a>4. Обнаружение атак с идентификацией

[Azure Advanced Threat protection (ATP)](/azure-advanced-threat-protection/what-is-atp) — это облачное решение для обеспечения безопасности, которое идентифицирует, обнаруживает и помогает исследовать сложные угрозы, скомпрометированные удостоверения и действия по предварительной проверке, направленные на локальный Active Directory PXE.

Этап 2 построен на работе, выполненной на этапе 1, и предназначен для выполнения в течение примерно 90 дней. На этой схеме показаны шаги этого этапа.

![Схема фазы 2: 1. Windows Hello для бизнеса или MFA, 2. ПРИВИЛЕГИРОВАННЫМ доступом, 3. Время простоя, 4. Credential Guard, 5. Утечки учетных данных 6. Обнаружение уязвимости перемещения бокового смещения](../media/securing-privileged-access/PAW_LP_Fig7.JPG)

### <a name="1-require-windows-hello-for-business-and-mfa"></a>1. Требовать Windows Hello для бизнеса и MFA

Администраторы могут воспользоваться преимуществами простоты использования, связанной с Windows Hello для бизнеса. Администраторы могут заменить свои сложные пароли на надежные двухфакторную проверку подлинности на компьютерах. Злоумышленник должен иметь как устройство, так и биометрические сведения или ПИН-код, гораздо труднее получить доступ без знания сотрудника. Дополнительные сведения о Windows Hello для бизнеса и пути для развертывания можно найти в статье [Обзор Windows Hello для бизнеса](/windows/security/identity-protection/hello-for-business/hello-overview) .

Включите многофакторную проверку подлинности (MFA) для учетных записей администратора в Azure AD с помощью Azure MFA. Как минимум включите [политику условного доступа с базовой защитой](/azure/active-directory/conditional-access/baseline-protection#require-mfa-for-admins) . Дополнительные сведения о многофакторной идентификации Azure см. в статье [Развертывание облачной многофакторной идентификации Azure](/azure/active-directory/authentication/howto-mfa-getstarted) .

### <a name="2-deploy-paw-to-all-privileged-identity-access-account-holders"></a>2. Развертывание ПРИВИЛЕГИРОВАННЫМ доступом для всех владельцев учетных записей доступа к привилегированным удостоверениям

Продолжая процесс отделения привилегированных учетных записей от угроз, обнаруженных в электронной почте, веб-просмотре и других задачах, не связанных с администрированием, следует реализовать рабочие станции с выделенным привилегированным доступом (ПРИВИЛЕГИРОВАННЫМ доступом) для всех сотрудников с привилегированным доступом к вашему информационные системы Организации. Дополнительные рекомендации по развертыванию ПРИВИЛЕГИРОВАННЫМ доступом можно найти в статье [привилегированный доступ рабочие станции](privileged-access-workstations.md#paw-phased-implementation).

### <a name="3-just-in-time-privileges"></a>3. Права на JIT

Чтобы уменьшить время получения привилегий и повысить видимость их использования, предоставьте привилегии JIT-КОМПИЛЯТОР с помощью соответствующего решения, такого как ниже или других сторонних решений:

* Для доменных служб Active Directory (AD DS) используйте [диспетчер привилегированного доступа](/microsoft-identity-manager/pam/privileged-identity-management-for-active-directory-domain-services), предоставляемый Microsoft Identity Manager.
* Для Azure Active Directory используйте [Azure AD Privileged Identity Management](/azure/active-directory/privileged-identity-management/pim-deployment-plan).

### <a name="4-enable-windows-defender-credential-guard"></a>4. Включение Credential Guard в Защитнике Windows

Включение Credential Guard помогает защищать хэши паролей NTLM, билеты Kerberos и учетные данные, хранящиеся в приложениях, как учетные данные домена. Эта возможность позволяет предотвратить атаки, направленные на хищение учетных данных, такие как Pass-a-hash или Pass-a-ticket, увеличивая сложность обработки в среде с помощью заимствованных учетных данных. Сведения о принципах работы Credential Guard и развертывании см. в статье [Защита производных учетных записей домена с помощью Credential Guard в Защитнике Windows](/windows/security/identity-protection/credential-guard/credential-guard).

### <a name="5-leaked-credentials-reporting"></a>5. Отчеты об утечке учетных данных

"Каждый день Корпорация Майкрософт анализирует свыше 6 500 000 000 000 сигналов для выявления новых угроз и защиты клиентов" — [Майкрософт по числу](https://news.microsoft.com/bythenumbers/cyber-attacks)

Включите Microsoft Azure AD защиту идентификации, чтобы сообщить о пользователях с потерянными учетными данными, чтобы их можно было исправить. [Защита идентификации Azure AD](/azure/active-directory/identity-protection/index) можно использовать, чтобы помочь вашей организации защищать облачные и гибридные среды от угроз.

### <a name="6-azure-atp-lateral-movement-paths"></a>6. Пути перемещения бокового смещения Azure ATP

Убедитесь, что владельцы учетных записей привилегированного доступа используют свои ПРИВИЛЕГИРОВАННЫМ доступом для администрирования, чтобы скомпрометированные непривилегированные учетные записи не могли получить доступ к привилегированной учетной записи с помощью атак с хищением учетных данных, таких как Pass-The-hash или Pass-the-Ticket. [Пути перемещения Azure ATP бокового смещения (лмпс)](/azure-advanced-threat-protection/use-case-lateral-movement-path) позволяют легко понять отчеты, чтобы определить, какие привилегированные учетные записи могут быть открыты для компрометации.

## <a name="phase-3-security-improvement-and-sustainment"></a>Этап 3. Повышение безопасности и непрерывность

Этап 3 стратегии строится на этапах 1 и 2, чтобы усилить безопасность. Этап 3 наглядно изображен на этой схеме:

![Этап 3. 1. Проверьте RBAC, 2. Уменьшение поверхностей атаки, 3. Интегрируйте журналы с СЕИМ, 4. Автоматизация утечки учетных данных](../media/securing-privileged-access/PAW_LP_Fig8.JPG)

Эти возможности посвящены этапам, описанным на предыдущих этапах, и перемещают средства защиты в более упреждающий уровень. На этом этапе не определена временная шкала, и ее реализация может занять больше времени в зависимости от конкретной организации.

### <a name="1-review-role-based-access-control"></a>1. Проверка управления доступом на основе ролей

Используя три многоуровневые модели, описанные в статье [Active Directory модель административного уровня](securing-privileged-access-reference-material.md), проверьте и убедитесь, что администраторы более низкого уровня не имеют административного доступа к ресурсам более высокого уровня (членство в группах, списки ACL для учетных записей пользователей и т. д.).

### <a name="2-reduce-attack-surfaces"></a>2. Уменьшение поверхностей атак

Защита рабочих нагрузок удостоверений, включая домены, контроллеры доменов, ADFS и Azure AD Connect, так как безопасность одной из этих систем может привести к компрометации других систем в Организации. Статьи, [снижающие Active Directory уязвимости](../ad-ds/plan/security-best-practices/reducing-the-active-directory-attack-surface.md) , и [пять действий по защите инфраструктуры идентификации](/azure/security/azure-ad-secure-steps) предоставляют рекомендации по защите локальных и гибридных сред удостоверений.

### <a name="3-integrate-logs-with-siem"></a>3. Интеграция журналов с SIEM

Интеграция ведения журналов в централизованное средство SIEM помогает Организации анализировать, обнаруживать и реагировать на события безопасности. Статьи, [отслеживаемые Active Directory для подписывания компромиссов](../ad-ds/plan/security-best-practices/monitoring-active-directory-for-signs-of-compromise.md) и [приложений L: Отслеживаемые события](../ad-ds/plan/appendix-l--events-to-monitor.md) предоставляют рекомендации по событиям, которые следует отслеживать в среде.

Это часть плана, за исключением того, что для агрегирования, создания и настройки оповещений в сведениях о безопасности и управлении событиями (SIEM) требуются квалифицированные аналитики (в отличие от Azure ATP в 30-дневном плане, который включает в себя оповещение).

### <a name="4-leaked-credentials---force-password-reset"></a>4. Утечки учетных данных — принудительный сброс пароля

Продолжайте повышать уровень безопасности, позволяя защита идентификации Azure AD автоматически принудительно сбрасывать пароли при подозрении на компрометации. В рекомендациях, приведенных в статье [Использование событий риска для активации многофакторной проверки подлинности и изменения паролей](/azure/active-directory/authentication/tutorial-risk-based-sspr-mfa) , объясняется, как включить это с помощью политики условного доступа.

## <a name="am-i-done"></a>Дальнейшие действия

Короткий ответ — «нет».

Злоумышленники никогда не останавливаются, поэтому ни одна из них не может. Эта схема поможет вашей организации защититься от известных угроз, так как злоумышленники постоянно развиваются и сдвигаются. Мы рекомендуем просматривать безопасность как непрерывный процесс, посвященный созданию затрат, и уменьшая степень успешности злоумышленников, предназначенную для вашей среды.

Хотя это не единственная часть программы безопасности Организации, которая защищает привилегированный доступ, является важнейшим компонентом стратегии безопасности.
