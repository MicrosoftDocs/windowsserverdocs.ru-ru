# <a name="privileged-access-workstations"></a>Рабочие станции c привилегированным доступом

>Область применения. Windows Server

Рабочие станции с привилегированным доступом предоставляют выделенную операционную систему для конфиденциальных задач, защищенную от атак из Интернета и векторов угроз. Отделение этих конфиденциальных задач и учетных записей от ежедневного использования рабочих станций и устройств обеспечивает очень надежную защиту от фишинг-атаках, приложения и уязвимости ОС, различных атак олицетворения и атак кражи учетных данных, таких как нажатие клавиши Ведение журнала, [Pass--Hash](https://aka.ms/pth)и Pass--Ticket.

## <a name="what-is-a-privileged-access-workstation"></a>Что такое рабочей станции привилегированного доступа?

Проще говоря рабочая станция c привилегированным доступом — это закрытая рабочая станция с высоким уровнем защиты, предназначенная для обеспечения высокого уровня безопасности для конфиденциальных учетных записей и задач.  Рабочие станции c привилегированным доступом рекомендуется использовать для администрирования систем удостоверений, облачных служб, структуры частного облака и конфиденциальных бизнес-функций.

> [!NOTE]
> Архитектура рабочей станции c привилегированным доступом не требует четкого соответствия между учетными записями и рабочими станциями, хотя это и стандартная конфигурация. Рабочая станция с привилегированным доступом создает среду надежных рабочих станций, которые могут использовать одна или несколько учетных записей.

Чтобы обеспечить максимальную безопасность, привилегированным доступом следует всегда запускать наиболее защиту и обновление операционной системы доступны: Корпорация Майкрософт настоятельно рекомендует Windows 10 Корпоративная, включающую несколько дополнительные функции безопасности не доступен в других выпусках (в частности, [Credential Guard](https://technet.microsoft.com/library/mt483740%28v=vs.85%29.aspx) и [Device Guard](https://technet.microsoft.com/library/dn986865(v=vs.85).aspx)).

> [!NOTE]
> Организации, имеющие доступ к Windows 10 Корпоративная, могут использовать Windows 10 Pro, содержащую множество важных базовых технологий для рабочих станций с привилегированным доступом, например надежная загрузка, BitLocker и удаленный рабочий стол.  Клиенты, связанные со сферой образования, могут использовать Windows 10 для образовательных учреждений.  Для рабочей станции с привилегированным доступом не следует использовать Windows 10 Домашняя.
>
> Таблицу сравнения различных выпусков Windows 10 см. в [этой статье](https://www.microsoft.com/en-us/WindowsForBusiness/Compare).

Элементы управления безопасностью с привилегированным ДОСТУПОМ направлен на предотвращение высокая степень влияния и риски высокая вероятность компрометации. К ним относятся Противостояние атакам на среду и риски, которые может привести к снижению эффективности элементов управления привилегированным ДОСТУПОМ со временем:

* **Интернет-атаки.** Большинство атак поступают напрямую или косвенно из источников в Интернете. При этом Интернет используется для эксфильтрации, а также командования и управления (C2). Чтобы не допустить компрометации рабочей станции с привилегированным доступом, следует прежде всего изолировать ее от доступа к Интернету.
* **Риск использования.** Если рабочая станция с привилегированным доступом слишком сложна для выполнения повседневных задач, администраторы будут заинтересованы в создании решения, упрощающего их работу. Часто такие обходные решения представляют значительную угрозу безопасности административных рабочих станций и учетных записей. Поэтому крайне важно предоставлять пользователям рабочих станций с привилегированным доступом возможности для безопасного устранения этих проблем с безопасностью. Это можно сделать путем прослушивания пользователей, установить средства и скрипты, необходимые для выполнения их работы, а также объяснить всем администраторам известно, почему они должны использовать привилегированным ДОСТУПОМ, а какие с привилегированным ДОСТУПОМ и способы ее использования и правильно.
* **Риски среды.** Так как множество компьютеров и учетных записей в среде напрямую или косвенно подвержены интернет-рискам, следует обеспечить защиту рабочей станции с привилегированным доступом от атак со стороны скомпрометированных ресурсов в рабочей среде. Для этого требуется, сводя к минимуму использование средства управления и учетными записями, имеющими доступ к привилегированным доступом для обеспечения безопасности и мониторинга этих специализированных рабочих станций.
* **Незаконное изменение логистической цепочки.** Невозможно устранить все возможные риски незаконного изменения в логистической цепочке для оборудования и программного обеспечения. Однако всего нескольких ключевых действий будет достаточно, чтобы устранить критические векторы атаки, легко доступные для злоумышленников. Для этого нужно выполнять проверку целостности всех установочных носителей ([принцип чистоты источника](https://aka.ms/cleansource)) и использовать оборудование и программное обеспечения от надежного и проверенного поставщика.
* **Физические атаки.** Так как рабочие станции с привилегированным доступом можно перемещать и использовать вне физически защищенных помещений, нужно обеспечить их защиту от атак с использованием несанкционированного физического доступа к компьютеру.

> [!NOTE]
> Рабочая станция с привилегированным доступом не сможет защитить среду от злоумышленника, уже получившего административный доступ к лесу Active Directory.
> Так как многие из существующих реализаций доменных служб Active Directory в течение многих лет эксплуатируются в условиях риска кражи учетных данных, организациям следует признать уже имеющуюся брешь в системе безопасности, а также то, что произошла не обнаруженная компрометация учетных данных администратора домена или предприятия. Организациям, предполагающим компрометацию домена, следует использовать профессиональные службы реагирования на инциденты.
>
> Дополнительные сведения о реагировании, а также рекомендации по восстановлению см. в разделе "Реагирование на подозрительные действия" и "Восстановление после сбоя" документа [Mitigating Pass-the-Hash and Other Credential Theft, version 2](https://aka.ms/pth) (Защита от атаки с передачей хэша и других видов кражи учетных данных, версия 2).
>
> Чтобы получить дополнительные сведения, посетите страницу [служб реагирования на инциденты и восстановления корпорации Майкрософт](https://www.microsoft.com/en-us/microsoftservices/campaigns/cybersecurity-protection.aspx).

### <a name="paw-hardware-profiles"></a>Профили оборудования станции с привилегированным ДОСТУПОМ

Административный персонал являются обычными пользователями, слишком — они должны с привилегированным ДОСТУПОМ как и рабочая станция обычного пользователя для проверки электронной почты, поиск в Интернете и получения доступа к бизнес-приложений.  Эффективность работы развертывания рабочей станции с привилегированным доступом в основном зависит от обеспечения эффективности работы и безопасности администратора.  Пользователи скорее выберут решение, повышающее эффективность работы, а не безопасное решение, существенно ограничивающее ее, даже если первое из них небезопасно.

Чтобы сбалансировать потребность в безопасности с необходимостью в повышении эффективности, специалисты корпорации Майкрософт рекомендуют использовать один из профилей оборудования рабочей станции с привилегированным доступом:

* **Выделенное оборудование** -отдельных выделенных устройствах задачи пользователей и административных задач.
* **Одновременное использование.** Задачи пользователей и задачи администрирования выполняются параллельно на одном устройстве за счет использования преимуществ ОС или виртуализации представления.

Организации могут использовать только один профиль или оба. Профили оборудования взаимодействуют без проблем, поэтому организации могут подобрать профиль оборудования для конкретных потребностей и в соответствии с ситуацией конкретного администратора.

> [!NOTE]
> Крайне важно, что во всех этих сценариях администраторам выдаются учетная запись обычного пользователя, который отделен от назначенного учетную запись администратора. Учетную запись администратора следует использовать только в административной операционной системе с рабочей станцией с привилегированным доступом.

В таблице ниже перечислены относительные преимущества и недостатки каждого профиля оборудования с точки зрения простоты использования, производительности и безопасности.  Оба аппаратных подхода обеспечивают надежную защиту учетных записей администраторов от кражи и повторного использования учетных данных.

|**Сценарий**|**Преимущества**|**Недостатки**|
|--------|---------|-----------|
|Выделенное оборудование|— Точное определение уровня конфиденциальности задач<br />— Эффективное разграничение прав безопасности|— Требует дополнительного места в физическом пространстве<br />— Большой объем (для удаленной работы)<br />— Высокая стоимость оборудования|
|Одновременное использование|— Низкая стоимость оборудования<br />— Использование одного устройства|— Совместное использование одной клавиатуры и мыши создает риск совершения случайных ошибок|

Это руководство содержит подробные указания по конфигурации рабочей станции с привилегированным доступом для подхода, предусматривающего использование выделенного оборудования. При наличии требований к профилям совместно используемого оборудования можно изменить указания на основе этого руководства самостоятельно или обратиться за помощью к специалистам организации, предоставляющей профессиональные услуги, например Майкрософт.

#### <a name="dedicated-hardware"></a>Выделенное оборудование

В этом сценарии рабочая станция с привилегированным доступом используется для администрирования, полностью отделенного от ПК, используемого для выполнения ежедневных действий, например отправки почты, редактирования документов и разработки. Все средства и приложения для администрирования устанавливаются на рабочую станцию с привилегированным доступом, а все рабочие приложения — на рабочую станцию обычного пользователя. Пошаговые указания в этом руководстве предназначены для этого профиля оборудования.

#### <a name="simultaneous-use---adding-a-local-user-vm"></a>Одновременное использование: Добавление локального пользователя виртуальной Машины

В этом сценарии одновременного использования один ПК используется для выполнения задач администрирования и ежедневных действий, например отправка почты, редактирование документов и разработка. В этой конфигурации операционная система пользователя доступна в отключенном состоянии (для редактирования документов и работы с почтой в локальном кэше), но требует наличия оборудования и процессов поддержки, обеспечивающих работу в таком состоянии.

![На схеме показан сценарий одновременного использования — один ПК применяется для выполнения задач администрирования и ежедневных действий, таких как отправка почты, редактирование документов и разработка](../media/privileged-access-workstations/PAWFig10.JPG)

Физическое оборудование работает под управлением двух операционных систем на локальном уровне:

* **ОС администратора**. Физический узел выполняется под управлением Windows 10 на узле рабочей станции с привилегированным доступом для выполнения задач администрирования.
* **ОС пользователя** Гостевая виртуальная машина с Client Hyper-V под управлением Windows 10 работает в корпоративном образе.

С Windows 10 Hyper-V, на гостевой виртуальной машине (также под управлением Windows 10) может иметь в многофункциональном пользовательском интерфейсе, включая звуковые, видео и приложений Интернета communications Скайп для бизнеса.

В этой конфигурации ежедневные задания, для которых не требуются права администратора, выполняются на виртуальной машине с ОС пользователя с постоянным корпоративным образом Windows 10, на которую не распространяются ограничения, применяемые к узлу рабочей станции с привилегированным доступом. Все задачи администрирования выполняются в ОС администратора.

Чтобы создать такую конфигурацию, следуйте указаниям в этом руководстве для узла рабочей станции с привилегированным доступом, добавьте компоненты Client Hyper-V, создайте виртуальную машину пользователя и установите в ней корпоративный образ Windows 10.

Дополнительные сведения об этом компоненте см. в статье [Client Hyper-V](https://docs.microsoft.com/virtualization/hyper-v-on-windows/index). Обратите внимание, что операционная система в гостевых виртуальных машинах должна использоваться в соответствии с условиями [лицензирования продуктов Майкрософт](https://www.microsoft.com/en-us/Licensing/product-licensing/products.aspx), также указанных [здесь](https://download.microsoft.com/download/9/8/D/98D6A56C-4D79-40F4-8462-DA3ECBA2DC2C/Licensing_Windows_Desktop_OS_for_Virtual_Machines.pdf).

#### <a name="simultaneous-use---adding-remoteapp-rdp-or-a-vdi"></a>Одновременное использование: Добавление RemoteApp, RDP или VDI

В этом сценарии одновременного использования один ПК используется для выполнения задач администрирования и ежедневных действий, например отправка почты, редактирование документов и разработка. В этой конфигурации осуществляется централизованное развертывание операционных систем пользователей и управление ими (в облаке или центре обработки данных). Однако они недоступны в отключенном состоянии.

![На рисунке показан сценарий одновременного использования — один ПК применяется для выполнения задач администрирования и ежедневных действий, таких как отправка почты, редактирование документов и разработка](../media/privileged-access-workstations/PAWFig11.JPG)

Для выполнения задач администрирования физическое оборудование использует операционную систему одной рабочей станции с привилегированным доступом локально, а для выполнения задач пользователя, например отправка почты и редактирование документов, и работы с бизнес-приложениями — службу удаленного рабочего стола Майкрософт или стороннего поставщика.

В этой конфигурации ежедневные задания, для которых не требуются права администратора, выполняются в удаленных ОС и приложениях, на которые не распространяются ограничения, применяемые к узлу рабочей станции с привилегированным доступом. Все задачи администрирования выполняются в ОС администратора.

Чтобы создать такую конфигурацию, следуйте указаниям в этом руководстве для узла рабочей станции с привилегированным доступом, обеспечьте сетевое подключение к службам удаленных рабочих столов и добавьте ярлыки для доступа к приложениям на компьютере пользователя рабочей станции с привилегированным доступом. Для размещения служб удаленных рабочих столов можно использовать множество способов:

* существующая служба удаленного рабочего стола или VDI (локально или в облаке);
* новая служба, устанавливаемая локально или в облаке;
* Azure RemoteApp с использованием предварительно настроенных шаблонов Office 365 или собственных установочных образов.

Дополнительные сведения об Azure RemoteApp см. на [этой странице](https://www.remoteapp.windowsazure.com).

## <a name="how-microsoft-is-using-administrative-workstations"></a>Как корпорация Майкрософт использует административных рабочих станций

В корпорации Майкрософт архитектурный подход рабочих станций с привилегированным доступом применяется к внутренним системам и к клиентам. Корпорация Майкрософт использует рабочие станции администратора несколько производственных мощностей, включая администрирование ИТ-инфраструктуры Microsoft, Microsoft cloud fabric инфраструктуру разработки и операций и других ценных ресурсов.

В этом руководстве используются принципы эталонной архитектуры рабочей станции с привилегированным доступом, развернутой специалистами отдела по разработке профессиональных служб кибербезопасности для защиты пользователей от кибератак. Рабочие станции администраторов также являются ключевым элементом самого надежного средства защиты, предназначенного для выполнения задач администрирования домена — эталонной архитектуры административного леса ESAE.

Дополнительные сведения о административного леса ESAE см. в разделе *подход к проектированию административного леса ESAE* статьи [защита справочные материалы привилегированного доступа](../securing-privileged-access/securing-privileged-access-reference-material.md#esae-administrative-forest-design-approach).

### <a name="why-dedicated-workstations"></a>Почему выделенных рабочих станций?

Текущая среда угроз для организаций изобилует сложными фишинг-атаками и другими видами интернет-атак, подвергающих учетные записи и рабочие станции с выходом в Интернет постоянному риску нарушения безопасности.

Эта среда угроз требует организаций применения стратегии «брешь» уровень безопасности при разработке средств защиты для ценных ресурсов, например административных учетных записей и конфиденциальных бизнес-ресурсов. Эти ценные ресурсы должны быть защищены от прямых атак из Интернета и атак с других рабочих станций, серверов и устройств в среде.

![Рисунок показывает, какому риску подвергаются управляемые ресурсы, если злоумышленник получает контроль над рабочей станцией пользователя, где используются конфиденциальные учетные данные](../media/privileged-access-workstations/PAWFig2.JPG)

На этом рисунке показано, какому риску подвергаются управляемые ресурсы, если злоумышленник получает контроль над рабочей станцией пользователя, в которой используются конфиденциальные учетные данные.

Злоумышленнику, получившему контроль над операционной системой, доступно множество способов для незаконного получения доступа ко всем действиям на рабочей станции и олицетворения правомерной учетной записи. Такой уровень доступа можно получить, используя множество известных и неизвестных атак. Из-за увеличения количества и уровня сложности кибератак появилась необходимость в расширении концепции разделения для полного разделения клиентских операционных систем для конфиденциальных учетных записей. Чтобы получить дополнительные сведения об этих типах атак, посетите [веб-сайт Pass The Hash](https://www.microsoft.com/pth), где вы найдете информативные технические документы, видеоматериалы и многое другое.

Подход, при котором применяется рабочая станция с привилегированным доступом, является расширением общепризнанного рекомендованного способа — сотрудники, отвечающие за администрирование, должны использовать отдельные учетные записи администратора и пользователя. При таком подходе используются отдельно назначенные учетные записи администратора, полностью отделенные от учетных записей обычного пользователя. Подход, при котором применяется рабочая станция с привилегированным доступом, основан на этом подходе разделения учетных записей. Он заключается в предоставлении защищенной рабочей станции для конфиденциальных учетных записей.

Это руководство по рабочим станциям с привилегированным доступом поможет реализовать эту возможность для защиты ценных учетных записей, например учетные записи ИТ-администраторов с высоким уровнем привилегий и конфиденциальные корпоративные учетные записи. Руководство поможет:

* защитить учетные данные от риска за счет использования только доверенных узлов;
* предоставить администраторам рабочую станцию с высоким уровнем безопасности для выполнения задач администрирования.

Использование для конфиденциальных учетных записей только рабочих станций с привилегированным доступом и высоким уровнем защиты — простой способ защиты этих учетных записей, так как при этом обеспечивается удобство для администраторов и усложняются условия для атаки злоумышленника.

### <a name="alternate-approaches"></a>Альтернативные подходы

В этом разделе представлено сравнение уровня защиты, обеспечиваемого при применении альтернативных подходов и рабочей станции с привилегированным доступом, а также сведения о том, как правильно внедрить эти подходы в архитектуре рабочих станций с привилегированным доступом. все эти подходы связаны со значительными рисками при применении по отдельности, но также внести вклад в реализацию с привилегированным ДОСТУПОМ, в некоторых сценариях.

#### <a name="credential-guard-and-windows-hello-for-business"></a>Credential Guard и Windows Hello для бизнеса

Представленное в Windows 10, решение [Credential Guard](https://technet.microsoft.com/library/mt483740%28v=vs.85%29.aspx) использует оборудование и средства безопасности на базе технологии виртуализации для устранения рисков распространенных атак, направленных на хищение учетных данных, например атаки типа pass-the-hash. Это достигается за счет обеспечения защиты извлеченных учетных данных. Закрытый ключ для учетных данных, используемых [Windows Hello для бизнеса](https://aka.ms/passport) может быть также можно защитить, оборудования доверенного платформенного модуля (TPM).

Это мощные средства защиты, но рабочие станции могут оставаться уязвимыми к определенным атакам даже если учетные данные защищены с помощью Credential Guard или Windows Hello для бизнеса. Атаки могут включать отправку прав и использованием учетных данных непосредственно из скомпрометированных устройств, повторное использование ранее украденных учетных данных перед включением Credential Guard и нарушениях использования средства управления и конфигурации слабые приложений на рабочей станции.

В этом разделе, посвященном рекомендациям по использованию рабочих станций с привилегированным доступом, представлены сведения об использовании этих технологий для конфиденциальных учетных записей и задач.

#### <a name="administrative-vm"></a>Административная виртуальная машина

Администратора виртуальной машины (Admin VM) имеет выделенную операционную систему для выполнения административных задач, размещенных на рабочий стол обычного пользователя. Хотя этот подход аналогичен использованию рабочих станций с привилегированным доступом (предоставление выделенной ОС для выполнения административных задач), он имеет существенный недостаток: обеспечение защиты административной виртуальной машины зависит от компьютера обычного пользователя.

На схеме ниже показано, как злоумышленники могут проследовать по цепочке управления к интересующему целевому объекту в административной виртуальной машине на рабочей станции пользователя. Кроме того, на ней показано, что сложно создать путь, используя обратную конфигурацию.

Архитектура с привилегированным ДОСТУПОМ позволяет разместить в административной виртуальной Машине на рабочей станции пользователя, но виртуальную Машину пользователя со стандартным корпоративным образом можно разместить на администратора ПРИВИЛЕГИРОВАННЫМ сотрудники получают один ПК для все функциональные обязанности.

![Схема архитектуры рабочей станции с привилегированным доступом](../media/privileged-access-workstations/PAWFig9.JPG)

#### <a name="shielded-vm-based-paws"></a>Экранированный привилегированным доступом на основе виртуальной Машины

Безопасный вариант административной модели виртуальной Машины, можно выполнить [экранированных виртуальных машин](../../security/guarded-fabric-shielded-vm/guarded-fabric-and-shielded-vms.md) для размещения один или несколько виртуальных машин администратора вместе с пользователя виртуальной Машины.
Экранированные виртуальные машины предназначены для выполнения защиты рабочих нагрузок в среде, где потенциально ненадежных пользователей или кода могут выполняться на рабочем столе физического компьютера обычного пользователя.
Экранированная виртуальная машина имеет виртуальный доверенный платформенный модуль, который позволяет шифровать свои данные при хранении, а также несколько административные элементы управления, такие как простые консольные доступа, PowerShell Direct и возможность отладки виртуальной Машины отключаются для дальнейшей локализации виртуальной Машины с рабочего стола обычного пользователя и другие виртуальные машины.
Ключи для экранированной виртуальной Машины хранятся на сервере доверенных управления ключами, который требует физического устройства к подтверждающим удостоверение и работоспособности, не отпуская клавишу, чтобы запустить виртуальную Машину.
Это гарантирует, что экранированные виртуальные машины можно запустить только на нужные устройства и что эти устройства работают конфигурации известного и надежного программного обеспечения.

Так как эти виртуальные машины изолированы друг от друга и рабочий стол обычного пользователя, это допустимо для запуска нескольких экранированных виртуальных машин с привилегированным ДОСТУПОМ на одном узле, даже в том случае, когда эти виртуальные машины администратора управлять различных уровней.

См. в разделе [развертывание привилегированным доступом с помощью защищенной структуре](#deploy-paws-using-a-guarded-fabric) Дополнительные сведения в приведенном ниже разделе.

#### <a name="jump-server"></a>Сервер перехода

В административной архитектуре с серверами перехода настраивается небольшое число серверов консоли администрирования, а сотрудникам разрешается использовать их только для выполнения задач администрирования. Обычно такие серверы работают на базе служб удаленных рабочих столов, стороннего решения по виртуализации представления или технологии инфраструктуры виртуальных рабочих столов (VDI).

Этот подход часто предлагается для снижения риска при администрировании и обеспечивает определенный уровень безопасности. Однако сам по себе подход с использованием сервера перехода предполагает уязвимость к определенным видам атак, так как он нарушает [принцип чистоты источника](../securing-privileged-access/securing-privileged-access-reference-material.md#clean-source-principle). Этот принцип заключается в том, что уровни надежности всех зависимых объектов системы безопасности и защищаемого объекта должны совпадать.

![Рисунок показывает простую взаимосвязь при управлении](../media/privileged-access-workstations/PAWFig3.JPG)

На этом рисунке представлена простая взаимосвязь при управлении. Любой субъект под управлением объекта зависит от уровня безопасности этого объекта. Если злоумышленник может управлять зависимостью безопасности целевого объекта (субъект), он может управлять и объектом.

Сеанс администрирования на сервере перехода зависит от состояния целостности локального компьютера, с которого осуществляется доступ. Если этот компьютер является рабочей станцией пользователя, на которую совершается фишинг-атака и другие виды интернет-атак, сеанс администрирования также подвержен риску этих атак.

![Рисунок показывает, как злоумышленники могут проследовать по установленной цепочке управления к интересующему их целевому объекту](../media/privileged-access-workstations/PAWFig4.JPG)

На рисунке выше показано, как злоумышленники могут проследовать по установленной цепочке управления к интересующему целевому объекту.

Хотя из-за некоторых расширенных средств безопасности, например многофакторной идентификации, злоумышленнику и будет сложнее захватить контроль над сеансом администрирования на рабочей станции пользователя, никакое средство безопасности не сможет защитить от технических атак в полной мере, если у злоумышленника есть административный доступ к исходному компьютеру (например, включение несанкционированных команд в санкционированный сеанс, захват санкционированных процессов и т. д.)

В конфигурации по умолчанию в этом руководстве на рабочей станции с привилегированным доступом устанавливаются средства администрирования. Однако при необходимости можно добавить также архитектуру сервера перехода.

![Рисунок показывает, как при изменении взаимосвязи на основе управления и получении доступа к приложениям пользователя с рабочей станции администратора злоумышленник не может проложить путь к целевому объекту](../media/privileged-access-workstations/PAWFig5.JPG)

На этом рисунке показано, как при изменении взаимосвязи на основе управления и получении доступа к приложениям пользователя с рабочей станции администратора злоумышленник не может проложить путь к целевому объекту. Сервер перехода пользователя по-прежнему подвержен риску, поэтому к компьютеру с выходом в Интернет нужно применять соответствующие средства защиты, обнаружения и реагирования.

В этой конфигурации администраторы должны четко следовать указаниям по выполнению операций, чтобы случайно не ввести учетные данные администратора в сеансе пользователя на компьютере.

![Рисунок показывает, как при получении доступа к административному серверу перехода с рабочей станции с привилегированным доступом злоумышленник не может получить доступ к ресурсам администрирования](../media/privileged-access-workstations/PAWFig6.JPG)

На этом рисунке показано, как при получении доступа к административному серверу перехода с рабочей станции с привилегированным доступом злоумышленник не может получить доступ к ресурсам администрирования. В этом случае сервер перехода с рабочей станции с привилегированным доступом позволяет объединить несколько расположений для наблюдения за действиями администраторов и распределения административных приложений и средств. Таким образом инфраструктура слегка усложняется. Однако мониторинг безопасности и обновление программного обеспечения упрощается, если в реализации рабочей станции с привилегированным доступом используется большое количество учетных записей и рабочих станций. При создании и настройке сервера перехода необходимо применять стандарты безопасности, сходные со стандартами безопасности в конфигурации с рабочей станцией с привилегированным доступом.

#### <a name="privilege-management-solutions"></a>Решения по управлению привилегиями

Решения по управлению привилегиями — это приложения, обеспечивающие временный доступ к скрытым привилегиям или привилегированным учетным записям по требованию. Эти решения являются крайне ценным компонентом в полной стратегии обеспечения защиты привилегированного доступа и критически важной визуализации и учета действий администраторов.

Обычно они используют гибкий рабочий процесс для управления доступом. Во многих из них предусмотрены дополнительные функции и возможности для обеспечения безопасности, например управление паролями учетных записей служб и интеграция с административными серверами перехода. На рынке представлено множество решений с возможностями для управления привилегиями. Одно из них — PAM Microsoft Identity Manager (MIM).

Для доступа к таким решениям специалисты корпорации Майкрософт рекомендуют использовать рабочие станции с привилегированным доступом. Доступ к ним должен предоставляться только для рабочих станций с привилегированным доступом. Однако специалисты корпорации Майкрософт не рекомендуют использовать эти решения в качестве замены рабочим станциям с привилегированным доступом, так как при получении доступа к привилегиям на потенциально скомпрометированном компьютере пользователя с помощью этих решений нарушается [принцип чистоты источника](https://aka.ms/cleansource), как показано на следующей схеме:

## <a name="paw-phased-implementation"></a>Реализация Phased станции с привилегированным ДОСТУПОМ

Так как рабочая станция с привилегированным доступом должна предоставлять надежный и безопасный источник для администрирования, при ее создании также следует обеспечить безопасность и надежность.  Этот раздел содержит подробные указания по созданию рабочей станции с привилегированным доступом в соответствии с общими принципами и подходами, подобными тем, которые используются в ИТ-отделах и отделах проектирования облаков корпорации Майкрософт, а также в организациях, предоставляющих услуги управления службами.

Указания разбиты на три этапа. В них основное внимание уделяется быстрому устранению наиболее серьезных уязвимостей и постепенному повышению использования рабочих станций с привилегированным доступом на предприятии.

* [Этап 1 - немедленное развертывание для администраторов Active Directory](#phase-1-immediate-deployment-for-active-directory-administrators)
* [Этап 2 - станций с привилегированным ДОСТУПОМ всем администраторам](#phase-2-extend-paw-to-all-administrators)
* [Этап 3 — расширенные станции с привилегированным ДОСТУПОМ безопасности](#phase-3-extend-and-enhance-protection)

Важно отметить, что этапы всегда должно выполняться по порядку, даже если они планируются и реализуются в рамках одного проекта.

### <a name="phase-1-immediate-deployment-for-active-directory-administrators"></a>Этап 1. Ускоренное развертывание для администраторов Active Directory

Назначение: Предоставляет станции с привилегированным ДОСТУПОМ, быстро, можно защитить роли администрирования локального домена и леса.

Область: Администраторы уровня 0 "Администраторы предприятия", "Администраторы домена" (для всех доменов) и Администраторы других полномочных систем идентификации.

Этап 1 затрагивает администраторов, управляющих локальным доменом Active Directory (критически важные роли, подверженные частым атакам злоумышленников). Эти системы идентификации эффективно обеспечивают защиту этих администраторов независимо от размещения контроллеров домена Active Directory (локальный центр обработки данных, в решении IaaS Azure или другого поставщика решений IaaS).

На этом этапе создается безопасная административная структура подразделений Active Directory для размещения и развертывания рабочей станции с привилегированным доступом.  Эта структура также включает групповые политики и группы, необходимые для поддержки рабочей станции с привилегированным доступом.  Большая часть структуры создается с помощью скриптов PowerShell, доступных в [коллекции TechNet](https://aka.ms/pawmedia).

Скрипты помогут создать следующие подразделения и группы безопасности:

* Подразделения
   * Шесть новых подразделений верхнего уровня:  Admin; Группы; Серверы уровня 1; Рабочих станций; Учетные записи пользователей; и карантин компьютера.  Каждое Подразделение верхнего уровня будет содержать несколько дочерних подразделений.
* Группы
   * Шесть новых глобальных группы безопасности:  Обслуживание репликации уровня 0; Обслуживание сервера уровня 1; Операторы службы поддержки; Рабочая станция обслуживания; Пользователи с привилегированным ДОСТУПОМ. Обслуживание с привилегированным ДОСТУПОМ.

Кроме того, вы создадите несколько объектов групповой политики: Настройка станции с привилегированным ДОСТУПОМ — компьютер; ПРИВИЛЕГИРОВАННЫМ конфигурации - пользователя. Требуется RestrictedAdmin — компьютер; Ограничения для исходящих подключений с привилегированным ДОСТУПОМ; Разрешить вход в систему рабочей станции; Разрешить вход в систему сервера.

Этап 1 включает следующие операции:

#### <a name="complete-the-prerequisites"></a>Выполните предварительные требования.

1. **Убедитесь, что все администраторы используют отдельные учетные записи для администрирования и выполнения пользовательских действий** (включая отправку почты, работу в Интернете, работу с бизнес-приложениями и другие стандартные действия).  Назначение учетной записи администратора для каждого авторизованного сотрудника, отдельной от учетной записи обычного пользователя — основа модели рабочей станции с привилегированным доступом, так как для входа в рабочую станцию с привилегированным доступом разрешено использовать только определенные учетные записи.

   > [!NOTE]
   > Каждый администратор должен собственную учетную запись для администрирования.  Не используйте учетную запись администратора совместно.

2. **Максимально сократите число привилегированных администраторов уровня 0**.  Так как каждому администратору нужно использовать рабочую станцию с привилегированным доступом, нужно сократить число администраторов для сокращения числа рабочих станций с привилегированным доступом. Связанные с этим расходы также снижаются. Кроме того, чем меньше число администраторов, тем ниже риск атаки этих привилегий, а также связанные с этим риски. Администраторы, находящиеся в одном месте, могут совместно использовать одну рабочую станцию с привилегированным доступом. Однако администраторам в разных физических местоположениях потребуются отдельные рабочие станции с привилегированным доступом.
3. **Приобретайте соответствующее всем техническим требованиям оборудование у надежного поставщика**. Корпорация Майкрософт рекомендует приобретать оборудование, соответствующее техническим требованиям, в статье [Protect domain credentials with Credential Guard](https://technet.microsoft.com/library/mt483740%28v=vs.85%29.aspx) (Защита учетных данных домена с помощью Credential Guard).

   > [!NOTE]
   > Рабочая станция с привилегированным доступом, установленная на оборудовании без этих возможностей обеспечивает высокий уровень защиты. Однако в ней будут недоступны расширенные компоненты безопасности, например Credential Guard и Device Guard.  Credential Guard и Device Guard не являются обязательными для развертывания на этапе 1. Однако их настоятельно рекомендуется использовать при выполнении этапа 3 (расширенное усиление защиты).
   >
   > Убедитесь, что оборудование, используемое для рабочей станции с привилегированным доступом, получено от производителя и поставщика, методам по обеспечению безопасности которых доверяет организация. Так принцип чистоты источника применяется для обеспечения безопасности логистической цепочки.
   >
   > Дополнительные сведения о том, почему важно обеспечивать безопасность логистической цепочки, см. на [этом сайте](https://www.microsoft.com/security/cybersecurity/).

4. **Получите и проверьте, требуется Windows 10 Enterprise Edition и прикладное программное обеспечение**. Получите программное обеспечение, необходимое для рабочей станции с привилегированным доступом, и проверьте его, используя рекомендации в разделе о [принципе чистоты источника для установочных носителей](https://aka.ms/cleansource).

   * Выпуск Windows 10 Корпоративная
   * [Средства удаленного администрирования сервера](https://www.microsoft.com/en-us/download/details.aspx?id=45520) для Windows 10
   * [Базовые показатели системы безопасности Windows 10](https://aka.ms/win10baselines)

      > [!NOTE]
      > Корпорация Майкрософт публикует хэш MD5 для всех операционных систем и приложений на сайте MSDN. Однако не все производители программного обеспечения предоставляют аналогичную документацию.  В таких случаях вам потребуются другие стратегии.  Дополнительные сведения о проверке программного обеспечения см. в разделе [о принципе чистоты источника](https://aka.ms/cleansource) для установочных носителей.

5. **Убедитесь, что в интрасети доступен сервер WSUS**. Он понадобиться для скачивания и установки обновлений для рабочей станции с привилегированным доступом. Для этого сервера WSUS должно быть настроено автоматическое утверждение всех обновлений системы безопасности для Windows 10 или администратор должен отвечать за быстрое утверждение обновлений программного обеспечения.

   > [!NOTE]
   > Дополнительные сведения см. в разделе "Автоматическое утверждение обновлений для установки" руководства [Approving Updates](https://technet.microsoft.com/library/cc708458(v=ws.10).aspx) (Утверждение обновлений).

#### <a name="deploy-the-admin-ou-framework-to-host-the-paws"></a>Разверните инфраструктуру подразделений администрирования для размещения рабочих станций с привилегированным доступом.

1. Скачайте библиотеку скриптов рабочих станций с привилегированным доступом из [коллекции TechNet](https://aka.ms/PAWmedia).

   > [!NOTE]
   > Скачать все файлы и сохранять их на том же каталоге и выполнять их в указанном ниже порядке.  Скрипт Create-PAWGroups зависит от структуры подразделений, созданных скриптом Create-PAWOUs, а скрипт Set-PAWOUDelegation зависит от групп, созданных скриптом Create-PAWGroups.
   > Не изменяйте скрипты или файл данных с разделителями-запятыми (CSV).

2. **Запустите скрипт Create-PAWOUs.ps1**.  Этот скрипт создаст структуру подразделений в Active Directory и заблокирует наследование объектов групповой политики для новых подразделений соответствующим образом.
3. **Запустите скрипт Create-PAWGroups.ps1**.  Этот скрипт создаст глобальные группы безопасности в соответствующих подразделениях.

   > [!NOTE]
   > Хотя он и создаст группы безопасности, он не заполнит их автоматически.

4. **Запустите скрипт Set-PAWOUDelegation.ps1**.  Этот скрипт назначит разрешения на новые подразделения соответствующим группам.

#### <a name="move-tier-0-accounts-to-the-admintier-0accounts-ou"></a>Переместите учетные записи уровня 0 в подразделение администратор\уровень 0\учетные записи

Переместите каждую учетную запись, принадлежащую группе "Администратор домена", "Администратор предприятия" или эквивалентные группы уровня 0 (включая вложенные элементы), в это подразделение. Если в организации есть группы, добавленные в эти группы, следует переместить их в подразделение "Администратор\Уровень 0\Группы".

   > [!NOTE]
   > Дополнительные сведения о том, какие группы принадлежат к уровню 0, см. в разделе "Эквивалентность уровня 0" статьи [Справочные материалы по обеспечению безопасности привилегированного доступа](../securing-privileged-access/securing-privileged-access-reference-material.md).

#### <a name="add-the-appropriate-members-to-the-relevant-groups"></a>Добавьте соответствующих участников в соответствующие группы.

1. **Пользователи рабочих станций с привилегированным доступом.** Добавьте администраторов уровня 0 с группами "Администратор домена" или "Администратор предприятия", определенные на шаге 1 этапа 1.
2. **Обслуживание рабочих станций с привилегированным доступом.** Добавьте по крайней мере одну учетную запись, которая будет использоваться для обслуживания рабочих станций с привилегированным доступом и устранения неполадок в них. Учетные записи для обслуживания рабочих станций с привилегированным доступом будут использоваться редко.

   > [!NOTE]
   > Не добавляйте одну и ту же учетную запись или группу пользователя в группы "Пользователи рабочих станций с привилегированным доступом" и "Обслуживание рабочих станций с привилегированным доступом".  Модель защиты рабочей станции с привилегированным доступом частично основана на предположении, что учетная запись пользователя рабочей станции с привилегированным доступом обладает правами на управляемые системы или рабочую станцию с привилегированным доступом, но не оба компонента.
   >
   > * Это важно для формирования эффективных методов администрирования на этапе 1.
   > * Это крайне важно для этапа 2 и следующих этапов во избежание повышения привилегий с помощью рабочей станции с привилегированным доступом, так как рабочие станции с привилегированным доступом используются на разных уровнях.
   >
   > В идеальном случае никому не назначаются обязанности, требующие работы на нескольких уровнях, в соответствии с принципом разделения обязанностей. Однако в корпорации Майкрософт понимают, что во многих организациях штат ограничен (или присутствуют другие организационные требования). Из-за этого невозможно полностью разделить обязанности. В этих случаях одному сотруднику могут назначить обе роли. Однако ему не следует использовать одну учетную запись для выполнения этих функций.

#### <a name="create-paw-configuration---computer-group-policy-object-gpo"></a>Создание «Привилегированным ДОСТУПОМ — компьютер конфигурации» объект групповой политики (GPO)

В этом разделе вы создадите новый «ПРИВИЛЕГИРОВАННЫМ конфигурации — компьютер» объект групповой Политики, которые обеспечивают защиту атак определенного типа для этих привилегированным доступом и связать его с подразделением устройства уровня 0 («устройства» Администратор\уровень 0\пользователи).

   > [!NOTE]
   > **Не добавляйте эти параметры в политику домена по умолчанию**.  Это может повлиять на работу всей среды Active Directory.  Настройте эти параметры только в созданных объектах групповой политики, описанных здесь, и примените их только к подразделению рабочей станции с привилегированным доступом.

1. **Доступ к группе "Обслуживание рабочих станций с привилегированным доступом"** . Этот параметр позволит назначить конкретных пользователей определенным привилегированным группам в рабочих станциях с привилегированным доступом. Выберите *Конфигурация компьютера\Параметры\Параметры панели управления\Локальные пользователи и группы* и сделайте следующее:
   1. Щелкните **Создать** и **Локальная группа**.
   2. Выберите действие **Обновить** и выберите "Администраторы (встроенная учетная запись)" (не используйте кнопку "Обзор" для выбора группы доменов "Администраторы").
   3. Установите флажки **Удалить всех пользователей-членов этой группы** и **Удалить все группы-члены этой группы**.
   4. Добавьте группы "Обслуживание рабочей станции с привилегированным доступом" (pawmaint) и "Администратор" (опять же, не используйте кнопку "Обзор" для выбора администратора).

      > [!NOTE]
      > Не добавляйте группу "Пользователи рабочей станции с привилегированным доступом" в список участников локальной группы администраторов.  Чтобы пользователи рабочей станции с привилегированным доступом не могли случайно или намеренно изменить параметры безопасности рабочей станции с привилегированным доступом, их не следует добавлять в локальную группу администраторов.
      >
      > Дополнительные сведения об использовании параметров групповых политик для изменения участников групп см. в статье TechNet [Настройка элемента локальной группы](https://technet.microsoft.com/library/cc732525.aspx).

2. **Ограничить участие в локальной группе**. Этот параметр позволит обеспечить отсутствие участников в локальных группах администраторов.
   1. Выберите "Конфигурация компьютера\Параметры\Параметры панели управления\Локальные пользователи и группы" и сделайте следующее:
      1. Щелкните **Создать** и **Локальная группа**.
      2. Выберите действие **Обновить** и выберите "Операторы архива (встроенная учетная запись)" (не используйте кнопку "Обзор" для выбора группы доменов "Операторы архива").
      3. Установите флажки **Удалить всех пользователей-членов этой группы** и **Удалить все группы-члены этой группы**.
      4. Не добавляйте участников в группу.  Просто нажмите кнопку **ОК**.  Назначив пустой список, групповая политика автоматически удалит всех участников и обеспечит отсутствие участников при каждом обновлении.
   2. Выполните приведенные выше действия для следующих дополнительных групп:
      * Криптографические операторы
      * Администраторы Hyper-V
      * Операторы настройки сети
      * Опытные пользователи
      * Пользователи удаленного рабочего стола
      * Репликаторами
   3. **Ограничения входа с привилегированным ДОСТУПОМ** -этот параметр ограничивает число учетных записей, которые могут запускаться на станции с привилегированным ДОСТУПОМ. Для настройки этого параметра выполните следующее:
      1. Выберите "Конфигурация компьютера\Политики\Параметры Windows\Параметры безопасности\Локальные политики\Параметры безопасности\Локальные политики\Назначение прав пользователя\Локальный вход в систему".
      2. Выберите определить следующие параметры политики и добавьте «Пользователи привилегированным ДОСТУПОМ» и администраторов (опять же, не используйте кнопку обзора для выбора администраторов).
   4. **Блокировать входящий сетевой трафик**. Этот параметр позволяет предотвратить передачу нежелательного входящего сетевого трафика на рабочую станцию с привилегированным доступом. Для настройки этого параметра выполните следующее:
      1. Перейдите в раздел "Конфигурация компьютера\Политики\Параметры Windows\Параметры безопасности\Брандмауэр Windows в режиме повышенной безопасности\Брандмауэр Windows" и сделайте следующее:
         1. Щелкните элемент "Брандмауэр Windows в режиме повышенной безопасности" и выберите **Импорт политики**.
         2. Щелкните **Да**, чтобы принять действие. При этом будут перезаписаны все существующие политики брандмауэра.
         3. Перейдите к файлу PAWFirewall.wfw и выберите **Открыть**.
         4. Нажмите кнопку **ОК**.

            > [!NOTE]

#### <a name="deploy-your-paws"></a>Разверните свои рабочие станции с привилегированным доступом.

   > [!NOTE]
   > Для автоматизации развертывания рабочей станции с привилегированным доступом можно использовать Microsoft Deployment Toolkit (MDT) или другую систему автоматизированного развертывания образов, но при создании уровень надежности должен совпадать с уровнем надежности рабочей станции с привилегированным доступом. Злоумышленники ищут именно корпоративные образы и системы развертывания (включая образы ISO, пакеты развертывания и т. д.) в качестве механизмов сохранения. Поэтому не следует использовать существующие системы развертывания или образы.
   >
   > При автоматизации развертывания рабочей станции с привилегированным доступом сделайте следующее:
   >
   > * Создайте систему с использованием установочных носителей, проверенных на соответствие указаниям в разделе о [принципе чистоты источника для установочных носителей](https://aka.ms/cleansource).

    2. Убедитесь, что в привилегированных группах на рабочей станции с привилегированным доступом нет дополнительных учетных записей пользователей, выполнив следующее:
       1. Откройте элемент **Изменение локальных пользователей и групп** (lusrmgr.msc), выберите **Группы** и убедитесь, что в локальной группе администраторов содержатся только локальная учетная запись администратора и глобальная группа безопасности "Обслуживание рабочей станции с привилегированным доступом".

          > [!NOTE]
          > Группа "Пользователи рабочих станций с привилегированным доступом" не должна входить в локальную группу администраторов.  В группу должны входить только локальная учетная запись администратора и глобальная группа безопасности "Обслуживание рабочей станции с привилегированным доступом" (эта глобальная группа не должна содержать группу "Пользователи рабочей станции с привилегированным доступом").

       2. Кроме того, с помощью элемента **Изменение локальных пользователей и групп** убедитесь, что следующие группы пусты: Пользователи удаленного рабочего стола пользователей репликаторами для конфигурации операторы Power сетевые администраторы Hyper-V криптографические операторы операторы резервного копирования

12. (Необязательно) Если ваша организация использует сведения о безопасности и решения по управлению (SIEM) событие, убедитесь, что рабочая СТАНЦИЯ [настроено перенаправление событий в систему с помощью пересылки событий Windows (WEF)](http://blogs.technet.com/b/jepayne/archive/2015/11/24/monitoring-what-matters-windows-event-forwarding-for-everyone-even-if-you-already-have-a-siem.aspx) или она зарегистрирована решение таким образом, чтобы активную передачу событий и сведений о станции с привилегированным ДОСТУПОМ.  Сведения об этой операции зависят от решения SIEM.

    > [!NOTE]
    > Если решению SIEM требуется агент, выполняющийся в качестве системной или локальной учетной записи администрирования на рабочей станции с привилегированным доступом, убедитесь, что уровень доверия при управлении для решений SIEM, контроллеров домена и систем удостоверений совпадает.

13. (Необязательно) Если вы решили выполнить развертывание LAPS для управления паролями для учетной записи локального администратора на ваш станции с привилегированным ДОСТУПОМ, убедитесь, что пароль успешно зарегистрирован.

    * Используя учетную запись с разрешениями на чтение паролей, управляемых с помощью LAPS, откройте оснастку **Пользователи и компьютеры Active Directory** (dsa.msc).  Убедитесь, что включен параметр "Дополнительные компоненты", а затем щелкните соответствующий объект-компьютер правой кнопкой мыши.  Выберите вкладку "Редактор атрибутов" и убедитесь, что в качестве значения параметра msSVSadmPwd указан действующий пароль.

### <a name="phase-2-extend-paw-to-all-administrators"></a>Этап 2. Рабочих станций с привилегированным ДОСТУПОМ всем администраторам

Область: Все пользователи с правами администратора на критически важных приложений и зависимостей.  Этот этап затрагивает по крайней мере администраторов серверов приложений, решения для обеспечения работоспособности и мониторинга безопасности, решения для виртуализации, системы хранения и сетевые устройства.

> [!NOTE]
> В указаниях на этом этапе предполагается, что этап 1 завершен полностью.  Не начинается этап 2, пока не будут выполнены все действия на этапе 1.

Убедившись, что все действия выполнены, выполните следующие действия, чтобы завершить этап 2:

#### <a name="recommended-enable-restrictedadmin-mode"></a>(Рекомендуется) Включить **RestrictedAdmin** режим

Включить эту функцию на существующих серверах и рабочих станциях, а затем задать принудительное использование этой функции. Эта функция требует целевые серверы, чтобы работать под управлением Windows Server 2008 R2 или более поздней версии и рабочие станции — работать под управлением Windows 7 или более поздней.

1. Включите **RestrictedAdmin** на серверах и рабочих станциях, следуя указаниям на этой [странице](https://aka.ms/rdpra).

   > [!NOTE]
   > Прежде чем включать эту функцию для серверов, подключенных к Интернету, следует учитывать, что злоумышленники могут пройти проверку подлинности на этих серверах с помощью хэша украденного пароля.

2. Создайте объект групповой политики "Требуется RestrictedAdmin — компьютер". В этом разделе создается объект групповой политики, который обеспечивает принудительное использование параметра /RestrictedAdmin для исходящих подключений удаленного рабочего стола и защиту учетных данных в целевой системе от кражи.
   * Выберите "Конфигурация компьютера\Политики\Административные шаблоны\Система\Передача учетных данных\Ограничить делегирование учетных данных удаленным серверам" и установите параметр **Включено**.
3. Свяжите объект групповой политики **Требуется RestrictedAdmin — компьютер** с соответствующими устройствами уровня 1 и/или 2, используя следующие параметры политики:
   * Конфигурация рабочей станции с привилегированным доступом — компьютер
      * -> Расположение связи: Администратор\Уровень 0\Устройства (существующая)
   * Конфигурация рабочей станции с привилегированным доступом — пользователь
      * -> Расположение связи: Администратор\уровень 0\учетные записи
   * Требуется RestrictedAdmin — компьютер
      * -> администратор\уровень 1\устройства или "->" Admin\Tier2\Devices (оба являются необязательными)

   > [!NOTE]
   > Это необязательно для систем уровня 0, так как они полностью управляют всеми ресурсами в среде.

#### <a name="move-tier-1-objects-to-the-appropriate-ous"></a>Переместите объекты уровня 1 в соответствующие подразделения

1. Переместите группы уровня 1 в подразделение "Администратор\Уровень 1\Группы". Найдите все группы, которые предоставляют следующие административные права, и переместите их в это подразделение:
   * права локального администратора на нескольких серверах;
      * административный доступ к облачным службам;
      * административный доступ к корпоративным приложениям.
2. Переместите учетные записи уровня 1 в подразделение "Администратор\Уровень 1\Учетные записи". Переместите в это подразделение все учетные записи групп уровня 1 (включая вложенные элементы).
3. Добавьте соответствующих участников в соответствующие группы.
   * **Администраторы уровня 1**. Эта группа будет содержать администраторов уровня 1 без разрешений на вход в узлы уровня 2. Добавьте все ваши административные группы уровня 1, имеющие административные привилегии на серверы или Интернет-служб.

      > [!NOTE]
      > Если администраторы управляют ресурсами на нескольких уровнях, необходимо создать отдельную учетную запись администратора для каждого уровня.

4. Включите Credential Guard, чтобы уменьшить риск кражи и повторного использования учетных данных.  Credential Guard — новый компонент Windows 10, ограничивающий доступ приложений к учетным данным, предотвращая таким образом атаки, связанные с кражей учетных данных (включая атаки типа pass-the-hash).  Это решение абсолютно прозрачно для пользователя и требует минимум времени и усилий для настройки.  Дополнительные сведения о Credential Guard, включая указания по развертыванию и требования к оборудованию, см. в статье [Protect domain credentials with Credential Guard](https://technet.microsoft.com/library/mt483740%28v=vs.85%29.aspx) (Защита учетных данных домена с помощью Credential Guard).

   > [!NOTE]
   > Для настройки и использования решения Credential Guard нужно включить Device Guard.  Однако для использования этого компонента не нужно настраивать другие средства защиты Device Guard.

5. (Необязательно) Разрешить подключение к облачным службам. Этот шаг позволяет настроить управление облачными службами, например Azure и Office 365, с соответствующим уровнем безопасности. Кроме того, этот шаг необходимо выполнить для обеспечения управления рабочими станциями с привилегированным доступом с помощью Microsoft Intune.

   > [!NOTE]
   > Этот шаг можно пропустить, если для администрирования облачных служб или управления с помощью Intune не требуется подключение к облаку.

   * При выполнении этих шагов обмен данными через Интернет будет осуществляться только с авторизованными облачными службами (но не с открытыми ресурсами в Интернете). Кроме того, будут добавлены средства защиты для браузеров и других приложений, которые будут обрабатывать содержимое из Интернета. Эти рабочие станции с привилегированным доступом для администрирования никогда нельзя использовать для выполнения задач обычного пользователя, например обмен данными в Интернете и рабочие задания.
   * Для обеспечения подключения к службам рабочей станции сделайте следующее:

   1. Настройте для рабочей станции с привилегированным доступом получение данных только из авторизованного назначения в Интернете.  При расширении развертывания рабочей станции с привилегированным доступом для обеспечения администрирования облака необходимо разрешить доступ к авторизованным службам и отфильтровать доступ из открытых ресурсов в Интернете, где риск атаки на администраторов более высокий.

      1. Создание **Администраторы облачных служб** группе и добавьте все учетные записи в него, которым требуется доступ к облачным службам в Интернете.
      2. Скачайте файл *proxy.pac* рабочей станции с привилегированным доступом из [коллекции TechNet](https://aka.ms/pawmedia) и опубликуйте его на внутреннем веб-сайте.

Не рекомендуется использовать его без PAC-файла, так как в этом случае доступ для рабочих станций с привилегированным доступом будет ограничен только при подключении к корпоративной сети.

      3. Настроив файл *proxy.pac*, обновите объект групповой политики "Конфигурация рабочей станции с привилегированным доступом — пользователь".

См. в разделе «Примечание: Нерекомендуемые сценарии file://-based прокси-сервера» раздела этого [основные сведения о конфигурации веб-прокси](http://blogs.msdn.com/b/ieinternals/archive/2013/10/11/web-proxy-configuration-and-ie11-changes.aspx) блог Дополнительные сведения о настройке требуемого значения регистра.

            8. На вкладке **Общие** выберите **Item level targeting** (Нацеливание на уровне элементов) и щелкните **Нацеливание**.
            9. Щелкните **Создать элемент** и выберите **группу безопасности**.
            10. Выберите кнопку "..." и найдите группу **Администраторы облачных служб**.
            11. Щелкните **Создать элемент** и выберите **группу безопасности**.

            14. Выберите **Не совпадает**.
            15. Нажмите кнопку **ОК** в окне определения цели.
            16. Нажмите кнопку **ОК**, чтобы завершить настройку групповой политики **AutoConfigUrl**.

   Примеры дополнительного программного обеспечения и приложений:

      * Microsoft Azure PowerShell;
      * Office 365 PowerShell (также называется модуль Microsoft Online Services);
      * программное обеспечение по управлению приложениями или службами на базе консоли управления (MMC);
      * защищаемое программное обеспечение по управлению приложениями и службами (отличное от MMC).

         > [!NOTE]
         > Теперь во многих приложениях управление осуществляется исключительно через веб-браузеры, включая множество облачных служб.  Таким образом сокращается число приложений, которые требуется установить на рабочей станции с привилегированным доступом. Однако при этом появляются риск возникновения проблем с взаимодействием браузера.  Для администрирования определенных служб может потребоваться развернуть сторонний браузер в конкретных экземплярах рабочей станции с привилегированным доступом.  При развертывании дополнительных веб-браузеров выполните все принципы чистоты источника и обеспечьте их безопасность в соответствии с рекомендациями по обеспечению безопасности поставщика.

8. (Необязательно) Скачайте и установите все требуемые агенты управления.

   > [!NOTE]
   > При установке дополнительных агентов управления (управление мониторингом, безопасностью, конфигурациями и т. д.) важно обеспечить соответствие уровней доверия систем управления, контроллеров домена и систем удостоверений. Дополнительные рекомендации см. в разделе "Обновление рабочих станций с привилегированным доступом и управление ими".

9. Оцените инфраструктуру, чтобы определить системы, требующие дополнительных средств защиты, предоставляемых рабочей станцией с привилегированным доступом.  Вам нужно точно знать, какие системы требуют защиты.  Ответьте на важные вопросы о ресурсах, например:

   * Где находятся целевые системы, которыми нужно управлять?  Они находятся в одном месте или подключены к одной четко заданной подсети?
   * Сколько у вас есть систем?
   * Зависят ли эти системы от других систем (виртуализация, хранилище и т. д.). Если да, то как осуществляется управление этими системами?  Насколько сильно зависят эти важные системы от других систем? Каковы дополнительные риски, связанные с такой зависимостью?
   * Насколько важны управляемые службы? Каков ожидаемый ущерб в случае компрометации этих служб?

      > [!NOTE]
      > Оцените облачные службы. Злоумышленники в первую очередь атакуют незащищенные облачные развертывания. Поэтому крайне важно, чтобы при администрировании этих служб поддерживался такой же уровень безопасности, как и при администрировании локальных критически важных приложений.

        В ходе этой оценки определите конкретные системы, требующие дополнительной защиты, а затем расширьте программу рабочих станций с привилегированным доступом на администраторов этих систем.  Стандартные примеры систем, получающих преимущества при администрировании с использованием рабочих станций с привилегированным доступом — SQL Server (в локальной среде и в SQL Azure), приложения для отдела кадров и программное обеспечение для финансовых операций.

        > [!NOTE]
        > Если управление ресурсом осуществляется в системе Windows, им можно управлять с помощью рабочей станции с привилегированным доступом, даже если приложение выполняется в операционной системе, отличной от Windows, или на базе сторонней облачной платформы.  Например, для управления учетной записью в подписке Amazon Web Services владельцу следует использовать только рабочую станцию с привилегированным доступом.

10. Создайте метод формирования запросов и распространения для развертывания рабочих станций с привилегированным доступом в масштабе организации.  В зависимости от количества рабочих станций с привилегированным доступом, развертываемых на этапе 2, возможно, потребуется автоматизировать этот процесс.

    * Установите процесс формирования запросов и утверждения для администраторов при получении рабочей станции с привилегированным доступом.  Этот процесс может помочь стандартизировать развертывание, обеспечить ведение учета устройств рабочих станций с привилегированным доступом и выявить недостающие элементы в развертывании рабочей станции с привилегированным доступом.
    * Как сказано выше, это решение развертывания необходимо отделить от существующих методов автоматизации (безопасность которых могла быть нарушена). Оно должно соответствовать принципам, описанным на этапе 1.

        > [!NOTE]
        > Управление любой системой, управляющей ресурсами, должно осуществляться на том же или более высоком уровне доверия.

11. Просмотрите и при необходимости разверните дополнительные профили оборудования рабочей станции с привилегированным доступом.  Профиль оборудования, выбранный для развертывания на этапе 1, может быть неприменим для всех администраторов.  Просмотрите профили оборудования и при необходимости выберите дополнительные профили оборудования рабочей станции с привилегированным доступом в соответствии с требованиями администраторов.  Например, профиль выделенного оборудования (рабочая станция с привилегированным доступом и рабочие станции для ежедневных задач используются отдельно) может не подходить для путешествующего администратора. В этом случае для администратора нужно развернуть профиль одновременного использования (рабочая станция с привилегированным доступом с виртуальной машиной пользователя).
12. Учтите требования к языку, эксплуатации, обмену данными и обучению, сопровождающие расширенное развертывание рабочей станции с привилегированным доступом.   При таком существенном изменении модели администрирования конечно понадобится в некоторой степени изменить способ управления. Очень важно внедрить эти изменения в проект развертывания.  Необходимо учесть по крайней мере следующее:

    * Как сообщить руководству более высокого уровня об изменениях, чтобы заручиться их поддержкой?  Любой проект без поддержки руководства, скорее всего, потерпит неудачу. По крайней мере при его реализации возникнут трудности с финансированием и принятием широким кругом людей.
    * Как будет задокументирован новый процесс для администраторов?  Эти изменения нужно задокументировать и передать не только существующим администраторам (которым потребуется изменить свои привычки и способ управления ресурсами), но и новым администраторам (получившие должность штатного или внештатного сотрудника).  Очень важно, чтобы в документации было представлено четкие и полные сведения о важности угроз, роли рабочей станции с привилегированным доступом в обеспечении защиты администраторов и ее правильном использовании.

      > [!NOTE]
      > Это особенно важно для ролей с высокой текучестью кадров, включая, помимо прочего, сотрудников службы технической поддержки.

    * Как будет обеспечиваться соответствие в новом процессе?  Хотя модель с привилегированным ДОСТУПОМ включает в себя несколько технических элементов, предотвращающих раскрытие привилегированных учетных данных, смогут обеспечить абсолютную защиту.  Например, несмотря на то, что невозможно предотвратить вход администратора в компьютер пользователя с использованием привилегированных учетных данных, даже попытка такого входа подвергает учетные данные риску воздействия вредоносных программ, установленных на этом компьютере пользователя.  Поэтому важно объяснить не только преимущества модели рабочей станции с привилегированным доступом, но и риск при несоответствии требованиям.  Для быстрого обнаружения и устранения проблем, связанных с раскрытием учетных данных, следует внедрить систему аудита и оповещений.

### <a name="phase-3-extend-and-enhance-protection"></a>Этап 3. Расширение и улучшение защиты

Область: Эти средства защиты улучшения системы, созданные на этапе 1, успехов основная защита с помощью дополнительных функций, включая многофакторную проверку подлинности и правила доступа к сети.

> [!NOTE]
> Этот этап можно выполнить в любое время после завершения этапа 1.  Он не зависит от завершения этапа 2. Таким образом, его можно выполнено до завершения этапа 2, в ходе или после его выполнения.

Для настройки выполните следующее:

1. **Включите многофакторную идентификацию для привилегированных учетных записей**.  Многофакторная идентификация повышает безопасность учетной записи пользователя, так как пользователь должен указывать учетные данные и физический маркер.  Многофакторная идентификация эффективно дополняет политики идентификации. При развертывании она не зависит от этих политик (это утверждение верно и в обратном порядке).  Корпорация Майкрософт рекомендует использовать одну из этих форм многофакторной идентификации:

   * **Смарт-карты**: Смарт-карты — это защищенное от несанкционированного доступа и портативных физическое устройство, который предоставляет второй проверки подлинности при входе в систему Windows.  Требование карты для входа может снизить риск повторного удаленного использования украденных учетных данных.  Дополнительные сведения о входе в Windows с использованием смарт-карты см. в статье [Общие сведения о смарт-картах](https://technet.microsoft.com/library/hh831433.aspx).
   * **Виртуальная смарт-карта**:  Виртуальные смарт-карт предоставляет тот же уровень безопасности, как физических смарт-карт с возможностью их, связанным с определенным оборудованием.  Дополнительные сведения о развертывании и требования к оборудованию, ознакомьтесь со статьями, [Общие сведения о виртуальных смарт-карт](https://technet.microsoft.com/library/dn593708.aspx) и [приступить к работе с помощью виртуальных смарт-карт: Пошаговое руководство](https://technet.microsoft.com/library/dn579260.aspx).
   * **Windows Hello для бизнеса**: Windows Hello для бизнеса позволяет пользователям проходить проверку подлинности учетной записи Майкрософт, учетную запись Active Directory, учетная запись Microsoft Azure Active Directory (Azure AD) или сторонних служб, который поддерживает проверку подлинности быстрого ID Online (FIDO). После первоначальной двухэтапной проверки во время Windows Hello для бизнеса регистрации Windows Hello для бизнеса установлен на устройстве пользователя, и пользователь задает жест, который может быть Windows Hello или ПИН-код. Windows Hello для бизнеса учетные данные являются парой асимметричных ключей, который можно создать в изолированных средах доверенных платформенных модулей (TPM).
      Дополнительные сведения о Windows Hello для бизнеса чтения [Windows Hello для бизнеса](https://docs.microsoft.com/windows/security/identity-protection/hello-for-business/hello-identity-verification) статьи.
   * **Многофакторная Идентификация Azure**:  Многофакторная Идентификация (MFA) Azure обеспечивает безопасность второго фактора проверки, а также улучшенную защиту за счет мониторинга и машины обучения на основе анализа.  Azure MFA обеспечивает защиту не только администраторов Azure, но и многих других решений, включая веб-приложения, Azure Active Directory и локальные решения, например удаленный доступ и удаленный рабочий стол.  Дополнительные сведения об Azure MFA см. в статье [Многофакторная проверка подлинности](https://azure.microsoft.com/services/multi-factor-authentication).

2. **Белый список доверенных приложений с помощью управления приложениями в Защитнике Windows и/или AppLocker**.  Ограничивая возможность запуска ненадежного или неподписанного кода на рабочей станции с привилегированным доступом, вы еще больше снижаете вероятность действий злоумышленников и компрометации.  В Windows предусмотрено два основных компонента для управления приложениями:

   * **AppLocker**:  AppLocker помогает администраторам контролировать, какие приложения могут выполняться в определенной системе.  Решением AppLocker можно централизованно управлять, используя групповую политику. Кроме того, его можно применять конкретным пользователям или группам (для целевых приложений пользователей рабочих станций с привилегированным доступом).  Дополнительные сведения об AppLocker см. в статье TechNet [Технический обзор AppLocker](https://technet.microsoft.com/library/hh831440.aspx).
   * **Управление приложениями в Защитнике Windows**: новая функция управления приложениями в Защитнике Windows предоставляет улучшенное приложение на основе оборудования элемент управления, который, в отличие от AppLocker, невозможно переопределить на затронутом устройстве.  Как и AppLocker управления через групповую политику управления приложениями в Защитнике Windows и предназначен для конкретных пользователей.  Дополнительные сведения об ограничении использования приложений с помощью управления приложениями в Защитнике Windows, обратитесь к [руководство по развертыванию Windows Defender приложения управления](https://docs.microsoft.com/windows/security/threat-protection/windows-defender-application-control/windows-defender-application-control-deployment-guide).

3. **Используйте защищенных пользователей, политики проверки подлинности и приемники команд проверки подлинности для обеспечения дополнительной защиты привилегированных учетных записей**.  На участников группы защищенных пользователей распространяются дополнительные политики безопасности, которые позволяют защитить учетные данные, хранимые в локальном агенте безопасности, и значительно снизить риск кражи и повторного использования учетных данных.  Политики проверки подлинности и приемники команд управляют доступом привилегированных пользователей к ресурсам в домене.  Вместе эти средства защиты значительно повышают уровень безопасности учетных записей привилегированных пользователей.  Дополнительные сведения об этих возможностях см. в веб-статье [How to Configure Protected Accounts](https://technet.microsoft.com/library/dn518179.aspx) (Настройка защищенных учетных записей).

   > [!NOTE]
   > Эти средства защиты предназначены для дополнения, а не замены существующих средств безопасности на этапе 1.  Администраторы по-прежнему должны использовать отдельные учетные записи для администрирования и общих задач.

## <a name="managing-and-updating"></a>Управление и обновление

Для обеспечения целостности рабочих станций с привилегированным доступом в них должны быть внедрены антивредоносные программы. Кроме того, программное обеспечение следует обновлять как можно скорее.

В рабочих станциях с привилегированным доступом можно также внедрить дополнительные возможности для управления конфигурациями, мониторинга операций и управление безопасностью. Однако их интеграцию следует тщательно продумать, так как каждая возможность для управления с помощью инструмента подвергает рабочую станцию с привилегированным доступом риску компрометации. Имеет ли смысл представить расширенных возможностей управления зависит от нескольких факторов, включая:

* состояние безопасности и методики, предоставляемые возможностью для управления (включая методы обновления программного обеспечения для средства, административные роли и учетные записи в этих ролях, операционные системы, используемые для размещения средства или управления им, а также другие зависимости средства от оборудования или программного обеспечения);
* частота и количество развертываний и обновлений программного обеспечения на рабочей станции с привилегированным доступом;
* требования к подробным сведениям о запасах и конфигурации;
* требования к мониторингу безопасности;
* стандарты организации и другие факторы, зависящие от организации.

Согласно принципу чистоты источника уровень доверия всех средств, используемых для управления рабочими станциями с привилегированным доступом и их мониторинга, должен быть не ниже уровня доверия рабочих станций с привилегированным доступом. Как правило управление этими средствами должно осуществляется из рабочей станции с привилегированным доступом для обеспечения независимости от уровня безопасности рабочих станций с привилегиями нижнего уровня.

## <a name="deploy-paws-using-a-guarded-fabric"></a>Развертывание с помощью Защищенная структура привилегированным доступом

Объект [защищенных fabric](https://aka.ms/shieldedvms) может использоваться для запуска рабочих нагрузок с привилегированным ДОСТУПОМ в экранированных виртуальных машин на ноутбуке или сервер перехода.
Этот подход требует дополнительной инфраструктуры и операций, но можно упростить его повторное развертывание образов с привилегированным ДОСТУПОМ с регулярными интервалами и позволяет объединять несколько разных привилегированным доступом многоуровневых (или классификации) в виртуальные машины Side-by-side на одном устройстве.
Подробное описание из защищенной структуры обещаний топологии и безопасности, обратитесь к [документацией по защищенной структуре](https://aka.ms/shieldedvms).

### <a name="changes-to-the-paw-gpos"></a>Изменения в объекты групповой политики с привилегированным ДОСТУПОМ

При с помощью экранированных привилегированным доступом на основе виртуальной Машины [рекомендуемые параметры объекта групповой Политики](#create-paw-configuration---computer-group-policy-object-gpo) определенные выше, необходимо будет изменить, чтобы поддерживать использование виртуальных машин.

1. Создание нового Подразделения для физических узлов с привилегированным ДОСТУПОМ. Физическими и виртуальными привилегированным доступом имеют различные требования к безопасности и должны быть разделены по Active Directory соответствующим образом.
2. Объект групповой Политики компьютера привилегированным ДОСТУПОМ должна быть связана как с физическими и виртуальными ПРИВИЛЕГИРОВАННЫМ подразделениями.
3. Создайте новый объект групповой Политики для физических с привилегированным ДОСТУПОМ добавить пользователей с привилегированным ДОСТУПОМ к группе "Администраторы" Hyper-V. Это необходимо для позволяют администраторам соединиться с администратора виртуальных машин и включать их включить или выключить мере необходимости. Очень важно, что пользователь, входящий в физических станции с привилегированным ДОСТУПОМ не имеет прав администратора, доступ к Интернету, или возможность скопировать данные вредоносной виртуальной машины из общих сетевых ресурсов или внешние устройства хранения данных на физических станции с привилегированным ДОСТУПОМ.
4. Создайте новый объект групповой Политики для администратора виртуальных машин, чтобы добавить пользователей с привилегированным ДОСТУПОМ в группу Пользователи удаленного рабочего стола. Это позволит пользователям использовать Hyper-V улучшены сеансы консоли, обеспечивающим удобство работы пользователей и обеспечивает passthrough смарт-карты к виртуальной Машине.

### <a name="set-up-the-host-guardian-service"></a>Настройка службы защиты узла

Служба защиты узла несет ответственность за заверяющую идентификации и работоспособности физического устройства с привилегированным ДОСТУПОМ.
Только компьютеры, на которые HGS и запуск доверенный [политика целостности кода](https://docs.microsoft.com/windows/security/threat-protection/windows-defender-application-control/windows-defender-application-control) разрешены для запуска экранированных виртуальных машин.
Это позволяет защитить эти виртуальные машины, которые выполняются доверенные рабочие нагрузки для управления ресурсами многоуровневых от угроз среду рабочего стола пользователя.

Поскольку HGS отвечает за определение, какие устройства можно запустить виртуальные машины с привилегированным ДОСТУПОМ, он считается ресурсом уровня 0.
Ее нужно развернуть параллельно с другими ресурсами уровня 0 и от несанкционированного доступа физических и логических.
HGS является кластерной роли, упрощая масштабное развертывание для развертывания любого размера.
Общее правило состоит в том, для планирования 1 сервер HGS на каждые 1000 устройств, которые имеют менее 3 узлов.

1. Чтобы установить первый сервер HGS, начните с [установить HGS - лесу-бастионе](../../security/guarded-fabric-shielded-vm/guarded-fabric-install-hgs-in-a-bastion-forest.md) статьи и присоединить к домену уровня 0 HGS.

2. Затем [создавать сертификаты для HGS](../../security/guarded-fabric-shielded-vm/guarded-fabric-obtain-certs.md) с помощью центра сертификации предприятия.
Любой пользователь, владеющий HGS шифрования и подписывания сертификатов может расшифровать экранированной виртуальной Машины, поэтому если у вас есть доступ к аппаратным модулем безопасности для защиты закрытых ключей, рекомендуется создавать эти сертификаты с помощью аппаратного модуля безопасности.
Для лучшей защиты выберите размер ключа не меньше 4096 бит.

3. Наконец, выполните действия, чтобы [инициализировать сервер HGS](../../security/guarded-fabric-shielded-vm/guarded-fabric-initialize-hgs-tpm-mode-bastion.md) в **режиме TPM**.
Инициализации устанавливает аттестации и защиты ключей веб-служб, используемых вашей привилегированным доступом.
Должно быть HGS [настроен сертификат TLS](../../security/guarded-fabric-shielded-vm/guarded-fabric-configure-hgs-https.md) для защиты этих сообщений и только порт 443, должен быть открыт из ненадежных сетей для HGS.

4. Выполните действия, чтобы [добавить дополнительные узлы](../../security/guarded-fabric-shielded-vm/guarded-fabric-configure-additional-hgs-nodes.md) для вашей второй, третий и дополнительные узлы HGS.

5. Если сервер HGS работает под управлением Windows Server 2019 или более поздней версии, вы можете включить дополнительный компонент ключи в кэш для экранированных виртуальных машин на привилегированным доступом, чтобы можно было использовать автономный режим. Ключи являются запечатанными к текущей конфигурации безопасности для предотвращения от использования кэшированных ключей на другом компьютере или компьютере в небезопасное состояние системы. Это может быть эффективна, если станции с привилегированным ДОСТУПОМ пользователей проходить без доступа к Интернету, но все равно нужно войти в виртуальные машины привилегированным ДОСТУПОМ. Чтобы использовать эту функцию, выполните следующую команду на любом сервере HGS:

      ```powershell
      Set-HgsKeyProtectionConfiguration -AllowKeyMaterialCaching:$true
      ```

### <a name="set-up-the-physical-paw-device"></a>Настройка физического устройства с привилегированным ДОСТУПОМ

Физическое устройство с привилегированным ДОСТУПОМ считается доверенным по умолчанию в решении защищенной структуры.
Он может доказать, что он заслуживает во время процесса подтверждения, после чего оно может получить ключи, необходимые для запуска экранированных администратора виртуальной Машины.
Устройства должны иметь возможность запускать Hyper-V и безопасная загрузка и TPM 2.0 включена в соответствии с [защищенных предварительные требования узлов](../../security/guarded-fabric-shielded-vm/guarded-fabric-guarded-host-prerequisites.md).
Минимальная версия операционной системы для поддержки всех функций с привилегированным ДОСТУПОМ — **Windows 10 версии 1803**.

Как и любое другое, за исключением, что все пользователи с привилегированным ДОСТУПОМ, необходимо будет администраторов Hyper-V, чтобы иметь возможность включить в виртуальную Машину администратора и подключиться к нему следует настроить физических станции с привилегированным ДОСТУПОМ.
В вашей среде чистую комнату необходимо создать конфигурацию "Золотая" для каждого сочетания уникальных оборудование и программное обеспечение, развертываемое как защищенных узлов для администрирования виртуальных машин.
В каждой конфигурации "Золотая" выполните следующие задачи:

1. Установите последние обновления для Windows, драйверов и встроенного по на компьютере, а также любой управления третьих лиц или агентов мониторинга.
2. [Записать сведения о необходимых базовых](../../security/guarded-fabric-shielded-vm/guarded-fabric-tpm-trusted-attestation-capturing-hardware.md), включая уникальный идентификатор доверенный платформенный модуль (ключ подтверждения) загрузки измерения (журнал TCG) и политика целостности для машины кода.
3. Скопируйте эти артефакты к серверу HGS и выполните команды аттестации HGS в предыдущей статье, чтобы зарегистрировать узел. Если все узлы использовать ту же политику целостности кода и/или используйте ту же конфигурацию оборудования, необходимо зарегистрировать журнал политики/TCG целостности кода.

### <a name="create-the-signed-template-disk"></a>Создайте подписанный диск шаблона

Экранированные виртуальные машины создаются с помощью подписанных дисках шаблонов.
Подпись проверяется во время развертывания для проверки целостности диска и подлинность, не отпуская секретные данные, например пароль администратора в виртуальную Машину.

Чтобы создать диск подписанного шаблона, сделайте этап 1 развертывания на обычный, виртуальная машина поколения 2.
Этот компьютер станет "золотого образа" для администратора виртуальной Машины.
Можно создать более одного диска шаблона специализированные средства, доступные в различных контекстах.

Когда виртуальная машина настроена в качестве требуемого, запустите `C:\Windows\System32\sysprep\sysprep.exe` и решили **Generalize** диска. **Завершение работы** завершении обобщение операционной системы.

Наконец, запустите [мастер шаблонов диска](../../security/guarded-fabric-shielded-vm/guarded-fabric-create-a-shielded-vm-template.md) на vhdx-файл из виртуальной Машины, чтобы установить компоненты BitLocker и создать подпись диска.

#### <a name="create-the-shielding-data-file"></a>Создайте файл данных экранирования

Диск универсального шаблона сопряжен с файл данных экранирования, который содержит секретные данные, необходимые для подготовки экранированной виртуальной Машины.
Файл данных экранирования включает в себя:
   - Список guardians, которые определяют структуры, где разрешено выполнение виртуальной Машины. Каждый кластер HGS — защиты для устройств с привилегированным ДОСТУПОМ, которые он защищает.
   - Список из подписи диска для развертывания. Файлы данных экранирования освободит только свои секреты для виртуальных машин, созданных с помощью носителя авторизованного источника.
   - Политика безопасности, определяет ли дополнительные средства защиты должны быть помещены в месте, чтобы защитить виртуальную Машину на узле и разрешены ли виртуальная машина для перемещения на другой компьютер. Станции с привилегированным ДОСТУПОМ администратора виртуальных машин должен всегда быть полностью изолирована.
   - Специализация файле unattend.xml, которая позволяет Windows, чтобы автоматически завершить установку и содержат секретные данные, такие как пароль локального администратора.
   - Дополнительные файлы, такие как сертификаты протокола удаленного рабочего СТОЛА или VPN.

См. в разделе [статье файл данных экранирования](../../security/guarded-fabric-shielded-vm/guarded-fabric-tenant-creates-shielding-data.md) инструкции о том, как создать файл данных экранирования.

Владелец ключей для экранированных виртуальных машин являются конфиденциальными и его следует в аппаратный модуль безопасности или автономного хранения в безопасном месте.
Они могут использоваться в сценарии аварийного break прозрачного стекла для загрузки экранированной виртуальной Машины без наличия HGS.

Настоятельно рекомендуется, что данные для администратора с привилегированным ДОСТУПОМ экранирования виртуальных машин включают параметр, чтобы заблокировать виртуальную Машину для первого физического узла, где она загружается.
Это будет препятствовать кто-то перемещение административную виртуальную Машину из одной станции с привилегированным ДОСТУПОМ в другой станции с привилегированным ДОСТУПОМ в той же среде.
Чтобы использовать эту функцию, создайте файл данных экранирования с помощью PowerShell и добавьте **BindToHostTpm** параметр:

```powershell
New-ShieldingDataFile -Policy Shielded -BindToHostTpm [...]
```

#### <a name="deploy-an-admin-vm"></a>Развертывание администратора виртуальной Машины

После подготовки диска шаблона и файл данных экранирования можно развернуть административную виртуальную Машину на любой станции с привилегированным ДОСТУПОМ, который был зарегистрирован в службе HGS.

1. Скопируйте шаблон диска (vhdx) и файл данных экранирования (PDK-файл) на доверенное устройство с привилегированным ДОСТУПОМ.
2. Следуйте инструкциям, чтобы [развертывание новой экранированной виртуальной Машины с помощью PowerShell](../../security/guarded-fabric-shielded-vm/guarded-fabric-create-a-shielded-vm-using-powershell.md)
3. Выполните оставшиеся действия на первом этапе процесса развертывания для защиты операционной системы виртуальной Машины и настроить его для его предполагаемой роли, соответствующим образом.
