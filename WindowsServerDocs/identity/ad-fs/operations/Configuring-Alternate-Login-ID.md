# <a name="configuring-alternate-login-id"></a>Настройка альтернативного идентификатора входа


## <a name="what-is-alternate-login-id"></a>Что такое альтернативного идентификатора входа?
В большинстве случаев пользователи использовать имени участника-пользователя (имена участника-пользователя) для входа с использованием учетной записи. Однако в некоторых средах из-за корпоративной политики или зависимостей локальных бизнес-приложения, пользователи могут использовать другие формы входа в систему. 

>[!NOTE]
>Рекомендованное корпорацией Майкрософт, что рекомендации в соответствии с основным адресом SMTP имени участника-пользователя. Эта статья затрагивает небольшой процент клиентов, которые не может устранить имени участника-пользователя, для сопоставления.

Например их можно создать с помощью идентификатора электронной почты для входа в систему, и, может отличаться от имени участника-пользователя. Это особенно часто в сценариях, где имени участника-пользователя не поддерживает маршрутизацию. Рассмотрим Jane Doe с именем участника-пользователя jdoe@contoso.local и адрес электронной почты jdoe@contoso.com. Мария может оказаться даже о имя участника-пользователя, так как она всегда использовал свой идентификатор электронной почты для входа в. Использование любого другого входа метода вместо имени участника-пользователя является альтернативного идентификатора. Для см. в разделе, создает дополнительную информацию о том, как имя участника-пользователя [заполнение Azure AD UserPrincipalName](https://docs.microsoft.com/azure/active-directory/connect/active-directory-aadconnect-userprincipalname).

Active Directory служб федерации (AD FS) позволяет федеративным приложениям с помощью AD FS для входа в систему с использованием альтернативного идентификатора. Это позволяет администраторам указать вместо имени участника-пользователя, используемый для входа в систему по умолчанию. AD FS уже поддерживает, с помощью идентификатора пользователя, которая принимается доменными службами Active Directory (AD DS) в любой форме. При настройке альтернативного идентификатора, AD FS позволяет пользователям входить с помощью настроенных альтернативное значение идентификатора, например идентификатор электронной почты. Использование альтернативного идентификатора позволяет внедрять поставщиков SaaS, таких как Office 365 без изменения вашей локальных имен участников-пользователей. Он также предоставляет возможность поддержки службы бизнес-приложений с удостоверениями потребителей подготовкой.

## <a name="alternate-id-in-azure-ad"></a>Альтернативный идентификатор в Azure AD
Организации может потребоваться использовать альтернативный идентификатор в следующих сценариях:
1. Имя локального домена не поддерживает маршрутизацию, например. Contoso.Local и в результате имя участника-пользователя по умолчанию не поддерживает маршрутизацию (jdoe@contoso.local). Существующие имя участника-пользователя нельзя изменить из-за зависимостей локального приложения или политики компании. Azure AD и Office 365 требует все суффиксы доменов, связанных с каталогом Azure AD для полного internet маршрутизацию. 
2. Имя участника-пользователя в локальной отличается от адреса электронной почты пользователя и войти в Office 365, пользователи использовать адрес электронной почты и имя участника-пользователя не может использоваться из-за ограничений организации.
   В указанных выше сценариев альтернативного идентификатора с AD FS дает пользователям возможность входа в Azure AD без изменения вашей локальных имен участников-пользователей. 

## <a name="end-user-experience-with-alternate-login-id"></a>Взаимодействие с пользователем с помощью альтернативного идентификатора входа
Взаимодействие с конечным пользователем зависит от метода аутентификации, используемого с альтернативного идентификатора входа.  В настоящее время существует три разных способа, в которых можно сделать с помощью альтернативного идентификатора входа.  Подробные сведения.

- **Регулярные проверки подлинности (для прежних версий)** -осуществляется по протоколу обычной проверки подлинности.
- **Современная проверка подлинности** -обеспечивает основе библиотеки АУТЕНТИФИКАЦИИ Active Directory входа к приложениям. Это позволяет функции например многофакторная Идентификация (MFA), на основе SAML-сторонних поставщиков удостоверений с клиентскими приложениями Office, смарт-карты и проверки подлинности на основе сертификатов входа.
- **Гибридной современной проверкой подлинности** — предоставляет все преимущества современной проверки подлинности и предоставляет пользователям возможность доступа к локальным приложениям с помощью авторизации маркеры, полученные из облака.

>[!NOTE]
> Для получения наилучших результатов Корпорация Майкрософт настоятельно рекомендует гибридной современной проверкой подлинности.

### <a name="manually-configure-alternate-id"></a>Настройка альтернативного идентификатора вручную
Для настройки альтернативного имени пользователя, необходимо выполнить следующие задачи: Настройка вашей отношений с поставщиками утверждений AD FS для включения альтернативного идентификатора входа

1.  Если у вас есть Server 2012 R2, убедитесь, что у вас есть KB2919355 установлена на всех серверах AD FS. Можно получить посредством службы обновления Windows или загрузить его напрямую. 

2.  Обновите конфигурацию AD FS, выполнив следующий командлет PowerShell на любом из серверов федерации в ферме (при наличии фермы внутренней базы данных Windows, необходимо запустить эту команду на основной сервер AD FS в ферме):

``` powershell
Set-AdfsClaimsProviderTrust -TargetIdentifier "AD AUTHORITY" -AlternateLoginID <attribute> -LookupForests <forest domain>
```

**AlternateLoginID** имя LDAP атрибута, который вы хотите использовать для входа в систему.

**LookupForests** — список DNS-сервер, пользователи относятся к лесу.

Чтобы включить функцию идентификатор альтернативного имени пользователя, необходимо настроить параметры - AlternateLoginID и - LookupForests со значением, отличных от null, допустимым.

В следующем примере включается функциональность альтернативного имени для входа идентификатор таким образом, что пользователей с учетными записями в contoso.com и fabrikam.com леса можно в систему AD FS приложений с поддержкой их атрибут «mail».

``` powershell
Set-AdfsClaimsProviderTrust -TargetIdentifier "AD AUTHORITY" -AlternateLoginID mail -LookupForests contoso.com,fabrikam.com
```

3. Чтобы отключить эту функцию, задайте значение для обоих параметров должен иметь значение null.

``` powershell
Set-AdfsClaimsProviderTrust -TargetIdentifier "AD AUTHORITY" -AlternateLoginID $NULL -LookupForests $NULL
```

### <a name="pre-requisites-for-exchange-and-skype-for-business"></a>Предварительные требования для Exchange и Скайп для бизнеса
Ниже приведены предварительные требования для обеспечения единого входа с помощью альтернативного идентификатора.

- Exchange Online необходимо включение современной проверки подлинности.
- Скайп для бизнеса (SFB) Online должны иметь включена современная проверка подлинности.
- В локальной организации Exchange должны иметь современная проверка подлинности включена.  Exchange 2013 CU19 или Exchange 2016 CU18 и копирования необходим на всех серверах Exchange. Нет Exchange 2010 в среде.
- Скайп для бизнеса в локальном должен иметь современная проверка подлинности включена.
- Необходимо использовать Exchange и Скайп клиентов, поддерживающих современную аутентификацию. Все серверы должны работать под CU5 2015 SFB сервера.
- Скайп для бизнес-клиентами, которые поддерживают современную проверку подлинности
   - iOS, Android, Windows Phone
   - SFB 2016 (MA включено по умолчанию, но убедитесь, что он не был отключен).
   - SFB 2013 (MA отключена по умолчанию, поэтому MA был включен.)
   - Рабочий стол SFB Mac
- Клиенты Exchange, которые поддерживают современную проверку подлинности и поддержки с помощью AltID
    - Office профессиональный плюс 2016 только

##### <a name="step-1-update-to-required-office-version"></a>Шаг 1. Требуется обновление для версии Office

## <a name="additional-details--considerations"></a>Дополнительные сведения и рекомендации

-   Эта функция идентификатор альтернативного имени пользователя доступна для федеративных средах с AD FS, развернутые.  Не поддерживается в следующих сценариях:
    -   Не поддерживающие маршрутизацию доменов (например, Contoso.local), не может быть проверен Azure AD.
    -   Управляемые среды, у которых нет AD FS, развернутые.


-   При включении альтернативного имени для входа идентификатор функция доступна только для проверки подлинности имя пользователя и пароль для всех пользователей имя и пароль проверки подлинности протоколов, поддерживаемых AD FS (SAML-P, WS-Fed, WS-Trust и OAuth).


-   При проведении интегрированной проверки подлинности Windows (WIA) случае (например, когда пользователи пытаются получить доступ к корпоративному приложению на компьютере, присоединенном к домену из интрасети и AD FS администратор настроил политику проверки подлинности для использования WIA для интрасети), имя участника-пользователя isused для проверки подлинности. Если вы настроили какие-либо правила утверждения для проверяющих сторон для альтернативного имени для входа идентификатор компонента, должны убедитесь, что эти правила, по-прежнему допустимы в случае WIA.

-   При включении функции Идентификаторов альтернативного имени для входа требуется хотя бы один сервер глобального каталога может быть достигнут из сервера AD FS для каждого леса учетной записи пользователя, который поддерживает AD FS. Не удалось связаться с сервером глобального каталога в лесу учетных записей пользователя приводит к AD FS, а не использовать имя участника-пользователя. По умолчанию все контроллеры домена, серверы глобального каталога.

-   Когда включена, если сервер AD FS находит более одного объекта пользователя с одинаковым значением идентификатор альтернативного имени пользователя, указанного во всех лесах учетной записи пользователя, происходит сбой, имя входа.

-   Если включена функция идентификатор альтернативного имени пользователя, AD FS пытается сначала проверки подлинности конечного пользователя с помощью альтернативного имени пользователя и затем откат для использования имени участника-пользователя, если не удается найти учетную запись, можно определить по альтернативного имени пользователя. Следует убедиться, что имеются нет конфликты между альтернативного идентификатора входа и имя участника-пользователя, если вы хотите по-прежнему поддерживают учетная запись имени участника-пользователя. Например параметр единицы атрибута mail с другой стороны имени участника-пользователя заблокировано для других вход с использованием его имени участника-пользователя.

-   Если один из лесов, настроенную администратором не работает, для поиска учетной записи пользователя с помощью альтернативного идентификатора входа в других лесах, настроенные продолжает AD FS. Если сервер AD FS находит уникального пользователя объектов между лесами, которые его поиск, пользователь успешно входит.

-   Кроме того можно настроить страницу входа AD FS по предоставлению конечным пользователям подсказка о альтернативного имени пользователя. Это можно сделать путем добавления описание настроенной страницы входа в систему (Дополнительные сведения см. в разделе [Customizing the AD FS Sign-in Pages](https://technet.microsoft.com/library/dn280950.aspx) или настройка строку «Войти с учетной записью организации» над полем имени пользователя (для получения дополнительных сведений сведения см. в разделе [Advanced Customization of AD FS Sign-in Pages](https://technet.microsoft.com/library/dn636121.aspx).

-   Новый тип утверждения, содержащий значение идентификатора альтернативного имени для входа является **http:schemas.microsoft.com/ws/2013/11/alternateloginid**
