# <a name="deploying-active-directory-federation-services-in-azure"></a>Развертывание службы федерации Active Directory (AD FS) в Azure
AD FS предоставляет упрощенную, защищенную федерацию удостоверений и возможности единого входа (SSO). Федерация с Azure AD или Office 365 позволяет пользователям проходить проверку подлинности с помощью локальных учетных данных и получать доступ ко всем ресурсам в облаке. Поэтому важно иметь высокодоступную инфраструктуру AD FS для обеспечения доступа к ресурсам как локально, так и в облаке. Развертывание AD FS в Azure поможет добиться высокой доступности, необходимой для минимальных усилий.
Существует несколько преимуществ развертывания AD FS в Azure. Некоторые из них перечислены ниже.

* **Высокая доступность** . Благодаря преимуществам групп доступности Azure вы гарантируете инфраструктуру высокой доступности.
* **Простота масштабирования** — требуется дополнительная производительность? Простая миграция на более мощные компьютеры с помощью нескольких щелчков в Azure
* **Географическая избыточность** . Благодаря геоизбыточности Azure вы можете быть уверены, что ваша инфраструктура обеспечивает высокую доступность по всему миру.
* **Простота управления** — благодаря высоко упрощенным средствам управления в портал Azure, управление инфраструктурой выполняется очень просто и без проблем. 

## <a name="design-principles"></a>Принципы проектирования
![Разработка развертывания](./media/how-to-connect-fed-azure-adfs/deployment.png)

На схеме выше показана Рекомендуемая Базовая топология для начала развертывания инфраструктуры AD FS в Azure. Ниже перечислены принципы различных компонентов топологии.

* **Серверы DC/ADFS**. Если у вас менее 1 000 пользователей, можно просто установить AD FS роль на контроллерах домена. Если вы не хотите оказывать влияние на производительность контроллеров домена или если у вас более 1 000 пользователей, разверните AD FS на отдельных серверах.
* **Сервер WAP** — необходимо развернуть прокси-серверы веб приложения, чтобы пользователи могли получить доступ к AD FS, если они не находятся в сети компании.
* **Демилитаризованная**зона. прокси-серверы веб приложения будут помещены в сеть периметра, и между демилитаризованной зоной и внутренней подсетью будет разрешен только доступ по протоколу TCP/443.
* Подсистемы **балансировки нагрузки**. чтобы обеспечить высокий уровень доступности AD FS и прокси-серверов, мы рекомендуем использовать внутреннюю подсистему балансировки нагрузки для AD FS серверов и Azure Load Balancer для прокси-серверов веб – приложений.
* **Группы доступности**. чтобы обеспечить избыточность развертывания AD FS, рекомендуется сгруппировать две или несколько виртуальных машин в группе доступности для схожих рабочих нагрузок. Эта конфигурация гарантирует, что во время запланированного или незапланированного обслуживания будет доступна как минимум одна виртуальная машина.
* **Учетные записи хранения**. рекомендуется использовать две учетные записи хранения. Наличие одной учетной записи хранения может привести к созданию единой точки отказа и может привести к недоступности развертывания в маловероятном сценарии, в котором учетная запись хранения будет отключена. Две учетные записи хранения помогут связать одну учетную запись хранения с каждой линией сбоя.
* **Разделение сети**: прокси-серверы веб приложения должны быть развернуты в отдельной сети DMZ. Одну виртуальную сеть можно разделить на две подсети, а затем развернуть в изолированной подсети серверы прокси веб-приложения. Можно просто настроить параметры группы безопасности сети для каждой подсети и разрешить только требуемый обмен данными между двумя подсетями. Дополнительные сведения приведены в следующем сценарии развертывания.

## <a name="steps-to-deploy-ad-fs-in-azure"></a>Действия по развертыванию AD FS в Azure
Шаги, описанные в этом разделе, описывают руководство по развертыванию описанной ниже AD FS инфраструктуры в Azure.

### <a name="1-deploying-the-network"></a>1. Развертывание сети
Как описано выше, можно либо создать две подсети в одной виртуальной сети, либо создать две совершенно разные виртуальные сети (VNet). В этой статье рассматривается развертывание одной виртуальной сети и разделение ее на две подсети. В настоящее время это более простой подход, так как две отдельные виртуальных сетей потребуют шлюза виртуальной сети для связи.

**1,1. Создание виртуальной сети**

![Создать виртуальную сеть](./media/how-to-connect-fed-azure-adfs/deploynetwork1.png)

В портал Azure выберите Виртуальная сеть, и вы можете развернуть виртуальную сеть и одну подсеть сразу, только одним щелчком мыши. Подсеть INT также определена, и теперь она готова к добавлению виртуальных машин.
Следующим шагом является добавление в сеть другой подсети, т. е. подсети DMZ. Чтобы создать подсеть DMZ, просто

* Выберите только что созданную сеть
* В окне Свойства выберите подсеть.
* На панели "подсеть" нажмите кнопку "Добавить".
* Укажите имя подсети и сведения об адресном пространстве для создания подсети

![Подсеть](./media/how-to-connect-fed-azure-adfs/deploynetwork2.png)

![ДЕМИЛИТАРИЗОВАНная зона подсети](./media/how-to-connect-fed-azure-adfs/deploynetwork3.png)

**1,2. Создание групп безопасности сети**

Группа безопасности сети (NSG) содержит список правил списка управления доступом (ACL), которые разрешают или запрещают сетевой трафик для экземпляров виртуальных машин в виртуальной сети. Группы безопасности сети можно связать с подсетями или отдельными экземплярами виртуальной машины в этой подсети. Если NSG связан с подсетью, правила ACL применяются ко всем экземплярам виртуальных машин в этой подсети.
В рамках этого руководства мы создадим два группы безопасности сети: по одному для внутренней сети и ДЕМИЛИТАРИЗОВАНной зоны. Они будут помечены NSG_INT и NSG_DMZ соответственно.

![Создание NSG](./media/how-to-connect-fed-azure-adfs/creatensg1.png)

После создания NSG будет установлено 0 входящих и 0 исходящих правил. После установки и функционирования ролей на соответствующих серверах можно установить правила для входящих и исходящих подключений в соответствии с требуемым уровнем безопасности.

![Инициализация NSG](./media/how-to-connect-fed-azure-adfs/nsgint1.png)

После создания группы безопасности сети свяжите NSG_INT с подсетью INT и NSG_DMZ с демилитаризованной зоной сети. Ниже приведен пример снимка экрана:

![Настройка NSG](./media/how-to-connect-fed-azure-adfs/nsgconfigure1.png)

* Щелкните подсети, чтобы открыть панель для подсетей.
* Выберите подсеть для связи с NSG 

После настройки панель для подсетей должна выглядеть следующим образом:

![Подсети после NSG](./media/how-to-connect-fed-azure-adfs/nsgconfigure2.png)

**1,3. Создание подключения к локальной среде**

Для развертывания контроллера домена в Azure потребуется подключение к локальной среде. Azure предлагает различные варианты подключения для подключения локальной инфраструктуры к инфраструктуре Azure.

* Точка-сеть
* Сеть-сеть виртуальной сети
* ExpressRoute

Рекомендуется использовать ExpressRoute. ExpressRoute позволяет создавать частные подключения между центрами обработки данных Azure и инфраструктурой на локальном компьютере или в среде совместного размещения. Подключения ExpressRoute не проходят через общедоступный Интернет. Они обеспечивают повышенную надежность, скорость, меньшую задержку и более высокий уровень безопасности по сравнению с обычными подключениями через Интернет.
Хотя мы рекомендуем использовать ExpressRoute, вы можете выбрать любой метод подключения, наиболее подходящий для вашей организации. Чтобы узнать больше о ExpressRoute и различных вариантах подключения с помощью ExpressRoute, ознакомьтесь с [техническим обзором expressroute](https://aka.ms/Azure/ExpressRoute).

### <a name="2-create-storage-accounts"></a>2. Создание учетных записей хранения
Чтобы обеспечить высокий уровень доступности и избежать зависимости от одной учетной записи хранения, можно создать две учетные записи хранения. Разделите компьютеры в каждой группе доступности на две группы, а затем назначьте каждую группу отдельной учетной записью хранения.

![Создание учетных записей хранения](./media/how-to-connect-fed-azure-adfs/storageaccount1.png)

### <a name="3-create-availability-sets"></a>3. Создание групп доступности
Для каждой роли (DC/AD FS и WAP) создайте группы доступности, которые будут содержать по крайней мере два компьютера. Это поможет обеспечить более высокую доступность для каждой роли. При создании групп доступности важно принять решение о следующих возможностях.

* **Домены сбоя**. виртуальные машины в одном домене сбоя совместно используют один источник питания и физический сетевой коммутатор. Рекомендуется не менее 2 доменов сбоя. Значение по умолчанию — 3. Вы можете оставить его как есть в целях этого развертывания
* **Домены обновления**. компьютеры, принадлежащие к одному домену обновления, перезапускаются вместе во время обновления. Требуется не менее 2 доменов обновления. Значение по умолчанию — 5. Вы можете оставить его как есть в целях этого развертывания

![Группы доступности](./media/how-to-connect-fed-azure-adfs/availabilityset1.png)

Создание следующих групп доступности

| Группа доступности | Role | Домены сбоя | Домены обновления |
|:---:|:---:|:---:|:--- |
| контосодксет |DC/ADFS |3 |5 |
| контосовапсет |WAP |3 |5 |

### <a name="4-deploy-virtual-machines"></a>4. Развертывание виртуальных машин
Следующим шагом является развертывание виртуальных машин, в которых будут размещаться различные роли в инфраструктуре. В каждой группе доступности рекомендуется использовать не менее двух компьютеров. Создайте четыре виртуальные машины для базового развертывания.

| Компьютер | Role | Подсеть | Группа доступности | Учетная запись хранения | IP-адрес |
|:---:|:---:|:---:|:---:|:---:|:---:|
| contosodc1 |DC/ADFS |INT |контосодксет |contososac1 |Static |
| contosodc2 |DC/ADFS |INT |контосодксет |contososac2 |Static |
| contosowap1 |WAP |DMZ |контосовапсет |contososac1 |Static |
| contosowap2 |WAP |DMZ |контосовапсет |contososac2 |Static |

Как вы могли заметить, NSG не указан. Это связано с тем, что Azure позволяет использовать NSG на уровне подсети. Затем можно управлять сетевым трафиком компьютера с помощью отдельных NSG, связанных с подсетью или другим объектом NIC. Дополнительные сведения см. в статье [что такое группа безопасности сети (NSG)](https://aka.ms/Azure/NSG).
При управлении DNS рекомендуется использовать статический IP-адрес. Вы можете использовать Azure DNS и вместо этого в записях DNS для своего домена, обратитесь к новым компьютерам по их доменным именам Azure.
После завершения развертывания область виртуальной машины должна выглядеть следующим образом:

Чтобы сэкономить дорогостоящий путь из Azure к локальному контроллеру домена для проверки подлинности, рекомендуется развернуть реплику контроллера домена в Azure. Для достижения высокого уровня доступности рекомендуется создать группу доступности, сопоставленную как минимум 2 контроллеры домена.

| Контроллер домена | Role | Учетная запись хранения |
|:---:|:---:|:---:|
| contosodc1 |Реплика |contososac1 |
| contosodc2 |Реплика |contososac2 |

* Повышение уровня двух серверов как реплик контроллеров домена с помощью DNS
* Настройте AD FS серверы, установив роль AD FS с помощью диспетчера сервера.

### <a name="6-deploying-internal-load-balancer-ilb"></a>6. Развертывание внутренних Load Balancer (ILB)
**6,1. Создание ILB**

Чтобы развернуть ILB, выберите подсистемы балансировки нагрузки в портал Azure и нажмите кнопку Добавить (+).

> [!NOTE]
> Если вы не видите подсистемы **балансировки нагрузки** в меню, нажмите кнопку **Обзор** в левом нижнем углу портала и прокрутите экран, пока не увидите подсистемы **балансировки нагрузки**.  Затем щелкните желтую звезду, чтобы добавить ее в меню. Теперь выберите значок нового балансировщика нагрузки, чтобы открыть панель, чтобы начать настройку балансировщика нагрузки.
> 
> 

![Обзор подсистемы балансировки нагрузки](./media/how-to-connect-fed-azure-adfs/browseloadbalancer.png)

* **Имя**: присвойте подсистеме балансировки нагрузки любое подходящее имя.
* **Схема**: так как подсистема балансировки нагрузки будет размещена перед серверами AD FS и предназначена только для внутренних сетевых подключений, выберите "внутренний".
* **Виртуальная сеть**. Выберите виртуальную сеть, в которой развертывается AD FS
* **Подсеть**: Выберите внутреннюю подсеть здесь.
* **Назначение IP-адресов**: статическое

![Внутренняя подсистема балансировки нагрузки](./media/how-to-connect-fed-azure-adfs/ilbdeployment1.png)

После нажатия кнопки Создать и развертывания ILB вы увидите его в списке подсистем балансировки нагрузки:

![Подсистемы балансировки нагрузки после ILB](./media/how-to-connect-fed-azure-adfs/ilbdeployment2.png)

Следующим шагом является Настройка внутреннего пула и серверной проверки.

**6,2. Настройка внутреннего пула ILB**

Выберите только что созданный ILB на панели подсистемы балансировки нагрузки. Откроется панель Параметры. 

1. Выберите серверные пулы на панели "Параметры".
2. На панели добавить серверный пул щелкните Добавить виртуальную машину.
3. Появится панель, где можно выбрать группу доступности.
4. Выбор группы доступности AD FS

![Настройка внутреннего пула ILB](./media/how-to-connect-fed-azure-adfs/ilbdeployment3.png)

**6,3. Настройка пробы**

На панели Параметры ILB выберите зонды работоспособности.

1. Щелкните "Добавить".
2. Укажите сведения для зонда a. **Имя**: имя пробы b. **Протокол**: HTTP c. **Порт**: 80 (http) d. **Путь**:/адфс/пробе e. **Интервал**: 5 (значение по умолчанию) — это интервал, с которым ilB будет проверять компьютеры во внутреннем пуле f. **Неработоспособное ограничение порогового значения**: 2 (значение по умолчанию) — это порог последовательных ошибок пробы, после которого ilB будет объявлять компьютер во внутреннем пуле Неотвечающий и прекращать отправку трафика.

![Настройка проверки ILB](./media/how-to-connect-fed-azure-adfs/ilbdeployment4.png)

Мы используем конечную точку/адфс/пробе, созданную неявно для проверок работоспособности в среде AD FS, где не удается выполнить полную проверку HTTPS-пути.  Это значительно лучше, чем базовая проверка порта 443, которая не отражает точное состояние современного развертывания AD FS.  Дополнительные сведения об этом можно найти по адресу https://blogs.technet.microsoft.com/applicationproxyblog/2014/10/17/hardware-load-balancer-health-checks-and-web-application-proxy-ad-fs-2012-r2/.

**6,4. Создание правил балансировки нагрузки**

Для эффективного распределения трафика ILB следует настроить с помощью правил балансировки нагрузки. Чтобы создать правило балансировки нагрузки, 

1. Выберите правило балансировки нагрузки на панели Параметры ILB.
2. Нажмите кнопку Добавить на панели правил балансировки нагрузки.
3. На панели Добавление правила балансировки нагрузки a. **Имя**: укажите имя для правила б. **Протокол**: выберите TCP c. **Порт**: 443 d. **Внутренний порт**: 443 e. **Серверный пул**. Выберите пул, созданный для кластера AD FS более ранней версии. **Зонд**: выбор пробы, созданной для AD FS серверов ранее

![Настройка правил балансировки ILB](./media/how-to-connect-fed-azure-adfs/ilbdeployment5.png)

**6,5. обновление DNS с помощью ILB**

Перейдите на DNS-сервер и создайте запись CNAME для ILB. Запись CNAME должна быть для службы федерации с IP-адресом, указывающим на IP-адрес ILB. Например, если ILB DIP-адрес — 10.3.0.8, а установленная служба федерации — fs.contoso.com, то создайте запись CNAME для fs.contoso.com, указывающую на 10.3.0.8.
Это обеспечит правильную маршрутизацию всех соединений, касающихся fs.contoso.com в ILB.

### <a name="7-configuring-the-web-application-proxy-server"></a>7. Настройка сервера прокси веб-приложения
**7,1. Настройка серверов прокси веб-приложений для доступа к AD FS серверам**

Чтобы серверы прокси веб-приложений могли достичь AD FS серверов за ILB, создайте запись в%systemroot%\System32\Drivers\Etc\hosts для ILB. Обратите внимание, что различающееся имя (DN) должно быть именем службы федерации, например fs.contoso.com. А IP-запись должна быть IP-адресом ILB (10.3.0.8, как в примере).

**7,2. Установка роли прокси веб-приложения**

Убедившись, что прокси-серверы веб приложения могут получить доступ к AD FSным серверам за ILB, можно далее установить прокси-серверы веб приложения. Серверы прокси веб-приложений не должны быть присоединены к домену. Установите роли прокси веб-приложения на двух прокси-серверах веб приложения, выбрав роль удаленного доступа. Диспетчер серверов поможет завершить установку WAP.
Дополнительные сведения о том, как развертывать WAP, см. в статье [Установка и настройка сервера прокси веб-приложения](https://technet.microsoft.com/library/dn383662.aspx).
