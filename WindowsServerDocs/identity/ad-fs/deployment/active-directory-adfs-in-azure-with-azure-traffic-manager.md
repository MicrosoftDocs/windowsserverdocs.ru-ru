# <a name="high-availability-cross-geographic-ad-fs-deployment-in-azure-with-azure-traffic-manager"></a>Развертывание высокой доступности в нескольких регионах AD FS в Azure с помощью диспетчера трафика Azure
[Развертывание AD FS в Azure](how-to-connect-fed-azure-adfs.md) предоставляет пошаговые инструкции о том, как развернуть простую инфраструктуру AD FS для вашей организации в Azure. В этой статье описаны дальнейшие действия по для создания нескольких регионах развертывания AD FS в Azure с помощью [диспетчера трафика Azure](https://docs.microsoft.com/azure/traffic-manager/). Диспетчер трафика Azure предоставляет создайте географически spread высокий уровень доступности и высокопроизводительную инфраструктуру AD FS для вашей организации, применяя методы маршрутизации в соответствии с потребностями разных от инфраструктуры.

Обеспечивает инфраструктуру AD FS с высоким уровнем нескольких регионах:

* **Отказ от единой точки отказа:** С возможностями отработки отказа диспетчера трафика Azure можно добиться высокой доступности инфраструктуры AD FS, даже в том случае, если один из центров обработки данных в любом регионе мира выходит из строя
* **Улучшенная производительность:** Предлагаемые развертывания в этой статье можно использовать для предоставления высокопроизводительную инфраструктуру AD FS, которое может помочь пользователям быстрее выполнять проверку подлинности. 

## <a name="design-principles"></a>Принципы проектирования
![Общая схема](./media/active-directory-adfs-in-azure-with-azure-traffic-manager/blockdiagram.png)

Базовыми принципами разработки будет тем же, как указано в принципы проектирования в этой статье развертывание AD FS в Azure. На схеме выше показано простое расширение базового развертывания в другом географическом регионе. Ниже приведены некоторые моменты, которые следует учитывать при расширении развертывания на новом географическом регионе

* **Виртуальная сеть:** Следует создать новую виртуальную сеть в географическом регионе, вы хотите развернуть дополнительную инфраструктуру AD FS. В приведенной выше схеме вы увидите: Geo1 VNET и Geo2 VNET как две виртуальные сети в каждом географическом регионе.
* **Контроллеры домена и серверы AD FS в виртуальной сети:** Рекомендуется развернуть домен, который контроллеров в новом географическом регионе, чтобы серверы AD FS в новом регионе нет необходимости связаться с контроллером домена в другой слишком далеко от сети для выполнения проверки подлинности и тем самым повышая производительность.
* **Учетные записи хранения:** Учетные записи хранения связаны с областью. Так как вы будете развертывать машин в новом географическом регионе, необходимо будет создать новые учетные записи хранения для использования в регионе.  
* **Группы безопасности сети:** Так как учетные записи хранения, группы безопасности сети, созданные в регион не может использоваться в другом географическом регионе. Таким образом необходимо будет создание новых групп безопасности сети аналогичны первого географического региона для подсетей INT и DMZ в новом географическом регионе.
* **Имена DNS для общедоступных IP-адресов:** Диспетчер трафика Azure может ссылаться на конечные точки только по имени DNS. Таким образом необходимо создать имена DNS для общедоступных IP-адресов внешних балансировщиков нагрузки.
* **Диспетчер трафика Azure.** Диспетчер трафика Microsoft Azure позволяет управлять распределением пользовательского трафика для конечных точек службы в разных центрах обработки данных по всему миру. Диспетчер трафика Azure работает на уровне DNS. Он использует ответы DNS для перенаправления трафика пользователей к глобально распределенным конечным точкам. Затем клиенты подключаются к этим конечным точкам напрямую. С различными параметрами маршрутизации объекта производительности, взвешенной нагрузки или приоритета можно легко параметра маршрутизации, лучше всего подходит для вашей организации. 
* **V-net для подключение между двумя регионами:** Необходимо иметь подключение между виртуальными сетями, сам. Так как каждая виртуальная сеть имеет доступ к контроллерам домена и имеет сервер AD FS и WAP само по себе, они могут работать без подключения между виртуальными сетями в разных регионах. 

## <a name="steps-to-integrate-azure-traffic-manager"></a>Интеграция диспетчера трафика Azure
### <a name="deploy-ad-fs-in-the-new-geographical-region"></a>Развертывание AD FS в новом географическом регионе
Выполните шаги и рекомендации [развертыванию AD FS в Azure](how-to-connect-fed-azure-adfs.md) развернуть аналогичную топологию в новом географическом регионе.

### <a name="dns-labels-for-public-ip-addresses-of-the-internet-facing-public-load-balancers"></a>Имена DNS для общедоступных IP-адресов подсистем балансировки нагрузки с выходом в Интернет (общедоступного)
Как упоминалось выше, диспетчер трафика может ссылаться только на метки DNS, как конечные точки, и поэтому очень важно создать имена DNS для общедоступных IP-адресов внешних балансировщиков нагрузки. Приведенном ниже снимке экрана показано, как настроить DNS-имя для общедоступного IP-адреса. 

![Метка DNS](./media/active-directory-adfs-in-azure-with-azure-traffic-manager/eastfabstsdnslabel.png)

### <a name="deploying-azure-traffic-manager"></a>Развертывание диспетчера трафика Azure
Выполните следующие действия, чтобы создать профиль диспетчера трафика. Дополнительные сведения можно также ссылаться на [Управление профилем диспетчера трафика Azure](https://docs.microsoft.com/azure/traffic-manager/traffic-manager-manage-profiles).

1. **Создайте профиль диспетчера трафика:** Присвойте профилю диспетчера трафика уникальное имя. Это имя профиля является частью DNS-имя и выступает в качестве префикса для метки доменного имени диспетчера трафика. Имя / добавляется префикс. trafficmanager.net, чтобы создать имя DNS для трафика manager. На следующем снимке экрана показан диспетчер трафика префикс DNS, установить в качестве mysts и итоговый метка DNS будет mysts.trafficmanager.net. 
   
    ![Создание профиля диспетчера трафика](./media/active-directory-adfs-in-azure-with-azure-traffic-manager/trafficmanager01.png)
2. **Метод маршрутизации трафика:** В диспетчере трафика доступны три варианта маршрутизации:
   
   * Priority 
   * Производительность
   * Со смещением
     
     **Производительность** рекомендуется для достижения высокой скоростью отклика инфраструктуры AD FS. Тем не менее можно выбрать любой метод маршрутизации, лучше всего подходит для вашего развертывания. С выбранным параметром маршрутизации не влияет на функциональность AD FS. См. в разделе [методы маршрутизации трафика диспетчером трафика](https://docs.microsoft.com/azure/traffic-manager/traffic-manager-routing-methods) Дополнительные сведения. В этом примере можно увидеть снимок экрана выше **производительности** выбранного метода.
3. **Настройка конечных точек:** На странице диспетчера трафика щелкните конечные точки и выберите Add. Откроется страница добавления конечной точки, как на снимке экрана ниже
   
   ![Настройка конечных точек](./media/active-directory-adfs-in-azure-with-azure-traffic-manager/eastfsendpoint.png)
   
   Для различных входных параметров следуйте рекомендациям ниже:
   
   **Тип:** Выберите конечную точку Azure, как мы будет указывать на Azure общедоступный IP-адрес.
   
   **Имя:** Введите имя, которое необходимо связать с конечной точкой. Это не DNS-имя и никак не влияет на записи DNS.
   
   **Тип целевого ресурса:** Выберите общедоступный IP-адрес в качестве значения этого свойства. 
   
   **Целевой ресурс:** Это даст вам возможность выбора из различных меток DNS, что у вас есть доступные в рамках вашей подписки. Выберите метку DNS, соответствующий этой конечной точке, которую вы настраиваете.
   
   Добавьте конечную точку для каждого географического региона, диспетчер трафика Azure для маршрутизации трафика.
   Дополнительные сведения и подробные инструкции о том, как добавление и настройка конечных точек в диспетчере трафика см. в разделе [Добавление, отключение, включение или удаление конечных точек](https://docs.microsoft.com/azure/traffic-manager/traffic-manager-manage-endpoints)
4. **Настройка пробы:** На странице диспетчера трафика щелкните конфигурацию. На странице настройки необходимо изменить параметры монитора пробы на HTTP-порт 80 и относительный путь/ADFS/Probe
   
    ![Настройка пробы](./media/active-directory-adfs-in-azure-with-azure-traffic-manager/mystsconfig.png) 
   
   > [!NOTE]
   > **Убедитесь, что из конечных точек находится в состоянии ONLINE после завершения настройки**. Если все конечные точки находятся в «ограниченном» состоянии, диспетчер трафика Azure сделает максимально точного для маршрутизации трафика при условии, что диагностику ошибочной и все конечные точки доступны.
   > 
   > 
5. **Изменения записей DNS для диспетчера трафика Azure:** Службы федерации должно быть запись CNAME, чтобы имя DNS диспетчера трафика Azure. Создайте запись CNAME в общедоступных записях DNS, таким образом, желающие подключиться к службе федерации обращались к диспетчеру трафика Azure.
   
    К примеру чтобы она указывала на диспетчер трафика fs.fabidentity.com службы федерации, необходимо обновить запись ресурса DNS на следующую:
   
    <code>fs.fabidentity.com IN CNAME mysts.trafficmanager.net</code>

## <a name="test-the-routing-and-ad-fs-sign-in"></a>Проверьте работу маршрутизации и входа в AD FS
### <a name="routing-test"></a>Проверка маршрутизации
Проще всего проверить маршрутизацию удалось, попробуйте проверить связь с DNS-имя службы федерации, к компьютеру в каждом географическом регионе. В зависимости от выбранного метода маршрутизации, выбираться конечной точки будет отражаться в отображении ping. Например если вы выбрали маршрутизации производительности, затем наиболее приближены к ценам клиента будет связаться с конечной точкой. Ниже приведен снимок двух запросы проверки связи от двух разных регионах клиентских компьютеров, экрана и Западная часть США. 

![Проверка маршрутизации](./media/active-directory-adfs-in-azure-with-azure-traffic-manager/pingtest.png)

### <a name="ad-fs-sign-in-test"></a>Проверка входа в AD FS
Самый простой способ проверки AD FS — использовать страницу IdpInitiatedSignon.aspx. Чтобы иметь возможность сделать это, необходимо включить IdpInitiatedSignOn в свойствах AD FS. Выполните следующие действия, чтобы проверить установку AD FS

1. Запустите ниже командлет на сервере AD FS, с помощью PowerShell, чтобы включить. 
   Set-AdfsProperties -EnableIdPInitiatedSignonPage $true
2. Из любого внешнего компьютера доступа https://<yourfederationservicedns>/adfs/ls/IdpInitiatedSignon.aspx
3. Должна появиться страница AD FS, например ниже:
   
    ![Тестирование ADFS — запрос проверки подлинности](./media/active-directory-adfs-in-azure-with-azure-traffic-manager/adfstest1.png)
   
    и на успешного входа в систему, он будет отобразится сообщение об успешном выполнении как показано ниже:
   
    ![Тестирование ADFS — Успешная проверка подлинности](./media/active-directory-adfs-in-azure-with-azure-traffic-manager/adfstest2.png)
