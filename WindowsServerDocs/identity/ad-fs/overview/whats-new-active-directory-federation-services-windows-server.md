>Область применения. Windows Server 2019, Windows Server 2016

## <a name="whats-new-in-active-directory-federation-services-for-windows-server-2019"></a>Новые возможности в Active Directory Federation Services для Windows Server 2019 г.

### <a name="protected-logins"></a>Защищенные имена входа
Ниже приведен краткий обзор обновлений для защищенных имен входа, доступные в AD FS 2019.
- **Внешние поставщики проверки подлинности в качестве основного** -клиенты могут использовать как первый фактор проверки подлинности сторонних производителей и не предоставляют пароли как первый фактор. В случаях, когда поставщик проверки подлинности внешних может оказаться 2 факторов он может запросить многофакторной проверки Подлинности. 
- **Проверка пароля в качестве дополнительной проверки подлинности** -клиенты имеют возможность полностью поддерживается папки "Входящие" использовать пароль, только для дополнительной factor после пароль меньше параметр как первый фактор. Это улучшает взаимодействие клиентов с ADFS 2016, где клиентам приходилось скачать адаптер github, который поддерживается как есть. 
- **Подключаемые Framework модуля угроз** -клиенты теперь могут создавать свои собственные подключаемые в модулях для блокировки определенных типов запросов во время стадии предварительной проверки подлинности. Это упрощает для клиентов использовать интеллектуальные облачные средства, такие как защита идентификации для блокировки имена входа для пользователей, представляющих риск или опасных операций.
- **Усовершенствования ESL** -улучшает ESL QFE в 2016, добавив следующие возможности
    - Позволяет клиентам работать в режиме аудита при, защищенном с помощью функции «классический» блокировка экстрасети, доступно с версии 2012 R2 служб федерации Active Directory. В настоящее время 2016 пользователи щелкнут без защиты в режиме аудита. 
    - Позволяет пороговое значение независимых блокировки для знакомых расположениях. Это позволяет несколько экземпляров приложения, работающие с общей учетной записью службы смены паролей с минимально влияния. 

### <a name="additional-security-improvements"></a>Усовершенствования для обеспечения дополнительной безопасности
В AD FS 2019 доступны следующие усовершенствования для обеспечения дополнительной безопасности.
- **Удаленный PSH, используя имя входа смарт-карты** — может смарт-карт теперь использования удаленным клиентам подключиться к ADFS через PSH и использования, что для управления все PSH функции включают командлеты PSH несколькими узлами.
- **Настройки HTTP-заголовка** -клиентов можно настроить заголовки HTTP, генерируемые за время ответов служб федерации Active Directory. Сюда входят следующие заголовки
     - HSTS: Это содержимое, что конечные точки служб федерации Active Directory может использоваться только в конечных точках HTTPS для соответствующих браузера для принудительного применения
     - x-frame-options: Разрешает администраторам служб федерации Active Directory позволяют конкретных проверяющих сторон внедрить элементы IFRAME для страницы интерактивного входа в систему AD FS. Это следует использовать осторожно и только на узлах HTTPS. 
     - Будущие заголовка: Также можно настроить дополнительные заголовки в будущем. 

Дополнительные сведения см. в разделе [заголовков ответа HTTP, настройка безопасности с AD FS 2019](../../ad-fs/operations/customize-http-security-headers-ad-fs.md) 

### <a name="authenticationpolicy-capabilities"></a>Возможности проверки подлинности и политики
Следующие возможности проверки подлинности и политики находятся в AD FS 2019:
- **Укажите метод проверки подлинности для дополнительной проверки подлинности на RP** -клиенты могут теперь использовать правила, чтобы решить, какие дополнительного поставщика проверки подлинности для вызова для дополнительного поставщика проверки подлинности утверждений. Это полезно для вариантов использования 2
    - Клиенты переходят из одного дополнительного поставщика проверки подлинности в другой. Таким образом, как только они подключение пользователей к более новый поставщик проверки подлинности они могут использовать группы для управления, какие дополнительной проверки подлинности поставщик называется.
    - Клиентам требуются для определенного дополнительного поставщика проверки подлинности (например, сертификат) для определенных приложений. 
- **Ограничить только для приложений, которые ее требуют проверки подлинности устройства на основе TLS** -клиенты теперь можно ограничить TLS на основе проверки подлинности устройства только для приложений, выполняющих устройства условный доступ на основе клиента. Это позволяет предотвратить нежелательные запросы на проверку подлинности устройства (или сбои, если не удается обработать клиентское приложение) для приложений, которые не требуют проверки подлинности устройства на основе TLS.
- **Поддержка актуальности многофакторной проверки Подлинности** -AD FS теперь поддерживает возможность для повторно выполните 2-й коэффициент учетных данных на основе свежести 2-й коэффициент учетных данных. Это позволяет клиентам осуществлять начальная транзакция с 2 факторов и запрашивать второй фактор периодически. Эта возможность доступна только для приложений, которые можно указать дополнительный параметр в запросе и не является значением настраиваемого параметра в службах ADFS. Этот параметр поддерживается службой Azure AD, когда настроен параметр «Запомнить мои MFA на X дней» и флаг «supportsMFA» имеет значение true в параметры доверия федеративного домена в Azure AD.

 - **Поток/профиль OAuth устройства** -AD FS теперь поддерживает профиль устройства поток OAuth для выполнения входа на устройствах, у которых нет пользовательского интерфейса контактная зона для поддержки возможностей входа с широкими возможностями.  - **Исправление ошибки: Утверждение отправки x5t и kid** — это небольшие исправления ошибки. AD FS теперь Кроме того отправляет утверждение «kid» для обозначения Указание идентификатора ключа для проверки подписи. Ранее AD FS отправляется только это как утверждения «x5t».

### <a name="proof-key-for-code-exchange-pkce-support-for-oauth"></a>Ключ проверки для обмена кодом (PKCE) поддержка oAuth 
Открытый клиентов OAuth с помощью предоставления кода авторизации уязвимы для атак перехват кода авторизации.  Атаки также описан в RFC 7636. Чтобы избежать подобной атаки, AD FS в 2019 г. сервер поддерживает ключа проверки для обмена кодом (PKCE) для потоке предоставления кода авторизации OAuth. 
 
Чтобы использовать поддержки PKCE, эта спецификация добавляет дополнительные параметры авторизации OAuth 2.0 и в запросе маркера доступа.

![Proofkey](media/whats-new-in-active-directory-federation-services-for-windows-server-2016/adfs2019.png)

A. Клиент создает и записывает секрет с именем «code_verifier» и является производным преобразованные версии «t(code_verifier)» (называется «code_challenge»), который отправляется в OAuth 2.0 Authorization Request вместе с метода преобразования «t_m». 

Б. Конечная точка авторизации отвечает обычным образом, кроме записей «t(code_verifier)» и метода преобразования. 

В. Затем клиент отправляет код авторизации в Access Token запроса как обычно, но включает секретный код «code_verifier», созданный в (A). 

D. AD FS преобразовывает «code_verifier» и сравнивает его с «t(code_verifier)» из (B).  Доступ запрещен, если они не равны.

## <a name="eliminate-passwords-from-the-extranet"></a>Исключить пароли из экстрасети  
AD FS 2016 включает три новых параметра для входа в систему без использования паролей, позволяя организациям, во избежание риска сети влияет на получаемые от фишинга, потерянных или украденных пароли. 

### <a name="sign-in-with-azure-multi-factor-authentication"></a>Войдите с помощью многофакторная Идентификация Azure
AD FS 2016 основан на многофакторную проверку подлинности (MFA) возможности AD FS в Windows Server 2012 R2, позволяя вход, используя только код Azure MFA, не вводя имя пользователя и пароль.

* С помощью Azure MFA как основной метод аутентификации пользователю предлагается ввести свое имя пользователя и OTP код из приложения Azure Authenticator.  
* С помощью Azure MFA в качестве метода проверки подлинности-получателя или дополнительное пользователь предоставляет основной проверки подлинности учетные данные (с использованием встроенной проверки подлинности Windows, имя пользователя и пароля, смарт-карта или сертификат пользователя или устройства), а затем видит запрос для текста, голоса и OTP на основе имени входа в Azure MFA.  
* С помощью новых встроенных адаптера Azure MFA Установка и настройка для Azure MFA с AD FS не было проще.
* Организации могут воспользоваться преимуществами Azure MFA, без необходимости в локальном сервере Azure MFA.
* Azure MFA можно настроить для интрасети или экстрасети или как часть любую политику контроля доступа.

### <a name="password-less-access-from-compliant-devices"></a>Доступ без пароля устройств, соответствующих требованиям
AD FS 2016 основан на предыдущем возможность регистрации устройств, чтобы включить функцию входа и управления доступом на основе состояния соответствия устройства. Пользователи могут входить в систему с помощью учетных данных устройства и соответствия требованиям повторно вычисляется при изменении атрибутов устройства, таким образом, чтобы вы всегда можете убедитесь, что, применения политик.  Это позволяет политики, такие как

* Разрешить доступ только на устройствах, управляемых и соответствующих  
* Включение доступа к экстрасети только на устройствах, управляемых и соответствующих  
* Требовать многофакторную проверку подлинности для компьютеров, которые не являются управляемыми или не соответствующих политике  

AD FS предоставляет компонент в локальной среде политик условного доступа в гибридном сценарии. При регистрации устройств в Azure AD для условного доступа к облачным ресурсам, можно использовать удостоверение устройства для политик AD FS, а также.

### <a name="moving-from-ad-fs-in-windows-server-2012-r2-to-ad-fs-in-windows-server-2016-is-easier"></a>Перенос из AD FS в Windows Server 2012 R2 AD FS в Windows Server 2016 — проще  
Ранее переход на новую версию службы федерации Active Directory требуется Экспорт конфигурации из старой фермы и импорт в ферму новый, parallel.  

Теперь переходе с AD FS в Windows Server 2012 R2 на AD FS на Windows Server 2016 стала гораздо проще. Просто добавьте новый сервер Windows Server 2016 в ферму Windows Server 2012 R2 и фермы будет действовать на уровне поведения фермы Windows Server 2012 R2, поэтому он работает так же, как ферму Windows Server 2012 R2.  

Затем добавьте новые серверы Windows Server 2016 в ферме, проверки функциональности и удалите старые серверы из подсистемы балансировки нагрузки. После запуска всех узлах фермы Windows Server 2016 все готово для обновления на уровне поведения фермы до 2016 и приступить к использованию новых функций.  

Дополнительные сведения см. в разделе [обновление до AD FS в Windows Server 2016.](../../ad-fs/deployment/Upgrading-to-AD-FS-in-Windows-Server-2016.md)
