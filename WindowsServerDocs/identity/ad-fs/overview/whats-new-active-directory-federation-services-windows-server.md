# <a name="whats-new-in-active-directory-federation-services"></a>Новые возможности служб федерации Active Directory


## <a name="whats-new-in-active-directory-federation-services-for-windows-server-2019"></a>Новые возможности службы федерации Active Directory (AD FS) для Windows Server 2019

### <a name="protected-logins"></a>Защищенные имена входа
Ниже приведен краткий обзор обновлений защищенных имен входа, доступных в AD FS 2019.
- **Внешние поставщики проверки подлинности как первичные** клиенты теперь могут использовать сторонние продукты проверки подлинности в качестве первого фактора и не предоставляют пароли в качестве первого фактора. В случаях, когда внешний поставщик проверки подлинности может доказать два фактора, он может запрашивать MFA. 
- **Проверка пароля в качестве дополнительной проверки подлинности** . клиенты имеют полностью поддерживаемый параметр "Входящие" для использования пароля только для дополнительного фактора, если в качестве первого фактора используется параметр "меньше пароля". Это улучшит взаимодействие с клиентами из ADFS 2016, где клиентам пришлось загрузить адаптер GitHub, поддерживаемый как есть. 
- **Модуль оценки подключаемых рисков** . Теперь клиенты могут создавать собственные модули подключаемых модулей для блокировки определенных типов запросов на этапе предварительной проверки подлинности. Это упрощает для клиентов использование средств облачной аналитики, таких как защита идентификации, для блокировки имен входа для рискованных пользователей или рискованных транзакций.  Дополнительные сведения см. [в статье подключаемые модули сборки с моделью оценки рисков AD FS 2019](../../ad-fs/development/ad-fs-risk-assessment-model.md) . 
- **Улучшения есл** — улучшают есл QFE в 2016, добавив следующие возможности.
    - Позволяет клиентам работать в режиме аудита, пока защита с помощью классической функции блокировки экстрасети доступна с 2012 R2 ADFS. В настоящее время 2016 клиенты не будут иметь защиты в режиме аудита. 
    - Включает независимое пороговое значение блокировки для привычных расположений. Это позволяет нескольким экземплярам приложений, работающих с общей учетной записью службы, переключаться на пароли с минимальным количеством последствий. 

### <a name="additional-security-improvements"></a>Дополнительные улучшения безопасности
В AD FS 2019 доступны следующие дополнительные улучшения системы безопасности:
- **Удаленный командном PSH с использованием учетной записи смарт-карты** . клиенты теперь могут использовать SmartCards для удаленного подключения к ADFS через командном PSH и использовать для управления всеми функциями командном PSH, включая командлеты с несколькими узлами командном PSH.
- **Настройка HTTP-заголовка** . Теперь клиенты могут НАСТРАИВАТЬ заголовки HTTP, генерируемые во время ответов ADFS. Сюда входят следующие заголовки
     - HSTS. Это дает возможность использовать конечные точки ADFS только на конечных точках HTTPS для обеспечения соответствия браузеру, совместимому с
     - параметры x-Frame-: позволяет администраторам ADFS разрешать конкретным проверяющим сторонам внедрять элементы iFrame для интерактивных страниц входа ADFS. Его следует использовать с осторожностью и только на узлах HTTPS. 
     - Заголовок будущего: также можно настроить дополнительные заголовки будущих периодов. 

Дополнительные сведения см. [в статье Настройка заголовков ответа безопасности HTTP с помощью AD FS 2019](../../ad-fs/operations/customize-http-security-headers-ad-fs.md) . 

### <a name="authenticationpolicy-capabilities"></a>Возможности проверки подлинности и политики
Следующие возможности проверки подлинности и политики относятся к AD FS 2019:
- **Укажите метод проверки подлинности для дополнительной проверки подлинности на RP** . клиенты теперь могут использовать правила утверждений, чтобы решить, какой дополнительный поставщик проверки подлинности будет вызываться для дополнительного поставщика проверки Это полезно в двух вариантах использования.
    - Клиенты переходят от одного дополнительного поставщика проверки подлинности на другой. Таким образом, при подключении пользователей к более новым поставщикам проверки подлинности они могут использовать группы для управления дополнительными поставщиками проверки подлинности.
    - Клиентам требуются определенные дополнительные поставщики проверки подлинности (например, сертификат) для определенных приложений. 
- Ограничьте проверку подлинности **устройства на основе TLS только приложениями, которые нуждаются в ИТ** -клиентах. Теперь клиенты могут ограничивать проверки подлинности устройств на основе TLS только приложениями, выполняющими условный доступ на основе устройств. Это предотвращает любые нежелательные запросы на проверку подлинности устройства (или сбои, если клиентское приложение не может работать) для приложений, которые не нуждаются в проверке подлинности устройства на основе TLS.
- **Поддержка свежести MFA** . AD FS теперь поддерживает возможность повторного использования учетных данных второго фактора в зависимости от актуальности учетных данных второго фактора. Это позволяет клиентам выполнять начальную транзакцию с двумя факторами и периодически выдавать запрос на второй фактор. Эта возможность доступна только для приложений, которые могут предоставлять дополнительный параметр в запросе и не являются настраиваемыми параметрами в ADFS. Этот параметр поддерживается в Azure AD, если настроено "запомнить MFA в течение X дней", а флаг "supportsMFA" имеет значение true в параметрах доверия федеративного домена в Azure AD. 

### <a name="sign-in-sso-improvements"></a>Усовершенствования единого входа для входа
В AD FS 2019 были внесены следующие улучшения единого входа входа:

- Интерфейс [с разбивкой на страницы с помощью центрированной темы](../operations/AD-FS-paginated-sign-in.md) . Теперь ADFS перемещен в поток интерфейса с разбивкой на страницы, что позволяет ADFS проверять и обеспечивать более гладкий вход в систему. Теперь ADFS использует центрированный пользовательский интерфейс (вместо правой части экрана). Для согласования с этим интерфейсом может потребоваться использовать новые логотипы и фоновые образы. Это также отражает функции, предлагаемые в Azure AD.
- **Исправление ошибки: постоянное состояние единого входа для устройств Win10 при выполнении проверки подлинности PRT**   Это устраняет проблему, при которой состояние MFA не было сохранено при использовании проверки подлинности PRT для устройств Windows 10. Результатом этой проблемы было то, что конечные пользователи часто получают запрос на получение учетных данных 2-го фактора (MFA). Это исправление также обеспечивает единообразную работу при успешном выполнении проверки подлинности устройства через клиент TLS и через механизм PRT. 


### <a name="suppport-for-building-modern-line-of-business-apps"></a>Встроенная для создания современных бизнес-приложений
В AD FS 2019 добавлена поддержка создания современных бизнес-приложений.

 - **Поток или профиль устройства OAuth** . AD FS теперь поддерживает профиль потока устройства OAuth для выполнения имен входа на устройствах, не имеющих контактной области пользовательского интерфейса для поддержки расширенного входа в систему. Это позволяет пользователю завершить процедуру входа на другом устройстве. Эта функция необходима для Azure CLI работы в Azure Stack и может использоваться в других случаях. 
 - **Удаление параметра "Resource"** -AD FS теперь удалило требование указать параметр ресурса, который находится в строке с текущими спецификациями OAuth. Теперь клиенты могут предоставить идентификатор доверия проверяющей стороны в качестве параметра области в дополнение к запрошенным разрешениям. 
 - **Заголовки CORS в ответах AD FS** . Теперь клиенты могут создавать одностраничные приложения, позволяющие клиентским библиотекам JS проверять подпись id_token путем запроса ключей подписывания из документа OIDC discovery на AD FS. 
 - **Поддержка PKCE** . AD FS добавляет поддержку PKCE, чтобы обеспечить безопасный поток кода проверки подлинности в OAuth. Это добавляет дополнительный уровень безопасности в этот поток, чтобы предотвратить перехват кода и его воспроизведение с другого клиента. 
 - **Исправление ошибки: send x5t и ребенок** — это незначительное исправление ошибки. AD FS теперь также отправляет заявку "ребенок", чтобы обозначить указание идентификатора ключа для проверки подписи. Ранее AD FS отправили это утверждение "x5t".

### <a name="azure-ad-style-resource-specification-in-scope-parameter"></a>Спецификация ресурсов в стиле Azure AD в параметре scope 
Ранее AD FS требовалось, чтобы требуемый ресурс и область были в отдельном параметре любого запроса проверки подлинности. Например, типичный запрос OAuth будет выглядеть следующим образом: 7 **https:&#47;&#47;FS.contoso.com/ADFS/OAuth2/Authorize?</br>response_type = Code & client_id = клаимсксрайклиент & ресурс = urn: Microsoft:</br>ADFS: клаимсксрай & Scope = OAuth & redirect_uri =&#47;&#47;HTTPS: adfshelp.Microsoft.com/</br> клаимсксрай/токенреспонсе & Prompt = login**
 
С AD FS на сервере 2019 теперь можно передать значение ресурса, внедренное в параметр Scope. Это согласуется с тем, как одна из них может выполнять проверку подлинности в Azure AD. 

Параметр области можно упорядочить в виде списка с разделителями-пробелами, где каждая запись представляет собой структуру как ресурс или область. 

> [!NOTE]
> В запросе проверки подлинности можно указать только один ресурс. Если в запрос включено более одного ресурса, AD FS вернет ошибку, и проверка подлинности не будет выполнена. 

### <a name="proof-key-for-code-exchange-pkce-support-for-oauth"></a>Ключ подтверждения для поддержки обмена кодом (PKCE) для oAuth 
Общедоступные клиенты OAuth, использующие предоставление кода авторизации, уязвимы для атаки перехвата кода авторизации.  Атака хорошо описана в RFC 7636. Чтобы устранить эту атаку, AD FS на сервере 2019 поддерживает ключ проверки подлинности для обмена кодом (PKCE) для потока предоставления кода авторизации OAuth. 
 
Чтобы использовать поддержку PKCE, эта спецификация добавляет дополнительные параметры к запросам авторизации OAuth 2,0 и маркеру доступа.

![пруфкэй](media/whats-new-in-active-directory-federation-services-for-windows-server-2016/adfs2019.png)

А) Клиент создает и записывает секрет с именем "code_verifier" и получает преобразованную версию "t (code_verifier)" (называемую "code_challenge"), которая отправляется в запросе авторизации OAuth 2,0 вместе с методом преобразования "t_m". 

Б) Конечная точка авторизации реагирует обычным образом, но записывает "t (code_verifier)" и метод преобразования. 

В. Затем клиент отправляет код авторизации в запросе маркера доступа обычным образом, но включает секрет "code_verifier", созданный в (а). 

Г. AD FS преобразует "code_verifier" и сравнивает его с "t (code_verifier)" из (B).  Если они не равны, доступ запрещен.

 [ADFS в Windows Server 2012 или 2012 R2](https://technet.microsoft.com/library/hh831502.aspx) и [AD FS 2,0](https://technet.microsoft.com/library/adfs2.aspx)  

 Службы федерации Active Directory (AD FS) обеспечивает управление доступом и единый вход для разнообразных приложений, включая Office 365, облачные приложения SaaS и приложения в корпоративной сети.  
* Для ИТ – организации вы предоставляете возможность входа и контроля доступа для современных и устаревших приложений локально и в облаке на основе того же набора учетных данных и политик.

В этой статье описаны новые возможности AD FS в Windows Server 2016 (AD FS 2016).  

## <a name="eliminate-passwords-from-the-extranet"></a>Удаление паролей из экстрасети  
AD FS 2016 включает три новых параметра для входа без паролей, позволяя организациям избежать риска компрометации сети от фишинговых, потерянных или украденных паролей. 

### <a name="sign-in-with-azure-multi-factor-authentication"></a>Вход с помощью многофакторной идентификации Azure
AD FS 2016 строится на основе возможностей многофакторной идентификации (MFA) AD FS в Windows Server 2012 R2, разрешая вход с использованием только кода Azure MFA, не вводя имя пользователя и пароль.

* При использовании Azure MFA в качестве основного метода проверки подлинности пользователю предлагается ввести имя пользователя и код OTP из приложения Azure Authenticator.  
* В Azure MFA в качестве дополнительного или дополнительного метода проверки подлинности пользователь предоставляет основные учетные данные для проверки подлинности (с использованием встроенной проверки подлинности Windows, имени пользователя и пароля, смарт-карты или сертификата пользователя или устройства), а затем видит запрос на ввод текста, для входа на основе голоса или OTP на базе Azure MFA.  
* Благодаря новому встроенному адаптеру Azure MFA Настройка и настройка для Azure MFA с AD FS никогда не были более простыми.
* Организации могут воспользоваться преимуществами Azure MFA без необходимости использовать локальный сервер Azure MFA.
* Azure MFA можно настроить для интрасети или экстрасети или в рамках любой политики управления доступом.

### <a name="password-less-access-from-compliant-devices"></a>Доступ без пароля с соответствующих устройств
AD FS 2016 построены на предыдущих возможностях регистрации устройств, чтобы включить вход и контроль доступа на основе состояния соответствия устройства. Пользователи могут войти в систему с помощью учетных данных устройства, и при изменении атрибутов устройства соответствие требованиям будет проверено, чтобы всегда гарантировать принудительное применение политик.  Это позволяет включать такие политики, как

* Разрешить доступ только с управляемых и/или соответствующих устройств  
* Разрешить доступ к экстрасети только с управляемых и/или соответствующих устройств  
* Требовать многофакторную проверку подлинности для компьютеров, которые не являются управляемыми или не соответствуют требованиям  

AD FS предоставляет локальный компонент политик условного доступа в гибридном сценарии. При регистрации устройств в Azure AD для условного доступа к облачным ресурсам удостоверение устройства можно также использовать для AD FS политик.

### <a name="sign-in-with-windows-hello-for-business"></a>Вход с помощью Windows Hello для бизнеса  

> [!NOTE]
> В настоящее время Google Chrome и [Новый Microsoft ребр](https://www.microsoft.com/edge?form=MB110A&OCID=MB110A) , основанные на браузерах проектов с открытым исходным кодом Chromium, не поддерживаются для единого входа (SSO) на основе браузера с помощью Microsoft Windows Hello для бизнеса. Используйте Internet Explorer или более старую версию Microsoft ребр.  

Устройства Windows 10 представляют собой Windows Hello и Windows Hello для бизнеса, заменяя пароли пользователей строгими учетными данными пользователя, защищенными с помощью жеста пользователя (ПИН-код, биометрический жест, например отпечаток пальца или распознавания лиц). AD FS 2016 поддерживает эти новые возможности Windows 10, чтобы пользователи могли входить в AD FS приложения из интрасети или экстрасети без необходимости указывать пароль.

### <a name="modern-authentication"></a>Современная проверка подлинности
AD FS 2016 поддерживает новейшие современные протоколы, которые обеспечивают лучшее взаимодействие с пользователем для Windows 10, а также новейшие устройства и приложения iOS и Android.  

Дополнительные сведения см. в разделе [сценарии AD FS для разработчиков](../../ad-fs/overview/ad-fs-openid-connect-oauth-flows-scenarios.md) .  


### <a name="configure-access-control-policies-without-having-to-know-claim-rules-language"></a>Настройка политик контроля доступа без необходимости знания языка правил утверждений  
Ранее AD FS администраторам пришлось настраивать политики с помощью языка правил утверждений AD FS, что усложняет настройку и поддержку политик. С помощью политик управления доступом администраторы могут применять встроенные шаблоны для применения общих политик, таких как
* Разрешите доступ только через интрасеть
* Разрешить всем и требовать MFA из экстрасети
* Разрешить всем и требовать MFA из определенной группы

Шаблоны можно легко настроить с помощью процесса, управляемого мастером, чтобы добавить исключения или дополнительные правила политики и применить к одному или нескольким приложениям для обеспечения единообразного применения политик.

Дополнительные сведения см [. в разделе политики управления доступом в AD FS.](../../ad-fs/operations/Access-Control-Policies-in-AD-FS.md)  

### <a name="enable-sign-on-with-non-ad-ldap-directories"></a>Включение входа с помощью каталогов LDAP, не относящихся к AD  
Многие организации имеют сочетание Active Directory и сторонних каталогов. С добавлением поддержки AD FS для проверки подлинности пользователей, хранящихся в каталогах, совместимых с LDAP V3, AD FS теперь можно использовать для:
* Пользователи в каталогах, совместимых со сторонними LDAP V3
* Пользователи в Active Directory лесах, к которым не настроено Active Directory двустороннее доверие
* Пользователи в службы Active Directory облегченного доступа к каталогам (AD LDS)

Дополнительные сведения см. в разделе [настройка AD FS для проверки подлинности пользователей, хранящихся в каталогах LDAP.](../../ad-fs/operations/Configure-AD-FS-to-authenticate-users-stored-in-LDAP-directories.md)  

## <a name="better-sign-in-experience"></a>Улучшение процесса входа
### <a name="customize-sign-in-experience-for-ad-fs-applications"></a>Настройка интерфейса входа для AD FS приложений  
Мы слышали о том, что возможность настройки входа для каждого приложения будет очень удобной, особенно для организаций, предоставляющих вход для приложений, представляющих несколько разных компаний или торговых марок.  

Ранее AD FS в Windows Server 2012 R2 поддерживали общую процедуру входа для всех приложений проверяющей стороны с возможностью настройки подмножества содержимого на основе текста для каждого приложения. Windows Server 2016 позволяет настраивать не только сообщения, но и изображения, эмблему и веб-тему для каждого приложения. Кроме того, можно создавать новые пользовательские веб-темы и применять их для каждой проверяющей стороны.  

Дополнительные сведения см. в разделе [AD FS Настройка входа пользователя.](../../ad-fs/operations/AD-FS-user-sign-in-customization.md)  



## <a name="manageability-and-operational-enhancements"></a>Возможности управления и улучшения в работе  
В следующем разделе описаны усовершенствованные рабочие сценарии, появившиеся в службы федерации Active Directory (AD FS) в Windows Server 2016.  

### <a name="streamlined-auditing-for-easier-administrative-management"></a>Упрощенный аудит для упрощенного управления администраторами  
В AD FS для Windows Server 2012 R2 существовало много событий аудита, созданных для одного запроса, и соответствующие сведения о действиях входа в систему или выдаче маркера либо отсутствуют (в некоторых версиях AD FS), либо распределены по нескольким событиям аудита. По умолчанию события аудита AD FS отключены из-за их подробной природы.  
В выпуске AD FS 2016 аудит стал более оптимизированным и менее подробным.  

Дополнительные сведения см. в разделе [улучшения аудита для AD FS в Windows Server 2016.](../../ad-fs/technical-reference/auditing-enhancements-to-ad-fs-in-windows-server.md)  

### <a name="improved-interoperability-with-saml-20-for-participation-in-confederations"></a>Улучшенное взаимодействие с SAML 2,0 для участия в конфедератионс  
AD FS 2016 содержит дополнительную поддержку протокола SAML, включая поддержку импорта доверий на основе метаданных, содержащих несколько сущностей. Это позволяет настроить AD FS для участия в конфедератионс, например нестандартная Федерация и другие реализации, согласованные со стандартом егов 2,0.  

Дополнительные сведения см [. в статье улучшенное взаимодействие с SAML 2,0.](../../ad-fs/operations/Improved-interoperability-with-SAML-2.0.md)  

### <a name="simplified-password-management-for-federated-o365-users"></a>Упрощенное управление паролями для федеративных пользователей Office 365  
Можно настроить службы федерации Active Directory (AD FS) (AD FS) для отправки утверждений об истечении срока действия пароля в отношения доверия проверяющей стороны (приложения), защищенные AD FS. Использование этих утверждений зависит от приложения. Например, в качестве проверяющей стороны в Office 365 были реализованы обновления для Exchange и Outlook, позволяющие уведомлять федеративных пользователей о сроках их действия в скором времени.  

Дополнительные сведения см [. в разделе настройка AD FS для отправки утверждений об истечении срока действия пароля.](../../ad-fs/operations/Configure-AD-FS-to-Send-Password-Expiry-Claims.md)  

### <a name="moving-from-ad-fs-in-windows-server-2012-r2-to-ad-fs-in-windows-server-2016-is-easier"></a>Переход с AD FS в Windows Server 2012 R2 на AD FS в Windows Server 2016 проще  
Ранее при переходе на новую версию AD FS требовалось экспортировать конфигурацию из старой фермы и импортировать ее в новую, параллельную ферму.  

Теперь переход с AD FS на Windows Server 2012 R2 на AD FS в Windows Server 2016 стал намного проще. Просто добавьте новый сервер Windows Server 2016 в ферму Windows Server 2012 R2, и ферма будет работать на уровне поведения фермы Windows Server 2012 R2, поэтому она выглядит и работает так же, как и ферма Windows Server 2012 R2.  

Затем добавьте в ферму новые серверы Windows Server 2016, проверьте функциональность и удалите старые серверы из балансировщика нагрузки. После того как все узлы фермы работают под Windows Server 2016, вы можете обновить уровень поведения фермы до 2016 и начать использовать новые функции.  

Дополнительные сведения см [. в статье обновление до AD FS в Windows Server 2016.](../../ad-fs/deployment/Upgrading-to-AD-FS-in-Windows-Server-2016.md)
