# <a name="whats-new-in-active-directory-federation-services-for-windows-server-2016"></a>Новые возможности Active Directory Federation Services для Windows Server 2016

>Область применения: Windows Server 2016

## <a name="whats-new-in-active-directory-federation-services-for-windows-server-2016"></a>Новые возможности Active Directory Federation Services для Windows Server 2016   
Если вы ищете сведения о предыдущих версиях Служб федерации Active Directory, см. в следующих статьях:  
 [Службы федерации Active Directory в Windows Server 2012 или 2012 R2](https://technet.microsoft.com/library/hh831502.aspx) и [AD FS 2.0](https://technet.microsoft.com/library/adfs2.aspx)  

 Службы федерации Active Directory предоставляет управление доступом и единого входа на на множестве различных приложений, включая Office 365, облачных приложений SaaS и приложений в корпоративной сети.  
* ИТ-организации позволяет предоставить входа и управления доступом для современных и старых приложений, на локальном компьютере и в облаке, на основе того же набора учетных данных и политики.    
* Для пользователей он предоставляет безупречное вход с использованием учетных данных же и привычный учетной записи.  
* Для разработчиков она предоставляет простой способ проверки подлинности, удостоверения live в каталоге организации, чтобы сосредоточить можно внимание на приложения, не проверки подлинности или удостоверений пользователей.  

В этой статье описаны новые возможности AD FS в Windows Server 2016 (AD FS 2016).  

## <a name="eliminate-passwords-from-the-extranet"></a>Исключить из экстрасети паролей  
AD FS 2016 включает три новые параметры для входа в систему без паролей, позволяя организациям избежать сети нарушить из phished утечками или кражи паролей. 

### <a name="sign-in-with-azure-multi-factor-authentication"></a>Войдите в систему многофакторной идентификации Azure
AD FS 2016 основан на многофакторной проверки подлинности (MFA) возможности AD FS в Windows Server 2012 R2, позволяя входа с использованием только кодом Azure MFA, без необходимости ввода имени пользователя и пароля.

* С помощью многофакторной проверки Подлинности Azure в качестве метода основной проверки подлинности пользователю предлагается ввести свое имя пользователя и код OTP из приложение Azure Authenticator.  
* С помощью многофакторной проверки Подлинности Azure в качестве метода дополнительный или дополнительной проверки подлинности пользователь вводит основной проверки подлинности учетных данных (с помощью встроенной проверки подлинности Windows, имя пользователя и пароль, смарт-карты или сертификата пользователя или устройства), а затем видит командную строку для текста, голосовых команд, или OTP на основе входа Azure MFA.  
* Новые встроенные адаптер Azure MFA установки и настройки для Azure MFA с AD FS еще никогда не было проще.
* Организации могут воспользоваться преимуществом многофакторной проверки подлинности Azure без необходимости локальным сервером Azure MFA.
* Можно настроить Azure MFA для интрасети или экстрасети или как часть любой политики управления доступом.

Дополнительные сведения об Azure MFA с AD FS
*  [Настройка AD FS 2016 и Azure MFA](https://docs.microsoft.com/windows-server/identity/ad-fs/operations/configure-ad-fs-and-azure-mfa)  

### <a name="password-less-access-from-compliant-devices"></a>Пароль доступ из совместимые устройства
AD FS 2016 основано на предыдущих возможностей регистрации устройств для разрешения входа и управления доступом на основе состояния соответствия устройства. Пользователи могут войти в систему с помощью учетных данных устройства и соответствие требованиям повторно оценивается при изменении атрибутов устройства, чтобы вы всегда можете убедитесь, что политики применялись.  Это позволяет политик, таких как

* Разрешить доступ только с устройств, управляемых или совместимые  
* Включение доступа к экстрасети только с устройств, управляемых или совместимые  
* Требовать многофакторную проверку подлинности для компьютеров, которые не являются управляемые или не соответствует требованиям  

AD FS обеспечивает компонент на локальных политик условного доступа в гибридном сценарии. При регистрации устройства с Azure AD для условного доступа к облачным ресурсам, удостоверения устройства можно использовать для Служб федерации Active Directory политик также.

### <a name="sign-in-with-windows-hello-for-business"></a>Войдите в систему Windows Hello для бизнеса   
Устройства с Windows 10 представлено Windows Hello и Windows Hello для бизнеса, заменив пароли пользователей надежный привязкой устройства учетные данные защищены жест пользователя (ПИН-код, биометрического жеста, такого как отпечатков пальцев или распознавание лиц). AD FS 2016 поддерживает эти новые эти новые возможности Windows 10, чтобы пользователи могут входить в AD FS приложениям из интрасети или экстрасети без необходимости указать пароль.

## <a name="secure-access-to-applications"></a>Безопасный доступ к приложениям

### <a name="modern-authentication"></a>Современные проверки подлинности
AD FS 2016 поддерживает последние современных протоколы, предоставляющие улучшает взаимодействие с пользователем для Windows 10, а также последние iOS и Android устройств и приложений.  

Дополнительные сведения см. [сценарии AD FS для разработчиков](../../ad-fs/overview/AD-FS-Scenarios-for-Developers.md)  


### <a name="configure-access-control-policies-without-having-to-know-claim-rules-language"></a>Настройка политики управления доступом, не зная языка правил утверждений  
Ранее, AD FS администраторам приходилось настраивать политики, с помощью языка правил утверждений Служб федерации Active Directory, что усложняет для настройки и обслуживания политик. С помощью политик управления доступа администраторы могут использовать встроенные шаблоны, для применения распространенных политик, таких как
* Разрешить доступ только к интрасети
* Разрешить всем пользователям и требует многофакторной проверки Подлинности из внешний доступ
* Разрешить всем пользователям и требует многофакторной проверки Подлинности из определенной группы

Шаблоны легко настроить с помощью мастера управляемый процесс Добавление исключений или правила Дополнительные политики и могут применяться для одного или нескольких приложений, для применения согласованного политики.

Дополнительные сведения см. [политики управления доступом в AD FS.](../../ad-fs/operations/Access-Control-Policies-in-AD-FS.md)  

### <a name="enable-sign-on-with-non-ad-ldap-directories"></a>Разрешения входа с каталогов LDAP, не являющихся AD  
Многие организации имеют сочетание Active Directory и каталогов сторонних производителей. С добавлением поддержки AD FS для проверки подлинности пользователей, хранящихся в каталогах несоответствия v3 LDAP для AD FS можно использовать для:
* Пользователи в сторонних разработчиков, совместимые каталогов LDAP v3
* Пользователи в лесах Active Directory, для которых не настроена двустороннее доверие Active Directory
* Пользователей в службах Active Directory облегченного доступа к каталогам (AD LDS)

Дополнительные сведения см. [Configure AD FS для проверки подлинности пользователей, хранящихся в каталогах LDAP.](../../ad-fs/operations/Configure-AD-FS-to-authenticate-users-stored-in-LDAP-directories.md)  

## <a name="better-sign-in-experience"></a>Лучше входа в систему
### <a name="customize-sign-in-experience-for-ad-fs-applications"></a>Настройка журнала в взаимодействие приложений Служб федерации Active Directory  
Мы слышали от вас, что позволяет настраивать процесс входа в систему для каждого приложения будет улучшения отлично удобство использования, особенно для организаций, которые предоставляют входа на для приложений, которые представляют несколько разных компаний или торговых марок.  

Ранее AD FS в Windows Server 2012 R2 предоставляются общие входа на взаимодействие для всех приложений проверяющей стороны, с возможностью настройки подмножество текст на основе содержимого каждого приложения. Windows Server 2016 вы можете настроить не только сообщения, но образов, эмблемы и веб-темы каждого приложения. Кроме того, можно создать пользовательские веб-темы и применять их каждой проверяющей стороной.  

Дополнительные сведения см. [настройки входа пользователя AD FS.](../../ad-fs/operations/AD-FS-user-sign-in-customization.md)  



## <a name="manageability-and-operational-enhancements"></a>Управляемость и рабочей улучшения  
Ниже приведена улучшенные рабочих сценариев, которые появились со службами федерации Active Directory в Windows Server 2016.  

### <a name="streamlined-auditing-for-easier-administrative-management"></a>Упрощенная аудита для упрощении администрирования управления  
В AD FS в Windows Server 2012 R2 были многочисленные события аудита, созданные для одного запроса и соответствующих информация о действиях выдачи входа в систему или маркера либо отсутствуют (в некоторых версиях AD FS) или распределены на несколько событий аудита. По умолчанию AD FS события аудита из-за своей природе verbose отключены.  
В выпуске AD FS 2016 аудит стала более простой и менее подробный.  

Дополнительные сведения см. [усовершенствования аудита в AD FS в Windows Server 2016.](../../ad-fs/technical-reference/auditing-enhancements-to-ad-fs-in-windows-server.md)  

### <a name="improved-interoperability-with-saml-20-for-participation-in-confederations"></a>Улучшенные возможности взаимодействия с SAML 2.0 для участия в confederations  
AD FS 2016 содержит дополнительные поддержку протокола SAML, включая поддержку Импорт отношений доверия на основе метаданных, содержащий несколько объектов. Это позволяет настроить AD FS для участия в confederations, например InCommon федерации и в других реализациях соблюдения eGov стандарт 2.0.  

Дополнительные сведения см. [улучшена взаимодействия с SAML 2.0.](../../ad-fs/operations/Improved-interoperability-with-SAML-2.0.md)  

### <a name="simplified-password-management-for-federated-o365-users"></a>Управление упрощенного пароль для федеративных пользователей Office 365  
Можно настроить службы федерации Active Directory (AD FS) для отправки утверждений об истечении срока действия пароля для доверия с проверяющей стороной (приложения), защищенные с помощью AD FS. Как используются эти утверждения зависит от приложения. Например с Office 365 как проверяющей стороной, обновления были реализованы в Exchange и Outlook, чтобы уведомить федеративных пользователей скоро к--срок действия паролей.  

Дополнительные сведения см. [Configure AD FS для отправки утверждений об истечении срока действия пароля.](../../ad-fs/operations/Configure-AD-FS-to-Send-Password-Expiry-Claims.md)  

### <a name="moving-from-ad-fs-in-windows-server-2012-r2-to-ad-fs-in-windows-server-2016-is-easier"></a>Переход с AD FS в Windows Server 2012 R2 в AD FS в Windows Server 2016 проще  
Ранее переход на новую версию Служб федерации Active Directory требуется Экспорт конфигурации из старой ферме и затем импортировать на новый, параллельных фермы.  

Теперь переход от AD FS Windows Server 2012 R2 AD FS в Windows Server 2016 стало гораздо проще. Просто добавьте новый сервер Windows Server 2016 в Windows Server 2012 R2 ферму и фермы будет действовать на уровне фермы поведение Windows Server 2012 R2, поэтому он выглядит и ведет себя так же, как ферме Windows Server 2012 R2.  

Добавьте новые серверы Windows Server 2016 в ферме, проверьте работоспособность и удалять старые серверы из подсистемы балансировки нагрузки. После всех узлах фермы под управлением Windows Server 2016, вы готовы обновление на уровне фермы поведение до 2016 и начать использовать новые функции.  

Дополнительные сведения см. [обновление до AD FS в Windows Server 2016.](../../ad-fs/deployment/Upgrading-to-AD-FS-in-Windows-Server-2016.md)
