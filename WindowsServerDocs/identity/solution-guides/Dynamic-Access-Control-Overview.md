# <a name="dynamic-access-control-overview"></a>Обзор динамического контроля доступа

>Область применения: Windows Server 2012 R2, Windows Server 2012

В этой обзорной статье для ИТ-специалистов описывается динамический контроль доступа и связанные с ним элементы, которые были представлены в Windows Server 2012 и Windows 8.  
  
На основе домена динамический контроль доступа позволяет администраторам применять разрешения управления доступом и ограничения, основываясь на четко определенных правилах, которые включают конфиденциальность ресурсов, работу или роль пользователя и конфигурацию устройства, которое используется для доступа к этим ресурсам.  
  
Например пользователь может выдаваться разные разрешения при доступе к ресурсу с офисного компьютера и с портативного компьютера через виртуальную частную сеть. Либо доступ может быть разрешен, только если устройство соответствует требованиям безопасности, определенным администраторами сети. Если используется динамический контроль доступа, разрешения пользователя меняются динамически, без дополнительного вмешательства администратора Если должность или роль пользователя меняется (что меняет атрибуты учетной записи пользователя в AD DS).  
  
Динамический контроль доступа не поддерживается в операционных системах Windows, предшествовавших Windows Server 2012 и Windows 8. Если динамический контроль доступа настроен в средах с поддерживаемых и неподдерживаемых версий Windows, изменения будут применяться только в поддерживаемых версиях.

### <a name="BKMK_Rules"></a>Централизованные правила доступа  
Централизованное правило доступа — это выражение правил авторизации, которые могут содержать один или несколько условий, затрагивающих группы пользователей, утверждения пользователей, утверждений устройств и свойств ресурсов. Несколько централизованных правил доступа можно объединить в централизованную политику доступа.  
  
Если для домена определены одно или несколько централизованных правил доступа, администраторы файл ресурсов общего доступа могут сопоставлять определенные правила с определенными ресурсами и бизнес-требований.  
  
### <a name="BKMK_Policies"></a>Централизованные политики доступа  
Централизованные политики доступа — это политики авторизации, содержащие условные выражения. Например предположим, что в организации имеются бизнес-требования, чтобы ограничить доступ к сведениями (PII) в файлах только владельцу файла и сотрудникам отдела персоналом (ч), которым разрешено просматривать Персональные данные. Эта политика организации применяется к файлам личные сведения, где бы они ни располагались на файловых серверах организации. Для реализации этой политики организация должна иметь возможность:  
  
-   Идентифицировать и пометить файлы, содержащие персональные данные.  
  
-   Идентифицировать группу сотрудников отдела Кадров, которой разрешен просмотр Персональных данных.  
  
-   Добавить централизованную политику доступа к централизованному правилу доступа и применения правила централизованного доступа ко всем файлам, содержащим персональные данные, где бы они ни располагались на файловых серверах в организации.  
  
Централизованные политики доступа в качестве зонтиками все безопасности, которые организация применяет его серверах. Эти политики дополняют (но не заменяют) политики локального доступа или списки управления доступом на уровне пользователей (DACL), применяемые к файлам и папкам.  
  
### <a name="BKMK_Claims"></a>Утверждения  
Утверждение — это уникальная деталь информации о пользователя, устройства или ресурса, которое было опубликовано с помощью контроллера домена. Должность пользователя, классификация отделом файла или состояние работоспособности компьютера, допустимые примеры утверждения. Объект может включать более одного утверждения, и любое сочетание утверждений можно использовать для авторизации доступа к ресурсам. В поддерживаемых версиях Windows доступны следующие типы утверждений:  
  
-   **Утверждения пользователей** атрибуты Active Directory, связанные с конкретным пользователем.  
  
-   **Утверждения устройств** атрибуты Active Directory, связанные с конкретным компьютером.  
  
-   **Атрибуты ресурса** глобальные свойства ресурсов, отмеченные для использования в решениях авторизации и публикуемые в Active Directory.  
  
Утверждения позволяют администраторам давать точные организации или предприятия уровне инструкции, касающиеся пользователей, устройств и ресурсов, которые можно включать в выражения, правила и политики.  
  
### <a name="BKMK_Expressions2"></a>Выражения  
Условные выражения — это усовершенствование управления доступом, разрешающее или запрещающее доступ к ресурсам при выполнении определенных условий, например, членство в группе, расположение или состояние безопасности устройства. Этими выражениями можно управлять через диалоговое окно дополнительных параметров безопасности редактора ACL или редактор централизованных правил доступа в Active Directory администрирования Center (ADAC).  
  
Выражения помогают администраторам управлять доступом к конфиденциальным ресурсам с помощью гибких условий в все более сложных бизнес-средах.  
  
### <a name="BKMK_Permissions2"></a>Предложенные разрешения  
Предложенные разрешения позволяют администратору более точно моделировать результаты потенциальных изменений настроек управления доступом, не изменяя их до.  
  
Прогнозирование действительного доступа к ресурсу помогает планировать и настраивать разрешения для этих ресурсов перед их реализацией.  
  
## <a name="additional-changes"></a>Дополнительные изменения  
Ниже перечислены дополнительные изменения в поддерживаемых версиях Windows, поддерживающие динамический контроль доступа:  
  
### <a name="support-in-the-kerberos-authentication-protocol-to-reliably-provide-user-claims-device-claims-and-device-groups"></a>Поддержку протокола проверки подлинности Kerberos для надежного предоставления утверждений пользователей и устройств, групп устройств.  
По умолчанию устройства под управлением любой поддерживаемой версии Windows могут обрабатывать билеты Kerberos, связанные с динамическим контролем доступа, которые включают данные, необходимые для комплексной проверки подлинности. Контроллеры доменов, могут выдавать и реагировать на билеты Kerberos с помощью комплексной сведения, относящиеся к проверки подлинности. Когда домен настраивается на распознавание динамического контроля доступа, устройства получают утверждения от контроллеров домена при начальной проверке подлинности и билеты комплексной проверки подлинности при подаче запросов на билет службы. Комплексная проверка подлинности ведет к созданию маркера доступа, включающего удостоверение пользователя и устройства для ресурсов, признающих динамический контроль доступа.  
  
### <a name="support-for-using-the-key-distribution-center-kdc-group-policy-setting-to-enable-dynamic-access-control-for-a-domain"></a>Поддержка использования ключей (KDC) групповой политики центра распространения для обеспечения динамического контроля доступа в домене.  
Каждый контроллер домена должен быть же Настройка политики административных шаблонов, расположенный в **компьютера Конфигурация компьютера\Политики\Административные шаблоны\система\kdc\поддержка динамического контроля, доступа и защиты Kerberos**.  
  
### <a name="support-for-using-the-key-distribution-center-kdc-group-policy-setting-to-enable-dynamic-access-control-for-a-domain"></a>Поддержка использования ключей (KDC) групповой политики центра распространения для обеспечения динамического контроля доступа в домене.  
Каждый контроллер домена должен быть же Настройка политики административных шаблонов, расположенный в **компьютера Конфигурация компьютера\Политики\Административные шаблоны\система\kdc\поддержка динамического контроля, доступа и защиты Kerberos**.  
  
### <a name="support-in-active-directory-to-store-user-and-device-claims-resource-properties-and-central-access-policy-objects"></a>Поддержка в Active Directory для хранения утверждений пользователей и устройств, свойств ресурсов и объектов централизованных политик доступа.  
  
### <a name="support-for-using-group-policy-to-deploy-central-access-policy-objects"></a>Поддержка использования групповой политики для развертывания объектов централизованных политик доступа.  
Следующий параметр групповой политики позволяет развертывать объекты централизованной политики доступа на файловых серверах в организации: **компьютера Конфигурация Windows\Параметры безопасности\Файловая система\централизованная политика доступа**.  
  
### <a name="support-for-claims-based-file-authorization-and-auditing-for-file-systems-by-using-group-policy-and-global-object-access-auditing"></a>Поддержка авторизации файлов на основе утверждений и аудита для файловых систем с помощью групповой политики и аудит доступа к глобальным объектам  
Необходимо включить поэтапный аудит централизованной политики доступа для аудита реального доступа, предоставляемого централизованной политикой доступа с помощью предложенных разрешений. Настроить этот параметр для компьютера в разделе **конфигурация расширенной политики аудита** в **параметры безопасности** из объекта групповой политики (GPO). После настройки параметров безопасности в объекте групповой Политики, можно развернуть объект групповой Политики к компьютерам в сети.  
  
### <a name="support-for-transforming-or-filtering-claim-policy-objects-that-traverse-active-directory-forest-trusts"></a>Поддержка преобразования или фильтрации объектов политики утверждений, просматривающих доверия лесов Active Directory  
Можно фильтровать или преобразовывать входящие и исходящие утверждения, просматривающие доверие леса. Существует три основных сценария фильтрации и преобразования утверждений.  
  
-   **Фильтрация на основе значения** фильтры могут быть основаны на значении утверждения. Это позволяет доверенному лесу предотвратить отправку утверждений с определенными значениями доверяющему лесу. Контроллеры доменов в доверяющих лесах могут использовать фильтрацию на основе значения для защиты от несанкционированного повышения привилегий, фильтруя входящие утверждения с определенными значениями из доверенного леса.  
  
-   **Фильтрация на основе типов утверждений** фильтры, основанные на типе утверждения, а не значение утверждения. Тип утверждения определяется по имени утверждения. Вы используете фильтрации в доверенном лесу на основе типов утверждений, и она не позволяет Windows отправлять утверждения, раскрывающие информацию доверяющему лесу.  
  
-   **Преобразование на основе типов утверждений** изменяет утверждение перед отправкой его намеченной цели. Преобразование на основе типов утверждений используется в доверенном лесу для обобщения известного утверждения, содержащего конкретную информацию. Преобразования можно использовать для обобщения типа утверждения и значение утверждения.  
  
## <a name="software-requirements"></a>Требования к программному обеспечению  
Поскольку утверждения и комплексную проверку подлинности для динамического контроля доступа требуют расширений проверки подлинности Kerberos, у домена, поддерживающего динамический контроль доступа должен иметь достаточно контроллеров домена под управлением поддерживаемых версий Windows для поддержки проверки подлинности Kerberos поддержкой контроля динамический доступ клиентов. По умолчанию устройства должны применять контроллеры доменов на других сайтах. Если не такие контроллеры домена недоступны, проверка подлинности завершится ошибкой. Таким образом должен поддерживать один из следующих условий:  
  
-   Во всех доменах, поддерживающих динамический контроль доступа должен иметь достаточно контроллеров домена под управлением поддерживаемых версий Windows Server для поддержки проверки подлинности всех устройств под управлением поддерживаемых версий Windows или Windows Server.  
  
-   Устройства под управлением поддерживаемых версий Windows, которые не защищают ресурсы с помощью утверждений или составных удостоверений, следует отключить поддержку динамического контроля доступа протоколом Kerberos.  
  
В доменах, поддерживающих утверждения пользователей каждый контроллер домена под управлением поддерживаемых версий Windows server должны быть настроены соответствующие параметры для поддержки утверждений и комплексной проверки подлинности и защиты Kerberos. Настройте параметры в политике административного шаблона KDC следующим образом:  
  
-   **Всегда предоставлять утверждения** используйте этот параметр, если все контроллеры домена работают под управлением поддерживаемых версий Windows Server. Кроме того установите режим работы домена Windows Server 2012 или выше.  
  
-   **Поддерживается** при использовании этого параметра отслеживайте контроллеры доменов, чтобы убедиться, что число контроллеров домена под управлением поддерживаемых версий Windows Server достаточно для числа клиентских компьютеров, которым требуется доступ к ресурсам, защищенным динамическим контролем доступа.  
  
Если домены пользователя и файлового сервера находятся в разных лесах, все контроллеры домена в корне леса файлового сервера должны быть установлены Windows Server 2012 или высокий функциональный уровень.  
  
Если клиенты не признают динамический контроль доступа, должно существовать двустороннее отношение доверия между двумя лесами.  
  
Если утверждения преобразуются, когда они покидают лес, все контроллеры домена в корне леса пользователя должны быть установлены Windows Server 2012 или высокий функциональный уровень.  
  
Файловый сервер под управлением Windows Server 2012 или Windows Server 2012 R2 должен иметь параметр групповой политики, который указывает, должен ли он получать утверждения пользователей для токенов пользователей, у которых нет утверждений. Этот параметр имеет значение по умолчанию для **автоматического**, что приводит к этот параметр групповой политики, чтобы включить **на** при наличии централизованной политики, содержащей утверждения пользователей или устройств для данного файлового сервера. Если файловый сервер содержит избирательные списки управления доступом, включающие утверждения пользователей, необходимо задать эту групповую политику **на**, чтобы сервер знал о необходимости запрашивать утверждения от имени пользователей, которые не предоставляют утверждений при доступе к серверу.
