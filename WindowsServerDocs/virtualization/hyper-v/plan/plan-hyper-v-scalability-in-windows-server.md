# <a name="plan-for-hyper-v-scalability-in-windows-server-2016"></a>Планирование масштабируемость Hyper-V в Windows Server 2016

> Область применения. Windows Server 2016
  
В этой статье приводятся сведения о максимальной конфигурации для компонентов, вы можете добавлять и удалять на узле Hyper-V или виртуальных машин, например виртуальных процессоров или контрольные точки. При планировании развертывания, рассмотрите возможность максимальные ограничения для каждой виртуальной машины, а также тех, которые применяются к узлу Hyper-V. 

Максимальные значения для логических процессоров и памяти являются крупнейших увеличивается с Windows Server 2012, в ответ на запросы для поддержки новых сценариев, таких как машины обучения и анализ данных. Блог Windows Server недавно опубликовала результатов производительности виртуальной машины с Exchange 5.5 терабайт памяти и 128 виртуальных процессоров, работающих с базы данных в памяти в 4 ТБ. Производительность была более 95% производительности физического сервера. Дополнительные сведения см. в разделе [крупномасштабных производительности виртуальной Машины Windows Server 2016 Hyper-V для обработки транзакций в памяти](https://blogs.technet.microsoft.com/windowsserver/2016/09/28/windows-server-2016-hyper-v-large-scale-vm-performance-for-in-memory-transaction-processing/). Другие числа похожи на те, которые применяются к Windows Server 2012. \(Максимальные значения для Windows Server 2012 R2 были так же, как Windows Server 2012.\) 
  
> [!NOTE]  
> Сведения о System Center Virtual Machine Manager (VMM), см. в разделе [Virtual Machine Manager](https://technet.microsoft.com/system-center-docs/vmm/vmm). VMM — это продукт Майкрософт, предназначенный для управления виртуализированным центром обработки данных, который продается отдельно.  
  
## <a name="maximums-for-virtual-machines"></a>Максимальные ограничения для виртуальных машин  
Эти значения применяются к каждой виртуальной машины. Не все компоненты доступны в обоих поколения виртуальных машин. Сравнение поколений, см. в разделе [следует ли создавать виртуальные машины поколения 1 или 2 в Hyper-V?](should-i-create-a-generation-1-or-2-virtual-machine-in-hyper-v.md) 
  
|Component|Максимум|Примечания|  
|-------------|-----------|---------|  
|Контрольные точки|50|Реальное количество может быть ниже в зависимости от доступного объема хранения. Каждая контрольная точка сохраняется как AVHD-файле, который использует физического хранилища.|  
|Память|12 ТБ для поколения 2; <br>1 ТБ для поколения 1|Сверьтесь с требованиями для конкретной операционной системы, чтобы определить минимальный и рекомендуемый объемы.|  
|Последовательные порты (COM)|2|Нет.|  
|Объем физических дисков, напрямую подключенных к виртуальной машине|Различается|Максимальный размер зависит от операционной системы на виртуальной машине.|  
|Виртуальные адаптеры Fibre Channel|4|Мы рекомендуем подключать каждый виртуальный адаптер Fibre Channel к отдельной виртуальной сети SAN.|  
|Виртуальные гибкие диски|1 виртуальный гибкий диск.|Нет.|
|Емкость виртуального жесткого диска|64 ТБ для формата VHDX;<br>2 040 ГБ для формата виртуального жесткого диска|Каждый виртуальный жесткий диск хранится на физическом носителе в виде файла .VHDX или .VHD в зависимости от используемого формата виртуального жесткого диска.|  
|Виртуальные IDE-диски|4|Загрузочный диск (иногда называется загрузочный диск) должен быть подключен к одному из устройств IDE. Диск запуска может быть виртуальным жестким диском или физическим диском, напрямую подключенным к виртуальной машине.|  
|Виртуальные процессоры|240 для поколения 2;<br>64 для поколения 1;<br>320, доступных для размещения операционной системы (корневой раздел)|Число виртуальных процессоров, поддерживаемых операционной системой на виртуальной машине, может быть ниже. Дополнительные сведения см. в разделе сведения, приведенные для конкретной операционной системы.|
|Виртуальные SCSI-контроллеры|4|Для использования виртуальных SCSI-устройств требуется службы integration services, которые доступны для поддерживаемых операционных систем. Дополнительные сведения, в которых поддерживаются операционные системы, см. в разделе [виртуальных машин поддерживается Linux и FreeBSD](../Supported-Linux-and-FreeBSD-virtual-machines-for-Hyper-V-on-Windows.md) и [операционной системы на виртуальных машинах Windows поддерживается](../supported-windows-guest-operating-systems-for-hyper-v-on-windows.md).|  
|Виртуальные SCSI-диски|256|Каждый SCSI-контроллер поддерживает до 64 дисков; это означает, что в каждой виртуальной машине может быть настроено до 256 виртуальных SCSI-дисков (4 контроллера x 64 диска).|  
|Виртуальные сетевые адаптеры|всего 12:<br> — 8 Hyper-V сетевые адаптеры<br>— 4 традиционных сетевых адаптеров|Конкретного сетевого адаптера Hyper-V обеспечивает более высокую производительность и требует драйвера, включенного в службах integration services. Дополнительные сведения см. в разделе [Планирование сетей Hyper-V в Windows Server](plan-hyper-v-networking-in-windows-server.md).|

Дополнительные сведения об обновлениях для отказоустойчивой кластеризации, включая новые функции для виртуальных машин, см. в разделе [новые возможности отказоустойчивой кластеризации в Windows Server 2016](../../../failover-clustering/whats-new-in-failover-clustering.md).

|-------------|-----------|---------|  
|Узлы на один кластер|64|Определите число узлов, которые следует зарезервировать для отработки отказов, а также выполнения задач обслуживания, например применения обновлений. Рекомендуется предусмотреть ресурсоемкость, позволяющую выделить 1 резервный узел для отработки отказов; это означает, что он будет бездействовать до момента отработки отказа на другом узле. (Такие узлы иногда называются пассивными.) При необходимости резервирования дополнительных узлов это число можно увеличить.
