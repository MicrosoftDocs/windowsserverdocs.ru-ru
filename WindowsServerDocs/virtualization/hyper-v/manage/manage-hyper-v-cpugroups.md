# <a name="virtual-machine-resource-controls"></a>Элементы управления ресурсами виртуальной машины

В этой статье описываются элементы управления ресурсов и изоляцию Hyper-V для виртуальных машин.  Эти возможности, которые мы будем называть групп ЦП виртуальной машины или просто «ЦП groups», были введены в Windows Server 2016.  Группы ресурсов ЦП позволяют администраторам Hyper-V, чтобы лучше управлять и распределять ресурсы ЦП узла по гостевых виртуальных машин.  С помощью групп ЦП, администраторы Hyper-V могут:

* Создание групп виртуальных машин с каждой группой необходимости выделение ресурсов для узла виртуализации общее количество ресурсов ЦП, общих для всей группы. Это позволяет администратору узла для реализации классов служб для различных типов виртуальных машин.

* Задания ограничений ресурсов ЦП для определенных групп. «Ограничение группы» задает верхнюю границу для узла ресурсов ЦП, вся группа может потреблять, эффективное применение нужного класса службы для этой группы.

* Ограничить группу ЦП для выполнения только на определенное число процессоров компьютера узла. Это можно использовать для изоляции виртуальных машин, принадлежащих к разным группам ЦП друг от друга.

## <a name="managing-cpu-groups"></a>Управление группами ЦП

Групп ЦП осуществляется через службу вычислений узла Hyper-V, то есть HCS. Хорошее описание HCS, его происхождения, ссылки на API-интерфейсы HCS и многое другое можно найти в блоге группы Microsoft Virtualization в учет [введение в службы вычислительных узлов (HCS)](https://blogs.technet.microsoft.com/virtualization/2017/01/27/introducing-the-host-compute-service-hcs/).

>[!NOTE] 
>Только HCS может использоваться для создания и настройки групп ЦП; приложение диспетчера Hyper-V, управление интерфейсов WMI и PowerShell не поддерживают групп ЦП.

Корпорация Майкрософт предоставляет командной строки служебная программа, cpugroups.exe, на [центра загрузки Майкрософт](https://go.microsoft.com/fwlink/?linkid=865968) с использованием интерфейса HCS для управления группами ЦП.  Эта программа также может отображаться топологии ЦП узла.

## <a name="how-cpu-groups-work"></a>Принципах работы групп ЦП

Распределение узла вычислительных ресурсов в группах ЦП обеспечивается низкоуровневой оболочки Hyper-V, используя ограничение вычисляемые группы. Ограничение ЦП группы — это часть с общей производительностью ЦП, для группы ресурсов ЦП. Значение политики авторизации подключений группы зависит от того, класс группы или уровнем приоритета. Ограничение вычисляемые группы может рассматриваться как «количество LP's стоит времени ЦП». Совместно этого бюджета группы, если только одной виртуальной Машины были активны, но использует ЦП для всей группы для себя.

Ограничение ЦП группы вычисляется как G = *n* x *C*, где:

    *G* is the amount of host LP we'd like to assign to the group
    *n* is the total number of logical processors (LPs) in the group
    *C* is the maximum CPU allocation — that is, the class of service desired for the group, expressed as a percentage of the system’s total compute capacity

Например рассмотрим группу ЦП, 4 логических процессоров (LPs) и ограничение 50%.

    G = n * C
    G = 4 * 50%
    G = 2 LP's worth of CPU time for the entire group

В этом примере группе ЦП G выделяется 2 LP's стоит времени ЦП.  

Обратите внимание, что группы ограничение применяется независимо от количества виртуальных машин или виртуальных процессоров, связанное с группой, а также независимо от состояния (например, завершение работы или запуска) виртуальных машин, назначенных группе ЦП. Таким образом каждой виртуальной Машины, привязаны к одной и той же группе ресурсов ЦП, получите небольшую часть всего ЦП группы, и это приведет к изменению с количеством виртуальных машин, присоединенных к группе ресурсов ЦП. Таким образом виртуальных машин привязаны или несвязанных виртуальных машин из группы ресурсов ЦП, общее ограничение ЦП группы необходимо откорректировать и задать для поддержания наконечника итоговый каждой виртуальной Машины требуемого. Управление администратора или виртуализации узла виртуальной Машины программное обеспечение отвечает за управление caps группы так, чтобы обеспечить выделение ресурсов ЦП требуемой каждой виртуальной Машины.

## <a name="example-classes-of-service"></a>Пример классов службы

Давайте рассмотрим несколько простых примеров. Для начала Предположим, что хотели бы администратора узла Hyper-V поддерживает два уровня службы для гостевых виртуальных машин:

1. С низким быстродействием уровень «C». Мы предоставим этот уровень 10% от всего узла вычислительные ресурсы.

1. На уровне среднего уровня «B». Этот уровень выделяется 50% от всего узла вычислительные ресурсы.

На этом этапе в нашем примере мы будем assert, никакие другие элементы управления ресурсами ЦП используются, например caps отдельных виртуальных Машин, весовые коэффициенты и оставляет за собой.
Тем не менее caps отдельных виртуальных Машин важны, как будет видно чуть позже.

Проще говоря, предположим, каждая виртуальная машина имеет 1 вице-Президент, а также имеет 8 LPs нашего основного приложения. Мы начнем с пустым узлом.

Чтобы создать на уровне «B», adminstartor узла задает ограничение группы на 50%:

    G = n * C
    G = 8 * 50%
    G = 4 LP's worth of CPU time for the entire group

Администратор узла добавляет один уровень «B» виртуальной Машины.
На этом этапе наши «B» уровень виртуальной Машины можно использовать не более 50% затраченных ЦП узла или в аналогичной 4 LPs в нашей системе пример.

Теперь администратор добавляет второй «уровня "" B» виртуальной Машины. Выделение группы ЦП — распределяется равномерно между всеми виртуальными машинами. У нас всего две виртуальные машины в группы B, поэтому теперь каждая виртуальная машина получит половины группы B всего 50%, 25% или эквивалент 2 LPs стоит вычислительного времени.

## <a name="setting-cpu-caps-on-individual-vms"></a>Установка ограничений по ЦП на отдельных виртуальных машинах

Помимо крепления группы каждой виртуальной Машины также могут иметь отдельные «виртуальной Машины cap». Элементы управления ресурсами ЦП для каждой виртуальной Машины, включая ограничение ЦП, вес и резервирование, были частью Hyper-V до сегодняшнего дня.
В сочетании с ограничением группы виртуальных Машин ограничение указывает максимальный объем ресурсов ЦП, который можно получить каждого вице-Президент, даже если в группе ресурсов ЦП, доступных.

Например администратор узла может потребоваться разместить политику авторизации подключений виртуальной Машины 10% на виртуальных машинах «C».
Таким образом, даже если большинство вице-президенты «C» в режиме простоя, каждый вице-Президент никогда не пробовал более чем на 10%.
Без ограничения виртуальной Машины виртуальные машины «C» рационально удалось добиться производительности, кроме режимов, допустимый уровнем их.

## <a name="isolating-vm-groups-to-specific-host-processors"></a>Изоляция группы виртуальных Машин к процессорам конкретного узла

Администраторов узла Hyper-V также может потребоваться возможность выделить вычислительные ресурсы для виртуальной Машины.
Например, предположим, нужно было Предложить категории "премиум" Администратор «A» виртуальной Машины с ограничением класса 100%.
Эти виртуальные машины уровня "премиум" также требуется минимальный планирования задержек и дрожание: возможно; то есть они могут не отмены запланировать с любой другой виртуальной Машине.
Чтобы обеспечить такое разделение, группу ресурсов ЦП можно настроить с определенной сопоставление и LP.

Например в соответствии с «A» виртуальной Машины на узле в нашем примере, администратор будет создать новую группу ресурсов ЦП и установить соответствие процессоров группы к подмножеству LPs главного приложения.
Группы B и C будут привязаны ко оставшиеся LPs.
Администратор может создать одну виртуальную Машину в группе A, будет иметь монопольный доступ для всех LPs из группы A, при предположительно нижний уровень группы B и C будут входить в одну оставшиеся LPs.

## <a name="segregating-root-vps-from-guest-vps"></a>Разделение вице-президенты корневой из гостевой вице-президенты

По умолчанию Hyper-V будет создавать корневые вице-Президент на каждый базовый физический LP.
Эти VPs корневой являются строго сопоставленные 1:1 с системой LPs и не выполняйте миграцию, т. е каждого корня вице-Президент всегда будет выполняться в одной физической LP.
Вице-президенты гостевой может быть запущена на любые доступные LP и выполнения будут совместно использовать с корнем вице-президенты.

Тем не менее возможно желательно к корню никак не связано действие вице-Президент из вице-президенты гостевой.
Рассмотрим наш пример выше где мы реализуем виртуальная машина уровня «A» уровня "премиум".
Чтобы убедиться, что вице-президенты «A» виртуальной Машины имеют Минимальная возможная задержка и дрожание «:», а также планирования, мы бы хотели запускать их на выделенном наборе LPs и убедитесь, что корневой не затрагивает эти LPs.

Это можно сделать с помощью сочетания конфигурации «minroot», которая ограничивает раздел операционной системы узла запущен на подмножестве общего объема системной логических процессоров, а также один или несколько родственных групп ЦП.

Узел виртуализации можно настроить для ограничения разделу узла к определенной LPs с одной или нескольких групп ЦП, оставшиеся LPs в соответствие.
Таким образом секции корневой и гостевой запускаются на выделенных ресурсов ЦП и полностью изолированным, без использования общих ресурсов ЦП.

Дополнительные сведения о конфигурации «minroot», см. в разделе [управления ресурсами ЦП узла Hyper-V](https://docs.microsoft.com/windows-server/virtualization/hyper-v/manage/manage-hyper-v-minroot-2016).

C:\vm\tools>CpuGroups.exe SetVmGroup /VmName:g1 /GroupId:36AB08CB-3A76-4B38-992E-000000000001

### <a name="example-11--attempting-to-delete-a-non-empty-cpu-group"></a>Пример 11 – попытка удалить непустой группы ЦП

Только пустые групп ЦП — то есть групп ЦП, не имеющий привязан виртуальных машин — могут быть удалены.
Попытка удалить группу ЦП непустой завершится сбоем.

```console
C:\vm\tools>CpuGroups.exe DeleteGroup /GroupId:36ab08cb-3a76-4b38-992e-000000000001
(null)
Failed with error 0xc0350070
```

### <a name="example-12--unbind-the-only-vm-from-a-cpu-group-and-delete-the-group"></a>Пример 12 – Отмена привязки только виртуальную Машину из группы ресурсов ЦП и удалить группу

В этом примере мы будем использовать несколько команд проверьте группу ЦП, удалите одной виртуальной Машины, входящие в эту группу, а затем удалить группу.

Во-первых Перечислим виртуальных машин в нашей группе.

```console
C:\vm\tools>CpuGroups.exe GetGroupVms /GroupId:36AB08CB-3A76-4B38-992E-000000000001
CpuGroupId                           VmName                                VmId
------------------------------------ ------ ------------------------------------
36AB08CB-3A76-4B38-992E-000000000001     G1 F699B50F-86F2-4E48-8BA5-EB06883C1FDC
```

Мы видим, что только одной виртуальной Машины, с именем G1, принадлежит к этой группе.
Давайте удалим G1 виртуальной Машины из нашей группы, указав идентификатор группы Виртуальных машин в значение NULL.

```console
C:\vm\tools>CpuGroups.exe SetVmGroup /VmName:g1 /GroupId:00000000-0000-0000-0000-000000000000
```

И проверить изменение...

```console
C:\vm\tools>CpuGroups.exe GetVmGroup /VmName:g1
VmName                                 VmId                           CpuGroupId
------ ------------------------------------ ------------------------------------
    G1 F699B50F-86F2-4E48-8BA5-EB06883C1FDC 00000000-0000-0000-0000-000000000000
```

Теперь, когда группа пуста, мы ее можно безопасно удалить.

```console
C:\vm\tools>CpuGroups.exe DeleteGroup /GroupId:36ab08cb-3a76-4b38-992e-000000000001
```

И убедитесь, что наша группа удаляется.

```console
C:\vm\tools>CpuGroups.exe GetGroups
CpuGroupId                          CpuCap                     LpIndexes
------------------------------------ ------ -----------------------------

36AB08CB-3A76-4B38-992E-000000000003 65536  12,13,14,15
36AB08CB-3A76-4B38-992E-000000000004 65536 24,25,26,27,28,29,30,31
```
