# <a name="windows-authentication-concepts"></a>Основные понятия проверки подлинности Windows

>Область применения. Windows Server (полугодовой канал), Windows Server 2016

Общие сведения об этом справочном разделе описываются основные понятия, на которых основан проверки подлинности Windows.

Проверка подлинности — это процесс проверки удостоверения объекта или пользователя. Задача проверки подлинности объекта заключается в проверке подлинности объекта. При проверке подлинности пользователя, призван убедитесь, что пользователь не посторонним лицом.

В контексте сети проверка подлинности — это подтверждение удостоверения сетевого приложения или ресурса. Как правило удостоверение подтверждается операцией шифрования, использующий, либо ключ только пользователь знает (как при использовании шифрования с открытым ключом) или общего ключа. Во время проверки подлинности на стороне сервера выполняется сравнение подписанных данных с известным криптографическим ключом для проверки попытки проверки подлинности.

Благодаря тому что криптографические ключи хранятся в безопасном центральном местоположении, процесс проверки подлинности можно масштабировать и поддерживать. Active Directory — это рекомендованная и по умолчанию технология хранения информации об удостоверениях, включающие криптографических ключей, являются учетными данными пользователя. Служба каталогов Active Directory необходима для реализаций NTLM и Kerberos по умолчанию.

Технологии проверки подлинности варьируются от простого входа для операционной системы или входа в службу или приложение, которое определяет то, что только пользователь знает, такие как пароль, чтобы более мощные механизмы безопасности, как использовать что-нибудь на основе пользователей, has'such пользователя как маркеры, сертификаты открытого ключа, рисунки или биологических атрибуты. В бизнес-среде пользователи могут открывать множество приложений на различных типах серверов из одного или многих местоположений. Поэтому проверка подлинности должна поддерживать среды для других платформ и других операционных систем Windows.

## <a name="security-principals-and-accounts"></a>Субъекты безопасности и учетных записей
В Windows любой пользователь, службы, группы или компьютера, которая может инициировать действие является субъектом безопасности. Субъекты безопасности имеют учетные записи, которые могут быть локальными для компьютера или домена. Например присоединенных к домену компьютеры Windows может участвовать в сетевой домен, связываясь с контроллером домена, даже в том случае, если человека пользователь не вошел в систему. Инициировать обмен данными, компьютер должен иметь активную учетную запись в домене. Прежде чем принимать подключения от компьютера, в локальный центр безопасности на контроллере домена проверяет подлинность удостоверения компьютера и затем определяет контекст безопасности компьютера, как и в случае относящихся к субъекту безопасности человека. Этот контекст безопасности определяет удостоверение и возможности пользователя или службы на конкретный компьютер или пользователя, службы, группу или компьютер в сети. Например он определяет ресурсы, например общей папки или принтер, к которым можно получить доступ, а также действия, такие как чтение, запись или изменение, которое может быть выполнено пользователем, службы или компьютера, на этот ресурс. Дополнительные сведения см. в разделе [субъекты безопасности](https://technet.microsoft.com/itpro/windows/keep-secure/security-principals).

Учетная запись — средства обнаружения заявитель--пользователя или службы--запрос доступа или ресурсов. Компьютер, который владеет подлинный passport может иметь учетную запись со страной узла. Пользователи, группы пользователей, объектов и служб можно все имеют отдельные учетные записи или совместно использовать учетные записи. Учетные записи может быть членом группы и могут назначаться права и разрешения. Учетные записи могут быть ограничены локального компьютера, рабочей группы, сети или быть членами домена.

Встроенные учетные записи и группы безопасности, в которых они являются членами, определяются в каждой версии Windows. С помощью групп безопасности, можно назначить те же разрешения безопасности для многих пользователей, которые успешно прошли проверку подлинности, что упрощает администрирование доступа. Правила для выдачи паспорта может потребоваться на компьютер должны быть назначены определенных групп, такие как бизнеса, туристов или для государственных организаций. Этот процесс гарантирует разрешения согласованной системы безопасности среди всех участников группы. С помощью групп безопасности для назначения разрешений означает, что доступ к контроль над ресурсами остается постоянным и удобные в управлении и аудита. Добавление и удаление пользователей, которым требуется доступ из соответствующие группы безопасности, при необходимости, можно свести к минимуму частоту изменений для управления доступом (ACL).

Автономные управляемые учетные записи служб и виртуальные учетные записи были введены в Windows Server 2008 R2 и Windows 7, чтобы предоставить необходимые приложения, такие как Microsoft Exchange Server и Internet Information Services (IIS), изоляцию собственным доменом учетные записи, устраняя необходимость в ручном администрировании имени участника-службы (SPN) и учетных данных для этих учетных записей. ## <a name="delegated-authentication"></a>Делегированная аутентификация
Для использования travel аналогию, странах может выдавать те же права доступа ко всем членам официальный государственных делегирования, так же, пока делегаты, хорошо известны. Это делегирование позволяет действовать полномочия другому члену один член. В Windows делегирование проверки подлинности происходит при сетевой службой, принимает запрос на аутентификацию от пользователя и предполагает наличие удостоверение пользователя для установки нового подключения второй сетевой службы. Чтобы обеспечить поддержку делегирование проверки подлинности, необходимо установить серверы переднего плана или первого уровня, например веб-серверы, которые отвечают за обработку запросов проверки подлинности клиента и серверной части и n уровневых серверы, например больших баз данных, которые отвечают за хранение информации. Вы можете делегировать право настроить делегирование проверки подлинности для пользователей в вашей организации, чтобы сократить административные нагрузки на администраторов.

Установив службы или компьютер доверенным для делегирования, позвольте данной службы или компьютера завершить делегирование проверки подлинности, получения билетов для пользователя, который выполняет запрос и получить доступ к информации для этого пользователя. Эта модель ограничивает доступ к данным на внутренних серверах только для этих пользователей или служб, присутствует учетные данные с помощью маркеров управления надлежащих прав доступа. Кроме того он позволяет аудит доступа к этих ресурсов серверной части. Требуя доступности всех данных с помощью учетных данных, которые делегируются на сервер для использования от имени клиента, убедитесь, что сервер не могут быть перехвачены и что можно получить доступ к конфиденциальной информации, хранятся на других серверах. Делегирование проверки подлинности полезна для многоуровневых приложениях, предназначенных для использования возможности единого входа на нескольких компьютерах.

### <a name="authentication-in-trust-relationships-between-domains"></a>Проверка подлинности в доверительных отношений между доменами
Большинство организаций, имеющих более одного домена, которые имеют законного должны пользователям доступ к общим ресурсам, которые находятся в другом домене, так же, как компьютер может переместиться в разных регионах в стране. Требует управления доступом, что пользователи в одном домене можно также выполнять проверку подлинности и прав на использование ресурсов в другом домене. Для обеспечения проверки подлинности и авторизации между клиентами и серверами в разных доменах, должно существовать отношение доверия между двумя доменами. Отношения доверия являются базовой технологии, по которому безопасных подключений Active Directory происходят и являются неотъемлемой безопасности компонентом архитектуры сети Windows Server.

Если существует отношение доверия между двумя доменами, механизмов проверки подлинности для каждого домена уверены, проверок подлинности, поступающих из другого домена. Отношения доверия помогают обеспечить управляемый доступ к общим ресурсам в домене ресурсов--доверяющем домене--, проверяя, проверку подлинности на входящие запросы поступают из доверенного центра--доверенного домена. Таким образом отношения доверия выступать в качестве мостов, которые позволяют только проверить travel запросы проверки подлинности между доменами.

Как определенные доверия передает запросы на проверку подлинности зависит от того, как оно настроено. Отношения доверия может быть односторонней связи, предоставляя доступ к ресурсам в доверяющем домене из доверенного домена или двустороннее, предоставляя доступ к ресурсам в другом домене из каждого домена. Отношения доверия, также являются либо Нетранзитивное, в котором существует вариантов доверие только между доменами партнеров два отношения доверия, либо транзитивными, в этом случае доверия автоматически распространяется на другие домены, которым доверяет участники.

Сведения о том, как работает отношение доверия, см. в разделе [как домены и рабочие отношения доверия леса](https://technet.microsoft.com/library/cc773178(v=ws.10).aspx).

### <a name="constrained-delegation"></a>Ограниченное делегирование
Ограниченное делегирование позволяет администраторам указывать и обеспечивать границы доверия приложений с помощью ограничения области, в которой службы приложения могут действовать от имени пользователя. Можно указать определенных служб, из которых компьютер, который является доверенным для делегирования могут запрашивать ресурсы. Гибкость для ограничения прав авторизации для служб повышает проектирование системы безопасности приложения за счет сокращения возможностей для компрометации с недоверенных служб.

Дополнительные сведения об ограниченном делегировании см. в разделе [Kerberos ограниченного делегирования Общие сведения о](../kerberos/kerberos-constrained-delegation-overview.md).
