# <a name="windows-authentication-architecture"></a>Архитектура проверки подлинности Windows

>Область применения. Windows Server (полугодовой канал), Windows Server 2016

В этой обзорной статье для ИТ-специалистов объясняется основные архитектурные схемы для проверки подлинности Windows.

Проверка подлинности — это процесс, по которому система проверяет, входа в систему или учетные данные пользователя. Имя пользователя и пароль сравниваются со списком доступа, и в случае обнаружения совпадения, доступа к область, указанную в списке разрешений для этого пользователя.

В рамках расширяемой архитектуры операционных систем Windows Server реализовать набор поставщиков поддержки безопасности проверки подлинности, в том числе Negotiate, Kerberos протокола, NTLM, Schannel (безопасный канал) и хэш-кода по умолчанию. Протоколы, используемые этими поставщиками включить проверку подлинности пользователей, компьютеров и служб, и процесс проверки подлинности позволяет авторизованным пользователям и службам для доступа к ресурсам в безопасном режиме.

В Windows Server приложений проверять подлинность пользователей с помощью SSPI для абстрагирования вызовов для проверки подлинности. Таким образом разработчикам не требуется понять сложности конкретных протоколов проверки подлинности или построения протоколы проверки подлинности в свои приложения.

Операционные системы Windows Server включают набор компонентов безопасности, которые составляют модель безопасности Windows. Эти компоненты убедитесь, что приложения не может получить доступ к ресурсам без проверки подлинности и авторизации. Элементы архитектуры проверки подлинности в следующих разделах.

### <a name="local-security-authority"></a>Локальная система безопасности
Local Security Authority (LSA) — это защищенная подсистема, выполняет проверку подлинности и входа пользователей на локальном компьютере. Кроме того локальный администратор безопасности содержит сведения обо всех аспектах безопасности локальной на компьютере (эти аспекты, совокупно называются локальной политики безопасности). Он также предоставляет различные службы для трансляции имена и идентификаторы безопасности (SID).

Подсистема безопасности отслеживает этой политики безопасности и учетные записи, в системе компьютера. В случае домен контроллера, эти политики и учетных записей, которые действуют для домена, в котором расположен контроллер домена. Эти политики и учетные записи хранятся в Active Directory. Подсистема LSA предоставляет службы для проверки доступа к объектам, проверка права пользователя и создание сообщения аудита.

### <a name="security-support-provider-interface"></a>Интерфейс поставщика поддержки безопасности
Интерфейс поставщика поддержки безопасности (SSPI) — это API, который получает встроенной безопасности службы для проверки подлинности, целостность, конфиденциальность сообщений и безопасности качества обслуживания для любой протокол распределенного приложения.

SSPI представляет собой реализацию универсального безопасности API службы (GSSAPI). SSPI предоставляет механизм, по которому распределенного приложения можно вызвать один из нескольких поставщиков безопасности для получения подключение с проверкой подлинности без ведома различными аспектами безопасности протокола.
