# <a name="authentication-policies-and-authentication-policy-silos"></a>Политики проверки подлинности и приемники команд политик проверки подлинности

>Область применения: Windows Server (канал точками годовой), Windows Server 2016

В этом разделе для ИТ-специалистов описываются приемники команд политик проверки подлинности и политики, которые могут заключить учетную запись в подобные приемники. Он также объясняется использование политик проверки подлинности для ограничения области действия учетных записей.

Приемники команд политик проверки подлинности и сопутствующие политики позволяют высокопривилегированные учетные данные в системы, актуальные лишь для определенных пользователей, компьютеров или служб. Приемники команд можно определены и управлять ею в доменных службах Active Directory (AD DS), используя Центр администрирования Active Directory и командлеты Windows PowerShell для Active Directory.

Приемники команд политик проверки подлинности — это контейнеры, которым администраторы могут присваивать учетные записи пользователей, учетные записи компьютеров и учетных записей служб. Набором учетных записей могут управлять затем политики проверки подлинности, которые были применены к этот контейнер. Это снижает необходимость отслеживания администратором доступа отдельных учетных записей к ресурсам и помогает предотвратить доступ к другим ресурсам через кражи учетных данных пользователей-злоумышленников.

Возможности, появившиеся в Windows Server 2012 R2 позволяют вам создавать приемники команд политик, которые включаться определенные высокопривилегированные пользователи проверки подлинности. Затем можно назначить для этого контейнера политики проверки подлинности, чтобы ограничить использование привилегированных учетных записей в домене. Если учетные записи в группу защищенных пользователей, применяются дополнительные элементы управления, такие как исключительное использование протокола Kerberos.

Благодаря этим возможностям можно ограничить использование ценных учетных записей высокого уровня ценности узлами. Например может создать новый лес приемник команд для администраторов, содержащий предприятия, схемы и Администраторы домена. Затем вы можете назначить приемнику политика проверки подлинности, чтобы пароль и проверки подлинности на основе смарт-карты из системы, отличные от контроллеров домена и консолей администрирования не удастся.

Сведения о настройке политик проверки подлинности и приемники команд политик проверки подлинности см. в разделе [Настройка защищенных учетных записей](how-to-configure-protected-accounts.md).

Вы можете создать приемник команд, основываясь на требованиях вашей организации. Приемники команд являются объектами Active Directory для пользователей, компьютеров и служб, как определяются схемой, представленной в следующей таблице.

### <a name="about-authentication-policies"></a>Политики проверки подлинности
Политики проверки подлинности определяет протокол билет предоставления билета (TGT) время жизни билета Kerberos и проверки подлинности условия управления доступом для разных типов учетных записей. Политика выполняется на основе и элементы управления в Доменных службах Active Directory контейнера, называемые приемник команд политик проверки подлинности.

Политики проверки подлинности управляют следующими:

-   Время жизни TGT для учетной записи, которая задается продлить.

-   Критерии, которым учетные записи устройства должны соответствовать, чтобы войти с помощью пароля или сертификата.

-   Критерии, которым должны удовлетворять для проверки подлинности для служб, работающих в составе учетных записей пользователей и устройств.

Тип учетной записи Active Directory определяет от роли вызывающей стороны выделяют следующие:

-   **Пользователь**

    Пользователь должен обязательно члены группы безопасности "защищенные пользователи", что по умолчанию блокирует попытки пройти проверку подлинности с помощью протокола NTLM.

    Политики можно настроить для меньшее значение время существования TGT учетной записи пользователя или ограничить устройства, к которым можно подключиться учетной записи пользователя. Доступны широкие можно настроить политики проверки подлинности для управления критериями подключения пользователей и их устройств для удовлетворения для проверки подлинности в службе.

    Дополнительные сведения см. [группу защищенных пользователей](protected-users-security-group.md).

-   **Службы**

    Изолированные управляемые учетные записи служб, групповые управляемые учетные записи службы или объект пользовательской учетной записи, который является производным от этих двух типов обслуживания, используются учетные записи. Политик можно установить условия управления, которые используются для ограничения для определенных устройств с удостоверением Active Directory управляемых служб учетные данные учетной записи доступа к устройству. Службы не должны члены группы безопасности "защищенные пользователи", так как все входящие проверка подлинности завершится ошибкой.

-   **Компьютер**

    Используется объект учетной записи компьютера или объекта пользовательской учетной записи, которое является производным от объекта учетной записи компьютера. Политики можно настроить доступ управления условия, которые необходимы для разрешения проверки подлинности учетной записи, в зависимости от параметров пользователя и устройства. Компьютеры не должны члены группы безопасности "защищенные пользователи", так как все входящие проверка подлинности завершится ошибкой. По умолчанию попытки использовать проверку подлинности NTLM отклоняются. Время жизни TGT, не должны быть настроены для учетных записей компьютеров.

> [!NOTE]
> Можно задать политику проверки подлинности для нескольких учетных записей, не связывая политику, чтобы приемник команд политики проверки подлинности. Вы можете использовать эту стратегию, при наличии одной учетной записи для защиты.

**Защищенные учетные записи**

Группа безопасности защищенных пользователей позволяет включить дополнительную ненастраиваемую защиту для устройств и главных компьютеров под управлением Windows Server 2012 R2 и Windows 8.1, а также на контроллерах домена в доменах с основного контроллера домена под управлением Windows Server 2012 R2. В зависимости от режима работы домена учетной записи члены группы безопасности "защищенные пользователи" дополнительной защите благодаря изменениям в способах проверки подлинности, поддерживаемые в Windows.

-   Член группы безопасности "защищенные пользователи" не удается пройти проверку подлинности с помощью NTLM, дайджест-проверка подлинности или CredSSP делегирование учетных данных по умолчанию. На устройстве под управлением Windows 8.1, который использует один из этих поставщиков поддержки безопасности (SSP) пройдет проверку подлинности в домен, когда учетная запись является членом группы безопасности "защищенные пользователи".

-   Протокол Kerberos не будет использовать более слабые типы шифрования DES и RC4 в процессе предварительной проверки подлинности. Это означает, что домен должен быть настроен для поддержки по крайней мере шифрования AES.

-   Учетная запись пользователя не может быть делегирована с Kerberos ограниченное или неограниченное делегирование. Это означает, что старые подключения к другим системам возможны, если пользователь является членом группы безопасности "защищенные пользователи".

-   Время жизни TGT Kerberos по умолчанию равный четырем часам настраивается с помощью политик проверки подлинности и приемников, в которых может осуществляться через Центр администрирования Active Directory. Это означает, что по прошествии четырех часов пользователь должен снова пройти проверку подлинности.

Дополнительные сведения об этой группе безопасности см. в разделе [как защищенные группы "пользователи" работает](protected-users-security-group.md#BKMK_HowItWorks).

**Приемники команд и политики проверки подлинности**

Приемники команд политик проверки подлинности и политики проверки подлинности используют существующую инфраструктуру проверки подлинности Windows. Использование протокола NTLM отклоняются, и используется протокол Kerberos с более новыми способами шифрования. Политики проверки подлинности позволяют дополнить группу безопасности защищенных пользователей, указав способ применения настроить ограничения учетных записей, а также ограничивая учетные записи служб и компьютеров. Политики проверки подлинности применяются при службы проверки подлинности протокол Kerberos (AS) или предоставления билетов (TGS) обмен. Дополнительные сведения о том, как Windows использует протокол Kerberos, и какие изменения были внесены для поддержки политик проверки подлинности и приемники команд политик проверки подлинности см. в разделе:

### <a name="BKMK_HowKerbUsed"></a>Использование протокола Kerberos с политиками и приемники команд политик проверки подлинности
Если учетной записи домена привязана к приемнику команд политику проверки подлинности и пользователь выполняет вход, диспетчер учетных записей безопасности добавляет тип утверждения приемник команд политик проверки подлинности приемник команд, как значение. Это утверждение в учетной записи предоставляет доступ к целевому приемнику команд.

Если политика проверки подлинности и контроллер домена получает запрос службы проверки подлинности для учетной записи домена, контроллер домена возвращает непродлеваемый TGT с установленным временем жизни (если время жизни TGT в домене не короче).

> [!NOTE]
> Учетная запись домена должна иметь настроено время жизни TGT и должны быть либо напрямую связываться с политикой, либо косвенно через принадлежность к.

Если политика проверки подлинности находится в режиме аудита, и контроллер домена получает запрос службы проверки подлинности для учетной записи домена, контроллер домена проверяет, если проверка подлинности разрешение для устройства могут входить предупреждение, если не удается. Политика проверки подлинности в режиме аудита влияет на результат, поэтому запросы проверки подлинности не будут отклонены, если они не отвечают требованиям политики.

> [!NOTE]
> Учетной записи домена необходимо связана с политикой напрямую или косвенно через принадлежность к.

Если политика проверки подлинности и служба проверки подлинности защищена, контроллер домена получает запрос службы проверки подлинности для учетной записи домена, контроллером домена проверяется, если проверка подлинности будет разрешен для устройства. В случае неудачи контроллер домена возвращает сообщение об ошибке и регистрирует событие в журнале.

> [!NOTE]
> Учетной записи домена необходимо связана с политикой напрямую или косвенно через принадлежность к.

Если политика проверки подлинности находится в режиме аудита, и получает запрос службы предоставления билетов контроллером домена для учетной записи домена, контроллер домена проверяет проверки подлинности, основываясь на данных сертификата атрибута привилегий (PAC) для билета запроса, и в случае неудачи регистрируется сообщение с предупреждением. PAC содержит различные типы данных авторизации, включая группы, которые пользователь является членом, права пользователя, и применение политик для пользователя. Эта информация используется для создания маркера доступа пользователя. Если это применяемая политика проверки подлинности обеспечивает проверку подлинности пользователя, устройства или службы, контроллер домена проверяет возможность этой операции, основываясь на данных PAC для билета запроса. В случае неудачи контроллер домена возвращает сообщение об ошибке и регистрирует событие в журнале.

> [!NOTE]
> Учетной записи домена должны быть напрямую связан или друг с другом через принадлежность к политику аудита проверки подлинности, который разрешает проверку подлинности пользователя, устройства или службы,

Можно использовать одну политику проверки подлинности для всех членов приемника команд или вы используете отдельные политики для пользователей, компьютеров и управляемых учетных записей служб.

Политики проверки подлинности можно настроить для каждого приемника команд с помощью консоли администрирования Active Directory или Windows PowerShell. Дополнительные сведения см. в разделе [Настройка защищенных учетных записей](how-to-configure-protected-accounts.md).

### <a name="BKMK_HowRestrictingSignOn"></a>Принципы ограничения входа пользователей в работает
Так как эти политики проверки подлинности применяются к учетной записи, они также распространяются на учетные записи, используемые службами. Этот параметр полезен, если требуется ограничить использование пароля службой для конкретных узлов. Например групповых управляемых учетных записей служб задать каких случаях узлам разрешается получать пароль от доменных служб Active Directory. Тем не менее этот пароль может использоваться с любого узла для начальной проверки подлинности. С помощью условий управления доступом, дополнительный уровень защиты можно повысить путем ограничения пароль набора узлов, которые можно получить пароль.

Если службы, выполняющиеся как система, сетевой службы или других локальных служб подключаются к сетевым службам, они используют учетную запись главного компьютера. Учетные записи компьютеров нельзя ограничить. Поэтому даже если службой используется учетная запись компьютера не предназначен для сервера Windows, его нельзя ограничить.

Чтобы ограничить подключения вход пользователя в систему к определенным узлам требуется контроллер домена для проверки удостоверения узлов. При использовании проверки подлинности Kerberos с защитой Kerberos (входящую в динамический контроль доступа), в центр распределения ключей предоставляется TGT узла, из которого пользователь проходит проверку подлинности. Содержание этого защищенного TGT используется для проверки доступа, чтобы определить возможность подключения узла.

Когда пользователь входит в Windows или вводит учетные данные домена в запрос учетных данных для приложения, по умолчанию, Windows отправляет контроллеру домена незащищенное сообщение AS-REQ. Если пользователь посылает запрос с компьютера, который не поддерживает защиту, например компьютеров под управлением Windows 7 или Windows Vista, запрос терпит неудачу.

Следующий список описывает процесс:

-   Контроллер домена в домене под управлением Windows Server 2012 R2, запрашивает учетную запись пользователя и определяет, если настроена политика проверки подлинности, ограничивающая начальную проверку подлинности, для которой требуются защищенные запросы.

-   Контроллер домена отклоняет запрос.

-   Поскольку требуется защита, пользователь может попытаться выполнить вход с помощью компьютера под управлением Windows 8.1 или Windows 8, для которого поддерживается защита Kerberos с целью повторной попытки входа в систему.

-   Windows обнаруживает, что домен поддерживает защиту Kerberos и посылает защищенное сообщение AS-REQ повторного запроса на вход.

-   Контроллер домена выполняет проверку доступа при помощи настроенных условий управления доступом и сведения об удостоверении операционной системы клиента TGT, которая использовалась для защиты запроса.

-   Если проверка доступа терпит неудачу, контроллер домена отклоняет запрос.

Даже в том случае, если операционная система поддерживает защиту Kerberos, требованиям к контролю доступа могут применяться и должны быть выполнены перед предоставлением доступа. Пользователи войти в Windows или вводят свои учетные данные домена в запросе приложением. По умолчанию Windows отправляет незащищенное сообщение AS-REQ к контроллеру домена. Если пользователь посылает запрос с компьютера, поддерживающего защиту, например Windows 8.1 или Windows 8, политики проверки подлинности производится следующим образом:

1.  Контроллер домена в домене под управлением Windows Server 2012 R2, запрашивает учетную запись пользователя и определяет, если настроена политика проверки подлинности, ограничивающая начальную проверку подлинности, для которой требуются защищенные запросы.

2.  Контроллер домена выполняет проверку доступа при помощи настроенных условий управления доступом и информации об удостоверениях системы TGT, используемый для защиты запроса. Проверка доступа проходит успешно.

    > [!NOTE]
    > Если настроены ограничения устаревших рабочей группы, те также необходимо выполнить.

3.  Контроллер домена отправляет защищенное ответное (AS-REP), и проверка подлинности продолжается.

### <a name="BKMK_HowRestrictingServiceTicket"></a>Принципы ограничения выдачи билетов служб
Если учетной записи не разрешается, и учетной записи пользователя с помощью TGT пытается подключиться к службе (например, открыв приложение, требующее проверки подлинности для службы, который идентифицируется по имя участника службы службы (SPN), происходит следующая последовательность:

1.  При попытке SPN подключиться к SPN1 Windows отправляет контроллера домена, запрашивающему билет службы у spn1, сообщение TGS-REQ.

2.  Выполняет поиск spn1, чтобы найти учетную запись доменных служб Active Directory для службы контроллера домена в домене под управлением Windows Server 2012 R2 и определяет, что учетная запись настроена политика проверки подлинности, ограничивающая выдачу билетов службы.

3.  Контроллер домена выполняет проверку доступа при помощи настроенных условий управления доступом и информации об удостоверениях пользователя TGT. Проверка доступа терпит неудачу.

4.  Контроллер домена отклоняет запрос.

Когда разрешается учетную запись, поскольку учетная запись соответствует условиям управления доступом, заданные политикой проверки подлинности и пытается подключиться к службе учетной записи пользователя с помощью TGT (например, открыв приложение, требующее проверки подлинности к службе, определяемой по SPN службы), происходит следующая последовательность событий:

1.  При попытке подключения к SPN1 Windows отправляет контроллера домена, запрашивающему билет службы у spn1, сообщение TGS-REQ.

2.  Выполняет поиск spn1, чтобы найти учетную запись доменных служб Active Directory для службы контроллера домена в домене под управлением Windows Server 2012 R2 и определяет, что учетная запись настроена политика проверки подлинности, ограничивающая выдачу билетов службы.

3.  Контроллер домена выполняет проверку доступа при помощи настроенных условий управления доступом и информации об удостоверениях пользователя TGT. Проверка доступа проходит успешно.

4.  Контроллер домена отвечает на запрос с ответ службы предоставления билетов (TGS-REP).
